Нагрузочное тестирование (tools/load_testing)

Реальные HTTP-запросы, в том числе с записью в БД. Подробный сценарий политики: Developer: API load testing, ADR 0012.

Раннер не вызывает Python-функцию обработчика напрямую (это был бы другой вид теста): запросы идут по сети, как с curl, чтобы в метрики попадали те же middleware и статусы. Чтобы не дублировать контракт ручки, тела для user строятся через UserCreateRequest из app.schemas.user, путь — USER_HTTP_BASE_PATH в app.api.v1.user.

Архитектура (обзор)

Раннер подхватывает Python-модули из scenarios/, строит план по весам и отправляет запросы. Цепочка совпадает с идеей планирования нагрузки: 

flowchart LR
  subgraph scenarioTree [scenarios_directory]
    W[weights_py]
    U[user_create_py]
    O[observability_http500_py]
  end
  R[runner_py]
  R --> scenarioTree
  R --> H[httpx_Client]
  H --> API[FastAPI]
  API --> DB[(database)]

Расчёт доли сценария

Итоговый вес одного именованного сценария:

flowchart TD
  GW["GROUP_WEIGHTS[group]"]
  SH["SHARE_OF_GROUP[file]"]
  MX["MIX[scenario_key]"]
  F["weight = GW * SH * MX"]
  GW --> F
  SH --> F
  MX --> F

В каждом файле сценариев MIX суммируется в ~1.0; по всем файлам одной группы сумма SHARE_OF_GROUP = ~1.0; GROUP_WEIGHTS в weights.py в сумме ~1.0.

Запуск

Из корня репозитория (виртуальное окружение активировано):

python -m tools.load_testing.runner --dry-run
python -m tools.load_testing.runner --total-requests 200
python -m tools.load_testing.runner --total-requests 200 -q
LOAD_TEST_BASE_URL=http://127.0.0.1:8000 python -m tools.load_testing.runner --delay-ms 0

Во время реального прогона в терминал пишутся строка «Старт» и периодический прогресс [N/total] сценарий → HTTP код; без этого вывод был бы только в конце (долгие паузы из‑за --delay-ms). Флаг -q / --quiet отключает этот прогресс.

Переменные окружения: LOAD_TEST_BASE_URL, LOAD_TEST_API_KEY, LOAD_TEST_DELAY_MS (по умолчанию ~1 с между запросами, чтобы не упираться в лимит API), LOAD_TEST_VERBOSE=1 или флаг -v / --verbose — показать, какие модули пропущены из‑за нулевого веса группы (например observability_5xx).

Если в логе «несовпадения» и 404 / 429

Это обычно не ошибка в коде раннера. 404 на /__loadtest/http500 — маршрут есть только при LOADTEST_HTTP_500=true на сервере; по умолчанию вес группы observability_5xx в scenarios/weights.py равен 0, сценарий не гоняется. 429 — сработал rate limit. Варианты: увеличить --delay-ms; для нагрузки только на своей машине поднять лимит на стороне API — make run-loadtest-api (поднимает API с высоким лимитом, ждёт /ready, запускает python -m tools.load_testing.runner, останавливает API; см. tools/load_testing/run_with_local_api.sh (перед стартом — предупреждение и вопрос «продолжить?»; отмена без прогона; для CI: LOADTEST_SKIP_CONFIRM=1). Лимит для прогонов по умолчанию 1000000000 запросов за окно (переменная API_RATE_LIMIT_REQUESTS_LOADTEST). Долгоживущий сервер с тем же лимитом — make run-loadtest-api-serve. Либо временно увеличить API_RATE_LIMIT_REQUESTS / API_RATE_LIMIT_WINDOW_SECONDS в .env (не для prod).

Структура каталога

  • scenarios/weights.py — GROUP_WEIGHTS (доли групп: user, observability_5xx, …)
  • scenarios/user/create.py — POST /api/v1/user, MIX, SCENARIOS, SHARE_OF_GROUP
  • scenarios/observability/http500.py — GET /__loadtest/http500 (нужен LOADTEST_HTTP_500=true)
  • scenarios/conspectus/ — заготовка под новые ручки (README.txt внутри каталога)

Как добавить ручку

  1. Создай файл tools/load_testing/scenarios/<сущность>/<имя>.py.
  2. Задай GROUP (ключ из weights.py), SHARE_OF_GROUP, MIX (сумма ~1.0), SCENARIOS (ключ → callable(ctx) -> BuiltRequest).
  3. При новой сущности добавь долю группы в GROUP_WEIGHTS.
  4. Ключи сценариев глобально уникальны (например conspectus.create.ok).

Несколько файлов в одной группе

Например user/create.py и user/get.py: сумма SHARE_OF_GROUP по этим файлам должна быть ~1.0; в каждом файле MIX суммируется в ~1.0.

422 при создании пользователя

См. user_payload.apply_break_field и поля в scenarios/user/create.py.
