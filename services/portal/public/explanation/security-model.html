Security model

What you get out of the box

The Study API ships with a defensive default for every category that has historically been the source of API security incidents. The defaults are normative — they are the contract, not a starting point that integrators are expected to relax.

  • Transport. HTTPS-only in any non-dev environment. The local-dev base URL is plain HTTP for convenience and is never reachable from the public internet.
  • Authentication. Long-lived API key in the X-API-Key header (see how-to: authenticate). Missing or wrong key returns 401 SECURITY_AUTH_REQUIRED with no information leak.
  • Rate limiting. Per-key budget; exceeding it returns 429 SECURITY_RATE_LIMIT_EXCEEDED with a Retry-After hint when known. Rate-limit settings live under env vars.
  • Body size cap. Hard upper bound; exceeding returns 413 SECURITY_REQUEST_BODY_TOO_LARGE. Prevents accidental denial-of-service via oversize JSON.
  • CORS. Disabled by default. Operators opt in to specific origins via CORS_ALLOW_ORIGINS; there is no * wildcard in production.
  • Header redaction. Structured logs redact X-API-Key and similar secret headers automatically — your key never leaks via log shipping.
  • Idempotency for writes. See how-to: handle idempotency — the contract that protects you from double-application during retries.

What is your responsibility

  • Store the API key in a secret manager — never in source control, never in build artifacts.
  • Rotate proactively if you suspect leakage. There is no expiry; rotation is operator-driven.
  • Use distinct keys per integration / environment so logs and rate-limit metrics are attributable.
  • Don't log the request body or the response body for write endpoints unless you have stripped sensitive fields client-side.
  • Always include the server-issued request_id when filing a support ticket — it is the fastest way to find the matching server-side trace.

Out of scope (today)

Some features that mature security postures eventually need are deliberately not part of v1: per-call signing, IP allow-lists, mTLS, audit-log export, role-based scope tokens, request-replay detection at the network layer. Each will land — or be intentionally rejected — through its own ADR. None of this is needed to integrate today; it is listed here so a security-review checklist does not get the wrong impression about a missing feature.
