Handle idempotency

Network failures and timeouts make it unsafe to retry a write blindly. The Study API uses an Idempotency-Key contract so a retried POST returns the original outcome instead of duplicating work.

Why it matters

Without a stable idempotency key, you can create two users for one request after a single retry. The contract on this page protects you from that class of bug end-to-end.

The recipe

  1. Generate a unique key per logical operation (UUID v4 is fine).
  2. Send the same key on the original request and on every retry of the same request.
  3. Send a fresh key when the operation is genuinely new.
  4. Store the key locally with the request payload until you receive a definitive response.
KEY=$(uuidgen)
curl -X POST "$BASE_URL/api/v1/user" \
  -H "X-API-Key: $API_KEY" \
  -H "Idempotency-Key: $KEY" \
  -H "Content-Type: application/json" \
      -d '{"system_user_id":"42","system_uuid":"b2c3…","full_name":"Ivan"}'

If you retry the same call within the TTL window, the server replies with the original 201 and original response body — no duplicate row, no extra side effect.

The contract

  • Header: Idempotency-Key. Required on every POST under /api/v1.
  • Key composition: a string ≤ 64 characters; UUID v4 strongly recommended. Free-form is permitted but the namespace is per-route, so keep keys globally unique to be safe across endpoints.
  • Replay: the original response (status + body) is returned. The server compares the SHA-256 of the canonicalised request body to detect mismatches.
  • TTL: 24 hours. After that, the same key may be reused with different payloads.
  • Conflict (409 IDEMPOTENCY_KEY_REUSED_WITH_DIFFERENT_PAYLOAD, COMMON_409): the server has seen this key before, but the body is different — you have either reused a key by mistake or sent a non-canonical body. Generate a fresh key.
  • Missing key (400 IDEMPOTENCY_KEY_REQUIRED, COMMON_400): writes without the header are rejected up front; there is no implicit fallback.

What "logical operation" means

One key per intent, not one key per HTTP call. If your code retries on transient failure (timeout, 5xx, connection reset), reuse the key. If a human user clicks "Create" twice in a UI, that is two intents — two keys. If you need to recreate the same row after a soft-delete, that too is a new intent — a new key.
