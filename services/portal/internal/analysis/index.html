SA portal — System Analysis

What this portal is

The SA portal is the project's explanation-layer documentation: methodology, system design, and — most importantly here — the principles behind how this project does documentation. The audience is hiring panels, new engineers reading the repository for the first time, and senior contributors looking for the canonical rationale behind a convention.

This portal does not own decisions (those live in ADRs and RFCs), nor templates (those live in RFC 0003 and the audit template), nor periodic measurements (those live in audits). It owns the narrative — what we believe, why we believe it, and the concrete file paths where the belief shows up in the code.

Documentation principles

Seven principles describe how documentation is authored, owned, and evolved in this repository. Each principle page follows the same structure: What, Why we use it, Where it applies in this project (with file:line evidence), Anti-patterns we have (an honest list of current violations), and References.

Principle In one sentence
SSOT — Single Source of Truth Every fact, schema, or decision has exactly one authoritative location; everything else is derived or referenced.
Docs as code Documentation lives in the same repository as the code it describes, reviewed and versioned through the same workflow.
Diátaxis Every page belongs to exactly one of four modes — tutorial, how-to, reference, explanation — and stays in that mode.
Decision records (ADR / RFC) Significant decisions are captured as numbered records with Context, Decision, and Consequences sections.
Single authoring model One shared template registry governs all major page types (API specs, screen specs, audits, principles).
Mobile contract Three viewport tiers (≥1025 / 761–1024 / ≤760); typography stays uniform across all of them.
PlantUML only All diagrams are PlantUML — never Mermaid — so they render server-side, version-pinned, and accessible.

See the principles hub for the card view and cross-references.

Practices
