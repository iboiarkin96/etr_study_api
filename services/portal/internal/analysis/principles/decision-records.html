Decision records (ADR / RFC)

What

Significant architectural and engineering decisions are captured as numbered, immutable records. Two related formats are used:

  • ADR (Architecture Decision Record) — captures a decision that has been made. Has Context, Decision, and Consequences sections. Superseded ADRs are not deleted; they remain in the index marked as superseded.
  • RFC (Request for Comments) — captures a proposal under discussion. Has Problem statement, Proposed approach, Alternatives, and Open questions sections. When ratified, an RFC may produce one or more ADRs.

Why we use it

Without decision records, the only way to reconstruct "why does the codebase look like this" is to ask the original author — who may have left, forgotten, or remember it wrong. The decision becomes folklore.

ADRs make the decision durable: anyone reading the repo six months later can find the rationale, the alternatives that were considered, and the trade-offs that were accepted. They also make changing the decision honest — a new ADR supersedes the old one with an explicit justification, instead of silently inverting the codebase.

RFCs separate proposing from deciding: the asynchronous review window forces stakeholders to engage before implementation starts, surfacing concerns when they are still cheap to address.

Where it applies in this project

  • ADR index. governance/adr/index.html currently lists 30+ ADRs spanning docs-as-code, OpenAPI governance, security defaults, testing policy, observability, CI/CD, etc.
  • RFC index. governance/rfc/index.html lists 3 active RFCs: docs search program (RFC 0001, 0002) and documentation authoring model (RFC 0003).
  • Lifecycle, ratification, and badges. ADR 0018 codifies the lifecycle (Proposed / Accepted / Superseded / Deprecated), the ratification process, and visual badges that appear on each ADR's header.
  • Assessment-document format. ADR 0024 defines the format for quality assessments (the audit pages), which use a decision-record-shaped structure.
  • Governance phases. RFC 0004 (governance consolidation phases) orchestrates how ADRs, RFCs, audits, and backlog interact.
  • ADR / RFC templates. governance/adr/0000-template.html and governance/rfc/0000-template.html fix the section order so every new record is shaped the same way.

Anti-patterns we have

  1. Some early ADRs lack a Consequences section. The Consequences section is mandatory in MADR — it records what the team is accepting in exchange for the decision. A small number of older ADRs are thin on consequences; back-fill tracked as Lifecycle & Governance gap.
  2. RFC 0003 still in Proposed state. The documentation-authoring RFC (RFC 0003) is the single authoring model for API specs, screen specs, and templates, but it has not yet been ratified to Accepted. Until it is, its guidance is advisory rather than normative.
  3. No automated cross-reference linting. ADRs cross-reference other ADRs informally; there is no link-checker step in CI that verifies the references resolve. scripts/check_asset_refs.py covers asset paths but not internal ADR/RFC links.
