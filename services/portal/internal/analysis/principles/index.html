Documentation principles

Overview

Seven principles describe how documentation is authored, owned, and evolved in this repository. They are not aspirations — each one is enforced (or honestly admitted as violated) on a concrete page somewhere in the repo, and every principle page lists the file paths that demonstrate the principle in action.

Page structure is uniform: What, Why we use it, Where it applies in this project (bullet list with file:line evidence), Anti-patterns we have (honest list of current violations), References (ADR/RFC/external standards), and a page-history table.

The seven principles

Principle In one sentence Primary references
SSOT — Single Source of Truth Every fact, schema, or decision has exactly one authoritative location; all other appearances are derived or referenced. ADR 0026, ADR 0001
Docs as code Documentation lives in the same repository as the code it describes, reviewed and versioned through the same workflow. ADR 0001
Diátaxis Every page belongs to exactly one of four modes — tutorial, how-to, reference, explanation — and stays in that mode. diataxis.fr
Decision records (ADR / RFC) Significant decisions are captured as numbered records with Context, Decision, and Consequences sections. ADR 0024, RFC 0004, MADR 4.0
Single authoring model One shared template registry governs all major page types (API specs, screen specs, audits, principles). RFC 0003
Mobile contract Three viewport tiers (≥1025 / 761–1024 / ≤760); typography stays uniform across all of them. ADR 0030
PlantUML only All diagrams are PlantUML — never Mermaid — so they render server-side, version-pinned, and accessible. ADR 0020
