Mobile contract

What

The mobile contract is the three-tier viewport canon that every portal page must respect. Each tier is defined by an inclusive width range; the rules at each tier are the same on every page so a reader (and an author) can predict the shape without re-reading the CSS.

Tier Viewport width Sidebar / drawer behaviour
Desktop ≥ 1025 px Persistent left sidebar, prose column capped at the shared max-width token.
Tablet 761 – 1024 px Sidebar collapses; top header gains a drawer-trigger button. Prose uses the full available width.
Mobile ≤ 760 px Drawer is the primary nav; drawer panel visually mirrors the desktop sidebar (not a body-sized "native list view").

Typography is uniform across all three tiers. No font-size rule lives inside a width-conditional media query — body type is identical on a desktop monitor and a phone. The 8-tier font ladder defined in docs.css applies everywhere.

Why we use it

Without a contract, every page picks its own breakpoints, every CSS author re-invents a sidebar collapse, and authors who add new pages either match the closest existing page (cargo cult) or invent something new (drift). After ten pages the project has four different mobile experiences.

Uniform typography across tiers prevents a subtle reading failure: scaling type with viewport produces "fluid" but unpredictable hierarchies. On a tablet, an H2 might collide visually with body type. By holding the ladder constant we keep the visual hierarchy intact at every width and force authors to think in structure (heading levels, lists, tables) rather than tweaking sizes for a tier.

The mobile drawer mirroring the sidebar (not replacing it) is a usability choice: a reader who learned the desktop nav should see the same tree on mobile, not a separate flat list. The cost is a slightly more complex drawer; the benefit is one navigation mental model.

Where it applies in this project

  • ADR 0030 — portal shell token contract. ADR 0030 codifies the shell tokens (max-widths, sidebar widths, breakpoints) shared between internal and public portals. services/frontend/portal/assets/docs-shell.css is the SSOT for these tokens.
  • Normative #mobile-contract anchor. Pages that ship UI work include a section anchored at #mobile-contract describing the page's behaviour at the three tiers. Examples: front/_shared/style-guide.html#mobile-contract, front/components/sticky-toc.html#mobile-contract.
  • Drawer = sidebar visual. services/frontend/portal/assets/docs-drawer.css and the shared docs-sidebar.js runtime ensure the drawer panel renders the same tree, same indentation, same icons as the desktop sidebar.
  • Uniform font ladder. The 8-tier ladder lives in services/frontend/portal/assets/docs.css; body size resolves to ~0.9rem (~15.3 px) identically across viewports. Audits (UI-UX consistency assessment) verify this periodically.
  • Validators check token consumption. scripts/validate_docs_design.py flags pages that re-declare shell tokens instead of consuming them, which is how breakpoint drift is caught early.

Anti-patterns we have

  1. Width-conditional font-size rules creeping back in. The contract forbids font-size inside width media queries; without a dedicated linter, any individual page or component can re-add one in a PR and only an attentive reviewer will catch it. Tracked as a candidate for a stylelint rule.
  2. "Mobile drawer ≠ desktop sidebar" regressions in new components. When a new screen spec adds a custom navigation widget (e.g. a horizontal tab strip), the drawer mirror gets bypassed unless the author wires it through the shared sidebar runtime. A handful of front component pages have had to be rewritten after the fact.
  3. Tablet tier underspecified. The contract names the 761 – 1024 px range, but several pages were originally designed for "desktop or mobile" and only retrofitted to handle the tablet midpoint. Audit pages occasionally still call this out under "Mobile contract" rows in the v2 maturity report.
