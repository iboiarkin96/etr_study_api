Internal docs

PlantUML only

What

Every diagram in this repository is authored in PlantUML. No Mermaid, no draw.io exports, no screenshot of a whiteboard, no Figma-only diagram. PlantUML .puml sources live next to the pages that reference them; SVG artefacts rendered from those sources are committed alongside, server-side-rendered, and served as static assets.

The convention applies uniformly: C4 context / container / component diagrams, sequence diagrams, state machines, ER diagrams, deployment views — all of them are PlantUML, all of them follow the same style and skin parameters.

Why we use it

Mermaid renders client-side in JavaScript. That means: every reader's browser executes the renderer, the diagram pops in after the page paints (layout shift), screen readers don't see it because it's a runtime canvas, and the diagram silently breaks when the Mermaid runtime version changes. None of those failure modes is acceptable for documentation that needs to render the same way for the next five years.

PlantUML renders server-side at build time. The output is a committed SVG with deterministic markup, an <title> and <desc> from the source, and stable file paths that link-checkers can validate. Version-pinning is explicit: the PlantUML jar / container that produced the SVG is named in the build script; no surprise upgrades.

A second benefit is uniform style. One skinparam block sets all diagrams' palette, font, and corner-radius; authors don't pick their own theme per diagram, and the project's diagrams visually belong to the same family.

Where it applies in this project

  • ADR 0020 — C4 / PlantUML diagram style and conventions. ADR 0020 is the canonical decision: PlantUML is mandatory, Mermaid is prohibited, and the C4 model is the default abstraction framework for architectural diagrams.
  • System design diagrams. The SA portal's system-design page (analysis/system-design.html) and the service-catalog Architecture pages (internal/catalog/api/architecture/) consume committed SVGs rendered from PlantUML sources.
  • Sequence diagrams in API operation pages. Pages under internal/api/ include sequence diagrams for non-trivial flows (auth refresh, error propagation, retries) rendered from co-located .puml sources.
  • Diagrams & lightbox component. front/components/diagrams-and-lightbox.html documents how diagrams are embedded — committed SVG plus a lightbox for zoom — and explicitly names PlantUML as the only source format.
  • Periodic diagram audits. The diagram audit prompt (notes/diagram audit promt.txt) verifies that every diagram in the repo has a corresponding .puml source, that the SVG matches the source, and that no Mermaid <pre class="mermaid"> or fenced ```mermaid blocks have leaked in.

Anti-patterns we have

  1. Mermaid leftovers as risk, not current state. A repo-wide grep on 2026-05-11 finds the literal "Mermaid" only inside prose that prohibits it (ADR 0020, audit reports, this principle page) — no actual diagram blocks. The anti-pattern to guard against is silent re-introduction by contributors used to Mermaid in other projects. Mitigation: add a pre-commit grep for ```mermaid and class="mermaid".
  2. No automated source-to-SVG drift check. A diagram's .puml source can diverge from the committed SVG if someone edits one without re-rendering the other. There is no CI step today that re-renders every .puml and diffs against the committed SVG. Tracked as a candidate validator script.
  3. Inconsistent skinparam application. A handful of older diagrams predate the shared style block and use their own fonts or palette. The diagram audit flags these but back-fill is sporadic.
