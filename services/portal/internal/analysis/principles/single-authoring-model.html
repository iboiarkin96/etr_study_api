Single authoring model

What

A single authoring model means that every major page type in this repository — API specs, screen specs, audits, principle pages, ADRs, RFCs, runbooks — is produced from one shared template registry. New pages clone an existing template rather than being designed from scratch, and template changes propagate by convention: the next page authored from that template picks up the updated structure.

The model is normative for shape (section order, naming conventions, page-history block, sidebar registration), not for content. Authors fill in their material; the skeleton is fixed.

Why we use it

Without a single authoring model, each new page is shaped by whoever writes it. After a year the project has six different audit formats, four different ADR layouts, three different ways of writing "Page history". Readers can't predict where to find information; tooling can't extract it; reviewers waste time arguing about structure instead of content.

One template per page type collapses that variance. An ADR always has Context / Decision / Consequences; an audit always has axis tables with the same score columns; a principle page (like this one) always has What / Why / Where / Anti-patterns / References. The reader builds a mental model once and reuses it everywhere.

It also makes templates auditable: if the template changes (e.g. "every page must have a maintainer-ids attribute"), one validator script can check every page that descends from the template instead of every page individually.

Where it applies in this project

  • RFC 0003 — documentation authoring model. governance/rfc/0003-documentation-authoring-model.html is the consolidated rationale and template index. It names the canonical templates and how they relate.
  • ADR / RFC templates. governance/adr/0000-template.html and governance/rfc/0000-template.html fix the section order for every decision record.
  • Audit template. governance/audit/AUDIT_TEMPLATE.html is the canonical skeleton for quality assessments (UI-UX, maturity, bug audits). Codified in ADR 0024.
  • Principle pages (this section). All seven principle pages under internal/analysis/principles/ share the structure What / Why / Where / Anti-patterns / References / See also / Page history. The first principle page authored (ssot.html) seeded the template; later pages clone it.
  • Internal API operation pages. Pages under internal/api/ share an operation skeleton (Endpoint summary, Request, Response, Errors, Sequence, See-also).
  • Screen spec pages. internal/front/screens/*.html share a uniform skeleton (Intent, Layout, States, Tokens, A11y, Tests).
  • "Overview" first-child convention. Sidebar groups in services/frontend/portal/assets/internal-sidebar.js now consistently use Overview as the label of the first child of every group — instead of mixing Hub — X, Index — Y, and ad-hoc labelHtml. The convention is part of the authoring model.

Anti-patterns we have

  1. RFC 0003 still in Proposed state. The authoring model itself is described in an RFC that has not yet been ratified to Accepted. Its guidance is therefore advisory; a future template change does not have a single source of authority to point at. Ratification is tracked as Lifecycle & Governance gap in the v2 maturity audit.
  2. No template-conformance validator. There is no script that checks "every audit page has the canonical section order" or "every principle page has a page-history table". Validators today cover HTML well-formedness, a11y, and design tokens; structural conformance to a template is still done by eye in PR review.
  3. Handbook pages predate the model. Pages under internal/handbook/developer/ were authored before RFC 0003 stabilized and don't follow a uniform skeleton. They mix reference, how-to, and explanation modes within a single page — a Diátaxis violation tracked separately on the Diátaxis principle page.
