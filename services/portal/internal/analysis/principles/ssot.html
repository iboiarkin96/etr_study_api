SSOT — Single Source of Truth

What

SSOT (Single Source of Truth) is the principle that any piece of data, configuration, decision, or documentation must have exactly one authoritative location. Every other appearance — in another file, another system, another team's wiki — is a derivative: generated from the source, imported, or linked to it. Derivatives never carry independent edits; if you change them in place, you are creating a divergent copy and breaking the SSOT.

Why we use it

When the same fact lives in two places, no rule tells a reader which is correct. The two copies drift — usually because one is updated and the other is forgotten — and the loudest copy wins, even when it's wrong. Decision-makers act on bad data; engineers ship code against stale specs; on-call follows runbooks that point at non-existent dashboards.

SSOT prevents drift by collapsing the number of authoritative copies to one. Everything else is either generated from the source (so it can't disagree) or links to it (so a reader can follow the chain back to the canonical version).

The cost SSOT pays: a single edit may need to regenerate derivatives (build step, CI job). We accept that cost because silent drift is more expensive than explicit regeneration.

Primary source → derivatives

The mental model: pick the SSOT, then list everything that must be regenerated or referenced from it.

Entity SSOT Derivatives
API contract openapi.yaml SDK clients, reference docs, mock servers, contract tests
Database schema Migration files (DDL) ORM models, ER diagrams, table-doc pages
Service in catalog catalog-info.yaml (Backstage) Service catalog UI, on-call routing, dashboards
Design tokens tokens.json CSS custom properties, Figma styles, theme files
Architecture decision ADR file (.html in /governance/adr/) ADR index, RFC cross-references, changelog entries

Where it applies in this project

Concrete places where SSOT is currently enforced in the repository:

  • Internal service docs as SSOT for engineering narrative — codified in ADR 0026. The internal portal under services/portal/internal/ is the canonical engineering narrative; external docs (under services/portal/public/) reference internal pages, not the other way around.
  • Docs as code — codified in ADR 0001. Documentation files in the repository are the SSOT; there is no parallel CMS / wiki to keep in sync.
  • ADR / RFC numbering — each decision has one numbered file under governance/adr/ or governance/rfc/. Indexes (ADR index, RFC index) and cross-links derive from the file list, not separate state.
  • External vs internal API documentation split — codified in ADR 0025. Public API reference is generated from the OpenAPI spec; the spec is the SSOT, not the rendered HTML.
  • Portal shell token contract — codified in ADR 0030. Layout geometry lives in CSS custom properties in docs-shell.css; both portals consume the same tokens. The token file is the SSOT, the two layout CSS files are derivatives.
  • Database table documentation — codified in ADR 0029. Each table has exactly one page under internal/catalog/api/data/; the API catalog entries link to that page rather than restating columns.

Anti-patterns we have

Honest list of CURRENT SSOT violations in this repository, with file references and remediation pointers. We track them as debt; the v2 maturity audit (Axis 3 — SSOT & Auto-gen) currently scores this project at 4.3 / 10 precisely because of these gaps.

  1. Hand-maintained sidebar nav. services/frontend/portal/assets/internal-sidebar.js contains an INTERNAL_SIDEBAR_NAV array that duplicates the filesystem structure under services/portal/internal/. Adding, renaming, or removing a page requires editing both the file and this array; divergence is silent until a reader hits a 404. The file carries an SSOT-TODO comment block at the top of the array marking the debt; the planned fix is to generate the nav from a directory walk or a single YAML manifest. Tracked as Phase 2 work (2026-05-11).
  2. No catalog-info.yaml for services. The service catalog under internal/catalog/ describes each service (api, portal, monitoring) in prose-and-HTML across multiple sibling pages (Architecture, API reference, Runbooks, Dependencies, On-call). There is no machine-readable manifest, so Backstage-style consumers (on-call routing, dashboard auto-discovery, owner lookup) would need to scrape HTML. Planned Phase 2 work: migrate to catalog-info.yaml per Backstage software-catalog schema, then generate the per-service HTML pages from it. Tracked separately from the sidebar fix.
  3. No openapi.yaml as primary source. The internal API documentation under internal/api/ is hand-written HTML per operation, with the OpenAPI YAML maintained alongside but not as the generator input. The ideal: write OpenAPI first, generate the HTML reference from it. Tracked as a separate strand of work referenced in v2 maturity audit Axis 8 (API/Data/Event).

These anti-patterns are visible, not hidden — calling them out here is part of the principle. A project that tells you the truth about its own SSOT violations is one you can trust to fix them.
