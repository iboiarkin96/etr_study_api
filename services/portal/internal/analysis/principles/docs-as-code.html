Docs as code

What

Docs as code treats documentation as a first-class engineering artefact: it lives in the same repository as the code it describes, follows the same review process, runs through the same CI checks, and ships through the same release flow. Documentation files are plain-text (HTML / Markdown), version-controlled in Git, and never edited through a separate CMS.

Why we use it

A separate documentation system (Confluence, Notion, a wiki) decouples docs from code review. Engineers can change behaviour without anyone noticing the docs are now wrong, because the docs change is a separate task in a different tool that nobody is blocked on. Docs rot quietly.

When docs live in the same pull request as the code, the reviewer is forced to look at both. A schema change that doesn't update the schema reference page fails review. A new endpoint without a spec page is incomplete. The cost of keeping docs current is paid at the point of change, not later as unscheduled debt.

Plain-text formats add their own benefits: diffable, scriptable, grep-able, copy-able into LLMs, durable across tool migrations.

Where it applies in this project

  • Repository layout. All documentation lives under services/portal/ in the same Git repository as the API code (services/api/) and frontend code (services/frontend/). Pull requests routinely touch both code and docs.
  • ADR / RFC discipline. Decision records under governance/adr/ and governance/rfc/ are HTML files in the repo, reviewed and merged like any other change.
  • Generated artefacts checked into the repo. Python code reference (pdoc output) lives at internal/catalog/api/code-reference/ as committed HTML, regenerated by a build script. Token gallery at internal/front/foundations/tokens.html follows the same pattern.
  • CI gates. Pre-commit and CI run validator scripts (scripts/validate_docs_html.py, scripts/validate_docs_a11y.py, scripts/validate_docs_design.py, scripts/check_asset_refs.py) on every documentation change.
  • Search index built from the repository. scripts/build_docs_search_index.py regenerates search-index.json from the on-disk HTML; no external indexer is involved.

Anti-patterns we have

  1. Stale CHANGELOG entries. Some changes land in the repository without a corresponding entry in services/portal/CHANGELOG.md. The CHANGELOG is in the repo (good), but updating it is not gated by CI (bad). Tracked as Lifecycle & Governance gap in the v2 maturity audit.
  2. No CODEOWNERS file. Docs-as-code implies a review owner for each path. We currently lack a CODEOWNERS file, so reviews fall on whoever sees the PR rather than on a domain-owning team. Tracked as a Trust & Compliance gap.
  3. Pre-commit not enforced everywhere. The .pre-commit-config.yaml exists but is not consistently installed across contributor environments. CI catches the gap, but local feedback is slower than it could be.
