Diátaxis

What

Diátaxis is a framework that organizes documentation into four distinct modes, each serving a different reader need. A page belongs to exactly one mode and stays in that mode — mixing modes within a page makes it serve nobody well.

Mode Reader is… Page shape
Tutorial learning by doing Step-by-step lesson with a guaranteed outcome.
How-to guide solving a specific problem Goal-oriented recipe — assumes prior knowledge.
Reference looking something up Information-oriented spec — exhaustive, neutral, dry.
Explanation seeking understanding Discussion of why & how — context, rationale, alternatives.

Why we use it

Without an explicit framework, documentation drifts toward a single mush: every page tries to be a little tutorial, a little reference, a little explanation. Readers can't predict what they will get, can't navigate by intent, and abandon the docs for Slack or source code.

Diátaxis gives every page a clear purpose. A reader knows that a page in /reference/ is exhaustive and dry; a page in /tutorials/ walks them through to a working result; a page in /how-to/ assumes they know what they want and just need the recipe; a page in /explanation/ answers why.

The framework also produces a coherent information architecture: sidebar and search results have natural structure (mode → topic → page) instead of a flat directory of documents.

Where it applies in this project

The public portal explicitly uses Diátaxis as its top-level IA. The internal portal uses Diátaxis-aligned folders but not as strictly. Concrete mapping:

  • Tutorials — public portal: services/portal/public/tutorials/ (e.g. "First API call").
  • How-to guides — both portals: services/portal/public/how-to/ (external integrators) and services/portal/internal/handbook/howto/ (internal engineers).
  • Reference — both portals: services/portal/public/reference/ (API reference, error catalogue, env vars) and services/portal/internal/api/ (per-endpoint specs).
  • Explanation — public portal at services/portal/public/explanation/ (architecture overview, security model, SLO & error budget); internal portal at services/portal/internal/analysis/ (this portal — methodology, system design, documentation principles).

Anti-patterns we have

  1. Handbook mixes how-to and reference. The internal/handbook/developer/ folder contains pages like "Requirements guide", "Business logic guide", and "Schemas and contracts" — some of which are reference, some how-to, some explanation. Diátaxis discipline would split each into its proper mode.
  2. ADR pages sometimes leak explanation into decisions. An ADR's purpose is to record a decision (Context / Decision / Consequences); when it also tries to teach the reader the underlying concept, the explanation belongs on a separate explanation page that the ADR links to. A handful of ADRs currently include extended tutorial content.
  3. Internal portal lacks a dedicated explanation section until now. This very SA portal closes that gap — prior to it, explanation-mode content was scattered across ADRs, RFCs, and handbook pages.
