ER model — the data domain, table by table

The database is documented one table at a time; the ER diagram is regenerated from the metadata on every migration.

TL;DR

Most data questions — 'is this PII?', 'what's the retention?', 'who owns this column?' — are answered once at schema time, then asked over and over by compliance, support, and analytics. An ER model with table-level metadata answers them once.

What it is

An ER model is two things together: a diagram and a set of table pages. The diagram shows tables and the foreign-key edges between them; each table page lists purpose, owner, PII tags, retention rule, indexes, columns with types, and enum values.

Both come from the same source: the SQL migrations plus a small YAML metadata file per table. CI extracts the structure, runs a Graphviz/PlantUML pass to draw the diagram, and writes a page per table.

Why we need it

Compliance, security, and analytics all ask the same kinds of questions about the schema — and they ask them often. Without table docs, every question turns into a Slack thread that ends in a code-walk by whoever wrote the table.

With table docs, the answer is one click away. The PR that adds a column also adds its metadata; if it doesn't, CI fails. The schema becomes self-describing.

How others do it

dbt Labs

Made .yml column documentation the standard in the modern data stack; every dbt model carries column descriptions, tests, and tags in YAML next to the SQL.

What we take: Put the column docs in the same PR as the schema change — CI fails the build when a column lacks a description.

Airbnb (Dataportal)

Built an internal portal that crawls warehouse metadata, links every table to its owner, downstream consumers, and the dashboards that depend on it.

What we take: An ER diagram is useful; a queryable graph of tables-and-consumers is what stops the next 'who uses this column?' Slack thread.

Open-sourced their data discovery tool: tables, columns, owners, freshness, lineage, all from machine-readable metadata.

What we take: Once metadata is structured, the same source feeds humans (search) and tools (lineage); it stops being 'documentation' and becomes infrastructure.

LinkedIn (DataHub, open-source)

Schema + ownership + lineage for tens of thousands of datasets; powers governance, compliance, and onboarding.

Anatomy

The parts a complete practice has, with one sentence on each. None of these are optional.

Migration files

The SQL. The source of truth for the structure.

Per-table YAML

Purpose, owner, PII tags, retention, an enum-values block when relevant.

Column comments

Each column has a one-sentence comment in the SQL itself.

ER diagram

Generated from FK edges plus the migration list.

Table page

One HTML page per table; columns rendered as a table; FK edges link to the related table page.

How it works in this project

Tables live under migrations/; metadata under services/portal/internal/catalog/api/data/_metadata/. CI runs scripts/build_table_docs.py, which writes catalog/api/data/<table>.html and updates catalog/api/data/er-diagram.svg.

The contract is in ADR 0029: a new column without an updated YAML entry fails the build. There is no way to silently ship a column that the docs don't know about.

Deep dive — the theory you need

Column metadata — three categories you should always capture

Not all metadata is the same. Three categories cover almost every legitimate question:

CategoryFieldsAsked by
TechnicalType, nullability, default, indexes, foreign keysEngineers writing queries
BusinessDescription, examples, source of truth, derivation ruleAnalysts, product, support
GovernanceOwner, PII tags, retention rule, classificationCompliance, security, legal

The three questions an ER document must answer

  1. What does this column mean? — Business category.
  2. Can I store an email here? — Governance (PII tag).
  3. What breaks if I drop this column? — Technical + lineage graph.

An ER page that answers all three needs about 15 lines of YAML per table. A page that answers only one of the three is a partial product; readers will keep asking the other two in chat.

Trade-offs — what this practice costs

  • Migration friction. The schema linter that requires a YAML entry per new table will reject migrations that used to ship in minutes. Worth it for tables that store user data; overkill for ephemeral analytics tables. Carve out the latter explicitly.
  • Diagram readability. An ER diagram with 50+ tables is unreadable as one picture. Split by bounded context; render one diagram per context. Resist the urge to draw 'the whole schema' — nobody reads it.
  • PII tags are someone's job. Without an owner who reviews PII tags, they drift towards 'no, this isn't PII'. Compliance must be a CODEOWNERS reviewer on any change to the metadata.

Lifecycle

migration written → metadata updated → CI extracts → page regenerated → ER diagram updated

A developer writes the migration and updates the YAML in the same PR. CI reads both, regenerates the table page, and the ER diagram. The PR review shows the diff in all three places: SQL, metadata, generated page.

Common pitfalls

PitfallWhy it happensWhat to do instead
Columns without commentsDeveloper adds the column and forgets the comment.CI fails on any column without a comment; pre-commit hook can catch it earlier.
PII tags are wrongThe author guesses.Compliance reviews the PII tag in every migration PR. Add a CODEOWNERS rule.
ER diagram becomes unreadableToo many tables in one diagram.Cap each sub-diagram at 15 tables; group by bounded context. The architecture page links to the sub-diagrams instead of rendering a 50-table picture.
Retention rules driftThe YAML says 30 days; the cleanup job deletes after 90.Single source: the cleanup job reads the YAML; mismatch fails the deploy.

Worked example

YAML metadata

sessions table — one record

table: sessions
purpose: One row per study sitting.
owner: backend-team
retention: 365d  # purged by daily cleanup job
pii: false
indexes:
  - { name: idx_sessions_user_started, on: [user_id, started_at] }
columns:
  - { name: id,             type: uuid,       null: false, pii: false, note: server-assigned }
  - { name: user_id,        type: uuid,       null: false, pii: false, fk: users.id }
  - { name: deck_id,        type: uuid,       null: false, pii: false, fk: decks.id }
  - { name: started_at,     type: timestamp,  null: false, pii: false }
  - { name: ended_at,       type: timestamp,  null: true,  pii: false }
  - { name: cards_reviewed, type: int,        null: false, pii: false }
  - { name: tags,           type: text[],     null: true,  pii: false }

From this single YAML, CI writes: the HTML table page (with each column rendered as a row), the entry in er-diagram.svg, and the row in the data-retention summary on the security audit.
