Design Docs — discuss before you code

A short written proposal that surfaces trade-offs before any code is written.

TL;DR

A Design Doc is a 2–10 page proposal that the team reads and comments on before the change is built. It is not an ADR (those record decisions after the fact) and not a spec (those describe the final shape). The Design Doc is the room where alternatives are still alive.

What it is

A Design Doc is a short written document that proposes a change to the system. It is written before the change is built. The author lays out the context, states the problem in one paragraph, walks through two or three real alternatives with trade-offs, recommends a path, names the risks they expect, and lists open questions.

It is not a specification — a spec describes the final shape. It is not an ADR — an ADR records a decision after it was made. The Design Doc is the conversation: the place where the team still argues, asks for numbers, and shifts the plan before the code is touched.

Why we need it

Most expensive mistakes are not bugs. They are designs that were committed to in chat, built quickly, and refactored painfully six months later. A Design Doc costs one to three days; the rebuild costs weeks and a hit to user trust.

The doc also leaves a trail. A new engineer who joins next year reads the doc and understands why the team picked option B, not option A. Without the doc, that knowledge lives only in the memory of people who may have left.

How others do it

Short tour of how teams known for the practice apply it. Each card is one company, what they do, and what we take from it.

Google

Engineers write a Design Doc (2–10 pages) for any non-trivial change. The doc is reviewed in a shared folder and the comment thread is part of the record.

What we take: The doc is the conversation, not a deliverable. Reviewers comment inline; alternatives stay alive until the team picks one.

Source: Industrial Empathy — Design Docs at Google

HashiCorp

Every meaningful change to Terraform, Vault, or Consul starts as an RFC in a public repo; the team comments, decides, and the RFC is merged before any code lands.

What we take: Public RFCs force the author to defend the trade-offs in writing. External contributors learn the team's reasoning at the same time.

Source: HashiCorp Terraform RFCs (public)

Rust language team

Language and standard-library changes go through an RFC process with a fixed template (Motivation, Guide-level explanation, Reference-level explanation, Drawbacks, Alternatives, Unresolved questions).

What we take: Forcing the same six sections on every author makes proposals comparable; reviewers know where to look.

Source: Rust RFC template

In each case the pattern is the same — the design lives as text before it lives as code, and the comment trail becomes the record of why the team chose what it chose.

Anatomy

The parts a complete practice has, with one sentence on each. None of these are optional.

Context

What the reader needs to know before they can judge the proposal. Two paragraphs, no more.

Problem

One paragraph. Concrete, with a measurable symptom — latency, drift rate, ticket count.

Goals & non-goals

What success looks like, and what this change explicitly does not try to fix.

Alternatives

Two or three real options. Each gets a paragraph and a trade-off note. No straw men.

Proposal

The path the author recommends, in enough detail that a reviewer can object.

Risks

What could go wrong. Include the boring ones: schema migrations, dependency drift.

Rollout & metrics

Order of work; how we will know it landed; what we watch in the first week.

Open questions

Things the author does not know yet. A short list signals an honest doc.

How it works in this project

Design Docs will live under services/portal/internal/design-docs/, with a numeric prefix that matches the related ADR — 0042-search-rewrite-design-doc.html maps to ADR 0042. The numbering keeps the doc and the eventual decision easy to cross-link.

A Design Doc is required when a change touches more than one service, introduces a new dependency, or changes a contract. Routine refactors, dependency bumps, and bug fixes do not need one.

The PR for a Design Doc is reviewed like code: at least two reviewers, comments inline, and a clear status chip at the top of the page (draft, under review, accepted, retired).

Deep dive — the theory you need

What goes in a Design Doc, and why

The structure is not arbitrary. Each section answers one question that a reviewer must answer before they can disagree responsibly:

  • Context — What do I need to know first? The reviewer needs the same background as the author. Two paragraphs cap.
  • Problem — Is this a real problem? One paragraph with a measurable symptom (latency, ticket count, drift rate). No symptom, no problem.
  • Goals & non-goals — What is in scope? Non-goals are as important as goals; they stop scope creep in the review.
  • Alternatives — Did the author look around? Two or three real options. Each must be defensible — no straw men.
  • Proposal — What do I do if I disagree? The recommended path in enough detail that a reviewer can write a counter-argument.
  • Risks — What is the author worried about? Includes boring ones (migrations, dependencies).
  • Rollout — How will we land it safely? Order of work and rollback path.
  • Open questions — What does the author not know? A short list signals honesty.

Design Doc vs ADR vs Spec — they are three different documents

This is the single most common confusion.

  • Design Doc — proposal stage. Alternatives are alive. Status: under review.
  • ADR — decision stage. One option won; the others are now history. Status: accepted.
  • Spec — implementation stage. Final shape of the code or API. Lives next to the code.

A change of consequence walks through all three: Design Doc proposes, ADR records the chosen path, Spec describes what shipped.

Lifecycle

draft → under review → accepted → retired

A doc stays in draft while the author writes. It moves to under review the moment the PR opens. Once the team agrees on a path, the doc is marked accepted and the matching ADR is opened the same day. After the work ships, the doc moves to retired — kept for the trail, but no longer the source of truth.

Common pitfalls

PitfallWhy it happensWhat to do instead
Spec masquerading as a Design DocThe author writes the final design with no alternatives.Force the section structure. If Alternatives has one entry, the doc isn't ready for review.
Decision is taken in chatReviewers leave their objections in Slack instead of on the PR.Move the discussion back to the PR; the Slack thread is lost in a week.
Doc is forgotten after it landsMonths later, no one knows which doc led to the change.Tie the Design Doc number to the ADR number; both link to the merge commit.
Doc is too longAn 18-page doc is read by no one.Keep it under 10 pages. If the topic is bigger, split it into a series.

Worked example

Worked example

0042 — Replace the in-process search with an inverted index

Context. Today the portal grep-walks the page tree at request time. P95 query latency is 1.6 s on the analysis subtree (8 k files). Users have complained twice this quarter.

Problem. Search is the second-most-clicked surface; 1.6 s P95 is above the 800 ms SLO.

Alternatives.

  • A. Cache the grep result. Cheap. Fixes cold-start only; warm path still iterates files.
  • B. Inverted index at CI time. ~200 ms cold, ~30 ms warm. Adds a build artefact and a CI step.
  • C. Off-the-shelf engine (Tantivy, Bleve). Strongest performance, biggest surface area to maintain.

Proposal. Option B. Build cost is acceptable (≈8 s in CI); the index file ships with the docs and is read into memory on portal start.

Risks. Build-time index can desync from sources if the script silently fails. Mitigation: a non-zero exit code in the indexer fails the docs CI job.

Rollout. Land behind a flag for two weeks; compare P95 in production; remove the old grep path.

Open questions. JSON or binary index format? Defer to the implementation PR.
