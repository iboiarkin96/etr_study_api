AsyncAPI — events as a contract

Async messages (Kafka, RabbitMQ, EventBridge, SNS) get the same contract discipline as HTTP — written before code, validated by a schema registry.

TL;DR

Event-driven systems break each other silently. There is no 4xx and no alert — just a stream of corrupted data that surfaces a week later. AsyncAPI makes events as contractual as REST. A schema registry catches incompatible producers before they ship.

What it is

AsyncAPI is the OpenAPI equivalent for async messages. It describes the channels (topics, queues, subjects), the producer, the consumer(s), and the payload schema for each event. Like OpenAPI, it's a YAML/JSON file in the repo.

A schema registry (Confluent, Apicurio, Karapace) stores the schemas, assigns them IDs, and refuses incompatible writes. Every produced message carries the schema ID; consumers fetch the schema from the registry to decode.

Why we need it

The hardest production incidents in event-driven systems come from silent schema drift. A producer adds a field; a consumer downstream that does strict decoding throws on the new field; messages back up; a queue fills; a dependent service stops processing — all without an alert that points at the cause.

A contract plus a schema registry catches that on the producer side. The producer cannot publish a message whose schema is not backward-compatible. The consumer never sees the bad data.

Anatomy

The parts a complete practice has, with one sentence on each. None of these are optional.

asyncapi.yaml

The spec, one per service that publishes or consumes.

Channels

Topics or queues. Each has a producer and one or more consumers.

Payload schemas

Avro, Protobuf, or JSON Schema. Versioned in the registry.

Schema registry

Stores schemas, enforces compatibility (backward, forward, or full).

Bindings

Transport-specific config: Kafka cluster, RabbitMQ exchange, EventBridge bus.

How it works in this project

The project will adopt AsyncAPI for the events that already exist (analytics, audit, search-index notifications). Each producing service ships an asyncapi.yaml. CI publishes the payload schemas to the registry on merge; consumers fetch by ID.

A short ADR will lock the compatibility mode (backward by default), the registry choice, and the layout. The reference page at internal/api/events/ will list every topic with its producer, consumers, and current schema version.

Deep dive — the theory you need

Compatibility modes — backward, forward, full

A schema registry enforces one of three compatibility modes when a producer tries to publish a new schema version:

ModeAllowed changeWho it protects
BackwardNew schema can read data written with the old schema.Consumers. New consumers + old data.
ForwardOld schema can read data written with the new schema.Producers. Old consumers + new data.
FullBoth directions. New ↔ old in both directions.Both, during long migrations.

Concrete examples

  • Backward-compatible: add an optional field (tags); remove a field that has a default; widen an enum.
  • Breaking: add a required field; rename a field; narrow an enum; change a type.
  • Forward-only (rarer): same shape, but readers now ignore a deprecated field that producers stopped sending.

Pick backward as the default. It is the only mode that lets producers move first; producers always outnumber consumers, and you want the side that moves first to be the side with the most code.

Trade-offs — what this practice costs

  • Registry is more infrastructure. A schema registry needs to be deployed, monitored, and operated. For a single-team project with one Kafka topic, AsyncAPI without a registry is fine; the cost trips when you add the second service.
  • Backward-compat constraints slow product. 'You cannot add a required field' is the right rule but hurts a product team that needs the field tomorrow. The escape hatch: ship a new schema version, run both in parallel for one release, retire the old.
  • Consumer ergonomics. Strict-mode consumers throw on unknown fields. Loose-mode consumers silently ignore data they could use. Library defaults matter — picking loose by default and strict for security-critical paths is the usual compromise.

Lifecycle

draft spec → spec accepted → schemas registered → compatible publishes → incompatible publishes rejected

A new event starts as a PR on the AsyncAPI file. Once accepted, the payload schema goes into the registry with version 1. The producer publishes with that schema ID; the registry refuses an incompatible v2 unless the team explicitly opts in.

Common pitfalls

PitfallWhy it happensWhat to do instead
'We'll add a schema later'Team ships the first event without a schema; consumer parses manually.CI gate (scripts/check_registry.py) blocks the producer image build until the event's payload schema is registered.
Schema compatibility mode is wrongDefault is 'none'; producers can ship anything.Set compatibility=BACKWARD at the registry level; a breaking change requires an ADR plus a 2-week parallel-run of v1 and v2.
Versioning by topic nameProducer publishes to orders-v2; consumers need to migrate.Use schema versioning inside the registry, not topic name. The registry's job is to handle versioning.
Consumers ignore unknown fieldsConsumer libraries that throw on extras break on any additive change.Configure consumers to ignore unknown fields by default (jackson.fail-on-unknown=false for JVM; strict=False for confluent-kafka-python); strict mode opt-in only for security-critical paths.

Worked example

AsyncAPI fragment

session.opened event

asyncapi: 3.0.0
info: { title: Sessions service, version: 1.0.0 }
channels:
  session.opened:
    address: sessions.events.opened
    messages:
      sessionOpened:
        $ref: '#/components/messages/sessionOpened'
operations:
  publishSessionOpened:
    action: send
    channel: { $ref: '#/channels/session.opened' }
components:
  messages:
    sessionOpened:
      payload:
        type: object
        required: [session_id, user_id, started_at]
        properties:
          session_id: { type: string, format: uuid }
          user_id:    { type: string, format: uuid }
          started_at: { type: string, format: date-time }
          tags:       { type: array, items: { type: string } }

On merge, the payload schema is registered as session.opened-v1. Adding tags as a new optional field is backward-compatible; adding a required field is not, and the registry will refuse it.

Outcome. The producer publishes sessions.events.opened with the registered schema ID; consumers fetch the schema from the registry, decode, and never see a corrupt event. Adding tags as an optional field passes the backward-compat check; adding a required field is rejected at registration time.
