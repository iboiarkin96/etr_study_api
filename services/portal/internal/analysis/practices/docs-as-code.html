Docs as code — PR, CODEOWNERS, CI

Documentation lives in the repo, is reviewed in PRs, is versioned with the code, and is validated by CI.

TL;DR

The docs and the code are managed identically. Same repo, same PR flow, same review tooling, same CI gates. No parallel CMS, no 'wiki for docs, repo for code'. The drift between code and docs is impossible because the same commit touches both.

What it is

Docs as code is a method, not a tool. It means: write docs in plain text in the same repo as the code; review them in the same PR as the code; check them in the same CI as the code. CODEOWNERS routes each docs folder to a named owner.

The discipline is the point. Once you cannot ship code without touching the docs, drift becomes a build failure, not a slow rot.

Why we need it

A parallel CMS (Confluence, Notion) is convenient until someone changes the API and forgets the wiki page. Then the wiki tells the integrator something the code no longer does. The team's options at that point are equally bad: keep updating two systems, or stop trusting the wiki.

Putting docs in the repo eliminates the choice. The PR for the API change also touches the doc. Reviewers see the diff in one place. CI runs the same linters and link-checks on both.

How others do it

Short tour of how teams known for the practice apply it. Each card is one company, what they do, and what we take from it.

GitLab

The entire GitLab handbook (3 000+ pages) lives in a Git repo; every change goes through merge requests; CI checks links, style, and structure.

What we take: The handbook is the company's operating system. Treating it like code is how it survives the scale.

Source: GitLab handbook (the source repo)

Stripe

Public API docs are generated from the same OpenAPI source as the SDKs; CI gates the spec on lint, lint, and consistency checks before deploy.

What we take: When the docs site is generated, doc PRs and code PRs have the same review weight.

Source: Stripe API docs

Cloudflare

Developer docs are in a public repo; community PRs land alongside Cloudflare's own; the docs site rebuilds on every merge.

What we take: Public repos invite community contributions; the CI gate is what makes those PRs safe to accept.

Source: Cloudflare Developer Docs (repo)

HashiCorp

Every product's docs are in the product's repo; the documentation team owns infrastructure, not content; product engineers write the content.

What we take: Docs as code is a staffing model too — engineers write, doc engineers build the platform.

Source: HashiCorp Learn (engineering)

Pattern across all four: the docs site is generated from text files in a Git repo; PR review and CI are the quality gates; no parallel CMS is allowed to host 'real' content.

How it works in this project

Codified in ADR 0001. The portal at services/portal/ is the canonical doc site. CI runs scripts/validate_docs_design.py, scripts/check_asset_refs.py, and scripts/check_h1_structure.py on every PR.

CODEOWNERS routes each top-level docs folder to a maintainer; merges require at least one approving review from a code-owner. No parallel CMS is allowed; if a doc cannot live in the repo, the team explains why in a Design Doc.

Deep dive — the theory you need

The maturity ladder for docs-as-code

Teams don't get to full docs-as-code overnight. Five stages, each strictly better than the previous:

  1. Wiki-first. Docs in Confluence/Notion. Code is in Git. They drift weekly.
  2. Mirror. Docs in Git, but mirrored to a wiki for 'readability'. Wiki is the source for most readers; drift continues.
  3. Repo-first, no CI. Docs in Git, PRs reviewed, but no automated checks. Drift is slower; broken links and dead refs accumulate.
  4. Repo-first, basic CI. Link checker, structure validator, style linter. Builds fail on common drift.
  5. Generated from source. Reference docs, API tables, and config schemas are auto-generated from the code/spec/schema. Drift impossible by construction.

What changes at each level

The big jumps are 1→2 (any move out of the wiki) and 4→5 (auto-generation eliminates a whole class of bugs). Most teams plateau at level 3 — that's where the rules exist but no one enforces them.

The check that pushes you from 3 to 4 is the cheapest single improvement: a CI job that fails on a broken internal link saves more time than any other gate.

Trade-offs — what this practice costs

  • Excludes non-technical contributors. A PM who wants to fix a typo now needs Git, a branch, and a PR. The cheapest fix: a 'Suggest edit' link on every page that opens a pre-filled GitHub issue or web-editor PR.
  • PR review is slower than wiki editing. A typo on Confluence is a one-click fix. The same on GitHub is a PR + review + merge. Worth it for content that must stay correct; overkill for ephemeral notes.
  • No WYSIWYG. Authors who don't read HTML find the source intimidating. Investing in a preview server (or a Markdown-first authoring layer) closes the gap.

Lifecycle

edit in branch → PR → CI gates → review → merge → publish

The doc edit follows the same path as a code edit. Open a branch, push, open a PR. CI runs linters, link-checkers, and the design baseline. Reviewers approve. Merge ships the doc — there is no separate publish step.

Worked example

PR that touches both

Adding a new endpoint

  • The PR title: feat(api): POST /api/v1/sessions/{id}/share
  • Files changed: services/api/sessions.py, services/api/tests/test_share.py, services/api/openapi.yaml, services/portal/internal/api/sessions/share.html.
  • CI runs: unit tests, OpenAPI lint, link-check, design baseline. All gates must pass.
  • Reviewers: backend code-owner + docs code-owner. Two approvals; merge.

The doc cannot drift because it travels with the code. The next reader sees the doc that matches what shipped.
