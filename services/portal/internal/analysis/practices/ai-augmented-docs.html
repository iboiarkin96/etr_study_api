AI-augmented docs — llms.txt and the MD source

Docs that read well for both humans and AI tools — llms.txt, MD source next to HTML, and an Ask-AI widget on high-traffic pages.

TL;DR

AI assistants give precise help when they can find clean source. Without llms.txt and a Markdown copy, the assistant guesses from an HTML soup and confidently returns wrong answers. The fix is small — a few canonical files and a build step — and pays back every day.

What it is

This practice has three pieces:

  • llms.txt — a small file at the site root that lists the canonical pages, in priority order, with one-line descriptions. The format is published at llmstxt.org.
  • MD source next to HTML — every doc page has a sibling .md file generated from the same source. AI assistants read Markdown an order of magnitude more reliably than rendered HTML.
  • Ask-AI widget — a small RAG-backed assistant on high-traffic pages that answers questions against the canonical content, not against the open web.

Why we need it

Engineers use AI assistants all day. The question is no longer 'will they', but 'will they get the right answer'. Without llms.txt, the assistant has no idea which pages are canonical, so it picks the loudest result; without an MD copy, it parses HTML poorly and hallucinates the rest.

The Ask-AI widget closes the loop. A reader who would have searched, asked a teammate, or guessed gets a precise answer from the docs themselves. The reader stays in the docs; the docs stay accurate.

The standard playbook is now: ship llms.txt at the site root, ship MD source next to every HTML page, and run a RAG widget on high-traffic pages. The format (llmstxt.org) was proposed in 2024 and is rapidly becoming the default.

Indexer

RAG widget

Small UI that takes a question and returns an answer plus citations.

Citations

Every AI answer links back to the canonical page sections it used.

Feedback loop

Thumbs-up / thumbs-down on answers; signals feed the next index rebuild.

How it works in this project

The MD source comes for free if we write doc pages from a small Jinja template that emits both HTML and MD. The indexer is a CI step on tag; the widget is a small JS component plus a backend that fans out to an embedding provider.

A short ADR will lock the choice of the embedding model and the retrieval policy: how many chunks, how to rank, when to refuse (rather than hallucinate).

Deep dive — the theory you need

The RAG pipeline — five stages

Retrieval-Augmented Generation is the technique behind an Ask-AI widget. The pipeline:

  1. Chunk. Split the docs into ~200–500 word chunks. Smaller chunks are more precise; larger ones carry more context.
  2. Embed. Pass each chunk through an embedding model (text-embedding-3-small, voyage-code, etc.). The output is a 1 500-dim vector per chunk.
  3. Search. Embed the user's question; find the top-K (typically 10–20) most-similar chunks by cosine similarity.
  4. Re-rank. Pass the top-K to a re-ranker (Cohere Rerank, BGE) that re-scores them against the question.
  5. Generate. Send the top 3–5 re-ranked chunks to an LLM with the question; ask it to answer only from the chunks.

The refusal threshold

The single most important parameter is the refusal threshold. If the top-K chunks have similarity scores below a cut-off, the widget should say 'I don't know' instead of generating an answer from nothing. The threshold is tuned per corpus; a typical value is 0.7 cosine similarity.

A widget that refuses 5% of questions is honest; a widget that refuses 0% is hallucinating.

Citations are non-negotiable

Every generated answer must include links to the source chunks. Two reasons:

  • The user can verify the answer.
  • The team can audit when the AI is wrong — without citations you cannot tell the difference between 'AI summarised badly' and 'docs are wrong'.

Trade-offs — what this practice costs

  • RAG infrastructure cost. Embedding API + vector store + ranker + LLM = real spend, often $200–$2 000/month even for a small docs site. Worth it on the most-visited pages; not worth it on every page.
  • Refusal threshold tuning. Set too low and the widget hallucinates; too high and it refuses useful questions. Plan a month of tuning + a feedback widget (thumbs-up/thumbs-down) to converge.
  • Privacy posture has legal review. A user query about an internal page must not leak to a third-party model. Either self-host the model, or strip PII at the boundary. Legal sign-off costs weeks; build it into the rollout schedule.

Lifecycle

llms.txt live → MD per page → indexed on release → widget on 5 pilot pages → feedback loop active

llms.txt ships first; it's a one-file lift. The MD pipeline follows. The index rebuilds on every release tag. The widget pilot picks the five most-visited pages. Feedback (positive and negative) flows back to the rebuild process.

Worked example

llms.txt

Root file (excerpt)

# Study App docs

## Canonical entry points
- [SA portal](/services/portal/internal/analysis/index.html): the systems-analysis home.
- [Practices](/services/portal/internal/analysis/practices.html): the 25-practice radar.
- [API reference](/services/portal/public/reference/api/openapi-explorer.html): public API.

## Architecture
- [Context](/services/portal/internal/front/architecture-map.html): C4 Level 1 across the system.
- [Catalog](/services/portal/internal/catalog/index.html): per-service Backstage entries.

## Decisions
- [ADR index](/services/portal/internal/governance/adr/index.html): accepted decisions.
- [RFC index](/services/portal/internal/governance/rfc/index.html): proposals.

Twelve canonical entry points; an AI assistant fetches this first and follows the links it needs. Pages outside the list are still reachable; they're just not the canon.

Outcome. AI assistants that read llms.txt fetch the listed canonical pages first; answer precision on questions about those pages rises measurably, and the team can audit which page was used via the citation links the widget emits.
