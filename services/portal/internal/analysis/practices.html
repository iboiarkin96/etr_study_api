Group A · 5 practices

Knowledge management

How decisions, facts and terminology of the project are captured. Without this group, a new engineer spends days asking questions, and every architectural decision is reopened three times. This is the team's institutional memory.

Group avg score: AS-IS 0.0→0.0 / 10
01

Design Docs — discuss before you code

2.0→8.0

What it is. Before any non-trivial change — a short 2–10 page Design Doc: context and problem, options with trade-offs, the chosen path, success metrics, risks, open questions.

Why it matters. At Google this is the primary engineering artefact: the team discusses a decision before the first line of code is written. ADRs capture accepted decisions; Design Docs propose them — different genres, and Google keeps them separate.

Design Docs at Google Document-driven design
02

ADR · RFC — numbered decisions

7.4→8.5

What it is. Every significant decision is a numbered file with Context · Decision · Consequences. ADRs for accepted decisions, RFCs for proposals under discussion. Lifecycle: draft → ongoing → done / declined / superseded.

Why it matters. So that a year from now nobody is trying to remember why it was done this way. A new engineer opens an ADR and sees context, alternatives, consequences — without verbal retellings or tribal knowledge.

principle ADR index RFC index MADR 4.0
03

SSOT — one source of truth per entity

5.0→8.0

What it is. Every fact (API contract, DB schema, service catalog, design tokens, terms) has exactly one authoritative file. Everything else is a derivative: generated from the source or referenced.

Why it matters. So that two copies don't drift apart. If a fact lives in two places, one copy will eventually become wrong — and nobody will know which one is the correct version.

principle ADR 0026
04

Glossary — unified domain language

2.5→7.0

What it is. One dictionary of project terms: canonical definition, synonyms, related terms, examples. Source is YAML, rendered to HTML and inline tooltips in prose. This is ubiquitous language from DDD.

Why it matters. So "contract", "API" and "endpoint" mean the same thing to everyone. A shared dictionary removes half the disputes in review, speeds up onboarding, and disciplines requirements before they reach implementation.

Fowler · Ubiquitous Language Good Docs glossary
05

Tech Radar — technology portfolio

1.0→7.0

What it is. An ADOPT / TRIAL / ASSESS / HOLD quadrant for the project's technologies. Source — tech-radar.json; rendered visually (Zalando / ThoughtWorks renderer). Reviewed quarterly.

Why it matters. So the team explicitly sees which technologies are adopted, which are experimental, which are forbidden. Transparent for hiring panels, contributors, and decisions like "should we adopt this new library".

ThoughtWorks Tech Radar Zalando renderer
Group B · 5 practices

Architecture & modeling

How we record what we built — at different levels of abstraction. Diagrams, schemas, contracts, domain boundaries. The artefacts an engineer must understand before touching the code.

Group avg score: AS-IS 0.0→0.0 / 10
06

C4 + PlantUML — diagrams as code

5.6→8.0

What it is. Architecture is drawn at three C4 levels: Context (what's outside), Container (what it's made of), Component (how it's built inside). Source — PlantUML / Structurizr DSL in the repo; rendered in CI.

Why it matters. So a new engineer understands the system in 10 minutes, not a day. And so the diagram doesn't lie: it lives next to the code and is updated in the same PR.

principle ADR 0020 C4 model
07

ER · data domain model

5.2→8.0

What it is. DB structure is documented at the table level: purpose, owner, PII tags, retention, indexes, foreign keys, enum values. The ER diagram is auto-regenerated from metadata on every migration.

Why it matters. To understand data before code. Is the "email" field PII or not? What is its retention? Who owns it? These questions are answered once — at schema time — not on every review or compliance audit.

ADR 0029 dbt column properties
08

API contract-first — OpenAPI as the source

6.4→8.5

What it is. The API spec (OpenAPI 3.1) is written before the implementation. SDKs, docs, mock servers and contract tests are generated from it. CI breaks the build when code diverges from spec.

Why it matters. So the integrator and the team speak the same language before the first line of code is written. The contract is the agreement; the code is its implementation, not the other way around.

ADR 0025 OpenAPI 3.1 Spectral
09

AsyncAPI · event schemas

1.0→7.0

What it is. For asynchronous messages (Kafka, RabbitMQ, EventBridge, SNS) — an AsyncAPI specification. Producer and consumer respect the contract; a schema registry checks compatibility.

Why it matters. So the event-driven surface is as contractual as REST. Without AsyncAPI, microservices drift schema-wise and break each other silently — no 4xx, no alert, just corrupted data.

AsyncAPI Confluent Schema Registry
10

DDD · bounded contexts

2.0→7.0

What it is. A map of bounded contexts: each service belongs to one context; contexts are connected by typed relationships (Customer-Supplier, Conformist, Anti-corruption Layer, Shared Kernel). This is a DDD context map.

Why it matters. So microservice and team boundaries align with domain model boundaries. Without this, "User" means different things in different services — and integration disasters follow at scale.

Fowler · Bounded Context Context Mapper
Group C · 5 practices

Information delivery

How the reader finds, understands and updates what was written. Content architecture, templates, style, lifecycle, accessibility. This is the "documentation UX" for engineers and auditors.

Group avg score: AS-IS 0.0→0.0 / 10
11

Diátaxis — four quadrants, no mixing

6.4→8.0

What it is. Every page belongs to one mode: tutorial (teach a beginner), how-to (solve a task), reference (precise data), explanation (build understanding). No "all-in-one" pages.

Why it matters. So the reader immediately knows what to expect. Tutorials teach but aren't reference; reference is precise but doesn't teach. Mixing modes is the main cause of "unreadable" documentation.

principle Diátaxis (Procida)
12

Docs as code — PR · CODEOWNERS · CI

6.5→8.5

What it is. Documentation is code: in the repo, reviewed via PR, versioned, validated in CI (linter, broken-link check, a11y). CODEOWNERS routes every docs folder; no parallel Confluence as SSOT.

Why it matters. So docs behave like tests — you can't forget to update them, can't introduce drift silently. A parallel CMS (Confluence / Notion) guarantees drift with code; the repo is the only SSOT.

principle ADR 0001 Write the Docs
13

Template registry — Design Doc · ADR · Postmortem · Runbook

5.4→8.0

What it is. One registry of templates for all major page types: Design Doc, ADR, RFC, Postmortem, Runbook, API spec, Screen spec, Audit, Principle. Each template is Jinja with required fields.

Why it matters. So 32 ADRs and 6 RFCs look like one project, not 38 different ones. Templates fix the grammar of the page (structure); the style guide fixes the grammar of sentences. They are different disciplines.

principle RFC 0003
14

Style guide as code — Vale + alex.js

1.5→7.5

What it is. Google Developer Documentation Style or Microsoft Writing Style Guide accepted as canon. Vale rules in .vale/, alex.js for inclusive language, cspell for typos. PR-blocking on style errors.

Why it matters. So the tone of voice is consistent across the portal. Without a linter, every author writes in their own voice; readers see patchwork and lose trust in the documentation.

Google Dev Style Vale alex.js
15

Onboarding · accessibility · i18n

3.5→7.0

What it is. README + CONTRIBUTING + .devcontainer/ + CODE_OF_CONDUCT + good-first-issue labels. Plus an a11y audit (WCAG 2.2 AA, axe-core, keyboard nav, screen-reader). Infrastructure for a second UI language.

Why it matters. So a new contributor lands their first commit in ≤ 1 working day. So everyone, including users with disabilities, can read and contribute — this isn't charity, it's compliance (EAA 2025).

GitHub Community Standards WCAG 2.2
Group D · 5 practices

Reliability & operations

How we know the system works "well enough", and what to do when it doesn't. This is the central group in Google SRE — without it the project has no operational maturity. Currently the weakest group in this project.

Group avg score: AS-IS 0.0→0.0 / 10
16

SLO · SLI · error budgets

0.5→7.0

What it is. Every user-facing service has a Service Level Objective (e.g. 99.9% availability), Service Level Indicators (metrics that measure the SLO), and an error budget. When the budget is exhausted — deployments slow down until reliability recovers.

Why it matters. This is the central concept of the Google SRE Book. Without SLOs you cannot answer "is the service working well enough". The team works on hype instead of metrics; ops and product argue without a shared frame.

Google SRE · SLO SLO generator
17

Runbooks — one per alert

1.5→7.5

What it is. For every alert — a runbook: symptom → diagnosis → actions → escalation → dashboard links. The template is standardized, tied to alert IDs, and updated after every incident.

Why it matters. So the on-call at 3 AM knows what to do instead of paging the architect. Cuts MTTR, reduces on-call burnout, makes on-call scalable to new engineers without years of tribal knowledge.

Google SRE · runbooks
18

Blameless postmortems

1.0→7.5

What it is. Every P1/P2 incident gets a blameless postmortem: timeline, impact, root cause, contributing factors, action items with owner and due date, lessons. Template is standardized; reviewed quarterly.

Why it matters. So the team learns from mistakes instead of repeating them. "Blameless" means: look for the process problem, not the guilty party. Google and Stripe publish postmortems publicly — a strong external maturity signal.

SRE · postmortem culture postmortem templates
19

Risk register — risk log

0.5→7.0

What it is. A risk log: list of risks (technical, business, security, compliance) with severity (probability × impact), mitigation, owner, status and review cadence. Reviewed at every quarterly planning.

Why it matters. So risks are visible and managed instead of living in people's heads. A Senior SA at PMI / PRINCE2 level always maintains a risk register; without it the project goes blind to strategic threats.

PMI · risk register
20

Capacity planning · FinOps

0.5→6.5

What it is. Back-of-envelope capacity planning per service: QPS, storage growth, cost projections for 6–12 months. FinOps documentation on TCO per service, budgets, optimizations.

Why it matters. So infrastructure doesn't run away on cost or hit a ceiling by end of quarter. A Senior SA thinks about cost-per-request and growth curves, not only functional features.

FinOps Foundation SRE · capacity planning
Group E · 5 practices

Process & governance

How the team operates the project as a system: security, service catalog, AI access, stakeholders, releases. This is the "governance frame" — the thing that separates a prototype from production.

Group avg score: AS-IS 0.0→0.0 / 10
21

SECURITY · threat modeling

1.0→7.5

What it is. At the repo root: SECURITY.md (responsible disclosure) and security.txt (RFC 9116). Key services have a one-pager STRIDE / LINDDUN threat-model. Vulnerability disclosure SLA defined.

Why it matters. So a researcher who finds a hole knows where to write — instead of publishing on Twitter. So the team thinks about threats as a process, not a fire drill. This is a SOC2 / ISO 27001 component.

RFC 9116 OWASP · threat modeling
22

Service catalog — Backstage YAML

3.0→7.5

What it is. Each service has a catalog-info.yaml in Backstage format: kind, metadata, owner, type, lifecycle, dependsOn, providesApis, consumesApis. HTML pages are generated from the YAML.

Why it matters. So the catalog is read not only by humans but by automation: on-call routing, dashboard auto-discovery, owner lookup, dependency graph. Without a machine format, every internal tool reinvents its own HTML parser.

Backstage catalog SSOT principle
23

AI-augmented docs — llms.txt

1.8→7.5

What it is. Documentation written so it can be read not only by humans but by Claude / ChatGPT / Cursor. llms.txt marks the canon, an MD source is read by LLMs tens of times better than HTML, an Ask-AI widget gives RAG answers.

Why it matters. So AI tools answer questions about the project correctly. An assistant giving a precise answer from the docs saves hours; an assistant hallucinating from an HTML soup misleads users and erodes trust.

llms.txt spec Kapa.ai · Ask-AI
24

Stakeholder map · personas · JTBD

1.0→7.0

What it is. A list of stakeholders (role, goal, influence, information needs) and user personas (JTBD statements, frustrations, expectations). A Power-Interest grid; refreshed quarterly.

Why it matters. So the product is built for real needs, not assumptions. Without personas, documentation and UX address "everyone" = "no one". This is the purest SA artefact.

NN/g · personas JTBD framework
25

RACI · release · review SLA

2.0→7.5

What it is. A RACI matrix (Responsible · Accountable · Consulted · Informed) on key processes; release process with semver, changelog, canary, rollback playbook; code review SLA (e.g. 2 working days), escalation process.

Why it matters. So processes are predictable and scalable as the team grows. Without RACI, decisions stall; without a release process, every release is improvisation; without a review SLA, PRs sit for weeks.
