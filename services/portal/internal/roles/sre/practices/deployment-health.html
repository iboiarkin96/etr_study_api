Deployment Health Gates — confirm the deployment is healthy before calling it done

Automated checks that run immediately after every deployment and block promotion or trigger rollback if any signal degrades — converting the post-deploy observation window from a manual watch into a structured, reproducible gate.

TL;DR

A deployment health gate is an automated check that runs for a defined observation window after every production deployment and compares post-deploy metrics against pre-deploy baseline. Three gate signals: error rate (must stay within 1.5× baseline for 10 minutes post-deploy), P99 latency (within 1.5× baseline), and success rate of the service's critical-path synthetic probe (must remain above 99%). If any signal degrades, the deployment pipeline pauses and the on-call is paged. If degradation persists past a 5-minute grace period, rollback is triggered automatically. The gate eliminates the "we deployed, it looks fine, we went home, something broke at 2 am" failure mode.

What it is

Deployment health gates are automated checks that run during a defined observation window after every production deployment and compare post-deploy metrics against a pre-deploy baseline. A gate has three components:

  • Gate signals: The specific metrics checked — typically error rate, P99 latency, and the success rate of the service's critical-path synthetic probe. Each signal has a pass threshold (e.g., error rate within 1.5× baseline) evaluated across the observation window.
  • Observation window: The time period after deployment during which signals are observed. 10 minutes for canary deployments; 30 minutes for ring 1 promotions. The window must be long enough to accumulate statistically significant traffic, but short enough to limit blast radius if something is wrong.
  • Failure response: If a gate signal degrades, the deployment pipeline pauses and the on-call is paged. A 5-minute grace period allows for transient noise. If degradation persists past the grace period, rollback is triggered automatically without human intervention.

Gates are defined per service as code — a YAML configuration specifying the signals, thresholds, and observation window. The gate runs as a pipeline stage, not a human check.

Why we need it

Most post-deploy outages happen in the hour after a deployment, not during it. The deployment itself goes cleanly, the first manual check looks fine, the team moves on — and then something fails at low traffic that wasn't visible in the observation minute the deployer spent looking at dashboards before going to lunch. Health gates close this gap by extending the observation window programmatically and defining explicit pass/fail criteria.

Health gates also provide the evidence for rollback decisions. Without a gate, the rollback decision requires an engineer to assess degradation in real time, under pressure, with partial information. With a gate, the rollback decision is made by the same criteria that would pass a production review — documented, reproducible, and fast.

How others do it

Netflix

Netflix's deployment system (Spinnaker) includes a "canary analysis" stage that compares the canary version's metrics against the baseline using a statistical comparison — not a simple threshold, but a Mann-Whitney U test across the distribution of metric values during the observation window. This detects regressions in the tail of the distribution that a simple average or P99 comparison would miss. The canary analysis score (0–100) is the gate criterion, not the raw metric values.

What we take: Statistical comparison is more rigorous than threshold comparison for detecting distribution shifts. A simple "error rate within 1.5× baseline" misses cases where the mean is fine but the tail has a new spike. Statistical gates are the correct approach for high-volume services where the distribution matters more than the average.

Source: Netflix Tech Blog — Kayenta

Weaveworks / Flagger

Flagger (open source, developed by Weaveworks) implements progressive delivery health gates as a Kubernetes controller. The controller watches a deployment, routes traffic through canary/stable splits, queries Prometheus or Datadog for gate signals, and advances or rolls back automatically based on configurable thresholds. The gate configuration lives in a CRD (Custom Resource Definition) in the same repository as the service — gates are reviewed in the same PR as the deployment change.

What we take: Gate configuration belongs in the service repository, not in a separate deployment system. When gates are code-reviewed alongside the deployment change, engineers are more likely to think about failure modes before deploying than after.

Source: Flagger — Progressive Delivery Operator

Anatomy

Gate signal selection

Three mandatory signals: error rate (4xx/5xx / total requests), P99 latency, and synthetic probe success rate. Optional: SLO burn rate for the window. Avoid business metrics (conversion rate, revenue) as gate signals — they're too noisy for a 10-minute window and confound A/B effects with reliability regressions.

Baseline calculation

Baseline = same signals averaged over the 24 hours before the deployment, excluding the previous deployment window. Day-of-week and hour-of-day adjustment: if deploying on Monday morning, the baseline is the previous Monday morning, not the Friday evening. Static thresholds (not compared to baseline) produce too many false positives on seasonal traffic patterns.

Grace period design

A 5-minute grace period after gate degradation absorbs transient metric spikes (alerting noise, burst traffic). During the grace period, the pipeline pauses and the on-call is notified but rollback has not triggered. The engineer can manually abort the grace period and trigger immediate rollback if the degradation is clearly severe. Never extend the grace period beyond 10 minutes — it defeats the purpose of the gate.

Gate-as-code requirement

Every Tier 0 and Tier 1 service must have a health gate configuration file (`deployment-gate.yaml`) in the repository root. CI validates that the file exists and parses correctly before the PR can merge. Services without a gate file cannot deploy to production.

Common pitfalls

PitfallWhy it happensWhat to do instead
Gates that never trigger because thresholds are too looseEngineers set 10× baseline thresholds to avoid false positives after being paged once for a transient spike.Track gate trigger rate. A gate that has never triggered in 6 months either means the service is flawless or the thresholds are wrong. Both are worth investigating. Review gate thresholds quarterly against actual post-deploy degradation events.
Gate signals chosen that aren't sensitive to the deployment"We track CPU utilisation because it's easy to instrument."Gate signals must be causally linked to the deployment. CPU is a resource metric — it changes with load, not just with code changes. Error rate and latency are output metrics that directly reflect user-visible quality. Choose user-visible output signals.
Manual override used routinely to bypass gate failures"We know this is a known metric spike — just override the gate."If a gate failure is routinely bypassed, the gate is wrong. Fix the gate (adjust the threshold or exclude the known spike from the signal) rather than building a culture of overrides. Every manual override is a data point that the gate needs tuning.
