Metrics & Dashboards — four golden signals, versioned as code

Every service exposes the four golden signals. Every dashboard is stored in git, reviewed before merge, and linked from the runbook. A service without a canonical dashboard is a service without observability.

TL;DR

The four golden signals (latency, traffic, errors, saturation) are the minimum viable metric set for any user-facing service. Every service exposes them via a standard Prometheus/StatsD library. Dashboards are versioned as Grafana JSON in git, reviewed by at least one other engineer before merge, and linked from the service's runbook. A dashboard built during an incident is too late; a dashboard reviewed before production is what enables fast diagnosis when it matters.

What it is

Google SRE identified four golden signals as the minimum viable metric set for any service:

  • Latency: Time to serve a request. Measure P50, P99, and P999 separately. Crucially, measure success and error latency separately — a spike in error latency that looks normal in the aggregate may be masking a serious failure mode.
  • Traffic: The volume of demand on the system — requests per second, messages per second, transactions per second.
  • Errors: The rate of requests that fail — both explicit failures (5xx responses) and implicit failures (200 responses with an error payload, or responses missing required data).
  • Saturation: How full the most constrained resource is. For CPU-bound services, CPU utilisation. For memory-bound services, memory utilisation. For I/O-bound services, queue depth. Saturation often predicts latency degradation before it becomes visible in the latency signal.

Dashboards-as-code: every Grafana dashboard is stored as JSON in a git repository, reviewed before merge (the same as application code), and deployed via a pipeline — not through the Grafana UI directly.

Why we need it

Ad hoc dashboards built during an incident are too late. They're built under stress, by an engineer who is simultaneously diagnosing the problem, and they're never maintained after the incident. Dashboards-as-code enforce a different discipline: the dashboard is reviewed before the service reaches production, by engineers who are not in crisis mode, and it's reproducible across environments.

The four golden signals give the minimum viable signal set. If all four look normal, the service is almost certainly healthy. If any one is degraded, it's a signal that something requires investigation. This simplicity is a feature — on-call engineers during an incident need fast answers, not 40-panel dashboards that take 5 minutes to interpret.

How others do it

Google SRE

Google codified the four golden signals in the SRE Book and builds their internal monitoring infrastructure around them. Every Google service has a canonical monitoring dashboard that shows the four signals plus SLO burn rate. The dashboard is linked from the service's on-call runbook and is the first thing an on-call engineer opens when paged.

What we take: Link the canonical dashboard from the runbook. When an on-call engineer is paged, they should not have to search for the right dashboard — it should be one click from the alert that fired. The runbook and the dashboard are the same artifact in two views.

Source: Google SRE Book — Monitoring

Grafana Labs

Grafana Labs ships "Mixin" — a pattern for packaging Grafana dashboards as code alongside the Prometheus alerts and recording rules for a specific service or library. Mixins are published as npm/Go modules. Teams import the mixin for a third-party service (PostgreSQL, Redis, NGINX) and get the dashboards, alerts, and recording rules pre-configured, without having to build them from scratch.

What we take: Adopt mixins for the infrastructure services. Grafana's postgres-mixin, redis-mixin, and nginx-mixin are production-quality dashboards built by experts for those specific systems. Spending 3 days building a custom Postgres dashboard when the mixin already exists is wasted engineering time.

Source: Grafana Dashboard Documentation

Anatomy

Shared instrumentation library

One Prometheus client library per language, pre-configured to expose the four golden signals with consistent metric names and label schemas. Services import it; they don't configure Prometheus by hand.

Dashboard-as-code

Grafana JSON stored in a dashboards/ git repo. Reviewed by at least one other engineer before merge. Deployed via a CI pipeline that pushes to Grafana via the API. Editing the Grafana UI directly is forbidden — changes must go through the git workflow.

Runbook link

Every canonical dashboard is linked from the service's runbook. Every alert rule includes a link to the dashboard panel showing the alerting signal. No hunting for the right dashboard during an incident.

Quarterly dashboard review

Stale panels are removed, new signals added, panel thresholds recalibrated. A dashboard with 40% stale panels is worse than no dashboard — it creates noise and erodes trust in the signals that are accurate.

Common pitfalls

PitfallWhy it happensWhat to do instead
40-panel dashboards no one understands"More data is better."The canonical dashboard for a service should have 8–12 panels: the four golden signals + SLO burn rate + 2–3 service-specific signals. Additional detail lives in sub-dashboards linked from annotations, not on the main dashboard.
Dashboards built in the UI, never in codeGrafana's UI is easy to use; git is friction.Lock down direct UI edits for the canonical dashboards (Grafana RBAC). Engineers who need to explore use temporary dashboards in a sandbox folder; those dashboards are not linked from runbooks or alerts.
Separate metric namespaces per serviceTeams pick their own label names without coordination.Enforce a canonical label schema: service, environment, region, version. Without it, cross-service queries are impossible and SLO calculations break when you try to aggregate across services.
