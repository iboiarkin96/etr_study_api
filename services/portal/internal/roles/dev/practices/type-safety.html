Type safety as default — strict mode, no escape hatches

Run the language's strictest type-checker in CI as a required gate. Block PRs that introduce any, cast, or # type: ignore without a reason comment.

TL;DR

A 2017 study by Gao, Bird, and Barr (Microsoft Research) measured what happens when JavaScript bug-fix commits are re-typed in TypeScript or Flow: 15% of the bugs would have been caught at compile time. Type safety as default means every codebase runs its language's strictest type-checker as a required CI gate — TypeScript strict: true, Python mypy --strict, Rust (no choice), Go with golangci-lint. Escape hatches (any, cast, type: ignore) are reviewed individually, must include a justifying comment, and are tracked in a monotonically-decreasing budget. The type system is a free exhaustive code reviewer that runs in seconds; disabling it is a strictly worse trade than disabling tests.

What it is

Type safety as default means the language's strictest type-checking mode is enabled, runs in CI as a required gate, and escape hatches are reviewed line-by-line. The concrete configuration per language:

  • TypeScript — "strict": true in tsconfig.json. This enables strictNullChecks, noImplicitAny, strictFunctionTypes, strictBindCallApply, strictPropertyInitialization, alwaysStrict, and noImplicitThis. Plus noUncheckedIndexedAccess: true to make array/object access return T | undefined.
  • Python — mypy --strict (equivalent: a mypy.ini / pyproject.toml with all strict flags on: disallow_untyped_defs, disallow_any_generics, warn_return_any, strict_equality, no_implicit_optional). Alternative: pyright in strict mode for faster IDE feedback.
  • Rust — type safety is non-optional. Add #![deny(clippy::all)] and #![warn(clippy::pedantic)] for additional rigor.
  • Go — Go's type system is weaker than TypeScript's or Rust's. Compensate with golangci-lint at high strictness: errcheck, staticcheck, govet, nilness, and project-specific lints. Avoid interface{} / any at API boundaries.

Escape hatches — any in TypeScript, cast() / Any / # type: ignore in Python, unsafe blocks in Rust — are not banned, but each use must include a comment explaining why and a tracking issue if it should eventually be removed. A repo-wide count of escape hatches is computed in CI and must decrease (or stay flat) quarter over quarter.

Why we need it

The type system is the cheapest, fastest, most exhaustive code reviewer available. It runs in under thirty seconds on a typical service, checks every line, and never has a bad day. The 2017 Microsoft Research paper by Gao, Bird, and Barr — To Type or Not to Type: Quantifying Detectable Bugs in JavaScript — replayed real-world bug-fix commits from open-source JavaScript projects with TypeScript and Flow type annotations and showed that 15% of the bugs would have been caught at compile time. That is one in seven production incidents prevented by turning on a config flag.

Disabling type-checking because "we don't have time" is a strictly worse trade than disabling tests: tests cover a subset of paths chosen by humans; types cover all paths automatically. A repo that has accumulated any in critical boundaries (API requests, database rows, third-party SDK returns) has paid no upfront cost — but every bug-fix in those boundaries is now a debug session instead of a compile error.

How others do it

Microsoft (TypeScript at Microsoft)

Microsoft built TypeScript at Anders Hejlsberg's instigation and migrated several internal JavaScript codebases, including parts of Office and Visual Studio, to it. The published guidance is uncompromising: enable strict on day one for new projects, and migrate existing projects file-by-file with // @ts-check first, strict later. The Microsoft Research paper that quantified the bug-catch rate at ~15% (Gao, Bird, Barr 2017) drove industry adoption.

What we take: Strict from day one is cheap. Retrofitting strict mode onto a non-strict codebase costs roughly one engineer-week per 10k lines, so the longer you delay, the more it costs.

Source: Gao, Bird, Barr (2017) — To Type or Not to Type

Stripe (Sorbet — typed Ruby)

Stripe's main monolith is in Ruby, a dynamically-typed language. Rather than accept the productivity tax, Stripe built and open-sourced Sorbet, a gradual type checker for Ruby. The checker now runs over millions of lines of Stripe's code, the rollout was incremental (file-by-file with explicit typed: true), and the engineering blog documents the migration honestly: it took years, it was hard, and the payoff was a quantifiable drop in production incidents tied to type errors.

What we take: Gradual typing works at very large scale if the migration is treated as a sustained engineering investment. File-level opt-in (typed: true) is the only way to move a multi-million-line codebase without a multi-quarter big-bang freeze.

Source: Stripe — Sorbet: Stripe's type checker for Ruby

Airbnb (TypeScript migration)

Airbnb migrated its frontend codebase from JavaScript to TypeScript over a multi-year programme, documented publicly on the Airbnb Engineering blog. The migration used codemods, file-by-file conversion, and a "no new JS files" rule once a directory had any TS file. Brie Bunge's React Conf talk reported that ~38% of bugs in their post-mortems were preventable by TypeScript. The migration's biggest internal blocker was untyped third-party libraries; the team contributed types upstream and used DefinitelyTyped extensively.

What we take: "No new untyped files" is a powerful policy. It stops the bleeding before any retro-conversion starts and naturally drives engineers to migrate adjacent files when they touch them.

Source: Airbnb Engineering — ts-migrate at scale

The pattern: type safety is treated as infrastructure, not preference. The migration is gradual, the budget for escape hatches is finite and tracked, and "no new untyped code" is a non-negotiable rule.

Anatomy of a strictly-typed codebase

Strict mode in the config

tsconfig.json with "strict": true + "noUncheckedIndexedAccess": true. pyproject.toml with [tool.mypy] strict = true. Cargo.toml with #![deny(clippy::all)]. Committed to the repo, no per-developer overrides.

CI as a required gate

The type-check step is a required PR check. A PR with a type error cannot merge. The check runs in under 60 seconds on a typical service so it is part of the normal feedback loop, not a separate slow job.

Typed boundaries

API requests/responses, database rows, and third-party SDK returns are typed at the boundary — never any. Use Zod / Pydantic / serde for runtime validation that produces a typed value. The internal code never has to second-guess what shape a value has.

Escape-hatch policy

Each any / cast / type: ignore includes an inline comment: // any: third-party SDK has no types, see issue #1234. A linter rule (@typescript-eslint/no-explicit-any as error with allowed-with-comment escape) enforces it.

Escape-hatch budget

A script in CI counts any / cast / type: ignore occurrences and stores the count. The count must be monotonically decreasing (or flat) across quarters. A PR that introduces a new one without removing another fails the gate.

"No new untyped files" rule

For gradual migrations: a CI check fails on any new .js file or any .py file without type annotations. Old files migrate when touched; new files are typed from inception. Within 6–12 months the codebase reaches full strict coverage with no migration sprint.

Common pitfalls

Pitfall Why it happens What to do instead
any / cast escape hatches accumulate silently Each escape hatch feels small ("just this one to unblock the PR"). There is no budget, no tracking, and no review of existing hatches. After a year the codebase has hundreds of anys and the type system is decorative — the most error-prone code paths are precisely the ones with no type information. Count escape hatches in CI on every push and store the count. Require a monotonically-decreasing budget. Lint rules (@typescript-eslint/no-explicit-any) make every new one a deliberate decision with a reviewer's name on it.
Third-party types stubbed as any and forgotten A third-party SDK has no types or has incomplete types. The fastest unblock is declare module "x" { export const foo: any; } or import x # type: ignore. The stub is committed, the engineer moves on, and the boundary to the SDK becomes a type-free zone forever. Type the boundary, not the SDK. Wrap the SDK call in a thin adapter that returns a fully-typed domain value, even if the SDK itself is any-typed underneath. The any is contained to a single function; the rest of the codebase only sees the typed adapter.
Strict turned on for new code only, never finished The team enables strict mode "going forward" but the existing 80% of the codebase stays under loose mode with per-file overrides. The migration was supposed to happen "in slack time" and three years later 65% of files are still on the override list. Engineers don't trust the type-checker because half the codebase lies to it. Adopt the "no new untyped files" rule and the "if you touch it, you migrate it" rule together. Track the override-list size as a monotonically-decreasing metric. Assign a quarterly time-box for sustained migration work, not "whenever".
