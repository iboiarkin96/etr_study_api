Feature Flags — decouple deploy from release

The deploy installs the code; the release flips the flag. Two different decisions, two different rollback paths, two different signals.

TL;DR

A feature flag is a runtime switch that gates a code path. The pattern was popularised at Facebook (Gatekeeper) and Flickr in the 2000s and commercialised by LaunchDarkly from 2014 onward; Pete Hodgson's 2017 essay on martinfowler.com is the canonical reference. With flags, "deploy" stops meaning "release to users." Code ships dark, then is enabled for an internal cohort, then a percentage of users, then everyone — each step independently reversible by a config flip rather than a redeploy. Large engineering orgs typically run with hundreds of active flags plus an explicit lifecycle process so flags don't accumulate forever.

What it is

A feature flag (or feature toggle) is a runtime conditional that selects between an old and a new code path based on an external decision: a config file, a flag-service lookup, a user attribute, or a percentage rollout. Pete Hodgson's taxonomy distinguishes four types: release toggles (decouple deploy from release), experiment toggles (A/B tests), ops toggles (kill-switches for expensive features), and permission toggles (entitlements per plan).

Operationally, a flag platform — LaunchDarkly, Statsig, GrowthBook, Unleash, or an in-house service — stores the current flag state and pushes changes to running processes within seconds. Application code asks if (flags.isEnabled("new-checkout", user)) { ... } and the platform decides per-call. The platform also stores targeting rules (by user ID, by plan, by region) and rollout percentages.

The discipline part is the flag lifecycle: every flag has an owner, a default-on date, and a removal date. After the removal date, CI fails if both branches still exist in code. This prevents the dominant failure mode where flags accumulate into permanent forks of the codebase.

Why we need it

Without flags, "rollback" means revert-and-redeploy. On a large monorepo with a multi-stage build, a multi-region rollout, and post-deploy migrations, that is a 20–40 minute operation — enough time during an active incident to triple the user impact. Flags collapse rollback to a single config flip applied in seconds.

Flags also let you separate two questions that genuinely deserve two different tests: "Is this code safe?" (verified by canary metrics and progressive rollout) and "Is this change a good idea?" (verified by an A/B test against an outcome metric). Without flags those two get tangled together, and a product decision masquerades as an engineering decision or vice versa. With flags, both can be answered independently — and a feature that is technically safe but commercially neutral can be turned off without reverting the code.

How others do it

Facebook — Gatekeeper

Facebook built Gatekeeper in 2007 as the original large-scale feature-flag system. Every product ships behind a Gatekeeper rule that can target by employee, country, region, friend graph, or random percentage. Releases happen via gatekeeper changes, not via deploys; deploys themselves are continuous and decoupled from product launches.

What we take: The targeting model is as important as the on/off switch. "Only employees" → "only employees and friends-of-employees" → "1% of US users" gives many cheap detection points before the change reaches everyone. Build targeting in from day one rather than retrofitting it.

Source: Pete Hodgson — Feature Toggles (martinfowler.com)

Etsy — gradual rollouts

Etsy's engineering blog (Code as Craft) has documented their use of feature flags since around 2011: every significant change ships behind a "config flag," initially set to 0%. The flag is ramped over hours or days with explicit dashboards comparing the new and old cohorts on conversion, error rate, and latency. Reverting a bad release is one curl call.

What we take: Pair every release flag with a dashboard that compares the new cohort to the control on the metric the change was supposed to move. Without the dashboard, "we rolled it out" is unmeasured and "rollback" is opinion-driven.

Source: Code as Craft — Etsy engineering

LaunchDarkly customers (GitHub, Atlassian)

LaunchDarkly's published case studies (with GitHub, Atlassian, IBM) describe production usage in the hundreds to low thousands of active flags per org. Both GitHub and Atlassian operate explicit "flag debt" reviews where stale flags are surfaced and either promoted to permanent config, deleted, or assigned an owner with a deadline.

What we take: Flags are operational debt with a half-life. Without a quarterly cleanup, the count grows monotonically and the code becomes a labyrinth. Build the cleanup process before you have 100 flags, not after you have 500.

Source: launchdarkly.com/blog

The repeating pattern: targeted rollouts (not on/off), paired metrics dashboards, and an explicit lifecycle policy. Teams that adopt flags without a lifecycle accumulate them into a worse codebase than they would have had without flags at all.

Anatomy of a feature flag

Flag key + owner

Every flag has a unique key, a named owner, and a one-line description. Anonymous flags are forbidden — no owner means no cleanup.

Targeting rules

Rules evaluated in order: internal users, beta cohort, percentage rollout (1% / 5% / 25% / 50% / 100%), default. Each rule is a runtime config change.

Default-on date

A target date when the flag should be at 100% and removable. Set when the flag is created; visible in the dashboard if missed.

Paired dashboard

For every release flag, one Grafana panel comparing new vs. control cohorts on the metric the change is supposed to move (and on error rate + p99 latency).

Kill-switch path

A documented "how to revert in 60 seconds" runbook: which flag, which UI, who has permission. Linked from the PR that introduced the flag.

Cleanup gate

CI check that fails if a flag is past its default-on date and still has both branches in code. Forces deletion to be a real task, not a Wishlist item.

Common pitfalls

PitfallWhy it happensWhat to do instead
Flag became a permanent fork in codeThe "old path" was never deleted after the flag reached 100%, because there's always something more urgent. Six months later, both branches must be maintained forever and nobody remembers which one is the truth.Every flag gets a default-on date at creation. CI fails after that date if both branches still exist. Removing a flag is a 30-minute task that lands on the owner's plate automatically.
Flags accumulate without cleanupAdding a flag is one line; deleting it is a real refactor. So the count grows monotonically. Eventually, evaluating a single user requires hundreds of flag checks and the codebase has dozens of stale forks.Quarterly "flag debt" review on the team calendar. Surface the oldest 10% of flags and either promote them to permanent config or assign a deletion ticket with an explicit owner.
Flag flip causes regression because no metricThe flag was treated as a deploy switch, not as an experiment. Nobody is watching the dashboard while it ramps. The 10% rollout damages 10% of users for hours before someone notices.Every release flag must have a paired dashboard linked from the flag definition. Ramping to a new percentage requires checking the dashboard. No dashboard, no ramp; the flag stays at the current step.
