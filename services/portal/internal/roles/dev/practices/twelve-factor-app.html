Twelve-Factor App — portable, scalable, ops-friendly

Twelve rules from Heroku that make a service behave the same on a laptop, in CI, and on autoscaling production.

TL;DR

The Twelve-Factor App is a methodology published by Adam Wiggins at Heroku in 2011 for building portable, cloud-native services. It defines twelve rules — one codebase, declared dependencies, config in environment variables, stateless processes, logs as event streams, dev/prod parity, and more. Every modern PaaS and orchestrator (Kubernetes, Cloud Run, Fly.io, Render, Heroku itself) assumes a service follows these rules; violations show up as ops pain, not compile errors. Retrofitting a non-conforming codebase costs weeks per service, so the rules are cheapest when applied from day one.

What it is

The Twelve-Factor App is a methodology by Adam Wiggins (Heroku co-founder), published in 2011 at 12factor.net, that codifies what makes a service "cloud-native". It captures the patterns Heroku saw in services that scaled gracefully versus ones that failed under autoscaling, container restarts, or multi-region deployment.

The twelve rules, in order, are:

  1. Codebase — one codebase tracked in version control, many deploys.
  2. Dependencies — explicitly declared and isolated (no system-wide packages).
  3. Config — stored in environment variables, never in the codebase.
  4. Backing services — treated as attached, swappable resources (DB, cache, queue).
  5. Build, release, run — three strictly separated stages.
  6. Processes — stateless and share-nothing; any local state is fair game to lose.
  7. Port binding — the app self-contains its HTTP server and binds to $PORT.
  8. Concurrency — scale out by running more processes, not bigger ones.
  9. Disposability — fast startup, graceful shutdown on SIGTERM.
  10. Dev/prod parity — keep development, staging, and production as similar as possible.
  11. Logs — treated as event streams written to stdout; the platform routes them.
  12. Admin processes — one-off tasks (migrations, REPL) run as identical-environment processes.

A service is "twelve-factor" only if every rule holds. The methodology is binary by design — a service that violates rule III alone is still vulnerable to leaked secrets, no matter how diligently it follows the other eleven.

Why we need it

The most common production failures in cloud-native systems are not algorithmic bugs — they are infrastructure-shaped bugs caused by 12-factor violations. A service that writes uploaded files to local disk works perfectly on a single VM and dies the first time it is run behind a load balancer. A service that holds in-memory user sessions handles 1 RPS in dev and corrupts every other login under autoscale. A service that reads config.prod.json at boot ships its secrets to GitHub the next time someone forks the repo.

Twelve-factor is cheap when followed from day one and expensive to retrofit. A non-conforming service typically takes one to three weeks per rule to fix at scale — moving sessions to Redis, extracting secrets, untangling boot-time config, adding SIGTERM handlers. Multiplied across a portfolio of services, this is the difference between a platform team that ships features and one that spends every quarter unblocking the previous quarter's mistakes.

How others do it

Heroku (origin)

Heroku published the methodology in 2011 based on years of running thousands of customer apps on a shared PaaS. Every Heroku buildpack and the platform's contract with apps assume strict twelve-factor conformance: the dyno can be killed at any moment, config arrives via env vars, logs are read from stdout. Apps that violate the rules do not run reliably on Heroku — the platform's contract is the methodology.

What we take: The methodology is the platform contract. Treat 12-factor as the floor: a service that violates a rule is broken, not "good enough but…".

Source: 12factor.net — Adam Wiggins

Google Cloud

Google Cloud's architecture guides treat twelve-factor as the baseline for Cloud Run, GKE Autopilot, and App Engine. Their reference document "Twelve-factor app development on Google Cloud" maps each rule to the GCP service that enforces it: Secret Manager for config, Cloud Logging for streams, Cloud Build for the build/release/run separation. Cloud Run's $PORT contract and 10-second graceful-shutdown window are literal implementations of rules VII and IX.

What we take: Map each rule to a concrete piece of infrastructure. "Config in env" is not abstract — it is "use Secret Manager and inject via deployment template".

Source: Google Cloud — Twelve-Factor on GCP

Kubernetes ecosystem

Kubernetes does not enforce twelve-factor, but it punishes violations harshly. Pods are killed and rescheduled on any node failure (disposability), config arrives via ConfigMap and Secret objects mounted as env vars (config), and logs are scraped from stdout by the kubelet (logs). A service that holds in-memory state survives until the first rolling deployment, at which point it loses all of it. The de-facto industry standard for cloud-native services is twelve-factor on Kubernetes.

What we take: If you run on Kubernetes, you have already implicitly committed to twelve-factor. Violations only delay the moment of failure; they do not prevent it.

Source: Kubernetes docs — ConfigMap

The pattern: the rules are not aspirational. They are the contract that every modern runtime — Heroku, Cloud Run, GKE, Fly.io, Render — assumes you have signed. Violations are tolerated only at low scale, and only until the first incident.

Anatomy of a twelve-factor service

Single repo, many deploys

One service ↔ one repository ↔ one Dockerfile. The same artefact is promoted from dev to staging to prod; the only difference between environments is configuration, never source code. No "prod branch".

All config in environment variables

Database URLs, API keys, feature flags — all from process.env at boot. Zero config.prod.json files. Secrets live in a secret manager (AWS Secrets Manager, GCP Secret Manager, Vault) and are injected by the deployment platform.

Stateless process model

No local disk writes that need to survive a restart. No in-memory user sessions, caches that store unique data, or upload buffers. Anything persistent goes to a backing service — Postgres, Redis, S3. The process can be killed and replaced at any second.

Logs to stdout

The app writes structured (JSON) log events to stdout and never opens a log file. The platform (Kubernetes, Cloud Run, journald) is responsible for capture, rotation, and shipping. Practice 16 — Structured Logging — covers the JSON schema.

Graceful SIGTERM handling

On SIGTERM the app stops accepting new requests, drains in-flight ones (typically within 10–30 seconds), and exits cleanly. Without this, every rolling deploy returns 502s to whichever percentage of requests were mid-flight.

Conformance checklist in service template

Every new service is generated from a template that already conforms (Dockerfile, env-driven config, stdout logging, SIGTERM handler). A pre-merge CI check flags violations. Drift is caught at the start, not at the next incident.

Common pitfalls

Pitfall Why it happens What to do instead
Secrets committed to git It's easier to drop an API key into a .env file and commit it than to set up a secret manager. The repo is "private", so it feels safe. Treat any secret in git as compromised. Move all secrets to a secret manager from day one (AWS Secrets Manager, GCP Secret Manager, Vault). Add a pre-commit hook (gitleaks / detect-secrets) and a CI step that scans diffs.
In-memory cache that breaks under autoscale A single-instance app uses an in-process LRU cache for sessions or computed values. It works fine in dev and at 1 RPS. Autoscaling adds a second instance, and now half of requests miss the cache, double-billing or invalidating logged-in users. If the cache is correctness-sensitive (sessions, idempotency keys, rate limits), move it to Redis or Memcached. If it is purely a hot-path speedup, accept that each instance has its own copy and size it accordingly.
Logs written to a file the autoscaler can't reach The app opens logs/app.log and rotates it with logrotate. On Cloud Run or Kubernetes the file goes to an ephemeral disk that the platform tears down with the pod, so logs vanish on every deploy and are unreachable during the incident that needed them. Write logs to stdout only. Let the platform (kubelet, Cloud Logging, journald) capture and ship them. Practice 16 — Structured Logging — defines the JSON schema and severity convention.
