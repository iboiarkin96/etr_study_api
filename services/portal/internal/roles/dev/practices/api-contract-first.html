API Contract-First — OpenAPI / GraphQL SDL / Protobuf before code

Write the schema first. Generate the server, the SDKs, the mocks, and the tests from it. Reject any PR that introduces an undocumented endpoint.

TL;DR

Contract-first API development treats the schema — OpenAPI for HTTP, GraphQL SDL for GraphQL, Protobuf for gRPC — as the source of truth, written before any handler. From that one document, CI generates the server stub, the client SDKs, mocks for consumers, and contract tests. Stripe famously generates SDKs in seven languages from a single OpenAPI source; GitHub's GraphQL schema is the canonical interface contract for thousands of integrators; Buf has built an entire Protobuf-first toolchain on the same premise. Code-first APIs drift; contract-first APIs stay in sync because the schema is the build input, not a side-effect.

What it is

Contract-first API development means the API's interface — its endpoints, fields, types, error codes — is written in a machine-readable schema language before any handler code, and that schema is the build input that produces everything else. The three dominant schema formats are:

  • OpenAPI (formerly Swagger) — for HTTP/JSON APIs. YAML or JSON. Maintained by the OpenAPI Initiative.
  • GraphQL SDL — for GraphQL APIs. The schema describes types, queries, mutations, and subscriptions.
  • Protobuf — for gRPC and other binary RPC protocols. .proto files. Buf provides the de-facto industry toolchain (lint, breaking-change detection, registry).

From the schema, CI generates:

  1. The server stub — routes, request/response models, validation. Engineers fill in business logic only.
  2. The client SDKs — one per supported language, all type-safe, all in sync.
  3. The mocks — runnable servers (Prism, WireMock) so consuming teams can integrate before the real backend exists.
  4. The contract tests — fuzz tests (Schemathesis) that hammer the running service against the schema and fail on any drift.

Breaking-change detection runs in CI on every PR (oasdiff for OpenAPI, buf breaking for Protobuf, GraphQL Inspector for GraphQL). A PR that removes a required field or changes a type without bumping the version is blocked at review.

Why we need it

Code-first APIs drift. The handler exists, the docs are written by hand in Confluence, the SDK is a different team's reimagination of the server's behaviour, and the mock is whatever the frontend team hand-rolled last quarter. Within six months, all four describe slightly different APIs. The bug reports that follow — "the docs say this field is optional but the server rejects null" — are not bugs in the code; they are bugs in the absence of a single source of truth.

Contract-first inverts the dependency: the schema is the input, everything else is the output. A breaking change at the schema layer fails CI before the code is even compiled. A new field appears in the SDK the same hour it appears on the server. A consuming team building against the mock cannot diverge from what the real service will eventually return. The compounding return is that integration cost — the largest hidden cost of multi-team API work — collapses.

How others do it

Stripe

Stripe maintains a single OpenAPI specification as the canonical description of its API and generates client libraries in seven languages (Ruby, Python, PHP, Java, Node, Go, .NET) from it. Versioning is date-based: every breaking change introduces a new dated version, and the schema preserves all historical versions so old clients keep working forever. The schema, the docs at stripe.com/docs/api, and the SDKs all derive from the same source — drift is structurally impossible.

What we take: The schema must be the only source. If the docs are written separately or the SDK is hand-rolled, you have already lost. Generate everything from one file.

Source: Stripe — API versioning at Stripe

GitHub (GraphQL)

GitHub's GraphQL API has been the recommended integration surface since 2017. The SDL schema is the contract — public, downloadable, versioned via schema-evolution rules rather than URL versions. Deprecated fields stay in the schema with @deprecated directives; breaking changes go through a public preview period. The Octokit SDKs are generated from the introspection result, not hand-written.

What we take: GraphQL's schema evolution model — additive changes are safe, deprecations stay visible — gives consumers a clear migration path without version sprawl. Use @deprecated, never silent removals.

Source: GitHub — GraphQL API docs

Buf (Protobuf platform)

Buf, founded by ex-Uber engineers who built Uber's internal Protobuf platform, ships the de-facto industry toolchain for contract-first gRPC: buf lint enforces naming and style; buf breaking blocks breaking changes against a registered baseline; the Buf Schema Registry hosts schemas the way npm hosts packages. Companies that ship gRPC at scale (Block/Square, Snowflake, Mux) have moved from hand-curated Protobuf workflows to Buf.

What we take: Treat breaking-change detection as a non-negotiable CI gate. Without it, "contract-first" decays into "contract-eventually". Buf for Protobuf, oasdiff for OpenAPI, GraphQL Inspector for GraphQL.

Source: buf.build — Protobuf platform

The pattern across Stripe, GitHub, and Buf is identical: one schema, automated generation, breaking-change detection in CI. The differences are only the schema language (OpenAPI / GraphQL / Protobuf) and the protocol it describes.

Anatomy of a contract-first pipeline

Schema file in the repo root

One openapi.yaml / schema.graphql / service.proto per service, version-controlled. Linted by spectral / graphql-eslint / buf lint on every PR. No endpoint exists until it is in this file.

Server stub generator

CI runs openapi-generator / graphql-codegen / protoc on every push and produces typed handler signatures, request/response models, and validation middleware. Engineers fill in business logic only — they cannot deviate from the schema's contract.

Client SDK generator

One generator pass per target language (TypeScript, Go, Python, Java). Outputs published to the internal package registry on every merge. Consumers depend on the generated package — never hand-roll an HTTP client. Stripe generates 7 SDKs this way.

Mock server

Prism (OpenAPI) / Apollo Server mock mode / a mock service generated from .proto — runs against the same schema and returns example data for every operation. Consumer teams develop against the mock before the real handler exists, eliminating cross-team blocking.

Breaking-change gate in CI

On every PR, run oasdiff breaking / buf breaking / GraphQL Inspector against the main-branch schema. A removed field, a tightened type, or a changed enum value fails the build. Bypassed only with an explicit version bump and reviewer sign-off.

Contract test suite

Schemathesis (OpenAPI) or buf curl (Protobuf) fuzz-tests the running service against the schema in CI: generates random valid requests, asserts the response matches the schema. Catches the case where the handler accidentally diverged from the contract — even though the schema didn't change.

Common pitfalls

Pitfall Why it happens What to do instead
Schema is generated AFTER the code (annotation-driven OpenAPI) The team adopts springdoc-openapi or drf-spectacular, sprinkles annotations on handlers, and calls it "contract-first". In practice the schema is whatever the handler happens to produce, which defeats the point: the contract cannot fail CI because it is downstream of the code. The schema file is the input. Generate handlers from it, not the other way around. If the framework only supports annotation-driven generation, write the schema by hand and pin the handler to it with a CI diff check that fails on drift.
Breaking change shipped because no compat check in CI The schema sits in the repo but no automated check compares the new version to main. An engineer removes a "deprecated" field that one customer is still using, ships, and the customer's integration breaks silently in production. Run oasdiff breaking / buf breaking on every PR as a required check. Treat "breaking change" as a structural concept (field removed, type tightened, enum value dropped), not a judgment call.
Client SDKs are hand-written and drift from the schema "The generator output is ugly" or "we want a more idiomatic SDK", so the SDK is hand-rolled. Six months later the SDK supports endpoints that no longer exist and is missing fields that have been there for a quarter. Use the generator output as the SDK, even if it is verbose. If idiomatic ergonomics matter, wrap the generated client with a thin hand-written facade — but never duplicate the transport layer. The generator is the only thing that can stay in sync.
