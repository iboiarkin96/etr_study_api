Pre-commit hooks — fast feedback before push

A bundle of fast, deterministic checks runs on staged files at commit time. The same hooks re-run in CI, but most issues never reach the PR.

TL;DR

Pre-commit hooks shift fast feedback to the cheapest moment: before the commit is even written. The pre-commit framework (pre-commit.com, by Anthony Sottile, 2014) is the de facto standard; Husky and lefthook are the JS/Node equivalents. A typical bundle: formatter, linter, type-check on staged files, secret scan (gitleaks), conventional-commits validator. The discipline rule: the same hook bundle runs in CI as the source of truth — laptops and CI never disagree, and engineers never spend reviewer time on indent comments again.

What it is

Pre-commit hooks are scripts that run automatically when an engineer runs git commit, before the commit is recorded. Git has native hooks (.git/hooks/pre-commit), but the practical pattern is to use a framework — pre-commit (Python), Husky (Node), or lefthook (Go) — that stores the hook configuration in the repo and installs it on every clone via a single command.

A typical bundle has six categories: formatters (Prettier, gofmt, Black) that rewrite the staged files in place; linters (ESLint, ruff, golangci-lint) that flag violations; type-checkers on changed files (tsc, mypy, pyright); secret scanners (gitleaks, ggshield) that block accidental credential commits; commit-message validators (commitlint) that enforce Conventional Commits; and a fast subset of tests for the files being committed.

The non-negotiable constraint is speed: the entire bundle must finish in < 5 seconds on staged files, otherwise engineers will routinely use --no-verify and the hooks become decorative. The same bundle re-runs in CI via pre-commit run --all-files as the source of truth, so a laptop pass and a CI pass are guaranteed to agree.

Why we need it

Without pre-commit hooks, style and format comments dominate PR review and crowd out substantive feedback. A reviewer spending 10 minutes on a 200-line PR has limited attention; if half of it is consumed by "wrong indent" or "missing trailing comma," the architecture problem in the middle goes unnoticed. Hooks turn that whole class of comment into a fix on the engineer's own keyboard, before commit.

Hooks also catch the small number of high-cost incidents — accidentally committing an AWS key, a private key, or a database password — at the strictly cheapest moment. Once a secret reaches the remote, removing it requires git-history rewriting (BFG, git-filter-repo) and credential rotation; catching it before commit costs zero. Gitleaks and similar tools take < 1 second on staged files and pay for themselves on the first prevented leak.

How others do it

Linux kernel — patch process

The Linux kernel uses a strict patch submission process where scripts/checkpatch.pl is the de facto pre-commit gate: it checks formatting, style, and a long list of conventions before a patch can even be sent to the relevant mailing list. Maintainers reject patches that haven't been run through checkpatch as a matter of course. The script lives in the repo, so every contributor uses the same rules.

What we take: The checker tool ships with the code, not as a separate doc. Contributors don't need to remember the rules; the tool tells them. Pair this with a CI re-run so the local check and the merge gate are literally the same script.

Source: pre-commit.com

Spotify / Airbnb — public pre-commit configs

Both Spotify and Airbnb publish open-source style guides and pre-commit-style configs (Airbnb's JavaScript style guide is among the most-starred repos on GitHub). Internal repos use centralised hook bundles distributed via a shared base config; new repos inherit the bundle automatically and can override only specific entries.

What we take: A centrally maintained hook bundle, inherited by every repo, is what makes the practice scale across an org. Per-repo bespoke hooks drift; a shared base config keeps every codebase consistent and reduces the cost of moving engineers between teams.

Source: github.com/airbnb/javascript

gitleaks — secret scanning

Gitleaks is the most widely deployed open-source secret scanner: it ships as a pre-commit hook and a CI action with a default ruleset covering AWS, GCP, Azure, Stripe, GitHub, Slack, and ~150 other providers. The hook runs in well under a second on staged files. Custom regex rules are added in-repo for org-specific token formats.

What we take: Secret scanning belongs at commit, not at PR. A leaked credential that reaches the remote has to be assumed compromised even if the PR is immediately reverted — once it's in the reflog, treating it as still secret is wishful thinking. Block it before it leaves the laptop.

Source: github.com/gitleaks/gitleaks

The repeating pattern: hooks live in the repo (not in personal dotfiles), the same hooks re-run in CI, and the bundle is fast enough that --no-verify is unnecessary. Teams that drift from any of these three see hooks erode within a quarter.

Anatomy of a pre-commit bundle

Formatter — rewrites in place

Prettier / gofmt / Black / ruff format. Auto-fixes; never argues with the engineer. Target: < 1s on staged files.

Linter — flags issues

ESLint / ruff / golangci-lint with project-tuned ruleset. Hard-fail on errors, no warnings (treat warnings as errors or remove them).

Type-check on changed files

tsc / mypy / pyright restricted to the modules touched. Full-repo type-check stays in CI; the hook is a fast subset.

Secret scan

gitleaks or trufflehog on staged content. Default ruleset + a regex for the org's own token formats. < 1s budget.

Commit-message validator

commitlint with Conventional Commits config. Runs on the commit message itself, not on the diff; rejects non-conformant messages with the example fixed up.

CI re-run as truth

pre-commit run --all-files in CI, blocking merge. The merge gate and the local gate are literally the same script with the same versions, pinned in .pre-commit-config.yaml.

Common pitfalls

PitfallWhy it happensWhat to do instead
Hooks too slow, engineers --no-verifySomeone added a full-repo test run or a heavyweight static analyzer to the hook. Now git commit takes 45 seconds. The first time engineers are mid-flow, they pass --no-verify "just this once" — and it becomes muscle memory.Hard budget the bundle at < 5 seconds on staged files. Move slow checks (full type-check, full test suite, dependency audit) to CI only. Measure hook duration in telemetry and block additions that exceed the budget.
Hooks differ between laptop and CIThe hook bundle is documented in a README; everyone installs it slightly differently. Local versions of tools drift. A commit passes locally and fails in CI on something nobody can reproduce.Pin every tool version in .pre-commit-config.yaml. CI runs pre-commit run --all-files, the same command an engineer can run locally to debug. Single source of truth, single failure mode.
Secret flagged after commit, history rewrite painfulThe secret scanner is in CI only. The credential reaches GitHub before the scan finds it; rotation + history rewrite + force-push to every clone is the only safe response. Half a day per incident.Secret scan goes in the pre-commit hook with a hard-fail. Catching at commit is < 1s; catching at CI is an incident. Both can exist, but the local one must be there.
