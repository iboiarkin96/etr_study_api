Test-Driven Development — red, green, refactor

Write the failing test first; let the test drive the smallest production change that makes it pass.

TL;DR

Test-Driven Development, formalised by Kent Beck in Test-Driven Development: By Example (2002), is a three-step micro-cycle: write a failing test (red), write the smallest code that makes it pass (green), then refactor with the safety of the green bar. TDD treats the test as the executable specification and the production code as a consequence. The output is not "more tests" — it is code with a small, composable public surface that survives change because every line was demanded by a test.

What it is

Test-Driven Development is a programming rhythm with three explicit, named steps:

  • Red. Write the smallest test that captures one new behaviour. Run the suite; the new test fails. The failure must be the failure you expect — wrong output, not a missing file. Red proves the test is exercising something.
  • Green. Write the simplest production code that makes the test pass. "Simplest" includes returning a hard-coded value if that is the minimum that satisfies the test. Resisting cleverness here is the point.
  • Refactor. With the green bar as a safety net, restructure the code — rename, extract, deduplicate — without changing behaviour. Run tests after every change. The refactor step is where design emerges; skipping it produces code that works but does not improve.

The cycle takes two to ten minutes. The shortness of the loop is the discipline. A 30-minute "TDD cycle" with one test and a finished feature is not TDD — it is test-after-with-a-trick.

Kent Beck's Test-Driven Development: By Example (2002) is the canonical reference; Martin Fowler's bliki entry summarises the practice and its common misunderstandings.

Why we need it

Code written test-after tends to be code written without testability in mind. The tests that result exercise implementation details — private methods, internal state, mocked collaborators — because there is no other way to reach the code. A test suite of that shape is brittle: any refactor breaks dozens of tests, so refactoring stops, so the code rots.

TDD inverts the pressure. Because the test exists before the code, the code must be reachable from the test, which means the public surface is small, the dependencies are explicit, and the behaviour is observable from outside. The shape of the resulting code is testable by construction, not by luck.

Good looks like: every new module has tests written first; refactors run weekly without breaking suites; the team trusts the test bar enough to deploy on green.

How others do it

Kent Beck (originator)

Beck described TDD in Test-Driven Development: By Example (2002), drawing on practices he had been using since the late 1990s as part of Extreme Programming at Chrysler's C3 project. The book works through two example projects step by step — a money library and a unit-testing framework — making the rhythm visible at the keystroke level. Beck's two rules: never write production code without a failing test; never write more production code than is needed to pass the current test.

What we take: Read the book's first three chapters in a pairing session and try the rhythm on a small ticket. The kata is more useful than any abstract argument for TDD.

Source: Kent Beck — TDD by Example (Addison-Wesley)

Martin Fowler — clarifying the practice

Fowler's bliki entry on TDD is a regularly-updated reference that addresses the most common misreadings: TDD is not about testing, it is a design technique; the refactor step is not optional; coverage is a side effect, not the goal. Fowler also documents the "Mockist vs Classicist" distinction — whether refactor-time tests rely on interaction mocks or end-to-end state — which is the most-litigated TDD style question.

What we take: Default to the classicist style (state-based assertions over mocks). Mockist tests couple to the call graph and become a refactor anchor; classicist tests assert on output and survive internal change.

Source: martinfowler.com — TestDrivenDevelopment

Pivotal Labs — TDD as the team default

Pivotal Labs (now part of VMware Tanzu Labs) made TDD the default working mode of every engagement, paired with full-time pair programming. Engineers do not commit code without a test that drove it. The combination of pairing and TDD is treated as the bar: a violation surfaces in real time because the pair partner notices the production-first move. The Labs blog and Tanzu engineering writing document this rhythm in many client engagement retrospectives.

What we take: TDD adoption sticks when paired with another practice that catches lapses in real time. Solo TDD requires more discipline than most engineers can maintain; pair TDD is self-enforcing.

Source: VMware Tanzu Labs (formerly Pivotal Labs)

Across teams that practise TDD at scale, the same observation recurs: TDD is a design technique that produces tests as a side effect, not a testing technique that produces design as a side effect.

Anatomy

Red step — a real failure

The test must fail with the expected assertion, not a missing-import or syntax error. If the first run is not red for the right reason, the test does not yet exercise the behaviour. Target cycle time: under 10 seconds from test write to red bar.

Green step — minimum viable code

Write the simplest thing that makes the test pass, even if it is a hardcoded return value. The "obvious implementation" rule applies only when the implementation is genuinely obvious; otherwise prefer the smallest change. Target cycle time: under 2 minutes.

Refactor step — design emerges here

After every green, look for duplication, unclear names, and seams that want to become functions. Run tests after each small change. The refactor step is non-optional; skipping it accumulates the same code-rot TDD was meant to prevent.

Short cycle time

A full red-green-refactor cycle should take 2–10 minutes. Test suites must run in under 30 seconds for the inner loop, or the rhythm collapses. Slow tests are a TDD anti-feature, not a TDD necessity.

Classicist test style

Tests assert on observable output and persisted state, not on which collaborators were called. Mocks are reserved for boundaries the test cannot reach (external APIs, clocks). Internal collaborators are exercised, not mocked.

Test-first ratio in CI

Heuristic check in CI that flags commits where production code was added in the same commit with no preceding test commit. Target: > 60% of new logic written test-first within the quarter, climbing toward 90%.
