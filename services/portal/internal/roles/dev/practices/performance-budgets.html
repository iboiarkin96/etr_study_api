Performance budgets — measured, not assumed

The engineer's PR runs Lighthouse-CI on the changed page, the bundlesize check on the changed package, k6 on the changed endpoint — and the merge button stays grey until the numbers are back inside the line drawn in perf-budget.json.

TL;DR

The idea was popularised by Tim Kadlec in Setting a Performance Budget (2013) and is the official guidance on web.dev — Performance Budgets 101. The mechanism is now standard: pick a small set of metrics (LCP, API p99, bundle size), set a numeric target, measure on every PR, fail the build above the target. Lighthouse-CI for the web, k6 or Artillery for APIs, bundlesize / size-limit for JS bundles. Budgets must be enforced on real-user metrics, not on dev hardware, otherwise you ship a slow site to a fast laptop.

What it is

A performance budget is a written, version-controlled numeric limit on a small set of performance metrics, plus an automated check that fails a PR when the limit is exceeded. The practice comes from Tim Kadlec's 2013 article and is the official web.dev guidance.

A useful budget specifies, per surface:

  • Metric — choose 2–5: Largest Contentful Paint (LCP), API p99 latency, JS bundle size (gzipped), Cumulative Layout Shift (CLS), Total Blocking Time. More than five is unmaintainable.
  • Target — a single number per metric. LCP < 2.5s, p99 < 500ms, bundle < 150 KB gz.
  • Measurement environment — the network, device, percentile. "LCP at the 75th percentile on a Moto G4 over 4G" is reproducible. "LCP" on its own is not.
  • Tool — Lighthouse-CI for web vitals, k6 / Artillery for API load, bundlesize / size-limit for bundle, RUM (Real-User Monitoring) for production validation.
  • CI gate — a required check on the PR. Over budget = the merge button is grey.
  • Budget change policy — the budget itself is in a file (perf-budget.json, lighthouse.json); changing it is a PR with a written justification, reviewed like any other PR.

Why we need it

Without budgets, performance regresses one millisecond at a time, no individual PR ever feels worth blocking, and after twelve months the page is five seconds slower than it used to be. The team then spends a quarter undoing the damage with no idea which commits caused it. Budgets force the regression to be visible at the moment it's introduced — when the offending code is still fresh in the author's head and the fix is a 10-minute change instead of a six-week rewrite.

For the engineer, the budget is also a design tool. "This new feature must fit in a 30 KB bundle increase and 20 ms of LCP" is a concrete constraint that informs the architecture choice up front, instead of a vague "make it fast" that no one can verify.

How others do it

web.dev (Google Chrome team)

The canonical industry guidance: pick metrics that map to user experience (Core Web Vitals — LCP, FID/INP, CLS), measure with a real device and a real network, set targets from the 75th percentile of real users, and fail the build when over budget. The web.dev guide includes the Lighthouse-CI budgets.json format and end-to-end CI setup.

What we take: Core Web Vitals are the right starting metrics for a public web surface, because they are what Google measures users with, and because they have established human-perception thresholds (LCP good < 2.5s, needs-improvement 2.5–4s, poor > 4s).

Source: web.dev — Performance Budgets 101

Lighthouse-CI (GoogleChrome)

Lighthouse-CI is the reference implementation: a GitHub Action / CLI runs Lighthouse on configurable URLs in CI, compares to budgets.json, and posts a PR comment with the deltas. Assertions ("assertions": {"largest-contentful-paint": ["error", {"maxNumericValue": 2500}]}) decide pass/fail. Used by Polymer, MUI, many open-source frontends.

What we take: Lighthouse-CI is the path of least resistance for web budgets. Don't invent your own runner. The PR comment with measured-vs-budget deltas is what makes the gate feel collaborative instead of adversarial.

Source: Lighthouse-CI on GitHub

Calibre / Smashing Magazine (industry practice)

Calibre's blog and Smashing Magazine's "Front-End Performance Checklist" have documented the practical operations of perf budgeting at companies like the BBC, Pinterest, Etsy — including the lesson that the budget number must be set from production RUM data (the 75th percentile is the standard target), not from a synthetic test that runs on the CI runner.

What we take: The CI synthetic test is the regression detector. The actual budget target is calibrated against production RUM. Otherwise the team is optimising for the CI runner's CPU, not the user's phone.

Source: Calibre — Performance Budgets & Page Weight

Anatomy of a performance budget

Metrics chosen for user experience, not vanity

LCP, INP, CLS for web; p99 latency for API; bundle gz size for JS. Avoid "load time" and "TTFB alone" — neither correlates strongly with user happiness. 2–5 metrics maximum.

Targets calibrated from production RUM

Pick the 75th-percentile real-user value over the last 28 days. Don't pull numbers from a blog post; don't pull them from a CI runner. The target must reflect what users actually experience.

CI gate on every PR

Lighthouse-CI / k6 / bundlesize as required checks. Above budget = the merge button is grey. The check posts a PR comment with measured-vs-budget deltas so the author sees the impact of their change.

Measurement throttled to user-realistic conditions

Lighthouse-CI uses mobile preset (Moto G4, 4G slow). k6 uses realistic concurrency and think-time. Measuring on a beefy CI runner without throttling produces numbers that bear no relation to user reality.

Budget changes are themselves PRs

The budget file (perf-budget.json) is in version control. Loosening it requires a PR with a written justification ("new feature X adds 20 KB; we accept the trade-off because Y"). Tightening it requires evidence of headroom.

Production RUM closes the loop

A real-user-monitoring tool (Calibre, SpeedCurve, Sentry Performance, Datadog RUM) tracks the same metrics in production. The CI synthetic catches regressions; RUM catches "actually a problem for users" — both are required.

Common pitfalls

PitfallWhy it happensWhat to do instead
Budget set once at project kickoff, never enforced afterThe team wrote perf-budget.json in week 1 with aspirational numbers, no CI check was wired up, and the file is now seven months stale. Everyone forgets it exists.CI gate from day one. If the budget exists but the gate doesn't, the budget is decorative. Tie the gate to a required status check on PRs to main; no merge without it.
Budget enforced only on dev hardware in CIThe Lighthouse run on a 16-core CI runner with no throttling reports LCP of 0.4s for a page that takes 5s on a mid-range Android phone. The team ships happily and the customer complaints arrive.Enable the mobile preset on Lighthouse-CI (Moto G4, simulated 4G slow). For API load tests, set concurrency that matches production p95. Validate the gate's numbers against production RUM weekly — if they diverge, the gate is lying.
Budget on a single synthetic test, not the real-user metricThe CI test hits one URL on one preset and passes. Production p75 is 4× the synthetic. The team optimised for the runner instead of for the user.The synthetic CI run is the regression detector. The budget target itself is calibrated against and validated by the production RUM 75th-percentile, refreshed monthly. If the synthetic passes but RUM regresses, the synthetic configuration is broken — fix it.
