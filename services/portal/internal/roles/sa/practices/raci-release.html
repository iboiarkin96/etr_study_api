RACI · release · review SLA

A one-page RACI for the key processes, a versioned release flow, and a written review SLA.

TL;DR

Processes scale with the team only if they're predictable. A RACI matrix says who is Responsible, Accountable, Consulted, and Informed on each process. A versioned release flow with a changelog and a rollback playbook makes shipping repeatable. A code-review SLA stops PRs from sitting for weeks.

What it is

This practice bundles three habits that show up wherever a team is more than five people:

  • RACI. A small matrix for each process — release, security review, on-call rotation, ADR review. Every row is a step; every column is a role; every cell is one of R/A/C/I.
  • Release flow. Semantic versioning, a generated changelog, a canary rollout, and a written rollback playbook. The same flow runs every release.
  • Review SLA. 'First reply within two working days.' Written down; a bot pings reviewers when the SLA is at risk; escalation path published.

Why we need it

Without RACI, decisions stall while everyone waits for someone else. Without a release process, every release is improvisation; mistakes repeat because there is no shared script. Without a review SLA, PRs sit for weeks and contributors lose momentum.

None of this is exciting; all of it pays back for the entire life of the project. Once the rules are written, the team spends time on the work, not on coordinating the work.

How others do it

Short tour of how teams known for the practice apply it. Each card is one company, what they do, and what we take from it.

GitLab

Public RACI matrices for every cross-functional process; a one-page RACI per process is part of the handbook standard.

What we take: One A per row, no exceptions. The discipline is in the rule, not in the matrix.

Source: GitLab handbook — RACI

Etsy

Pioneered continuous deployment in 2010; ~50 deploys per day; a written rollback playbook for every deployment path.

What we take: Speed and safety are not in tension. A fast release process with a tested rollback is safer than a slow one without.

Source: Etsy Code as Craft — Continuous Deployment

Stripe

Public 'How Stripe protects developers' explains the release process for the Stripe API: API versioning by date, canary rollouts, automated rollback on SLO breach.

What we take: Versioning + canary + auto-rollback is the standard production release shape. Each piece compounds with the others.

Source: Stripe — Designing robust APIs

Atlassian

Maintains the public 'Incident Management Handbook' and a Release Management guide; explicit RACI for release roles.

What we take: Atlassian's templates are good defaults — fewer roles than you'd guess. Most teams overcomplicate RACI.

Source: Atlassian — Release management

The repeated lessons: keep RACI to one A per row; ship a generated changelog; canary every release; rehearse the rollback quarterly. The complete process is unglamorous and pays back for years.

How it works in this project

The RACI sheets will live under internal/governance/raci/ — one HTML page per process. The release flow is documented at internal/handbook/release.html. The review SLA is in CONTRIBUTING.md with a backing GitHub Action.

Two enforcement loops: CI checks tag format on every release and fails non-conformant tags; the review-SLA bot opens an issue when a PR has been waiting more than two working days without a comment.

Deep dive — the theory you need

RACI rules — the five that prevent every common mistake

  1. Exactly one A per row. Two As means no one is accountable; zero As means the row should not exist.
  2. R and A can be the same person. One person can do the work and be accountable for it; no need to invent a second role.
  3. C and I should be sparing. A row with 8 Cs is process theatre. Three Cs maximum per row.
  4. The matrix is per process, not per project. One matrix for 'releases', one for 'security review'. Mixing processes makes the matrix unreadable.
  5. Review on a calendar. Roles change; the matrix must change with them. Quarterly review is the minimum.

The release flow — five steps that compound

  1. SemVer — MAJOR.MINOR.PATCH. CI enforces the format.
  2. Changelog — Keep a Changelog format; generated from PR labels.
  3. Canary — 1–5% of traffic to the new version; observe for N minutes.
  4. Auto-rollback on SLO breach — if the canary trips the burn-rate alert, roll back automatically.
  5. Rollback playbook — manual rollback steps, rehearsed quarterly, available even when CI is down.

Each step is small. Together they turn 'every release is improvisation' into 'every release looks the same'.

Review SLA — three numbers

  • Time to first comment — target 2 working days; bot reminds at 1 day; escalates at 2.
  • Time to merge — target 5 working days for non-blocked PRs.
  • Escalation path — published in CONTRIBUTING; second ping goes to the team lead; missed SLA shows on the team dashboard.

Trade-offs — what this practice costs

  • RACI matrix maintenance. Reorgs change the column headers. Quarterly review of every RACI keeps it current; the alternative is a matrix that names people who left a year ago.
  • DACI vs RACI. RACI distributes; DACI (Driver / Approver / Contributor / Informed) names a single decision-maker explicitly. Pick one per project — mixing them creates double-A rows ('who has the A — the Accountable or the Approver?').
  • Review SLA is hard to enforce. The bot pings reviewers but cannot make them act. Escalation must be a real human path (team lead → manager) and missed SLAs must show on the team dashboard, or the SLA is decoration.

Lifecycle

proposed → piloted → enforced → reviewed

Each artefact starts as a draft. The release flow is piloted on the next three releases before becoming the only way to ship; the RACI is piloted in two processes before being applied broadly; the review SLA starts as a warning before becoming a build gate.
