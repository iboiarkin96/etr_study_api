Service catalog — machine-readable Backstage YAML

Each service ships a catalog-info.yaml; HTML catalog pages and tooling are generated from it.

TL;DR

A catalog page that only a human can read is half a catalog. With catalog-info.yaml per service, the team gets ownership routing, dependency graphs, alert ownership, and dashboard discovery for free. The HTML page is generated from the same YAML.

What it is

The service catalog is the team's inventory of services with the metadata each one needs: name, owner, type, lifecycle, dependencies, APIs provided, APIs consumed, dashboards, runbooks. The Backstage catalog-info.yaml schema is a published, well-tested format that captures all of this.

Each service ships its YAML in its own repo folder. CI aggregates them, validates the schema, and generates two things: the public HTML catalog under internal/catalog/ and a machine-readable index for tools (on-call routing, dashboard discovery, ownership lookup).

Why we need it

A pure HTML catalog needs a human reader to be useful. Once we want tooling — automatic on-call routing for an alert, a dependency graph that updates itself, a 'who owns this service' lookup — the HTML is the wrong shape. We end up with bespoke scrapers per tool.

A machine-readable catalog avoids that. The YAML is the source; the HTML is generated; the tools read the YAML directly. There is one truth, and it is automatable.

How others do it

Short tour of how teams known for the practice apply it. Each card is one company, what they do, and what we take from it.

Spotify

Built Backstage internally, open-sourced it in 2020. Now a CNCF incubating project with hundreds of corporate adopters.

What we take: Backstage is the de-facto standard for internal developer portals. Adopt the schema even if you don't run the UI.

Source: Backstage software catalog

Netflix

Eureka (the service registry) was open-sourced from Netflix; Spinnaker uses a similar idea. Both treat service metadata as machine-readable infrastructure.

What we take: A service catalog is infrastructure, not documentation. It should be available via API, not just via a web page.

Source: Netflix OSS — Eureka

Cortex (commercial)

Built a 'production readiness' product around the catalog idea: scorecards against catalog entries (has a runbook, has an owner, has SLOs).

What we take: Once the catalog exists, scorecards on top come almost for free — and they drive measurable engineering hygiene.

Source: Cortex Internal Developer Portal

Adobe

Public case study on adopting Backstage across 300+ services; biggest gain was 'no more hunting for who owns this service'.

What we take: At ~50 services or more, the catalog pays for itself in saved hunting time alone.

Source: Adobe Tech blog — Backstage

The same recipe across adopters: YAML in each service repo, aggregator that builds an index, UI on top of the index, and scorecards to drive hygiene. The YAML schema is Backstage; the UI is replaceable.

Anatomy

The parts a complete practice has, with one sentence on each. None of these are optional.

apiVersion / kind

Backstage envelope; constant per entry.

metadata

Name, description, tags, owner annotation.

spec.type

service / library / website / documentation.

spec.lifecycle

experimental / production / deprecated.

spec.dependsOn

Other catalog entities; renders as the dependency graph.

providesApis / consumesApis

OpenAPI/AsyncAPI references; ties catalog to API contracts.

Custom annotations

Dashboard URL, on-call rotation, repo URL, runbook URL.

How it works in this project

The schema is the Backstage software-catalog descriptor. The file lives at services/<name>/catalog-info.yaml. CI runs scripts/build_catalog.py which validates each YAML, resolves references, and writes:

  • internal/catalog/<name>/index.html + sibling pages (architecture, dependencies, on-call, runbooks).
  • internal/catalog/_index.json for tooling.

The HTML pages get the same skeleton no matter the service. The dependency graph on each page is rendered from dependsOn.

Deep dive — the theory you need

Backstage entity kinds — what each describes

Backstage's catalog distinguishes several entity kinds. The five that matter for most projects:

KindWhat it isExample
ComponentA piece of software you build and runThe sessions service; the iOS app
ResourceInfrastructure a Component depends onA PostgreSQL cluster; an S3 bucket
APIAn interface a Component provides or consumesAn OpenAPI doc; an AsyncAPI doc; a gRPC service
SystemA group of Components delivering a capabilityStudy App platform; Identity platform
DomainA business area that contains SystemsCustomer engagement; Operations

The relationship graph

The kinds compose: Components belongTo Systems; Systems belongTo Domains; Components provideApi APIs and consumeApi other APIs; Components dependsOn Resources and other Components.

The graph that comes out of this is the dependency map for free. The same YAML feeds the catalog UI, the on-call routing, and the architecture documentation.

Trade-offs — what this practice costs

  • YAML maintenance. Each service has a YAML file that drifts the moment a dependency changes. CODEOWNERS pinning the YAML to the service owner is the only durable fix.
  • Relationships can be wrong. Hand-listed dependsOn entries miss the unintentional dependency that production traces will reveal. Reconcile with runtime traces quarterly; the gap is the alert.
  • Large-org catalog sprawl. Once Backstage is installed, every team wants their own entity kind. Resist; stick to the canonical kinds (Component, Resource, API, System, Domain) for at least the first year.

Lifecycle

proposed → experimental → production → deprecated → retired

A new service starts as experimental in its YAML. The lifecycle value is moved to production with a Design Doc and a working SLO. Deprecated means we will retire it; the YAML stays so consumers know. Retired archives the entry.

Common pitfalls

PitfallWhy it happensWhat to do instead
YAML drifts from realityAuthor changes the deploy without updating the YAML.CODEOWNERS routes the YAML to the same person as the deploy; PR fails if one is touched without the other.
Two sources of truthCatalog YAML + a hand-maintained ownership spreadsheet.Spreadsheet stops being authoritative the day the YAML lands; one source only.
Lifecycle stays at 'production' foreverService is unused but no one moves it to deprecated.Quarterly review: any service with zero traffic for 90 days is reviewed for retirement.
Dependency list is incompleteAuthor lists the obvious deps; forgets the rare ones.Reconcile catalog-info.yaml's dependsOn with Jaeger traces from staging once per quarter; diffs are posted to #catalog-drift in Slack.

Worked example

catalog-info.yaml

sessions service entry

apiVersion: backstage.io/v1alpha1
kind: Component
metadata:
  name: sessions
  description: Owns the study-session lifecycle.
  tags: [backend, fastapi]
  annotations:
    github.com/project-slug: study-app/services-sessions
    grafana/dashboard-url: https://grafana.example/d/sessions
    pagerduty/integration-key: sessions-primary
    runbook-url: ../sre/runbooks/0007-sessions-5xx.html
spec:
  type: service
  lifecycle: production
  owner: backend-leads
  system: study-app
  providesApis:
    - sessions-v1
  consumesApis:
    - decks-v1
    - identity-v1
  dependsOn:
    - component:decks
    - component:identity
    - resource:sessions-pg

From this single YAML, CI writes the service's catalog page, its row in the dependency graph, its on-call route, and the dashboard link on every related page.
