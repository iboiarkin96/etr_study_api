SECURITY · threat modeling

SECURITY.md, security.txt, and a one-page STRIDE / LINDDUN threat model for critical services.

TL;DR

A researcher who finds a hole in your service should know exactly where to write. A team that thinks about threats as a process catches issues early. SECURITY.md plus security.txt give the disclosure path; one-page threat models per critical service give the team a structured way to find weaknesses before users do.

What it is

This practice has three parts that work together:

  • SECURITY.md at the repo root describes how to report a vulnerability (preferred channel, PGP key, expected acknowledgement time, disclosure timeline).
  • security.txt (RFC 9116) at /.well-known/security.txt is the machine-discoverable form of the same information.
  • Threat models are one-page documents per critical service that walk through the surfaces (entry points, data stores, trust boundaries) and the threats against each. STRIDE for security; LINDDUN for privacy.

Why we need it

Researchers who can't find a disclosure path either post on social media or stay quiet. Both are bad outcomes — the first hurts users; the second leaves the hole open. A clear SECURITY file makes responsible disclosure the path of least resistance.

Threat models change how the team thinks. A one-hour STRIDE pass on the auth flow surfaces issues that none of the four engineers around the table had thought of individually. The threat model also becomes the input to security review — the reviewer doesn't have to start from zero.

How others do it

Short tour of how teams known for the practice apply it. Each card is one company, what they do, and what we take from it.

Microsoft

Originated STRIDE in the late 1990s; ships the Microsoft Threat Modeling Tool and trains every product team in its use.

What we take: Threat modeling becomes routine when it's a tool, not a workshop. Engineers model threats at their desk, not in a quarterly offsite.

Source: Microsoft Threat Modeling Tool

OWASP

Maintains the Threat Modeling Cheat Sheet, the SAMM model for security maturity, and the open-source 'Threat Dragon' tool.

What we take: Start with OWASP. Everything you'd write from scratch is already there, peer-reviewed, with examples.

Source: OWASP — Threat Modeling

GitHub

Public Security policy + a dedicated security@github.com mailbox + structured advisories via Security Advisories feature.

What we take: A researcher who can't find your disclosure path will post on Twitter. Make the path so obvious that posting elsewhere takes more effort.

Source: GitHub Security

Adam Shostack

Wrote 'Threat Modeling: Designing for Security' (the canonical book) and runs threat-modeling workshops for major tech companies.

What we take: The four questions ('What are we building? What can go wrong? What do we do about it? Did we do a good job?') are enough to start.

Source: Shostack — Threat Modeling (Wiley)

Common thread: structured frameworks (STRIDE, LINDDUN), public disclosure path with an SLA, threat modeling per service rather than per project. Tools matter less than the discipline.

The parts a complete practice has, with one sentence on each. None of these are optional.

security.txt

Per-service threat model

One page per service. STRIDE table, LINDDUN if PII is in scope.

Trust boundaries

Drawn on the C4 Container diagram; threats are listed per boundary.

Disclosure SLA

Acknowledge within 48 h; fix non-critical within 30 days; embargo period for coordinated disclosure.

CVE process

Who files, what cadence, where the public note lives.

How it works in this project

SECURITY.md and security.txt are the cheapest wins; both are one file each. Threat models start with the auth flow (every project has one and it's always sensitive), then extend to the data services.

STRIDE = Spoofing, Tampering, Repudiation, Information disclosure, Denial of service, Elevation of privilege. The model is a table: per entry point, list which STRIDE categories apply and what mitigates them. LINDDUN adds the privacy lens: Linkability, Identifiability, Non-repudiation, Detectability, Disclosure, Unawareness, Non-compliance.

Deep dive — the theory you need

STRIDE — six categories, explained

LetterThreatExampleCommon mitigation
SSpoofing identityStolen credentials, replayed tokensMFA, rate-limit, signed tokens
TTampering with dataModified request in transitTLS, signed payloads, integrity checks
RRepudiationUser denies doing an actionAudit log of every action, with user + time
IInformation disclosureError reveals account existenceConstant-time response, generic errors, least-privilege queries
DDenial of serviceBrute force, request floodRate limit per IP + per account, exponential backoff
EElevation of privilegeRole escalation via login flowRoles loaded from JWT, never from request; principle of least privilege

LINDDUN — the privacy complement

LINDDUN is STRIDE for privacy. Use it when the system handles personal data:

  • L — Linkability of two records about the same person
  • I — Identifiability of a person from anonymised data
  • N — Non-repudiation (privacy: user wants to deny an action)
  • D — Detectability of activity
  • D — Disclosure of information
  • U — Unawareness (user doesn't know what's being collected)
  • N — Non-compliance with policy/regulation

Run STRIDE on every service. Run LINDDUN on services that touch personal data. Both fit on one page per service.

Trade-offs — what this practice costs

  • STRIDE tables are repetitive. The sixth threat model uses 80% of the same rows. Build a template with common mitigations pre-filled; review them out where they don't apply, in where they do.
  • DREAD vs CVSS. STRIDE finds threats; DREAD (Damage, Reproducibility, Exploitability, Affected users, Discoverability) and CVSS score severity. Pick one scoring scheme per project — mixing them confuses the audit.
  • Security review becomes a bottleneck. If every PR needs a threat-model update, the security engineer is the slowest path. Cap threat-model updates to: new entry point, new data type, new external dependency. Routine changes don't trigger one.

Lifecycle

disclosure path published → first threat model drafted → reviewed → accepted → quarterly refresh → feeds security review

The disclosure files go live in the first sprint. Threat models start as drafts in a PR, get one focused review, and are accepted. Quarterly refreshes match the risk-register review; major changes (new external service, new auth method) trigger an ad-hoc update.

Common pitfalls

PitfallWhy it happensWhat to do instead
Disclosure path is a generic emailAll security mail goes to info@.Dedicated security@ with PGP key; auto-acknowledgement within minutes.
Threat model is 40 pagesLong document, never read.One page per service. If you need more, you have multiple services in one threat model.
Mitigations are aspirational'We will validate inputs.'Each mitigation row has an evidence: field pointing at a file or test path; CI greps the targets and fails the build when a link does not resolve.
Old findings get lostFirst threat model surfaces 8 issues; team fixes 5; the other 3 disappear.Open issues for everything; close them deliberately or accept them in writing.

Auth flow — login endpoint

Entry point: POST /api/v1/auth/login. Trust boundary: external → internal.

Every mitigation links to a file or a test. A reviewer can verify in 10 minutes; an auditor can sample five rows and see that the team isn't bluffing.

Outcome. Each row's mitigation links to a concrete file or test; a security reviewer samples 5 rows during quarterly review and verifies the links resolve; mitigations without evidence are flagged in the next PR cycle.
