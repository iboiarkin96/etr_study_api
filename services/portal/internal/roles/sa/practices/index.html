Systems analysis practices

Overview

This is the reference handbook for the 25 systems-analysis practices. Each page is written to be the only document you need to read to understand the practice — definition, why it matters, how Google / Stripe / GitLab / Spotify and friends apply it, the parts it has, the theory it stands on, the lifecycle, the pitfalls, a worked example, and the quality bar.

The radar with all 25 axes lives on practices.html. This index lets you jump to any page directly.

Group A · Knowledge management (5)

01Design Docs

A short written proposal that surfaces trade-offs before any code is written.

2.0 → 8.0 / 10

02ADR · RFC

Every significant decision is a numbered file with Context, Decision, and Consequences.

7.4 → 8.5 / 10

03SSOT

Every fact in the project has exactly one authoritative location. Everything else is derived from it or links to it.

5.0 → 8.0 / 10

04Glossary

One curated YAML file of terms with canonical definitions; the HTML page and inline tooltips are generated from it.

2.5 → 7.0 / 10

05Tech Radar

Quarterly chart of the libraries, languages, and tools we adopt, trial, assess, or hold.

1.0 → 7.0 / 10

Group B · Architecture & modeling (5)

06C4 + PlantUML

Architecture drawn at three levels — Context, Container, Component — with PlantUML as the only diagram source.

5.6 → 8.0 / 10

07ER model

The database is documented one table at a time; the ER diagram is regenerated from the metadata on every migration.

5.2 → 8.0 / 10

08OpenAPI contract-first

The OpenAPI 3.1 document is written before the implementation; SDK, mocks, and tests are generated from it.

6.4 → 8.5 / 10

09AsyncAPI

Async messages (Kafka, RabbitMQ, EventBridge, SNS) get the same contract discipline as HTTP — written before code, validated by a schema registry.

1.0 → 7.0 / 10

10DDD bounded contexts

A map of bounded contexts: each service belongs to one context; relationships between contexts are typed.

2.0 → 7.0 / 10

Group C · Information delivery (5)

11Diátaxis

Every page belongs to one mode — tutorial, how-to, reference, or explanation — and stays in that mode.

6.4 → 8.0 / 10

12Docs as code

Documentation lives in the repo, is reviewed in PRs, is versioned with the code, and is validated by CI.

6.5 → 8.5 / 10

13Template registry

One folder of Jinja templates for the major page types: ADR, RFC, Design Doc, Postmortem, Runbook, API spec, Audit, Practice.

5.4 → 8.0 / 10

14Style guide as code

One published style canon (Google or Microsoft), enforced by Vale rules that block PRs with style errors.

1.5 → 7.5 / 10

15Onboarding · accessibility · i18n

README, CONTRIBUTING, devcontainer, WCAG 2.2 AA audit, and the path to a second UI language.

3.5 → 7.0 / 10

Group D · Reliability & operations (5)

16SLO · SLI · error budgets

A measurable target for each user-facing service, with a budget that slows releases when reliability dips.

0.5 → 7.0 / 10

17Runbooks

Every alert points at a runbook with symptom, diagnosis, actions, escalation, and dashboard links.

1.5 → 7.5 / 10

18Blameless postmortems

Every P1/P2 incident gets a postmortem with timeline, impact, root cause, action items, and a 'what we learned' section.

1.0 → 7.5 / 10

19Risk register

A list of risks (technical, business, security, compliance) with severity, mitigation, owner, status, and review cadence.

0.5 → 7.0 / 10

20Capacity planning · FinOps

Back-of-envelope plans per service: QPS, storage growth, cost projection — refreshed every quarter.

0.5 → 6.5 / 10

Group E · Process & governance (5)

21SECURITY · threat modeling

SECURITY.md, security.txt, and a one-page STRIDE / LINDDUN threat model for critical services.

1.0 → 7.5 / 10

22Service catalog

Each service ships a catalog-info.yaml; HTML catalog pages and tooling are generated from it.

3.0 → 7.5 / 10

23AI-augmented docs

Docs that read well for both humans and AI tools — llms.txt, MD source next to HTML, and an Ask-AI widget on high-traffic pages.

1.8 → 7.5 / 10

24Stakeholder map · personas · JTBD

A list of stakeholders, three or four user personas, JTBD statements, and a Power-Interest grid — refreshed every quarter.

1.0 → 7.0 / 10

25RACI · release · review SLA

A one-page RACI for the key processes, a versioned release flow, and a written review SLA.

2.0 → 7.5 / 10
