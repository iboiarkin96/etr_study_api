Hexagonal Architecture — isolate the domain core

Separate the pure domain model from all external dependencies so the core is independently testable and technology-agnostic.

TL;DR

Hexagonal Architecture (Ports and Adapters), coined by Alistair Cockburn in 2005, separates the domain core from all external dependencies — databases, HTTP clients, message brokers, frameworks. The domain defines interfaces (ports); technology-specific implementations (adapters) plug into those ports. The result is a domain that can be tested with no infrastructure, migrated to a new database in hours, and understood without reading framework code.

What it is

Hexagonal Architecture (also called Ports and Adapters) is a structural pattern introduced by Alistair Cockburn in 2005. Its central rule: the application core (domain model + use cases) has no dependencies on external infrastructure. All dependencies point inward, toward the domain.

Two sides of the hexagon:

  • Driving side (primary) — the actors that initiate use cases: HTTP controllers, CLI commands, message consumers. They call the domain via use-case interfaces.
  • Driven side (secondary) — the infrastructure the domain needs: databases, external APIs, message brokers. They implement the domain's repository or notification ports.

The domain defines the ports (interfaces). The adapters implement the ports. The domain never imports an adapter. This inversion of control means the domain can be tested with in-memory fake adapters, and migrating from one database to another requires only writing a new adapter — the domain code does not change.

Why we need it

When domain logic is entangled with Spring annotations, SQLAlchemy models, or Kafka consumers, every test requires a running database and a running broker. The test suite takes 5 minutes instead of 5 seconds. Database migrations require updating domain code. Framework upgrades touch business logic. The accidental complexity of the infrastructure invades the domain and makes it fragile.

Hexagonal Architecture makes the domain a clean, independently deployable unit. A domain test suite with fake adapters runs in under 500 ms with no I/O. The team can swap the database (Postgres to DynamoDB) or the framework (Flask to FastAPI) without rewriting a single use case. At startup speed, this kind of reversibility is worth the initial investment.

How others do it

Alistair Cockburn (originator)

Published the original pattern in 2005 at alistair.cockburn.us. The motivating example: a system that must be driven by both a GUI and a test suite, and that must work with both a real database and an in-memory fake. The hexagonal shape (6 sides) is arbitrary — the pattern has nothing to do with six; it was chosen because a hexagon communicates "many equal ports" better than a rectangle with two sides.

What we take: The most valuable application: making the domain unit-testable without infrastructure. Write domain tests first; the architecture follows from the testing discipline.

Source: alistair.cockburn.us — Hexagonal Architecture

Stripe

Stripe's payment processing core is designed with clear separation between domain logic and infrastructure adapters. The charge creation logic has no direct dependencies on the database layer — it operates on domain objects and emits events. Infrastructure adapters handle persistence and external API calls. This enables Stripe to test the entire charge lifecycle in under 100 ms per test with no external calls. Their public engineering blog describes the principle as "pure core, effectful shell."

What we take: "Pure core, effectful shell" — all side effects (I/O, external calls) at the boundary; pure domain logic at the centre. This makes the core trivially testable.

Source: Stripe Engineering Principles

Netflix (DGS Framework)

Netflix's Domain Graph Service (DGS) framework enforces a ports-and-adapters structure for GraphQL services. The domain layer defines data fetcher interfaces (ports); infrastructure adapters implement them using databases, REST services, or caches. The framework's module structure prevents adapters from importing domain classes — enforced by Maven module boundaries. DGS is open-source at github.com/Netflix/dgs-framework.

What we take: Enforce the architecture at the module/package level, not just by convention. Module boundaries that prevent wrong-direction imports make the pattern maintainable as the team grows.

Source: Netflix DGS Framework

Anatomy

Domain Model

Pure domain classes: aggregates, entities, value objects, domain events. Zero imports from frameworks, ORMs, or HTTP libraries. Measured: the domain module must have zero transitive dependencies on spring-web, sqlalchemy, or kafka-python. Run the module dependency graph in CI.

Use Case (Application Service)

Orchestrates domain objects to fulfil one business scenario (PlaceOrder, CancelBooking). Accepts input ports; returns output. No HTTP status codes, no DB queries. Each use case is a class/function with one public method. Max complexity: if the use case exceeds 50 lines, it is probably doing too many things.

Port (Interface)

An interface defined in the domain that describes what the domain needs from the outside world. Repository ports (find, save), notification ports (notify), and event ports (publish) are the most common. Named from the domain perspective: OrderRepository, not PostgresOrderDAO.

Adapter (Implementation)

The technology-specific implementation of a port. Lives in a separate module/package outside the domain. Examples: PostgresOrderRepository implementing OrderRepository; KafkaEventPublisher implementing DomainEventPublisher. Adapters import the domain; the domain never imports adapters.

Fake Adapter (Test Double)

An in-memory implementation of a port used in domain unit tests. InMemoryOrderRepository stores orders in a map. Enables the domain test suite to run in under 500 ms with no I/O. Must implement the full port interface; must not skip validation.

Common pitfalls

Pitfall Why it happens What to do instead
Domain model importing ORM models or framework annotations Developers annotate domain classes with @Entity or @Column for convenience. The domain and the persistence layer become inseparable over time. Define separate persistence models (mapped separately by the adapter). The domain Order class knows nothing of @Table. The adapter maps between domain objects and persistence models.
Use cases that accumulate too many dependencies (god service) A single use case class is given every port it might ever need. It becomes a 500-line orchestrator that is untestable and hard to reason about. One use case, one public method, max 3–4 port dependencies. When a use case needs 5+ dependencies, split it into smaller use cases with a coordinator on top.
Leaky abstractions in port interfaces (technology bleed) A repository port's method signature returns QueryResult or a Pandas DataFrame — a type from the adapter layer. The domain now depends on the adapter's technology vocabulary. All port method signatures use only domain types. List[Order] is valid; QueryResult is not. If you need to name a return type, define it in the domain package.
