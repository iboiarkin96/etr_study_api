Event-Driven Architecture — decouple through events

Replace synchronous point-to-point calls with immutable domain events to achieve temporal decoupling and resilient integration.

TL;DR

Event-Driven Architecture (EDA) is a design style in which services communicate by publishing and consuming immutable domain events rather than calling each other directly. CQRS (Command Query Responsibility Segregation) separates write and read models; Event Sourcing stores the full state history as an ordered event log. Together they enable temporal decoupling, free audit trails, and independent service scaling. Netflix, Uber, and LinkedIn all run core systems on event-driven backbones.

What it is

Event-Driven Architecture is a system integration style where components communicate by publishing and consuming events. An event is an immutable record of something that happened in the domain: OrderPlaced, PaymentFailed, UserVerified. The producer does not know who will consume the event.

Three related patterns that often appear together:

  • Pub-Sub — a broker (Kafka, SQS, Google Pub/Sub) delivers events from producers to one or more consumer groups. Each consumer group processes every event independently.
  • CQRS (Command Query Responsibility Segregation) — separates the write model (commands that change state) from the read model (queries on projections). Martin Fowler and Greg Young popularised the pattern around 2009. The read model is eventually consistent with the write model.
  • Event Sourcing — stores every state change as an immutable event in an append-only log. The current state is the result of replaying the event log. First described by Martin Fowler in 2005. Enables time travel (rewind to any point), audit trails, and replayable migrations.

Why we need it

Synchronous point-to-point calls create two failure modes at startup speed. First, availability coupling: when Service A calls Service B synchronously, A's uptime is bounded by B's uptime. A slow B cascades into a full A outage. Second, team coupling: adding a new consumer of data from Service A requires a contract negotiation with A's team and an API change.

Events eliminate both. A producer publishes to a broker and moves on. Consumers subscribe independently. The producer team never needs to know about new consumers. Failure in one consumer does not propagate upstream. The event log also provides a free audit trail — a regulatory requirement that is expensive to add after the fact.

How others do it

Netflix

Netflix processes over 1 trillion events per day through its Keystone pipeline (Apache Kafka-based). Every user interaction — play, pause, search — is an event. Consumers include the recommendation engine, A/B testing, billing, and fraud detection. The entire analytics and personalisation stack is event-driven; no service polls another for state.

What we take: Events as the primary integration primitive from day one, not as a migration from RPC. Designing for event replay enables the recommendation engine to be retrained on any historical window.

Source: Netflix Tech Blog — Keystone pipeline

Uber

Uber uses Apache Kafka as the central nervous system for its real-time systems. Every trip state change (requested, matched, started, completed) is published as a domain event. Over 1 million messages per second at peak. The driver location system, ETA calculation, and surge pricing all consume the same event stream independently. Uber publishes its Kafka deployment patterns via the open-source uReplicator project.

What we take: Make the trip/order lifecycle an event stream from day one. Every business question ("how many active rides at 3 pm?") becomes a stream query rather than an expensive DB aggregation.

Source: Uber Engineering — Apache Kafka use case

Martin Fowler / Greg Young (CQRS + Event Sourcing)

Greg Young formalised CQRS as a pattern in 2009 and popularised Event Sourcing for financial systems where immutable audit trails are a regulatory requirement. His EventStore is the canonical reference implementation. Fowler codified the patterns on martinfowler.com. The canonical guidance: "Don't use Event Sourcing unless you genuinely need the audit log or replay capability — the complexity cost is real."

What we take: Apply Event Sourcing selectively to the Core Domain aggregates with regulatory audit requirements. Do not use it as a default persistence mechanism — the query complexity is significant.

Source: Fowler — Event Sourcing

Anatomy

Domain Event Schema

Every event has: eventId (UUID), eventType (namespaced string), occurredAt (ISO-8601), correlationId, and a typed payload. Defined in AsyncAPI 2.x. Breaking changes follow a versioning policy (new topic per major version). Schema registry validates compatibility before deployment.

Event Broker

The delivery infrastructure (Kafka, SQS/SNS, Google Pub/Sub). Retention: minimum 7 days for replay. Consumer groups: each downstream system has its own consumer group so it can replay independently. Partition count: sized to expected peak throughput with 2× headroom.

Idempotent Consumer

Every consumer must be idempotent — processing the same event twice produces the same result. Implemented via deduplication on eventId with a 24-hour TTL. Required for at-least-once delivery guarantees. Non-idempotent consumers cause duplicate charges, duplicate emails, and duplicate inventory decrements.

Dead Letter Queue (DLQ)

Events that fail processing after 3 retries are routed to a DLQ. The DLQ is monitored and alerted. DLQ messages are investigated and replayed manually or through a retry job. Without a DLQ, failed events are silently lost.

Outbox Pattern

To guarantee that a database write and an event publication happen atomically, the event is written to an "outbox" table in the same transaction, then a separate relay process publishes it to the broker. Prevents the dual-write problem (DB write succeeds, broker publish fails). Required for any event that drives financial or inventory state.

Common pitfalls

Pitfall Why it happens What to do instead
Event payload contains too much data ("fat events") Teams include the entire entity state in the event payload to avoid consumers making follow-up calls. Consumers become coupled to the full data model and every schema change is a breaking change. Publish thin events (IDs and key facts only). Let consumers call the source service for full state if needed, or maintain their own projections. Use the "notification + fetch" pattern.
Non-idempotent consumers causing duplicate side effects Developers assume at-most-once delivery. Network retries and consumer restarts cause duplicate delivery. Duplicate emails, duplicate charges, and duplicate inventory decrements follow. Design every consumer for idempotency from day one. Use a deduplication store keyed on eventId. Test with duplicate event injection in CI.
Event schema changes break consumers silently A producer adds or renames a field. Consumers in other teams do not know until their processing silently fails or produces incorrect results. There is no contract enforcement. Use a schema registry (Confluent Schema Registry or AWS Glue). Enforce backward-compatible changes only (add optional fields, never remove or rename required fields). Use separate topics for breaking major versions.
