Proof of Concept & Spike — de-risk before committing

Answer architectural unknowns with a focused, time-boxed experiment before committing engineering capacity to the full implementation.

TL;DR

A Proof of Concept (PoC) answers "can we build it?" — a focused experiment with a clear success criterion, a time box (max 3 days), and an explicit throw-away policy. A Spike (from Extreme Programming, Kent Beck, 2000) answers "how long will it take?" — a time-boxed technical exploration with a written output. Both produce a written report, not production code. The discipline of "define the question before starting and throw away the code after" prevents PoC code from silently becoming production code — the most common failure mode.

What it is

A Proof of Concept (PoC) answers one technical feasibility question. Before building a full feature, the architect identifies the riskiest assumption and builds the minimum code needed to validate or invalidate it. The PoC is explicitly not production code: no tests, no error handling, no cleanup. After the question is answered, the code is deleted.

A Spike (from Kent Beck's Extreme Programming, 2000) answers a different question: "how long will the real implementation take?" It is a time-boxed exploration (max 2 days) of an uncertain area, producing an estimate and a set of findings, not a deliverable. The spike output is a short written document, not code. Code written during a spike is deleted unless it is also production-quality.

Both share the same structure: a question defined before starting, a time box that must not be exceeded, and a written output summarising the findings and a recommendation.

Why we need it

Architecture mistakes committed to production cost 10–100× more to fix than if they were caught in a 2-day PoC. A team that spends 3 weeks implementing a WebSocket integration before discovering that the chosen broker does not support their messaging pattern has wasted 15 engineer-days that a 1-day PoC would have saved.

The throw-away discipline is equally important. PoC code that silently becomes production code is the most common failure mode: the developer finishes the PoC, it works, and the path of least resistance is "this already runs, let's just keep it." Three months later the team is maintaining code with no tests, no error handling, and design choices made purely for speed of exploration. The "delete the code" rule is the most valuable part of the practice.

How others do it

Kent Beck / Extreme Programming (Spike originator)

Kent Beck introduced the Spike in Extreme Programming (2000) as a direct response to the problem of uncertain estimates. When the team cannot estimate a story because the technical approach is unknown, a spike is scheduled as a separate story with a fixed time box. The spike output is a revised estimate, not a feature. Beck's rule: "always destroy spike code" — the spike is a learning activity, not implementation. The spike story is tracked on the board; when the time box expires the spike ends regardless of completeness.

What we take: The spike is an estimation tool, not a "free work" mechanism. Define the question at the start: "What is the estimate for implementing X?" The output is an answer to that specific question.

Source: Agile Alliance — Spikes

Google

Google uses PoCs extensively as part of its Design Doc process. Before writing a design doc, engineers typically run a 1–2 day technical spike to validate the most uncertain assumption. The spike findings inform the options section of the design doc: "we spiked approach X and found Y, which is why we are recommending Z." This prevents design docs from being purely theoretical — they are grounded in experimental evidence. The spike findings are included as an appendix or inline in the design doc.

What we take: Link PoC findings to design docs. The PoC provides the experimental evidence that makes the design doc's option recommendation credible rather than speculative.

Source: Industrial Empathy — Design Docs at Google

Airbnb

Airbnb's infrastructure team uses PoCs to validate migration strategies before committing to full implementation. Their React Native adoption (2016–2018) started with a 2-week PoC in the Airbnb app before a team was allocated to the full migration. The PoC revealed performance characteristics that shaped the migration approach. When the PoC results were later published (after the migration was ultimately reversed), they demonstrated that a good PoC produces honest negative results — the PoC should be able to say "this doesn't work" as readily as "this works."

What we take: A PoC must be designed to produce a negative result if the answer is negative. Success criteria must be defined before the experiment, not after. A PoC that cannot fail is not a PoC — it is a justification exercise.

Source: Airbnb Engineering — Sunsetting React Native

Anatomy

Success Criterion

A specific, binary, measurable condition defined before the PoC starts. "The system can handle 10k concurrent WebSocket connections on a t3.medium" is a valid criterion. "The approach seems feasible" is not. If the success criterion cannot be defined before starting, the question being asked is not precise enough — iterate on the question first.

Time Box

A hard limit: max 1 day for a Spike, max 3 days for a PoC. When the time box expires, the experiment ends regardless of how close it is to a result. A PoC that needs more time is either asking the wrong question or has underestimated the complexity — both are useful signals. Exceeding the time box without explicit sign-off from the architect is not permitted.

One-Page Report

The required output of every PoC and Spike. Sections: (1) question asked, (2) approach taken, (3) findings (what was learned), (4) recommendation (proceed / abort / run another PoC). Max 1 page. Stored in the design docs folder. Referenced by the ADR if the PoC informed an architecture decision.

Throw-Away Policy

PoC code is explicitly tagged as throw-away: a top-level # POC: DO NOT MERGE comment in the main file, a separate git branch, and a ticket to delete the branch after the report is written. No exceptions. Production implementation must start from scratch, informed by the PoC findings. The throw-away policy is enforced in code review.

Common pitfalls

Pitfall Why it happens What to do instead
PoC code shipped to production The PoC works. Shipping the PoC code is faster than rewriting it. Deadlines are tight. Three months later the team is maintaining untested, unstructured code that nobody dares refactor. Enforce the throw-away policy in the PR process. PoC branches are never merged to main. If the PoC answered yes, open a new ticket for the production implementation and link the PoC report.
Success criterion defined after the experiment (confirmation bias) The developer runs the experiment, it produces a result they like, and then writes the success criterion to match. The PoC is used to justify a decision that was already made. Write the success criterion in the ticket before the PoC starts. The criterion is reviewed by the architect before work begins. Changing the criterion after the experiment starts requires explicit justification.
Spike used to avoid estimation rather than enable it The team adds a spike when they are uncomfortable providing an estimate. The spike becomes a way to defer the estimation conversation indefinitely. A spike must have a defined output: "After the spike, we will provide a story-point estimate for the full implementation." If the spike cannot produce an estimate, define why the uncertainty remains — that is also a valuable output.
