Scenario-based Testing — validate end-to-end user journeys

Derives test cases from realistic user goals to verify that components compose correctly from the outside in.

TL;DR

Scenario-based testing derives test cases from end-to-end user journeys — "create a conspectus, assign a due date, and verify its appearance in the list" — rather than from individual functions. A system can pass every unit and integration test and still fail the most common user workflow due to integration gaps. Scenarios validate that components compose correctly. Each user-facing feature has at least one happy-path and one failure-path scenario, documented before implementation and automated in CI at the system test layer.

What it is

Scenario-based testing derives test cases from realistic end-to-end user journeys rather than from individual functions or API endpoints. Each scenario represents a user goal: a concrete thing a user wants to accomplish with the system. The scenario walks through the steps the user takes and asserts the outcome they expect.

The canonical format for scenario-based testing is Given-When-Then, introduced by Dan North as the cornerstone of Behaviour-Driven Development (BDD) in 2006:

  • Given [context/precondition]: The state the system must be in for the scenario to make sense.
  • When [action]: The action the user takes.
  • Then [expected outcome]: The result the user expects.

Example: "Given a registered user with an active session / When they create a new conspectus with a title and a due date / Then the conspectus appears in the list view with the correct title and due date."

Scenario-based tests live at the system test layer of the test pyramid. They are slower than unit and integration tests but test the composition of the entire stack. They are automated using tools like Cucumber, Playwright, or Cypress, and run in CI as part of the system test stage.

Each user-facing feature requires at least: one happy-path scenario (the normal, successful flow) and one failure-path scenario (what happens when something goes wrong). The scenarios are documented before implementation begins, in the Three Amigos session.

Why we need it

Unit tests verify individual functions. Integration tests verify that two components interact correctly. But neither layer verifies that the complete user journey — spanning multiple components, API calls, database writes, and UI renders — works end to end. A system can pass every unit test and every integration test and still fail the most common user workflow due to an integration gap nobody tested.

Scenario tests fill this gap. They are the only test layer that validates the system from the user's perspective rather than the developer's. If the scenario passes, the user journey works. If it fails, something in the composition is broken — and the failure message points to the broken step, not just a failed assertion.

How others do it

Cucumber / BDD community

Cucumber is the most widely used tool for scenario-based testing. It introduced the Given-When-Then format and the Gherkin language for writing scenarios in plain English. Cucumber scenarios serve two purposes: as executable tests and as living documentation. The Cucumber team advocates writing scenarios before implementation (specification by example) so they serve as a shared understanding of the feature before a line of code is written.

What we take: Write scenarios in plain English before implementation. The scenario is a communication artefact first, a test artefact second. If the developer, PM, and QA can't all read it, it's not a good scenario.

Source: Cucumber — BDD overview

Dan North (BDD originator)

North introduced BDD in 2006 as a response to test-driven development's difficulty in communicating intent. His key insight was that test names should describe behaviour, not implementation: "should return the correct total" not "testCalculateTotal". The Given-When-Then format was designed to make scenarios readable by non-technical stakeholders without losing precision. North advocates writing scenarios at the story level (one scenario per acceptance criterion) rather than at the function level.

What we take: One scenario per acceptance criterion. If a story has three acceptance criteria, it has three scenarios — not one scenario with three assertions.

Source: Agile Alliance — BDD

Google

Google's testing culture, as described in Software Engineering at Google, includes scenario tests (called "large tests" or "system tests") at the top of the test pyramid. Google requires that every user-facing feature has at least one large test that covers the primary user journey end to end. These tests are deliberately kept small in number (the "10%" in the 70/20/10 split) because they are slow and expensive to maintain. Google's test engineering teams own large tests; developers are expected to own unit and integration tests.

What we take: Keep the scenario test count low and focused on the highest-value user journeys. Every new scenario test added to the suite must justify its cost in terms of coverage provided.

Source: Software Engineering at Google

Anatomy

User goal statement

One sentence describing what the user wants to accomplish. Not a technical action but a business goal: "complete a purchase," not "submit a POST request."

Given-When-Then steps

The scenario written in Given-When-Then format. Each step is a user-level action or assertion, not an implementation detail.

Happy path + failure path

At minimum two scenarios per feature: the successful path (user achieves goal) and one failure path (user provides invalid input or an error occurs).

CI automation

Scenarios are automated using a BDD tool (Cucumber, Playwright, Cypress) and run in CI at the system test stage. Failures block the release pipeline.

Common pitfalls

PitfallWhy it happensWhat to do instead
Scenarios written after implementation QA writes scenarios from the completed feature rather than from the acceptance criteria, producing tests that describe what was built rather than what was wanted. Write scenarios in the Three Amigos session before implementation. Scenarios are a specification, not a description of the finished product.
Too many scenarios per feature Every edge case gets a scenario, producing a system test layer that is as large as the unit test layer and significantly slower. Restrict scenarios to user-level goals. Edge cases and boundary conditions belong in unit and integration tests, not in scenarios. Keep scenarios at the 10% target of the pyramid.
Implementation-level steps Steps describe API calls and database queries rather than user actions: "When POST /api/v1/orders is called with body X" instead of "When the user places an order." Write steps at the user's level of abstraction. The scenario should be readable by the PM without technical knowledge. Technical details belong in the step definitions, not the scenario text.
