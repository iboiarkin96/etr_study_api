GET Reference API · Conspectus Draft

GET /api/v1/schedule/summary

Aggregates per-slot counts of due and upcoming conspectuses. operationId: getScheduleSummary. Owner: Ivan Boyarkin. Last reviewed 2026-04-29.

Overview

Aggregate counts per slot for the learner's schedule: how many conspectuses sit in slots A / B / C / D, how many are due now, and how many are due in the next 24 hours. Read-only.

1. Business context

Drives "study load" headers in client UIs (the strip "3 due now, 17 in slot A, 5 due tomorrow"). The aggregate is cheap on the index but expensive to recompute client-side from GET /due alone, so the server takes care of it.

2. User stories

  • US-CONSP-SUMMARY-1. As a learner, I want to see at a glance how much study load is queued, so I can plan my day.
  • US-CONSP-SUMMARY-2. As an integrator, I want a single read for the home-screen badge counts, instead of multiple GET /due calls with different filters.

4. Functional requirements

IDShort descriptionDetailed characteristic
FR-CONSP-SUMMARY-001Resolve owner from query.Same composite-key resolution as listDueConspectuses.
FR-CONSP-SUMMARY-002Aggregate counts in one query.SELECT slot, COUNT(*) FROM conspectus_schedules WHERE owner_client_uuid = ? GROUP BY slot + two filtered counts for due_now and due_next_24h.
FR-CONSP-SUMMARY-003Always return all four slot keys.Slots with zero count are returned as 0, not omitted.
FR-CONSP-SUMMARY-004Server-side time anchor.due_now uses now() UTC; due_next_24h uses now() < next_review_at <= now() + P1D.
FR-CONSP-SUMMARY-005Read-only.No DB writes.

API key per shared auth; learner from query. Aggregate scoped to owner_client_uuid.

8. Response contract

200 OK — schedule summary

by_slot object
Per-slot counts. Keys are A, B, C, D; values are non-negative integers.
due_now integer · int64
Items with next_review_at <= now().
due_next_24h integer · int64
Items with now() < next_review_at <= now() + P1D.
total integer · int64
Sum of by_slot values; redundant for client convenience.
computed_at string · date-time
Server timestamp at which the aggregate was computed.

Example 200 response

Content-Type: application/json

        }

13. Edge cases checklist

  • No conspectuses. All zeros.
  • Brief disagreement with GET /due. Reads are snapshot-isolated; UI may briefly show due_now=4 while /due returns 5 if a review committed in between. Acceptable.
  • Slot D with rungs. All D-rungs collapsed into single by_slot.D count in v1.
  • Idempotency-Key sent. Silently ignored.

  • Q1. Add D-rung breakdown (by_slot.D = {0: n0, 1: n1, ...})? v1: no — collapsed counts are enough for badges. Revisit if product asks.

  • Per-policy breakdowns.
  • Calendar-day buckets — client computes those locally.
