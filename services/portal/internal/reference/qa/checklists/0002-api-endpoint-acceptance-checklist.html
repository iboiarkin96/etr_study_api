Reference QA Checklist 0002

API endpoint acceptance

Run this for every new or changed HTTP endpoint, against the operation's spec under internal/reference/api/<resource>/. Mark each row Pass / Fail / Needs follow-up; attach evidence per the API endpoint testing playbook.

Preconditions

  • Spec page exists and passes the spec DoD.
  • API is reachable on a test environment with the build SHA known.
  • You have a valid bearer token for an account with the required role.
  • HTTP client (curl, httpie, Postman) is ready and capturing.

Contract — schema and shape

#CheckExpected
1.1Path, method, version segment match the spec exactly.Identical.
1.2Request body required/optional/types match the spec.Identical.
1.3Response body schema matches for every documented status.Identical.
1.4Field naming follows field conventions.snake_case keys; UTC ISO-8601 timestamps; UUIDs as strings.
1.5OpenAPI baseline test (tests/openapi/) is green.Pass; if the contract changed, baseline is regenerated and committed.

Happy path

#CheckExpected
2.1Smallest valid body returns 2xx success.200 / 201 per spec; body conforms to schema.
2.2All optional fields, when supplied, round-trip unchanged.Echoed in the response and in a follow-up read.
2.3List endpoints — pagination first → next → empty next_cursor.Per pagination.
2.4Read-after-write — follow-up GET returns the just-created resource.Visible immediately.

Error paths

#CheckExpected
3.1Missing required field.422; envelope; field_errors points to the field.
3.2Wrong type / out-of-range value.422; envelope.
3.3Resource not found (UUID for a missing row).404; envelope; catalog code.
3.4Conflict (forbidden state transition or unique constraint).409; envelope; catalog code.
3.5Wrong HTTP method.405 with Allow header.
3.6Wrong / missing Content-Type on body methods.415; envelope.
3.7Forced 5xx (e.g. break a downstream).5xx; envelope; X-Request-ID traces in logs; no stack trace in the response.

Authentication / authorization

#CheckExpected
4.1No token.401; envelope.
4.2Malformed token / wrong scheme.401; envelope; never 500.
4.3Authenticated but lacking permission.403; envelope.
4.4Authenticated with sufficient role.2xx.

Idempotency (write methods only)

#CheckExpected
5.1Same key + same body, twice within window.Both calls return identical body and status; no duplicate row.
5.2Same key + different body.409; envelope.
5.3Same key after window expires.Treated as new request.

Observability

#CheckExpected
6.1Every response carries X-Request-ID.UUID; unique per request.
6.2Structured log shows one record per request.Includes method, path, status, latency_ms, correlation id.
6.3Sensitive headers / fields not logged in clear.Per observability conventions.

Docs & OpenAPI

#CheckExpected
7.1Spec page passes DoD.All required sections non-empty.
7.2Operation visible in Swagger UI with the expected request/response.Render is correct in both light and dark.
7.3Examples in the spec match the live API.Copy-paste runs.
7.4Sidebar entry exists under API endpoints documentation.Active state highlights for this page.

Evidence required for sign-off

  • Request/response capture for one happy path, one validation error, one auth error.
  • One X-Request-ID traced in the structured log.
  • Screenshot of Swagger UI showing the operation expanded.
  • Result line per row above (Pass / Fail / Needs follow-up), pasted into the ticket.

Pass / fail rules

  • Pass: all rows above pass; evidence attached.
  • Fail: any contract, error envelope, auth, idempotency, or observability row fails.
  • Needs follow-up: non-blocking wording issues with a tracked ticket.
