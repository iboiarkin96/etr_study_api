Glossary index

Where the canonical definition of each vocabulary lives. Glossaries are kept close to the domain that owns the terms — project acronyms with Onboarding, QA vocabulary with the QA reference, HTTP conventions in the API shared specs. This page is the single jump-off so readers do not have to remember which sub-tree owns which term.

Project-wide acronyms & domain terms

Acronyms (API, ADR, RFC, SRE, gRPC, AsyncAPI, …), the Study App domain vocabulary (conspectus, cue sheet, ETR, attempt, slot ladder, schedule revision), and the SA-craft terms (Diátaxis, traceability matrix, spine matrix).

  • Canonical: Onboarding · Glossary — 48 entries, A–Z. Link here from page prose instead of expanding inline.

QA vocabulary

Acceptance test, charter, contract test, equivalence partition, exploratory testing, flake, HICCUPPS-F, oracle, risk-based testing, smoke vs sanity, severity vs priority. Aligned with ISTQB.

  • Canonical: Reference · QA · Glossary — 36 entries, A–Z, ISTQB-aligned.
  • Related: Bug severity & priority matrix · Defect lifecycle

HTTP & API conventions

Idempotency, error envelope, error catalog, pagination, versioning, observability conventions, field conventions. Each shared spec is a definition of the terms used inside per-operation specs.

  • Canonical: Shared API standards — 10 pages, one per cross-cutting concern.
  • Related: Error catalog (code-synced) · Error matrix by status

SRE & observability vocabulary

SLI, SLO, SLA, error budget, burn rate (1h / 6h / 3d windows), the four golden signals, page vs ticket, SEV-1 through SEV-4. Naming and rationale live in the Explanation pages; the matrices are in shared specs.

  • Canonical: Explanation · SLOs (SLI/SLO/SLA + burn rates), Observability (three pillars), Incident response (severity matrix).
  • Conventions: Observability conventions — log event / metric / alert naming applied to the API.

Data & persistence vocabulary

Composite natural key, client_uuid (server-issued PK), is_row_invalid (soft-delete flag), schedule_revision (monotonic counter), content_version, idempotency_keys table.

  • Canonical: Data reference — one page per table with schema, constraints, lifecycle.
  • Conventions: Field conventions — naming, types, timezones, PII classification.

Forbidden vocabulary patterns

  • Inline expansion in prose. Writing "Architecture Decision Record (ADR)" the first time on every page is noise. Link to the glossary entry once, use the acronym the rest of the way.
  • Made-up acronyms. If it isn't in the glossary, it isn't an acronym yet. Add the entry first, use it second.
  • Casing drift. API, ADR, SRE, gRPC, AsyncAPI, QA, CI are not negotiable — see feedback_sidebar_acronym_casing.
  • Synonyms for the same domain term. "conspectus" only. Not "note", not "study card", not "summary". Synonyms degrade search; one canonical word per concept.
