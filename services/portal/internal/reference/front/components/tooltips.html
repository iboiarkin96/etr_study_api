Docs frontend: tooltips and inline hints

Docs portal · Surfaces & overlays.

Summary: Premium tooltip surface and content contract. Status: Availability: v1 · Internal Owner: Ivan Boyarkin Last reviewed: 2026-04-30 Spec template: v1

Overview

Hand-authored HTML under services/portal/ uses two complementary patterns for short, optional explanations. This page is the normative contract for which pattern to use so hints stay visually consistent, keyboard-accessible, and not clipped by layout chrome.

  • Default for UI labels, table headers, cockpit copy: data-tooltip only (no dotted underline).
  • Optional glossary term in prose: class="docs-tooltip" with data-tooltip text, or docs-tooltip--etr for the shared ETR definition.
  • Stylesheet source of truth: services/frontend/portal/assets/docs.css (not per-page ad-hoc tooltip CSS unless the page deliberately does not load shared assets).
  • Authoring model: Frontend reference shape from RFC 0003 — anatomy diagram, visual gallery, do/don't, page history.

Pattern A — data-tooltip (compact "backlog-style" hint)

Use when: explaining a control, column header, filter, or short label without turning the term into a glossary link.

Markup

  • Non-focusable host (e.g. <span>): add tabindex="0" so keyboard users can focus the term and see the hint.
  • Native controls (<button>, <input>, <select>): no extra tabindex required.
  • Put the explanation in data-tooltip="…". Keep copy short; wrap is handled in CSS.
html<span tabindex="0" data-tooltip="What this column means in one or two sentences.">Score</span>

Visual contract

  • No dotted underline on the trigger (unlike .docs-tooltip).
  • Light surface bubble, small type (0.7rem), centered above the anchor by default.
  • Typography resets inside the bubble so text does not inherit <strong> uppercase or bold from cockpit labels.

Pattern B — class="docs-tooltip" (glossary / "look up this term")

Use when: the trigger is a deliberate glossary entry or you want the dotted underline to signal "more context on hover/focus".

  • Requires data-tooltip="…" on the element unless you use the docs-tooltip--etr modifier (ETR uses shared copy from CSS variables).
  • Dark tooltip surface — distinct from Pattern A — so readers can tell glossary from field hints.
html<span class="docs-tooltip" tabindex="0" data-tooltip="Your definition here.">Term</span>
html<span class="docs-tooltip docs-tooltip--etr" tabindex="0">ETR</span>

Clipping, tables, and sticky chrome

Global table { overflow: hidden; } (for rounded table frames) clips absolutely positioned bubbles that extend above a cell. For <th> (and the same combined rules for horizontal scroll regions and a few other hotspots), shared CSS opens hints below the trigger so they stay inside the paint region.

  • If a hint still disappears under a sticky strip (narrow internal layout title row), prefer Pattern A in that region or keep the trigger lower on the page — the downward rules also cover .docs-feedback-card labels.
  • Wide tables wrapped in .docs-table-scroll use the same downward behavior for both patterns.

JavaScript-generated UI

Example: the "Page feedback" card in services/frontend/portal/assets/docs-nav.js builds label spans with data-tooltip only (no docs-tooltip class) so it matches Pattern A automatically wherever the card is injected.

The backlog cockpit sets data-tooltip on rendered chips from services/frontend/portal/assets/backlog.js; styles for [data-tooltip] live in docs.css (backlog still loads that file first).

Contributor checklist

  1. Prefer Pattern A for filters, cockpit labels, assessment table headers, RFC KPI columns, and similar.
  2. Use Pattern B for intentional glossary terms; use docs-tooltip--etr only for the standard ETR expansion.
  3. Do not duplicate tooltip CSS on standalone pages — either load docs.css or copy the [data-tooltip] rules explicitly and keep them in sync.
  4. After editing, spot-check hover and Tab + focus on the trigger in light and dark theme.

Per-component states and accessibility

Component States Role / ARIA Keyboard Reduced motion Screen-reader announcement
Pattern A — data-tooltip hidden, on-hover, on-focus, clipped (when ancestor has overflow: hidden) role="tooltip" via aria-describedby on the trigger Tab focus shows the tooltip; Esc dismisses while focused Show instantly (skip 120 ms fade) Tooltip text is read after the trigger label, e.g. "Apply review, tooltip: posts the user's tag"
Pattern B — .docs-tooltip underlined inactive, on-hover, on-focus, dismissed Term acts as <dfn> with paired tooltip Tab to the underlined term reveals the definition; Esc dismisses Definition appears without animation "DFEAT-04, definition: docs frontend feature ledger entry…"

Do's and Don'ts

DoDon't
Use Pattern A (data-tooltip) for compact backlog-style hints — short labels under ~10 words. Use it for paragraph-length explanations — use a .callout from the UI kit instead.
Use Pattern B (.docs-tooltip) only for glossary terms that need an in-place definition. Sprinkle <dfn>-like markup across narrative text — readability drops fast.
Verify the tooltip appears on keyboard Tab focus, not only on mouse hover. Ship a tooltip that disappears the moment focus moves to it — keyboard users get nothing.
Test inside scrollable tables and under sticky chrome — both clip tooltips by default. Position a tooltip absolutely without checking that it escapes overflow: hidden ancestors.
