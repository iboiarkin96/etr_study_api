Docs frontend: spec metadata block

Docs portal · Cross-cutting spec chrome.

Summary: The single approved metadata block used at the top of every internal spec page (operations, screens, components, data tables). Status: Availability: v1 · Internal Owner: Ivan Boyarkin Last reviewed: 2026-05-08 Spec template: v1

Overview

.internal-spec-meta is the normative metadata block that sits at the top of every internal spec page in this repo. It exists so that — regardless of whether you are reading an HTTP-operation spec, a screen spec, a component spec, or a database-table spec — the answer to "who owns this, what status is it in, when was it last reviewed?" appears in the same shape, with the same labels, in the same place. Reusing one block keeps the corpus visually coherent and lets cross-cutting tooling (status pill, page-history widget) attach to a single contract.

  • Stylesheet source of truth: services/frontend/portal/assets/internal-layout.css — never reimplement the layout per page.
  • Status pill: the <span data-spec-status-mount=""></span> slot is filled by services/frontend/portal/assets/docs-spec-status.js; the page must declare data-spec-status on <body> for the script to render correctly.
  • Authored at the top of the page — between <h1> + #docs-top-nav + #docs-page-meta-mount chrome and the first <section class="card">.

Anatomy

The block is a flat list of <span class="internal-spec-meta__item"> children inside one <div class="internal-spec-meta"> wrapper. Each item has a label (uppercase eyebrow) and a value (text, link, code, or status mount).

Field vocabulary by spec type

The label set is fixed per spec type. Use the matching column when authoring a new spec page; do not invent labels.

Spec type (data-spec-page) Required fields, in order
operation — HTTP endpoint per RFC 0003 Summary · Status · Availability · Owner · Resource · Last reviewed · Spec template
hub — resource hub or section index Summary · Status · Availability · Owner · Resource · Last reviewed · Spec template
screen — frontend screen spec Summary · Status · Availability · Owner · Implementation · Last reviewed · Spec template
component — frontend component spec Summary · Status · Availability · Owner · Last reviewed · Spec template
foundation · contract · pattern Summary · Status · Availability · Owner · Implementation · Last reviewed · Spec template
data-table — database-table spec per ADR 0029 Table · Category · Status · Source · Owner · Last reviewed · Spec template

Status uses <span data-spec-status-mount=""></span> always — never plain text — so the pill is rendered consistently from data-spec-status on <body>.

Do / don't

Do

  • Use this block on every spec page (operation / hub / screen / component / data-table). One block per page, at the top.
  • Match the label vocabulary from the field-vocabulary table — do not introduce ad-hoc labels.
  • Set data-spec-page, data-spec-status, data-spec-owner on <body> so docs-spec-status.js can mount the status pill.
  • Wrap every value in a semantic element (<a> for owner / source / implementation, <code> for filenames / paths / template version, <time> for dates, <span> otherwise).

Don't

  • Don't roll your own <dl> definition list at the top of a spec page — drift makes the corpus look amateur and breaks status-pill mounting.
  • Don't put the metadata block inside a <section class="card"> — it lives between the page chrome and the first card.
  • Don't omit aria-label="Spec metadata" on the wrapper — assistive tech needs the landmark.
  • Don't mix metadata fields with prose; if a field needs a paragraph of explanation, put it in the Overview / Summary <section class="card"> below.

References

  • Authoring model: RFC 0003 — closed taxonomy of spec types and their templates.
  • Templates that already use this block: component-spec-template, foundation-spec-template, contract-spec-template, API operation spec-template, database-table template.
  • Stylesheet: services/frontend/portal/assets/internal-layout.css.
  • Behavior: services/frontend/portal/assets/docs-spec-status.js mounts the status pill.
  • Database-table specs that adopt this block (2026-05-08): users, systems, invalidation_reasons, timezones.
