Contract spec — template

One-line description of the cross-cutting surface this contract governs.

How to use this template

Golden template for any normative behaviour contract under services/portal/internal/front/contracts/. A contract describes what the portal must do at the product level — the rules every implementation must follow, regardless of which JS or CSS file delivers them. Implementation details belong in the foundations and components pages the contract references.

Required vs optional. Required sections must exist and be non-empty. Optional sections may be removed when they do not apply — never reorder, only remove.

Contracts use RFC 2119 keywords (MUST, MUST NOT, SHOULD, MAY) for normative requirements. If you are not making a normative statement, do not use these keywords.

Lint checklist (block-merge gate, future)

This block lives only on the template page. Real contract specs do not duplicate it.

  • Body attributes: data-spec-page="contract", data-spec-status, data-spec-owner on <body>.
  • Page-type badge in the H1 reads Contract.
  • Every required section exists as <section data-spec-section="…" data-spec-required="true"> with non-empty content.
  • Scope & breakpoints names the viewport tiers covered (≥ 1025 / 761–1024 / ≤ 760).
  • Normative requirements uses MUST / SHOULD / MAY; each clause is independently verifiable.
  • Acceptance checklist is a numbered or bulleted list of pass/fail conditions.
  • Implementation entry-points names every JS/CSS file the contract relies on.
  • Page history section present with at least one entry.

3. Normative requirements required

Numbered list of MUST / SHOULD / MAY clauses. Each clause is independently verifiable: a tester reads the clause and can pass/fail the implementation against it without further interpretation.

  1. R1. The implementation MUST <observable behaviour>.
  2. R2. The implementation MUST NOT <forbidden behaviour>.
  3. R3. The implementation SHOULD <recommended behaviour>, unless <explicit exception>.
