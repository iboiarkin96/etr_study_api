Screen spec: Backlog cockpit

Overview

This page is the implementation reference for the backlog cockpit screen: services/portal/internal/governance/backlog/index.html. Use it as the canonical source when changing filters, priority logic, sorting, view modes, the page hero, the ticket-style cards, the kanban board, the ETA-bucketed timeline, and the task details modal.

2026-05-08 premium pass: page now opens with a dedicated hero block with status tickers; cards switched to a ticket-style layout with a priority-tinted left bar and a status pill anchored top-right; the board view became a real kanban with status columns; the timeline view groups tasks by ETA bands on a vertical axis; the inline expand toggle was replaced by a centered details modal opened from the clickable card title. All inline-SVG mocks below were redrawn to match.

Screen layout map

  1. Page hero (.backlog-hero): eyebrow / gradient title / lead / 4 status tickers (To do · In progress · Blocked · Done). Tickers are populated by paintBacklogHeroTickers().
  2. Cockpit / control bar (.backlog-cockpit): small contextual header (eyebrow + title + new-task primary CTA), hero search input with leading icon, two pill-shaped segmented controls (Quick filters, View), button-styled Detailed filters disclosure (collapsing the Advanced nest), and the live insights row (filter summary + reset + removable active-filter chips).
  3. Intelligence panel: attention, bottlenecks, and reprioritization hints (desktop-only).
  4. Grouped orientation blocks: block-level grouped lists for quick scan.
  5. Main task stream (#backlog-task-list): ticket-style task cards. The layout switches with data-backlog-view: list = flat stack, board = 5 status columns rendered into .backlog-board-column flex wrappers, timeline = 6 ETA bands rendered into .backlog-timeline-segment rows.
  6. Operational helpers: new-task template and page history.

Data contract and anti-duplication rules

  • data-group (domain): frontend, backend, devops, docs.
  • data-tags (task type): bug, feature, research, tech-debt.
  • data-risk (risk profile): high, medium, low.
  • data-confidence (estimation confidence): high, medium, low.
  • data-priority: P0, P1, P2, P3.
  • data-status: todo, in-progress, done, blocked, rejected.
  • Status visual pill is rendered from data-status; markup can stay as <span class="status-pill"></span> without manual text/class duplication.
  • data-created-at (optional): ISO date used for task age calculation when explicit age is not set.
  • data-age-days (optional): explicit age override in days.
  • data-blocked-days (optional): integer for blocked aging, used by SLA breach risk KPI.
  • data-done-days-ago (optional): integer for throughput window, used by Flow health KPI.
  • data-blocked-at (optional): ISO date fallback for blocked aging if data-blocked-days is missing.
  • data-done-at (optional): ISO date fallback for done recency if data-done-days-ago is missing.
  • data-actual-hours (optional): factual effort for completed tasks, used in flow meta as average actual effort.
  • data-eta-days (optional): explicit ETA range string in days (for example 1-1.5); has highest priority for card ETA line.
  • data-eta-min-days / data-eta-max-days (optional): numeric ETA override in days.
  • Never duplicate domain meaning in tags (for example, do not use backend in data-tags).

Page hero and at-a-glance tickers

The .backlog-hero block at the top of services/portal/internal/governance/backlog/index.html announces the page (eyebrow INTERNAL · ROADMAP, gradient-accented Backlog title, single-paragraph lead) and exposes a calm 4-tile snapshot of the current backlog state. The hero never reflects the filter set — it is a stable "where are we overall" summary; the cockpit below is what reacts to filters.

DOM and tone contract

  • Container: <section class="backlog-hero"> with two decorative siblings (.backlog-hero__bg with two tinted orbs, .backlog-hero__dots) and a 2-column grid .backlog-hero__layout (copy + tickers).
  • Title accent: .backlog-hero__title-accent uses the same monochromatic ramp as .portal-hero__title-accent on the Hall of Contributors page (linear-gradient(140deg, var(--text) 20%, color-mix(var(--accent) 68%, var(--text)) 100%)) — no rainbow stops.
  • Tickers: a <dl class="backlog-hero__tickers"> with 4 cards .backlog-hero__ticker[data-tone]. Each card has a 3 px tone-colored left signature bar (--ticker-tone) and a soft tone halo on the top-right corner.

Tones (per data-tone)

  • todo → #94a3b8 (slate)
  • progress → #f59e0b (amber)
  • blocked → #a855f7 (purple)
  • done → #10b981 (green)

Counts contract

  • Each ticker has a <span data-backlog-ticker="todo|in-progress|blocked|done"> that paintBacklogHeroTickers() writes into on init (one-shot).
  • Counts come from document.querySelectorAll(".backlog-item") grouped by data-status — the entire backlog, not the filtered view.
  • Unknown statuses are silently ignored; data-status defaults to todo when missing.

Implementation map (page hero)

  • Markup: .backlog-hero block in services/portal/internal/governance/backlog/index.html.
  • Counter wiring: paintBacklogHeroTickers() in services/frontend/portal/assets/backlog.js, called once at the tail of the init IIFE after applyFilter().
  • Visuals: .backlog-hero* rules in services/frontend/portal/assets/backlog.css.

Filters and presets

Detailed filters

  • Group is single-select.
  • Priority, Task type, Risk, Confidence, and Status are multi-select (All resets each dimension).
  • Filter expression is an AND across active dimensions.

Quick filters

  • All — no quick preset restrictions.
  • My focus — open statuses + P0/P1.
  • Open — todo, in-progress, blocked.
  • Blocked — blocked status only.
  • High risk — P0 + task type bug or tech-debt (preset, not status).

Sync behavior

  • Clicking quick presets updates detailed filters.
  • Changing detailed filters recalculates quick preset state; non-matching combinations become internal custom state.

Implementation map (filters)

  • Main filter predicate: matches(item) in services/frontend/portal/assets/backlog.js.
  • Quick preset rule evaluation: matchesQuickPreset(item, status, priority).
  • Quick → detailed sync: applyDetailedFromQuickPreset(preset).
  • Detailed → quick sync: syncQuickPresetFromDetailed().
  • Detailed filter state attributes: data-filter-group, data-filter-priority, data-filter-task-type, data-filter-risk, data-filter-confidence, data-filter-status in services/portal/internal/governance/backlog/index.html.

Sorting and view behavior

Stable sorting policy

  • Open tasks first; closed tasks after.
  • Within open tasks: open P0 first, then bug, then P1, P2, P3.
  • Final tie-break by numeric item id (item-XX) to keep order deterministic.

Implementation map (sorting)

  • Sort orchestrator: sortItemsInPlace() in services/frontend/portal/assets/backlog.js.
  • Open vs closed ordering: comparator block using isOpenStatus(status) and openRank.
  • Open P0 → bug → P1/P2/P3 ordering: priorityBucketRank(item, status, priority) with prioritySortRank.
  • Status tie-breaker: statusSortRank map in comparator.
  • Final deterministic tie-break: itemNumber(item) parsing id="item-XX".

View switch

Three view modes share the same item set; data-backlog-view on <body> drives the layout. applyViewMode() performs DOM restructure for board and timeline modes (and tears it down on switch); list mode is the bare flat stack.

  • List (data-backlog-view="list"): flat vertical stack of .backlog-item cards inside #backlog-task-list. No wrappers, no per-status grouping.
  • Board (data-backlog-view="board"): a 5-column kanban. buildBoardColumns() creates one .backlog-board-column[data-board-column] wrapper per status (in fixed order: todo → in-progress → blocked → done → rejected) and moves items into the matching wrapper. Each column has a sticky .backlog-board-column-header (status-color dot, uppercase label, count badge) and a .backlog-board-column-empty placeholder shown only when no visible items remain. The outer #backlog-task-list is display: grid; grid-template-columns: repeat(5, minmax(260px, 1fr)); each column is its own display: flex; flex-direction: column stack so column heights are independent (the previous flat-grid attempt left empty space above shorter columns because CSS Grid rows are global).
  • Timeline (data-backlog-view="timeline"): a vertical ETA axis. buildTimelineSegments() creates 6 .backlog-timeline-segment[data-timeline-segment] rows (now ≤1d, soon 1–3d, mid 3–5d, later >5d, blocked, shipped) and routes items via timelineSegmentKeyForItem(). ETA is the midpoint of data-eta-days="A-B" (or data-eta-min/max-days, or estimateForItem() as the final fallback). Status overrides ETA bucketing for blocked, done, and rejected so finished or stuck work always lands in its dedicated lane. Each segment has a left rail (tone dot + uppercase label + hint + count) and a right items column; a continuous neutral 1px axis line is drawn via a single pseudo-element on the parent. Palette is a single-accent gradient (vivid → faded muted) for ETA bands so the axis never competes with priority-tag colors; reserved tones are amber for blocked and green for shipped.

Board / timeline counts on filter changes

  • refreshBoardColumnCounts() and refreshTimelineSegmentCounts() are called from the tail of applyFilter() so the badge counts reflect the currently visible (post-filter) item set.
  • mountTaskList() rebuilds the active board/timeline scaffolding after the rAF-deferred re-mount (which wipes #backlog-task-list.innerHTML), keeping the structure intact across re-mounts.

Implementation map (view modes)

  • State setter: applyViewMode() sets data-backlog-view on body.
  • View controls: buttons with data-view-mode in services/portal/internal/governance/backlog/index.html are wired in wireButtons().
  • Visual behavior: CSS selectors body[data-backlog-view="board|list|timeline"] ... in services/frontend/portal/assets/backlog.css.

Task card PM-style UI

Cards switched to a ticket-style layout in the 2026-05-08 pass. Color cue lives on the left edge of the card (priority bar) and on the status pill in the top-right corner; the body stays calm — no corner-glow gradient, no tag rainbow. The corner gleam was removed entirely so the card reads as a flat sheet; overflow: visible on .backlog-item lets [data-tooltip] hints on chips render above the card without clipping.

Anatomy (current, 2026-05-08)

  • Priority left bar: 5 px border-left in --priority-tone (P0 rose #ef4444, P1 amber #f59e0b, P2 blue #3b82f6, P3 slate #94a3b8); the same tone tints the hover shadow.
  • Header (<h2>): CSS Grid with grid-template-areas: "priority . status" / "title title title" — the priority badge sits top-left, the status pill is anchored top-right, and the title (with stable id #N inside .backlog-heading-main) takes a full-width second row. Title never wraps under the pill.
  • Priority badge: small mono-typeset chip (.priority-tag) with hairline border + 10% tonal fill; the Priority: P0 chip in the meta row was removed (it duplicated this badge).
  • Status pill (.status-pill): existing palette per status; --blocked gets a soft 2.6 s pulse animation (respects prefers-reduced-motion).
  • Clickable title: .backlog-heading-main carries role="button" + tabindex="0"; click / Enter / Space opens the task details modal.
  • Meta row (.backlog-task-meta): two chips — Owner (avatar + name) and Age (days since data-created-at). The owner avatar resolves from __DOCS_PORTAL_DATA__.people by display name (case-insensitive); when no portal record matches, a monogram fallback (initials in a tonal circle) is rendered.
  • Group + tag chips: one row of .backlog-chip.backlog-chip--group + .backlog-chip (one per data-tag).
  • Risk / Confidence chips: each chip carries a leading colored dot via ::before (backlog-chip--risk-{level} / backlog-chip--confidence-{level}). Risk: high = red, medium = amber, low = green. Confidence is inverted (high = green = "we know what we're doing").
  • ETA mini-bar (.backlog-task-eta): single row with uppercase mono ETA label, a 0–7-day track with thin tick marks at 1 d / 3 d / 5 d, an accent-gradient fill spanning etaBarMin..etaBarMax, and a compact range readout (1–2.5d, format from etaLine.replace(/\s*days?$/i, "d")). On viewports ≤520 px and inside board cells, the bar wraps onto its own row to keep the range readable.
  • No inline action buttons. The previous Open details / Copy link footer was removed — the title is the sole open trigger; Copy link lives in the modal footer.
  • Expanded view: the .backlog-task-expanded block (with auto-injected sections 4/5 when missing) stays in the DOM but is rendered only inside the modal (cloned, never moved).
  • TOC safeguard: expanded title uses h4 so it does not appear in On this page.

Task details modal

Clicking a card title opens a centered, scrollable modal with the full task description plus action footer. The modal is the sole entry point for the long-form view — there is no inline expand on the card anymore.

DOM contract

  • Singleton mount: <div id="backlog-task-modal" class="backlog-task-modal" hidden> appended to <body> on first open by ensureTaskModal().
  • Structure: .backlog-task-modal__backdrop (blurred dim layer, dismiss target) + .backlog-task-modal__panel (role="dialog" aria-modal="true" aria-labelledby="backlog-task-modal-title") → header (.backlog-task-modal__head with priority badge, title, status pill, ✕) + body (.backlog-task-modal__body, scrollable) + footer (.backlog-task-modal__foot with Copy link ghost + Close primary).
  • Body content: openTaskModal(card) clones the card's .backlog-item-summary, .backlog-item-risk-confidence, .backlog-task-meta, .backlog-task-eta, .backlog-task-progress, and .backlog-task-expanded in that order. Cloning preserves card styling without detaching the originals from the card.

Open / close

  • Open trigger: click / Enter / Space on .backlog-heading-main--clickable on any .backlog-item. The opener element is stashed on modal.__openerButton for focus restoration.
  • Close triggers: three paths — ✕ button (.backlog-task-modal__close[data-modal-close]), backdrop click (.backlog-task-modal__backdrop[data-modal-close]), and the global Escape keydown handler installed at module init.
  • Body lock: document.body.classList.add("is-task-modal-open") freezes background scroll while the modal is open.
  • Focus return: closeTaskModal() calls opener.focus() if the stash is still in the DOM.

Footer actions

  • Copy link (data-modal-action="../../../front/screens/copy-link"): copies window.location.origin + pathname + "#" + card.id via navigator.clipboard.writeText; falls back to setting window.location.hash = card.id if clipboard is denied. Visible feedback swaps the label to Link copied for 1.4 s.
  • Close: dismisses the modal (same path as the ✕ button).

Animation and motion

  • Backdrop fades in over 220 ms; panel scales-in 240 ms with cubic-bezier(0.22, 1, 0.36, 1).
  • prefers-reduced-motion: reduce disables both animations.
  • Mobile (≤640 px): the panel becomes a full-bleed sheet (border-radius: 0, height: 100vh).

Implementation map (modal)

  • Builders: ensureTaskModal(), openTaskModal(card, opener), closeTaskModal() in services/frontend/portal/assets/backlog.js.
  • Trigger wiring: renderTaskCardWorkspace() attaches click + keydown listeners to .backlog-heading-main on each card (guarded by dataset.taskTitleClickable).
  • Visuals: .backlog-task-modal* rules at the bottom of services/frontend/portal/assets/backlog.css.

KPI strip metrics

KPI tiles are computed from the currently visible set (after all filters/search are applied), so numbers always reflect the exact manager view on screen.

Core metrics

  • Open, In progress, Blocked — count plus percentage of visible scope.
  • Done % — done ratio of the current visible scope.

Risk and flow metrics

  • SLA breach risk — blocked P0/P1 tasks with data-blocked-days > 5.
  • ETA pressure — open tasks with narrow estimate window (max ETA <= 3 days, from estimate model).
  • Flow health — throughput in /wk based on done tasks over a 14-day window.

Calculation formulas (current implementation)

  • Scope base: all KPI values are calculated from visible tasks after current filters/search.
  • Open % / In progress % / Blocked %: count(status) / visible_count * 100.
  • Done %: done_count / visible_count * 100.
  • SLA breach risk count: status=blocked AND priority in (P0,P1) AND blocked_days > 5.
  • ETA pressure count: open tasks with eta_max_days <= 3 (explicit override first, estimate-model fallback second).
  • Flow health: (done_in_14_days / 14) * 7, rendered as X/wk.

Implementation map (KPI strip)

  • Main KPI renderer: renderCockpitStats() in services/frontend/portal/assets/backlog.js.
  • SLA configuration: SLA_BLOCKED_THRESHOLD_DAYS constant.
  • Flow window: FLOW_HEALTH_PERIOD_DAYS constant.
  • ETA pressure threshold: ETA_PRESSURE_MAX_DAYS constant.
  • Card-level KPI attributes: data-blocked-days, data-blocked-at, data-done-days-ago, data-done-at, data-actual-hours, data-eta-days, data-eta-min-days, data-eta-max-days.

Fallback behavior

  • data-blocked-days falls back to data-blocked-at (days since date) when provided.
  • data-done-days-ago falls back to data-done-at (days since date) when provided.
  • Card ETA line defaults to recalibrated estimate at 3 h/day; explicit ETA fields override this default.
  • ETA pressure uses data-eta-max-days when set; otherwise it falls back to computed estimate model.
  • If no done recency attributes are present, flow KPI uses current done snapshot as proxy and marks it in tile meta.
  • If both explicit and fallback fields are missing for SLA or flow recency, that card is excluded from the corresponding strict age calculation.
  • If visible_count = 0, percentage tiles show safe zero-based output from guarded denominator logic (no divide-by-zero errors).
  • KPI severity visuals: healthy, watch, risk with dedicated tile styling.

Intelligence panel behavior

  • Rendered by renderIntelligencePanel() using current visible task set.
  • Includes: today attention, top 3 bottlenecks, and reprioritization hints.
  • Desktop-only visibility is controlled in services/frontend/portal/assets/backlog.css (#backlog-intelligence hidden below desktop breakpoint).

Taxonomy validation

  • Runtime validates each card against allowed group/tag sets.
  • Invalid cards are highlighted with backlog-item--invalid-taxonomy.
  • Cockpit warning panel (#backlog-validation-alert) lists invalid items with direct anchors.

Implementation map (validation)

  • Validator entry point: validateTaxonomy() in services/frontend/portal/assets/backlog.js.
  • Allowed taxonomy sets: allowedGroups and allowedTags constants.
  • Alert mount: ensureValidationMount() creates #backlog-validation-alert.
  • Error highlighting class: backlog-item--invalid-taxonomy styled in services/frontend/portal/assets/backlog.css.

Where to change what

  • services/portal/internal/governance/backlog/index.html — page hero markup (.backlog-hero + [data-backlog-ticker] mounts), cockpit chrome, filter controls, task cards (the 48 source section.card.backlog-item entries), content labels, and the Page history table.
  • services/frontend/portal/assets/backlog.js — filter state, quick presets, sorting, validation, grouping, hero ticker counts (paintBacklogHeroTickers), kanban board build/teardown (buildBoardColumns / teardownBoardColumns / refreshBoardColumnCounts), timeline segments (buildTimelineSegments / teardownTimelineSegments / refreshTimelineSegmentCounts / itemEtaMidDays / timelineSegmentKeyForItem), task details modal (ensureTaskModal / openTaskModal / closeTaskModal), and clickable-title wiring inside renderTaskCardWorkspace.
  • services/frontend/portal/assets/backlog.css — cockpit look and feel, responsive behavior, hero (.backlog-hero*), ticket-style cards (.backlog-item with --priority-tone per data-priority, h2 grid header), board view (body[data-backlog-view="board"] + .backlog-board-column*), timeline view (body[data-backlog-view="timeline"] + .backlog-timeline-segment* with single-accent gradient palette and neutral axis line), and task details modal (.backlog-task-modal*).
