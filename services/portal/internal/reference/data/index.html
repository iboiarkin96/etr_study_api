Database tables

Authoritative table-level documentation for the api service's persistence layer. Every page on this hub follows the conventions in ADR 0029: a metadata block, a column-by-column schema, constraints and relationships, lifecycle, and cross-references to the HTTP-operation spec pages that read or write the table.

Storage engine today: SQLite (ADR 0015). Schema is portable to PostgreSQL — conventions in ADR 0029 hold regardless.

Tables

Core business

  • users — end-user identity per source system.

Reference / lookup

  • systems — catalog of upstream systems that own user identities.
  • invalidation_reasons — reasons a row may be marked invalid (soft-delete pair).
  • timezones — IANA timezone catalog with UTC offset.

Templates

  • data-table.html — copy-paste shell for a new table spec (lives under reference/templates/).

Conventions at a glance

Full normative text is in ADR 0029. Quick reference:

  • Table names: snake_case, plural.
  • Core PK: UUID string named <entity>_uuid; reference PK may be code.
  • Audit columns on core tables: created_at, updated_at (timezone-aware UTC).
  • Soft-delete pair: is_row_invalid (Integer 0/1) + nullable invalidation_reason_uuid FK.
  • Foreign keys are always indexed (index=True).
