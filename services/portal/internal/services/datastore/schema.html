datastore · Schema reference

Every table in the runtime database, generated by reading the SQLAlchemy models under services/api/app/models/ — the models and Alembic migrations are the source of truth; this page is the human-readable map. Storage engine and its rationale: ADR 0037.

Relationships that matter

  • users is the ownership root. conspectuses, conspectus_schedules, conspectus_events, conspectus_review_logs and learning_errors all carry owner_client_uuid — every read path filters by it, which is what makes the 404-not-403 cross-tenant rule cheap to enforce.
  • conspectuses ⇄ conspectus_schedules is strictly one-to-one (shared PK, ON DELETE CASCADE). Content and scheduling deliberately live in separate rows so a content edit never touches the review state, and vice versa.
  • Audit rows cascade with their conspectus (ON DELETE CASCADE on conspectus_events / conspectus_review_logs) — but learning_errors.conspectus_uuid uses ON DELETE SET NULL: the learner's error line outlives the note it pointed at.
  • Reference tables are FK targets only: systems and invalidation_reasons into users; timezones.code into users.timezone; schedule_policies into conspectus_schedules.

Migrations

  • Alembic owns the schema. Versions live in services/api/alembic/versions/; two on file today (0001 initial schema, 0002 conspectus / schedule / error-log tables). Every migration ships with a working downgrade().
  • Additive-first. Column renames are a remove + add on the wire and are treated as removals — same rule as the API schema contract (schemas as the public contract). Destructive migrations need an explicit call-out in the PR description.
  • When a migration fails, the recovery procedure is runbook 0002 — migrations failing.
  • Open gap (honest): the app role can run DDL — the DML-only application role / separate migration role split from the threat model is not implemented yet.
