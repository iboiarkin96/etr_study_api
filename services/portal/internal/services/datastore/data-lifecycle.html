Reference PII Retention

datastore · Data lifecycle — classification, retention, PII

What class of data each table holds, how long it lives, and where the honest gaps are. Companion to the schema reference: that page says what the columns are, this page says what we owe the people whose data sits in them.

Data classification

ClassDefinitionWhere it lives
PIIIdentifies a natural person directly or in combinationusers.full_name, users.username, users.system_user_id (pseudonymous external id, e.g. a Telegram user id), users.timezone (coarse location signal); actor columns on conspectus_events / conspectus_review_logs
User contentLearner-authored material — private by default, readable only by its ownerconspectuses (title, cue_sheet, dense_paragraph, bullets), learning_errors.message (free text — learners may put anything here, treat as sensitive)
Behavioural / auditRecords of what the user did and whenconspectus_events, conspectus_review_logs, review timing fields on conspectus_schedules
InfrastructureOperational plumbing that may embed copies of the classes aboveidempotency_keys.response_body stores the full serialized response — including user content and PII of the original write
ReferenceNon-personal dictionariessystems, invalidation_reasons, timezones, schedule_policies

The least obvious PII surface is idempotency_keys.response_body. It replays whole responses, so it inherits the classification of whatever the endpoint returned. Its 24-hour TTL is therefore a privacy control, not only a storage control — cleanup must stay reliable.

Honest gaps — known, owned, not yet closed

  1. No hard-erasure path. Soft delete hides rows from the API but the bytes stay. A "delete my data" request today would be a manual SQL operation touching users, content tables, audit logs and any live idempotency_keys rows. Before real third-party users arrive, this needs a designed erasure procedure (and a decision on whether audit rows are anonymised or removed).
  2. No purge job for soft-deleted rows. is_row_invalid=1 rows accumulate indefinitely; a retention ceiling (e.g. purge N days after invalidated_at) is undefined.
  3. No backup automation. The ./var/postgres/ volume is the only copy; encrypted pg_basebackup + restore drills are the open follow-ups named in ADR 0037. Until a restore has actually been rehearsed, "we have the data" is a claim, not a fact.
  4. No audit-log retention ceiling. Indefinite behavioural history is a liability as much as an asset; a ceiling (or aggregation-after-N-months rule) should be decided deliberately, not by default.

Each gap above is a candidate backlog item; the point of this page is that the gaps are written down where a reviewer will find them, instead of being discovered during the first deletion request.
