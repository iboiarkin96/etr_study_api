datastore · Dependencies

What the datastore depends on (build-time and runtime) and who depends on it. The datastore is not a standalone process — it lives inside the api service as the persistence layer (SQLAlchemy session, Alembic migrations, an embedded SQLite file in dev). The Postgres swap for production is wired but not yet active.

Upstream — build-time dependencies

The «build-time» layer here is the set of tools the engineer runs to author models, generate migration scripts, and apply them to a database. Source of truth is services/api/app/db/.

ComponentUsed for
Python (≥ 3.12)Runs Alembic CLI (alembic upgrade head / alembic revision --autogenerate), drives make -C services/api migrate.
SQLAlchemyDeclarative ORM under services/api/app/models/. Defines the schema the migrations are diffed against.
AlembicMigration tool. Owns forward (upgrade) and backward (downgrade) revisions under services/api/alembic/versions/; produces deterministic SQL from model deltas.
pre-commit (Alembic head check)Refuses commits that introduce two unmerged Alembic heads. Catches a forgotten alembic merge before push.

Runtime dependencies

ComponentRole
SQLite (embedded, dev)Single file under services/api/var/study_app.sqlite served via SQLAlchemy. Single-writer semantics — see the 2026-04-02 write-lock burst postmortem.
Postgres (planned, prod)Planned production engine per the datastore entity card. The SQLAlchemy URL is the only switch — no app-level code changes anticipated.
SQLAlchemy engine + sessionThe api process owns one engine and short-lived per-request sessions. Pool size: default (5 + overflow); a dedicated datastore/architecture.html page will hold the rationale once stood up.

There is no separate database process in development — SQLite is opened in-process by the api on request. Production Postgres will be a network dependency over TCP; the SLO and on-call rules in that case fall under the api on-call rotation by default until the data team owns a separate slot.

Downstream — who consumes the datastore

ConsumerWhat they read / write
api serviceThe only direct consumer. Reads + writes via SQLAlchemy session; idempotency-key dedup and request-correlation columns are part of the schema (per ADR 0006).
Alembic CLI (operator)Out-of-band consumer at deploy time — alembic upgrade head applies pending migrations before the api starts serving traffic.
monitoring (indirect)No direct SQL — observes through the api's /ready probe (the probe touches the DB) and through structured logs of slow / failed queries.
