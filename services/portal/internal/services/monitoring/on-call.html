monitoring · On-call

Monitoring is the source of alerts for the rest of the platform — but it is itself unmonitored. When monitoring is broken, the platform's pages stop firing. Triage starts here when an operator can't see metrics, can't query logs, or suspects the alert chain itself is broken.

Alert routing (what fires where)

Per ADR 0011, alerts are error-budget-driven. Today, "routing" is a Grafana alert rule that posts to a manually-checked surface; paging is unconfigured (solo project).

AlertTrigger (sketch)Severity
FastBurn14.4× burn over 1 h on api:error_rate:5mPage (when paging is wired)
SlowBurn6× burn over 6 h, sustainedTicket / backlog item
ProbeDownBlackbox /ready down for 2 consecutive intervalsPage
ScrapeDownPrometheus target up == 0 for 5 minTicket — monitoring is broken, not the api
LogPipelineSilentElasticsearch ingest rate = 0 for 10 min while api receives trafficTicket
