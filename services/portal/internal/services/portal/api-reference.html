Internal docs

portal · API reference

The portal exposes no HTTP API. Its surface is a static URL tree served over HTTPS. "API reference" here means the contract a reader (or a deep-link) can rely on: top-level URLs, the search index format, and the asset versioning rules. Catalog spine slot kept for parity with the api / monitoring entries — see RFC 0004 §4.4.

URL surface — what a reader can deep-link

Two trees with stable top-level paths. Internal links inside the portal use relative paths so the build can move between hosts without breaking.

Search index — JSON contract

The portal ships a client-side search index at a fixed path. Per ADR 0027, the format is documented and stable so future tooling can re-rank without rebuilding the source map.

  • Path: /internal/assets/search-index.json (and a public counterpart for /).
  • Schema: array of { url, title, headings[], excerpt, audience }. audience = "public" | "internal".
  • Producer: scripts/build_docs_search_index.py; runs in make docs-fix.
  • KPI / SLO: per RFC 0002.

Asset versioning

Shared CSS / JS / fonts under services/frontend/portal/assets/ are referenced by relative path from every HTML page. There is no fingerprinting at build time today; cache busts ride on edits to the asset filenames themselves (see CSS architecture).

If long-term caching becomes a problem, the right fix is to add a fingerprint suffix in the build step — not to add a runtime layer. Track in backlog.
