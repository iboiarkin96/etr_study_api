api · On-call

Who responds when the api service is unhealthy, what they look at first, and how alerts route. The project is in solo-maintainer mode — there is one owner; "rotation" and "escalation" reduce to a single backlog queue. The shape of this page mirrors the Backstage on-call descriptor so it scales when the team grows.

Health signals — what to look at first

The api exposes three operational endpoints. Standard order when triaging: /health → /ready → /metrics → logs.

EndpointMeaningConsumer
GET /healthProcess is alive.Container orchestrator liveness probe.
GET /readyService can accept traffic (DB reachable, migrations applied).Blackbox exporter (monitoring); orchestrator readiness probe.
GET /metricsPrometheus exposition (request rate, latency, error rate).Prometheus scrape (monitoring).
NDJSON log filePer-event structured records.Filebeat → Elasticsearch / Kibana (optional).

Authority for these signals: ADR 0009; SLO and error budget per ADR 0011; structured logging per ADR 0023.
