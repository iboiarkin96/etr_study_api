0007 — OpenAPI canon-parity failing

The parity gate detected divergence between the analyst-authored canon (etr_study_app/merged-spec.json) and the shipped FastAPI runtime spec. Two failure classes, two recovery paths — pick the one that matches the message.

Impact

  • Local developer loop blocked until the failing stage clears.
  • No production user impact unless the failure is in shipped artefacts.

Follow-up

  • Capture the root cause in the related postmortem if SEV-2 or higher.
  • Open backlog items for any preventive change discovered during recovery.

Trigger

  • make openapi-check fails with «Canon operation missing in code: …» or «Operation in code but not in canon or exceptions: …» or «Canon-required … missing or optional in code for …».
  • make verify fails on the openapi-check stage.
  • CI reports canon-parity issues on the PR build.

Fast triage

  1. Run make openapi-check locally to read the exact message.
  2. Match it to one of the two failure classes below.
  3. Pick the recovery branch based on which side of the parity is «right»: canon (analyst) or code (dev).

Most common causes

  • Field renamed or type-changed in a Pydantic model on the code side, but the canon fragment still declares the old shape.
  • New endpoint added in code without a matching fragment under openapi/etr_study_app/fragments/.
  • Endpoint removed in code while its fragment is still in canon.
  • Response status set changed on the code side (e.g. removed 422 or 404) while the canon still lists it.
  • Field or parameter flipped between required and optional on either side.
  • A newly-added internal telemetry / infra endpoint hasn't been added to KNOWN_NON_CANON_OPERATIONS.

Recovery paths

Case A · Canon says X exists, code doesn't (or code shape weaker than canon)

Messages: «Canon operation missing in code…», «Canon-required … missing or optional in code for …», «Canon-documented response N missing in code…».

Question to answer: is the canon aspirational (analyst hasn't been implemented yet) or is the code broken?

  • Canon is aspirational. Leave the fragment alone; the ratified spec must be implemented before the PR can land. Implement the missing operation / field / response in services/api/app/…; re-run make openapi-check. This is the normal API-first flow: canon leads, code follows.
  • Code is broken. Something in the FastAPI handler / Pydantic model / response definition drifted from what the canon promises. Fix it in the code (revert the rename, add back the required field, restore the response code). Do not update the canon to match — that hides the break from every future review.

Case B · Code has an operation the canon doesn't cover

Message: «Operation in code but not in canon or exceptions: METHOD /path…».

Question to answer: is this a public product endpoint, or infrastructure?

  • Public product endpoint. The API-first flow requires a canon fragment before shipping. Author it: add services/portal/internal/services/api/openapi/etr_study_app/fragments/<resource>/<operationId>.yaml using the createUser fragment as reference. Run make api-check to regenerate merged-spec.json. Re-run make openapi-check.
  • Infrastructure endpoint (health probe, telemetry). Add (method, path) to KNOWN_NON_CANON_OPERATIONS in tools/api/openapi_governance.py. This flags the exception for PR review and lets the parity gate pass. Every entry there should be justifiable in one sentence.

Final recovery sequence

make api-check      # regenerate canon merged-spec.json if a fragment changed
make openapi-check  # confirm canon parity passes
make verify         # full local gate before push

Exit criteria

  • make openapi-check passes with both «OpenAPI lint» and «OpenAPI canon parity (etr_study_app)» green.
  • make verify passes end-to-end.
  • If you added an entry to KNOWN_NON_CANON_OPERATIONS, the PR description names the endpoint and explains why it's not public API surface.
  • If you touched a canon fragment, the corresponding portal-spec page (services/portal/internal/services/api/reference/<resource>/operations/…html) has been updated to match.

Escalation

If 30 minutes of triage didn't get make openapi-check green, or the failure mode isn't on the «Most common causes» list:

  1. Capture state — make openapi-check 2>&1 | tee /tmp/runbook-0007-state.txt, the current services/portal/internal/services/api/openapi/etr_study_app/merged-spec.json, git status, and the recovery steps tried.
  2. Open a SEV-3 incident in the team backlog and link this runbook from it. A breaking contract change shipped to a published version jumps to SEV-2 immediately — downstream consumers will hit it on the next deploy.
  3. Page on-call (or, for solo work, the project incident commander) with the captured state and the last known-good canon commit hash.
  4. Do not weaken the canon fragment or grow KNOWN_NON_CANON_OPERATIONS just to «make the check pass» — canon is the analyst contract, not a movable target. If the code is right and the canon is wrong, update the canon deliberately (as a documented change) rather than silently.

Follow-up

  • If compatibility rules changed, update an ADR and developer docs.
  • If this keeps happening in the same shape, tighten run_parity_check in tools/api/openapi_governance.py — the gate should get harder to bypass over time, not easier.
