Overview

Return the full ConspectusResponse for the given conspectus_uuid when the row is owned by the calling tenant. Read-only. Rows soft-invalidated via deleteConspectus are still returned by this endpoint — the client inspects is_row_invalid and invalidation_reason_uuid to decide what to render.

§1Business context

The point look-up is the workhorse behind every «open one conspectus» flow: entering a note from a search result, resolving a link out of the study-load badge, resuming a review card after a page reload. It's also the read half of every write-then-refetch pattern the client uses — patchConspectus already returns the row, but a subsequent refetch here confirms external cache invalidations.

§2User stories

  • US-CONSP-GET-1. As a learner, I want to open a specific note from any surface (list, search, notification) and see its current state.
  • US-CONSP-GET-2. As an integrator, I want a stable identity endpoint so I can round-trip an edit: PATCH → local state update → GET for the confirmed shape.
  • US-CONSP-GET-3. As a data auditor, I want to inspect a soft-invalidated row (via ?include_invalid pattern elsewhere) without exposing it in the day-to-day learner UX.

§13Edge cases checklist

  • Row soft-invalidated. Return the row with is_row_invalid = 1. Client decides whether to render or hide.
  • Cross-tenant row. 404 CONS_404. Never leak existence with 403.
  • Idempotency-Key present. Ignored. No 400.
  • Path UUID has valid syntax but no row. 404 CONS_404.
  • Repeated calls during a single second. Each returns a fresh row snapshot; small drift on updated_at is expected after a concurrent patchConspectus.
  • System clock skew between DB and API pod. Timestamps come from DB, not Python datetime.now().

§17Open questions

  • Q1. Should soft-invalidated rows be gated behind an explicit ?include_invalid=true flag on this endpoint too? v1: no — return unconditionally, let clients filter. Revisit if audit tooling ends up needing a distinct read path.
  • Q2. Do we support If-None-Match / ETag for conditional fetches? Deferred — no evidence the client bandwidth cost matters yet.

§18Out of scope

  • Batch fetch by uuid list — separate endpoint if a batch use case emerges.
  • Field projection (?fields=title,slot) — deferred; response is small enough.
  • Response caching headers (see Q2).
  • Cross-tenant admin lookup — never via this endpoint.
