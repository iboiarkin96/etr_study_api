Overview

Cursor-paginated list of the learner's conspectuses. Distinct from listDueConspectuses, which is a bounded scan capped at 100 items for the study-load widget — this endpoint is the full library view with stable forward-only traversal. Sort is fixed at created_at DESC, conspectus_uuid DESC in v1.

§1Business context

Every «my notes» surface, every export flow, every audit UI needs a way to walk the full conspectus set. The due-list is a specialised, small-window view intended for the review flow; this endpoint is the general-purpose catalogue. Cursor pagination is a decision — it keeps the DB query cheap regardless of catalogue size and keeps client behaviour deterministic across late writes.

§2User stories

  • US-CONSP-LIST-1. As a learner, I want to see all my notes ordered newest-first so I can scroll through my library.
  • US-CONSP-LIST-2. As a learner, I want to narrow the view by slot or by creation window without changing endpoints.
  • US-CONSP-LIST-3. As a data auditor, I want to include soft-deleted rows via include_invalid=true so I can inspect the retention state without a separate admin API.
  • US-CONSP-LIST-4. As an integrator, I want cursor pagination so my client can page through arbitrary catalogue sizes without missing or duplicating items during concurrent writes.

§4Functional requirements

IDShort descriptionDetailed characteristic
FR-CONSP-LIST-001Resolve owner.Resolve users.client_uuid from (system_user_id, system_uuid); 404 USER_404 when missing.
FR-CONSP-LIST-002Cursor decoding.Server-signed opaque token encoding (created_at, conspectus_uuid) of the last item returned. Rejected with 422 if signature invalid or clock TTL exceeded (7 days).
FR-CONSP-LIST-003Query shape.WHERE owner_client_uuid = ? AND (created_at, conspectus_uuid) < (?, ?) when cursor present.
FR-CONSP-LIST-004Filters.Compose with AND: slot = ?, created_at >= ?, created_at < ?, is_row_invalid = 0 (unless include_invalid=true).
FR-CONSP-LIST-005Ordering.ORDER BY created_at DESC, conspectus_uuid DESC LIMIT (?+1) — request one extra to know whether more exist.
FR-CONSP-LIST-006Cursor generation.When rows.count > limit, trim to limit and emit next_cursor from the last returned row; otherwise next_cursor = null.
FR-CONSP-LIST-007Envelope.Response is an object { items, next_cursor, count, has_more } — not a bare array.
FR-CONSP-LIST-008Read-only.No writes. Idempotency-Key silently ignored.

§5Non-functional requirements

IDShort descriptionDetailed characteristic
NFR-CONSP-LIST-001Latency.P95 ≤ 200 ms, P99 ≤ 500 ms per page (default limit=50).
NFR-CONSP-LIST-002Index coverage.Composite index on (owner_client_uuid, created_at DESC, conspectus_uuid DESC) covers both the base query and every filter combination.
NFR-CONSP-LIST-003Cursor stability.Cursor decoded value produces identical page even under concurrent writes (rows inserted after the cursor point are picked up on later pages).

NFR-CONSP-LIST-005Payload cap.items.maxItems = 100. Request limit > 100 → 422.
NFR-CONSP-LIST-006Backwards compatibility.Adding a new response field or filter param is non-breaking; changing sort order or cursor format is breaking.

§10Side effects

None. Read-only.

§11Idempotency & concurrency

Not applicable — read-only. Cursor pagination is stable under concurrent writes because it anchors on (created_at, conspectus_uuid); rows inserted after the cursor point appear on the next page.

§13Edge cases checklist

  • Zero conspectuses. 200 OK; items = [], next_cursor = null, count = 0, has_more = false.
  • Race with concurrent create. New rows created between page 1 and page 2 will only surface if their (created_at, uuid) is beyond the cursor — usually they show on the earlier page next time the client restarts.
  • Cursor exceeds TTL. 422 COMMON_000; the message says «restart pagination without cursor».
  • include_invalid=true + slot filter. Compose freely; the invalidated rows show their last-known slot.
  • Deleted mid-page. deleteConspectus is soft — the row is still returned unless include_invalid = false, in which case it's dropped from the next page but never «vanishes» mid-page.
  • Server clock skew. Cursor uses DB now() and DB-stored created_at only.

§14Observability requirements

Log events

  • Standard middleware request_done.
  • conspectus_list_user_not_found (info) on USER_404.
  • conspectus_list_bad_cursor (warn) on cursor validation failure.

Metrics

  • conspectus_list_total (counter).
  • conspectus_list_duration_seconds (histogram).
  • conspectus_list_page_size (histogram, observed count).
  • conspectus_list_pages_walked (histogram, distinct cursors per session — proxy for catalogue depth).

Alerts

  • Page: 5xx rate > 1 % over 5 min.
  • Ticket: P95 > 400 ms over 30 min — likely missing index or over-broad filter.

§15Test plan

  • Unit: cursor encode/decode roundtrip; signature tampering rejected; TTL enforced.
  • Integration: AC-001..AC-011.
  • Contract: Schemathesis on listConspectuses; envelope shape checks.
  • End-to-end: create 60 rows → page through → assert no gaps / duplicates / order violations.
  • Property: for any interleaving of creates and reads across two pages, union of item uuids ⊆ full-set uuids and no duplicates.

§17Open questions

  • Q1. Do we support server-side full-text search (?q=) here or as a separate searchConspectuses endpoint? Deferred — separate op is cleaner for observability and indexing.
  • Q2. Sort direction — should we support ?sort=next_review_at:asc? Deferred; use listDueConspectuses for review-order use cases.
  • Q3. Cursor TTL — 7 days is a guess. Tune once we have telemetry on conspectus_list_pages_walked.

§18Out of scope

  • Total-count field. Requires an extra query; not needed by any client in v1.
  • Backward cursors. Forward-only is sufficient for infinite-scroll UI.
  • Field projection (?fields=…).
  • Full-text search (Q1) and alternative sort orders (Q2).
