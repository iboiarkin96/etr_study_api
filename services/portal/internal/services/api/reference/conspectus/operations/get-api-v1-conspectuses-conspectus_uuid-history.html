Overview

Immutable audit trail for a Conspectus — every reviewConspectus and every patchConspectus emits an event on this stream. Cursor-paginated in chronological order. Two event kinds share one discriminated union: review events carry slot transitions, content_patch events carry the list of changed fields.

§1Business context

The history stream is the analytics primitive for the ETR loop: it makes «did this note actually get better over time?» a queryable question. It's also the backbone of any «why is my schedule off?» support ticket — the review series shows exactly which tags landed, in what order, and how the ladder walked. Immutability is the value: a mutable log stops being a truthful traceback (see the error-log resource for the mirror-image decision).

§2User stories

  • US-CONSP-HIST-1. As a learner, I want to see how a note has evolved so I can decide whether to keep drilling it or retire it.
  • US-CONSP-HIST-2. As a support engineer, I want to inspect the full review series when a learner reports «my schedule is broken».
  • US-CONSP-HIST-3. As a data auditor, I want to distinguish content changes from schedule changes so I can compute «retention lift per content revision».
  • US-CONSP-HIST-4. As an integrator, I want cursor pagination so I can replay long histories deterministically.

IDShort descriptionDetailed characteristic
FR-CONSP-HIST-001Resolve owner + conspectus.Same guard as getConspectusById; miss → 404 USER_404 or CONS_404.
FR-CONSP-HIST-002Merge two source tables.UNION ALL over review_events and content_events tagged with event_type.

FR-CONSP-HIST-004Since filter.WHERE created_at >= ? — inclusive lower bound.
FR-CONSP-HIST-005Ordering.ORDER BY created_at ASC, sequence ASC. sequence is a per-conspectus monotonic counter that breaks same-timestamp ties.
FR-CONSP-HIST-006Cursor.Opaque token encoding (created_at, sequence) of the last event returned; server-signed with 7-day TTL.
FR-CONSP-HIST-007Discriminated union.Each item has an event_type discriminator. Exactly one of review / content_patch is present (not both, not neither).
FR-CONSP-HIST-008Immutable.No PATCH, no DELETE on events. Even a deleteConspectus keeps its history readable.

§13Edge cases checklist

  • Simultaneous review + patch. Ordered by sequence — the write that acquired the DB lock first gets the earlier sequence.
  • Soft-deleted parent. History still returns. Client rendering may need to badge the events as «historical».
  • Filter matches zero. 200 OK; not a 404.
  • Cursor from a different conspectus. 422 COMMON_000 — the cursor encodes conspectus_uuid; mismatch is a client bug.
  • Very old history (> 1 year). Pagination still walks; no retention truncation in v1.
  • New event_type shows up. Client must switch on event_type; unknown discriminator is a client bug (see §16).

§15Test plan

  • Unit: UNION query planner uses both indexes; discriminator marshalling.
  • Integration: AC-001..AC-009. Include a case with 30 mixed events and paginate through.
  • Contract: Schemathesis on getConspectusHistory; assert discriminator invariant (exactly one union arm per item).
  • End-to-end: create → patch → review easy → patch → review hard → history returns 4 events in order with correct slot transitions.
  • Property: for any interleaving of writes and history reads, per-conspectus sequence is strictly monotonic.

§16OpenAPI authoring hints

  • operationId: getConspectusHistory.
  • Tags: Conspectus.
  • Response schema: ConspectusHistoryPage — object with conspectus_uuid, items, next_cursor, count, has_more.
  • Item schema: ConspectusHistoryEvent — discriminated union with event_type as discriminator. Consider oneOf + discriminator in v2 for stronger client codegen; v1 keeps both optional-object branches for forward-compat.
  • Examples: mixed_events, only_reviews, empty.
  • Canon fragment: openapi/etr_study_app/fragments/conspectus/getConspectusHistory.yaml — already authored, x-implementation-status: pending.

§17Open questions

  • Q1. Do we include a diff blob (before/after) on content_patch events? Deferred — expensive to compute, few consumers today. Revisit when the analytics team asks.
  • Q2. Should we also emit lifecycle events for createConspectus and deleteConspectus? Deferred — for now, «existence» is derivable from getConspectusById.
  • Q3. Reverse-chronological order option (?order=desc) — deferred until UI needs it.

§18Out of scope

  • Cross-conspectus history queries.
  • Backward cursors / reverse-chronological pagination (Q3).
  • Diff blobs on patches (Q1).
  • Lifecycle events for create/delete (Q2).
  • Webhook / streaming push — polling only in v1.
