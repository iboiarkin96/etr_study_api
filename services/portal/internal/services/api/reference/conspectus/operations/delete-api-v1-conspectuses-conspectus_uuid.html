Overview

Soft-delete a Conspectus by setting is_row_invalid = 1, invalidation_reason_uuid, and invalidated_at. The row is retained in storage — getConspectusById still returns it, listConspectuses hides it unless ?include_invalid=true, listDueConspectuses always hides it. Linked error-log entries have their conspectus_uuid cleared to null — the errors survive.

§1Business context

Learner-authored content carries a retention obligation. Hard-delete is a separate administrative operation (out of scope in v1). Soft-delete gives the client an immediate «gone from my view» response while keeping the row available for audit exports, undo flows, and downstream analytics. The required invalidation_reason_uuid is not a UX nag — it's the audit trail that lets the retention sweep decide when a row is finally purgeable.

§2User stories

  • US-CONSP-DEL-1. As a learner, I want to remove a note I no longer need without losing its history.
  • US-CONSP-DEL-2. As a learner, I want repeated «delete» requests to be safe — accidental double-click doesn't confuse the server.
  • US-CONSP-DEL-3. As a data auditor, I want every deletion tagged with a machine-readable reason so retention sweeps can act on them.
  • US-CONSP-DEL-4. As an integrator, I want the errors I logged against the deleted note to survive with a cleared link — they capture real learning context.

§8Response contract

200 OK — the soft-deleted ConspectusResponse

Full echo of the row post-update. Same shape as getConspectusById. Key differences on a successful delete: is_row_invalid = 1, invalidation_reason_uuid populated, invalidated_at populated, updated_at moved to now.

§10Side effects

  • Conspectus row mutated. is_row_invalid flipped; timestamps set.
  • Linked error-logs modified. conspectus_uuid nulled on all rows referencing the deleted uuid. The error entries themselves survive.
  • Idempotency cache updated. Standard idempotency middleware records the response for 24 h.
  • No schedule mutation. The schedule row is left as-is — listDueConspectuses filters on is_row_invalid.
  • No hard delete. Storage cost persists until the retention sweep.

§11Idempotency & concurrency

Idempotency-Key required. Standard 24 h window. Concurrency: soft-delete of an already-invalidated row is a no-op (see AC-003) — updated_at is not moved and no new event is emitted. Concurrent DELETE + PATCH: the PATCH lands only if it arrives before the DELETE; a PATCH after the DELETE sees is_row_invalid = 1 and the FastAPI handler is expected to reject it with a 404 CONS_404 per the resource ownership check (patch on invalidated row is undefined — patchConspectus should clarify).

§13Edge cases checklist

  • Already deleted. Return the current row; no-op on writes.
  • No linked errors. Cascade UPDATE affects zero rows; not an error.
  • Delete during evening-preview session. getSchedulePreview replay uses computed_at anchor; a deletion after materialisation still returns the item in the preview but the next call won't include it. Client should handle the case that a preview'd item is now soft-deleted.
  • Delete → resurrect. Not supported in v1 (no restoreConspectus). The row is discoverable via listConspectuses?include_invalid=true.
  • Unknown reason UUID. Reject at write time — don't create dangling FK-violating rows.
  • Concurrent double-delete. Two clients with different keys — both see 200 OK; second one is a no-op.

§14Observability requirements

Log events

  • Standard middleware request_done.
  • conspectus_deleted (info) — payload: {conspectus_uuid, reason_uuid, unlinked_error_count}.
  • conspectus_delete_noop (info) — row already invalid.
  • conspectus_delete_not_found (info) on CONS_404.
  • conspectus_delete_user_not_found (info) on USER_404.

Metrics

  • conspectus_delete_total (counter, labels: outcome=deleted|noop|error).
  • conspectus_delete_duration_seconds (histogram).
  • conspectus_delete_unlinked_errors_count (histogram, observed cascade size).

Alerts

  • Page: 5xx rate > 1 % over 5 min.
  • Ticket: Sudden spike in conspectus_delete_total (> 3× 24 h baseline) — possible client bug or account compromise.

§15Test plan

  • Unit: reason-UUID resolver rejects unknown FKs; soft-delete update sets all fields atomically.
  • Integration: AC-001..AC-009. Include a case with 5 linked error-logs and assert the cascade.
  • Contract: Schemathesis on deleteConspectus.
  • End-to-end: create → list (present) → delete → list (absent) → list include_invalid=true (present) → get by uuid (still resolves).
  • Property: deleting a random subset then computing listConspectuses without include_invalid returns the complement set.

§17Open questions

  • Q1. Should patchConspectus on an invalidated row succeed (silent resurrect) or fail with 404? v1: fail — resurrect is a separate deliberate operation.
  • Q2. Do we support hard-delete via ?hard=true? Deferred — administrative concern, out of the API-first product surface.
  • Q3. Do we need a restoreConspectus op? Deferred until product tells us learners want it.

§18Out of scope

  • Hard-delete (Q2).
  • Undo / restore (Q3).
  • Cascade to review-log entries — those live in getConspectusHistory and remain fully readable.
  • Bulk delete by filter.
