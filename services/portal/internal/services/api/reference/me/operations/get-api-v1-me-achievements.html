Overview

The learner's achievement set, derived entirely on read from existing data — per-day review counts (streaks), all-time review count, active conspectus count, and the miss-log count. No unlock rows are persisted, so a badge can never disagree with the data behind it.

§1Business context

Feeds the achievement chips on the Telegram Mini App Profile screen (T-23). The set is deliberately closed and additive-only — twelve keys the client maps to icon + copy (clients ignore unknown keys). The noticer_10 badge rewards logging misses: in the ETR methodology noticing a miss IS the practice, so the achievement system celebrates it instead of hiding it.

§2User stories

  • US-ME-ACH-1. As a learner, I want to see which milestones I've reached so consistency feels rewarded.
  • US-ME-ACH-2. As a learner, I want progress toward locked badges (e.g. «3/10») so the next milestone feels reachable.
  • US-ME-ACH-3. As a learner, I don't want a badge revoked when my current streak breaks — earned is earned.
  • US-ME-ACH-4. As an integrator, I want one call returning the whole set with stable keys, not per-badge endpoints.

FR-ME-ACH-005Unlock rule.unlocked = progress_raw >= target; response progress = min(progress_raw, target).
FR-ME-ACH-006No persistence.Derived on every read; no unlock rows, no timestamps. A wiped dev account loses its badges — by design.
FR-ME-ACH-007Read-only.No DB writes.

§13Edge cases checklist

  • Fresh account. Six items, all locked, all zero — the client renders the full grid dimmed, not an empty state.
  • Badge stickiness. Streak badges ride max(longest, current); a broken current streak never re-locks streak_7/streak_30.
  • Data wipe. Achievements are recomputed on read; wiping learner rows (dev --wipe-all) resets all badges — intended.
  • Invalidated notes. Soft-invalidated conspectuses (is_row_invalid = 1) never count toward notes_10.
  • Clamping. progress in the response never exceeds target, even when the raw counter does.
  • Binary badges. perfect_day / comeback / early_bird / night_owl only ever report 0/1 — the client hides the fraction on their locked chips.
  • Timezone change. The local-hour badges are re-evaluated against the CURRENT users.timezone on every read — moving timezone can newly unlock (or, in principle, re-lock) early_bird / night_owl. Accepted: derived-on-read semantics apply to inputs too.
  • Idempotency-Key sent. Silently ignored.

§17Open questions

  • Q1. Persist unlocked_at timestamps for «earned on» copy? v1: no — derived-on-read keeps the set consistent with data; revisit if product wants unlock dates or notifications.
  • Q2. Localised titles server-side? v1: no — the key is the contract; the client owns copy (i18n en/ru).

§18Out of scope

  • Unlock notifications (bot push on new badge).
  • Seasonal / rotating achievement sets.
  • Per-badge share images («share your streak» is a Profile v2 idea).
