Overview

Current and longest streak of consecutive UTC calendar days with at least one review, plus a fixed 30-day goal for the orb progress arc. Read-only, aggregated from conspectus_review_logs.

§1Business context

Feeds the streak orb on the Telegram Mini App Today screen — the single most-visible signal above the fold. Called on every Today open. The 24-hour grace (walk starts yesterday when today has zero reviews) prevents a spurious «streak lost» reading when the learner opens the app before their first review of the day.

§2User stories

  • US-ME-STATS-1. As a learner, I want to see my current streak so I feel the momentum of consecutive practice days.
  • US-ME-STATS-2. As a learner, I want the streak to survive opening the app in the morning before I've reviewed anything today.
  • US-ME-STATS-3. As an integrator, I want one call for the orb (current + longest + goal), not three separate reads.

§4Functional requirements

IDShort descriptionDetailed characteristic
FR-ME-STATS-001Resolve owner from query.Same composite-key resolution (system_user_id + system_uuid → owner_client_uuid) as getScheduleSummary.
FR-ME-STATS-002Per-day counts.SELECT DATE(reviewed_at), COUNT(*) FROM conspectus_review_logs WHERE owner_client_uuid = ? GROUP BY 1 across the last 365 UTC calendar days.
FR-ME-STATS-003Current streak walk.Start from today when today has ≥1 review; otherwise start from yesterday (24-hour grace). Walk backward while every prior day has ≥1 review; stop at the first zero.
FR-ME-STATS-004Longest streak.Sweep the sorted set of active days once; track the longest run of consecutive dates (delta = 1 day).
FR-ME-STATS-005Invariant.Coerce longest_days = max(longest_days, current_days) so a fresh run never violates the pair semantics.
FR-ME-STATS-006Goal constant.goal_days = 30. Sourced from the DEFAULT_STREAK_GOAL_DAYS constant; not per-learner in v1.
FR-ME-STATS-007Server-side time anchor.today is now() UTC's calendar date; day boundaries are UTC midnight.
FR-ME-STATS-008Read-only.No DB writes.

Equivalent raw SQL

per-day counts + streak walk SQL 
1-- Per-day review counts (base fact used by both current + longest):2SELECT DATE(reviewed_at) AS day, COUNT(*) AS n3FROM conspectus_review_logs4WHERE owner_client_uuid = :owner5GROUP BY 16ORDER BY 1;7 8-- Current streak: walk backwards from today (or yesterday if today is 0).9-- Longest streak: largest gap-free consecutive-days run in the above set.

§13Edge cases checklist

  • Zero reviews ever. All counters zero; goal still 30.
  • Grace boundary. Today zero + yesterday ≥1 → walk starts from yesterday; today zero + yesterday zero → current_days = 0.
  • Fresh 1-day run. Walk yields current_days = 1; service coerces longest_days = max(1, historic).
  • Year-boundary lookback. Lookback window is 365 days; any streak that pre-dates this window is silently truncated in v1.
  • Idempotency-Key sent. Silently ignored.
  • Timezone. Day boundaries are UTC midnight; a learner reviewing at 23:59 local in UTC+3 is recorded on the next UTC calendar day.

§15Test plan

  • Unit: _current_streak (grace boundary, empty set, single-day run) and _longest_streak (single sweep, gaps, single-day set).
  • Integration: AC-001..AC-007; assert longest_days >= current_days invariant.
  • Contract: Schemathesis on getMeStats.
  • End-to-end: seed N consecutive review-log days → stats returns matching counts.

§16OpenAPI authoring hints

  • operationId: getMeStats.

  • Define component: MeStatsResponse wrapping StreakStats.
  • Examples: empty (no reviews) and populated (5-day run with 12-day historic max).

§17Open questions

  • Q1. Per-learner goal_days? v1: no — fixed constant. Revisit if product wants seasonal goals.
  • Q2. Timezone-aware day boundaries? v1: UTC-only. Requires a per-learner tz field on user and a rewrite of the day bucket.

§18Out of scope

  • Multi-year history (year-cap at 365 days).
  • Weekly / monthly cadence buckets.
  • Streak-break notifications (belongs to a separate reminder path).

§19Traceability

  • System design: Telegram Mini App Today hero (T-15d).

  • OpenAPI: getMeStats.
