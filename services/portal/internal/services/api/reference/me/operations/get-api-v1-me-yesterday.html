Overview

Recap of the previous UTC calendar day: reviewed (count of review-log rows in yesterday's window), target (reviewed + still-due-from-yesterday), accuracy_pct (share of easy reviews, rounded to a %), and missed (target minus reviewed, floored at 0).

§1Business context

Feeds the yesterday-recap card on the Telegram Mini App Today screen. Answers «how did I do yesterday?» in one visible line: reviewed / target, an accuracy chip when reviewed > 0, and a «missed» pill when non-zero. The target proxy is intentionally cheap — a schedule-window count, not a replay of yesterday's expected due-list.

§2User stories

  • US-ME-YEST-1. As a learner, I want a one-glance recap of yesterday so I know whether to catch up today.
  • US-ME-YEST-2. As a learner, I want the accuracy chip only when I actually reviewed something (no misleading «0 %» on an idle day).
  • US-ME-YEST-3. As an integrator, I want a single call for the recap card instead of aggregating from /schedule/history + /schedule/summary.

§4Functional requirements

IDShort descriptionDetailed characteristic
FR-ME-YEST-001Resolve owner from query.Same composite-key resolution as getScheduleSummary.
FR-ME-YEST-002Yesterday window.UTC calendar day (today - 1); window is [midnight_utc, midnight_utc + 1d).
FR-ME-YEST-003Reviewed count.COUNT(*) on conspectus_review_logs filtered to owner + yesterday window.
FR-ME-YEST-004Easy count.Same filter + tag = 'easy'.
FR-ME-YEST-005Still-due proxy.COUNT(*) on conspectus_schedules filtered to owner + is_row_invalid = 0 + next_review_at inside yesterday's window.
FR-ME-YEST-006Target formula.target = reviewed + still_due_from_yesterday.
FR-ME-YEST-007Accuracy formula.accuracy_pct = round(easy_count * 100 / reviewed) when reviewed > 0; else 0.
FR-ME-YEST-008Missed formula.missed = max(0, target - reviewed).
FR-ME-YEST-009Read-only.No DB writes.

Equivalent raw SQL

yesterday counts SQL 
 1WITH y AS ( 2  SELECT (CURRENT_DATE - INTERVAL '1 day')::date AS d 3), 4log_bounds AS ( 5  SELECT d::timestamptz AS start_ts, 6         (d + INTERVAL '1 day')::timestamptz AS end_ts FROM y 7) 8SELECT 9  (SELECT COUNT(*) FROM conspectus_review_logs, log_bounds10     WHERE owner_client_uuid = :owner11       AND reviewed_at >= start_ts AND reviewed_at < end_ts) AS reviewed,12  (SELECT COUNT(*) FILTER (WHERE tag = 'easy')13     FROM conspectus_review_logs, log_bounds14     WHERE owner_client_uuid = :owner15       AND reviewed_at >= start_ts AND reviewed_at < end_ts) AS easy_cnt,16  (SELECT COUNT(*) FROM conspectus_schedules, log_bounds17     WHERE owner_client_uuid = :owner AND is_row_invalid = 018       AND next_review_at >= start_ts19       AND next_review_at <  end_ts) AS still_due_yesterday;20-- target = reviewed + still_due_yesterday21-- accuracy_pct = ROUND(easy_cnt * 100 / NULLIF(reviewed, 0))22-- missed = GREATEST(0, target - reviewed)

§13Edge cases checklist

  • No reviews yesterday. All zero; accuracy_pct = 0 (guard against divide-by-zero).
  • Reviewed exceeds target. Learner caught up on prior back-log; missed = 0 per the floor.
  • Soft-deleted schedule row. Excluded from the still-due proxy via is_row_invalid = 0.
  • Rounding boundary. 71.43 % → 71; 71.5 % → 72 (Python round() semantics — banker's rounding is not asserted).
  • Date field. Serialised as YYYY-MM-DD; matches the yesterday-window date, always one UTC day behind computed_at.
  • Idempotency-Key sent. Silently ignored.

§15Test plan

  • Unit: accuracy formula (zero-reviewed, all-easy, mixed rounding); missed floor.
  • Integration: AC-001..AC-008; assert soft-deleted rows excluded from still-due.
  • Contract: Schemathesis on getMeYesterday.
  • End-to-end: seed a review-log window straddling midnight → yesterday's count excludes today's rows.

§17Open questions

  • Q1. Should target reflect the historical due-list (replay of schedules at yesterday's start) instead of the window-count proxy? v1: no — proxy is cheaper and «good enough» for the recap card. Revisit if the value visibly under-counts.
  • Q2. Learner-timezone-aware boundaries? v1: UTC-only. Same open question as getMeStats.

§18Out of scope

  • Per-slot breakdown of yesterday's reviews.
  • Time-of-day distribution (belongs to analytics).
  • Arbitrary calendar-day recap (only «yesterday» in v1).
