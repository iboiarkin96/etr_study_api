Me resource

Learner-scoped aggregates over conspectus_review_logs: read-only projections that power the Today-hero of the Telegram Mini App (streak orb, yesterday recap). Mount: /api/v1/me. Owner: Ivan Boyarkin. Last reviewed 2026-07-19.

Overview

Me is a computed projection, not a writable resource. All endpoints read from conspectus_review_logs (with a small helper join into conspectus_schedules for the «still-due» count on yesterday, and owner-scoped COUNTs over conspectuses + learning_errors for achievements) and return an aggregated view scoped to the caller. The client never writes here — writes happen through reviewConspectus, which appends the log rows this resource projects over.

Two operations serve the two hero blocks on the Today screen:

  • Stats — current + longest streak of consecutive review days for the streak orb.
  • Yesterday — recap of the previous UTC calendar day (reviewed / target / accuracy / missed) for the recap card.

Business context

The Telegram Mini App opens on a Today hero that has to render in one screen with one API round-trip per block. Two questions drive the visible copy above the fold, and each one gets its own cheap endpoint instead of an over-fetched «everything» payload:

  • «How long is my streak?» — getMeStats. Feeds the streak orb; called on every Today open. 24-hour grace so opening the app before today's first review doesn't visually break the streak.
  • «How did I do yesterday?» — getMeYesterday. Feeds the recap card. Client renders reviewed / target, an accuracy chip, and a «missed» pill when non-zero.

Methodology anchor: methodology · retrieve cadence. Streak days are computed from the review log (append-only, one row per reviewConspectus call); yesterday's target is inferred from conspectus_schedules. The daily heat-map that sits alongside the Today hero is served by getScheduleHistory — same aggregation, hosted on the Schedule mount because it echoes the schedule cadence.

Domain model

Source tables: conspectus_review_logs (append-only fact — one row per completed review) and conspectus_schedules (one row per active conspectus, joined for the «still-due» proxy on yesterday). Both endpoints project over the review log; canonical column references live on the Conspectus resource page.

Lifecycle

Me is read-only: the client never posts here. State transitions happen on the Conspectus resource; Me endpoints observe the resulting review-log fact.

  • Row creation. A fresh conspectus_review_logs row is appended by reviewConspectus in the same transaction as the schedule advance.
  • Immutable log. Review logs are append-only — there is no update or delete path. Streak / yesterday counts always reflect the full history.
  • Soft delete of a note. deleteConspectus flips is_row_invalid = 1 on the schedule row — the «still-due» proxy on yesterday filters it out. Review-log rows are retained.
  • Reads. getMeStats, getMeYesterday, and getMeAchievements. No Idempotency-Key.

Decision · 24-hour grace on the streak walk

Why the streak counts backward from yesterday when today is empty. A learner who opens the app at 08:00 UTC before today's first review would otherwise see the streak reset to zero — a jarring, incorrect signal. The walk starts from today when today has ≥1 review; otherwise it starts from yesterday. If yesterday is also zero, the streak is zero.

Longest is monotonically ≥ current. Because the walk can start yesterday, a fresh current run can momentarily match longest even if a longer historical run exists; the service coerces longest_days = max(longest_days, current_days) to keep the pair sane.

Goal is fixed at 30. Client renders current_days / goal_days as an orb progress arc. Product may swap the constant in future via a repo-level setting; the field is present for that path.

Technical overview

Both endpoints share the same access pattern: resolve owner_client_uuid from the query composite key, run one indexed scan on conspectus_review_logs (plus one extra count on conspectus_schedules for yesterday), compute the projection in memory, return. No writes, no idempotency middleware. The composite index (owner_client_uuid, reviewed_at) covers streak's per-day grouping and yesterday's window filter.
