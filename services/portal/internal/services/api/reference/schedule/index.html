Schedule resource

Schedule projection over conspectus_schedules: two read-only endpoints that turn the underlying schedule rows into presentation-ready views for the client. Mount: /api/v1/schedule. Owner: Ivan Boyarkin. Last reviewed 2026-07-09.

Overview

Schedule is a computed projection, not a writable resource. Both endpoints read from conspectus_schedules and return an aggregated or shuffled view of it — the client never writes here. Writes happen through reviewConspectus, which is the only surface that mutates a schedule row.

Two operations serve two distinct client flows:

  • Summary — a compact count-per-slot payload for the study-load badge (rendered on every screen).
  • Preview — a server-shuffled, deterministic lookahead for the evening-review session.

Business context

The methodology's retrieve step needs two answers from the schedule surface, and giving the client each one via a specialised endpoint costs less than shipping the full due-list every time:

  • «How much is on my plate right now?» — getScheduleSummary. Cheap enough to poll on every UI mount; four small integers per response.
  • «Give me an evening batch, mixed so I don't cram same-slot» — getSchedulePreview. Server owns the shuffle so the order is reproducible for support and consistent across the learner's devices.

Methodology anchor: methodology · retrieve cadence. Schedule state is written by reviewConspectus (state transitions per the reference SchedulePolicy) and by createConspectus (initial slot A · revision 1 · +1h next review).

Storage table: conspectus_schedules (one row per active conspectus, joined on conspectus_uuid). This hub summarises the columns that both Schedule endpoints project on; canonical column reference lives on the Conspectus resource page (Schedule is a projection, not a distinct storage entity).

Lifecycle

Schedule is read-only: the client never posts here. State transitions happen on the Conspectus resource; Schedule endpoints observe the resulting view.

  • Row creation. A fresh conspectus_schedules row is inserted by createConspectus in the same transaction as the conspectus.
  • State transition. reviewConspectus advances slot, next_review_at, and schedule_revision per the reference SchedulePolicy.
  • Soft delete. deleteConspectus flips is_row_invalid = 1 on the parent conspectus — Schedule reads filter it out. The schedule row itself is not modified.
  • Reads. getScheduleSummary and getSchedulePreview. No Idempotency-Key.

Decision · Server-side shuffle on getSchedulePreview

Why the server owns the shuffle. A client-side shuffle spreads the same responsibility across every integrator — mobile app, web app, CLI, future partners — and each implementation carries the risk of a biased Fisher-Yates or a non-seeded PRNG. Server-side keeps the algorithm in one place, protects the learner from same-slot cramming even on brand-new clients, and lets support engineers reproduce a specific evening session by replaying the seed.

Determinism boundary. Same (random_seed, computed_at.minute) → identical order. The client's session anchor is the echoed random_seed; mid-session refresh preserves order for up to 60 seconds. Beyond that the minute rolls over and the order can change.

Payload shape. Compact projection (five fields: preview_order_index, conspectus_uuid, title, slot, next_review_at) — the client calls getConspectusById on tap for the full content.

Technical overview

Both endpoints share the same access pattern: resolve owner_client_uuid from the query composite key, run one indexed scan on conspectus_schedules, compute the projection in memory, return. No writes, no idempotency middleware. The composite index (owner_client_uuid, next_review_at) covers Summary's grouping and Preview's window filter.

Specification only. Canon fragments are ratified (x-implementation-status: pending); FastAPI handlers land alongside the Conspectus review-loop implementation because the two share the same schedule state.
