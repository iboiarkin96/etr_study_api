Overview

One row per UTC calendar day for the last days days (default 90), zero-filled for inactive days. Each row carries a count of that day's reviews and an intensity bucket in 0..4 that maps to the Ember-tinted heat-map cells on the Today screen.

§1Business context

Feeds the 90-day heat-map that sits underneath the Today hero blocks on the Telegram Mini App. The intensity bucket is server-owned so every client renders the same tint scale (0=none · 1 <3 · 2 <7 · 3 <12 · 4 ≥12). Placed on the Schedule mount because it echoes the schedule cadence — the Me mount owns learner-scoped hero blocks, but the heat-map is a schedule-shaped view over the review log.

§2User stories

  • US-SCHED-HIST-1. As a learner, I want to see my last 90 days as a heat-map so I can spot patterns and gaps.
  • US-SCHED-HIST-2. As an integrator, I want the intensity bucket pre-computed by the server so every client renders the same colour ramp.
  • US-SCHED-HIST-3. As a learner, I want inactive days to be present in the payload (as count = 0) so the client doesn't have to fill gaps.

Equivalent raw SQL

per-day counts + intensity SQL 
 1SELECT day::date AS day, 2       COALESCE(n, 0) AS count, 3       CASE 4         WHEN COALESCE(n, 0) = 0 THEN 0 5         WHEN n < 3  THEN 1 6         WHEN n < 7  THEN 2 7         WHEN n < 12 THEN 3 8         ELSE 4 9       END AS intensity10FROM generate_series(CURRENT_DATE - (:days - 1), CURRENT_DATE, '1 day') AS day11LEFT JOIN (12  SELECT DATE(reviewed_at) AS day, COUNT(*) AS n13  FROM conspectus_review_logs14  WHERE owner_client_uuid = :owner15  GROUP BY 116) logs USING (day)17ORDER BY day;

§13Edge cases checklist

  • All zero days. 90 rows, all count = 0, all intensity = 0. Client renders a full grey grid.
  • Bucket boundaries. count = 3 → intensity 2; count = 7 → intensity 3; count = 12 → intensity 4. Strictly-less thresholds.
  • Timezone. Day boundaries are UTC midnight; a review at 23:59 local UTC+3 lands in the next UTC day.
  • Custom days=1. Only today's row is returned; useful for smoke tests.
  • Custom days=365. Full year of rows; still one indexed scan.
  • Idempotency-Key sent. Silently ignored.

§15Test plan

  • Unit: _bucket_intensity (each boundary), zero-fill loop (materialises exactly days rows).
  • Integration: AC-001..AC-007; assert row order is ascending.
  • Contract: Schemathesis on getScheduleHistory.
  • End-to-end: seed 15 reviews on a single day → that day's row is count=15, intensity=4.

§17Open questions

  • Q1. Should bucket thresholds be per-learner (adaptive to their historical density)? v1: no — global constants keep the heat-map comparable across screens.
  • Q2. Per-slot breakdown of the daily count? v1: no — one scalar per day is enough for the heat-map cell.

§18Out of scope

  • Arbitrary date-range windows (only trailing days from today).
  • Timezone-aware day boundaries.
  • Streak information (belongs to getMeStats).

§19Traceability

  • System design: Telegram Mini App Today heat-map (T-15d).
  • ADRs: 0003, 0005.
  • OpenAPI: getScheduleHistory.
