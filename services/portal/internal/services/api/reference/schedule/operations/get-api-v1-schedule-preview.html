Overview

Return conspectuses due within a bounded window, server-shuffled. Intended for the «evening review» client flow — a presentation-ready lookahead that doesn't require the client to fetch the full due-list. Each item is stamped with a preview_order_index so client rendering is deterministic across a mid-session refresh.

§1Business context

Randomization of the review order matters — sequential same-slot practice induces cadence bias, so a shuffled queue is pedagogically stronger. The decision was where to do it. Server-side wins on three points: (1) one source of truth, no per-client shuffle bugs, (2) the client only receives the window it needs, small privacy win, (3) deterministic replay via a seed lets support engineers reproduce a specific evening session.

§2User stories

  • US-SCHED-PREV-1. As a learner, I want an «evening review» surface that gives me the next 4 hours of due notes in a mixed order.
  • US-SCHED-PREV-2. As a learner, I want to refresh mid-session without the order jumping around.
  • US-SCHED-PREV-3. As a support engineer, I want to reproduce a specific learner's evening session by replaying the seed.
  • US-SCHED-PREV-4. As an integrator, I want a compact preview payload (uuid + slot + title + next_review_at) — I'll getConspectusById for the render.

§10Side effects

None. Read-only. Shuffle happens in-process.

§11Idempotency & concurrency

Read-only + deterministic within a 60-second bucket. Concurrent calls with the same seed within the bucket return identical bodies. Across buckets the order can differ because computed_at minute is part of the shuffle seed.

§13Edge cases checklist

  • Empty window. 200 OK; items = []; random_seed still echoed.
  • Seed omitted. Server generates a UUIDv4-derived slug ([A-Za-z0-9_-]{22}) and echoes it.
  • Item soft-deleted between preview calls. Second call excludes it; first call's preview_order_index stays but that item is gone.
  • Window crosses DST boundary. Server uses UTC; DST is a client display concern.
  • P1D vs PT24H. Treated identically in v1 — both are «one day of lookahead»; keep both for ergonomics.
  • Clock skew. computed_at comes from DB, not Python.
  • Same-timestamp ties. Broken by conspectus_uuid ASC before shuffle, so shuffle is over a stable ordering.

§15Test plan

  • Unit: Fisher-Yates unbiased; seed → identical order; seed omitted → server-generated slug matches pattern.
  • Integration: AC-001..AC-009.
  • Contract: Schemathesis on getSchedulePreview; assert preview_order_index is a permutation of 0..N-1.
  • End-to-end: create 12 items due within PT4H → call twice with same seed → identical order; call again with new seed → same set, different order.
  • Property: for random seeds S₁ ≠ S₂ and the same due set, P(same permutation) ≤ 1/N! (over 10 000 pairs).

§17Open questions

  • Q1. Should we weight the shuffle so slot-A items come first (soft priority)? v1: pure random. Revisit if learners report «I always burn energy on new-material first».
  • Q2. Should the seed be namespaced per learner-day so a single seed produces the same order across devices? v1: yes — the seed alone is enough because computed_at minute is factored in; but «per learner-day» convenience wrapper could be a client concern.
  • Q3. Do we expose the shuffle algorithm (Fisher-Yates, seeded)? v1: yes — documented here so support engineers can reproduce.

§18Out of scope

  • Custom windows (arbitrary PT2H, etc.).
  • Weighted / policy-driven ordering (Q1).
  • Server-side session state — the client owns whether the learner has «finished» the preview.
  • Push notification of new due items — polling only in v1.
