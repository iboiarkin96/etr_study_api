OpenAPI fragment anatomy

Every key you can write inside one of our OpenAPI 3.1 fragments — what it does, whether it's required, and how it surfaces in the Swagger UI preview (which defaults to the combined view of every fragment, or you can pin to one via ?spec=fragments/course/createCourse.yaml). Scoped to what we actually use in this repo's fragments; for the full OpenAPI 3.1 spec see spec.openapis.org/oas/v3.1.0.

Scope & how to read this page

This is a lookup reference for one fragment file — the unit governed by ADR 0036 at services/portal/internal/services/api/openapi/test/fragments/{resource}/{operationId}.yaml. A fragment is a self-contained OpenAPI 3.1 document that describes one operation. Cross-cutting policy (versioning, public vs. internal split, error governance) lives in the related ADRs and in the public-translation guide; this page is about the YAML shape.

  • Required? column refers to the OpenAPI 3.1 spec. The Project rule column adds our local conventions on top (some optional keys are mandatory for us; some optional keys we never use).
  • Renders as column tells you what Swagger UI shows when the key is present — read it side-by-side with preview.html open on createCourse.yaml.
  • Tables list only keys we actually use. If a key isn't here, treat it as «not for this project» until an ADR adds it.

1 · Document skeleton

The five top-level keys of a fragment. openapi, info, and paths are required by the spec; we additionally require components on any fragment that owns local schemas.

KeyRequired?TypePurposeProject rule
openapiYesstringSpec version. Determines which keywords are legal.Always 3.1.0. We don't author 3.0 fragments.
infoYesInfo ObjectDocument metadata: title, version, contact, license.Title Study App API, version v1. No termsOfService, no contact — those belong to the public spec, not per-fragment.
serversNoarray of ServerBase URLs for the API. If absent, Swagger UI uses the same origin as the spec file.Never set per-fragment. Server config is a runtime concern.
pathsYesPaths ObjectOne key per URL template, e.g. /api/v1/courses. Each value is a Path Item Object with one method block.Exactly one path per fragment. Mixing operations from different paths in one file breaks the «one fragment per operation» rule.
componentsNo (spec)Components ObjectReusable building blocks — schemas, parameters, responses, headers, examples. Referenced from anywhere in the document via #/components/….Required when the fragment owns request/response schemas (it usually does). Shared building blocks live in _shared/, not in per-fragment components.
tagsNoarray of TagTop-level tag metadata (descriptions, external docs). Distinct from the tags: list inside an Operation.Never set top-level. Tag descriptions live in the merged public spec, not in per-operation fragments.
securityNoarray of Security RequirementDefault security schemes for every operation in the document.Not used today. When auth lands, will be declared per-operation, not at document level.

Renders in Swagger UI as — the page header (info.title + info.version), one collapsible block per paths entry, and a «Schemas» section at the bottom built from components.schemas.

2 · Operation Object — paths.{path}.{method}

The body of post:, get:, put:, patch:, delete:, head:, or options: inside a Path Item. This is where most of the contract lives.

KeyRequired?TypePurposeRenders asProject rule
operationIdNo (spec) / Yes (us)stringUnique identifier for the operation. SDK generators use it as the method name.Shown in the operation header as a grey badge; also used as the anchor in deep links.Required. {verb}{Resource} in camelCase (createCourse, listCourses, getCourse). Unique across the whole API. Stable forever.
summaryNostring (≤120 chars)One-line description. The «title» of the operation.The bold text next to the method pill (POST/GET/…) in the collapsed block header.Required. Imperative mood, ends with a period. «Create a new course.»
descriptionNostring (CommonMark)Long-form description. Supports Markdown.Markdown-rendered block right under the operation header when expanded.Optional. Use it when the operation has non-obvious semantics (idempotency window, side effects, ordering guarantees). Skip when summary + acceptance criteria say enough.
tagsNoarray of stringGrouping. Swagger UI clusters operations by their first tag.Section grouping in the side index; pill in the operation header.Required. Exactly one tag — the resource's PascalCase name (Course, User, Conspectus).
parametersNoarray of Parameter/ReferencePath / query / header / cookie inputs.«Parameters» table when expanded. Required ones marked with a red required badge.Use $ref to _shared/headers/* for cross-operation headers (Idempotency-Key). Inline only operation-specific params.
requestBodyNoRequestBody/ReferencePayload for POST/PUT/PATCH.«Request body» panel with a media-type dropdown, schema view, and example payload.Required on every write. Always one media type: application/json. Always at least one examples entry.
responsesYesResponses ObjectMap status code → Response Object.«Responses» table with one row per status code; click a row to expand the body.Document every status the handler can return — success, validation errors (400), idempotency conflicts (409), authentication (401 when auth lands). Use $ref to _shared/responses/* for envelope reuse.
deprecatedNobooleanMarks the operation as deprecated.Strikethrough on the operation summary; warning banner.Use only after a deprecation ADR. Set to true, point description at the successor operation and the deprecation date.
securityNoarray of Security RequirementPer-operation security overrides.Lock icon next to the operation; security scheme listed in the «Authorize» dialog.Not used yet — set per-operation once auth lands, not at document level.
callbacksNomap of CallbackWebhook contracts.«Callbacks» panel.Not used. We don't ship webhooks.
serversNoarray of ServerPer-operation base URL override.Server dropdown.Not used.
externalDocsNoExternal DocumentationLink out to extended docs.«Find more info» link in the operation header.Not used at operation level — the portal Operation spec page is the external doc.

3 · Parameter Object

One entry per element of an Operation's parameters: array. Reusable parameters (e.g. Idempotency-Key) live in _shared/headers/ and are pulled in via $ref.

KeyRequired?TypePurposeRenders asProject rule
nameYesstringParameter name as it appears on the wire.The «Name» column in the Parameters table.For in: path, must match the placeholder in the path (/{course_id} ↔ name: course_id). For headers and queries, follow our naming convention (snake_case for query, Title-Case-Hyphenated for HTTP headers).
inYesenumWhere the parameter lives. One of path / query / header / cookie.«Located in» column.We use path, query, header. No cookies.
requiredYes for in: path, else NobooleanWhether the parameter must be provided.Red required badge in the Parameters table.Always set explicitly, even when false, so the contract is unambiguous on review.
descriptionNostring (CommonMark)What the parameter means.Markdown-rendered description column.Required for every parameter. Mention units, valid ranges, and any cross-references (e.g. «UUID of the course, see GET /api/v1/courses»).
schemaYes (in practice)Schema ObjectType and constraints of the value.Type column («string», «integer», …) plus the format / pattern in the expanded view.Always inline (small) or $ref to a shared schema (_shared/schemas/Uuid.yaml, etc.). Constrain with pattern / minLength / maxLength when the underlying type allows.
exampleNoany (matches schema)One representative value.Pre-fills the «Try it out» form with this value.Required on every required: true parameter. Optional otherwise.
examplesNomap of ExampleMultiple named examples — useful when the same parameter has distinct shapes per use case.Dropdown above the «Try it out» form.Use when example isn't enough (e.g. a filter that takes either a UUID or a slug).
deprecatedNobooleanMarks the parameter as deprecated.Strikethrough on the parameter name.Same rule as Operation-level deprecated — only after an ADR.

Header-parameter pattern (lives at _shared/headers/IdempotencyKey.yaml and is referenced by every write operation):

name: Idempotency-Key
in: header
required: true
description: >
  Caller-supplied opaque key (UUIDv4 recommended). Replays the cached
  response for the configured idempotency window. See ADR-0006.
schema:
  type: string
  minLength: 8
  maxLength: 128
  pattern: "^[A-Za-z0-9_-]+$"
example: "8f5a2c19-3b4e-4d8a-9f3b-1c2d3e4f5a6b"

4 · Request Body Object

The body of an Operation's requestBody:. Only used on POST / PUT / PATCH operations.

KeyRequired?TypePurposeRenders asProject rule
descriptionNostring (CommonMark)One-paragraph description of the payload.Text above the schema view.Optional. Use when the payload's invariants (e.g. «exactly one of a or b») aren't expressible in the schema.
requiredNo (default false)booleanWhether the body must be present.Red required badge on the «Request body» panel.Always true for POST/PUT/PATCH on this project. We don't author write ops that accept an empty body.
contentYesmap of Media TypeMedia-type → MediaType Object. The key is the MIME type (application/json, multipart/form-data, …).Media-type dropdown above the schema view.Exactly one entry: application/json. Multipart is out of scope until a file-upload operation lands (separate ADR).

5 · Response Object

One entry per status code under an Operation's responses:. Reusable error envelopes live in _shared/responses/ and are pulled in via $ref.

KeyRequired?TypePurposeRenders asProject rule
descriptionYesstring (CommonMark)What this response means — when the handler returns it, what state it implies.The expanded row's primary description.One sentence, present tense («Created.» «Conflict — operation conflicts with current state.»).
headersNomap of HeaderResponse headers (e.g. Location, ETag).«Headers» subsection inside the response row.Document any header the client must read (e.g. Location on a 201 that returns a redirect). Use $ref for cross-operation headers.
contentNomap of Media TypeThe body shape per media type.Media-type dropdown above the body schema + example.Required for 2xx and 4xx responses. Omit for 204 No Content.
linksNomap of LinkHATEOAS — declares how a returned value can drive a subsequent operation.«Links» subsection.Not used today. Add once the first cross-operation relationship needs it.

Shared error responses (under _shared/responses/) are full Response Objects — they ship description + content together. That's why a fragment can pull one in with one line: "400": { $ref: "../../_shared/responses/Error400.yaml" }.

Sharing tiers — where the 4xx response lives

OpenAPI's $ref is a full substitution — it replaces the entire Response Object. You can't $ref a shared response and then «add one more example» to it. So we have three tiers, ordered cross-resource → per-resource → per-operation, and pick per status code per operation:

TierLives atUse when
A · Cross-resource shared openapi/test/_shared/responses/Error{NNN}.yaml The 4xx description is generic and identical across every resource — «request failed validation» with a non-specific example.
B · Per-resource shared openapi/test/fragments/{resource}/error-{NNN}/{specifics}.yaml Multiple operations on the SAME resource raise the same 4xx with the same semantics. createCourse and a future updateCourse both check «course name already exists» — the response file lives next to them, ready to be $ref'd from both.
C · Inline per-operation Inside the fragment itself The 4xx is unique to THIS operation: the examples name its specific fields, the description mentions its specific rules.

Tier A — generic cross-resource:

responses:
  "400": { $ref: "../../_shared/responses/Error400.yaml" }

Tier B — per-resource (the file at fragments/course/error-409/name-conflict.yaml is a Response Object with course-specific examples; both POST and a future PATCH on the same resource reuse it):

responses:
  "409": { $ref: "./error-409/name-conflict.yaml" }

Tier C — inline per-operation (used in createCourse.yaml for the 400 because the validation messages name name and length, which only exist on this operation):

responses:
  "400":
    description: Bad Request — request body failed schema validation.
    content:
      application/json:
        schema: { $ref: "../../_shared/schemas/ErrorBody.yaml" }
        examples:
          missing_name:
            summary: "name is missing"
            value:
              code: validation_error
              message: "Field 'name' must not be empty."
          length_out_of_range:
            summary: "length not in [1..1000]"
            value:
              code: validation_error
              message: "Field 'length' must be between 1 and 1000."

All three tiers reuse the same envelope schema (_shared/schemas/ErrorBody.yaml) — only the wrapping Response Object and its examples change.

How to pick.

  1. Is this 4xx specific to this resource's domain (e.g. «course name conflict»)? → tier B, sibling under {resource}/error-{NNN}/. Other operations on the same resource will reuse it.
  2. Are the examples truly unique to this one operation (named after its specific fields)? → tier C, inline.
  3. Is the 4xx genuinely generic and identical across resources? → tier A, cross-resource $ref.

Anti-pattern. Putting course-specific messages into _shared/responses/Error409.yaml. The cross-resource file should stay generic; the moment a message names a field, that's the cue to migrate to tier B or C.

How the validator treats partials. Tier-B files live under fragments/ but are Response Objects, not full OpenAPI 3.1 documents. The validator detects them (no top-level openapi: key) and skips them with a ↺ Partial note — the partial gets validated transitively when its parent fragment's $ref walk pulls it in.

6 · Media Type Object

The body shape for one media type, inside either a Request Body or a Response.

KeyRequired?TypePurposeRenders asProject rule
schemaYesSchema/ReferenceThe body's type and constraints.Schema view in the panel.Always $ref to a named schema — either local (#/components/schemas/Foo) or shared (../../_shared/schemas/Foo.yaml). Inline schemas only when the body is trivial (single primitive).
exampleNoany (matches schema)One representative body.Pre-fills the «Try it out» form and shows on the request-body / response-body tab.Use examples instead — even when there's only one, named examples make a clearer review.
examplesNomap of ExampleNamed examples.Dropdown on the panel labelled with each example's name.Required on every documented response status and request body. At least one named example. Example keys are snake_case (minimal, with_optional, duplicate_name).
encodingNomap of EncodingPer-property encoding rules (for multipart/form-data and friends).«Encoding» subsection.Not used — we're JSON-only.

Example Object inside examples has four keys: summary (short label), description (longer), value (the actual example), and externalValue (a URL — never used). We almost always use only value: { value: { name: "Introduction to Python" } }.

7 · Schema Object

JSON Schema 2020-12 with a small OpenAPI delta. Used in requestBody, response content, parameter schema, and components/schemas. We only use a subset — listed below.

Core keys

KeyTypePurposeProject rule
typestring or arrayJSON type: string, integer, number, boolean, object, array, null.Always set, even when the schema also has $ref children. Nullable fields use type: [string, "null"] (3.1 style — never the deprecated nullable: true).
formatstringHint about the concrete representation: date-time, uuid, email, uri, int64, etc.Use format: uuid on UUID fields; format: date-time on RFC-3339 timestamps. Never invent custom formats — link to a shared schema instead.
$refURI referenceReplace this schema with the referenced one.Always quoted: $ref: "../../_shared/schemas/Foo.yaml". Paths are relative to the fragment's directory; the validator resolves them via its base URI.
descriptionstring (CommonMark)What this field means.Required on every field exposed in a request or response. Required on every named schema in components.
examplesarraySample values for the type. Distinct from the Media-Type-level examples.Used inside _shared/schemas/* to make the «Schemas» bottom panel readable. Optional inside per-operation fragments — the operation's examples map is the primary source.
defaultanyValue the server uses when the client omits the field.Document defaults explicitly. A missing default on a non-required field means «the absence is meaningful» — make sure that's intentional.
enumarrayClosed set of allowed values.Pull frequently-reused enums into _shared/schemas/ rather than redefining inline.

Object schemas

KeyTypePurposeProject rule
propertiesmap of SchemaProperty name → schema.Property names are snake_case. Always declare every documented property; never rely on additionalProperties.
requiredarray of stringProperty names that must appear.Always present (even when empty []) for any object schema. Reviewers need it to read the contract at a glance.
additionalPropertiesboolean or SchemaWhether/which extra properties are allowed.Default to omitting (which means «allowed»). Set to false when the schema must be closed (e.g. a write body that mustn't accept stray fields).
minProperties / maxPropertiesintegerProperty-count bounds.Rare. Use only when the cardinality has business meaning.

String constraints

KeyPurposeProject rule
minLength / maxLengthLength bounds.Always set on user-supplied strings. Trace the upper bound to a real storage / UI limit.
patternJS-flavoured regex.Use for IDs, slugs, opaque keys. Anchor with ^…$. Test against the example: the example must match the pattern.
formatSemantic hint.Combine with pattern when format alone isn't enough (Swagger UI shows both; clients honour the pattern).

Number / integer constraints

KeyPurposeProject rule
minimum / maximumInclusive numeric bounds.Set on every numeric field that has a business range.
exclusiveMinimum / exclusiveMaximumStrict numeric bounds (3.1 — boolean style).Use when zero is meaningful and you need «strictly positive».
multipleOfStep size.Used on quantised values (e.g. minute-resolution timestamps).

Array schemas

KeyPurposeProject rule
itemsSchema of each element.Always set on arrays — never use untyped arrays.
minItems / maxItemsCardinality bounds.Set maxItems on every user-supplied array. Set minItems: 1 when an empty array is meaningless.
uniqueItemsWhether duplicates are allowed.Set to true for set-like arrays (tag lists, IDs); skip for ordered logs.

Composition

OpenAPI inherits all four JSON Schema combinators: allOf (intersection), oneOf (xor), anyOf (union), not (negation). We use them sparingly:

  • allOf — to extend a base schema. Example: a paginated-response wrapper that allOf-merges a generic envelope with the concrete items schema.
  • oneOf — to express «exactly one of these shapes». Pair with discriminator when the shapes have a tag field that picks between them.
  • anyOf — only when an input can validly match more than one branch. Rare; usually oneOf is what you actually want.
  • not — not used.

8 · $ref resolution

Every $ref is a URI. The validator (tools/governance/validate_openapi.py) resolves it against the fragment's own file URL. Concretely:

  • Local refs like #/components/schemas/CreateCourseRequest resolve inside the same fragment. The fragment is parsed once; refs are looked up against the parsed tree.
  • Relative file refs like ../../_shared/responses/Error400.yaml resolve as filesystem paths from the fragment's directory. From fragments/course/createCourse.yaml, two levels up lands on openapi/, where _shared/ lives.
  • URL refs (https://…) are technically valid but not used here — we want the canon to live in the repo.

Count ../ by directory depth from the fragment:

Fragment pathTo _shared/headers/X.yamlTo _shared/responses/X.yamlTo _shared/schemas/X.yaml
fragments/course/createCourse.yaml"../../_shared/headers/X.yaml""../../_shared/responses/X.yaml""../../_shared/schemas/X.yaml"
fragments/course/subdir/foo.yaml (hypothetical nested resource)"../../../_shared/headers/X.yaml""../../../_shared/responses/X.yaml""../../../_shared/schemas/X.yaml"

If the validator says Unresolvable: ../../_shared/responses/Error400.yaml, the path is wrong — count again. The Swagger UI preview honours the same relative resolution, so a passing validator means a working preview.

9 · Project-specific rules in one place

Hard rules that every fragment must satisfy. Reviewers check these before everything else; the validator catches most of them automatically.

  1. OpenAPI version 3.1.0. Period.
  2. Exactly one path under paths:. Exactly one method under that path. One fragment = one operation.
  3. operationId is required and follows {verb}{Resource} camelCase.
  4. summary is required (≤120 chars, imperative mood, ends with a period).
  5. tags is required, exactly one entry, PascalCase resource name.
  6. Write operations (post, put, patch, delete) include the Idempotency-Key header via $ref to _shared/headers/IdempotencyKey.yaml.
  7. Every documented response code has content (except 204) and at least one named example.
  8. Every 4xx and 5xx response uses a shared envelope from _shared/responses/. Never inline an error body.
  9. Every request-body and response schema is a $ref (local or shared) — no anonymous inline schemas in content.schema.
  10. Object schemas always declare required: [...] explicitly, even when empty.
  11. Property names are snake_case. operationId is camelCase. Schema names are PascalCase. Tag names are PascalCase.
  12. The validator (make doc-api-swagger-check FILE={resource}/{operationId}.yaml) exits 0 before the spec PR opens.
  13. The Swagger UI preview (preview.html?spec=fragments/{resource}/{operationId}.yaml) renders the operation with every documented response code and example.

10 · Quick map — «where will this key show up in Swagger UI?»

Open the Swagger UI preview (combined view) alongside this list:

  • info.title + info.version → page header at the very top.
  • Each tags entry → collapsible section grouping operations.
  • Inside a tag: one block per operation. Block header = method pill (POST) + summary + grey operationId badge.
  • Click the block → description, then «Parameters» table, then «Request body» panel, then «Responses» table.
  • Parameters table: one row per parameters[] entry — name (with required badge), description, schema preview, example.
  • Request body panel: media-type dropdown → schema view + named examples dropdown.
  • Responses table: one row per status code → click to expand → description, schema view, named examples.
  • Bottom of the page: «Schemas» section listing every entry in components.schemas (and, after $ref resolution, the shared schemas too).
