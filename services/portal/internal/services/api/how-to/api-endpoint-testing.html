API How-to

Verify an API endpoint against its contract

How to verify any HTTP operation under /api/v1/… against its internal specification. The playbook is the procedure; the line-by-line outcomes go in checklist 0002.

Where to find the contract for an endpoint

Each operation has three artefacts you must keep in sync while testing:

  1. The internal specification page — services/portal/internal/api/<resource>/operations/<method>-…html. The oracle for behavior.
  2. The OpenAPI document — served at /openapi.json and rendered as static Swagger UI in the public developer portal. The oracle for shape.
  3. Cross-cutting standards under _shared/ — error envelope, error catalog, auth, idempotency, pagination, versioning, observability, field conventions.

If the spec, OpenAPI, and behavior disagree, file a defect against whichever is wrong (the spec is usually the contract). Don't paper over the disagreement.

Workflow at a glance

  1. Open the spec page for the operation.
  2. Run the contract tests for that resource: pytest tests/api/v1/<resource>/.
  3. Open the static Swagger UI in the public developer portal and confirm the operation appears with the expected schema.
  4. Run the steps in happy path, error paths, auth, idempotency, observability from this playbook.
  5. Annotate checklist 0002 with pass/fail per row, attach evidence.
  6. If anything fails, file a defect using bug template.

Verify the contract (schema)

  • Path, method, version segment match the spec exactly.
  • Request body schema matches — required, types, formats, enum values, length limits.
  • Response body schema matches for every documented status — 200/201, 4xx, 5xx.
  • OpenAPI baseline test (tests/openapi/) is green; if you changed the contract, it must be regenerated.
  • Field conventions — UUIDs, timestamps in UTC ISO-8601, snake_case keys, enum values lowercase. See field conventions.

Happy path

  • Smallest valid body produces success status and well-formed response.
  • All optional fields, when supplied, round-trip and appear in the response unchanged.
  • Pagination (where applicable) — first page, next page using cursor, last page returns empty next_cursor. See pagination.
  • Mutations are visible on a follow-up GET (read-after-write).

Error paths — what every operation must do correctly

Every error response must conform to the error envelope and the specific code must exist in the error catalog.

ClassTriggerExpected
ValidationMissing required field, wrong type, value out of range, oversize string.422 + envelope + field_errors for each offender.
Not foundUUID for a resource that does not exist.404 + envelope + catalog code.
ConflictState transition not allowed; unique constraint.409 + envelope + catalog code.
UnauthenticatedMissing/invalid auth.401 + envelope; never 500.
ForbiddenAuthenticated, lacks permission.403 + envelope.
Method not allowedWrong method on the path.405 with Allow header.
Unsupported mediaContent-Type wrong or absent on a body-bearing method.415 + envelope.
Server errorForced (e.g. break a downstream).5xx + envelope; X-Request-ID traceable in logs.

Authentication and authorization

  • Confirm the operation requires the auth class declared by the spec (see auth).
  • No token → 401. Wrong scope/role → 403. Valid token → 2xx.
  • Token in wrong scheme (Basic vs Bearer) → 401, not 500.

Idempotency (write operations)

  • Send the same request twice with the same Idempotency-Key within the window — second call returns the same response without creating a duplicate.
  • Send same key with a different body — 409.
  • Resend after the key window expires — treated as a new request.
  • See idempotency for the policy.

Observability

  • Every response carries X-Request-ID (UUID).
  • Search the structured log for that X-Request-ID; one record per request, with method, path, status, latency_ms.
  • Sensitive headers and bodies are not logged in clear text — see observability conventions.
  • Health and readiness endpoints respond appropriately while load is in flight.

Evidence to capture

  • Full request and response (including headers) for one of each: success, validation error, auth error.
  • The X-Request-ID values used.
  • One log excerpt matching one of the request IDs.
  • Screenshot of Swagger UI with the operation expanded (verifies user-visible doc shape).

cURL examples to copy

# Happy path: POST with idempotency key
curl -sS -X POST "$BASE/api/v1/conspectuses" \
-H "Authorization: Bearer $TOKEN" \
-H "Content-Type: application/json" \
-H "Idempotency-Key: tc-001-$(date +%s)" \
-d '{"title":"Demo","content":"…"}' \
-i

# Validation error: missing required field
curl -sS -X POST "$BASE/api/v1/user" \
-H "Authorization: Bearer $TOKEN" \
-H "Content-Type: application/json" \
-d '{"first_name":"I","last_name":"B"}' \
-i

# Auth missing
curl -sS -X GET "$BASE/api/v1/conspectuses/due/" -i

# Pagination: follow next_cursor
curl -sS "$BASE/api/v1/errors/?limit=10" -H "Authorization: Bearer $TOKEN"
# then
curl -sS "$BASE/api/v1/errors/?limit=10&cursor=<next_cursor>" -H "Authorization: Bearer $TOKEN"

# Idempotency replay
KEY="tc-replay-$(date +%s)"
for i in 1 2; do
curl -sS -X POST "$BASE/api/v1/conspectuses" \
-H "Authorization: Bearer $TOKEN" \
-H "Content-Type: application/json" \
-H "Idempotency-Key: $KEY" \
-d '{"title":"Replay","content":"x"}' -i
      done

Set BASE and TOKEN in your shell. Keep captures in a folder named after the ticket — paste them into the defect or sign-off note as evidence.
