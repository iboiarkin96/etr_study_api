Event spec

Every PostHog event the client emits, with its trigger, its property shape, and whether it participates in a funnel. Adding a new event without a row here is a review block — orphan events poison analytics.

Conventions

  • Naming: <object>_<action_past_tense> — review_completed, error_logged, focus_session_ended. Never verb-first, never present-tense.
  • Properties: snake_case keys, primitive values only (string / number / boolean). No nested objects, no arrays.
  • PII: zero. Never send conspectus content, never send free-text errors, never send usernames. Server-generated UUIDs are OK.
  • Timing: emitted synchronously with the user action, not after the resulting API response. Optimistic UX means the analytics is optimistic too.

Event lifecycle

Adding, deprecating, and removing an event follows the same three-step rule as an API change:

  1. Add: row in this table → PostHog project has the event registered (property types validated) → code emits it → PR merges.
  2. Deprecate: mark row with deprecated 2026-MM-DD in the «Fires when» column; keep emitting for 30 days so dashboards migrate.
  3. Remove: after the deprecation window, delete the row and the emit site. Grep for orphans first.
