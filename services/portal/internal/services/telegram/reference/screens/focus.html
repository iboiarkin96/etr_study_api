Screen map: Focus

The SRS review flow — a chrome-free full-screen surface with one card at a time, tap to reveal, four grade buttons. This is the screen users spend the most seconds in per ADR 0038, so its DNA follows the «single tap surface deserves a moment» pick. Sibling of the design narrative; this is the machine-readable anatomy.

The whole screen is a state machine driven by useFocusSession. Every visible variant maps to exactly one phase; nothing else can be on screen.

Session-complete visual

The complete phase renders a SessionCompleteOrb — the streak-orb primitive wrapped in .tma-orb-ring, its conic arc filled by the session's accuracy. Ring ink + orb data-state flip together per scenario so the outcome reads before the copy does. Three scenarios; the resolver is pure and lives in components/session-scenario.ts.

Session debrief (SessionRecap)

Under the orb + copy, the complete state renders a SessionRecap ledger — one row per card the user marked Again or Hard this round (from useFocusSession.sessionMisses). Each row quotes the ink of the grade that produced it (danger = Again, warn = Hard); tapping a row opens the shared MissSheet composer pre-linked to that conspectus, so the miss note is saved via POST /api/v1/errors without leaving Focus. A saved row cools to success («✓ Noted») and locks for the rest of the session. Clean sweep → the ledger doesn't render at all; «Back to Today» stays the one hero action (D2 intact).

This replaces the earlier quiet «Log a miss →» link that deep-linked to /errors?prefill_from=session with only the last miss — a session with N misses could log exactly one. The deep-link stays a valid URL contract on /errors; the in-app producer is now the recap.

The screen is composed of three stacked regions plus two end-states. Every region names the React component that draws it, the data source that feeds it, and the CSS primitive that styles it.

Grade → tag mapping

Server accepts three tags — easy, hard, forgot. The Focus screen shows four buttons because the SM-2 tradition of Again · Hard · Good · Easy matches how users self-report recall, and analytics needs the finer resolution. Until the server contract grows a fourth tag, Good collapses to hard — both mean «keep in rotation soon-ish»; only Easy earns the big interval bump and only Again resets. The distinction is preserved client-side in SessionSummary.perGrade for analytics.

Grade Hotkey Tone Server tag SM-2 meaning
Again1dangerforgotComplete failure — reset the schedule.
Hard2warnhardGot it but struggled — short next-interval.
Good3accenthardNormal recall — same interval treatment as Hard until 4-tag lands.
Easy4successeasyTrivial — big interval bump.

Hooks & endpoints

Hook Endpoint Query / mutation key
useConspectusesDue (via useFocusSession) GET /api/v1/conspectuses/due ['today.conspectuses.due', client_uuid] — snapshot on first success, capped at SESSION_CAP = 20
useReviewConspectus POST /api/v1/conspectuses/{uuid}/actions/review Pessimistic — row leaves the due-list cache on onSuccess; response's next_review_at drives the interval preview.

Reuse note: the mutation hook is the same one Today's swipe primitive uses (T-16). Both screens share pessimistic semantics, so the queue is trustworthy across route changes.
