Screen map: Search

The keyboard-first jump-to-anything surface — a full-viewport command palette that opens on Cmd+K / Ctrl+K or the header ⌕ icon on Today. Unlike every other screen in the TMA, Search is not a route: it's a React portal mounted at document.body from a provider that sits inside the router, so any route can trigger it without a navigation. Filtering is pure client-side substring over a pre-warmed 100-title cache (the server hard-caps limit at 100); there is no search endpoint. Sibling of the design narrative; this is the machine-readable anatomy.

Purpose

As a user's library grows past a screen or two of conspectuses, browse-only navigation becomes a bottleneck: the muscle-memory shortcut to «that Python decorators note I made last month» is typing, not scrolling. Search closes that loop with a single keystroke and zero server hops — the whole title index is fetched once per session and filtered locally, so results appear on the same event-loop tick as the keypress. The overlay-not-route shape is a deliberate choice: opening Search must never lose the underlying context (a half-graded Focus card, a scroll position on Today), and closing it must return the user exactly where they were.

Because the whole surface is a portal to document.body, it sits above every route's own <main> without needing any z-index gymnastics on the caller. The provider is intentionally mounted inside the router (moved there in code review) so useNavigate() is available when Enter is pressed on a result — a naive «mount above the router» split would break navigation from within the palette.

States

State Trigger What renders
loadinglist.fetchStatus === 'fetching' — cache is cold, the one-shot library fetch is in flight.Empty-state slot: «Loading library…» copy plus a subdued spinner glyph. Input is enabled but filter is a no-op until data lands.
hintCache resolved, query is empty (or whitespace-only).Empty-state slot: «Start typing to find any conspectus». No list.
matchCache resolved, query non-empty, hits.length > 0.Results list — up to N rows, first row selected. Each hit renders its title with a <mark>-wrapped substring plus the slot label.
no-matchCache resolved, query non-empty, hits.length === 0.Empty-state slot: «No matches for «xyz»» — the current query is echoed back verbatim.

The four states are mutually exclusive and cover every possible render. There is no separate error state — the underlying list query is staleTime: Infinity and fires once per session; a fetch failure surfaces as loading that never resolves. If that becomes user-visible, the fix is a retry inside useConspectusesList, not a new UI state.

Data hooks

Hook Endpoint Query key / config
useConspectusesList GET /api/v1/conspectuses?limit=100 ['conspectuses.list', client_uuid] · staleTime: Infinity · enabled: auth.status === 'authenticated'

Exactly one hook. It fires once per session (opening the overlay for the first time), warms the shared TanStack Query cache, and never re-fetches on its own — a mutation elsewhere in the app is what invalidates the key. The 100-row cap is a hard server ceiling (Query(ge=1, le=100)); past it the overlay surfaces a «showing first N» hint via hasMore, and we'll switch to a server-side ?q= endpoint (see Non-goals). Filtering itself is a synchronous, allocation-free pass in search-filter.ts against a pre-lowercased index built once on cache resolve.

Accessibility

  • Dialog semantics. Overlay root carries role="dialog" + aria-modal="true" and an aria-label derived from the search placeholder.
  • Combobox pattern. The input owns aria-controls pointing at the list, and aria-activedescendant pointing at the highlighted option's id. Cursor moves never take focus off the input, so keystrokes keep flowing to the filter without a focus round-trip.
  • Listbox. The <ul> carries role="listbox"; each <li> carries role="option" plus aria-selected. The cursor row is the single aria-selected="true" option.
  • Live region. A visually-hidden role="status" + aria-live="polite" node announces «Loading library…», «N matches», or «No matches for …» when the outcome changes. Screen readers get the same feedback sighted users get from the visible copy.
  • Focus save + restore. On open, the overlay records document.activeElement and moves focus into the input; on close, focus is restored to that element. This is what makes Cmd+K from a mid-workflow surface non-destructive.

Non-goals

  • Fuzzy matching. No Levenshtein, no trigram, no typo tolerance — a substring match against 100 rows is fast, deterministic, and easy to explain. Fuzzy scoring gets added the same day we hit the 100-row cap.
  • Server-side ranking. Ranking is «earliest-match, then shorter title», computed on the client. There is no ?q= parameter on the endpoint; there is no relevance score.
  • Recent-history section. No «recently opened» rail above results — deferred to T-25 with the rest of the palette-expansion work.
  • Pull-down mobile gesture. Opening Search from a mobile pull-down on Today is deferred to T-25; today the mobile entry is the header ⌕ icon only.
  • Body search. Only conspectus titles are indexed. Searching inside dense_paragraph / bullets / notes is a separate epic and needs a server endpoint.
  • Non-Latin normalisation. Only toLocaleLowerCase() is applied — no diacritic-strip, no Cyrillic-to-Latin transliteration. «é» does not match «e», «Ё» does not match «е». Good enough for the current corpus; revisit if users complain.
