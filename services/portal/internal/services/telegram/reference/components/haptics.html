Haptics vocabulary

Seven tactile patterns the TMA speaks — the same seven every screen picks from. One entry point (haptic()), one call-site catalogue, one lookup table for «which tone means what». Progressive-degrade to a no-op when no Telegram SDK is present, so the code compiles in vitest, boots in a plain browser tab, and shakes wrists in real Telegram.

The seven tones

Named for what they mean, not for the SDK call they wrap — impactMedium isn't the interesting bit at the call site; «swipe committed» is.

Tone SDK call Meaning · when to fire
selectionselectionChanged()Softest tick. Discrete list highlight, tab switch, spotlight open, card-reveal acknowledgement.
impactLightimpactOccurred('light')Primary button press, minor confirm (Focus «Good/Easy» grade, Onboarding start, «Log a miss» opener).
impactMediumimpactOccurred('medium')Decisive tap — swipe commit, Focus «Hard» grade.
impactHeavyimpactOccurred('heavy')Miss acknowledgement — Focus «Again», swipe left past the deep threshold («forgot»), destructive confirms.
notifySuccessnotificationOccurred('success')Save landed. Focus session complete, miss saved, nudge updated, streak advanced.
notifyWarningnotificationOccurred('warning')Soft error — rate-limit, throttle, retry recommended. Unused today; reserved.
notifyErrornotificationOccurred('error')Hard error — save failed, mutation rolled back, grade error banner.

Degrade paths

  • No SDK / plain browser tab (dev). The mock-env shim installed by installTelegramMock() maps every call to console.debug('[haptic] …'). Wiring is verifiable without a phone in hand — open devtools, tap a graded card, watch the log.
  • WebView throws on unsupported pattern. Some Android WebViews cough on selectionChanged() when the OS is running an outdated system haptics service. The wrapper's try/catch swallows the exception — haptics are decorative acknowledgement, never load-bearing behaviour, so a swallowed call must never regress a save or a navigation.
  • vitest. No window.Telegram unless a test stubs it. The wrapper's SSR guard returns early — 88 passing test cases don't trip on ambient globals.
  • SSR / node build. Same guard covers typeof window === 'undefined'.

Hand-testing

A dedicated /debug/haptics route lists every tone as a row you can tap. Not linked from the app nav — you either type the URL, or open the same tunnel URL from real Telegram.

  • Local browser — http://localhost:5173/debug/haptics. Tap a row → watch devtools log [haptic] impactOccurred(medium) etc.
  • Real device — make -C services/telegram tunnel, open the printed trycloudflare.com URL through the BotFather test bot in real Telegram, navigate to /debug/haptics, tap each row → feel the pattern.

Accessibility

  • Not load-bearing. Every state change that fires a haptic also updates a visible / announced surface (toast, phase-swap, error banner). Users who never feel the vibration lose no information.
  • Reduced motion. Haptics are not motion — they're acknowledgement. Telegram exposes no prefers-reduced-haptics media query; users who want silence disable Telegram's own haptic setting. We deliberately do not collapse haptics under prefers-reduced-motion: the two concerns decouple.

Fixes that had to land before the on-device handshake was green

Haptics only fire inside a real Mini App WebView. Getting that state green from a plain make dev uncovered five infra bugs that were all silent until the very last hop — «vibration doesn't fire» sat on top of a stack of masked failures. The record is here so the next «SDK-adjacent» feature doesn't hit the same wall.

  • Missing <script src="https://telegram.org/js/telegram-web-app.js"> in services/telegram/index.html. Without it Telegram doesn't inject window.Telegram.WebApp, our mock-telegram-env.ts shim takes over, and every haptic call routes through the no-op mock. Symptom on /debug/haptics: platform: weba, initData: missing inside real Telegram.
  • Bot API 7.2+ signature field IS part of the HMAC data-check-string. Initially assumed it should be excluded (some third-party libs do that); on iOS Telegram 9.6 the assumption breaks every real handshake while our dev signer (which doesn't emit signature) round-trips green. Kept in services/api/app/core/telegram_init_data.py, pinned by test_signature_field_participates_in_hmac_check_bot_api_7_2.
  • Stale-cache in bootstrapAuth: cached JWT + user from a previous dev-fallback session (under telegram_user_id=42) were reused when the real session belonged to a different id — client thought it was logged in, UI hung on «Connecting to the server…». Fix: readInitDataUserId() compares against cached user.telegram_user_id; mismatch wipes both cache keys and forces a fresh exchange.
  • CloudStorage callbacks can silently never fire on some builds. Added a 3 s withTimeout() around getItem/setItem/removeItem in shared/auth/cloud-storage.ts — falls back to «cache empty» so bootstrap proceeds instead of hanging forever.
  • Two tunnels vs one. The old tunnels-up.sh launched two Cloudflare quick tunnels (frontend + API) and wrote VITE_API_BASE_URL into .env.local — every restart handed out a fresh URL and a stale value in the env silently broke auth. Vite dev-server now proxies /api/* to the local API, the client uses relative URLs, and one tunnel covers everything.
