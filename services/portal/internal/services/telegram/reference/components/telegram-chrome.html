Telegram SDK chrome

Native BackButton / MainButton / SettingsButton wired through three declarative hooks — pass a handler while the button should be visible, pass null (or unmount) to hide it. The SDK API is imperative (show / hide / onClick(cb) / offClick(cb)); the hooks own the lifecycle so callsites read as intent, not as a subscription dance.

Three hooks, identical lifecycle contract

Hook Signature Visible when
useTelegramBackButton (handler: (() => void) | null) => void handler !== null — clicking calls the current handler via a ref
useTelegramMainButton ({ text, onClick } | null) => void config !== null — setText re-runs on text change; click still routed through a ref
useTelegramSettingsButton (handler: (() => void) | null) => void handler !== null — same shape as BackButton

Handler identity is stashed in a useRef: changing a closure's captured props doesn't churn the SDK subscription (would leak offClick without onClick). Effect deps depend only on active (and text for MainButton), so re-renders are cheap.

Design policy — native chrome complements, not replaces

The on-canvas «Start Focus» button, «Review now» ember CTA, and «← Back» links are not removed when the native chrome lands the same action. Three reasons:

  • Plain-browser dev loop. Vite dev / Storybook / vitest have no window.Telegram.WebApp. The on-canvas affordance is the only way to complete the flow there.
  • Older Telegram builds. Some Android WebViews from before Bot API 6.9 lack SettingsButton; some pre-6.1 builds drop HapticFeedback. The on-canvas CTA is the last-known-good baseline.
  • Muscle memory in-viewport. Even on-device, a user scrolled inside the note doesn't always look at the very bottom of the sheet. The native MainButton is a supplement, not a substitute.

Degrade paths

  • No SDK. The hook returns early when window.Telegram?.WebApp?.[Button] is undefined. Same shape the plain-browser dev shim and vitest use, so wiring is verifiable outside Telegram — the wired handler simply never runs.
  • SDK present, button subobject missing (pre-6.9 SettingsButton, pre-6.1 haptics-adjacent builds). Same early-return; no throw.
  • SSR / node build. typeof window === 'undefined' guard covers server render.

Accessibility

  • Never load-bearing. Every action wired through the native chrome also exists on-canvas — users who can't see or feel the native shell (screen-reader, older build) lose no navigation.
  • Focus safety. The hooks don't move focus when they hide the button; the parent screen's tab order is untouched.
