Component spec: EtrMoves

The editorial peek on the Onboarding screen — three E · T · R moves as a plain typographic list. No card chrome, no illustrations, no expand/collapse — the entire method fits on one screen so the user can read it in one glance and start.

Overview

A semantic <ol> of three rows. Each row is a two-column grid: an ember-inked mono digit on the left (the numeric spine of the method), a title + one-sentence prose gloss on the right. Everything localised — the copy set is a closed six-string table (extract, extractBody, transform, transformBody, retrieve, retrieveBody). The row order is stable; the numbers 1 · 2 · 3 read as an ordered sequence, not a checklist.

Props

None. The move set is a hard-coded closed tuple — a fourth move means a new component (the whole method is «three»; adding a fourth row would break the ETR mnemonic on which the app is named).

Design tokens

  • Numeric spine. --tma-font-mono at --tma-fs-small, weight 800, ink --tma-tone-accent. The mono ink echoes the digit inside the day-zero orb without repeating it.
  • Title. --tma-fs-lead semi-bold on --tma-text-primary, letter-spacing −0.01em.
  • Body. --tma-fs-body at 1.55 line-height on --tma-text-secondary.
  • Gaps. Row gap var(--tma-sp-4); column gap var(--tma-sp-3). No border, no divider — whitespace does the separating.

Accessibility

  • Semantic ordered list. Real <ol> + <li>. Screen readers announce «1 of 3, Extract, …»; the numeric spine is aria-hidden to avoid double-reading the count.
  • Labelled group. The list carries the localised onboarding.tagline as its aria-label so the group's purpose is announced before the rows are enumerated.
  • Not interactive. Rows are not focusable, do not pretend to be links or steps — the CTA below is the only affordance on the screen.
