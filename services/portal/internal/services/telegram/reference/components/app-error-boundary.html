AppErrorBoundary

The app's crash surface — two error tiers rendering one shared BoundaryFallback: TanStack Router's defaultErrorComponent for route-component throws, and the AppErrorBoundary class for throws in the provider shell above the router. Both show the same warm ErrorScreen the auth handshake failure uses — instead of the raw fallbacks (router's built-in «Something went wrong!» plate, Vite's dev overlay) that used to reach the user on a runtime crash.

Behaviour

  • Two tiers, one surface. TanStack Router wraps every route in its own catch boundary, so a throw from a route component (Today, Focus, …) never reaches the app-level boundary. The router tier is covered by defaultErrorComponent: RouterErrorFallback — without that override the router renders its built-in ErrorComponent («Something went wrong!» + a red <pre> quoting the raw message), in production too, since it's runtime router code rather than a dev overlay. The AppErrorBoundary class covers the second tier: throws from the provider shell above the router (auth bootstrap, theme, toasts). Both render the shared BoundaryFallback.
  • What the tiers catch. Errors thrown from a component's render() or from an useEffect/useLayoutEffect callback. Same coverage as React's built-in error boundary contract.
  • What it does NOT catch. Errors inside event handlers (onClick, onChange …) — those propagate normally and either bubble to window.onerror or need per-callsite try/catch. Also does not catch anything thrown by code that lives outside React's render cycle (a raw setTimeout callback, a fetch().then() chain that isn't awaited in an effect).
  • Recovery paths. Two:
    • Try again — resets the owning boundary so the tree re-renders (on the router tier also router.invalidate() to re-run the route). Fine for transient errors (race condition, one-off network blip).
    • Reload the app — the underlined link below the primary CTA fires window.location.reload(). Nuclear option for anything that survives an in-place retry.
  • Observability. componentDidCatch writes a single tagged console.error today: [AppErrorBoundary] caught render/effect error:, the error, and the component stack. When Sentry lands (T-26d), that line gets replaced with Sentry.captureException(error, {contexts: {react: {componentStack}}}).

Mount order — why not further out

Boundary sits inside ThemeProvider + ViewportProvider so the fallback resolves the app's Ember tokens and safe-area vars — otherwise a broken app would render an unstyled broken orb over the wrong background. Sits outside AuthProvider so an error thrown from the auth bootstrap itself surfaces here instead of vanishing under a broken AuthGate.

<QueryClientProvider>
  <ThemeProvider>
    <ViewportProvider>
      <AppErrorBoundary>         ← tier 2: provider-shell errors
        <AuthProvider>
          <ToastProvider>
            <AuthGate>
              <Router />         ← tier 1 inside: router's own catch
            </AuthGate>            boundary per route, rendered by
            <Toaster />            defaultErrorComponent
          </ToastProvider>
        </AuthProvider>
      </AppErrorBoundary>

</QueryClientProvider>

Why a class component

React's error boundary API is class-only — there is no hook equivalent (useError proposals live in RFC-land, not in the runtime). The class body is kept minimal so the class-vs-hook mismatch never leaks past this file: all rendering (and hooks-based i18n) lives in the shared BoundaryFallback function component; the class only holds the caught-error state and the console tag.
