Motion vocabulary + View Transitions

Three named springs, three named durations, two named easings — everything else is a copy of one of them with a comment on why. Plus router-level opt-in to the browser View Transitions API, so route swaps cross-fade without any per-callsite work.

Springs

Framer-motion transition presets. Pick from the list before minting a new one — two nearly identical springs feel indistinguishable to the eye but drift on paper.

TokenStiffness · DampingWhen to reach for it
SPRING_SOFT220 · 24Content that arrives — hero mount, ambient fade-ins. Overshoot is visible but subtle.
SPRING_MEDIUM300 · 28Default. Cards, panels, sheets. Reads as «placed, not thrown». Used on Focus card mount + FocusCard's item variants.
SPRING_STIFF400 · 40Controls settling to their resting position — swipe rows returning to x=0, focused card seating into the stage. Snaps hard.

Durations + easings

TokenValueWhen to reach for it
DURATION_FAST_MS150Fast exit / discreet fade — «this is leaving, don't linger». Focus card exit, Today's due-list AnimatePresence exit.
DURATION_BASE_MS220Default for in-place changes (colour, opacity, small transforms). Also drives the View Transitions cross-fade timing.
DURATION_SLOW_MS550The «watchable» transition — reserved for the FocusCard flip and full-screen movements where the motion itself is content. Pinned at 550 ms because dropping the flip below reads as «snap», not «reveal».
EASE_GLASS[0.7, 0, 0.3, 1]Asymmetric cubic-bezier that leans into the halfway point — peak visual interest at ~55 % progress. FocusCard's 180° rotation.
EASE_OUT[0.2, 0, 0, 1]Standard ease-out for exits — quick handoff to the next state without the material-design bounce-back.

Framer expresses durations in seconds, DOM APIs in milliseconds. durationSec(ms) and durationMs(sec) live next to the tokens so callsites never sprinkle / 1000 arithmetic in their JSX.

Small drifts we accepted during migration

Standardising four callsites onto three springs and three durations moved a handful of values by up to 40 ms / 40 stiffness units. FocusCard's flip duration was pinned back to its original 550 ms (see the token note above); the rest are within visual tolerance and were left as the round-trip of standardisation.

  • FocusCard item variants: spring 260/26 → 300/28 (SPRING_MEDIUM). Slightly stiffer entrance — items settle a hair sooner.
  • Focus stage spring: 320/30 → 300/28 (SPRING_MEDIUM). Slightly softer — cards seat with a touch more give.
  • Focus card exit: 180 ms → 150 ms (DURATION_FAST_MS). 30 ms quicker exit fade.
  • DueCardsList exit: 250 ms → 220 ms (DURATION_BASE_MS). 30 ms quicker row collapse.
  • Today empty appear: 200 ms → 220 ms (DURATION_BASE_MS). 20 ms slower fade-in — imperceptible.

If a future callsite needs an off-preset value, add a named token here — do not sprinkle magic numbers back into JSX.

View Transitions API

TanStack Router's createRouter({ defaultViewTransition: true }) hooks every navigation into document.startViewTransition. The browser snapshots the outgoing and incoming DOM into ::view-transition-old(root) and ::view-transition-new(root) pseudo-elements and cross-fades them — no per-callsite code, no framer wrappers on route boundaries.

The pseudo-element animation duration is set to DURATION_BASE_MS so it matches the rest of our motion vocabulary. Under prefers-reduced-motion: reduce the CSS drops it to 1 ms, so the surface flips without a visible cross-fade — same reduced-motion story as the springs.

Browser support: Chromium 111+, Safari 18.2, Android WebView on recent Telegram builds. Older WebViews (iOS pre-18.2, Android WebView on some legacy Telegram builds) silently fall back to a plain state swap — TanStack Router feature-detects and calls startViewTransition only when it exists on document. Progressive enhancement, zero runtime cost on unsupported browsers.

How to test

  1. Unit — npx vitest run src/shared/motion: only the token module exists in this slice; there are no runtime tests here because springs / durations are declarative constants, not code paths. Coverage lives at the callsite (existing FocusCard / DueCardsList / Focus / Today tests still pass at 96/96).
  2. Visual, browser — make -C services/telegram dev, open http://127.0.0.1:5173 in Chrome:
    • Route transitions: click a Today due card → detail opens with a subtle cross-fade (220 ms). Back via the browser back button — same fade in reverse.
    • Focus reveal: Start Focus → card flips 180° in ~450 ms with the glassy easing.
    • Swipe row settle: drag a Today due card halfway right, release — springs back to x=0 quickly (stiffness 400).
    • Reduced motion: DevTools → three-dot → Run command → «Emulate CSS media prefers-reduced-motion: reduce». Every animation collapses instantly.
  3. Visual, Storybook — make -C services/telegram storybook, view Screens / Focus and Screens / Today / DueCardsList stories. Same primitives; stories run without a router so no View Transitions, but framer springs / durations render.
  4. On-device — make up boots the tunnel + syncs Menu Button; reload Mini App. On iOS 18.2+ Telegram, route swaps cross-fade; on older builds nothing appears to change (correct fallback).
