On-call

Small tester cohort — no paging, no rotation. This page is the checklist Ivan walks when the Mini App misbehaves. Ordered by frequency, not by severity. Each item names the exact command / URL to check first.

30-second triage

Before diagnosing anything specific, answer three questions in order. Most incidents resolve at question one.

  1. Is the API running? Hit http://localhost:8000/ready (dev) or the production API URL. If not 200 — the client is fine, the API is down. Jump to api on-call.
  2. Does the bundle load? Open DevTools → Network → filter «Doc» and «Script». Missing / 404 / MIME-mismatch → deploy is broken (see Bundle broken below).
  3. Does the console show anything red? Uncaught exception → open Sentry issue link (once W5 · T-26 ships). Before Sentry: expand the stack trace in DevTools, note the file + line, look at the source.

Common failures

Bundle broken (blank screen, MIME-mismatch)

  • Symptom: Telegram opens the WebApp, blank screen for > 3 s, DevTools console shows «Failed to load module script».
  • Root cause 90% of the time: a build with a broken base or a CDN cache serving an old index.html referencing hashed assets that no longer exist.
  • First check: curl -I https://study-app.pages.dev/ — look at etag. Compare with a fresh vite build output locally.
  • Fix: re-run the Cloudflare Pages deploy from the latest main. CDN purges automatically on new deploy.

API unreachable (all screens show error toasts)

  • Symptom: Every screen paints its skeleton, then shows an «Retry» toast within 5 s. TanStack Query is retrying in the background.
  • First check: curl -s https://api.study.example.com/ready. If not 200 — API is the problem.
  • If dev: is make -C services/api run still up? Docker containers healthy (docker compose ps)?
  • If prod (once W7 ships): Cloudflare Tunnel / VPS / Fly.io — depends on the hosting decision. Runbook to be finalised with ADR 0039.

CORS-blocked (browser console «CORS policy»)

  • Symptom: One specific origin — usually a new tunnel URL or a new port — sees requests fail with a CORS pre-flight error in the browser console.
  • Root cause: the origin isn't in CORS_ALLOW_ORIGINS in env/.
  • Fix: add the origin to env/dev (or env/example if it's a documented recipe) and restart the API.

Auth-rejected (401 on every request)

  • Symptom: Every /api/v1/* call returns 401 Unauthorized, error toast reads «Session expired».
  • Root cause A: POST /api/v1/auth/telegram failed at cold-open — usually mismatched TELEGRAM_BOT_TOKEN between bot and API (the HMAC key must be identical).
  • Root cause B: JWT_SECRET was rotated on the API without invalidating cached JWTs on the clients; existing tokens fail signature verification.
  • First check: in DevTools → Network → /auth/telegram. If it's absent, the bootstrap didn't fire — check the SDK loaded. If it's 401, tokens diverged. If it's 200 but subsequent calls are 401, JWT_SECRET mismatch.
  • Fix: confirm TELEGRAM_BOT_TOKEN parity across bot and API; if JWT_SECRET was rotated, notify clients to re-bootstrap (evict CloudStorage cache).

Deep-link opens the wrong screen

  • Symptom: Tap a bot notification with startapp=conspectus_<uuid> → Today screen opens instead of that conspectus.
  • Root cause: Either the payload wasn't parsed, or the router doesn't know the target route.
  • First check: in DevTools console — window.Telegram.WebApp.initDataUnsafe.start_param. If empty → bot didn't send the payload (bot bug). If present → router bug.
  • Grammar reference: deep-link grammar.

Idempotency violated (duplicate rows)

  • Symptom: User marks one review, two conspectus_events rows appear.
  • Root cause: The client sent two different Idempotency-Key values for what should have been one gesture (retry-loop bug), or the API idempotency middleware didn't see the header.
  • First check: tail API access log filtered by user's correlation_id. Two request lines with two different keys → client bug in shared/api/. Two lines with the same key returning 200 twice → API-side idempotency bug.

Where to look

Client-side signals live in tools designed for browsers; API-side signals live in the backend stack. Both are joined by correlation_id — the client sets it on every request, the API logs it.

Where What it answers
Sentry (client, ships W5 · T-26) «What broke inside the browser?» — uncaught exceptions, unhandled rejections, source-mapped to the TypeScript file.
PostHog (client, ships W5 · T-26) «What did the user do before it broke?» — event stream, funnels, session replay.
Grafana → Prometheus (API) «Did the backend spike?» — request latency, error rate, per-endpoint counters. Filter by time window.
Kibana → Elasticsearch (API) «What was the API doing at that exact moment?» — structured logs, joinable by correlation_id.
DevTools (immediate) «What does my browser see right now?» — Network tab for request/response bodies, Console for JS errors.

Paging model

None while the audience is a tester cohort. No alerts, no rotation, no PagerDuty. If the app is broken and nobody's using it, nothing gets escalated — someone will notice next time they open it. Adding paging infrastructure at this scale is over-engineering that will decay before it's ever needed. Revisit once the tester cohort grows beyond a handful of people.

The day the tester cohort grows past ~10 people or the app opens to public onboarding (P2 epic), this page gets a rotation table, a real SLO, and links to alert rules in Prometheus. Not before.
