Why React + Vite

The Mini App is a React 18 + TypeScript app bundled by Vite 6. Not vanilla JS. Not Svelte. Not Next.js. This page explains what those trade-offs bought us, what they cost, and when the answer would change.

Alternatives considered

Option Why it looked attractive Why it lost
Vanilla JS + HTML Zero Node in the project. Homogeneous Python stack. No node_modules to police. Nine screens with cross-screen shared components, swipe gestures, i18n, motion + haptics vocabulary, session-scoped state → ~4× the code, no type-safety, and a Telegram-UI component library that has no vanilla equivalent. Estimated 3-week delivery cost for the same MVP surface.
HTMX + server-rendered pages Server-first. Python-native. No JS framework. Telegram SDK (haptics / viewport / back-button / theme) is JS-side. Motion + swipe are JS-side. The result would be «HTMX for navigation + custom JS for TMA behaviour» — worst of both.
Svelte / Solid / Vue Smaller bundles. Cleaner reactivity models. All have Vite plugins. No @telegram-apps/telegram-ui equivalent — we'd re-implement Cell / Section / Button ourselves. Also: React is the ecosystem TanStack Router / Query, framer-motion, and the Telegram SDK-React target first.
Next.js / Remix / TanStack Start File-based routing. SSR. Batteries included. SSR is a liability for a Telegram Mini App — the WebView loads a static bundle from CDN. Nothing to server-render. File-based routing is nice; TanStack Router gives type-safe routing without the framework tax.
React Native (compile to native) Real native apps. Best performance. Telegram Mini App is a WebView. There's no path to native inside Telegram. RN would be a different product.

Why React specifically

Three concrete things React does better than the alternatives for this problem shape:

  1. Ecosystem gravity around Telegram tooling. @telegram-apps/sdk-react and @telegram-apps/telegram-ui are React-first — they ship as hooks and components. Non-React consumers use the vanilla SDK and re-implement the UI kit. That's weeks of work we skip.
  2. TanStack Router + Query are React-native. Both work with other frameworks but the React versions are the flagship. Type-safe routing with built-in View Transitions is worth its bundle bytes for a nine-screen app.
  3. Component reuse in a habit product. The due-card appears on Today, on Search results, on Conspectus list. One <ConspectusCard> component, three places. Svelte / Solid would deliver this too — but React's ecosystem for portfolio-grade component patterns (Storybook, Chromatic, testing-library) is mature enough that «boring» is a feature.

Why Vite specifically

Vite competes with webpack, Rspack, Parcel, esbuild, Turbopack, Rollup, Rolldown. For this project's shape the shortlist was Vite vs Rspack. Vite won on:

  • HMR that stays fast at 200+ modules. Cold start ≈ 300 ms, HMR ≈ 30 ms. Rspack is competitive but still catching up on the plugin ecosystem.
  • First-class React + TS integration. Zero-config for the 80% path; escape hatches for the 20%.
  • Storybook 8 works out of the box. Rspack's Storybook path is newer.
  • Cloudflare Pages has native Vite support. Deploy = vite build, no adapter layer.

Why React 18 and not 19

React 19 is stable and the target we'd pick in a green-field project. Two reasons we're on 18.3.1 instead:

  1. Peer-lock in @telegram-apps/telegram-ui@2.1.13. It declares peerDependencies.react: "^18.2.0". Newer telegram-ui builds haven't shipped as of the epic date.
  2. Forcing 19 via --legacy-peer-deps breaks lockfile reproducibility. CI installs from the lock and can end up with a different tree than dev, or subtly hit removed React APIs (unstable_batchedUpdates, etc.) inside telegram-ui at runtime — silent bugs.

The upgrade is a two-line change (package.json react/react-dom, main.tsx createRoot arguments) on the day telegram-ui publishes a 19-compatible build.

What this costs

  • Node.js in the toolchain. Dev machines and CI need Node 20+. Not «free» — it's a runtime to install, a lockfile to police, a security surface to monitor via pip-audit-equivalent (npm audit).
  • 253 npm packages in node_modules/. A supply-chain surface. Mitigated by pinning via package-lock.json, running npm audit in CI, and reviewing every dep before it lands.
  • ~144 KB / 46 KB gz initial bundle. React alone is ~40 KB gz. Tanstack Router + Query add ~15 KB. framer-motion is code-split — only Focus and Today pay for it upfront.

None of these show up at runtime for the user. The user gets a static HTML + JS + CSS bundle served from a CDN. Node runs zero times when they open the app.

When this decision would flip

  • The bundle budget breaks (> 200 KB gz) and can't be recovered by code-splitting — reconsider Svelte / Solid for lower baseline weight.
  • Telegram ships a native app-framework that replaces WebApps — this whole page becomes historical.
  • The project expands to a public product with SEO needs — Next.js becomes relevant for the marketing surface (still not for the Mini App itself).
  • React ecosystem consensus moves elsewhere and telegram-ui / SDK abandon React bindings.

None of these are foreseeable within the epic's timeframe.
