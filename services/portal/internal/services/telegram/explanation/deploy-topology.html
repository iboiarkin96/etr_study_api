Deploy topology

Two independent components need to be online for the Mini App to work: the static bundle (frontend) and the API (backend). The frontend story is settled — Cloudflare Pages. The backend has three viable paths on the table; the choice is deferred until on-device testing arrives (W6). This page explains today's setup, tomorrow's options, and how they connect.

Today: localhost loop

During active development, everything runs on the maintainer's Mac:

┌─────────────────────────────────────────────────────────────┐
│ Chrome / Safari on the Mac                                  │
│    ↓ loads                                                  │
│ http://localhost:5173  (Vite dev server)                    │
│    ↓ serves the Mini App bundle                             │
│    ↓ fetch(...)                                             │
│ http://localhost:8000  (FastAPI via docker compose)         │
│    ↓                                                        │
│ postgres:16-alpine  (docker container, same compose)        │
└─────────────────────────────────────────────────────────────┘

No cloud, no real Telegram. tools/dev/sign_init_data.py mints a synthetic initData against the same TELEGRAM_BOT_TOKEN the API uses; the app boots, exchanges it for a real JWT at /auth/telegram, and behaves the same as it would on-device. See Run locally.

Today (on-device): tunnel loop

When testing haptics / safe-area insets / real Telegram chrome, both the frontend and the API get temporarily exposed via Cloudflare Tunnel:

Telegram WebView (phone)
    ↓ opens
https://xxxxx.trycloudflare.com   (Vite tunnel)
    ↓ fetch(...)
https://yyyyy.trycloudflare.com   (API tunnel)
    ↓ still hits the Mac's docker compose stack

Same Mac, same processes — just wrapped in HTTPS by cloudflared. Tunnel URLs change on every restart of cloudflared, which is why Test on device reads like a checklist.

Tomorrow · frontend: Cloudflare Pages

Frontend deploy is settled. In W7 · T-29:

  1. git push to main triggers Cloudflare Pages via the GitHub integration.
  2. Pages runs make -C services/telegram build, produces services/telegram/dist/.
  3. Pages uploads dist/ to its global CDN, gives us a study-app.pages.dev URL (or a custom domain).
  4. The URL is registered in BotFather as the Mini App URL for the bot.

Runtime: pure static-file serving. No Node on the CDN edge. The Mini App is HTML + JS + CSS + a handful of assets. Time-to-first-byte is essentially network latency.

Why Cloudflare Pages over alternatives (GitHub Pages, Vercel, Netlify): existing Cloudflare relationship, native Vite support, generous free tier, and it composes with Cloudflare Tunnel (same account) for on-device testing.

Tomorrow · backend: three viable paths (undecided)

The API doesn't have a production home yet. The local docker-compose stack carries early dev, but on-device Telegram testing (W6 · T-27 / T-28) needs a public HTTPS endpoint. The three paths on the table:

Cloudflare Tunnel from the Mac ($0/mo)

The dev loop, promoted to «production». cloudflared tunnel run alongside docker compose up, backed by a Cloudflare-registered domain. Works when the Mac is on; doesn't when it isn't. Only suitable for the tester cohort while the Mac is available.

Hetzner CX22 VPS (~€4/mo)

Cheapest «install once, forget» option. Copy the docker-compose.yml, add nginx + Let's Encrypt for HTTPS, add a Postgres backup cron. Runs 24/7 independent of the Mac. ~1 hour first setup, ~5 min/mo maintenance after.

Fly.io ($0-5/mo)

PaaS. fly deploy from the Dockerfile, Postgres as a separate managed component, HTTPS + restart + health checks included. Portfolio-friendly. Free tier covers a small tester cohort.

ADR 0039 will pick one when we hit W6. Until then, the client's VITE_API_BASE_URL stays parameterised so the switch is a build-config change, not a code change.

Runtime shape (target)

Once both halves are deployed:

User taps a bot notification in Telegram
    ↓
Telegram opens the WebView, loads https://study-app.pages.dev
    ↓ HTML + JS + CSS from Cloudflare Pages CDN (edge-cached)
    ↓
React mounts, reads themeParams + language_code, dispatches to /
    ↓ fetch("https://api.study.example.com/api/v1/auth/telegram",
    ↓        body: window.Telegram.WebApp.initData)
    ↓ Backend: HMAC-verify initData → upsert users + telegram_users rows → mint 24 h JWT
    ↓ Client: cache JWT in WebApp.CloudStorage, keep hot in memory
    ↓
    ↓ fetch("https://api.study.example.com/api/v1/conspectuses/due",
    ↓        headers: { Authorization: "Bearer <jwt>",
    ↓                   "X-Request-Id": "abc123" })
    ↓
Backend (wherever it lives): FastAPI → Postgres → JSON scoped by JWT sub
    ↓
Response paints on Today screen (< 2 s LCP budget)

What this page doesn't decide

  • Which backend hosting path wins. That's ADR 0039 — drafted in W6, accepted before the first on-device test.
  • Custom domain scheme. Whether the Mini App lives at app.study.example.com, study-app.pages.dev, or something else — pinned when the backend host is chosen (so both are registered at the same time).
  • Bot hosting. The bot runs inside the API container (aiogram Dispatcher on FastAPI lifespan startup), so it deploys with whatever hosts the API.
