Observability model

Two observability stacks — one for the client, one for the backend — joined by a single request_id. This page explains why the split exists, why we don't collapse it to a single tool, and how the two halves connect end-to-end.

The two stacks

Stack Tools Runtime it observes Question it answers
Client observability Sentry · PostHog Browser / WebView on the user's device. «What happened in the user's session?» — clicks, funnels, uncaught exceptions with source-mapped stack traces.
Server observability Prometheus · Grafana · Elasticsearch (+ Filebeat) The FastAPI process + Postgres. «What was the backend doing?» — request rate, latency, error rate, per-endpoint counters, structured logs.

Why not use one tool for both?

The difference isn't «backend vs frontend» in a taxonomic sense. It's a runtime-model mismatch.

Tool How it works What that assumes
PrometheusPull: polls a URL like http://api:8000/metrics every 15 s.Long-lived process with a reachable IP + port.
Elasticsearch (via Filebeat)Log tail: Filebeat reads *.log files, ships to ES.Local filesystem where the process writes logs.
GrafanaViewer: dashboards over any data source.Data is somewhere queryable.
PostHogPush: browser calls posthog.capture(...), ships to PostHog HTTP endpoint.Runs in a browser, no local storage required.
SentryPush: browser catches unhandled exceptions, ships stack trace + context to Sentry.Runs in a browser, source-maps hosted separately.

A Mini App running in the user's browser has no long-lived process, no reachable port, and no filesystem. Prometheus can't scrape it. Filebeat can't tail it. You can invert Prometheus (Pushgateway) — but Pushgateway is designed for short-lived batch jobs, not per-user client sessions, and it collapses everything into a single time-series indistinguishable across users.

PostHog and Sentry are built the other way around: the browser initiates. Both handle the specifics of the client environment — capturing session context, source-mapping stack traces back to TypeScript, dealing with browser noise (extension errors, network aborts on tab close). Reinventing this on top of ELK is a multi-month project that ends less good.

How the two stacks join

Every request the client makes carries an X-Request-Id header — a UUID generated client-side per user gesture. The API middleware logs it on every request; PostHog and Sentry tag every event with the same UUID.

Result: any incident is triangulable across the split.

User: "Today screen shows an error"
    │
    ▼
PostHog:  distinct_id=ab12cd34-...  screen=today  event=screen_error  request_id=abc123  ts=14:23:07
    │
    ▼
Sentry:   request_id=abc123  TypeError in useDueList.ts:15  release=v1.2.3
    │
    ▼
Grafana:  /api/v1/conspectuses/due  p99 latency spike at 14:23  error_rate 12%
    │
    ▼
Kibana:   request_id=abc123  psycopg.OperationalError: connection timeout
    │
    ▼
Root cause: Postgres connection pool exhausted at 14:23 due to a slow migration lock.

Four tools, one request_id, four minutes from user report to root cause. The distinct_id ties every event to a users.client_uuid; Telegram-side identifiers stay server-side — see the identity model on event-spec.

One dashboard, four sources

Grafana is source-agnostic. Instead of standing up separate UIs, we compose:

  • «API health» dashboard — Prometheus data.
  • «Client analytics» dashboard — PostHog data (via the Grafana PostHog plugin or PostHog's HTTP API).
  • «Errors end-to-end» dashboard — Sentry + Elastic side by side, filterable by request_id.

Sentry and PostHog also have their own product UIs — better than Grafana at what they specialise in (Sentry: exception-triage workflow; PostHog: funnel-building UX). Grafana is the on-call landing page; the specialised UIs are drill-downs.

«What if we just used X for both?»

The two most-asked counter-proposals:

«Skip PostHog + Sentry, ship client logs to Elastic»

Technically possible — add an HTTP endpoint /api/v1/client-logs that accepts JSON, write a client-side logger. Then reinvent: session replay, source-mapping, funnels, cohort analysis, browser-noise filtering, source-map hosting, release tracking. Estimated 2-3 months to reach parity with what PostHog and Sentry give you in a day. Result is worse across every dimension except «one fewer vendor».

«Skip Prometheus / ELK, use PostHog for the backend too»

Doesn't work at all. PostHog is designed to observe user behaviour, not process health. There's no path to «scrape /metrics» or «join API logs to a request» through PostHog. Backend needs the backend stack; this proposal breaks on architecture.

Cost of the split

  • Two vendors. PostHog (client analytics) + Sentry (client errors) are external SaaS. Both have free tiers that cover a small tester cohort; paid tiers if usage grows.
  • Two on-call habits. «Check Grafana first» + «check Sentry when the console screams». Documented in on-call.
  • Request-id discipline. Every request must carry X-Request-Id; every log must record it; every PostHog and Sentry event must tag it. Enforced by the shared API client — never optional.

The cost is real. The alternative — pretend one stack fits both, then patch around the mismatch — is worse in every dimension that matters.
