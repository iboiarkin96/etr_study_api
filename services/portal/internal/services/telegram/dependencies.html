Dependencies

Everything this service reaches for at runtime, at build time, or through its host platform. Grouped by what breaks if the dependency goes away, not by where it lives in package.json. Blast radius is stated honestly per row — no «critical» / «non-critical» hand-waving.

Runtime dependencies

What must be alive while a user has the Mini App open. Each row is «if this goes away, this specifically stops working».

Dependency Role Blast radius if unavailable
services/api · /api/v1 The only backend surface the client consumes. Cold open first hits POST /api/v1/auth/telegram to exchange initData for a Bearer JWT; every subsequent screen is a view over the /api/v1/* contract. Total. If the API is unreachable the auth bootstrap itself fails — the app shell renders (offline-friendly HTML) but every screen shows an error toast on first fetch. No degraded mode — read-only cache is not in MVP.
PostgreSQL (via API) Persistence for every read and write. Also stores the two identity rows minted at cold open: one in users, one in telegram_users keyed on telegram_user_id. Client never touches Postgres directly. Same as API — surfaced as a 500-family response with an error toast. Retries are automatic (TanStack Query) but bounded.
Telegram WebApp platform The runtime host. Supplies themeParams, viewport, hapticFeedback, back/main/settings buttons, deep-link payload. None in dev (browser tab with mockTelegramEnv()). In prod: no Telegram = no launch. Not a component we can page.
Telegram Bot API (via services/api) Delivers the daily push and hosts the launch button (/start). Client keeps working for the user who already has the app open. New sessions can't launch — no push, no button.
Cloudflare Pages CDN Serves the static bundle over HTTPS. Auto-invalidates on new deploy. Cold opens fail (no bundle to download). Sessions already loaded keep working from browser cache until reload.

Build-time dependencies

Live in devDependencies. Present on the dev machine and in CI; never shipped in the bundle.

Package Role Failure mode
Node.js Runs Vite, tsc, ESLint. Required on dev machine + CI runner. Not installed → make targets exit at which node. Rare in practice — Homebrew / mise / nvm cover it.
vite + @vitejs/plugin-react Dev server (HMR) + production bundler. Config errors surface immediately at make dev / make build; no silent failures.
typescript Strict type-checking via project references (tsconfig.app.json + tsconfig.node.json). make typecheck fails in CI before merge. Never a runtime concern.
eslint (flat config) + typescript-eslint + react-hooks + react-refresh plugins Style + pattern gate. make lint fails in CI. Auto-fixable rules run via make lint-fix.
@types/react, @types/react-dom, @types/node Type definitions. Type errors caught at compile-time; never a runtime concern.

Upstream contracts we assume unchanged

Contracts owned by other services / vendors that this service reads as fixed points. Change to any of them is a coordinated cross-service PR, not a unilateral upgrade.

  • OpenAPI canon at services/api/openapi.json — every generated client type comes from here. Drift is caught by make openapi-check. If the API renames a field, the generated client won't compile → immediate loud failure at make typecheck, not a silent runtime bug.
  • Telegram initData signature scheme — secret_key = HMAC_SHA256(key="WebAppData", msg=bot_token); the hash field is then compared server-side against HMAC_SHA256(secret_key, data_check_string). Owned by Telegram; monitored via their Bots API changelog. A silent change here would break the auth bootstrap for every client.
  • Telegram WebApp protocol — window.Telegram.WebApp shape (themeParams, viewport, hapticFeedback, back-button, main-button). Owned by Telegram; monitored via the SDK release notes.
  • Cloudflare Pages build API — the build.command + build.output contract. Change to it is rare; deploy config lives in ops/.

Who consumes this service

Zero other services. The Mini App is a leaf in the internal graph — it takes from services/api and delivers to the user. Nothing in the monorepo imports its bundle or its source. Removing this service breaks the Telegram surface only; the API, portal, monitoring, datastore and ui-kit are unaffected.
