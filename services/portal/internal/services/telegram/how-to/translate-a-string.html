Add or update a translation

Every user-facing string in the Mini App goes through i18next. This recipe covers the three-file change: add the key in both locales, reference it in code, verify by flipping language_code. ~2 minutes.

Before you start

  • Locale files live at services/telegram/src/locales/{ru,en}/*.json. One JSON per screen (or shared namespace).
  • Every key exists in both ru and en. Missing key in either = build fails.
  • User's locale is chosen from Telegram's WebApp.initDataUnsafe.user.language_code. Fallback is en.

Steps

  1. Add the key to both locale files. Example: add a «star» tooltip to the Today screen.

    src/locales/en/today.json

    {
  "due_cards": {
    "title": "Due today",
    "star_tooltip": "Star to keep visible"
  }
}

    src/locales/ru/today.json

    {
  "due_cards": {
    "title": "К повторению сегодня",
    "star_tooltip": "В избранное — не уйдёт из списка"
  }
}
    Keep the key structure identical across locales. Nested keys are namespaced by screen; shared strings live in common.json.
  2. Reference the key in the component. 
    import { useTranslation } from 'react-i18next';

export function StarButton() {
  const { t } = useTranslation('today');
  return <button title={t('due_cards.star_tooltip')}>⭐</button>;
}
    Never hardcode a user-facing string. Even placeholders like «TODO» must go through t() so the missing translation surfaces during review.
  3. Verify in the browser. With the dev server up, open the app. Then in DevTools console: 
    window.i18next.changeLanguage('ru');
window.i18next.changeLanguage('en');
    The tooltip should flip between the two languages instantly. This confirms the key resolves in both.
  4. Run the local gate. 
    make -C services/telegram verify
    A build-time check confirms both locale files have the same set of keys (guard against dropped translations).

Verify

  • Switching window.i18next.changeLanguage(...) in DevTools flips the string between ru and en in-place — no missing-key placeholder.
  • make -C services/telegram verify passes: the parity check confirms both locale files carry the same key set.
  • Storybook renders the string correctly in both language stories (if a story exists for the component).

Style rules

  • No format strings for numbers or dates. Use Intl.NumberFormat and Intl.RelativeTimeFormat. i18n keys handle labels, not formatting.
  • Interpolation via {{name}}. Example: "greeting": "Привет, {{name}}", then t('greeting', { name: 'Иван' }).
  • Pluralization via _one / _other keys. Example: "cards_one": "{{count}} карточка", "cards_other": "{{count}} карточек".
  • No HTML in translations. Rich formatting is composed in code, not in JSON.
  • Keep RU concise. Russian tends to be ~15 % longer than English; if a phrase overflows, shorten the RU version rather than expand the layout.

Troubleshooting

SymptomFix
Key renders as literal string («today:due_cards.star_tooltip»).Namespace is wrong or file not registered. Check src/i18n.ts — every new locale JSON must be added to the resources list.
Build fails: «Missing key foo in en».You added it in one locale but not the other. Add it, keep the same nested path.
Text renders in English on a Russian Telegram.language_code from initData isn't reaching i18next. Check src/i18n.ts reads WebApp.initDataUnsafe.user.language_code. In dev, mockTelegramEnv() defaults to en.
Long RU translation overflows the button.Shorten the RU. Don't expand the button (breaks the design token). If shorter isn't possible, ellipsize with a title-attribute tooltip.
