Run the Mini App locally

Cold-start recipe: from a fresh clone to the app open in Chrome at http://localhost:5173, connected to a local API. Plain browser tab — no tunnel, no phone, no real Telegram. The API still requires a Bearer JWT (per architecture · request path), so tools/dev/sign_init_data.py mints a synthetic initData against the local TELEGRAM_BOT_TOKEN and the client feeds it to POST /auth/telegram the same way real Telegram would. That's the primary dev loop.

Before you start

  • Node.js 20+ installed. Check: node --version. Get it via Homebrew (brew install node), mise, or nvm.
  • Repo cloned and .venv set up (make setup from repo root).
  • API running locally at http://localhost:8000. Bring it up with make up from the repo root (starts Postgres + API + monitoring via docker compose). Verify: curl -s http://localhost:8000/ready returns 200.
  • TELEGRAM_BOT_TOKEN exported in the shell you'll run the signer from. Same value the API reads from env/dev. Pick anything shape-compatible for local work (e.g. 1234567890:AAAA-fake-bot-token-for-tests-only); the token only exists to sign / verify the same initData, no real Telegram round-trip involved.
  • A signed initData ready. Mint one on demand with python tools/dev/sign_init_data.py --user-id 42 --first-name Dev. Use --format env to get a VITE_DEV_INIT_DATA=… line you can drop straight into services/telegram/.env.local. The scaffold will consume that env var once T-12 wires the auth bootstrap (W3).

Steps

  1. Install dependencies. 
    make -C services/telegram install
    Runs npm ci against package-lock.json. First run takes ~2 min (installs 250+ packages into services/telegram/node_modules/). Subsequent runs are near-instant if nothing changed.
  2. Start the dev server. 
    make -C services/telegram dev
    Vite starts on http://localhost:5173 and prints a URL. Hot module replacement is on — edits to any .tsx reflect in the browser in ~100 ms without a full reload.
  3. Open the URL. Any modern browser works — Chrome is the reference target because DevTools and React DevTools are best there. The scaffold page renders «study_app · Mini App scaffold» in dark mode.
  4. Verify the auth handshake. Once the client bootstrap wires up (W3 · T-12), DevTools → Network on a fresh load should show POST /api/v1/auth/telegram return 200 with a JWT body, followed by /api/v1/* calls carrying Authorization: Bearer <jwt>. If step (a) returns 401, jump to Troubleshooting. If (a) is 200 but (b) is 401, also Troubleshooting.

    Today (2026-07-19): the scaffold at services/telegram/src/ renders a static page and makes no API calls — the auth bootstrap + generated client land in T-12. Steps 1–3 will still work; step 4's Network-tab check becomes real once T-12 ships.

The daily dev loop

Once installed, this is what «open the Mini App to work on it» looks like:

make up                        # (from repo root) start API + Postgres + monitoring
make -C services/telegram dev  # (in another terminal) start Vite

Open http://localhost:5173 in the browser and edit files in services/telegram/src/. Changes reflect immediately. Kill both with Ctrl-C; kill the docker side with make down from repo root.

Verify

Run the local gate. Same commands CI runs — pass locally = pass in CI.

make -C services/telegram verify   # typecheck + lint + build

Individual steps if you want to run them one at a time:

make -C services/telegram typecheck   # tsc --noEmit
make -C services/telegram lint        # eslint .
make -C services/telegram build       # vite build → dist/

Troubleshooting

SymptomFix
Port 5173 already in use. Kill the process holding it (lsof -i :5173 then kill) or launch on another port: make -C services/telegram dev VITE_PORT=5174.
Every API call returns 401 Unauthorized. The auth bootstrap failed. Open DevTools → Network → /auth/telegram. If it 401s, TELEGRAM_BOT_TOKEN differs between the API and the dev signer at tools/dev/sign_init_data.py — align them. If /auth/telegram is 200 but subsequent calls 401, the JWT in CloudStorage is stale after a JWT_SECRET rotation; clear WebApp.CloudStorage from DevTools and reload.
Browser console: «CORS policy: No 'Access-Control-Allow-Origin'». The API's CORS_ALLOW_ORIGINS doesn't include http://localhost:5173. Check env/dev. Restart the API after any change.
make dev exits with «Cannot find module 'react'» or similar. node_modules/ is missing or corrupt. Run make -C services/telegram install again. If still failing: rm -rf services/telegram/node_modules services/telegram/package-lock.json && npm install.
Page loads but is blank; DevTools console shows a JS error. Copy the error, look at the stack trace, find the file. In dev, Vite maps errors back to the source. If the error is in a dependency, check the browser cache — try Shift-Reload or open a new incognito window.
make typecheck or make lint fails after git pull. Someone bumped a dep. Run make -C services/telegram install to sync node_modules/ with the updated lockfile.
