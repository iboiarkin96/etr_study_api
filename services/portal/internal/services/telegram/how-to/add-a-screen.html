Add a new screen

End-to-end recipe: add a screen that has a route, hits an endpoint, renders loading / empty / error / success states, and ships a smoke test. Uses «Bookmarks» (/bookmarks) as the running example — a hypothetical screen that lists starred conspectuses.

Before you start

  • Design mock approved (or absent — the mock can follow, per Milestone 1).
  • Screen entry added to Screens and — if it has a URL — to Route table. Docs first, then code.
  • Endpoint documented in the API canon (services/api/openapi.json). If a new endpoint is needed, that's a separate PR against services/api.

Steps

  1. Create the screen directory. 
    services/telegram/src/screens/Bookmarks/
├── index.tsx           # <BookmarksScreen /> — top-level React component
├── useBookmarks.ts     # data hook (wraps generated API client + TanStack Query)
├── BookmarkRow.tsx     # per-row presentation
└── Bookmarks.stories.ts  # Storybook stories (loading / empty / error / success)
    One directory per screen; nothing shared with other screens goes here — put shared logic in src/shared/.
  2. Wire the route. Add to src/app/router.tsx: 
    createRoute({
  path: '/bookmarks',
  component: BookmarksScreen,
})
    TanStack Router is type-safe — TypeScript now knows about the new route. Deep-linking to it works immediately.
  3. Declare the Telegram chrome preset. Add to src/app/shell.tsx: 
    '/bookmarks': {
  back: 'today',            // Back → Today
  main: 'hidden',
  settings: 'profile',
}
    See Telegram chrome for the vocabulary. The shell auto-applies this on route change.
  4. Fetch data via the generated API client. In useBookmarks.ts: 
    import { useQuery } from '@tanstack/react-query';
import { api } from '@/shared/api';

export const useBookmarks = () =>
  useQuery({
    queryKey: ['bookmarks'],
    queryFn: () => api.GET('/api/v1/conspectuses', { params: { query: { starred: true } } }),
  });
    The generated client already carries Authorization: Bearer <jwt> (from the auth bootstrap on cold open) and, for mutations, Idempotency-Key. Never bypass it — a hand-rolled fetch is a review block.
  5. Render four states — always. Loading / empty / error / success. In index.tsx: 
    const { data, isLoading, error } = useBookmarks();
if (isLoading) return <BookmarksSkeleton />;
if (error) return <ErrorToast onRetry={refetch} />;
if (!data?.length) return <EmptyState illustration="bookmark" tip="Star a conspectus to bookmark it." />;
return <ul>{data.map(c => <BookmarkRow key={c.id} conspectus={c} />)}</ul>;
    A screen without an empty state is a review block. See Screens — every entry lists its states.
  6. Add i18n keys. Every user-facing string goes through t(). Add to both src/locales/{ru,en}/bookmarks.json. See Add or update a translation.
  7. Ship a smoke test. Playwright, one test per new screen: 
    test('bookmarks screen', async ({ page }) => {
  await page.goto('/bookmarks');
  await expect(page.getByRole('heading', { name: /bookmarks/i })).toBeVisible();
});
    Runs under make -C services/telegram test:e2e.
  8. Add Storybook stories. One per state (loading / empty / error / success). Storybook is the visual QA loop — don't test states in the browser by mangling MSW every time.
  9. Update the design docs. Add the screen row to screen matrix and the full per-screen section. Add the route to route table.
  10. Run the local gate. 
    make -C services/telegram verify
    Typecheck + lint + build all green → open a PR.

Verify

  • [ ] Route added to router.tsx.
  • [ ] Chrome preset added to shell.tsx.
  • [ ] Data hook uses the generated API client (no hand-rolled fetch).
  • [ ] All four states render (loading / empty / error / success).
  • [ ] Every string goes through t(); ru + en keys added.
  • [ ] Playwright smoke test lands.
  • [ ] Storybook stories cover all states.
  • [ ] Screens and Information architecture updated.
  • [ ] make -C services/telegram verify green locally.
