What this service is

A Telegram-native client for the ETR daily loop. Not a general-purpose app — a purpose-built surface for one habit: open, review what's due, log what went wrong, close.

The service is a React + TypeScript bundle that Telegram opens as a WebApp when the user taps «Open study_app» in the bot. Nine screens sit on top of the shipped /api/v1 surface — no separate BFF, no new persistence, no server-rendered pages. Every write carries an Idempotency-Key so a lost 4G reply never doubles an action.

Multi-user by design. Every open verifies the Telegram user's signed initData against TELEGRAM_BOT_TOKEN, upserts on telegram_user_id, and mints a 24 h JWT. Every /api/v1/* request from the client thereafter carries Authorization: Bearer <jwt>; the API resolves the caller to a real users row so schedule, conspectus and error data isolate cleanly per person. Public onboarding, conspectus editing and moderation stay out of scope — they are follow-on epics, not gate-keepers.

The request path — which endpoint each screen hits, what triggers what, where the branches sit — is documented once in the API reference: see the business flow & UML sequence on the api service page. This service does not duplicate it.
