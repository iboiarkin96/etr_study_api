01 · Today

Purpose: the daily open. What's due for review right now, with the fewest possible clicks to start.

Route: /. Landing screen. Also the destination of the back-button from every other screen.

Endpoints: GET /api/v1/conspectuses/due, GET /api/v1/schedule/summary, POST /api/v1/conspectuses/{id}/actions/review.

Primary action: tap a due card → jump into Focus for that card.

Secondary actions: swipe card (4 intents — see gesture vocabulary), open Schedule via header, open Profile via corner avatar.

States:

  • loading — content-aware skeleton: 3 due-card silhouettes + heat-map stub.
  • empty — «All caught up for today» + rotating tip + next-review-in copy.
  • error — toast with «Retry»; skeleton stays visible so retry lands in the same layout.
  • success — Yesterday-digest strip, streak ProgressRing, Recently-reviewed carousel, 90-day HeatmapCalendar, due list.
  • edge — 100+ due cards → virtualised list; day with 0 due but ≥1 upcoming → «Nothing due, but 3 land tomorrow».

Mock: today.html

02 · Focus

Purpose: the review flow. Full-screen card → tap to reveal → grade → next. Nothing else on screen — no navigation chrome, no distractions.

Route: /focus. Session-scoped state (progress, streak count) held in a Zustand store; leaves scope when the user exits.

Endpoints: POST /api/v1/conspectuses/{id}/actions/review per grade.

Primary action: grade the current card — Again / Hard / Good / Easy.

Secondary actions: undo last grade (5 s window), skip card (marks as «not today» tag), open the underlying conspectus in read-only, exit → session summary.

States:

  • loading — first-card spinner (only once per session).
  • empty — «No cards to review» → route back to Today.
  • error — grade fails → inline toast + button re-enables; card stays.
  • success (per card) — reveal → grade → interval preview flies up → next card slides in.
  • session end — «You reviewed N cards, streak +1» → button to Today.

Mock: focus.html

03 · Conspectus list

Purpose: browse everything you've ever noted, chronologically.

Route: /conspectus. Cursor pagination — one loads N, scroll-end loads the next.

Endpoints: GET /api/v1/conspectuses with cursor.

Primary action: tap a row → detail.

Secondary actions: long-press → context menu (open in Focus, archive, share).

States: loading (skeleton rows), empty («No conspectuses yet» + link to authoring), error (retry chip inline), success, edge (very-long-title row truncates with ellipsis + tooltip on hover).

Mock: conspectus.html

04 · Conspectus detail

Purpose: read one conspectus. Cue-sheet + bullets, formatted for scanning.

Route: /conspectus/:id. Deep-linkable — startapp=conspectus_<uuid> lands here directly (bypasses Today).

Endpoints: GET /api/v1/conspectuses/{id}, GET /api/v1/conspectuses/{id}/history.

Primary action: «Review now» → Focus for this card.

Secondary actions: AI micro-actions (Explain differently, Generate quiz — stubs in MVP), view history.

States: loading (skeleton with cue-sheet grid), empty (impossible — 404 shows «Not found» card), error (retry inline), success.

Mock: conspectus.html

05 · Schedule

Purpose: «when's the next wave?» — the 90-day pipeline at a glance.

Route: /schedule.

Endpoints: GET /api/v1/schedule/summary — server returns the pre-aggregated per-day counts. Contract: no client-side aggregation.

Primary action: tap a day cell → filtered due-list for that day.

Secondary actions: switch view (week vs month), scroll to today.

States: loading (skeleton heat-map), empty («Nothing scheduled» — rare), error, success, edge (day with 100+ due → single darkest tone; hover shows exact count).

Mock: schedule.html

06 · Errors

Purpose: log recall failures with context, browse the timeline to spot patterns.

Route: /errors. Reachable from Focus (inline «Log this error» when a grade is Again).

Endpoints: GET /api/v1/errors, POST /api/v1/errors (with Idempotency-Key).

Primary action: «+ Error» → inline compose (source conspectus pre-filled if coming from Focus).

Secondary actions: group by pattern (auto-cluster by conspectus), filter by date range.

States: loading, empty («No errors logged — that's good or you're not honest»), error, success (timeline + optional grouped view).

Mock: errors.html

07 · Search

Purpose: jump to any conspectus by title.

Route: /search — but usually reached via Cmd+K (desktop) or pull-down (mobile), overlay pattern.

Endpoints: GET /api/v1/conspectuses?q=… (server-side title-prefix match).

Primary action: type → results narrow live → Enter or tap → detail.

Secondary actions: arrow keys navigate results, Esc closes.

States: idle (recent-searches or hints), typing (debounced 200 ms), no-match, error, results.

Mock: search.html

08 · Profile

Purpose: the «you» screen. Achievements, weekly retention chart, entry points to settings.

Route: /me.

Endpoints: GET /api/v1/schedule/summary (for the retention chart), stats derived client-side from cached data.

Primary action: none — this screen is glanceable, not actionable.

Secondary actions: open settings (Telegram SettingsButton), toggle language, sign out (destructive — BiometricManager confirm).

States: loading (skeleton chart), success. No empty state — every user has at least a start date.

Mock: profile.html

09 · Onboarding

Purpose: first-run only. Explain the loop in three cards, land the user on Today.

Route: /onboarding. Auto-shown when WebApp.CloudStorage.getItem('onboardingDismissed') resolves to nothing. CloudStorage — not localStorage — so the dismissal state syncs across every device the same Telegram account opens the app on, and the memory-only cache rule (per architecture · request path) stays intact.

Endpoints: none — pure UI. CloudStorage read/write goes through the Telegram SDK, not /api/v1.

Primary action: «Get started» on the last card → route to Today, write CloudStorage['onboardingDismissed'] = '1'.

Secondary actions: «Skip» at any step (same dismissal write), navigate cards via swipe.

States: three static cards. No loading, no empty, no error — nothing to fetch.

Mock: onboarding.html
