Design

The design contract for the Mini App — what the nine screens are, how they're arranged, how motion and haptics reinforce the loop, how deep-links navigate. Visual language and component primitives live in the TMA design section of the UI Kit; this section is the client-side product-design layer that composes those primitives into a real product.

Product principles

Five constraints that decide every screen-level trade-off. If a proposed screen violates one, the screen changes — not the principle.

① One primary action per screen

Every screen has exactly one thing the user came to do. Today = review what's due. Focus = grade the card. Errors = log an error. Secondary actions are one tap away but never compete visually with the primary. Habit products win or lose in the first two seconds — that budget is spent on the primary action.

② Optimistic by default

Every mutation renders as if it succeeded, then rolls back on error. The user never waits on the network for feedback. Idempotency-Key on every write guarantees retries are safe; a lost 4G reply never doubles an action.

③ Native Telegram, not a website

The app uses Telegram's themeParams, hapticFeedback, back/main/settings buttons, and safe-area insets. The user should feel Telegram — same colours as their chats, same button-bar as the platform — not a webpage that happens to run inside it.

④ Motion tells the truth

A card swiped right doesn't just disappear — it exits with a small acknowledgment. A grade in Focus doesn't just advance — it shows the next-interval preview flying up. Motion earns its milliseconds by making the state change legible. prefers-reduced-motion collapses everything to instant with no functional loss.

⑤ Silence over noise

No badges. No toast for every action. No confetti except at real milestones (streak 7 / 30 / 100). Empty states are calm. Success is invisible — the card leaves the list and that's the confirmation. The user comes back to a habit tool, not a slot machine.

How this section composes

Four pages, each answering one question:

Screens — «what are they?»

Nine-screen catalog. Per screen: purpose, endpoints hit, states (loading / empty / error / success / edge), primary + secondary actions, links to the mock in the UI Kit and the reference in this catalog. The complete inventory a new contributor scans before touching code.

Interactions — «how do they feel?»

Motion vocabulary (7 spring configurations), haptics vocabulary (7 patterns), gesture vocabulary (4 swipe intents on cards), Telegram chrome (back-button / main-button / settings-button binding rules). Consumed by every screen; centralised here so screens don't reinvent.

Information architecture — «how do they connect?»

Sitemap, route table, deep-link grammar (startapp=… payload → route), navigation model (when back returns to the previous screen, when it returns to Today, when it closes the WebApp).

What lives here vs in the UI Kit

Two design surfaces, two audiences. The split matters because they change on different clocks and one shouldn't be blocked by the other.
