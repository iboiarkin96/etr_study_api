HEART Framework — user experience metrics that matter

Google's five-dimension framework for selecting UX metrics that reflect real user experience — not just activity signals that look good on a dashboard.

TL;DR

HEART (Google, Kerry Rodden, 2010) is a UX metrics framework with five dimensions: Happiness (user satisfaction), Engagement (frequency/depth of use), Adoption (new user uptake), Retention (returning user rate), and Task Success (efficiency and error rate). For each dimension the team defines Goals, Signals, and Metrics (GSM). The framework prevents teams from measuring only what is easy to measure and forces deliberate choices about which user experience dimensions matter for the product.

What it is

HEART was developed at Google by Kerry Rodden, Hilary Hutchinson, and Xin Fu and published in 2010. It defines five dimensions of user experience quality:

  • Happiness: Attitudinal metrics — how satisfied users are. Measured through surveys (CSAT, NPS, ease-of-use rating). The only dimension that requires primary research rather than log data.
  • Engagement: Depth and frequency of interaction. Sessions per week, actions per session, content consumed. Measures how much users actually use the product.
  • Adoption: New user uptake and feature discovery. New users per period, percentage of users who have tried a new feature, time to first key action.
  • Retention: Whether users come back. 30/60/90-day retention curves, churn rate, reactivation rate. The most direct leading indicator of LTV.
  • Task Success: Efficiency and error rate on core tasks. Completion rate, error rate, time-on-task. Often measured through usability testing or log analysis.

For each dimension, the team uses the Goals-Signals-Metrics (GSM) process: define the goal (what aspect of user behaviour matters?), identify signals (what user behaviour indicates success or failure?), and choose the metric (how do we quantify that signal in available data?).

Why we need it

Without a framework, teams default to measuring what is easy to measure: page views, signups, and click-through rates. These are not bad metrics — they are simply incomplete. A feature can increase page views while simultaneously increasing user frustration (Engagement up, Happiness down). HEART prevents this by requiring deliberate selection of which dimensions matter for each product goal and explicitly measuring the selected dimensions.

The GSM process is particularly valuable because it prevents metric confusion. "User satisfaction" is a goal, not a metric. "Average CSAT score after checkout" is a metric. The process of translating goals into signals into concrete metrics eliminates ambiguity about what will be measured and why.

How others do it

Google

Google uses HEART across multiple product areas including Google Docs, Gmail, and Google Maps. For Google Docs, the primary HEART dimensions are Engagement (documents edited per user per week), Retention (weekly retention rate), and Task Success (documents successfully shared and collaboratively edited). Happiness is measured quarterly through in-product surveys. Adoption tracks new feature uptake (e.g., comments, suggestions mode).

What we take: Not all five HEART dimensions are equally important for every product. Google Docs focuses on E, R, and T — the dimensions most predictive of collaboration value. Selecting two to four relevant dimensions per product focus area, rather than always measuring all five, makes HEART tractable.

Source: Google Research — HEART paper

Spotify

Spotify's squad metrics process uses HEART as the starting framework: each squad selects the two HEART dimensions most relevant to their mission and defines GSM for each. Discovery squads focus on Adoption (discovery of new music) and Happiness (satisfaction with recommendations). Listening squads focus on Engagement (minutes per session) and Retention (weekly return rate).

What we take: Different squads in the same product can legitimately focus on different HEART dimensions. What matters is that the selections are deliberate and connected to the squad's mission — not that every squad uses the same dimensions.

Source: Spotify Engineering Blog

Anatomy — GSM process

Select dimensions

Choose 2–4 HEART dimensions most relevant to your product's current focus. Not all five. Write down why the others were excluded.

Goals

For each selected dimension, write the goal in user-outcome terms: "Users can find a relevant document within 30 seconds." Goals are not metrics.

Signals

What user behaviours indicate progress toward the goal? "User finds a document by search rather than scroll." Signals are observable, not yet quantified.

Metrics

Quantified version of the signal using available data: "% of document opens that came from search results." Metrics must be measurable in existing analytics.

Counter-metrics

For each primary metric, define what must not get worse. Prevents local optimisation at the expense of adjacent dimensions.

Common pitfalls

PitfallWhy it happensWhat to do instead
Measuring all five dimensions with equal weightTeam doesn't want to exclude any dimension.Select 2–4 dimensions per product focus. All five simultaneously is analytically unmanageable. Make explicit trade-off choices.
Goals written as metricsJumping to measurement before defining what success looks like.Force the three-step GSM sequence. If you write a metric before a goal, reverse it and ask "what user behaviour would this metric reflect?"
Happiness measured only once a yearSurveys are costly and the team deprioritises them.Embed lightweight happiness signals in the product: post-task satisfaction prompts, periodic in-app micro-surveys. One question per session beats one annual survey.
