Tester onboarding — your first verified ticket

A 5-day path for a new tester. The goal is to feel the full QA loop end-to-end — orient → run a checklist → find a defect → file it → verify the fix — on real artefacts in this repo. Follow the days in order; each one ends with a concrete deliverable, not a reading assignment.

📅 5 days 📚 First-time onboarding 🧪 QA portal

What this is not. Not a reference for severity grading, not a playbook for API or accessibility testing. Once the loop feels natural, move on to the QA how-to playbooks for the real recurring work.

– Before you start

Day 1 assumes everything below is already in place. None of these belong inside the 5-day loop — they're one-time setup.

  • Repo access (read at minimum; contributor if you want to file PRs).
  • Local toolchain — follow local development. Confirm make, Python 3.x, Docker, and a Chromium-based browser run.
  • A defect tracker you can post issues to (GitHub Issues by default).
  • Browser helpers: a JSON-formatted view, an HTTP client (Postman / Insomnia / httpie), and an annotating screenshot tool.

1 Orient yourself

Three things you need to know before Day 2 — read them in this order, then stop. Everything else lives in "Bookmarks to keep open" at the bottom of this page; you'll pull it in when you actually need it.

  1. Test strategy — what we test, what we don't, and why. Deliverable: in one sentence, what is in scope for this product.
  2. Severity matrix. Deliverable: the one-line definition of Severity 1, 2, 3, 4 in your own words.
  3. Defect lifecycle. Deliverable: the state diagram on paper or in a note — you'll use it on Day 4 and Day 5.

✓ Check your progress

You can answer, without re-opening the pages: "what does Severity 2 mean here?", "what state follows Reopened?", and "what is out of scope for our test strategy?". If any answer is fuzzy, re-read just that page — do not pull in more reading on Day 1.

2 Run the app locally

  1. Follow local development to start the API on its default port.
  2. Open Swagger UI at /openapi/index.html (served by the docs portal) and issue a GET to a read endpoint.
  3. Open the docs portal index and click around — use the search box, switch theme, open the drawer, scroll to bottom.
  4. Run the test suite once:
make test
# or
        pytest

Note how long it takes — you'll come back to this baseline.

3 Run your first checklist on a real docs page

The cheapest first target is a docs page on your local portal — no API state to set up, easy to re-run, and the checklist is fully owned by us.

  1. Open the portal locally (it is served alongside the API) and pick one docs page you have never seen before.
  2. Open checklist 0001 — documentation pages visual checklist.
  3. Walk through every row against that page. Take a screenshot for each section the checklist requires.
  4. Record pass / fail / needs-follow-up against each row in a notebook or ticket.
  5. If a row is ambiguous, stop and ask — ambiguity in a checklist is a defect against the checklist itself, not against you.

✓ Check your progress

You have one annotated checklist with screenshots attached. That is a real QA artefact — keep it. If everything passed, your Day-4 defect will come from exploratory testing instead.

4 File your first defect

  1. Find a defect. If checklist 0001 came back clean, run a 60-minute exploratory session against the docs portal. Use this charter verbatim — you don't need to open the full how-to today:
    • Mission: tab order and keyboard reachability on internal portal pages.
    • Time box: 60 minutes, single browser, no extensions.
    • Capture: every focus stop, every trap, every missing visible focus ring.
    The full method is in exploratory testing — open it after Day 5 when you'll run another session.
  2. Reproduce the defect three times. If you cannot, note the conditions and either keep digging or file as Severity 3 — intermittent, needs reproduction.
  3. Copy bug report template.
  4. Grade severity using the severity matrix.
  5. Attach evidence — annotated screenshot, console log, request/response capture.
  6. File the defect. Assign yourself as reporter; leave assignee blank for triage (we don't have a CODEOWNERS file yet — see Day 5 for the handoff).

5 Close the loop

The point of Day 5 is to feel the back-half of the defect lifecycle — verify → close. Don't wait on a developer fix; author the smallest possible patch yourself and verify against your own build. Real QA often waits on a dev; the tutorial doesn't, so the loop closes today.

  1. Patch the defect yourself. Most Day-4 defects are typos, missing aria attributes, or a one-line CSS tweak — within reach for day five. Open a PR with the smallest fix and mark the defect Pending verification.
  2. Verify against your own PR build. Re-run the exact reproduction from Day 4. Pass → move the issue to Verified; fail → Reopened with new evidence. Use the defect lifecycle diagram you drew on Day 1.
  3. See a release sign-off. Walk checklist 0003 solo against the current main build. (If a senior tester happens to be running one this week, shadow them — the cadence is identical.)

✓ Shortcut — defect already fixed?

If someone landed a fix for your defect before you got here, skip step 1: re-run the Day-4 reproduction against the new build and jump straight to step 2 to move the issue to Verified.

✓ You're done

You ran the app, oriented yourself in the references, ran a real checklist on a real page, filed a defect from your own evidence, patched and verified it, and walked a release sign-off. That is the full QA loop — orient → setup → checklist → defect → patch → verify — and every recurring piece of QA work in this repo is a specialisation of it.
