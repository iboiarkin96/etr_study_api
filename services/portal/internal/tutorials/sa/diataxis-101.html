Diátaxis 101 for this portal

Open one page of each Diátaxis mode in this portal and answer four questions about each. The point is to feel the IA of the docs with your own hands so future you knows where to write next. The deliverable is a short cheatsheet you'll keep for yourself.

⚙️ 4 phases ⏱ 20–30 min 📚 First-time 📝 Paper-only

– Before you start

  • Portal serves locally on http://127.0.0.1:3000.
  • One scratch markdown file you'll create as you go — no other tools.

0 Goal

  • One cheatsheet at services/portal/internal/scratch/diataxis-cheatsheet.md with one row per Diátaxis mode (Tutorial, How-to, Reference, Explanation).
  • For each row, four answers: reader, verb, shape, portal example.
  • By the end, you can say in one sentence what each mode is and where it lives in this repo.

The four-question rubric is borrowed from the Diátaxis framework's own summary table — re-deriving it from real pages is what makes it stick.

1 Tutorial — open the dev onboarding

Visit SWE · Your first endpoint. Answer in your cheatsheet:

  • Reader: who is this for? (someone learning, not yet productive).
  • Verb: what does it ask you to do? (follow steps, not solve a problem).
  • Shape: prereqs → goal → numbered phases with ✓ checks → "you're done".
  • Where it lives: internal/tutorials/<role>/....

✓ Check your progress

Row 1 of your cheatsheet has all four cells filled. The "shape" cell mentions the phase-with-checkpoint structure — not a generic "intro, body, conclusion".

2 How-to — open a recipe

Visit How-to · Local development (or any recipe under internal/how-to/). Answer:

  • Reader: someone who already knows what they want and needs the steps.
  • Verb: achieve a specific outcome (not learn the loop).
  • Shape: focused goal in the lede; ordered steps without checkpoints; assumes context.
  • Where it lives: internal/how-to/<area>/....

Contrast with Phase 1: a tutorial promises learning by safe practice; a how-to assumes you're already in the middle of doing something real.

✓ Check your progress

Row 2 filled. The "verb" cell uses achieve or do, not learn. If you can't tell whether the page you opened is a tutorial or a how-to, that's a real signal — write the ambiguity into the row.

3 Reference — open a field manual

Visit API endpoint acceptance checklist (or anything under internal/reference/). Answer:

  • Reader: someone looking up a fact, mid-task.
  • Verb: find / look up, not follow.
  • Shape: lists, tables, schemas; austere; no narrative; scannable.
  • Where it lives: internal/reference/<area>/....

✓ Check your progress

Row 3 filled. Notice that reference reads like a dictionary, not a story — your "shape" cell should reflect that (tables, structured lists, terse one-liners).

4 Explanation — open a background piece

Visit anything under internal/explanation/ (e.g. Explanation hub). Answer:

  • Reader: someone off-task, building a mental model.
  • Verb: understand why, not do.
  • Shape: prose, history, trade-offs, alternatives considered; sometimes diagrams.
  • Where it lives: internal/explanation/<area>/....

Add a fifth column to your cheatsheet: where new content goes. For each row, write the rule of thumb in one sentence:

  • Tutorial: "If a new teammate has to learn a loop they've never done — write here."
  • How-to: "If someone with context will keep doing it on the regular — write here."
  • Reference: "If it's a lookup, a list, or a schema — write here."
  • Explanation: "If it answers why this and not that — write here."

✓ You're done

  1. Cheatsheet has four rows × five columns, all cells filled.
  2. You can name, in one sentence each, the difference between tutorial and how-to, and between reference and explanation.
  3. You know which folder under internal/ to open the next time you have a doc to write.

That is the IA loop — open one page of each mode → answer four questions → derive the placement rule. Every new portal page goes through this filter before it's written.
