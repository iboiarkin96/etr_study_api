Group A · 5 practices

Reliability foundations

The bedrock of SRE — defining what "working" means in numbers, keeping toil from consuming the team, and deliberately validating failure recovery before an incident forces the test.

AS-IS — → — TO-BE
1

SLO & error budget

3.5→8.5

What: Service Level Objectives are the reliability targets expressed as a percentage of good events over a rolling window (e.g. 99.9% of HTTP requests return 2xx in ≤ 300 ms over 28 days). The error budget is what remains: 0.1% of bad events the team is allowed to spend. When the budget is full, teams ship fast; when it's nearly exhausted, reliability work takes priority over features.

Why: Without SLOs, "reliability" is a feeling. With SLOs, every release decision becomes data-driven: is the remaining error budget large enough to risk this deployment? SLOs also remove the perennial conflict between product velocity and engineering caution — both sides agree in advance on what "reliable enough" means.

How we roll it out: Start with the two or three user-facing endpoints that generate the most support tickets. Agree on a 28-day SLO with the product team (not just engineering). Hook the SLO to a burn-rate alert so the team sees budget consumption in real time. Review the budget at each sprint retrospective before accepting new feature work.

2

Toil reduction

1.5→7.5

What: Toil is manual, repetitive, tactical work directly tied to running a production service that scales with load and provides no enduring value. Examples: manually restarting a service after each deploy, rotating credentials by hand, clicking through a dashboard to confirm a deploy health. Google's SRE standard: toil should consume no more than 50% of each SRE's working time; the rest goes to engineering work that permanently reduces future toil.

Why: Uncontrolled toil is the primary driver of SRE burnout and the reason on-call rotations collapse. As load grows, toil grows linearly with it — unless engineering work systematically eliminates it. The 50% rule is a forcing function: when toil crosses the ceiling, the team must automate or stop accepting new load.

How we roll it out: Track toil explicitly in the on-call log: each ticket classified as toil or non-toil. Monthly retro: top 3 toil sources by hours consumed. Assign each a reduction project with a 90-day deadline and a named owner. Measure toil percentage at the end of every quarter.

3

Reliability tiering

2.0→8.0

What: A formal classification of every service into reliability tiers (e.g. Tier 0 = business-critical, 99.99% SLO; Tier 1 = important, 99.9%; Tier 2 = best-effort, 99.5%), with explicit per-tier commitments for on-call response time, DR requirements, and allowed maintenance windows. Tiering prevents gold-plating less-critical services and lets teams prioritise investment proportionally.

Why: Without tiering, every service is implicitly treated as equally critical — which means everything is under-resourced for actual critical paths and over-engineered for internal tools. Tiering makes the trade-off explicit: a Tier 2 service can use best-effort infra; a Tier 0 service needs redundancy, geographic failover, and the team's pager attention.

How we roll it out: Audit current services and assign initial tiers collaboratively with product. Publish a tier registry in the service catalog. Enforce tier-appropriate SLOs in the monitoring stack (no Tier 0 alert → PagerDuty; Tier 2 alert → Slack only). Review tier assignments at each major architecture change.

4

Capacity planning

2.0→7.5

What: A quarterly process that projects resource consumption 6–12 months forward based on traffic growth models, upcoming feature launches, and peak event calendars, then translates those projections into provisioning decisions before the capacity headroom runs out. Includes load tests that validate peak-load projections against actual system behaviour.

Why: "We'll add more servers when we need them" is the most common path to an avoidable capacity incident. Cloud elasticity solves the provisioning lag, but not the budget surprise, the lead time for reserved capacity, or the architectural limits (connection pool sizes, DB connections, third-party rate limits) that don't scale automatically.

How we roll it out: Instrument the 5 key resource dimensions (CPU, memory, DB connections, storage, external API quota). Build a growth model from 6 months of historical data. Run the capacity review quarterly with infrastructure and product leads. Action items: reserved capacity purchases, architectural changes, hard limits in config.

5

Chaos engineering

1.0→7.5

What: Intentional injection of failures into a production or production-like environment to verify that the system's redundancy, failover paths, and alert coverage actually work. Experiments are run in controlled conditions (business hours, low traffic), with a predefined blast radius, and a clear "steady state" hypothesis: "If we kill one replica, P99 latency stays below 500 ms and no alerts fire for more than 2 minutes."

Why: The worst time to discover that a failover path doesn't work is during a real incident at 3 am. Chaos engineering surfaces the gap between the system you designed and the system you actually have — in controlled conditions where you can fix the problem rather than scrambling to contain it. Netflix, Google, Amazon all run chaos as a continuous process.

How we roll it out: Start with a GameDay (structured simulation, not automated tooling). Define one hypothesis per service per quarter. Run in staging first; promote to production after evidence of safety. Document findings in the blameless postmortem format. Automate experiment execution with Chaos Mesh or Gremlin once manual GameDays are mature.

Group B · 5 practices

Observability

You can't fix what you can't see. This group covers the signals, dashboards, and alert discipline that give the team genuine situational awareness of production at any moment.

AS-IS — → — TO-BE
6

Structured logging

4.0→8.5

What: All application logs emitted as machine-parseable JSON with a consistent schema: timestamp, level, service, trace_id, span_id, message, and domain-specific fields. No free-form printf strings. Every log line is a structured event that a query engine (Loki, OpenSearch, BigQuery) can filter, aggregate, and join with traces.

Why: Unstructured logs are read by humans; structured logs are queried by machines. When every log line carries the same trace ID as the distributed trace for that request, engineers can pivot from a metric anomaly to a log event to a distributed trace in one click. That pivot is the core of fast incident diagnosis — it turns hours of log grepping into a 2-minute query.

How we roll it out: Establish a canonical log schema in a shared library. Lint log schema compliance in CI. Route all service logs to the central log aggregator with retention policy. Build 3 golden-path dashboards using structured fields. Remove all console.log("debug info") style statements from the codebase within one sprint per service.

See also: logging.html · observability.html · OpenTelemetry Logs
7

Metrics & dashboards

4.5→8.5

What: A structured approach to metrics instrumentation and dashboard design. Every service exposes the four golden signals (latency, traffic, errors, saturation) via a standard metrics library (Prometheus, StatsD). Dashboards are versioned as code (Grafana JSON stored in git), reviewed before merge, and linked from the service runbook. A service without a dashboard is a service without observability.

Why: Ad hoc dashboards built during an incident are useless for the next incident. Dashboards-as-code mean every team member knows where the canonical view is, the view is reviewed for correctness before it matters, and it's reproducible across environments. The four golden signals give the minimum viable signal set: if all four look normal, the service is almost certainly healthy.

How we roll it out: Implement a shared Prometheus client library with the four golden signals pre-instrumented. Require a Grafana dashboard PR for every new service before it reaches production. Review existing dashboards for stale panels quarterly. Link each dashboard to the matching runbook via annotation.

8

Distributed tracing

1.5→8.0

What: Propagation of a trace_id across every service in a request path, with each service emitting spans that record latency, status, and context for its segment of work. The collected spans form a Gantt-style trace that shows exactly where time is spent across the distributed call graph for any individual request. Implemented via OpenTelemetry with Tempo, Jaeger, or a cloud-native backend.

Why: In a microservices system, a single user request may touch 10–20 services. Metrics tell you the aggregate is slow; a trace tells you which service's DB query is the bottleneck for this specific request, at this specific percentile. Without tracing, performance investigations require custom logging patches, redeployments, and guesswork. With tracing, the answer is a single query.

How we roll it out: Instrument the most user-facing service first with OpenTelemetry. Add automatic context propagation to the HTTP client / gRPC interceptor layer in the shared library. Set 5–10% sampling rate for normal traffic, 100% for errors. Build a trace-to-log correlation panel in Grafana. Add tracing to the DoD checklist for new services.

See also: observability.html · OpenTelemetry Tracing
9

Alerting design

2.0→8.5

What: A principled approach to alert design where every alert has a clear owner, an explicit severity, a documented response playbook, and a review cadence. Alerts fire on symptoms (SLO burn rate, error rate spike) not causes (CPU at 90%, memory at 85%). Alert noise is a first-class metric: alert fatigue from false positives kills on-call effectiveness faster than alert gaps.

Why: The most common on-call pathology is alert fatigue. When on-call engineers acknowledge 60 alerts per shift and most are noise, they stop treating alerts as signals. When the one real alert fires, it's ignored. Good alerting design inverts this: fewer alerts, each one meaningful, each one actionable. Google's target: every alert is actionable within 5 minutes.

How we roll it out: Audit existing alerts — classify each as "actionable", "informational", or "noise". Delete or demote informational and noise alerts. Require every new alert to have a runbook link. Build a "alert quality" dashboard: pages per engineer per week, MTTA (mean time to acknowledge). Target: fewer than 5 pages per engineer per week.

10

Synthetic monitoring

2.0→7.5

What: Scheduled scripts that simulate real user journeys (login → search → checkout) from multiple geographic locations and cloud regions, reporting success/failure and latency as metrics. Unlike passive monitoring (which only fires when users hit errors), synthetics detect availability and performance degradation before users encounter it — including degradation in regions with low real traffic.

Why: Internal metrics say the service is healthy; synthetic tests from a geographic region say it's unreachable. Real traffic is uneven — low-traffic regions may have an outage that affects 5% of users but never generates enough error volume to breach an alert threshold. Synthetics catch these dark spots and also validate that critical user journeys work end-to-end, not just that individual services respond.

How we roll it out: Identify the 3 critical user journeys. Implement synthetic tests with Playwright or k6. Run every 2 minutes from 3 regions. Route synthetic failures to PagerDuty at severity P2. Add synthetic SLO contribution to the overall service SLO calculation. Review synthetic test coverage quarterly.

Group C · 5 practices

Incident response

Procedures and culture for when things break. Strong incident response minimises customer impact, prevents repeat failures, and builds team confidence through practiced, blameless recovery.

AS-IS — → — TO-BE
11

On-call handbook

3.5→8.5

What: A written guide that answers every question an on-call engineer has before their shift starts: what to do when paged at 3 am, how to escalate, what tools to use, which services they're responsible for, how to declare an incident, and where to find runbooks. The handbook is a living document, updated after every incident and reviewed by new team members during onboarding.

Why: Every minute an on-call engineer spends searching for the escalation path or the right dashboard is a minute the customer-facing incident continues. The handbook eliminates the institutional knowledge problem: a new engineer on their first on-call shift can respond effectively because the process is documented, not because they've memorised it through experience.

How we roll it out: Write the handbook collaboratively — every on-call engineer contributes at least one section from their own experience. Link every runbook from the handbook. Shadow rotation: new engineers shadow 2 shifts before solo on-call. Handbook review as part of on-call handover. Update after every P0/P1 incident.

12

Incident severity framework

3.0→8.0

What: A classification system that maps incidents to severity levels (P0–P3) based on customer impact, business impact, and spread. Each severity level has a defined response SLA: who is notified, how quickly, through which channel, and what escalation path applies. Severity determines process, not engineering gut feel.

Why: Without a shared severity framework, every engineer classifies incidents differently. The result: P0-level customer impact sometimes gets a Slack message; internal tool slowdowns get 3 am pages. Shared severity criteria make the response proportional and predictable. They also enable post-incident analysis: "we had 12 P1 incidents this quarter; 8 had the same root cause."

How we roll it out: Define 4 severity levels with 3–4 specific qualifying criteria per level. Run calibration exercises: present past incidents, have teams classify independently, discuss divergence. Post the framework in the on-call handbook. Review classification accuracy in monthly incident retrospectives.

13

Incident command structure

2.0→7.5

What: A defined command structure for incidents — Incident Commander (IC) who coordinates and communicates, Tech Lead who diagnoses and implements, Communications Lead who writes customer status updates. During a P0/P1, roles are assigned within the first 5 minutes. The IC owns the incident timeline; the Tech Lead owns the technical investigation. Neither does the other's job.

Why: In the absence of a command structure, P0 incidents turn into chaotic Slack threads where 10 engineers are debugging simultaneously, nobody knows what's been tried, and the most senior engineer ends up doing diagnosis AND writing customer updates AND answering manager questions simultaneously. The command structure prevents this: everyone has one clear job during the incident.

How we roll it out: Define IC responsibilities in the on-call handbook (who to page, what to say, when to escalate). Run IC training: tabletop exercises with fake incidents where participants rotate the IC role. Require incident command in every P0/P1 postmortem retrospective. Track IC coverage — every P0 should have a named IC within 10 minutes.

14

Runbooks

4.5→8.5

What: Per-alert, per-symptom operational guides that tell an on-call engineer exactly what to do when a specific alert fires: what the alert means, how to confirm the diagnosis, what the remediation steps are (in order), what the expected outcome is, and when to escalate. A runbook is not a general how-to — it is a decision tree for a specific failure mode, optimised for 3 am use.

Why: The best engineers in the team can diagnose any alert from memory. But at 3 am on a Saturday, you want the most junior on-call engineer to resolve 90% of alerts without waking anyone up. Runbooks encode the senior engineer's diagnostic reasoning so anyone can execute it. Every minute saved in diagnosis is a minute less downtime for users.

How we roll it out: Write runbooks for the 10 most frequent alerts first (Pareto: they account for 80% of pages). Every runbook links to the alert definition and the relevant dashboards. Require runbooks as part of the DoD for any new alert. Validate runbooks in GameDays. Update after every incident where the runbook failed or was missing a step.

15

Blameless postmortem

3.5→9.0

What: A written retrospective published within 5 business days of every P0/P1 incident, covering: timeline (factual, not interpreted), impact (quantified), root causes (5 Whys to systemic cause — not individual error), what went well, and action items (named owner + deadline). The document is shared across engineering and never names individuals as causes of failure.

Why: Organisations that assign blame after incidents produce two outcomes: engineers hide problems and avoid transparency, and the same incident recurs. Blameless postmortems produce three outcomes: the systemic cause is fixed, the team learns faster than organisations that hide failure, and individuals feel safe to surface problems early. Google's SRE culture treats postmortem publication as the most important artifact of incident response.

How we roll it out: Adopt the standard 5-section postmortem template. Run a 60-minute postmortem review meeting within 5 days of P0/P1. Share the postmortem in the #engineering channel. Action items go to the engineering backlog with named owners. Monthly postmortem digest: what patterns emerged, which action items are overdue.

Delivery reliability

How releases become safer. Feature flags, canary deployments, and rigorously tested rollback paths mean the team can ship frequently and recover in minutes when something goes wrong.

AS-IS — → — TO-BE
16

Progressive delivery

2.0→8.5

What: A deployment strategy that exposes new code to increasingly larger user segments — canary (1%), ring 1 (10%), ring 2 (50%), GA (100%) — with automated SLO gate checks between each ring. If error rate, latency, or SLO burn rate degrades during promotion, the pipeline pauses or rolls back automatically without human intervention. Feature flags decouple deploy from release for high-risk changes.

Why: Big-bang deployments move all blast radius from "zero users" to "all users" in one step. A 1% canary converts that all-or-nothing risk into a 1% blast radius, detectable in minutes, reversible in seconds. Progressive delivery is the engineering practice that makes continuous deployment safe at scale — Spotify, Netflix, and Google deploy hundreds of times per day using this model.

How we roll it out: Start with canary deployments for the highest-traffic service. Define automated rollback triggers based on SLO burn rate. Build feature flag infrastructure (LaunchDarkly or a home-built variant). Graduate new features through beta → GA with explicit promotion criteria agreed before the beta starts. Require progressive delivery for every service with an SLO ≥ 99.9%.

17

CI/CD pipeline reliability

5.0→8.5

What: A set of practices that keep the CI/CD pipeline itself reliable and fast: flaky test elimination (any test that fails intermittently more than 0.5% of runs is quarantined), pipeline duration budgets (PR build ≤ 10 minutes, deploy ≤ 20 minutes), build infrastructure monitored with the same rigor as production, and a "pipeline on-call" rotation that treats build breaks as incidents.

Why: A slow or flaky CI/CD pipeline is a hidden reliability crisis. When engineers can't trust the pipeline, they start bypassing it — cherry-picking changes, skipping tests, or deploying from local machines. The consequence is that the pipeline becomes a formality rather than a safety gate. A reliable pipeline is a prerequisite for safe frequent deployment.

How we roll it out: Track pipeline duration and success rate as explicit SLOs. Quarantine flaky tests within 24 hours of detection. Set monthly flaky test elimination targets. Build dashboard for pipeline health (success rate, mean duration, queue depth). Make pipeline reliability a shared engineering responsibility — not just the platform team's problem.

18

Deployment health gates

2.5→8.0

What: Automated health checks that a deployment must pass before traffic is cut over: readiness probe (service responds), smoke tests (critical endpoints return 200), synthetic test pass, error rate within 2× of baseline, and latency P99 within 1.5× of baseline. All gates must pass within a defined window (e.g. 5 minutes) or the deployment is automatically rolled back. No manual approval needed.

Why: Manual post-deploy verification is slow, inconsistent, and often skipped when engineers are rushing or tired. Automated health gates make verification instantaneous and non-negotiable. They also reduce the "deploy and watch" ritual — engineers don't need to monitor dashboards for 20 minutes after every deploy because the pipeline does it for them.

How we roll it out: Define the minimum gate set for each service tier. Implement gates in the deploy pipeline. Run first with automatic pause (human approves rollback) before moving to automatic rollback. Track gate failure rate — a gate that fires every 5th deploy needs tuning. Add deployment health verification to the DoD.

19

Rollback playbook

2.5→8.0

What: A per-service runbook for reverting a deployment: the exact commands to run, who has access to run them, how long a rollback takes, what data migration or schema migration complications exist, and the expected outcome. The rollback playbook is written before the forward deployment, tested in staging, and linked from the deployment runbook. MTTR target: rollback within 5 minutes of the decision to roll back.

Why: "We can always roll back" is a statement of intent, not a capability. Without a tested rollback procedure, a rollback in production becomes an improvised, high-stress operation performed by a team that's already managing an active incident. A tested playbook converts rollback from a last resort into a routine recovery step that anyone on the team can execute.

How we roll it out: Write rollback playbooks for the 5 highest-traffic services first. Include DB schema rollback scenarios explicitly — schema changes are the most common rollback blocker. Run a rollback drill quarterly (planned rollback of a staging environment). Track "rollback capability" as a binary readiness check in the deployment DoD.

20

Game days & DR testing

1.0→7.5

What: Scheduled, facilitated exercises where the team deliberately induces failures (zone outage, DB failover, dependency degradation) and validates that recovery procedures work as documented. A Game Day is a controlled experiment, not a fire drill: it has a hypothesis ("failover completes within 2 minutes"), a blast radius limit, a rollback plan, and a postmortem-style retrospective.

Why: DR documentation that has never been tested is fiction. Every year of untested DR plans adds 12 months of configuration drift, tooling changes, and undocumented dependencies that make the actual DR scenario longer and riskier than the plan assumes. Regular Game Days surface the drift before a real disaster does. Netflix runs Game Days weekly; Amazon's Wheel of Misfortune is a quarterly mandatory exercise.

How we roll it out: Run one Game Day per quarter. Rotate the scenario type: zone failover Q1, DB failover Q2, dependency degradation Q3, data restore Q4. Publish a Game Day report (same format as a postmortem). Track hypothesis accuracy — how close was the actual recovery time to the plan? Feed findings back into runbooks.

Security & supply chain

The practices that keep the system trustworthy under adversarial conditions — from vulnerability management and secret hygiene to supply chain integrity and compliance automation.

AS-IS — → — TO-BE
21

Vulnerability management

2.0→8.0

What: A continuous process that scans every deployed container image, package dependency, and OS layer for known CVEs, classifies findings by severity (critical/high/medium/low), assigns remediation SLAs per severity (critical: 24h, high: 7d, medium: 30d, low: 90d), and tracks SLA compliance. Scanning is integrated into CI (block builds with critical CVEs) and runs on deployed images continuously.

Why: The average time between CVE publication and exploitation in the wild is now measured in days, not months. Without a systematic vulnerability management process, critical CVEs can sit unpatched in production for months because no one is tracking them. The SLA framework creates urgency proportional to severity and generates a measurable compliance metric that engineers and leadership can track.

How we roll it out: Integrate Trivy or Grype into the CI pipeline — fail builds with critical CVEs. Configure Dependabot or Renovate for automatic dependency PRs. Build a vulnerability dashboard (open critical CVEs, SLA breaches). Monthly security review: SLA compliance rate per service. Target: zero critical CVEs older than 24h in production.

See also: NIST NVD · Trivy
22

Secret management

3.5→8.5

What: A centralised, audited secret store (HashiCorp Vault, AWS Secrets Manager, GCP Secret Manager) that is the single source of truth for all credentials, API keys, certificates, and tokens. No secrets in environment variables, code, or config files. Secrets are rotated on a defined schedule, access is scoped to the minimum required, and every access is logged. Secret scanning in CI blocks commits that contain credentials.

Why: Credentials in git repositories, in Slack messages, or in unencrypted environment variables are the most common initial access vector in breaches. Secret management eliminates this entire attack surface and provides audit logs when credentials are accessed unexpectedly. Secret scanning in CI catches the problem before it reaches the repository history — where it's notoriously difficult to purge.

How we roll it out: Audit current secret handling — identify every place credentials are stored outside a vault. Migrate the highest-risk secrets (DB passwords, payment provider keys) to Vault first. Implement secret scanning in CI with gitleaks or truffleHog. Rotate all migrated secrets immediately after migration. Track secret hygiene as a quarterly security metric.

Read full practice →
See also: HashiCorp Vault · Gitleaks
23

Supply chain security

1.5→7.5

What: A set of practices that verify the integrity and provenance of every dependency, container image, and build artifact: SBOM generation (Software Bill of Materials) for every release, image signing (Cosign / Sigstore), dependency pinning with hash verification, and artifact policy enforcement (OPA / Kyverno) that blocks unsigned or unverified images from reaching production. Addresses SLSA (Supply chain Levels for Software Artifacts) framework requirements.

Why: The SolarWinds, Log4Shell, and XZ Utils incidents demonstrated that supply chain attacks can compromise thousands of organisations through a single dependency. An attacker who compromises a build pipeline or injects malicious code into a dependency can bypass all perimeter security. SBOMs and signed artifacts create a verifiable chain of custody from source code to production deployment.

How we roll it out: Generate SBOMs for every container image in CI (Syft). Sign all images at build time (Cosign). Require signature verification in the deployment pipeline. Pin all dependencies with hash verification. Achieve SLSA Level 2 as a 6-month target. Quarterly SBOM audit: identify new dependencies with no established provenance.

See also: SLSA Framework · Sigstore / Cosign
24

Compliance as code

1.5→7.0

What: Encoding compliance requirements (SOC 2, ISO 27001, GDPR controls) as machine-verifiable policies (OPA, Conftest, Terraform Sentinel) that run in CI, produce audit evidence automatically, and block non-compliant changes before they reach production. Evidence collection is automated — compliance audit preparation goes from weeks to hours because the evidence is generated continuously rather than assembled manually before each audit.

Why: Manual compliance is expensive, error-prone, and reactive — it finds violations after they've been in production for months. Automated compliance is cheaper, consistent, and preventive — it blocks violations before they ship. As regulatory requirements grow (GDPR, SOC 2, DORA) the manual approach doesn't scale; the automated approach scales with the codebase.

How we roll it out: Map the top 10 compliance requirements to machine-verifiable policies. Implement policies in Conftest against Terraform plans. Run policy checks in CI. Build an automated evidence collection dashboard. Reduce audit prep time from the current baseline by 60% within 12 months. Assign a named policy owner for each compliance domain.

See also: Open Policy Agent · Conftest
25

Security posture management

2.0→7.5

What: A continuous visibility program that aggregates signals from vulnerability scanners, SIEM, cloud configuration auditors (CSPM), IAM reviews, and network posture assessments into a unified security dashboard. A weekly security posture score tracks the aggregate health of the security programme. Threats are triaged in a security backlog with the same engineering rigour as feature work — prioritised by risk, assigned to owners, tracked to closure.

Why: Individual security tools produce individual findings — but without aggregation, no one knows whether the overall security posture is improving or degrading. The posture score gives the team and leadership a single number that encodes the current security health. Treating security findings as engineering backlog items removes the false separation between "security team" and "engineering team" that causes security work to be perpetually deprioritised.

How we roll it out: Implement a CSPM tool (Wiz, Lacework, or AWS Security Hub). Connect findings to the engineering backlog with automatic severity-based labels. Build a weekly posture score dashboard. Monthly security review with engineering leads: posture score trend, top risk drivers, overdue findings. Annual third-party penetration test with findings tracked in the same backlog.
