Synthetic Monitoring — detect outages before users do

Scheduled scripts that simulate real user journeys from multiple geographic locations, reporting availability and latency as metrics — detecting outages before users generate enough error volume to breach alert thresholds.

TL;DR

Synthetic monitoring runs scheduled Playwright or k6 scripts that simulate the 3 most critical user journeys (login → search → checkout) from multiple geographic regions, every 2 minutes. Unlike passive monitoring (which fires only when users hit errors), synthetics detect availability problems before users encounter them — including degradation in low-traffic regions that never generate enough error volume to breach a threshold alert. Synthetic test failures route to PagerDuty at P2 severity and contribute to the service SLO calculation.

What it is

Synthetic monitoring runs scripted test scenarios against the production system on a schedule, from multiple geographic locations. Unlike passive monitoring (which observes real user traffic), synthetic monitoring generates artificial traffic to test the system in the absence of real users.

A synthetic test typically:

  • Simulates a complete user journey — login, search, add to cart, checkout — not just a single API endpoint.
  • Runs every 1–2 minutes from 3+ geographic locations.
  • Reports success/failure and per-step latency as metrics.
  • Fires an alert when the journey fails or exceeds a latency threshold.
  • Contributes to the service's SLO calculation as a synthetic SLI.

Implementation options: Playwright (browser-based, tests the full frontend stack), k6 (HTTP-based, tests the API layer), or Grafana Synthetic Monitoring (hosted probe network, no infrastructure to manage).

Why we need it

Passive monitoring has a fundamental limitation: it requires real user traffic to detect failures. In low-traffic regions, a complete service outage may never generate enough error volume to breach an alert threshold. The first time the team learns about the outage is when a customer in that region reports it on social media — which is too late.

Synthetics also catch a class of failures that passive monitoring is structurally blind to: CDN misconfigurations, DNS failures, and TLS certificate expiration. These failures prevent users from reaching the service at all — so they generate zero events in the application monitoring stack, but they're immediately visible to a synthetic monitor that can't complete its test run.

How others do it

Cloudflare

Cloudflare runs synthetic tests from their global PoP network — 300+ locations. This lets them detect geographic degradation with a granularity that a 3-region setup cannot. Their synthetic tests include DNS resolution time, TLS handshake time, and time-to-first-byte as separate metrics — enabling precise diagnosis of where in the network stack degradation is occurring.

What we take: Break the synthetic test into phases (DNS, TCP connect, TLS, first byte, full page load) and report each phase as a separate metric. When the test fails, the phase breakdown tells you where to look without additional investigation.

Source: Cloudflare DEX Monitoring

New Relic

New Relic's synthetic monitoring includes "scripted browser" tests (full Selenium/Playwright journeys that render JavaScript and interact with the DOM) and "API test" probes (lightweight HTTP checks). Their recommendation: one scripted browser test per critical user journey; API probes for every health check endpoint. The browser tests are the SLO input; the API probes are the fast-feedback alerting layer.

What we take: Two levels of synthetics: lightweight API probes that run every minute with fast alert times (2-minute failure before alerting), and heavier browser journey tests that run every 5 minutes with a higher alert threshold (10 minutes of failure). Don't page on a single failure — wait for 2–3 consecutive failures to avoid flaky test noise.

Source: New Relic Synthetic Monitoring

Anatomy

Journey identification

Identify the 3 most business-critical user journeys. These are the journeys that, if broken, produce the most customer impact and support tickets. Not all endpoints — complete journeys from the user's perspective.

Multi-region probe network

Run synthetics from at least 3 geographic locations that represent the actual user distribution. If 30% of users are in APAC, one probe must be in APAC. A failure in APAC that's invisible in EU probes is a real outage for 30% of users.

Alert threshold

Alert on 2 consecutive failures from the same region. A single failure from one probe is often a network blip at the probe location. Two consecutive failures from the same region is a genuine signal. Never alert on single failures.

SLO contribution

Synthetic test results are SLI events — every successful test run is a "good event" in the SLO calculation. A 5-minute outage that produces 5 failing test runs from 3 regions is 15 bad events against the SLO budget. This makes synthetic failures budget-visible, not just alert-visible.

Common pitfalls

PitfallWhy it happensWhat to do instead
Synthetic tests that use test accountsThe test must have valid credentials to log in.Use dedicated synthetic test accounts with synthetic-specific email addresses and tokens. Never share test accounts with developer logins — a password change breaks all synthetics simultaneously. Rotate synthetic credentials on a separate schedule.
Alerting on single failures"One failure means it's down."Alert on 2 consecutive failures from the same region. Probe-level network issues cause single failures constantly — alerting on them creates noise that degrades on-call effectiveness.
Synthetic tests not maintainedThe login flow changes; the synthetic test still uses the old UI.Assign a named owner for each synthetic test. Require synthetic test update as part of the DoD for any user-facing feature that changes the critical journey flow. Synthetics that always fail because the UI changed are worse than no synthetics.
