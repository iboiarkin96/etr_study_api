SLOs & error budget

Why we need SLOs

Without numbers, "the service is fine" is just an opinion. An SLO (service-level objective) gives us a clear target. As long as we hit the target, we can focus on building features. When we miss it, we slow down and spend time on stability. The SLO is also what tells an alert "this is worth waking somebody up for."

The contract for this service is defined in ADR 0011 — SLO, SLA, and error budget. This page explains what each term means and how we use them day to day.

The three terms

SLI — Service-Level Indicator
A measurement we can read from the system. Examples: "share of HTTP requests with status 2xx or 3xx", "p95 response time of GET /conspectuses/due", "share of background jobs that finished without error".
SLO — Service-Level Objective
The target value of the SLI over a window. Example: "99.5% of API requests succeed over a rolling 30 days." This is an internal promise — we set it ourselves and we measure ourselves.
SLA — Service-Level Agreement
A contract with a paying customer that includes money penalties. This solo project has no SLA today; the section exists so we can plug one in later without renaming concepts.

Error budget — what it is and how to spend it

An error budget is "100% minus the SLO". If the SLO is 99.5%, the budget is 0.5%. Over 30 days that is about 3.6 hours where we are allowed to be unhealthy without breaking our promise.

The budget is a tool, not a punishment. The simple rules:

  • Budget remaining: ship features. Risky migrations, new endpoints, dependency upgrades — go ahead.
  • Budget low (less than 25% left): slow down. Prefer reversible changes, add extra tests, postpone optional refactors.
  • Budget exhausted: feature work pauses. Only changes that improve reliability or fix the SLO miss are merged. Follow runbook 0009 — error budget exhaustion.

Burn-rate alerting

We do not wait for the budget to run out before we react. Burn-rate alerting watches how fast the budget is being spent. If we are burning the monthly budget in two hours, that is an emergency. If we are burning it in three days, that is a "look soon" signal.

How we review SLOs

  1. Once a month, look at the previous 30 days for each SLI.
  2. Record the result in the changelog ("met / missed / N/A").
  3. If we missed an SLO, write a short note: was it one big incident, slow drift, or a measurement bug?
  4. If the same SLO is missed twice in a row, either fix the system or change the target through an ADR — never silently lower a number.

Common mistakes to avoid

  • "100% uptime" SLO. Impossible to hit and impossible to spend a budget against. Always pick a target less than 100%.
  • Vanity metrics. Picking an SLI that always looks green (e.g., "the homepage returns HTML"). Pick something that reflects user pain.
  • Silent SLO changes. If a number stops being realistic, change it through an ADR. Never quietly relax a target after a bad month.
  • No window. "99.5% availability" with no window can mean anything. Always state the window (7 / 30 days).
