How pages are arranged in this portal

The folders on disk and the groups in the sidebar are two different views of the same files. On disk, every page sits under the service it belongs to — predictable, refactor-safe. In the sidebar, the same pages are grouped by what the reader is trying to do (learn, do, look up, understand). One source of truth, two views.

The rule

Disk is flat. Sidebar is grouped.

Files sit predictably under services/<name>/. The sidebar takes those same files and renders them into four groups: tutorials, how-to, reference, explanation. The grouping is declared in each service's catalog-info.yaml — one source of truth, two views over it.

This is the pattern used by Django docs, Microsoft Learn, Cloudflare Workers, and Backstage TechDocs. It's the industry default because it's refactor-safe — links survive, bookmarks survive, and changing a page's group is a one-line edit, not a folder move.

What it looks like for one service

The API service has six files in its folder. The sidebar takes them and shows the reader four groups, in order of how they're typically used.

On disk In the sidebar
services/api/index.html Overview (above the groups)
services/api/architecture.html Reference → Architecture
services/api/api-reference.html Reference → API reference
services/api/dependencies.html Reference → Dependencies
services/api/runbooks.html How-to → Runbooks
services/api/on-call.html How-to → On-call
services/api/catalog-info.yaml (not shown — declares the grouping above)

Tutorials and Explanation are empty for this service today. Empty groups collapse in the sidebar so they don't take up space.

Where the grouping is declared

The grouping lives in one block of each service's catalog-info.yaml:

nav:
 - group: tutorials
 label: Tutorials
 pages: [] # empty → collapsed in sidebar
 - group: how-to
 label: How-to
 pages: [runbooks, on-call]
 - group: reference
 label: Reference
 pages:
 - architecture
 - api-reference
 - dependencies
 - group: explanation
 label: Explanation
 pages: []

Moving a page from How-to to Reference is one line of YAML. No git mv, no broken links, no redirect. The rendering script walks every service's catalog-info.yaml and emits the sidebar manifest the browser reads.

What this gets us, and what it costs

What we gain

  • Links don't break. Moving a page between groups doesn't move the file, so bookmarks and search-index URLs survive every reorganisation.
  • Two views, no duplication. «By service» (the folders) and «by reader's intent» (the sidebar) both render from the same files. No need to pick one ontology and lose the other.
  • The pattern is familiar. Anyone who has worked with Django, Microsoft Learn, Cloudflare, or Backstage will recognise it.

What it costs

  • The folder alone doesn't tell you the group. Running ls services/api/ shows a flat list — you have to open catalog-info.yaml to see how those files group in the sidebar.
  • We maintain the renderer. The nav: block is our extension; the script that turns it into a sidebar manifest is something we own. When it breaks, no one else has fixed it before us.
  • One page belongs to exactly one group. A page can't appear in both Reference and Explanation. When content genuinely straddles, we split it into two linked pages — cleaner long-term, but more files.
