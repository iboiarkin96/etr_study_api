Internal · Operating model · How-to

Six questions that place any page in exactly one folder

Walk the six questions top-down before you write any HTML. Each answer sets one or two of the six data-* attributes from the page-type contract and narrows the target folder. The ten worked examples at the bottom cover the page shapes you'll hit most often.

Use this when

  • You are about to create a new page and need a folder.
  • You are moving an existing page and need a new folder.
  • A reviewer asked «is this in the right place?» and you want one source of truth.

If you are editing an existing page in place, you do not need this tree — keep the folder and the metadata it already has.

The six questions — overview

Click any question to jump to its detailed answer below. Walk them in order; later questions assume earlier answers.

  1. Q1 · sets data-service

    Is this page about a specific service?

    If yes, it lives under services/<svc>/ and declares data-service="<svc>". The service catalog is the spine of the portal per ADR-0032 D1.

    Open the answer→
  2. Q2 · sets folder

    Otherwise: is it cross-cutting craft, governance, or onboarding?

    Cross-cutting goes to handbook/<discipline>/; new ADR/RFC/audits to governance/; role-onboarding paths to onboarding/. The remaining cases are tutorials, team pages, or this operating-model section.

    Open the answer→
  3. Q3 · sets data-page-type

    Which Diátaxis quadrant does the reader want?

    Tutorial (learning), how-to (recipe), reference (lookup), explanation (background), or landing-section (hub). The quadrant is metadata, not a folder — per ADR-0032 D3.

    Open the answer→
  4. Q4 · sets data-role

    Which role primarily reads this page?

    One of swe, sa, qa, sre, architect, manager — space-separated if it covers several. Empty value means cross-role. Role hubs aggregate via this tag (ADR-0032 D4).

    Open the answer→
  5. Q5 · sets data-topic

    What topic(s) does this page touch?

    Space-separated free tags: security, observability, authoring, runbooks, data, performance, operating-model, etc. Catalog views catalog/by-topic/ aggregate on this.

    Open the answer→
  6. Q6 · sets data-lifecycle + data-updated

    Is the page draft, review, published, or deprecated?

    Lifecycle is one of those four; aligned with the ADR-0018 vocabulary. data-updated is today's ISO date — bump it on every meaningful edit.

    Open the answer→

The six questions — in detail

Each answer below sets one or two data-* attributes and (sometimes) the folder. Outcomes are the values you write on <body>. Tip: the page-type contract is the reference list of allowed values; this page is the procedural walk-through.

Q1 — Is this page about a specific service?

Why this is the first question. Per ADR-0032 D1 the service catalog is the spine of the portal. If a page documents a deployable artefact, its home is that artefact's folder — runbooks, postmortems, API references, and architecture notes all sit under the service they describe.

If yes: set data-service to the canonical service slug and place the file under services/<svc>/.

data-serviceFolder rootExamples of pages that live here
apiservices/api/endpoint specs, API runbooks, OpenAPI conspectus, contract-test failures
portalservices/portal/portal how-to's, sync-staging runbook, portal postmortems
datastoreservices/datastore/schema notes, migration runbooks, write-lock postmortems
monitoringservices/monitoring/scrape runbooks, dashboard reference, on-call docs
ui-kitservices/portal/ui-kit/component showcases, token reference

If no: set data-service="none" and go to Q2.

Q2 — Otherwise: is it cross-cutting craft, governance, or onboarding?

Why this question. Pages that are not about one service still need a home. The portal has five non-service trees; each has a load-bearing purpose. Per ADR-0032 D2 cross-cutting craft lives under handbook/; per ADR-0026 governance artefacts live under governance/.

FolderWhat it holdsdata-service
handbook/<discipline>/cross-service practice — SA, SWE, QA, SRE, security, managersnone
governance/adr/ · rfc/ · audits/ · backlog/decisions, design proposals, dated assessments, planned worknone (or the service the decision affects, e.g. portal)
onboarding/role onboarding journeys, cold-start paths, the domain glossarynone
tutorials/<role>/«your first <task>» tutorialsmatches the artefact being built (e.g. api for «your first endpoint»)
team/people/ · team/roles/person pages, role radars + 25 practices per rolenone
operating-model/this section — hub home, routing tree, lifecycle, gates map, page-type contract, feed, glossaryportal

Q3 — Which Diátaxis quadrant does the reader want?

Why this question. Per ADR-0032 D3 the quadrant is metadata, not a folder. The same service folder mixes tutorials, how-to's, references, and explanations side by side; the lens that lets you slice them is data-page-type. Pick by what the reader is doing, not by what the page is about.

data-page-typeReader's stateTypical shapes
tutoriallearning — beginner, needs a guided path«Your first ADR», «Your first endpoint»
how-todoing — knows the task, needs the recipe«Add a page», «Release to staging», «Run a load test»
referencelooking up — needs an exhaustive spec or tableAPI operations, ADRs, runbooks, postmortems, templates, gates map
explanationunderstanding — wants the «why» and the trade-offsDiátaxis, topology, freshness, principles
landing-sectionarriving — wants the map of a sectionSection index.html (this page is a how-to; operating-model/index.html is a landing-section)

If two quadrants both feel right, pick the one closest to the reader's state when they land on the page. A runbook is a reference even though it reads like a how-to — operators look up steps, they do not learn from them.

Q4 — Which role primarily reads this page?

Why this question. Role hubs (team/roles/<role>/) aggregate pages via this attribute — per ADR-0032 D4 a page is never duplicated in a role hub, only tagged.

Allowed values: space-separated subset of swe, sa, qa, sre, architect, manager. Per ADR-0032 D6 the role token is swe, not dev / developer / engineer. Empty value (data-role="") marks the page cross-role; the catalog renders it as cross-role.

Rule of thumb. If you can name only one role that opens this page in a normal week, use that role. If three or more would, leave it empty.

Q5 — What topic(s) does this page touch?

Why this question. data-topic drives catalog/by-topic/ — the topical lens. Unlike data-page-type and data-service, the topic vocabulary is open: write the tags that match the page.

Common values. security, observability, authoring, runbooks, data, performance, release, onboarding, operating-model, career. Multiple tags are space-separated. The catalog only renders the four currently curated lenses (runbooks, postmortems, tests, on-call); other tags still flow into search.

Leave data-topic="" if the page is truly a standalone artefact (e.g. a person profile).

Q6 — Is the page draft, review, published, or deprecated?

Why this question. data-lifecycle is the freshness signal. Aligned with ADR-0018 lifecycle vocabulary; consumed by the freshness audit and (later) by the staleness SLA.

data-lifecycleMeaningVisibility
draftstructure is in place, content is unfinishedvisible in sidebar; not advertised in role hubs
reviewcomplete, awaiting a peer reviewsame as draft; reviewers find via PR
publishedthe supported state; everyone reads thisfull visibility in catalog views and hubs
deprecatedkept for archaeology; pointer to its successorgreyed out in catalog; superseded badge

And data-updated. Set to today's ISO date when you ship a meaningful edit (typo fixes do not count). The recent-changes view is sorted on this value; the freshness audit measures «days since data-updated».

After the tree

Once you have the folder and the six attributes, copy the matching template from handbook/sa/templates/ — that's Rule 4 of the operating-model hub. Then run make -C services/portal docs-fix && make -C services/portal docs-check (Rule 3). The drift gate will tell you if you missed an autogen step.
