Internal · Operating model · How-to

Where does my new page go?

Six questions that tell you exactly where to put a new page and how to label it. Walk them in order — each answer narrows down the right folder and fills in one of the labels every page must declare. Ten worked examples at the end cover the cases that come up most often.

When to use this

  • You're about to create a new page and need to know where to put it.
  • You're moving an existing page and need to know where it should go now.
  • Someone asked «is this in the right place?» and you want a single answer to point at.

If you're just editing a page in place, you don't need this — keep the folder and labels it already has.

The six questions

Click any question to jump to its detailed answer below. Walk them in order — later questions assume earlier answers.

  1. Q1 · Which service

    Is this page about a specific service?

    If yes, the page lives in that service's folder. The service catalog is the backbone of the portal, so anything that documents a service goes there.

  2. Q2 · Which folder otherwise

    Not about a service — where does it go then?

    Practice everyone shares goes in the handbook. New decisions, proposals, and audits go in governance. Cold-start guides go in onboarding. Tutorials, team pages, and this section have their own homes.

    Open the answer→
  3. Q3 · What kind of page

    What is the reader trying to do?

    Tutorial (learning something new), how-to (doing a known task), reference (looking something up), explanation (understanding why), or a section hub (arriving at an index). This is a label on the page, not where the page lives.

    Open the answer→
  4. Q4 · Who reads it

    Which role primarily reads this page?

    SWE, SA, QA, SRE, architect, or manager — list more than one if several roles read it. Leave blank for pages everyone reads. This is how role pages collect the right material without copying it.

  5. Q5 · Topic tags

    What topics does this page touch?

    Free-form tags like security, observability, runbooks, on-call, performance. Several tags can apply. These power the «by topic» views and feed the search index.

    Is the page draft, in review, live, or retired?

    One of those four. Together with today's date, this tells the rest of the portal whether the page is the current answer, a work in progress, or kept around for history.

The six questions in detail

Each answer below fills in one of the labels every page must declare, and sometimes the folder. The page-type contract lists all allowed values in one place; this page walks you through choosing them.

Q1 — Is this page about a specific service?

Why this is the first question. The service catalog is the backbone of the portal. If a page documents anything that belongs to a service — runbooks, postmortems, API references, architecture notes — its home is that service's folder.

If yes: set the service label and place the file in that service's folder under services/<service>/.

data-serviceFolder rootExamples of pages that live here
apiservices/api/endpoint specs, API runbooks, OpenAPI conspectus, contract-test failures
portalservices/portal/portal how-to's, sync-staging runbook, portal postmortems
datastoreservices/datastore/schema notes, migration runbooks, write-lock postmortems
monitoringservices/monitoring/scrape runbooks, dashboard reference, on-call docs
ui-kitservices/portal/ui-kit/component showcases, token reference

If no: mark the service label as «none» and go to Q2.

Q2 — Where do non-service pages go?

Why this question. Pages that aren't about a single service still need a home. The portal has five non-service folders, each with a clear purpose: practice and craft go in the handbook, decisions and proposals go in governance, cold-start guides go in onboarding, and so on.

Q3 — What is the reader trying to do?

Why this question. The kind of page (tutorial, how-to, reference, explanation) is a label on the page, not a folder. The same service folder contains all four kinds side by side, and the portal builds separate views from the label. Pick the label that matches what the reader is doing when they land on the page, not what the page is about.

data-page-typeReader's stateTypical shapes
tutoriallearning — beginner, needs a guided path«Your first ADR», «Your first endpoint»
how-todoing — knows the task, needs the recipe«Add a page», «Release to staging», «Run a load test»
referencelooking up — needs an exhaustive spec or tableAPI operations, ADRs, runbooks, postmortems, templates, gates map
explanationunderstanding — wants the «why» and the trade-offsDiátaxis, topology, freshness, principles
landing-sectionarriving — wants the map of a sectionSection index.html (this page is a how-to; operating-model/index.html is a landing-section)

If two labels both feel right, pick the one that matches what the reader is doing when they land. A runbook reads like a how-to but is consulted under pressure, not learned — so it's a reference.

Q4 — Which role primarily reads this page?

Why this question. Each role has its own hub that gathers the pages relevant to it. A page is never copied into a role hub — it's tagged with the role, and the hub picks it up.

Allowed roles: SWE, SA, QA, SRE, architect, manager. List several with spaces between if the page covers more than one. Note the role label is swe, not dev or developer or engineer — the value is fixed. Leave it blank if the page is for everyone; the catalog will mark it «cross-role».

Rule of thumb. If only one role opens this page in a normal week, name that role. If three or more would, leave the value blank.

Q5 — What topics does this page touch?

Why this question. Topic tags are how the portal builds its «by topic» views. Unlike the page kind and the service, the topic vocabulary is open — write the tags that fit the page.

Common tags. security, observability, authoring, runbooks, data, performance, release, onboarding, operating-model, career. List several with spaces between. The portal renders four curated topic views today (runbooks, postmortems, tests, on-call); any other tag still feeds the search index.

Leave the topic empty if the page is a standalone artefact (a person profile, for example).

Q6 — Is the page draft, in review, live, or retired?

Why this question. The status tells the rest of the portal whether the page is the current answer, work in progress, or kept around for history. Together with today's date, it's how the «recent changes» view and the staleness audit work.

data-lifecycleMeaningVisibility
draftstructure is in place, content is unfinishedvisible in sidebar; not advertised in role hubs
reviewcomplete, awaiting a peer reviewsame as draft; reviewers find via PR
publishedthe supported state; everyone reads thisfull visibility in catalog views and hubs
deprecatedkept for archaeology; pointer to its successorgreyed out in catalog; superseded badge

And the date. Set the update date to today (in ISO format, like 2026-06-13) whenever you ship a meaningful change — a new section, restructured content, a status change. Don't bump it for typo fixes, or you'll mask real staleness in the audit.

After you have the answers

Once you know the folder and the six labels, copy a matching template as your starting point — it gives you the structure the page needs to have. Then run make docs-fix and make docs-check before you commit: the first regenerates everything that's auto-generated (search index, indexes), the second confirms nothing fell out of sync.
