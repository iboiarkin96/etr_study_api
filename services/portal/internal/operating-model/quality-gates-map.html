Internal · Operating model · Reference

Every automated check — what it does and how to fix it

A reference of every automated check that runs on this portal. Each row tells you the name of the check, when it runs, what it looks at, how to fix it when it fails, and whether the fix needs to touch any other file. Start with the «coupling» column: that's the one most likely to save you from a second failed commit.

Why one reference and not the raw configs

The checks live in three different places — the commit-time config, the build script, and the cloud workflow files — and the rules that connect them live in nobody's head. The lifecycle page tells you when each check runs; this page tells you what it looks at, how to fix a fail, and what else must change at the same time.

Read the coupling column first. A dash (—) means the check is self-contained: fix it and move on. Anything else means there's an edit that has to travel with the fix, and the cloud build will catch you if you skip it.

Three places checks run

  1. A · On your computer, on commit

    Pre-commit checks (about 20)

    Run automatically when you commit, on the files you staged. Fast and cheap. Includes generic correctness (YAML / JSON / merge-conflict markers), Python style and type-checks, portal-specific validators, and the two checks that enforce paired edits.

    Table A→
  2. B · On demand, with one command

    Make targets (14 total)

    Run when you ask, with a make command. Two tables: Table B — portal-specific targets (the drift check, granular checks, verify = CI gate, verify-local = full pre-push gate that adds visual-test). Table B2 — five top-level entries (verify-api, verify-portal, verify-monitoring, verify-frontend, and the composer verify): one shortcut per service, one composer.

    Table B + B2→
  3. C · In the cloud, on every push

    Cloud build workflows (7)

    Run automatically when you push, when a PR opens, on a schedule, or when triggered. Includes the main build matrix, the accessibility score check, dead-code scan, draft changelog, weekly feedback digest, and the two deploys (staging and production).

    Table C→

B · Make targets — portal-specific

Portal-side make targets. The granular checks are at the top; the composites at the bottom. Two composites carry distinct contracts: verify is the CI gate (platform-independent), verify-local adds the visual pixel-diff that only runs locally. For the root-level entries that cover every service (verify-api, verify-portal, verify-monitoring, verify-frontend, and the composer verify), see Table B2 below.

verify (portal-local) Called by root verify-portal; same target CI runs Composite: docs-check + docs-a11y-check. Platform-independent — no Chromium, no PNG diff. Resolve whichever inner target failed and re-run —
visual-test local only On demand, after intentional UI Kit / template edits Pixel-diff the registered template pairs in tests/visual/pages.json against PNG baselines under tests/visual/baselines/ at three viewports (desktop-light/dark, mobile-light). 0.1 % threshold. If the visual changed intentionally, run make visual-test-update to refresh the baselines on the same platform where they will be checked; commit the new PNGs alongside the UI change. Twin-template coupling — see § Coupling rules
verify-local full local gate Before push, when you've touched UI Kit components / tokens / template structure verify + visual-test chained in order. Catches drift that the CI-side verify can't (visual baselines are platform-dependent, see below). Resolve verify first; if visual fails, inspect tests/visual/_artifacts/diffs/ and decide: real drift → fix the page; intentional change → visual-test-update. —

Why visual-test is local-only. PNG baselines bake the rendering platform — font hinting and sub-pixel anti-aliasing differ between macOS Chromium (typical dev machine) and Linux Chromium (the GitHub runner). A baseline that's pixel-perfect locally fails on CI without any UI change. Until we either bake baselines on the runner (workflow_dispatch + commit-back) or split baselines/<os>/, the gate runs only where the baselines were created. The structural canon tests under tests/visual/test_*_canon.py still run in CI — they read HTML through html5lib and need no browser.

D · Edits that must travel together

Some changes affect more than one place at once. When you edit one file, you also have to edit another — and there's no way a person can reliably remember every pairing. Each rule below names the trigger, the edit that has to ride with it, the reason, and the check (or memory rule) that holds it. One rule became an automated check on 2026-06-13; the rest are written down here so they're not invisible.

  1. Page template ↔ its design-example twin.
    • When you edit: a page template under handbook/sa/templates/.
    • You must also edit: the matching design example under portal/ui-kit/pages/templates/ (pairs are listed in tests/visual/pages.json). If the visual changed, refresh the baseline with make visual-test-update.
    • Why: the two pages are pixel-diffed against the same baseline. A one-sided change drifts silently until the visual test runs.
    • Enforcement: check_template_twins.py at commit time. Hard gate.
  2. Sidebar tree ↔ pages on disk.
    • When you edit: the sidebar tree, or you add or remove any page under the internal portal.
    • You must also edit: the matching page on disk for every entry in the sidebar; or the sidebar entry for every page.
    • Why: every internal page must be reachable from the sidebar. Orphan pages waste effort; broken nav links waste reader time.
    • Enforcement: a sidebar-coverage check in the cloud build where present; memory rule otherwise. Lands as a hard gate with the staging pipeline.
  3. Service catalog file ↔ rendered service page.
    • When you edit: a service's catalog descriptor.
    • You must also edit: the rendered service-overview page under the portal.
    • Why: the portal serves the rendered page, but the descriptor is the source of truth. Drift between them hides config changes from readers.
    • Enforcement: a catalog-render check that runs at commit time, on the Make target, and in the cloud build. Hard gate.
  4. Autogen source ↔ rebuilt artefacts.
    • When you edit: any file that feeds the auto-generators — diagram source, doc-comments in Python, catalog descriptors, or content blocks with autogen markers.
    • You must also edit: nothing by hand — just run make -C services/portal docs-fix and stage everything it touches.
    • Why: the drift gate in the build script will fail the commit otherwise.
    • Enforcement: the drift gate on the Make target and in the cloud build. Hard gate.

Adding a new check or a new pairing

This page is the single place to update when a new check ships. Steps:

  1. Decide where it runs. On commit if it's fast and only reads the files you staged. As a make target if it's something you'd want to run by hand on demand. In the cloud build if it needs a clean environment, a secret, or a schedule.
  2. Write the script. Place it under tools/<area>/<name>.py. Exit with a non-zero status on failure, and print enough context that a reader can fix the problem without opening your code.
  3. Wire it in. Add either a local hook to the pre-commit config, a target to the portal's Makefile (and into docs-check if it should run automatically before push), or a workflow file under .github/workflows/.
  4. Add a row to one of the tables above (A, B, B2, or C, depending on where the check runs). All five columns.
  5. If your check enforces a paired edit, add a row to Table D as well. What triggers it, what must change with it, why, what enforces it, and whether it's an automated check or still a memory rule.
