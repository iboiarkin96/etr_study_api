Internal · Operating model · Reference

Every gate, what it checks, how to fix a fail, and what travels with it

One inventory of every quality gate this portal runs — pre-commit hooks, Make targets, CI workflows — with five columns per row: name, when it runs, what it checks, how to fix a fail, and the coupling rule (what other edits must travel with it). The coupling column is the surface this portal was missing.

Why an inventory and not just hook configs

The hooks are defined in three different files — .pre-commit-config.yaml, services/portal/Makefile, .github/workflows/*.yml — and the coupling rules between them live in nobody's head. Pages like the lifecycle of a change tell you when a gate fires; this page tells you what it checks, how to fix a fail, and what other files have to change at the same time.

Read the coupling column first. Empty cell (—) means the gate is self-contained: pass it and move on. Populated cell means there is a structural rule a human cannot remember reliably — the gate exists or is being added precisely so CI does the remembering.

  1. A · 20 hooks · pre-commit

    Run at git commit on the staged diff. Defined in .pre-commit-config.yaml. Fail-fast; cheap. Includes generic correctness, Python lint/type, repo-specific docs validators, and the two coupling gates.

B2 · Root verify slots — one named gate per service

Five composite targets live in the root Makefile. Each one is a thin wrapper that delegates to its service's verify (or, for frontend, prints a no-op note). verify-all chains them in declared order; verify stays as a backward-compat alias for verify-all. This is the same composition ci.yml runs across its four matrix lanes.

Gate When it runs What it checks How to fix a fail Coupling

D · Coupling rules

This is the surface this portal was missing. A coupling rule is a structural «edit X ⇒ also edit Y» a single author cannot reliably remember. Each rule has a trigger path, a required co-change, an enforcer, and a state (hard gate or memory rule). The two pre-commit coupling gates landed on 2026-06-13; the remaining rules stay as memory rules with a documented home so they are not invisible.

Adding a new gate or a new coupling rule

The page is intentionally one place to edit. Steps:

  1. Decide where it runs. Pre-commit if the check is fast and reads the staged diff; Make if it is a recipe an author runs by hand; CI if it requires the cloud (matrix, deploy keys, schedule).
  2. Write the script. Under tools/<area>/<name>.py. Exit non-zero on failure; print enough context for an author to fix it without opening the source.
  3. Wire it in. One of: a repo: local hook in .pre-commit-config.yaml; a target in services/portal/Makefile (and into docs-check if it should run in CI's verify chain); a workflow under .github/workflows/.
  4. Add a row to the table above. Pick the right environment (A / B / C). Five columns.
  5. If it encodes a coupling rule, add a row to table D too. Trigger path, required co-change, why, enforced by, state.
  6. Add an entry to the whats-new feed. Four columns: date · what · why · what changes for you. The meta-changes gate refuses the commit otherwise.
