Domain model — six bounded contexts

The Study App splits into six bounded contexts, each with one owner, one purpose, and its own vocabulary. Words mean different things across them — the «user» in Identity is not the same as the «user» in Progress, and that's by design, not a mistake. This page lists the six contexts, how they connect, the shared vocabulary, the event shape they use to talk to each other, and the rules we don't break.

The six contexts

Each context owns one slice of business meaning, one team, and one source of truth. Same word can mean different things in different contexts — that's the point.

Context What it owns Owner Storage
Learning Authoring, structure, and delivery of study material. Owns course, module, lesson, exercise. Content guild Relational + object store
Identity & access Who the actor is and what they're allowed to see. Owns user, session, role, entitlement. Platform team Relational + key-value cache
Progress Learner state over time — what was attempted, what passed, what is still pending. Owns attempt, mastery, streak. Learning team Append-only log + materialised views
Assessment Scoring, grading policy, and certificate issuance. Owns rubric, grade, certificate. Learning team Relational; certificates in object store
Catalog (commerce) What is on offer, at what price, with what entitlements attached. Owns offering, price, promotion. Commerce team Relational + cache
Observability How the system reports its own state. Owns metric, log, alert, incident, runbook. SRE Time-series + S3 archive

Reading rule. «User» in Identity means «account holder»; «user» in Progress means «the learner whose attempts we count». Same word, two definitions, two contexts — no shared schema.

How they connect

Each row says how one context depends on another and what shape that dependency takes. The pattern names — published language, customer/supplier, anti-corruption layer, conformist, shared kernel — are the standard ways two contexts integrate.

Upstream Downstream Pattern Why this pattern
Identity Learning · Progress · Assessment · Catalog Published language One canonical session/token shape; every downstream parses it the same way.
Learning Progress Customer / supplier Progress's schema follows Learning's course/module IDs; Learning publishes changes ahead.
Assessment Progress Customer / supplier Progress consumes Assessment's grade events; Assessment owns the rubric versioning.
Catalog (commerce) Identity Anti-corruption layer Catalog talks to a third-party billing vendor; the layer keeps vendor terms out of our domain.
Catalog (commerce) Learning Conformist Catalog adopts Learning's course entity verbatim — no remapping, simplest integration.
All contexts Observability Shared kernel Single tracing schema; every team owns a slice of it together.

Shared vocabulary

Terms that appear in API contracts, in the UI, and in code. These spellings are exact — no synonyms inside a single context.

Term Meaning Owner
courseA bundle of modules sold or assigned as one offering.Learning
moduleA chapter inside a course. Owns lessons.Learning
lessonOne unit of study content — text, video, or exercise.Learning
exerciseA graded question or task inside a lesson.Learning (graded by Assessment)
attemptOne submission of an exercise by a learner. Immutable.Progress
masteryA derived score (0–1) of how well a learner knows a module.Progress
rubricVersioned grading rules that turn an attempt into a grade.Assessment
gradeThe output of applying a rubric to an attempt.Assessment
certificateSigned credential issued after course completion.Assessment
offeringA course or bundle listed for sale.Catalog
entitlementA purchased right to access an offering.Catalog (consumed by Identity)
sessionAn authenticated period bound to a user agent.Identity
roleCoarse-grained authorisation bundle (learner, author, admin).Identity
incidentAn open production failure tracked end-to-end with a postmortem.Observability
runbookThe on-call playbook for a specific alert.Observability

How contexts talk to each other

When one context emits a fact for others to consume, it uses this envelope. The envelope itself is shared by every context; the data block is whatever the producer publishes.

event_id: "01HVXY7M2K1S8KQKZJ4N5R7T9P" # ULID, source of truth for dedup
event_type: "assessment.grade.published.v1"
occurred_at: "2026-05-24T10:14:33Z"
producer:
 bounded_context: "assessment"
 service: "assessment-api"
 version: "1.7.2"
subject:
 user_id: "usr_3K9..."
 attempt_id: "att_8X1..."
data:
 rubric_version: "2026.03"
 grade: 0.82
 passed: true

Rules we don't break

  • One owner per term. Every term in the vocabulary belongs to exactly one context. Cross-context use happens through the shared language, not by sharing schemas.
  • No upstream calls downstream. Identity does not know about Progress; Learning does not know about Catalog. Cycles in the context map are bugs.
  • Events are immutable. A grade event published yesterday is still true today; corrections are new events, not edits.
  • Rubric versioning is visible. Every grade carries the rubric version that produced it. Re-grading uses a newer rubric and emits a new event.
  • Anti-corruption layers are explicit. Vendor terms (billing products, identity providers) never leak past their layer. The mapping code lives in one file per layer.

What this looks like for one team

One file a team commits alongside its code. The same five sections as above (contexts, relationships, vocabulary, events, invariants), no sprawling folders.

version: "2026.05"
team: "learning-platform"
owners: ["@learning-eng", "@learning-pm"]

contexts:
 - name: "learning"
 purpose: "Author and serve courses, modules, lessons, exercises."
 owns: [course, module, lesson, exercise]
 data_store: "postgres"
 - name: "assessment"
 purpose: "Apply rubrics to attempts; mint grades and certificates."
 owns: [rubric, grade, certificate]
 data_store: "postgres"
 - name: "progress"
 purpose: "Append-only learner history; derive mastery."
 owns: [attempt, mastery]
 data_store: "postgres + materialised views"

relationships:
 - from: progress
 to: assessment
 pattern: "customer-supplier"
 note: "Progress sends attempts; Assessment returns grades."
 - from: assessment
 to: learning
 pattern: "conformist"
 note: "Assessment grades inherit learning's exercise shape."

language:
 attempt: "One submission of an exercise. Immutable in Progress."
 grade: "Output of applying a rubric to an attempt. Owned by Assessment."
 mastery: "Derived score (0–1) of how well a learner knows a module."

events:
 - type: "assessment.grade.published.v1"
 producer: "assessment-api"
 consumers: ["progress-projector", "notification-fanout"]
 retention: "180d"

invariants:
 - "An attempt has exactly one grade."
 - "A certificate exists only if all required grades are 'passed'."
 - "Mastery is recomputed from attempts; it is never written directly."

A team that opens a new repo copies this file, swaps the names, and is most of the way to a published domain model.

What we gave up to get here

  • Six contexts, not three. Splitting Identity and Catalog from Learning increases integration cost — we took it to keep team ownership clean.
  • Progress as append-only. Materialised views add latency on schema changes — we took it because audit replay is more valuable than write-path simplicity.
  • Catalog as a conformist to Learning. Catalog inherits Learning's pace of change — we took it because Catalog is small and the integration is read-mostly.
  • Shared tracing kernel. Every team has to coordinate on schema edits — we took it because consolidated dashboards beat per-team observability silos.
