Cross-service workflows

Workflows that cross more than one service: a learner completing an exercise, an alert turning into an incident, a documentation page going live. Each one has a sequence diagram, a numbered step list, and (for the hot path) an error matrix. The diagrams below are ASCII previews of the canonical PlantUML sources; the real diagrams ship as SVG.

Why these live here

A workflow that crosses services has no single owner. No one service can document it alone, and if we leave it to chance, it ends up duplicated and out of sync across team docs. This page is where these workflows live, with one source file each, so the description doesn't fragment.

  • One source per flow. Each workflow has one diagram source. Service pages link to it; they don't copy it.
  • When it changes. A workflow edit needs a written decision when it changes who owns a step or what the contract is — not for cosmetic tweaks.
  • What lives here. Workflows that touch three or more services land on this page. Smaller flows live inline on the service that owns them.
  • Diagram format. PlantUML, rendered to SVG. Mermaid isn't accepted — the GitHub renderer for it drifts from what you see locally.

Workflow 1 — Learner submits an exercise

The hot path. A learner clicks «submit» in the web app and a grade comes back. Touches Identity (session check), Learning (content lookup), Assessment (grading), Progress (mastery update), and Observability (tracing). Synchronous on the response; the mastery update is async.

Learner          Web            Identity        Learning        Assessment      Progress
  |                |               |                |                |                |
  | submit attempt |               |                |                |                |
  | ──────────────►|               |                |                |                |
  |                | verify session|                |                |                |
  |                | ─────────────►|                |                |                |
  |                | ◄─────────────|                |                |                |
  |                | fetch exercise                 |                |                |
  |                | ──────────────────────────────►|                |                |
  |                | ◄──────────────────────────────|                |                |
  |                | POST grade                                      |                |
  |                | ───────────────────────────────────────────────►|                |
  |                |               |                |   grade.published (event)       |
  |                |               |                |                | ──────────────►|
  |                | ◄───────────────────────────────────────────────|                |
  | grade visible  |               |                |                |                |
  | ◄──────────────|               |                |                |                |
  |                |               |                |                |                | update mastery (async)

Steps

  1. Web receives the submit request with the learner's payload.
  2. Identity verifies the session token — a rejected request returns 401 immediately.
  3. Learning fetches the exercise definition (the rubric version is pinned at this read).
  4. Web forwards the attempt and the pinned rubric to Assessment.
  5. Assessment grades synchronously and returns the grade to Web.
  6. Assessment emits a grade.published event on the event bus.
  7. Progress consumes the event asynchronously and updates the mastery view.
  8. Observability captures a trace for every hop through the shared tracing layer.

Workflow 2 — Alert turns into an incident

The reliability spine. Starts when an alert fires with a runbook link in its payload and ends when a postmortem is published and attached to the incident record.

Prometheus     Alertmanager  PagerDuty     On-call       Runbook       Postmortem
  |               |              |              |              |              |
  | rule fires    |              |              |              |              |
  | ─────────────►|              |              |              |              |
  |               | route + group|              |              |              |
  |               | ────────────►|              |              |              |
  |               |              | page on-call |              |              |
  |               |              | ────────────►|              |              |
  |               |              |              | open runbook |              |
  |               |              |              | ────────────►|              |
  |               |              |              | execute      |              |
  |               |              |              | ─────────────|              |
  |               |              |              | open incident                |
  |               |              |              | ───────────────────────────►|
  |               |              |              |              |              | draft + review (48h)
  |               |              |              |              |              | publish

Steps

  1. Prometheus evaluates an alert rule and fires when the condition holds.
  2. Alertmanager deduplicates, groups, and routes by severity and service.
  3. PagerDuty pages the on-call engineer; the payload carries a runbook link.
  4. On-call opens the runbook and follows the diagnostic steps.
  5. If mitigated within 30 minutes, the alert auto-resolves and on-call files a short note.
  6. Otherwise, on-call opens an incident record with severity, services touched, and customer impact.
  7. A postmortem draft is owed within 48 hours of incident close (for severity 1 and 2 only).
  8. The postmortem is reviewed by someone other than the author, then published and linked from the incident.

Workflow 3 — A page goes live

The same pipeline product code uses. Automated checks run first, then a human review, then a merge that rebuilds the search index. The PR stays fast because the index doesn't rebuild on every CI run.

Author          PR             CI (auto)       Reviewer        Merge job       Portal
  |                |               |                |                |                |
  | open PR        |               |                |                |                |
  | ──────────────►|               |                |                |                |
  |                | trigger checks|                |                |                |
  |                | ──────────────►|                |                |                |
  |                | ◄──────────────|                |                |                |
  |                | request review                 |                |                |
  |                | ──────────────────────────────►|                |                |
  |                | ◄──────────────────────────────|                |                |
  |                | merge                                          |                |
  |                | ───────────────────────────────────────────────►|                |
  |                |               |                |                | regenerate index
  |                |               |                |                | ──────────────►|
  |                |               |                |                |                | sidebar refresh

Steps

  1. Author opens a pull request from a feature branch. The PR template asks what changed, why now, and who reviewed.
  2. CI runs the automated checks — heading structure, asset references, page metadata, accessibility baseline, file paths.
  3. A failing automated check blocks the PR. There is no human override. The author fixes and re-pushes.
  4. Reviewer applies the human review: does the page fit its kind, does the narrative hold, are the facts right, are the links the right ones.
  5. The reviewer requests changes inline or approves. Approval requires someone other than the author.
  6. Merge job rebuilds the search index before publishing the new page to the live site.
  7. Portal picks up the new sidebar entry on the next page load (cached navigation has a short time-to-live).

Owner: Documentation platform · Last reviewed: 2026-05-24

Adding a new workflow

  • One source file. Each workflow has one diagram source; pages render the SVG, they don't duplicate the diagram in another doc.
  • Only for cross-service flows. Promote a flow to this page only when it touches three or more services. Smaller flows live inline on the service that owns them.
  • Three blocks per flow. Sequence diagram, numbered step list, and either an error matrix (for hot paths) or an owner table (for everything else).
  • Dated. The «last reviewed» date is mandatory. The freshness check warns at 90 days and fails at 180.
  • Anchored. If a step's owner or contract changes, the workflow update needs a written decision first.
