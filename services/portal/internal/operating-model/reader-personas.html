Internal · Operating model · Explanation

Who reads this portal

Four reader personas drive every decision about how this portal is arranged — who each one is, what they came to find, and where the sidebar should take them. A new page that doesn't serve at least one of them doesn't ship. Authors and reviewers use this page to keep that bar honest.

Why personas instead of «just write good docs»

A six-hundred-page portal is no longer a single document — it's a small product. Without a model of who reads it, authoring drifts into «what looks complete» rather than «what gets read».

  • It makes the question testable. «Is this discoverable?» becomes «does the reader reach this page in three clicks or fewer?».
  • It keeps scope honest. A page tuned for a newcomer reads differently than one tuned for an architect. Naming the audience stops middle-ground prose that helps no one.
  • It gives reviewers a single question. «Which persona does this serve, and does the first viewport prove it?» — instead of chasing taste.
  • It surfaces hubs. If a page tries to serve three personas equally, it's probably a hub — split it into a hub plus three deep-dives, not one wall of headings.

The four personas

Newcomer engineer

Day 1, Week 1, first PR. «I have a laptop, a git clone, and a vague brief. Where do I start, what runs locally, and which PR won't embarrass me?»

GoalGet to a green build locally, then ship a docs-only PR through the same pipeline as production code.
FrictionDoesn't yet know the domain vocabulary; needs the glossary inline, not two levels deep.
Time budgetAbout two hours on Day 1, about one day across Week 1.
Done whenFirst PR merged via the docs-as-code pipeline; all checks pass.
Top pagesCold-start path, Glossary, How pages are arranged

SA peer — systems analyst

Reviewing a proposal. «Show me the requirement, the domain term it touches, the decision that froze it, and which API operation implements it.»

GoalTrace one business requirement end-to-end: requirement → decision → API operation → test → runbook.
FrictionIf the chain is broken anywhere, that gap is the actual finding.
Time budget10–15 minutes per requirement during async review; longer for a new feature kickoff.
Done whenEvery requirement row in the traceability matrix has no empty cells, or each empty cell is filed as work.
Top pagesTraceability matrix, Domain model, Cross-service workflows

SWE — software engineer

Building or shipping. «I need the exact request shape, the failure modes, and the on-call page if my change breaks production. Anything else is reading for fun.»

GoalImplement against a contract, prove the implementation matches it, know who to page when it breaks.
FrictionDoesn't want prose between sections of YAML. Wants the snippet first, the explanation second.
Time budget2–5 minutes per lookup, many times a day.
Done whenAn API page renders the request shape, status codes, and the runbook link without a page reload.
Top pagesAPI reference, API runbooks, service-level pages

Architect — staff or above

Reasoning about the system as a whole. «Show me the system in one diagram, the decisions that froze it, who owns each piece, and what we'd have to undo to change it.»

GoalReason about the system as a whole — components, contracts, blast radius, ownership.
FrictionDiagrams without an «as-of» date are noise. Every diagram should be linked to the source it was rendered from.
Time budgetShort bursts during design review; longer during quarterly audits.
Done whenCan answer «if I remove component X, what breaks?» from the catalog and the architecture pages alone.
Top pagesHow pages are arranged, Domain model, Decision records, Catalog taxonomy

Every page is a four-step journey

Each reader takes the same four-step trip across the portal, regardless of persona. Writing a page means asking «which step am I writing for?». Pages that try to cover all four become hub pages.

  1. Land. The reader arrives from search, the sidebar, or an outbound link. The first viewport must answer «am I on the right page?».
  2. Orient. The breadcrumb and the lede establish where in the portal they are and which persona this page targets.
  3. Act. The body delivers the thing the persona came for — a snippet, a table, a diagram, a checklist.
  4. Leave. Inline «See also» links and the sidebar push the reader to the obvious next page, not back to a hub.

Anti-patterns — pages that fail at least one persona

Patterns reviewers reject. Each one quietly ignores one of the four personas.

  • The universal hub. One page that tries to onboard the newcomer, satisfy the SA peer, and brief the architect at once. Result: no persona reaches what they need within their time budget. Fix: split into a hub plus per-persona deep-dives.
  • The diagram without a date. A diagram with no «as-of» timestamp. The architect can't tell whether it reflects reality or last quarter's plan. Fix: every diagram has a date and a link to the source it was rendered from.
  • The contract-as-prose. An API page that describes the request in English instead of a code block. The SWE has to mentally translate the prose back into code. Fix: snippet first, prose second. Prose explains «why» — it doesn't re-state the schema.
  • The acronym wall. A page that opens with five abbreviations in the first paragraph without expansion. A newcomer bounces inside 30 seconds. Fix: spell out every acronym the first time it appears on the page.
  • The broken trace. A requirement with no link to a decision; a decision with no link to the operation it freezes. The SA peer can't complete the chain. Fix: the traceability matrix shows empty cells as bugs to fix, not gaps to live with.
  • The dead-end leaf. A page that delivers its content and stops. The reader has to go back to the hub to continue. Fix: the last block of every leaf page suggests the next page inline, not as a separate footer.

How authors and reviewers use this page

  • Authors. Before writing, name the persona in your hero eyebrow. If you can't pick one, the page is probably a hub — split it.
  • Reviewers. Open the PR side-by-side with this page. Ask «which persona does this serve, and does the first viewport prove it?». If not, request a hero rewrite, not body edits.
  • LLM-assisted writing. Tell the assistant which persona to target. Output drifts less when the audience is explicit.
  • Periodic audit. A quarterly check picks a handful of random pages, scores each against its declared persona's «done when». Failing pages go to the backlog with an owner.
