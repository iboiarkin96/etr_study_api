What's changed in how we work

When something shared changes — a build command, an automated check, a workflow, a template — it gets logged here, so the rest of the team can see what changed without reading the commit log. Each entry answers four questions: when, what changed, why, and the one that matters most — what changes for you. The hub home shows the most recent few; this page has the full history.

Why a written feed and not the commit log

The commit log tells you what changed in the source, but not why it changed or what you need to do differently. This feed is written by hand precisely so each entry forces the author to write that last column. An automated check refuses to commit a change to anything shared (a build command, a check, a template, a workflow, or a new architecture decision) without an entry added here in the same commit.

What counts as «something shared»: pre-commit configuration, cloud build workflow files, the portal's Makefile, page templates, a new architecture decision record, or any script under tools/docs/. If you're touching one of those, this feed needs an entry from you.

Entry shape

Every entry has the same shape. Authoring rules below mirror what check_meta_changes_logged.py checks at every commit.

SlotWhat it isRule of thumb
docs-history__dateISO date the change shipped (merge date, not commit date)One date per entry; backdate only if writing the entry late
docs-history__authorAuthor chip pointing at the person who shipped the changeAlways one person, even when the work was a pair. Reviewer credit lives in the ADR, not here
<strong>Title.</strong>One-sentence headline ending in a periodThe headline is what shows in the hub-home preview; keep it scan-friendly
What:The concrete change — files, scripts, ADRsName paths; no generalities. «Added X, deleted Y, rewrote Z»
Why:The reason the change happenedOne or two clauses; cite the audit, the bug, or the constraint that forced it
What changes for you:The author-facing consequenceThe slot the gate exists for. «Now you must…», «From now on, X is silently…»; if there is no author-facing consequence, the change probably did not need a meta-log entry
«See …»Pointer to the ADR / RFC / commit that decided the changeOne link; if the change involved several ADRs, list the primary one

How to add a new entry

Six steps. The pre-commit hook fails the commit unless you do them.

  1. Confirm the change is a meta-change. Look at the coupling rule on the quality gates map § Meta-change ↔ feed entry. If your diff matches any of the trigger paths (.pre-commit-config.yaml, .github/workflows/*.yml, services/portal/Makefile, an SA template, a new operations-class ADR, anything under tools/docs/), the entry is required.
  2. Copy a recent entry. Any of the five above is fine — the shape is identical. Paste it at the top of the docs-history__list in this file.
  3. Set the date. Today's ISO date in the docs-history__date span.
  4. Fill the title and the four slots — What, Why, What changes for you, and the «See …» link. The «What changes for you» slot is the one a reader will actually read; spend most of the time there.
  5. Mirror the entry to the hub home feed. The hub home (operating-model/index.html) shows the last ten entries inline; if you push the bottom-most one off, remove it from the hub home in the same PR.
  6. Bump data-updated on both this page and the hub home to today.

What this feed is not

  • Not a changelog of every commit. The repository changelog (CHANGELOG.md) lives at the root per ADR-0013. This feed is the meta-process slice only.
  • Not a release-notes page. User-facing release notes are an API-contract concern; see the OpenAPI baseline diff per ADR-0007.
  • Not the ADR index. ADRs live at the ADR index and have their own lifecycle. The feed cites the ADR for context but never duplicates its content.
  • Not auto-generated. Auto-generating the «What changes for you» column from the commit message is impossible because the message rarely contains it. The gate exists precisely to force the author to write it.
