Internal · Operating model · Explanation

From idea to live page — ten steps

Every change to this portal — a small edit, a new section, or a new tool — goes through the same ten steps. The first five happen on your computer; the next two move into the cloud; the last three put the change in front of readers. The diagram below shows the path; each step has its own block with what you do, what runs automatically, and what tends to fail there.

Why one path for every change

Every change ships through the same path because the automated checks only catch what you'd expect them to when the path is predictable. Skipping a step to save a minute usually costs an hour debugging the drift it creates. The ten steps below are the only path; the checks at each step exist to keep you on it.

This page is the visual map. The longer reasoning — why we picked this path and why each check is there — lives in the SA process page. Use this page when you're mid-task and need «what comes next»; use the handbook when you want to understand why.

Three stages

The ten steps split into three stages by where the change lives. Each stage ends with a publish.

  1. A · Steps 1–5

    On your computer

    Decide what to change, branch, copy a template, write the content, run the automated check. Everything here runs on your laptop, and the point is to fail fast and cheap before anything reaches the cloud.

    Steps 01–05→
  2. B · Steps 6–8

    Review and staging

    Push, open the PR, the cloud build checks everything again, a reviewer approves, the change merges to the staging branch and auto-publishes to the staging URL. This is where bugs that only show up under a real web server surface — before any reader sees them.

    Steps 06–08→
  3. C · Steps 9–10

    Live for readers

    Smoke-test on the staging URL, open a second PR from staging to main, merge after review, and the change auto-publishes to the live URL. Going live is a merge, not a deploy script. Rolling back is a single revert.

    Steps 09–10→

The ten steps

The full path, colour-coded by stage. Click any circle to jump to its detail block below.

Each step in detail

Each block below has the same three parts: what you do, what runs automatically, and what tends to fail here.

01 — Decide what you're changing

What you do. Name the change in one sentence. Decide which kind it is: a single-page edit, a refactor that touches several pages, or a change to something everyone uses (a build command, a shared check, a template, a workflow). The last category may need a written decision first.

What runs automatically. Nothing here — judgment is the only check.

What tends to fail here. Scope creep. A «quick edit» turns into a refactor without anyone noticing. If a single PR touches more than three top-level sections of the portal, split it before you write code.

02 — Branch from main

What you do. Create a feature branch from main, not from staging. The convention is feature/<short-name>. Staging is an integration branch — branching from it would carry someone else's unmerged work into your PR.

What runs automatically. Nothing until the first commit.

What tends to fail here. Branching from staging by accident. The PR then carries other people's pending changes into live when it eventually promotes.

03 — Copy a template

What you do. Open the templates and copy the one that matches what you're writing (a decision record, a runbook, a how-to, a tutorial, and so on). Rename it to the path the routing questionnaire gave you. Fill in the six required labels at the top of the file.

What runs automatically. If you forget the maintainer block at the bottom of the page, a pre-commit check adds it for you. Everything else is on you.

What tends to fail here. Rearranging the template's structure. The section order is part of the contract — readers and the automated design check both expect it.

04 — Write the content and check locally

What you do. Write the page. After meaningful saves, run make docs-fix followed by make docs-check. The first regenerates everything that's auto-generated (search index, indexes, references, diagrams). The second confirms nothing fell out of sync — if it complains, something needs to be re-generated or committed.

What runs automatically. The auto-fix runs ten steps in order: re-render diagrams, sync content from code, render any companion HTML, repair structure, normalise formatting, refresh the service catalog, ensure the maintainer block, refresh the code reference, rebuild the indexes, and rebuild the search index.

What tends to fail here. «Drift on a file you didn't edit.» That's the auto-fix updating something auto-generated — usually the index or the search. Re-run the auto-fix, commit everything it changed, try again.

05 — Commit, and the checks run

What you do. Stage your changes and commit. About twenty automated checks then run on what you staged.

What runs automatically. The checks look at: the PR description, file types that shouldn't be staged, the maintainer block on each page, the design structure, links to decision records, asset references, the page heading structure, Python import boundaries, the rendered service catalog, and the standard YAML / JSON / large-file / merge-conflict guards. Plus the two newer pairing checks — they refuse the commit if you changed a shared command but didn't update the «what's new» feed, or changed a template without its matching design example.

What tends to fail here. A skipped step 4. The structure or asset checks catch the result. Re-run the auto-fix, re-stage, try the commit again.

06 — Push and open the PR

What you do. Push your branch to GitHub and open a pull request against the staging branch (not main).

What runs automatically. A pre-push hook copies your local PR_BODY.md into the PR description so the two stay in sync.

What tends to fail here. Opening the PR against main by accident. Re-target it before you ask for review.

07 — The cloud build runs

What you do. Wait, and watch the Actions tab for the result.

What runs automatically. The cloud build runs make verify-all on a fresh copy of the repository — the same composition you'd run locally with make verify-all. A separate workflow runs the accessibility score check (axe-core) on every push.

What tends to fail here. Something the local auto-fix skipped — usually a Python version difference. The canonical Python version is 3.11; pin your local environment to it.

08 — Review and merge to staging

What you do. Respond to review comments. When approved, squash-merge the PR into staging.

What runs automatically. The merge triggers the staging deploy workflow, which builds the site and publishes it to the staging URL. The URL refreshes within about a minute.

What tends to fail here. The deploy token expires (it's a personal access token used to push to the staging repository). The workflow's deploy step fails with a 403; rotate the token.

09 — Smoke-test on the staging URL

What you do. Open the staging URL and click through the change. Open the browser's network panel and look for any 404s — there shouldn't be any. Check the theme toggle, the search box, and in-page navigation.

What runs automatically. Nothing automated here yet. A visual-regression test suite is being prepared and will run on PRs once it lands on main.

What tends to fail here. Asset 404s caused by a wrong base path. Check the network panel for the failing requests.

10 — Promote to live

What you do. Open a second PR, this time from staging to main. After review (the live environment requires an additional reviewer), merge.

What runs automatically. The live deploy workflow publishes the site to the live URL within about a minute. No build artefacts are committed to the repository.

What tends to fail here. The staging branch has diverged from main — someone landed a hotfix directly on main and didn't forward-port it. Pull main into staging, then re-open the promotion PR.

Rolling back

Taking a live change back is a git operation, not a deploy procedure.

  1. Find the bad commit on main. It's usually the most recent merge from staging. Look at the last ten commits.
  2. Open a revert PR against main. Create a branch with git revert <commit> and open a PR with revert: in the title. Get one review.
  3. Merge the revert. The live deploy workflow re-publishes the previous state within about a minute.
  4. Carry the revert back into staging. Cherry-pick the revert into the staging branch so the next promotion doesn't bring the bad change back.
  5. Write up what happened. If readers saw the broken state, write a postmortem. If the problem exposed a gap in this process, add an entry to the «what's new» feed.

If you ever need to disable a deploy: removing the staging deploy workflow and its access token disables staging without affecting live. Removing the live deploy workflow freezes the live URL at the last published version until the workflow is added back.

Two paired-edit rules that fire here

Two of the paired-edit rules from the checks reference fire inside this path, and you'll meet them at step 5:

  • Template change requires its matching design example. If you change a page template, its matching design example must change with it. An automated check refuses the commit otherwise.
  • Shared-process change requires a feed entry. If you change a build command, a shared check, a workflow, a template, or write a new architecture decision, you must add an entry to the «what's new» feed in the same change. An automated check refuses the commit otherwise.

The full list of paired-edit rules and where they're enforced is on the checks reference.
