From idea to production — ten stations, three branches

The full path of any change to this portal — a runbook edit, a new ADR, a Make target, a pre-commit hook. Ten stations grouped into three branches (local → staging → prod), with the gates that fire at each one and what you do versus what CI does for you.

Why one canonical lifecycle

Every change ships through the same path because the gates only work when authors trust the path. Skipping docs-fix, merging without a PR, deploying directly from a feature branch — each saves a minute and costs an hour of debugging the drift it creates. The ten stations below are the one path; the gates pin it in place.

This page is the visual map of the lifecycle. The deep narrative — the eight operating rules behind these stations, the rationale for each gate — lives in handbook/sa/process. Use both: this page when you're inside a session and need «what comes next»; the handbook when you want to understand why.

Three branches at a glance

The path splits into three branches by environment. Each branch ends in a deploy.

  1. A · stations 1–5

    Local

    Idea → branch → template → write loop → pre-commit. Every gate that runs here runs on your laptop; the goal is to fail fast and cheap before anything reaches CI.

    Stations 01–05→
  2. B · stations 6–8

    Staging

    Push → PR → CI matrix → review → merge to staging → auto-deploy to the staging URL. Pagefind base paths, asset paths, theme persistence — the things that only fail under a real GH-Pages domain — surface here, not in prod.

    Stations 06–08→
  3. C · stations 9–10

    Prod

    Smoke-test on staging → PR staging → main → auto-deploy to the prod URL. The promotion is a git merge, not a deploy script. Rollback is a single revert.

    Stations 09–10→

The ten stations

Walk top-down. Each station has a detail block at § Each station in detail below; the timeline title links to that block's anchor.

  1. Decide scope. Is this a one-page edit, a multi-page refactor, or a process change? Operations-class changes need an ADR; cross-page refactors should have one too.
  2. Open a feature branch. feature/<short-name> from main. Per ADR-0017.
  3. Pick a template. Copy the matching shape from handbook/sa/templates/. Rule 4 of the operating-model hub.
  4. Write + local loop. Write content; run docs-fix to regenerate autogen; run docs-check to gate drift; iterate.
  5. Pre-commit gates fire. Twelve local hooks scan the staged diff at git commit — PR body, artefact files, maintainer chip, design baseline, ADR/RFC links, asset refs, h1 structure, service-import boundaries, catalog drift.
  6. Push and open the PR. git push runs sync_pr_body which syncs your local PR_BODY.md to the PR description; the PR opens against staging.
  7. CI matrix verifies. GitHub Actions runs make verify across four service matrices (api / portal / monitoring / cross-cutting); the same drift gate from station 04 runs here against a clean clone.
  8. Review, merge to staging, auto-deploy. Squash-merge into the staging branch triggers .github/workflows/deploy-staging.yml which publishes to the mirror repo (per ADR-0034). The staging URL updates within ~60 s.
  9. Smoke-test on the staging URL. Open the page on the real GH-Pages domain. Network panel filtered to 404s should show zero asset failures. This is where Pagefind base-path and asset-path bugs surface.
  10. Promote to prod. Open a second PR staging → main. Merge after review (a required-reviewer rule guards the production environment). deploy-prod.yml runs actions/deploy-pages and the prod URL updates.

Each station in detail

Each block names: your action, automation, what gates fire, and what fails most often here.

01 — Decide scope

Your action. Name the change in one sentence. Decide whether it's (a) a one-page edit, (b) a multi-page refactor, or (c) an operations-class change (new Make target / pre-commit hook / CI workflow / SA template / ADR). Categories (b) and (c) trigger downstream coupling (see Rule 5 of the operating-model hub).

Automation. None at this station — judgment is the gate.

What fails most often here. Scope creep. Pages that started as «one quick edit» grow into a refactor without their own ADR. If the same PR touches more than three top-level sections, split it.

02 — Open a feature branch

Your action. git switch -c feature/<short-name> from main (not from staging — staging is the integration branch). Naming follows ADR-0017.

Automation. None until the first commit.

What fails most often here. Branching from staging instead of main — your PR then carries someone else's unmerged staging work into prod when it eventually promotes.

03 — Pick a template

Your action. Open handbook/sa/templates/, copy the file that matches the artefact (ADR, RFC, runbook, postmortem, how-to, tutorial, audit, bug-report, screen-spec, etc.), and rename it to the target path the routing decision tree told you. Fill in the six data-* attributes on <body>.

Automation. The maintainer chip block is auto-injected by the docs-maintainers-fix pre-commit hook if you forget it. Everything else is on you.

What fails most often here. Reinventing the structure. The template's section order is a contract — readers and the design baseline both expect it.

04 — Write + local loop

Your action. Write content. After every meaningful save, run make -C services/portal docs-fix && make -C services/portal docs-check. docs-fix regenerates UML, marker-based docs, catalog views, pdoc, and Pagefind. docs-check compares the working tree before and after — if anything moved, you have drift.

Automation. Ten docs-fix steps run in order: regenerate UML → sync markers → render docs HTML → repair → normalise → render service catalog → ensure maintainers → pdoc Python reference → build catalog → build Pagefind index.

What fails most often here. «Drift on a file you did not edit.» That's docs-fix picking up a stale autogen output — usually the catalog or the search index. Re-run docs-fix, stage everything it changed, and try again.

05 — Pre-commit gates fire

Your action. git add + git commit.

Automation. Twelve hooks run against the staged diff:

  • require-pr-body — PR_BODY.md exists with one change type selected;
  • no-artifact-files — refuses to stage .coverage, *.db, PR_BODY.md, changelog drafts;
  • docs-maintainers-fix — auto-injects the maintainer chip block;
  • docs-design-maintainers — the design baseline (page-head, lede, breadcrumbs, card conventions);
  • check-adr-rfc-links — every href and anchor inside governance/adr/ + rfc/ resolves on disk;
  • check-asset-refs — every <link href>, <script src>, image, SVG resolves;
  • check-h1-structure — every page has exactly one <h1>;
  • check-service-imports — Python imports do not cross service boundaries (ADR-0028);
  • catalog-render-check — catalog-info.yaml matches the rendered HTML;
  • plus generic hooks (check-yaml, check-json, check-toml, end-of-file fixer, trailing whitespace, large-file guard, merge-conflict guard).

After C8 lands, two more local hooks gate coupling: check_meta_changes_logged.py (no meta change without a feed entry) and check_template_twins.py (no SA template change without the UI-kit twin).

What fails most often here. A missed docs-fix run from station 04 — the design or asset-ref hook catches it. Re-run docs-fix, re-stage, retry the commit.

06 — Push and open the PR

Your action. git push -u origin feature/<short-name>; open the PR against the staging branch on GitHub.

Automation. The pre-push hook sync-pr-body-on-push updates the PR description from your local PR_BODY.md using gh pr edit (silent if gh isn't installed or the PR doesn't exist yet).

What fails most often here. PR opened against main instead of staging. Re-target the PR before review.

07 — CI matrix verifies

Your action. Wait. Watch the Actions tab.

Automation. .github/workflows/ci.yml runs make verify on a clean clone. verify chains docs-check (drift gate) + docs-a11y-check (WCAG baseline). The matrix splits across four touchpoints: api, portal, monitoring, and cross-cutting; each runs the same gate against the same tree. a11y-formal-checks.yml (axe-core) runs on every push to confirm the formal a11y score is unchanged.

What fails most often here. A drift on a file the local docs-fix skipped — usually because the local Python minor differs from CI Python (3.11 is the canonical version; see CONTRIBUTING.md). Pin your venv to 3.11 to match.

08 — Review, merge to staging, auto-deploy

Your action. Respond to review comments. When approved, squash-merge into staging.

Automation. The merge triggers .github/workflows/deploy-staging.yml, which builds the _site artefact and uses peaceiris/actions-gh-pages@v3 with external_repository pointing at the <app>-staging mirror repo. force_orphan: true keeps the mirror's history a single commit. Staging URL refreshes in ~60 s. Per ADR-0034.

What fails most often here. The cross-repo PAT (STAGING_DEPLOY_TOKEN) expires. The workflow's «push to external repo» step fails with 403; rotate the PAT.

09 — Smoke-test on the staging URL

Your action. Open the staging URL. Click through the change. Open the network panel and filter on 404s — zero asset failures expected. Check theme toggle, search, in-page nav.

Automation. None (yet). A Playwright-based visual-test corpus exists locally (tests/visual/ on a feature branch); once it lands on main, make visual-test will run on a per-PR basis and compare against baselines.

What fails most often here. Asset 404s from a wrong base path — the portal's runtime project-base resolution sometimes lags an ADR-0034 amendment. Look at portal-href.js and search.js in the network panel.

10 — Promote to prod

Your action. Open a PR staging → main. After review (the production GitHub environment has a required reviewer), merge.

Automation. .github/workflows/deploy-prod.yml runs actions/upload-pages-artifact + actions/deploy-pages on the main branch. The official Pages deployer publishes within ~60 s. No artefact lands in git.

What fails most often here. Diverged staging — someone else's hotfix went direct to main and was not forward-ported. Pull main into staging before re-opening the promotion PR.

Rollback

Reverting prod is a git operation, not a deploy procedure.

  1. Identify the bad commit on main. Usually the most recent squash-merge from staging. git log --oneline -10 main.
  2. Open a revert PR against main. git revert <sha> on a new branch; PR with revert: prefix; require one review.
  3. Merge the revert. deploy-prod.yml re-publishes the previous state within ~60 s.
  4. Forward-port the revert to staging. Cherry-pick the revert into staging so the next promotion does not bring the bad change back.
  5. Log the incident. A postmortem at services/portal/postmortems/<date>-<title>.html if the broken state reached users; a feed entry at whats-new-in-how-we-work if the rollback exposed a process gap.

Disabling staging entirely is removing deploy-staging.yml and the STAGING_DEPLOY_TOKEN secret; the prod path is untouched. Disabling prod is removing deploy-prod.yml; the live URL freezes at the last published bytes until the workflow is re-added.

Coupling that hooks into this lifecycle

Two coupling-loops fire inside this lifecycle and you need to recognise them:

  • SA template ↔ UI-kit twin (station 05). If station 03 picked a template under handbook/sa/templates/ with a twin under portal/ui-kit/pages/templates/ (pairs declared in tests/visual/pages.json), both must change together. After C8 lands, check_template_twins.py blocks the commit otherwise; until then it's a memory rule. The visual baseline refreshes via make -C services/portal visual-test-update.
  • Meta-change ↔ feed entry (station 05). If the change touches .pre-commit-config.yaml, .github/workflows/*.yml, services/portal/Makefile, handbook/sa/templates/*.html, a new operations-class ADR, or a script under tools/docs/, the same PR must add an entry to whats-new-in-how-we-work. After C8 lands, check_meta_changes_logged.py blocks the commit; until then it's Rule 5 of the operating-model hub.
