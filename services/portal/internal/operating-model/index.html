How a page fits together

What's in this codebase

In three sentences

1. What you're working in

One codebase, four services: a backend API, this documentation site, a monitoring stack, and a shared design system. The documentation site is the largest of the four. The same risk shows up in all of them — a change in one place that doesn't get reflected somewhere else it needed to.

2. How a page is put together

Every page has to declare what kind of page it is, which service it belongs to, who owns it, and whether it's still live. It also has to use one of the shared templates and appear in the navigation. If any of this is missing, an automated check stops the page from going out.

3. What catches mistakes for you

Three layers of automated checks. Some run on your computer when you commit. Some you run yourself before pushing. The rest run in the cloud after your change leaves your machine. Together they cover almost every common mistake — you don't have to remember every rule by hand.

The five rules

Every rule below is enforced by an automated check. You don't need to memorise them — if you break one, the check will tell you what's wrong and what to do. Click any rule to read more.

  1. 01 · Where the page goes

    Put a page where its topic lives, not where its type lives

    A page about a particular service belongs in that service's folder. A page about something shared — like a guide everyone reads, or a procedure used across teams — belongs in the handbook. The kind of page (tutorial, how-to, reference, explanation) is a label on the page itself; it doesn't dictate where the page lives. If you're unsure, the routing tree asks a handful of questions and tells you the answer.

    Find the right place →
  2. 02 · How a page is labelled

    Every page declares six things about itself

    Every page must say what kind of page it is, which service it belongs to, who it's for, what topic it covers, whether it's still live, and when it was last updated. These labels are how the portal builds its indexes and search. Without them, the page silently disappears from those views — an automated check refuses to let you commit a page with any of them missing.

    See what to declare →
  3. 03 · Before you commit

    Run the auto-fix, then the check

    Before every commit, run two commands in this order: make docs-fix regenerates anything that's auto-generated (search index, indexes, references). make docs-check then confirms nothing drifted. Without this, the page itself can be fine while the search index falls behind — pre-commit hooks alone don't catch that. For changes to a particular service, also run that service's check (for example make verify-api).

    See every check →
  4. 04 · How you start a new page

    Always copy from a template

    A new page always starts as a copy of one of the existing templates — there's one for each common kind of page (a decision record, a how-to, a runbook, a postmortem, and so on). The template's structure is part of how the portal works, not a starting point to rearrange. If a template feels wrong for what you're writing, that's a signal to either pick a different one or talk about adding a new template — not to break the structure of the one you started from.

    Browse the templates →
  5. 05 · Some changes go in pairs

    Some edits must travel together

    Some kinds of changes affect more than one place at once. If you change a build command, a check, or a workflow that everyone else uses, you must also add an entry to the «what's new» feed in the same change, so others can see what changed and what they need to do differently. If you change a page template, the matching design example must change with it. Both of these are enforced automatically — the system won't let you commit one half on its own.
