Operating model

You are about to write into this portal. This page is the one entry point that answers «what are the rules, what is my next step, what gate will check me, what has changed in the process this month, and what edits drag other files along with them?» — without you having to remember any of it. Five rules below; three quick-paths after; the curated feed at the bottom.

Five rules

Every rule is enforceable. A deviation requires an ADR.

1 · Placement is driven by service or topic, not quadrant

where

A page about a service goes under services/<svc>/; a cross-cutting page goes under handbook/ with data-topic. The Diátaxis quadrant is metadata (data-page-type), not a folder. Use the routing decision tree if unsure.

Source — ADR-0032 D1.

2 · Every page declares six data-* attributes on <body>

what

The six are data-page-type, data-service, data-role, data-topic, data-lifecycle, data-updated. Without them the page silently falls out of catalog views. Allowed values and per-type examples live on page-type contract; tools/docs/validate_docs_html.py hard-enforces the contract after C7 lands.

Source — ADR-0032 D7.

3 · Before every commit — docs-fix then docs-check

before commit

Run make -C services/portal docs-fix && make -C services/portal docs-check. The drift gate fails on any uncommitted autogen delta. If you touched service pages, add make -C services/portal verify on top — same drift gate plus a11y. Pre-commit hooks alone do not catch autogen drift (catalog, search index, maintainers).

Why hooks aren't enough — quality gates map.

4 · Every new page starts from a template

how

Copy from handbook/sa/templates/. There are 16 templates — one per artefact (ADR, RFC, runbook, postmortem, how-to, tutorial, audit, bug-report, screen-spec, etc.). Structure is contract, not self-expression. Replacing prose in a template is the supported workflow; inventing new shapes is not.

Master five-step workflow — handbook/sa/authoring/add-a-page.

5 · Process changes leave a trail; coupled edits travel together

close the loop

A new Make target, pre-commit hook, CI workflow, SA template, or operations-class ADR must update whats-new-in-how-we-work in the same PR. A change to a SA template must travel with its UI-kit twin (pairs in tests/visual/pages.json) and a refreshed visual baseline. Both are enforced by pre-commit gates after C8 lands.

Coupling inventory — quality gates map § coupling column.

  • 2026-06-12 @ivan Operating model section ships. What: top-level operating-model/ with seven router pages (this kassa, routing tree, lifecycle diagram, gates map, page-type contract, what's-new feed, glossary). Why: portal scale (35 ADRs, 16 SA templates, 9 Make targets, 12 pre-commit hooks, 7 CI workflows) means placing a page and remembering what gates it now needs six pages of handbook to answer — no single entry point. What changes for you: the kassa is your front door; once C8 lands, any change to Make / workflow / pre-commit hook / SA template / operations-class ADR must update this feed in the same PR, and any change to a SA template must travel with its UI-kit twin.   • 2026-06-07 @ivan Portal staging via dual GH Pages (Proposed). What: source repo publishes prod from main; mirror repo <app>-staging publishes staging from a CI-populated branch. Why: the gap between «works on my laptop with python -m http.server» and «live on the prod URL» — Pagefind base path, asset paths, theme persistence — only manifests under a real GH-Pages domain. What changes for you: branch flow is now feature/* → PR → staging → PR → main; promotion is a git merge, not a deploy script; staging has its own URL, cache, and 404. See ADR 0034.
  • 2026-06-06 @ivan Service-catalog-first IA ratified (Variant C). What: services own runbooks / postmortems / specs; cross-cutting craft lives under handbook/; Diátaxis becomes data-page-type metadata rendered via catalog/by-quadrant/; role hubs aggregate via tags. Why: the previous IA (foundations/reference/) put SA-craft at the spine, which scaled poorly as services grew. What changes for you: Rule 1 and Rule 2 of this kassa are direct consequences — six data-* attributes on every <body>, choice rule «service or topic» before HTML. See ADR 0032.
  • 2026-06-06 @ivan Pagefind replaces the in-house BM25 indexer. What: ADR-0033 ratified, ADR-0027 superseded; legacy scripts/build_docs_search_index.py, scripts/check_search_index_subset.py, search-index.json, and search-index-public.json deleted; search.js rewritten on top of pagefind.search(). Why: 2.3 MB cold-payload → 92 kB gz; English-only stemming for free; single bundle splits internal/public via facets. What changes for you: nothing to maintain by hand; the index rebuilds via step 9 of docs-fix. See ADR 0033.
  • 2026-06-06 @ivan Portal shell token contract + foundations handbook canon ratified. What: ADR-0030 (one docs-shell.css as single source of shell geometry; fourteen CSS custom properties on body classes) and ADR-0031 (methodology pages are Diátaxis Explanation with one of seven lens chips per section and a «What to read next» closer) both moved Proposed → Accepted. Why: shell drift between internal/public portals; methodology pages needed a consistent shape. What changes for you: page-skeleton.html is the markup contract every page must satisfy; methodology pages declare a lens chip and a closer. See ADR 0030 and ADR 0031.
