Internal · Operating model · Reference

The six data-* attributes every page declares

Source of truth for Rule 2 of the operating-model hub. Every page on this portal carries six metadata attributes on <body>. The catalog views, the sidebar tags, the freshness audit, and (since 2026-06-13) the HTML lint all read these. This page lists allowed values, where each attribute surfaces, what enforces it, and ready-to-paste examples per page type.

Why six attributes

Per ADR-0032 the portal's IA does not nest on three independent axes — Diátaxis quadrant, owning service, audience role. Folders carry the service axis (D1); the other two travel as <body> metadata and surface as lens-views under catalog/by-quadrant/ and team/roles/<role>/. Two more attributes — topic and lifecycle — drive the topical lens and the freshness audit. One more — data-updated — feeds the recent-changes view. Six in total. Pick by the routing decision tree; this page is the allowed-values reference.

Authors who skip an attribute lose the page from the views that attribute drives. data-page-type missing ⇒ the page silently disappears from every catalog lens (tools/docs/build_catalog.py drops it). Since 2026-06-13 the silent drop is also a hard fail at the HTML lint — tools/docs/validate_docs_html.py asserts the three required attributes plus the closed vocabularies.

Each attribute in detail

data-page-type — Diátaxis lens

Vocabulary. Closed; one of six values.

ValueReader stateTypical shapeWhere it surfaces
tutoriallearning, beginner«Your first ADR», «Your first endpoint»catalog/by-quadrant/tutorial.html
how-todoing, knows the task«Add a page», «Release to staging», «Run a load test»catalog/by-quadrant/how-to.html
referencelooking upAPI operations, ADRs, runbooks, postmortems, templates, this contractcatalog/by-quadrant/reference.html
explanationunderstanding, wants «why»Diátaxis theory, topology, freshness, lifecycle-of-a-changecatalog/by-quadrant/explanation.html
landing-sectionarriving at a section hubSection index.html like operating-model home, governance homenot in by-quadrant view; visible in by-service
landingarriving at the portal rootOnly the top-level internal/index.html uses thisnot in by-quadrant view

Where used. build_catalog.py reads this for every page; TONE_BY_TYPE picks the spine-tile pill colour (tutorial=info, how-to=accent, reference=neutral, explanation=warn, landing=info). Missing value ⇒ the page is silently dropped from all catalog tables.

Pick by reader state. If two values both feel right, pick the one matching what the reader is doing when they land on the page. A runbook reads like a how-to but is consulted on incident, not learned — so reference.

data-service — service lens

Vocabulary. Closed; one of six slugs.

ValueOwnsWhere pages live
apithe HTTP API serviceservices/api/, services/portal/internal/services/api/
portalthe documentation portal itselfservices/portal/ (this page included)
datastoreSQLite + migrationsservices/portal/internal/services/datastore/
monitoringPrometheus + dashboardsservices/portal/internal/services/monitoring/
ui-kitUI Kit v2 components and tokensservices/portal/ui-kit/
nonecross-cutting craft, governance, onboarding, team — anything outside the service cataloghandbook/, governance/, onboarding/, team/

Where used. build_catalog.py reads this for catalog/by-service/. data-service="none" is rendered as «cross-service» in some views.

Default if unsure. none. Wrong service slug is worse than no slug — the page surfaces under a service that does not own it.

data-role — role aggregation

Vocabulary. Closed; space-separated subset of six tokens.

ValueFull role nameRole hub
sweSoftware Engineerteam/roles/swe/
saSolutions Architect / SWE-Architect blend (the canonical authoring role)team/roles/sa/
qaQuality Assuranceteam/roles/qa/
sreSite Reliability Engineerteam/roles/sre/
architectArchitect (the «think-first» discipline)team/roles/architect/
managerEngineering Manager / Tech Leadteam/roles/manager/

Canonical vocabulary. Per ADR-0032 D6 the token is swe, not dev / developer / engineer. The full table above is the closed list; nothing else passes the lint.

Multi-role. Space-separated: data-role="qa swe" means «both an engineer and a tester are first-class readers». Order does not matter.

Cross-role. Empty string: data-role="". The catalog renders this as «cross-role». Use when three or more roles open the page in a normal week — sprinkling all six tags is anti-pattern.

data-topic — topical lens

Vocabulary. Open; space-separated free tags. Unlike the other attributes, the value space is not fixed — write the tags that match the page.

Common tags in use today.

Where used. Today only the four curated topics (runbooks, postmortems, tests, on-call) render their own catalog/by-topic/<tag>.html. All tags flow into Pagefind search. Empty value (data-topic="") is legal for standalone artefacts (person profiles).

data-lifecycle — freshness state

Vocabulary. Closed; one of four states. Aligned with the ADR-0018 lifecycle vocabulary.

ValueMeaningVisibilityAllowed transitions
draftstructure is in place, content unfinishedvisible in sidebar; not promoted on role hubs→ review, → published
reviewcomplete, awaiting peer reviewsame as draft; reviewers find via PR→ published, → draft (rare)
publishedthe supported state; everyone reads thisfull visibility in catalog views and role hubs→ deprecated
deprecatedkept for archaeology; pointer to its successorgreyed in catalog; superseded badgeterminal — replace by a new page

Default for a new page in the same PR. Start at draft; flip to published in the commit that lands the real content.

data-updated — recency signal

Format. ISO-8601 date, YYYY-MM-DD. No time component; no timezone.

When to bump. On every meaningful edit — structural change, new section, new examples, status change, lifecycle transition. Typo fixes do not count. The freshness audit measures days since data-updated; bumping every typo would mask real staleness.

Where used. catalog/recent.html sorts on this value; the freshness audit ranks pages by «days since update»; the History footer in each page rows by date.

Validation — what enforces each attribute

Three enforcement layers per attribute: the HTML lint, the catalog builder, and (for lifecycle / updated) the freshness audit.

Attribute Validated by Before lint (historical) Today
data-page-type build_catalog.py reads it; validate_docs_html.py asserts it silent drop (page disappeared from catalog) hard fail — value must be in the closed list of six
data-service build_catalog.py reads it; validate_docs_html.py asserts it silent fallback to none if missing hard fail — value must be in the closed list of six
data-role build_catalog.py + validate_docs_html.py silent fallback to cross-role if missing hard fail if any token is outside the closed list of six (empty string still legal)
data-topic build_catalog.py only (open vocabulary) no enforcement — empty is legal no change; open vocabulary stays open
data-lifecycle build_catalog.py + validate_docs_html.py + freshness audit silent fallback to published if missing hard fail — value must be in the closed list of four
data-updated recent-changes view; freshness audit; validate_docs_html.py format check page fell off recent list if missing hard fail if format is not ISO-8601

C7 landed on 2026-06-13. tools/docs/validate_docs_html.py now asserts the three required attributes (data-page-type, data-service, data-lifecycle) drawn from the closed vocabularies in ADR-0032 D7, plus closed-list validation for data-role and ISO-8601 format for data-updated when those are present. The lint is wired into make docs-html-check (and through it into docs-check + verify) and runs in CI. This finishes ADR-0032 W6; the «silently falls out of the catalog» pattern is now a hard fail.

Migration — getting existing pages into compliance

How the migration to a hard lint went on 2026-06-13:

  1. Sweep filled defaults across internal/. Sixteen API-reference pages backfilled with data-service="api", data-role="", data-lifecycle="published", data-updated="2026-06-12"; sixteen team/ pages migrated from legacy page-types to canonical ones.
  2. Vocabulary widened pragmatically. The blog page-type was already in use via build_catalog.py's TONE_BY_TYPE map; the lint accepts it as a seventh allowed value pending a successor ADR that either ratifies it or migrates blog posts to explanation.
  3. Lint extended. tools/docs/validate_docs_html.py now asserts the schema on every commit; wired through docs-html-check → docs-check → CI.
  4. The contract is hard. New pages must pass the lint at docs-html-check; the operating-model section is already compliant.
