The six labels every page must declare

Every page on this portal must declare six things about itself: what kind of page it is, which service it belongs to, who it's for, what topic it covers, whether it's still live, and when it was last updated. These six labels are how the portal builds its indexes, its search, and its «recent changes» view. This page lists the allowed values for each one, with ready-to-paste examples.

Why six labels

The portal sorts pages along three independent axes: by service, by kind of page (tutorial, how-to, reference, explanation), and by who reads them. The folder carries the service axis; the other two travel as labels on the page itself, and the portal builds separate views from each label. Two more labels — topic and status — power the topic views and the staleness audit. One more — the update date — powers the «recent changes» view. Six in total. Use the routing questionnaire if you're unsure which values to pick; this page is the full list of allowed values.

Skipping a label costs the page its place in the views that label powers. A page with no «kind» label silently disappears from every index. An automated check now refuses to commit a page with any of the three required labels missing, or with a value outside the allowed list.

Each label in detail

Allowed values. One of six, listed below.

ValueReader stateTypical shapeWhere it surfaces

how-todoing, knows the task«Add a page», «Release to staging», «Run a load test»catalog/by-quadrant/how-to.html

Where it shows up. The «by quadrant» views read this label for every page, and the colour of each tile in the indexes comes from it. A missing or unknown value silently drops the page from every index.

Pick by what the reader is doing. If two values both feel right, pick the one that matches the reader's state when they land. A runbook reads like a how-to but is consulted under pressure, not learned — so it's a reference.

Default if unsure. Use «none». A wrong service is worse than no service — the page will appear under a service that doesn't own it.

data-role — who reads it

Allowed values. Up to six tokens, with spaces between several. Listed below.

ValueFull role nameRole hub
sweSoftware Engineerteam/roles/swe/
saSolutions Architect / SWE-Architect blend (the canonical authoring role)team/roles/sa/
qaQuality Assuranceteam/roles/qa/
sreSite Reliability Engineerteam/roles/sre/
architectArchitect (the «think-first» discipline)team/roles/architect/
managerEngineering Manager / Tech Leadteam/roles/manager/

The exact tokens matter. The token for software engineer is swe, not dev, developer, or engineer — the table above is the only allowed list, and an automated check rejects anything else.

Several roles. List them with spaces between, like qa swe for «both engineers and testers read this». Order doesn't matter.

Pages for everyone. Leave the value blank. The portal will mark the page «cross-role». Use this when three or more roles open the page in a normal week — sprinkling all six tags is the wrong fix.

data-lifecycle — status

Allowed values. One of four states.

ValueMeaningVisibilityAllowed transitions
draftstructure is in place, content unfinishedvisible in sidebar; not promoted on role hubs→ review, → published
reviewcomplete, awaiting peer reviewsame as draft; reviewers find via PR→ published, → draft (rare)
publishedthe supported state; everyone reads thisfull visibility in catalog views and role hubs→ deprecated
deprecatedkept for archaeology; pointer to its successorgreyed in catalog; superseded badgeterminal — replace by a new page

Default for a new page. Start at draft, then flip to published in the commit that lands the real content.

data-updated — last updated

Format. ISO date in YYYY-MM-DD form — no time, no timezone. For example: 2026-06-13.

When to bump. On every meaningful edit — a structural change, a new section, new examples, a status change. Don't bump for typo fixes: the staleness audit reads this date, and bumping for typos would mask real staleness.

Where it shows up. The «recent changes» view sorts pages by this date. The staleness audit ranks pages by «days since update». The History block at the bottom of each page lists its own entries by date.

What enforces each label

Each label is checked at up to three places: the HTML check that runs on every commit, the index builder that runs before each push, and (for status and last-updated) the staleness audit.

Attribute Validated by Before lint (historical) Today
data-page-type build_catalog.py reads it; validate_docs_html.py asserts it silent drop (page disappeared from catalog) hard fail — value must be in the closed list of six
data-service build_catalog.py reads it; validate_docs_html.py asserts it silent fallback to none if missing hard fail — value must be in the closed list of six
data-role build_catalog.py + validate_docs_html.py silent fallback to cross-role if missing hard fail if any token is outside the closed list of six (empty string still legal)
data-topic build_catalog.py only (open vocabulary) no enforcement — empty is legal no change; open vocabulary stays open
data-lifecycle build_catalog.py + validate_docs_html.py + freshness audit silent fallback to published if missing hard fail — value must be in the closed list of four
data-updated recent-changes view; freshness audit; validate_docs_html.py format check page fell off recent list if missing hard fail if format is not ISO-8601

The HTML check became a hard fail on 2026-06-13. From that day on, a page with a missing or unknown value for any of the three required labels — data-page-type, data-service, data-lifecycle — won't commit. The check also enforces the allowed list of role tokens (when one is set) and the date format (when one is set). Before this change, a missing label silently dropped the page from the indexes; now you get an explicit error and an explanation.

Background — when this became a hard check

The HTML check became a hard fail on 2026-06-13. Older pages were brought up to spec at the same time — about thirty pages got the three required labels backfilled in one sweep. From that day on, a new page that doesn't follow the contract above won't pass commit. Existing pages are already compliant; you just need to follow the contract when you create or edit one.

One small exception: the blog kind of page is also accepted today, even though it's not in the main list of six. Blog posts may move to explanation in a future cleanup; for now both work.
