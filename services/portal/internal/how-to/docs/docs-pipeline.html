Docs How-to

Documentation pipeline (docs-as-code)

What make docs-fix generates, which paths are machine-owned, and how make docs-check catches drift. Policy: ADR 0001.

make docs-fix stages

Each stage runs in order. All steps are idempotent — running twice produces the same result.

  1. UML rendering — scripts/regenerate_docs.py sends services/portal/internal/uml/**/*.puml to Kroki and writes SVGs under services/portal/internal/uml/rendered/. Requires network. Commit the SVGs after running.
  2. Marker sync — scripts/sync_docs.py overwrites marker blocks in: README.md, services/portal/internal/analysis/system-design.html, services/portal/internal/handbook/howto/0003-make-commands-inventory.html, and services/portal/internal/api/errors.html. Source data: Makefile targets and FastAPI routes. Prose outside markers is never touched. Environment variables are documented in env/example only — not injected into README.
  3. Markdown → HTML (optional) — scripts/render_docs_html.py rewrites the .html file next to any .md in services/portal/internal/governance/adr/, services/portal/internal/handbook/developer/, or services/portal/internal/sre/runbooks/. Only runs when companion .md files exist.
  4. HTML normalization — scripts/format_docs_html.py normalizes all services/portal/**/*.html except services/portal/internal/governance/backlog/ and services/portal/pdoc/.
  5. Python API reference — pdoc runs on the app package and writes HTML under services/portal/pdoc/. Commit this tree so GitHub Pages serves pdoc/index.html alongside the rest of services/portal/.
  6. Search index — scripts/build_docs_search_index.py writes services/frontend/portal/assets/search-index.json for client-side search.

make docs-check (CI gate)

docs-check saves git diff HEAD, runs docs-fix, then compares the two diffs. If they differ, docs-fix changed something — commit the result before pushing.

Files that docs-fix never touches are unaffected. If generated output already matches HEAD, the two diffs are identical and the check passes.

Python API reference (pdoc)

HTTP contracts are covered by OpenAPI and developer guides. Internal Python modules — packages, classes, functions — are documented from docstrings via pdoc, similar in spirit to Rust's cargo doc.

Use Google-style docstrings; template and checklist: ADR 0016. Run make install then make docs-fix to refresh. Output goes to services/portal/pdoc/; commit it with other doc updates.

In-page "On this page" sidebar

Pages that load docs-nav.js automatically render a sticky outline beside the article — no HTML mount is required. The script builds anchor links from all h2 / h3 headings (generating an id where missing). Headings inside figure.sys-diagram and p.lead are excluded.

To opt a single page out of the auto-TOC, mark its <main> with data-inpage-toc="off".

Do not write a manual "Contents" block on pages that use this script — it will duplicate the generated sidebar. If the sidebar is missing, see Runbook: In-page sidebar missing or empty.

Versioning and optional publishing

Default: docs live in git alongside code. After a feature merges to main, services/portal/ reflects the current state. Release history uses ADR 0013 and git tags.

GitHub Pages from the services/portal/ folder: Settings → Pages → Deploy from branch → /docs. Add services/portal/.nojekyll so GitHub skips Jekyll. Project-site URLs use /{repo}/… without a /docs/ segment — the top nav script handles this layout automatically.

Versioned prose site (optional): add MkDocs or Docusaurus, publish from CI on version tags, keep docs-fix for UML and marker sync until you migrate those steps. Docs-only PRs may merge before implementation lands; follow-up PRs carry code.
