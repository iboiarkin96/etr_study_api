Postmortems How-to

Write a postmortem

After an incident is over, capture what happened, why, and what changes so it doesn't recur — blamelessly, with named action items. The reusable scaffold is the postmortem template (Reference); the culture this procedure depends on lives in the incident-response Explanation; this page is the procedure.

Before you start

  • The incident is over — alarms cleared, recovery confirmed, no live mitigation in flight. Write postmortems after, not during.
  • You're inside the required cases: every SEV-1 and SEV-2 (mandatory) and SEV-3 if any of the following holds — same alert fired ≥ 2× in the past month, a workaround now exists that no one has scheduled to remove, the incident exposed a class of bug not previously tracked, or a user complained publicly. Severity matrix in incident response.
  • You know who's playing which role — see the role bar below:
    Author
    Writes the document. Usually the incident commander or the engineer who fixed it.
    Reviewer
    Reads the draft and pushes back on hand-waving («the system was unstable» → «the connection pool ran out at 14:32»). For a single-person team the author waits 24 hours and re-reads with fresh eyes.
    Action-item owner
    Each action item has one named owner and a target date. No owner = the action will not happen.
  • You've started within the 5-working-day target window after the incident closed — memory fades fast.

Steps

  1. Copy the template. Clone foundations/reference/sa/templates/postmortem.html into services/portal/internal/how-to/incidents/postmortems/<YYYY-MM-DD>-<incident-slug>.html (date = the day the incident started; slug = short kebab-case, e.g. connection-pool-saturation).
  2. Write blamelessly. Describe actions without judging the person — same facts, opposite effect on the system:
    • Blame: «Ivan deployed without checking the migration.»
    • Blameless: «The deploy was shipped before the migration finished. There was no automatic check in CI for the order.»
    The first sentence makes future engineers hide mistakes; the second points at a missing check we can add.
  3. Fill in the timeline. Stick to facts and exact timestamps (UTC). Drop adjectives like «huge» or «scary». If you can't pin a time, write «<~time>» rather than glossing — uncertainty is data.
  4. Build the root-cause section as contributing factors, not «the» cause. Real incidents almost always have several. List each factor as a separate bullet and link each one to the evidence (a log excerpt, a graph, a commit hash).
  5. Write action items per the SMART quality bar below. «Improve monitoring» is not an action item; «Add alert for connection-pool saturation, owner: @ivan, target 2026-05-20, ticket BL-XXX» is.
  6. Link from the changed surfaces. Add a one-line entry to the History footer of every page whose content changed because of this incident (runbook, ADR, code-reference). Cross-link the postmortem URL from each.
  7. Get a review. Send the draft to the Reviewer; for a solo team, sleep on it 24 h and re-read. Apply changes; ratify.

Verify

  • The file exists at postmortems/<YYYY-MM-DD>-<slug>.html and renders with the standard internal-portal shell.
  • Every required section in the template is filled in — no «TODO» or «TK» placeholders survive into the published version.
  • Every action item passes the SMART bar (next section) — Specific, Owned, Dated, Tracked, Sized.
  • Timeline timestamps are UTC and monotonic (no entry references a time earlier than the previous row).
  • The Reviewer signed off (or, for solo teams, the author re-reviewed after 24 h with no remaining issues).
  • The postmortem index below lists the new file; every surface whose content changed because of the incident links to the postmortem from its History footer.
05

Action-item quality bar (SMART)

The five tests every action item must pass to land in the document. Items that fail one of these are the ones that quietly stall — flag them before the postmortem is ratified, not at the next on-call review.

  • Specific. Names a file, an alert, or a runbook. No vague verbs («improve», «harden», «consider»).
  • Owned. One name. Not a team. Not «whoever is on-call next».
  • Dated. A real date, not «soon» / «next sprint» / «ASAP».
  • Tracked. Linked to a backlog item or a ticket — not just prose inside the postmortem.
  • Sized. Less than two weeks of effort each. Bigger work splits into several action items.

Review open action items at the start of each on-call shift handoff. Items older than 60 days either get done, get reassigned, or get explicitly closed with a reason.
