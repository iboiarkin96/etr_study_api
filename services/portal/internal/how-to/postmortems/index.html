Postmortems

What a postmortem is

A postmortem is a short document written after an incident is over. It explains, honestly and without blame, what happened, why, and what we will do so it does not happen the same way next time.

Postmortems are not court records and they are not punishment. They are the main way a team turns one bad day into a permanently better system. The model we follow is the SRE Book chapter on postmortem culture.

Blameless — what it really means

"Blameless" does not mean "ignore who did what". It means we describe actions without judging the person:

  • Blame: "Ivan deployed without checking the migration."
  • Blameless: "The deploy was shipped before the migration finished. There was no automatic check in CI for the order."

The first sentence makes future engineers hide mistakes. The second sentence points at a missing check we can add. Same facts, different effect on the system.

When we write one

A postmortem is required for every SEV-1 and SEV-2 incident (see severity matrix in incident response). It is recommended for SEV-3, especially for any of:

  • The same alert fired more than twice in a month.
  • A workaround is now in place that nobody has time to remove.
  • The incident exposed a class of bug we had not thought about.
  • A user complained publicly.

Target turnaround: 5 working days after the incident closes. Sooner is better; memory fades quickly.

Roles

Author
Writes the document. Usually the incident commander or the engineer who fixed it.
Reviewer
Reads the draft and pushes back on hand-waving ("the system was unstable" → "the connection pool ran out at 14:32"). For a single-person team the author waits 24 hours and re-reads with fresh eyes.
Action item owner
Each action item has one named owner and a target date. No owner = the action will not happen.

How to write one

  1. Copy _template.html into a new file named YYYY-MM-DD-incident-slug.html (date = the day the incident started, slug = short, kebab-case).
  2. Fill each section. Stick to facts and timestamps; avoid adjectives like "huge" or "scary".
  3. For "Root cause", list contributing factors, not one single cause. Real incidents almost always have several.
  4. For "Action items", make each one a single ticket with an owner and a date. "Improve monitoring" is not an action item; "Add alert for connection pool saturation, owner: Ivan, by 2026-05-20" is.
  5. Add the new file to the index below and link it from the page-history of the page that changed because of the incident.

Action items — what makes them stick

  • Specific. Names a file, an alert, or a runbook. No vague verbs.
  • Owned. One name. Not a team.
  • Dated. A real date, not "soon".
  • Tracked. Linked to a backlog item or a ticket — not just text in the postmortem.
  • Sized. Less than two weeks of effort each. Bigger work splits into several action items.

Review the open action items at the start of each on-call shift handoff. Items older than 60 days either get done, get reassigned, or get explicitly closed with a reason.
