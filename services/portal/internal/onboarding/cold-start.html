Onboarding Cold start Per-persona reading order

Cold-start paths — get productive on day one

If you joined this codebase today and have one day to be productive in a specific role, here is the curated reading order. The portal has 560+ pages; this page is the answer to "what do I actually read first?". Seven paths — one universal (Newcomer, ~2 hours) and six per-role — each between 6 and 15 pages. Read them in order: every step was chosen so the next page makes sense because the previous one set the context.

How to use this page

Pick the role that matches what you will do this week, not your job title. A senior backend engineer asked to triage an SLO breach reads the SRE path, not the SWE one. The paths are independent — you do not need to read them in order, and you do not need to read one before starting another. Most are sized for a single working day; Newcomer is the 2-hour entry point if you do not yet know which role-path applies, and Architect is the 1–2 day deep cold-start for staff+ engineers who need to review the whole posture.

If you are not in any of these roles, read the architectural narrative first — it sets the context every other path assumes.

Newcomer · ~2 hours

You will be productive when: you know what this portal is, where pages live, which voice we use, and you can post a small docs-only PR (fix a typo, broken link, stale snippet) without asking. Generic first-day reading for any role — choose this path if you do not yet know which role-path applies.

  1. Onboarding hub — single page that names every onboarding step (repo access, dev env, first PR). (~10 min)
  2. Glossary — «conspectus», «ETR», «attempt» mean specific things here; don't guess from English. (~10 min)
  3. Reader personas — which one are you. After this page you know which of the role-specific paths below to take next. (~12 min)
  4. SA reference hub — single landing for how this portal is authored. Tells you which sub-page answers which "how do I…" question. (~10 min)
  5. Diátaxis on this portal — every page belongs to one of four quadrants. Picking the wrong one is the #1 newcomer mistake. (~15 min)
  6. Style guide — voice, sentence rules, page skeleton. A docs PR with the wrong voice gets bounced. (~15 min)
  7. How-to · Add a page — the procedure that lands a new page in under an hour. (~15 min)
  8. Principles — the eight rules every page in this repo obeys. Skim now, return when reviewing. (~15 min)

Senior backend engineer · ~1 day

You will be productive when: you can confidently open a feature branch that adds a new endpoint, get it through CI, and respond to review comments about the contract, idempotency, observability, and tests without re-reading the rules.

  1. Architectural narrative — what this system is, what shape it has, what was deliberately not built. Sets every other piece of context. (~10 min)
  2. Where business logic lives — the four-layer architecture and why the direction of dependency is strict. After this page you can predict where any code change belongs. (~15 min)
  3. Schemas as the public contract — the additive-only rule and why renames are removals. Pairs with the OpenAPI baseline check that will block your first PR if you get it wrong. (~12 min)
  4. Reference · Idempotency conventions — every write you touch needs this. The 24-hour TTL, the body-hash key, the 409 on hash mismatch. (~10 min)
  5. POST /api/v1/user spec — the exemplar 18-section operation page. Future specs you author will copy this shape. (~15 min)
  6. How-to · Add a POST contract — the recipe that ties the four documents above into a single PR. By the end of this page you have a mental "what do I write where?" map. (~20 min)
  7. Test pyramid — the layered test discipline; tells you which level your new test belongs at. (~10 min)
  8. ADR 0007 — OpenAPI governance — the policy enforced by the contract test that will fail your first naive rename PR. (~10 min)
  9. Postmortem · OpenAPI baseline drift — read this last. It is what happens when you skip step 3 above. (~8 min)

Senior Systems Analyst · ~1 day

You will be productive when: you can author a new API specification page that passes make docs-spec-check, decide where a new document belongs in the Diátaxis IA, and review a junior author's PR with concrete suggestions instead of taste.

  1. Architectural narrative — sets context for what the system is. (~10 min)
  2. Diátaxis on this portal — the four quadrants applied to this repo. After this page you can answer "where does this content go?". (~15 min)
  3. API spec template (Reference) — the 21-section form; this is what every per-operation spec page is shaped against. (~15 min)
  4. How-to · Add an API spec page — the procedure that applies the template. (~10 min)
  5. _shared/ index — cross-cutting concerns (auth, idempotency, error envelope, versioning, observability, OpenAPI authoring) live here once and are linked from every spec. (~12 min)
  6. Why a Definition of Done — the artefact-agnostic DoD model; applies to specs, ADRs, runbooks, postmortems equally. (~10 min)
  7. Domain-model reference — bounded contexts, language, events, invariants; the worked YAML example shows what a complete one-file domain projection looks like. (~15 min)
  8. Deprecation policy — the five-stage flow that prevents soft-deprecation-forever. Read with the architectural-narrative's "what we deliberately did not build" section open in the next tab. (~10 min)

Architect · ~1–2 days

You will be productive when: you can run an architectural review of the whole system without asking a single colleague — locate any feature on the bounded-context map, name the load-bearing ADRs, defend the SLO numbers, and read one real postmortem to verify the team actually learns. This is the deepest path; staff+ cold-start due-diligence.

  1. Architectural narrative — the arc: initial gamble, hit constraints, current shape. After this page you can answer "why is the system like this and not otherwise". (~30 min)
  2. Reader personas — who else reads the portal; tells you which writing style each section is calibrated for. (~10 min)
  3. Diátaxis on this portal — the writing framework. Decoding the structure is decoding the portal. (~15 min)
  4. IA contract — flat disk, Diátaxis sidebar. After this page you can navigate by URL alone. (~15 min)
  5. Domain model — six bounded contexts. After this page you can locate any feature on the context map. (~25 min)
  6. Spine matrix — what every service must document and where; spot a missing artefact at a glance. (~20 min)
  7. Workflows — three Tier-1 cross-service flows; understand the seams where one bad assumption breaks everything. (~25 min)
  8. ADR 0008 — Make taxonomy — make help is the canonical entrypoint to everything we run; read this and you read the build system. (~15 min)
  9. ADR 0011 — SLO / SLA / error budget — defines what "working" means here. Numbers, not adjectives. (~20 min)
  10. ADR 0020 — PlantUML / C4 conventions — all diagrams in this portal are PlantUML; Mermaid is forbidden. After this page you can predict where any diagram's source lives. (~15 min)
  11. ADR 0026 — Internal portal = source of truth — the provenance rule; the internal portal wins on any conflict with the public site. (~15 min)
  12. Explanation · SLOs — SLO targets with the multi-window burn-rate table. Real numbers, not aspirations. (~20 min)
  13. Explanation · Observability — what we measure, why those signals and not others. Predict what a new feature must emit. (~20 min)
  14. Postmortem · SQLite write-lock burst (2026-04-02) — one real incident end-to-end; the only way to know if a team learns is to read one of their postmortems in full. (~15 min)
  15. Traceability matrix — requirements → release; empty cells are the backlog. Spot any orphan requirement in 30 seconds. (~15 min)

SRE / on-call · ~1 day

You will be productive when: you can take a page at 3 a.m., open the right runbook in under 30 seconds, decide severity in under a minute, drive triage with named roles, and file a postmortem within five working days that passes the SMART-action-item bar.

  1. Incident response — the four-moves model (detect, mitigate, recover, learn), severity rationale, roles. The frame every runbook and postmortem assumes. (~15 min)
  2. SLOs — SLI / SLO / SLA nesting, error-budget math, multi-window burn-rate thresholds. Tells you when to declare and when to stand down. (~15 min)
  3. Observability — metrics, logs, traces, what each is for, cardinality discipline. The vocabulary every runbook reuses. (~12 min)
  4. Runbook index — skim the 10 numbered runbooks. Memorise the URL pattern. (~8 min)
  5. Runbook 0009 — Error-budget exhaustion — the canonical SRE surface; SEV-2 default, tighter contract. Read this one in full. (~10 min)
  6. Runbook 0006 — API security failing — the other tighter-contract runbook. Knowing the security family's faster SLA stops you from defaulting to the generic 30-minute timeout. (~8 min)
  7. How-to · Write a postmortem — the procedure and the SMART action-item bar; followed by the three postmortems on file as exemplars. (~20 min)
  8. On-call — rotation cadence, hand-off rituals, the trade-offs. Tells you what to expect from the rest of the team. (~12 min)

QA engineer · ~1 day

You will be productive when: you can take a new endpoint specification and produce a complete test matrix (unit / integration / contract / E2E), identify which checklist applies, and authoritatively pass or fail an endpoint at the acceptance gate.

  1. QA process & SDLC integration — where testing sits in the lifecycle, entry/exit criteria, hand-offs. (~12 min)
  2. Test pyramid — distribution of unit / integration / E2E and the cost-vs-signal tradeoff. (~10 min)
  3. Test strategy — the risk-surface and coverage-stops framing. (~10 min)
  4. API endpoint acceptance checklist — the most-used checklist; you will run it at least weekly. (~10 min)
  5. QA glossary — 38 terms, A–Z. Bookmark and refer back. (~5 min skim)
  6. Defect lifecycle — the 8-state machine; how a bug travels from filed to closed. (~10 min)
  7. Bug severity & priority — the matrix you will reach for on every triage call. (~10 min)
  8. How-to · API endpoint testing — the procedure that ties the checklist and the test pyramid into a single runnable workflow. (~15 min)

Manager / product owner · ~half day

You will be productive when: you can walk the team's DoR/DoD, name what a release candidate must have to ship, decide severity on an incoming bug, and skim a postmortem and tell whether the action items will actually move.

  1. Architectural narrative — what the system is, where it earns its keep, what was deliberately not built. The same context everyone reads. (~10 min)
  2. ETR methodology — the learning loop the product is built around. After this page the product roadmap stops feeling arbitrary. (~12 min)
  3. Why a Definition of Done — the artefact-agnostic DoD. Read with your team's actual DoD next to it. (~10 min)
  4. Bug severity & priority — the matrix; you will be asked to call SEV on incoming reports. (~10 min)
  5. SLOs — error-budget model; what "we're out of budget" means in product terms. (~12 min)
  6. How-to · Write a postmortem — read the procedure, then skim the SMART action-item bar. You will not be writing postmortems, but you will be reviewing them. (~12 min)
  7. Deprecation policy — the five-stage flow with sunset windows; how the team decides when a feature gets retired. (~10 min)

Why curated paths instead of a search bar

The portal already has search. Search answers "where is X?" — it does not answer "what should I read first?". The cost of being wrong about reading order is small for any one page, but the cost is cumulative: a senior who reads observability before incident-response builds the wrong mental model of why the SLO matters, and only realises during the first SEV-2. Curated paths front-load that small cost so the day-one productivity gradient is steeper.

These paths are deliberately short. Adding more pages makes the path more "complete" and less likely to be finished. A path that takes a week is read by nobody; a path that takes a day is read by everybody.

These paths get pruned, not extended. If a new page becomes load-bearing for a persona, it replaces a less load-bearing one on the same path. Keep the budget.
