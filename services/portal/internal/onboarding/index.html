Three role-agnostic checkpoints for new joiners. Day 1 gets the portal running locally. Week 1 ships your first page through the docs-as-code gates. Month 1 lands your first ADR or RFC. Each step deep-links into the role-specific walkthroughs below.

Day 1 · 30 min

Get the portal running locally

By the end of Day 1 you have the repo cloned, the portal served on localhost:3000, and you know which role hub is yours.

  1. 01
    Clone the repo and install make

    Repo is monorepo-style. Run make help to see the docs-as-code targets.

    Run python3 -m http.server 3000 -d services/portal/internal. Open localhost:3000.

Week 1 · first PR

Ship your first page

Pick a real page you'd want to exist (a runbook, an explanation, a template fill-in), follow the five-step authoring workflow, and ship it through the docs-as-code gates.

  1. 01
    Pick the Diátaxis type

    Tutorial · How-to · Explanation · Reference — one quadrant only. Read principles if unsure.

    Principles →
  2. 02
    Copy a template

    Never start blank. Templates cover all common page types.

    Templates →
  3. 03
    Wire the sidebar

    Add an entry to nav-tree-internal.json. Mind acronym casing (API / AsyncAPI / SRE / gRPC).

    Update sidebar →
  4. 04
    Run the gates

    make docs-fix formats; make docs-check runs link, h1, acronym, a11y checks.

    Docs pipeline →
  5. 05
    Open the PR

    Link the relevant ADR / RFC / audit in the description. History block at the bottom of the file documents the change.

    Add a page →
Month 1 · first decision

Author an ADR or RFC

By Month 1 you've seen enough of the system to propose a decision. Draft an ADR (small, single decision) or an RFC (larger proposal needing review). Both go through the same lifecycle: Draft → Review → Accepted → Active → Superseded.

  1. 01
    Pick the right vehicle

    ADR for an accepted decision (one page). RFC for a proposal that needs review.

    Governance →
  2. 02
    Draft from the template

    Use the canonical ADR / RFC scaffold. Context · Decision · Consequences · Alternatives.

    Templates →
  3. 03
    Run review

    Open the PR with a status-timeline at Proposed. Move to Accepted only after consensus.

    Add an ADR →
  4. 04
    Cite it

    The point of an ADR is to be cited. Link it from service docs, runbooks, and follow-up RFCs.

    ADR index →

Senior cold-start paths — skip the spine if you already know how to onboard

The Day-1/Week-1/Month-1 spine above is for first-time engineers. If you joined as a senior and have one day to be productive, the spine is too slow — you don't need to be taught what a PR is. The cold-start paths are five curated reading sequences (senior backend, senior SA, SRE/on-call, QA, manager), each 6–9 pages, ~1 working day, with an explicit "you are productive when…" criterion.
