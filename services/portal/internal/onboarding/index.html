Map of the portal — where things live

Per ADR 0032 the portal follows a service-catalog-first IA (Backstage-style). Six top-level sections, each with a single load-bearing purpose. Read this once; the choice rule below tells you where to put anything new.

Services catalog backbone

One folder per deployable artefact (api, portal, datastore, monitoring, ui-kit). Each has its own overview · architecture · dependencies · API reference · on-call · runbooks · postmortems. Backstage catalog-info.yaml at the root of each service.

Handbook cross-service craft

Anything that applies across services. SA (24 canon pages + templates + authoring) · SWE / QA / SRE (cross-service practice) · data · security · UML · manager (topical reference) · personas · glossary.

Team people + role hubs

People — person pages, blog. Roles — six hubs (Architect · SWE · SA · QA · SRE · Manager), each with a radar, 25 practices, and a 60-day onboarding.

Governance decisions

ADRs · RFCs · Audits · Backlog · Traceability. Decisions, proposals, and the live status grid. Off-limits to routine edits — touched only by the decision flow.

Catalog lens views

Generated indexes from page metadata — by quadrant (Tutorial / How-to / Reference / Explanation), by service, recently updated. Use it to find anything you can't remember the path of.

Onboarding this page

60-day journeys per role, the Day-1 spine, the cold-start path, and the glossary. The first stop when you join.

Choice rule — where does a new page go?

  1. Is it tied to one deployable service? → services/<svc>/{runbooks,postmortems,how-to,reference,...}/.
  2. Is it a template, page skeleton, or authoring procedure? → handbook/sa/{templates,authoring}/.
  3. Is it cross-service craft (SRE practice, QA strategy, data table, threat model)? → handbook/{sre,qa,swe,security,data,uml,manager}/.
  4. Is it role onboarding (sequenced tutorial)? → team/roles/<role>/onboarding.html (or practices/ for a single craft pattern).
  5. Is it a decision, contract, or proposal? → governance/{adr,rfc,audits,backlog}/.
  6. Is it a lens or aggregator view? → catalog/by-*/ — autogenerated from metadata, never hand-written.

Decision time per page: ~10 seconds. Full rules and rationale: ADR 0032 · Service-catalog-first IA.

Day-1 spine · universal

Four steps every new joiner does on day one, regardless of role. About thirty minutes end to end. Once it is done, switch to your role's journey above.

  1. Clone the repo and install make. Monorepo-style layout. Run make help to see the docs-as-code targets you will use this week. Local dev → (~5 min)
  2. Serve the portal locally. Run python3 -m http.server 3000 -d services/portal/internal and open localhost:3000. Portal home → (~3 min)
  3. Open your role hub. Pick one of the six journeys above; the hub gives you the radar (AS-IS vs TO-BE) and a 25-practice map. This is your map. (~10 min)
  4. Skim the Diátaxis quadrant table. One read; refer back when authoring anything — every role contributes docs eventually. SA principles → (~10 min)

Senior? Skip the spine.

The spine above is for first-time engineers. If you joined as a senior and have one day to be productive, you don't need to be taught what a PR is. The cold-start paths are seven curated reading sequences (Newcomer, Senior backend, Senior SA, SRE/on-call, QA, Architect, Manager) — each 6–15 pages, ~1 working day, with an explicit «you are productive when…» bar.
