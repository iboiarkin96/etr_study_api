Non-Functional Requirements — make the ilities explicit

Capture quality attributes as measurable acceptance criteria so "fast enough" and "secure enough" mean the same thing to every engineer.

TL;DR

Non-Functional Requirements (NFRs) are the measurable quality attributes of a system: performance (p99 latency ≤ 200 ms), availability (99.9%), scalability (10k RPM), security (no stored plaintext passwords), and maintainability (MTTR ≤ 1 hour). ISO 25010 codifies the standard categories. NFRs are captured per service, validated where possible in CI, and reviewed in architecture reviews. Without explicit NFRs, "good enough" means whatever the most recent engineer thought it meant.

What it is

Non-Functional Requirements are the measurable quality attributes a system must satisfy. They are distinct from functional requirements (what the system does) — NFRs describe how well the system must do it. ISO/IEC 25010 (the standard quality model for software) defines eight characteristics; in practice most engineering teams use a reduced set of five:

  • Performance — latency (p50, p95, p99), throughput (RPM/RPS), and resource utilisation under load.
  • Availability — uptime targets (99.9% / 99.99%), recovery time objectives (RTO), recovery point objectives (RPO).
  • Security — authentication strength, data-at-rest and in-transit encryption, threat model coverage, OWASP Top 10 compliance.
  • Scalability — ability to handle 10× current load, horizontal scaling boundaries, statelessness requirements.
  • Maintainability — MTTR (mean time to recovery), test coverage thresholds, runbook coverage, deployment frequency.

Each NFR is captured per service in a YAML or Markdown file alongside the code, with a measurable target and a verification mechanism. Where automation is possible (latency assertions, security scans), NFRs run as fitness functions in CI. Where automation is not possible (threat model coverage), they are reviewed manually during architecture review.

Why we need it

Without explicit NFRs, "fast enough", "secure enough", and "scalable enough" mean different things to every engineer. The team ships a service that works at 100 requests per minute and discovers at 10× load that it falls over — because no one had specified the 10× target. The team builds an authentication system that meets the engineer's intuition for "secure" and discovers in a SOC 2 audit that the standard required something specific.

NFRs turn implicit expectations into testable acceptance criteria. They make trade-offs explicit at design time rather than discoverable in production. An NFR document also creates a contract with stakeholders — when an executive asks "can we handle Black Friday traffic?", the answer is a referenced NFR with a verification record, not a guess.

How others do it

arc42 quality requirements framework

arc42 is an open-source architecture documentation template, maintained by Gernot Starke and Peter Hruschka since 2003. Its quality requirements model uses the "quality tree" — a hierarchical decomposition of quality characteristics into scenarios with stimulus, environment, response, and response measure. Used by thousands of teams globally and aligned with ISO 25010. The format encourages teams to write quality scenarios rather than vague targets — e.g. "If the order rate exceeds 1000 RPM, the order service must continue to accept orders with degraded notifications."

What we take: Capture NFRs as scenarios, not numbers. "p99 ≤ 200ms" becomes "Under 5000 RPM sustained load, 99% of GET /orders requests return within 200ms." Scenarios make the test conditions explicit.

Source: arc42 quality requirements

ISO/IEC 25010

ISO 25010 is the international standard for software product quality. It defines 8 characteristics: Functional Suitability, Performance Efficiency, Compatibility, Usability, Reliability, Security, Maintainability, Portability. Each is decomposed into sub-characteristics. The standard is the canonical reference for compliance frameworks (SOC 2, ISO 27001) and is cited by many enterprise procurement contracts.

What we take: Use the ISO 25010 vocabulary in NFR docs even if you do not use the full standard. Auditors and enterprise customers recognise the terms and they map cleanly to compliance frameworks.

Source: ISO 25010 reference

Google SRE

Google's SRE practice operationalises performance and availability NFRs through SLOs (Service Level Objectives) — measurable targets backed by error budgets. The SRE Book documents the approach: every customer-facing service has an SLO, and engineering work is prioritised based on error budget consumption. Google has trained the industry to think of "availability" as a quantitative target rather than a qualitative aspiration.

What we take: Convert availability NFRs into SLOs from day one. An SLO with an error budget creates accountability that an NFR alone does not. See Practice 17 on SLO/SLA design.

Source: Google SRE — SLOs chapter

Anatomy

Per-Service NFR File

A YAML or Markdown file stored next to the service code (docs/nfrs.yaml). Captures the agreed NFRs for that service with categories, targets, and verification mechanisms. Updated in the same PR as architectural changes that affect them.

Measurable Target

Each NFR has a quantitative target. "Fast" is not measurable; "p99 latency ≤ 200ms at 5k RPM sustained" is. Targets without a load context are incomplete — they invite gaming ("we're at 200ms at 100 RPM").

Verification Mechanism

For each NFR, the method to verify it. Categories: automated (load test in CI, security scan), audited (manual review with named owner), or operational (production metric continuously monitored). NFRs without verification are aspirations, not requirements.

Architecture Review Gate

The NFR set is reviewed in every architecture review. The checklist asks: "Have NFRs been defined for this service?" and "Have changes affecting NFRs been re-validated?" A missing NFR document is a review blocker.

Common pitfalls

Pitfall Why it happens What to do instead
NFRs that are aspirational, not measurable The team writes "the system should be fast and reliable" because writing measurable targets requires committing to specific numbers, which feels risky. Vague NFRs are uncomfortable for nobody and useful to nobody. Force quantitative targets with load conditions. "p99 latency ≤ 200ms at 5000 RPM sustained" is uncomfortable to commit to but useful. Coach engineers to start with conservative targets and refine over time.
NFRs defined once and never re-validated The NFR document is created at service launch and never updated. Two years later the system has 10× the traffic and the original NFRs are obsolete or unmet — but no one checks. Re-validate NFRs annually or after any significant architecture change. The architecture retrospective is a natural forcing function. Stale NFRs are a higher risk than missing NFRs.
Too many NFRs (paralysis) The team copies ISO 25010 wholesale and ends up with 40 NFRs per service. None of them are verified. The document becomes a checkbox exercise that nobody reads. Limit to 5–8 NFRs per service. Cover the categories that matter most to this service. A backend transactional service emphasises performance and availability; a batch pipeline emphasises throughput and resilience.
