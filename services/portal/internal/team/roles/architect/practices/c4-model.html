C4 Model Diagrams — architecture as code

Communicate architecture at four levels of zoom so every audience gets the right abstraction without a whiteboard session.

TL;DR

The C4 model, created by Simon Brown around 2010, defines four zoom levels for software architecture diagrams: Context (the system in its environment), Container (what the system is made of), Component (internals of a container), and Code (class-level, rarely needed). Diagrams are stored as code (PlantUML, Structurizr DSL) in the repository and rendered in CI, so they stay in sync with the system they describe. A new engineer reading a C4 Context diagram understands the system in five minutes instead of five days.

What it is

The C4 model is a set of four hierarchical diagram types for software architecture, created by Simon Brown. C4 stands for Context, Container, Component, Code. Each level answers a different question for a different audience:

  • Level 1 — Context: Where does this system fit in the world? Shows users, external systems, and the system being described. Audience: executives, new joiners, product managers.
  • Level 2 — Container: What is the system made of? Shows apps, services, databases, and message brokers. Audience: developers, architects, ops teams.
  • Level 3 — Component: What is inside a container? Shows the major components and their responsibilities. Audience: developers working on that service.
  • Level 4 — Code: Class-level detail. Usually replaced by the IDE. Rarely worth maintaining manually.

The diagrams are stored as code (PlantUML !include C4Context.puml macros, or Structurizr DSL) in the same repository as the system they describe. CI renders them to PNG or SVG on every push, so they cannot drift from the code.

Why we need it

Architecture knowledge without diagrams lives only in the heads of the three engineers who were in the original design meeting. When one of them leaves, or when a new engineer joins, the knowledge has to be reconstructed from code — which takes days and produces guesses, not understanding.

C4 stored as code solves both problems. The diagram is in the PR. The review is in the PR. When the system changes, the diagram changes in the same commit. The five-minute onboarding experience — read the L1 Context diagram, understand the system's position in the world — is what separates teams that scale their documentation from teams that accumulate knowledge silos.

How others do it

Simon Brown (originator)

Developed C4 around 2010 after observing that UML diagrams were either too formal (nobody drew them) or too informal (nobody trusted them). C4 is opinionated enough to produce consistent diagrams across teams but simple enough that an engineer can create an L1 diagram in 20 minutes. Simon Brown also maintains the Structurizr toolchain (the reference C4 renderer). The c4model.com site is the authoritative reference; the notation is freely available and used by thousands of teams.

What we take: Start with L1 and L2 only. Resist the urge to draw L3 before L2 is stable. L2 is the most valuable diagram for cross-team communication.

Source: c4model.com — official reference

Thoughtworks

ThoughtWorks adopted C4 as its default architecture diagramming notation internally and lists "Diagrams as Code" at Adopt level on the Tech Radar since 2021. ThoughtWorks teams store C4 diagrams in Structurizr DSL or PlantUML alongside the services they describe. The toolchain generates architecture documentation sites from the DSL automatically — no manual rendering step needed for developers.

What we take: Use the Structurizr DSL for complex multi-service systems (it models relationships once and renders multiple views). Use PlantUML macros for single-service teams (simpler dependency footprint).

Source: ThoughtWorks Radar — C4 model

ING Bank

ING standardised on C4 as the architecture documentation format across 200+ squads after a 2-year effort to reduce architecture knowledge silos. They extended the model with a "Deployment" view (not part of standard C4) and a "Data Flow" view for regulatory compliance. Each service has a required L1 and L2; L3 is optional but reviewed in architecture reviews. New services without an L2 fail the architecture checklist.

What we take: Make L1 + L2 a mandatory gate in the architecture review checklist. Optional diagrams never get drawn; required diagrams do.

Source: InfoQ — C4 at scale

Anatomy

Context Diagram (L1)

Shows the system as a single box and its external relationships: users, external systems, and data flows. Max 5–8 elements on one diagram. Audience: non-technical stakeholders and new joiners. Required: one per product or system boundary.

Container Diagram (L2)

Shows the deployable units inside the system boundary: services, databases, queues, frontends. Technology labels are shown (PostgreSQL, React, Kafka). Max 10–12 containers before splitting into sub-diagrams. Required: one per service. Updated in the same PR as any container-level change.

Diagram-as-Code Source

PlantUML file using the official C4 macro library (!include C4Container.puml) or Structurizr DSL. Stored in docs/architecture/ alongside the service. CI renders to PNG on every push. The rendered images are committed to the repo so they are viewable in GitHub without a render step.

Notation Legend

Every C4 diagram includes the standard shape legend (Person = filled circle, Software System = large box, Container = box with technology label). Enforced by the C4 macro library. No custom shapes — consistency is more valuable than expressiveness.

Review Gate

Architecture review checklist includes: "Does the PR include an updated L2 diagram if a container was added, removed, or renamed?" Automated check is aspirational; manual checklist is the minimum. Target: 100% of container changes accompanied by a diagram update.

Common pitfalls

Pitfall Why it happens What to do instead
Diagrams become stale within weeks of creation The diagram lives in Confluence or a Google Doc, detached from the code. Engineers update the code but forget the diagram. After 3 months the diagram is fiction. Store diagrams as code in the repo. Add a CI step that renders them. Add the diagram update to the PR checklist for container-level changes.
Too many elements on one diagram (spaghetti L2) Teams add every microservice and every database to a single L2 diagram. After 15 services the diagram is unreadable and nobody refers to it. Apply the "max 12 elements" rule. Split by domain (each bounded context gets its own L2). Use the L1 to show relationships between L2 domains.
Skipping L1 in favour of jumping to L2 Engineers are comfortable with the container level and skip the context level. New joiners and executives never get a system-in-world view. Draw L1 first, always. It is the fastest diagram to create (5 minutes) and the most valuable for non-technical stakeholders. Required before any L2 work.
