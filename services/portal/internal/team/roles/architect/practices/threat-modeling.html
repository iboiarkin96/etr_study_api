Threat Modeling — security by design

Identify and mitigate security threats at design time, when they cost minutes instead of months to fix.

TL;DR

Threat modeling is a structured process for identifying security threats before building. STRIDE (Spoofing, Tampering, Repudiation, Information disclosure, Denial of service, Elevation of privilege), developed by Microsoft in the 1990s, is the canonical framework. Applied per service at design time, it produces a threat model document with mitigations recorded as security requirements. Vulnerabilities found in production cost 100× more to fix than ones caught at design — threat modeling shifts security left and gives the team a shared threat vocabulary.

What it is

Threat modeling is a structured analysis performed at design time to identify security threats and the mitigations required to defend against them. The STRIDE framework, developed at Microsoft in the late 1990s by Loren Kohnfelder and Praerit Garg, categorises threats into six types:

  • Spoofing — impersonating another user or system. Mitigated by authentication.
  • Tampering — modifying data in transit or at rest. Mitigated by integrity controls (signatures, hashes).
  • Repudiation — denying that an action occurred. Mitigated by non-repudiation (audit logs, signed actions).
  • Information disclosure — exposing sensitive data. Mitigated by confidentiality controls (encryption, access controls).
  • Denial of service — making the system unavailable. Mitigated by rate limiting, resource isolation, redundancy.
  • Elevation of privilege — gaining permissions beyond what is authorised. Mitigated by least privilege, authorisation checks.

The session input is a Data Flow Diagram (DFD) — boxes for processes, arrows for data flows, cylinders for data stores, parallel lines for trust boundaries. The team walks through each element and asks each STRIDE question. Identified threats are scored (CVSS or DREAD) and assigned mitigations. The output is a threat model document with required security controls listed as engineering requirements.

Why we need it

Security vulnerabilities found in production are catastrophically more expensive than ones caught in design. Industry data from IBM and others puts the cost ratio between 30:1 and 100:1 for design-time vs. production-time security findings. Beyond direct cost, a single high-severity vulnerability can erode customer trust, trigger compliance penalties, or end a startup. The famous 2017 Equifax breach (147M people affected) originated from a known vulnerability that better threat modeling and patching processes would have caught.

Threat modeling also creates a shared security vocabulary on the team. When the team has run a STRIDE session together, "what about authorisation here?" becomes a natural question in code review. Without threat modeling, security is the responsibility of whoever happens to think of it — which is unreliable at startup speed.

How others do it

Microsoft (STRIDE originator)

STRIDE was developed at Microsoft in the late 1990s and adopted as the threat modeling standard in the Microsoft Security Development Lifecycle (SDL) in the early 2000s. Microsoft maintains the Threat Modeling Tool (free, supported), which generates threat lists from DFDs automatically. The SDL was a response to the security crises of the late 1990s (Code Red, Nimda) and has been documented externally as the canonical example of shifting security left.

What we take: Use the Microsoft Threat Modeling Tool for the first 5–10 threat models as training wheels. The tool's automatic threat enumeration teaches the team to think about STRIDE before they internalise the framework. Once internalised, free-form diagram + STRIDE walkthrough is faster.

Source: Microsoft Threat Modeling Tool

OWASP

OWASP (Open Web Application Security Project) maintains the canonical open-source guidance on threat modeling, including the Threat Modeling Cheat Sheet, the Application Threat Modeling Guide, and OWASP Threat Dragon (open-source modeling tool). OWASP's guidance covers both STRIDE and PASTA (Process for Attack Simulation and Threat Analysis), a heavier methodology used by financial institutions. OWASP also publishes the Top 10 Web Application Security Risks, updated every 3–4 years, which serves as a starting checklist for web-focused threat models.

What we take: Use the OWASP Top 10 as a parallel checklist alongside STRIDE for web applications. STRIDE is comprehensive; Top 10 is prioritised. Combining them catches both systematic threats and high-frequency real-world attacks.

Source: OWASP Threat Modeling

Adam Shostack — Threat Modeling: Designing for Security

Adam Shostack (former Microsoft, now independent) wrote the canonical textbook on threat modeling (2014, Wiley). His "Four Questions Framework" simplifies threat modeling to: (1) What are we working on? (2) What can go wrong? (3) What are we going to do about it? (4) Did we do a good job? Shostack argues that STRIDE is the most learnable threat framework for engineering teams because the categories map directly to security properties developers already understand. The book is the canonical reference and Shostack runs training that has reached thousands of engineering teams.

What we take: Use Shostack's Four Questions as the session structure. The questions are simpler than the full STRIDE methodology and produce 80% of the value. Reserve full STRIDE for high-risk services.

Source: Shostack — Threat Modeling book

Anatomy

Data Flow Diagram (DFD)

A diagram showing: processes (circles), data stores (cylinders), external entities (boxes), data flows (arrows), and trust boundaries (dashed lines). The DFD is the input to threat enumeration. Without a DFD the session devolves into ad-hoc threat brainstorming.

STRIDE Walkthrough Session

A 2-hour session per service. Participants: service owner, architect, security engineer (if available), one other senior engineer. Walk through each DFD element and ask each STRIDE question. Record threats with: description, affected element, severity, status (open/mitigated/accepted).

Risk Scoring

Each threat gets a severity score. CVSS 3.1 (open standard) is preferred over DREAD (subjective). For threats above a configurable threshold (typically CVSS ≥ 7.0), mitigation is mandatory before launch. Lower-severity threats can be accepted with documented rationale.

Mitigation Requirements

Each unmitigated threat translates to a security requirement on the implementation team. Tracked as a security checklist item in the design doc. Verified before launch and re-verified annually. Without explicit tracking, mitigations are aspirational.

Annual Re-Review

Threat models go stale as services evolve. New features add new attack surface; new dependencies introduce new threats. An annual re-review (or after major architecture changes) catches drift. Without re-review, threat models become historical documents that no longer reflect the live system.

Common pitfalls

Pitfall Why it happens What to do instead
Threat model produced once at launch and never updated The team runs a threat modeling session before launch, ships the service, and never revisits. Two years later the service has tripled in scope; the original threat model covers 30% of the current system. Schedule annual re-reviews. Add a checklist item to architecture review: "Has the threat model been updated for this change?" Treat threat models as living documents, not launch artefacts.
Threats identified but never converted to engineering tickets The session produces a list of 30 threats, but nobody owns the conversion to mitigation tickets. The threat model document sits in a folder; the engineering work proceeds without the mitigations. End every session with a ticket-creation step. Each unmitigated threat above the severity threshold becomes a ticket assigned to a named engineer with a due date before launch. Block launch on the tickets.
Threat modeling treated as a security-team-only activity The security team runs threat models in isolation and presents results to engineers. Engineers do not learn the methodology; they passively receive findings. The security team becomes a bottleneck. Train engineers to run threat models themselves. The security team facilitates the first 2–3 sessions per service team, then steps back. Engineers running their own threat models is the scalable end state.
