Domain-Driven Design — model the business first

Structure software around the business domain so that model boundaries match organisational boundaries.

TL;DR

Domain-Driven Design (DDD) is a software design approach introduced by Eric Evans in his 2003 book of the same name. It centres the software model on the business domain through a shared ubiquitous language and explicit Bounded Contexts. Each Bounded Context owns a coherent domain model; a Context Map describes how contexts integrate. The outcome is a system whose seams match the business's natural boundaries, enabling independent team autonomy and reducing cross-team coordination cost.

What it is

Domain-Driven Design is a body of patterns and principles for building software whose structure mirrors the business problem. Eric Evans formalised it in 2003 after observing that the most successful projects shared a common trait: the code's vocabulary matched the domain expert's vocabulary.

The key building blocks:

  • Ubiquitous Language — a shared vocabulary agreed between domain experts and engineers. The same terms are used in code, tests, documentation, and conversations. Removes translation friction.
  • Bounded Context — an explicit boundary within which a domain model is valid and consistent. The same word ("product") can mean different things in different contexts; inside a context it has exactly one meaning.
  • Context Map — a diagram of all bounded contexts and the relationships between them (Customer-Supplier, Conformist, Anti-Corruption Layer, Shared Kernel, Published Language).
  • Aggregates — clusters of domain objects treated as a single unit for data changes. Each aggregate has a root that enforces invariants.
  • Domain Events — immutable records of something that happened in the domain (OrderPlaced, PaymentFailed). The primary integration mechanism between contexts.

Why we need it

Without domain boundaries, microservices share database tables, the same concept accumulates three different names across three teams, and every cross-service change triggers a coordination spiral. "User" means the authentication record to the Identity team, the billing record to Finance, and the analytics profile to Growth — and none of them agree on which one is authoritative.

DDD gives teams autonomy with explicit ownership. A team that owns an Order bounded context can evolve its model independently, because the boundary makes the integration points explicit and stable. Startups that invest in DDD early grow their teams without a proportional increase in coordination overhead.

How others do it

Eric Evans (originator)

Published Domain-Driven Design: Tackling Complexity in the Heart of Software in 2003 after observing patterns in complex enterprise projects. Introduced Bounded Contexts, Ubiquitous Language, Aggregates, and the Context Map. The "Blue Book" remains the canonical reference. Evans later introduced the "Distillation" concept — identifying the Core Domain (the 10–20% of the model that delivers competitive advantage) vs. supporting subdomains.

What we take: Focus DDD investment on the Core Domain first. The supporting domains can use simpler patterns (Active Record, CRUD) without losing the strategic benefit.

Source: domainlanguage.com — DDD reference

Spotify

Used context mapping to design the Squad model. Each Squad owns a bounded context (Playlists, Radio, Discovery). Context boundaries align with team boundaries, enforcing Conway's Law deliberately. Spotify's "Tribes and Squads" structure is DDD applied at the organisational level — the system architecture and the team topology mirror each other by design.

What we take: Draw the context map before drawing the org chart. The team structure should follow the domain boundaries, not precede them.

Source: Spotify Engineering Culture (2014)

Airbnb

Migrated from a monolith to service-oriented architecture using DDD bounded contexts as the seam finder. The "Service Decomposition" initiative identified 30+ domains (Listings, Bookings, Payments, Identity, Pricing) by running Event Storming workshops with domain experts. Each domain became an independent service owned by one team. The migration took 3 years; without DDD context maps it would have been ad-hoc service splits that recreated the monolith's coupling across the network.

What we take: Event Storming workshops are the fastest way to discover bounded contexts in an existing monolith — faster than reading code.

Source: Airbnb Engineering — Building services

Anatomy

Ubiquitous Language Glossary

A written dictionary of 20–50 domain terms agreed between engineers and domain experts. Each term has exactly one definition per bounded context. Reviewed quarterly; any new term added to the codebase must appear in the glossary first.

Bounded Context Map

A diagram showing all contexts and their relationships. Relationships are typed: Customer-Supplier, Conformist, ACL, Shared Kernel, Published Language, or Separate Ways. Stored as a PlantUML file in the repo. Reviewed at each architecture retrospective.

Aggregate Root

The single entry point to a cluster of domain objects. All state changes go through the aggregate root. Max recommended aggregate size: 3–5 entities. An aggregate that spans more than one database transaction is a smell.

Domain Events

Immutable records of domain occurrences. Verb past tense (OrderPlaced, UserVerified). Published as integration events between contexts. Each event carries a correlation ID. Retained for at least 30 days in the event log.

Anti-Corruption Layer (ACL)

A translation layer that prevents a downstream context from being contaminated by the upstream context's model. Implemented as a mapper or adapter. Required when two contexts have different ubiquitous languages for the same concept.

Common pitfalls

Pitfall Why it happens What to do instead
Applying DDD everywhere regardless of subdomain type Engineers read the Blue Book and apply Aggregates and Domain Events to CRUD admin screens. The overhead is unjustified for supporting subdomains. Identify the Core Domain first. Apply full DDD (Aggregates, Domain Events, ACL) only there. Use Active Record or simple CRUD for generic and supporting subdomains.
Contexts that are too large ("one context per service" but services are monolithic) Teams draw context boundaries to match existing team boundaries rather than domain boundaries. A 50-engineer "platform team" ends up owning one gigantic bounded context. Size contexts by cognitive load: one context should fit in one team's head (max 5–8 engineers). When a team grows past 8, the context is likely too large and should split.
Ubiquitous language that is code-centric, not domain-centric Engineers name things after implementation details (UserRecord, OrderDTO) rather than domain concepts (Customer, Booking). Domain experts stop engaging because the vocabulary is foreign to them. Run Event Storming with domain experts present. Use the sticky-note vocabulary in class names. Reject PR names that use technical suffixes (DTO, Record, Manager) when a domain term is available.
