Design Docs — discuss before you build

A short written proposal that catches wrong paths before a week of coding instead of after.

TL;DR

A Design Doc is a 2–10 page written proposal covering context, problem, two or three options with trade-offs, the chosen path, success metrics, risks, and open questions — written before implementation begins. The format was made canonical at Google, where any engineer can author a design doc and any engineer can comment. The Design Doc proposes; the ADR records the result. When designs are discussed in writing first, the team catches the wrong path in a 90-minute review instead of after a week of coding.

What it is

A Design Doc is a written proposal for a non-trivial technical change. It is the document an engineer writes before implementation begins, not after. The canonical format covers:

  • Context — what currently exists, what changed, what forces are driving this work.
  • Problem statement — the specific problem to solve, with success criteria.
  • Options considered — two or three approaches with pros, cons, and trade-offs.
  • Recommended path — the chosen approach and the reasoning.
  • Success metrics — how will we know the change worked?
  • Risks & rollout plan — what could go wrong; how the change is deployed.
  • Open questions — what the author wants feedback on specifically.

Length: 2–10 pages. Less is more. A design doc that exceeds 10 pages typically has scope problems — it is solving multiple problems at once and should be split. The reviewer audience comments asynchronously over 3–7 days. After resolution, the resulting decision is captured as an ADR; the design doc is archived as the analysis that informed it.

Why we need it

The cheapest moment to discover a wrong design is when it is written down on paper. A 90-minute reading by three senior engineers can catch a flawed assumption that would otherwise take 3 weeks of implementation to surface. The act of writing also forces the author to confront vagueness: ideas that sound coherent in a verbal pitch reveal their gaps when written.

Design docs also create institutional memory. A reader who returns to the doc a year later can see what alternatives were considered, what the team chose, and why. Without a design doc, the same conversation is re-litigated every time a new engineer joins. Google's Design Docs at Google post explicitly identifies this artifact as the most consistently valuable technical writing investment at scale.

How others do it

Google

Google made design docs a universal engineering practice — any engineer can write one, any engineer can comment. Google's docs follow a standard template (context, scope, requirements, design, alternatives, considered) and are stored in a shared Google Docs folder. Senior engineers (L6+) routinely review docs across teams as a form of mentoring and cross-pollination. Malte Ubl (Google) documented the practice in his 2020 essay Design Docs at Google: the doc is the design, the implementation follows from it.

What we take: Design docs are not just for senior engineers writing major proposals — they are for any engineer working on a non-trivial change. The act of writing scales the engineer's thinking before the implementation is committed.

Source: Industrial Empathy — Design Docs at Google

Stripe

Stripe's engineering culture is famously documentation-heavy, with design docs as one of the central artefacts. Stripe popularised the practice of writing "Memos" (modelled on Amazon's 6-page memo culture) for major decisions. Each memo follows a structured format and is read silently for the first 20 minutes of any major decision meeting. Stripe also publishes its API design principles as written guidance, derived from internal design docs that became canonical.

What we take: The "silent reading at the start of the meeting" practice (from Amazon, adopted by Stripe) is a powerful adaptation. It guarantees that everyone has read the doc before discussion starts.

Source: Stripe Engineering Principles

Amazon (6-page memo culture)

Amazon's "6-page memo" culture, established by Jeff Bezos, replaces PowerPoint presentations with narrative documents for major decisions. The format requires complete sentences, full paragraphs, and explicit FAQ sections that anticipate objections. The first 20 minutes of any decision meeting is silent reading. This format is closely related to design docs and has shaped how many ex-Amazon engineers run their teams at startups and at other tech companies.

What we take: The FAQ section at the end of the design doc is high-value. Anticipate the questions reviewers will ask and answer them in the doc itself. Reduces review iteration time.

Source: Bezos 2016 letter to shareholders

Anatomy

Header Block

Title, author(s), reviewers (named), status (draft / in review / accepted / superseded), created date, target decision date. The named reviewer list is the gate for distribution — if a stakeholder is not named, they may not see it in time to comment.

Context Section

2–4 paragraphs describing what currently exists and what forces are driving the proposed change. A reader new to this area should understand the situation without consulting external sources. Cites prior ADRs and design docs where relevant.

Options Comparison

2–3 distinct options with pros, cons, and effort estimates. One option is the recommended path; the others provide the comparison that makes the recommendation defensible. An option set of one is not a comparison — it is a proposal disguised as analysis.

Success Metrics

Explicit, measurable outcomes that will tell the team whether the change worked. "p99 latency reduced from 800ms to 200ms within 30 days" is a valid metric; "performance will improve" is not. Without metrics, the project ships and the team has no way to know if the design was correct.

FAQ & Open Questions

Two sections at the end. FAQ anticipates objections and answers them ("Why not use Kafka? Because we already operate SQS at scale and the additional broker introduces operational overhead disproportionate to the throughput need"). Open Questions list what the author explicitly wants feedback on.

Common pitfalls

Pitfall Why it happens What to do instead
Design doc written after implementation is mostly done The engineer codes first because coding feels productive and writing feels like overhead. The doc is written retroactively to satisfy a process requirement. Reviewers cannot meaningfully change the design because the code already exists. Write the design doc as the first deliverable of the work. The first commit of the implementation references the design doc PR. No design doc, no implementation start.
Single-option design docs (proposal disguised as analysis) The author already knows what they want. The Options section contains one real option and one straw-man. Reviewers cannot meaningfully push back because there is no real alternative to compare to. Require two genuine options in any Options section. If only one option exists, write a short ADR — be honest about the decision being already made. The Options section requires real analysis or it should not exist.
Design doc length grows to 30+ pages The author adds every detail anticipating questions. The doc is too long to read in one sitting. Reviewers skim and miss critical issues. Cap at 10 pages with linked appendices for detail. If the doc exceeds 10 pages, the scope is probably too broad — consider splitting into two docs for separable decisions.
