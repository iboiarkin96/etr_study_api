Build vs Buy vs OSS — structured decision framework

Evaluate every significant component systematically to avoid building what commodities provide and buying what differentiates you.

TL;DR

Build vs Buy vs OSS is a structured decision framework for evaluating whether to custom-build, purchase, or adopt open-source for any significant system component. The key dimensions are: competitive differentiation (does this provide a moat?), 3-year TCO, vendor lock-in risk, integration complexity, and team capability. The output is a scored comparison table documented as an ADR. Startups that skip this framework either build infrastructure they should buy or buy products that become strategic constraints.

What it is

Build vs Buy vs OSS is a decision framework applied to any significant new component or capability. "Significant" means: above $5k/year in SaaS cost, above 2 engineer-months to build, or strategically important to the product roadmap. The framework evaluates five dimensions:

  • Differentiation — does building this provide competitive advantage? If competitors can buy the same SaaS solution, building it is not differentiating.
  • 3-year Total Cost of Ownership (TCO) — full lifecycle cost including engineering time, infrastructure, support, and migration risk.
  • Vendor lock-in risk — how hard is it to migrate away? Data portability, API openness, and contract terms.
  • Integration complexity — how many touch points does this have with core systems? High-integration components are higher-risk purchases.
  • Team capability — does the team have the expertise to build and operate this well? Building something the team does not understand creates a different kind of risk.

The output is a 5-column ADR: option, differentiation score, 3-year TCO estimate, lock-in risk, verdict.

Why we need it

Startups fail in two opposite directions. The first: building infrastructure that commodities provide for free. A team that spends 6 months building an email delivery system when SendGrid exists for $50/month has misallocated six months of engineering capacity that should have gone to the core product. The second: buying SaaS products for core differentiators. A team that buys a third-party recommendation engine for an e-commerce business cannot differentiate on recommendations — which is often the core business value.

The framework prevents both failure modes by forcing an explicit answer to "is this a differentiator?" before any build decision is made. At startups, the default answer should be "buy unless you have a compelling differentiation argument."

How others do it

AWS / Bezos (originator context)

Jeff Bezos's famous "commodity or differentiator" framing underpins AWS's own creation: Amazon realised that compute infrastructure was a commodity for their retail business and chose to "buy" it internally (build AWS) so they could then sell it. The same logic applies in reverse — for a startup, compute is a commodity: buy it from AWS rather than building a data centre. The "two-pizza team + APIs" rule extended this to internal services: if a capability can be wrapped in a clean API, it becomes a buyable service.

What we take: Infrastructure is almost always commodity — buy it. Core business logic is almost always differentiating — build it. The grey area is data: data pipelines and ML models may or may not be differentiating depending on the business.

Source: a16z — The case for buying not building

Stripe

Stripe's engineering principles include "use commodity tools for commodity problems." Their payments core is entirely custom-built (differentiating). Their developer tooling (Homebrew packages, GitHub Actions, Datadog) is all bought or OSS. Stripe publishes its engineering principles publicly and explicitly states that building your own monitoring system is a waste of time when Datadog or New Relic exist. The principle has a counterpart: "build what you need to be the best in the world at."

What we take: Define the "best in the world at" list explicitly. Everything else is a candidate for buy/OSS. The list should be short: max 3–5 capabilities for a startup.

Source: Stripe Engineering Principles

Linear

Linear, a project management startup, explicitly builds their own data sync infrastructure (ElectricSQL-based multiplayer sync) because real-time sync is their core differentiator over Jira. They buy everything else: hosting (AWS), monitoring (Datadog), auth (Clerk). The Linear Method blog documents their build/buy decisions explicitly — their architecture choices follow directly from "what makes Linear faster than Jira" as the guiding question.

What we take: Ask "what makes us 10× faster/better than the competition?" That answer defines what to build. Everything else is a buy or OSS candidate.

Source: Linear Method

Anatomy

Differentiation Assessment

Score 1–5: Does this capability provide competitive advantage that cannot be replicated by buying a SaaS product? 1 = pure commodity (logging, monitoring), 5 = core differentiator (recommendation engine for a personalisation business). Score ≥ 4 is a build signal.

3-Year TCO Estimate

For Build: estimate initial engineering cost (days × rate) + ongoing maintenance (% of build cost per year) + infrastructure cost. For Buy: license fee + integration cost + migration risk cost (probability of migration × estimated migration cost). Most build decisions underestimate maintenance cost — use 30% of build cost per year as the default.

Vendor Lock-in Risk Score

Score 1–5: How hard is it to migrate away from this vendor? 1 = no lock-in (open protocol, data portability), 5 = catastrophic lock-in (proprietary format, no data export, single vendor API). Score ≥ 4 for a core capability is a build or OSS signal.

ADR Output

The decision is documented as an ADR with a comparison table: option, differentiation score, 3-year TCO, lock-in risk, team capability, verdict. Required for any component above $5k/year or 2 engineer-months to build. Reviewed annually — decisions made 2 years ago may have a different answer today.

Common pitfalls

Pitfall Why it happens What to do instead
NIH syndrome — building because it's more interesting "Not Invented Here" syndrome: engineers prefer to build because building is more interesting than configuring. The differentiation argument is post-rationalised. The team builds a custom logging system instead of using Datadog. Apply the "if a competitor buys this capability tomorrow, does that hurt us?" test. If the answer is no, it is commodity. Buy it. Reserve build for capabilities where a yes answer is credible.
Underestimating the true cost of OSS self-hosting The team adopts open-source software to avoid SaaS fees. The initial cost is zero. The ongoing operations cost (upgrades, security patches, on-call for the service, 3-year accumulation) often exceeds the SaaS alternative. Add a "managed vs self-hosted OSS" sub-decision. Always estimate the self-hosting maintenance cost (typically 20–40% of initial setup cost per year). Compare to the SaaS alternative explicitly.
Lock-in discovered after procurement, not before The vendor lock-in dimension is not evaluated at purchase time. Two years later the team discovers data is in a proprietary format with no export, or the API is the only access path to the data and the vendor has raised prices 10×. Evaluate lock-in before signing the contract. Read the data export terms and the API terms. For any score ≥ 4 on the lock-in dimension, negotiate data portability terms into the contract before signing.
