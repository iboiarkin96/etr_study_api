Architecture Decision Records — numbered, durable

A one-page record of why each architectural decision was made, stored next to the code it describes.

TL;DR

An ADR (Architecture Decision Record) is a short numbered file with three sections: Context, Decision, Consequences. Michael Nygard introduced the format in 2011. Each ADR has a permanent number, a status (proposed / accepted / superseded), and is stored in the repo alongside the code it describes. It is never deleted — only superseded by a later ADR. New hires read ADR 0042 once and stop asking the question in chat.

What it is

An Architecture Decision Record is a one-page document capturing a single architectural decision. The format was popularised by Michael Nygard in his 2011 essay Documenting Architecture Decisions. The structure has just three required sections:

  • Context — what is happening, what constraints we face, what forces are pushing the decision. Plain prose. The reader should understand why a decision was needed without consulting external sources.
  • Decision — the chosen path, stated in the active voice: "We will use PostgreSQL for the primary datastore." Specific, not aspirational.
  • Consequences — what now changes because of this decision. Both good and bad. An ADR with no downside listed is half-written.

The MADR 4.0 template (Markdown ADR) extends Nygard's original with fields for Status, Date, Deciders, and a Considered Options table — useful at organisational scale. The file has a permanent number (ADR 0042) that never changes, even after supersession.

Why we need it

A year from now, the team will not remember why the project chose gRPC over REST for internal services. The decision will become tribal knowledge held by the three engineers who were in the original meeting — and when they leave, that knowledge leaves with them. The next engineer to question the choice will re-argue it from scratch.

An ADR fixes this in 30 minutes. The decision is written down with its context. When a new engineer asks "why gRPC?", the answer is "read ADR 0017." The number gives every decision a stable link — across reorgs, repo moves, and team changes. The discipline of writing down decisions also improves the quality of decisions: writing the Consequences section forces the team to be honest about trade-offs.

How others do it

Michael Nygard (originator)

Wrote the original ADR essay in November 2011 on the Cognitect blog: Documenting Architecture Decisions. The format was deliberately minimal — three sections, one page, plain prose — because Nygard observed that heavier documentation formats (architecture documents, design specs) were never written and never read. The minimalism is the feature: an ADR is fast to write and fast to read, which is the only way it gets done at all.

What we take: Resist the urge to expand the template. Three sections work because they fit on one screen. Adding a "Risks" or "Open Questions" section reduces adoption faster than it improves quality.

Source: Cognitect — Documenting Architecture Decisions (2011)

Spotify

Adopted MADR (Markdown ADR) as the canonical template across all squads. Spotify's ADRs include a "Deciders" field (named individuals, not roles) and a "Considered Options" table that captures the alternatives explicitly. Every significant architectural choice across 200+ squads has an ADR. Spotify also surfaces ADRs in their internal developer portal (Backstage) so engineers can find decisions by service, by author, or by date.

What we take: The "Deciders" field with named individuals (not roles) is high-value at organisational scale. Roles change; names are searchable. Add the Considered Options table to document the alternatives — the trail is the value.

Source: MADR — Markdown ADR template

AWS Well-Architected

AWS Well-Architected Framework recommends ADRs as the canonical artifact for the Operational Excellence pillar. AWS provides their own template in the Well-Architected Tool. Their guidance: ADRs that document why a particular AWS service was chosen are gold during compliance audits — "why are you using DynamoDB and not Aurora?" has a documented answer in an ADR rather than a meeting transcript that no longer exists.

What we take: ADRs are an audit asset. When a cloud provider, regulator, or customer asks "why did you choose X?", an ADR is the answer. Treat them with the rigour appropriate for an audit document.

Source: AWS Prescriptive Guidance — ADRs

Anatomy

Permanent Number

Sequential integers (0001, 0042) assigned at PR creation time, never reused. The number is the stable link. Even when an ADR is superseded, the file keeps its number. Numbers above 9999 are exceptionally rare — most projects have 50–300 ADRs over their lifetime.

Status Field

One of: proposed / accepted / declined / superseded by NNNN. Status changes are themselves PRs. An accepted ADR is never edited (typos can be fixed, content cannot); if the decision changes, a new ADR supersedes the old one. The supersession trail is the value.

Three Required Sections

Context, Decision, Consequences. Plain prose, no headings inside sections. Max one page total. The Consequences section must list both positive and negative outcomes — an ADR with no downside listed has not been thought through.

Supersession Link

When ADR 0099 supersedes ADR 0017, ADR 0017's status becomes "superseded by 0099" with a forward link. ADR 0099 includes a backward link to 0017 and explains what changed. The reader can follow the chain to understand the decision evolution.

Repository Location

Stored in docs/adr/ or governance/adr/ in the repo, next to the code it describes. Named NNNN-kebab-case-title.md or .html. An ADR index page is generated in CI from the directory listing so the team always has an up-to-date catalogue.

Common pitfalls

Pitfall Why it happens What to do instead
ADRs that grow into design documents (10+ pages) Authors include implementation detail, code snippets, and exhaustive trade-off analysis. The ADR becomes a design doc. Nobody reads 10-page ADRs; they revert to "ask the author." Hard limit: one page. If the decision needs more context, write a separate design doc and link it. The ADR captures the decision; the design doc captures the analysis.
Major decisions made without an ADR ("decision archaeology") Big choices (which database, which auth mechanism) happen organically without a formal decision moment. There is no ADR until something breaks years later and the team cannot explain the choice. Schedule a quarterly "decision archaeology" review: identify undocumented major decisions and write retroactive ADRs. Tag them with a "retroactive" note in the Status to be honest about their origin.
No supersession discipline — ADR list grows but nothing is ever superseded The team writes ADRs but never marks them as superseded. The list grows; conflicts accumulate; readers cannot tell which ADRs are still in effect. Add supersession review to the quarterly architecture retrospective. Any ADR that no longer reflects current practice should either be superseded by a new ADR or have its status changed to "deprecated" with rationale.
