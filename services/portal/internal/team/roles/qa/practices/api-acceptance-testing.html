API Acceptance Testing — systematic per-endpoint sign-off

A structured checklist pass that verifies every HTTP endpoint against its specification before it is marked QA-verified.

TL;DR

API acceptance testing is a systematic pass that verifies every HTTP endpoint across six dimensions: contract conformance, happy-path responses, error codes (4xx/5xx), authentication gates, idempotency behaviour, and observability signals (X-Request-ID in headers and logs). It is the QA sign-off gate for each endpoint. Without this pass, endpoints routinely reach production missing authentication edge cases, incorrect error shapes, or dropped correlation IDs. The team uses checklist 0002 as the acceptance criterion for every endpoint.

What it is

API acceptance testing is a structured QA pass that verifies every HTTP endpoint of a service against its specification before the endpoint is marked ready for integration. The pass is not exploratory — it follows a fixed checklist (checklist 0002 in this project) that covers six dimensions:

  1. Contract conformance: The endpoint path, HTTP method, request schema, and response schema match the OpenAPI specification. No extra fields in the response; no missing required fields.
  2. Happy-path responses: The endpoint returns the correct status code (200, 201, 204, etc.) with the correct response body shape for valid inputs.
  3. Error codes: The endpoint returns the correct error status (400 for bad input, 404 for missing resource, 422 for validation failure, 401/403 for auth failures) with the correct error envelope shape.
  4. Authentication gates: The endpoint correctly rejects requests with missing tokens (401), expired tokens (401), and tokens with insufficient permissions (403).
  5. Idempotency behaviour: For endpoints that should be idempotent (PUT, PATCH with idempotency key, POST with idempotency key), duplicate requests return the same result as the first request without creating duplicate resources.
  6. Observability signals: The X-Request-ID header is present in the response and appears in the application logs for every request, enabling distributed tracing.

The acceptance pass is performed by the QA engineer using Postman, Insomnia, or a purpose-built test script. The checklist is annotated with evidence (request/response captures, X-Request-ID traces) and stored with the release artefacts.

Why we need it

Without a systematic per-endpoint acceptance pass, endpoints routinely reach integration or production with systematic defects: missing authentication checks on one method, incorrect error envelope shape for one status code, or X-Request-ID not propagated to the database query logs. These defects are invisible during development (the happy path passes) and surface only when consumers integrate the endpoint or when an incident requires log tracing.

The checklist turns the acceptance pass from a subjective "looks good to me" into an objective, repeatable test with documented evidence. Every future QA engineer can run the same checklist and verify the same properties.

How others do it

Postman

Postman's API testing platform is the most widely used tool for API acceptance testing. Teams create Postman collections with test scripts that automate the six-dimension check. Postman's "Collection Runner" executes the entire collection and produces a test report. Postman also integrates with CI: the Newman CLI runs Postman collections in GitHub Actions, blocking the PR if any acceptance check fails.

What we take: Automate the acceptance checklist as a Postman collection that runs in CI. Manual acceptance testing is not repeatable at scale.

Source: Postman — API testing

Swagger / OpenAPI community

The OpenAPI community has produced several tools for automated contract conformance checking: Schemathesis (generates test cases from the OpenAPI spec), Dredd (runs request/response pairs from the spec against a running server), and Spectral (lints the spec itself). Using these tools, the contract conformance dimension of the acceptance checklist can be fully automated — the tool verifies that every response matches the spec without manual test case authoring.

What we take: Automate contract conformance with Schemathesis or Dredd. The remaining five dimensions (auth, idempotency, errors, observability, happy path) still require hand-crafted test cases.

Source: Swagger — API testing best practices

Stoplight

Stoplight's API design platform includes a "Contract Testing" module that verifies running services against their OpenAPI specification. Stoplight's approach to acceptance testing starts during design: the spec is written first, mock servers are generated from the spec, and consumers test against the mock before the provider is implemented. When the provider is ready, the acceptance test runs against the real implementation. This "design-first" workflow means the acceptance checklist is partly automated before a line of code is written.

What we take: Use a mock server generated from the OpenAPI spec to unblock consumer integration while the provider is still under development.

Source: Stoplight — API design platform

Anatomy

Acceptance checklist (0002)

The canonical checklist document that QA uses for every endpoint. Six sections, each with specific pass criteria and an evidence field.

Request/response captures

Saved request/response pairs for each checklist item — evidence that the check was performed and what the system returned.

X-Request-ID trace

Evidence that the X-Request-ID header is present in the response and appears in the application log entry for the same request.

QA-verified status

The endpoint's status in the tracking system. An endpoint is not "QA verified" until the completed checklist 0002 is attached.

Common pitfalls

PitfallWhy it happensWhat to do instead
Skipping the auth dimension QA tests the happy path and error codes but doesn't test authentication edge cases because they are "obviously correct." Authentication failures are among the most commonly shipped defects. Make auth testing mandatory in the checklist, not optional.
No observability check X-Request-ID propagation is treated as a developer concern and not checked by QA. QA owns the observability check. X-Request-ID not in the logs is a defect — it prevents incident tracing. Verify it explicitly on every endpoint.
Acceptance test without evidence QA marks the checklist as complete without attaching request/response captures. The evidence is not available for future reference. Require evidence for every checklist item. A completed checklist without evidence is not a completed checklist.
