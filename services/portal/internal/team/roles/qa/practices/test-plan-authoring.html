Test Plan Authoring — a living document that drives test execution

A concise, structured test plan written before sprint execution that makes the QA approach explicit, reviewable, and reproducible.

TL;DR

A test plan documents six things: scope (what is being tested), out-of-scope (what is not), approach (which techniques — BVA, exploratory, scenario-based), environments, entry/exit criteria, and risk. It is written in one to two pages before sprint execution begins. Without a test plan, the QA approach is implicit — different QA engineers test differently, coverage is not tracked, and the same work cannot be reproduced. A test plan does not mean waterfall; it means writing down what you plan to do before you do it, then updating it when reality diverges.

What it is

A test plan is a concise, structured document (one to two pages) that makes the QA approach for a sprint or feature explicit before execution begins. It is not a comprehensive test specification — it is a plan, which means it describes intent, not every test case.

The six sections of a test plan for this project are:

  1. Scope: What is being tested in this sprint or release. Include specific stories, API endpoints, UI flows, or components by name. Also explicitly list what is out of scope (regression for unmodified features, performance testing, localisation).
  2. Approach: Which test techniques will be applied to which areas. Example: "New API endpoints — API acceptance checklist 0002. UI changes — BVA on input fields, exploratory charter for new flows. Regression — smoke checklist on critical paths."
  3. Environments: Which environments will be used for which test types. CI for unit/integration/lint; staging for API acceptance and scenario tests; production-like for performance.
  4. Entry and exit criteria: When does testing start (entry: all stories in PR review, CI gates green)? When does testing finish (exit: all acceptance criteria met, escape rate target met, release sign-off issued)?
  5. Risk register: What could go wrong with the test plan itself? Example risks: staging environment unstable, QA resource unavailable for 2 days due to other commitments, test data dependency on external service.
  6. Schedule: Timeline with owner for each phase. Not a detailed Gantt chart — a one-line-per-day summary of who is testing what.

The test plan is a living document. When reality diverges from the plan — a story is descoped, the environment is unavailable, a new risk emerges — the plan is updated. The update is as important as the original: a plan that is never updated is a dead document.

Why we need it

Without a test plan, the QA approach is implicit and varies between QA engineers and between sprints. Implicit approaches cannot be reviewed, challenged, or improved. A senior developer reviewing the sprint plan cannot assess whether the QA coverage is adequate for the risk level because the QA plan exists only in the QA engineer's head.

A written test plan also enables knowledge transfer. When a QA engineer is sick or leaves the team, the test plan tells the next engineer what was planned and how far through the plan they were. Without a plan, the testing is restarted from scratch — or skipped.

How others do it

ISTQB (IEEE 829 test plan standard)

The IEEE 829 standard for software test documentation defines the canonical structure of a test plan. ISTQB's Foundation Level certification teaches the IEEE 829 structure as the basis for test planning. The standard defines eight sections: test plan identifier, introduction, test items, features to be tested, features not to be tested, approach, item pass/fail criteria, and test deliverables. The ISTQB adaptation for agile (ISTQB Agile Tester extension) reduces the plan to a lightweight "just enough" document rather than the full IEEE 829 — emphasising that the plan should serve the team, not be a compliance exercise.

What we take: The six-section structure is our lightweight adaptation of IEEE 829. All six sections are necessary; no section is optional. But the plan must be one to two pages, not a 20-page artefact.

Source: ISTQB — Foundation Level Syllabus

Google (testing blog)

Google's testing blog (now part of the Google Testing Blog and Google SRE documentation) documents their approach to "test plans in an agile world." Google's SWE practice requires a one-page test plan for every significant feature before QA engagement begins. The plan is reviewed by the tech lead and the product manager — not just by QA. Google's key insight: the test plan review catches gaps in the test approach before testing begins, when the cost of correction is low. A test plan that is not reviewed is not a plan — it is a QA engineer's private notes.

What we take: The test plan must be reviewed by the engineering lead and PM, not just by QA. Review by non-QA stakeholders catches coverage gaps that QA alone would not notice.

Source: Google Testing Blog

Microsoft (one-page test plan)

Microsoft's internal QA practice for Office and Azure features uses a "one-page test plan" template that covers scope, approach, risks, and exit criteria in a single-page format. The template is enforced — QA engineers cannot begin formal testing without a completed, reviewed test plan. Microsoft's rationale: a test plan that cannot fit on one page is too detailed for a sprint-level plan — the detail belongs in the test cases, not the plan. The plan is strategic; the test cases are tactical.

What we take: One to two pages maximum. Strategic document, not tactical. If you are writing individual test steps in the plan, you are writing test cases, not a plan — separate them.

Source: Microsoft — Azure DevOps test plans

Anatomy

Scope and out-of-scope

What is explicitly in scope for this sprint or feature, and what is explicitly excluded. Both are required — absence of an explicit out-of-scope list implies everything is in scope.

Technique-to-area mapping

Which test techniques apply to which feature area. Makes the coverage rationale explicit and reviewable.

Entry and exit criteria

Objective conditions that start and end the test phase. Entry: CI green, stories in PR review. Exit: all criteria met, sign-off issued.

Review and update cadence

Plan is reviewed by engineering lead and PM before testing begins. Updated when scope or constraints change. Final state archived with the release record.

Common pitfalls

PitfallWhy it happensWhat to do instead
Plan written after testing starts QA starts testing immediately to save time. The plan is written after testing to satisfy a process requirement, documenting what happened rather than planning what will happen. The plan is an entry criterion for testing — testing cannot begin without a reviewed plan. Enforce this in the sprint policy, not just as guidance.
Plan too detailed (test cases in the plan) QA confuses the test plan with the test case specification. The plan becomes a 20-page document that nobody reads. The plan is strategic — approach, scope, risks. Test cases are tactical — specific steps and expected results. Keep them separate. The plan is one to two pages; the test cases are in the test management tool.
Plan not reviewed by non-QA stakeholders The plan is reviewed only by QA. The engineering lead and PM do not see it. Coverage gaps that a non-QA reviewer would catch are missed. Route the plan to the engineering lead and PM for review before testing starts. Five minutes of their time may save days of rework.
