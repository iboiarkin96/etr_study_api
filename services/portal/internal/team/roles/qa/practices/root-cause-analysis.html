Root Cause Analysis — identify why defects escaped detection

Applies the 5-Whys technique to escaped defects to find the systemic cause and prevent recurrence — not just to fix the symptom.

TL;DR

Root cause analysis (RCA) for QA asks not just "what broke" but "why did our test process not catch this before it reached production?" The 5-Whys technique iterates down from the observable defect to the systemic cause: missing test case → missing scenario in specification → Three Amigos session skipped → no time boxed for it → no sprint policy requiring it. Each "why" peels back a symptom to reveal the process gap. The output of RCA is a process improvement, not a bug fix.

What it is

Root cause analysis (RCA) in QA is the discipline of asking why a defect escaped the test process, not just what the defect was. Fixing the defect is necessary but insufficient — without understanding why QA did not catch it, the same class of defect will escape again through the same gap in the test process.

The primary technique is the 5-Whys, developed at Toyota and applied across quality disciplines:

  1. Start with the observable symptom: "a defect reached production."
  2. Ask "why?" and state the immediate cause.
  3. Ask "why?" about that cause.
  4. Continue until you reach a cause that is a process gap (something the team controls) rather than a technical fact.
  5. The root cause is the process gap. The action item is the process change that closes the gap.

RCA is applied to every escaped Blocker and Critical defect, and to any pattern of escaped defects (three or more Minor/Major defects in the same feature area within a sprint). The output is a brief RCA note (5–10 lines) attached to the defect record, plus an action item in the sprint backlog for the process change.

RCA is blameless — the goal is to find the process gap, not to identify who made a mistake. A blameless RCA produces honest answers; a blame-oriented one produces self-protective answers that don't address the real cause.

Why we need it

Without RCA, the response to an escaped defect is: fix the defect, add a test for that specific case, move on. This produces a test suite that is reactive — it grows to cover yesterday's defects but not tomorrow's. The underlying process gap (missing scenario definition session, inadequate risk-based test coverage, CI gate not configured for this code path) remains open and produces the next escape.

With RCA, each escaped defect improves the process rather than just adding a specific test. The test suite becomes proactive — it covers not just known defects but the classes of code paths and scenarios that have historically been missed.

How others do it

Toyota / Taiichi Ohno (5-Whys origin)

Taiichi Ohno developed the 5-Whys technique at Toyota as part of the Toyota Production System in the 1970s. Ohno's insight was that visible defects are always symptoms of systemic process problems — fixing the symptom without identifying the systemic cause means the next defect of the same class is already in the pipeline. Ohno used an example of a robot stopping: the observable cause (blown fuse) leads through four whys to the root cause (no strainer on the pump, allowing metal chips to contaminate the lubricant). The fix is adding the strainer, not replacing the fuse.

What we take: Stop when you reach a cause you can fix systemically. Adding a single test case is not a root cause fix — it is a symptom treatment. The root cause fix changes the process so the class of defect is caught systematically.

Source: 5-Whys — Toyota Production System

Google (blameless postmortems)

Google's Site Reliability Engineering (SRE) practice establishes blameless postmortems as the standard for production incidents. Blameless means the RCA focuses on process and system failures, not individual failures. Google's postmortem template includes: timeline of events, contributing causes (multiple, not one), action items with owners and due dates, and lessons learned. Google requires that every production incident above a severity threshold has a postmortem published internally within 72 hours. The blameless culture is explicitly stated in the SRE book as a prerequisite for honest postmortems.

What we take: Blameless RCA and a postmortem template. The template forces structured thinking; blameless culture produces honest answers. Both are required for RCA to be effective.

Source: Google SRE — Postmortem culture

Netflix (chaos engineering)

Netflix extends blameless RCA to proactive failure injection through chaos engineering (Chaos Monkey, Simian Army). Rather than waiting for defects to escape, Netflix deliberately introduces failures in production to discover system weaknesses before they cause customer-impacting incidents. After each chaos experiment, the team conducts a "game day retrospective" — an RCA of the failure modes discovered. This proactive RCA is paired with the reactive RCA of actual incidents, creating a feedback loop that continuously improves system resilience.

What we take: Treat game days (deliberate failure exercises) as RCA opportunities, not just reliability tests. The RCA of a controlled failure is more productive than the RCA of an uncontrolled incident.

Source: Netflix — Chaos Engineering

Anatomy

Trigger criteria

RCA is mandatory for every escaped Blocker or Critical defect, and for any pattern of three or more escaped defects in the same feature area within a sprint.

5-Whys chain

The documented why chain: symptom → cause 1 → cause 2 → … → root cause (process gap). Stop when the cause is something the team can change.

Blameless framing

RCA focuses on process and system gaps, not individual errors. No names appear in the why chain — only process steps and missing artefacts.

Action item

One or more sprint backlog items to close the identified process gap. Owned by QA lead. Status tracked in the following sprint retrospective.

Common pitfalls

PitfallWhy it happensWhat to do instead
Stopping at "human error" The 5-Whys chain ends at "the developer made a mistake" or "QA missed it." Human error is never a root cause — it is always a symptom of a process that does not prevent or detect human errors. Ask why the human error was possible. What process or tool should have caught it? Human errors are expected; the process must be designed to catch them.
RCA without action items The RCA identifies the root cause but produces no action item. The process gap remains open; the next similar defect escapes the same way. Every RCA must produce at least one action item in the sprint backlog, with an owner and a due date. An RCA without an action item is a description, not an analysis.
Blame-oriented RCA The RCA names individuals as causes rather than process gaps. Team members give self-protective answers; the real cause is not surfaced. Explicitly state blameless framing at the start of every RCA session. No names in the why chain. Review the RCA output before publishing: if it names a person, reframe to name the process gap.
