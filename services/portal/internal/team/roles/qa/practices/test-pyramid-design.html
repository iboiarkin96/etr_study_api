Test Pyramid Design — shape the automation portfolio

Defines the right ratio of unit, integration, and end-to-end tests so CI stays fast and the feedback loop stays tight.

TL;DR

The test pyramid (Mike Cohn, 2009) divides automated tests into three layers: many fast unit tests at the base, fewer integration tests in the middle, and a thin layer of slow end-to-end tests at the top. Inverted pyramids — too many E2E, too few unit tests — produce test suites that take 40 minutes to run, flake constantly, and give no signal on which line broke. The pyramid shape is an explicit design decision documented in an ADR and reviewed quarterly.

What it is

The test pyramid is a mental model for the composition of an automated test suite, introduced by Mike Cohn in his 2009 book Succeeding with Agile. It divides tests into three layers based on their speed, cost, and granularity:

  • Unit tests (base): Test individual functions or classes in isolation. They run in milliseconds, are deterministic, and are authored by developers. The base is the widest — the project should have the most of these.
  • Integration / contract tests (middle): Test interactions between components or services. They are slower than unit tests but faster than E2E. In our stack, this layer includes contract tests (Pact or OpenAPI-based) and database integration tests.
  • End-to-end / system tests (top): Test the system from the outside as a user would. They are the slowest and the most prone to flakiness. The top is the narrowest — the project should have the fewest of these.

The pyramid shape is not merely descriptive — it is a design constraint. Teams that allow the pyramid to invert (more E2E than unit tests) systematically accumulate technical debt in their test infrastructure until the suite is slower than manual testing and nobody trusts its results.

At Study App, ownership across layers is explicit: developers own the unit and integration layers; QA co-owns the contract layer and owns the system/E2E layer. The split is documented in an ADR and ratios are reviewed in the quarterly quality retrospective.

Why we need it

An inverted pyramid — a suite dominated by E2E tests — produces a predictable set of problems: a CI run that takes 40 minutes, a flake rate that means every third build fails for no reason, and test failures that point to the wrong layer of the stack. Developers stop trusting the suite and start ignoring failures. When a real defect occurs, it is buried in noise.

The pyramid gives the team an explicit target shape for the suite. When a new feature is delivered without unit tests, the pyramid has moved in the wrong direction — that is visible and correctable. When an E2E test is added without an equivalent integration test, the pyramid has moved in the wrong direction — again visible. The shape is the contract.

How others do it

Mike Cohn (originator)

Introduced the three-layer pyramid in Succeeding with Agile (2009). The original formulation used "unit tests / service tests / UI tests." The key insight was that tests at each layer serve a different purpose — unit tests find regression quickly; service tests verify integration; UI tests verify user-visible behaviour. The pyramid was a corrective to the prevailing pattern of test suites dominated by UI automation.

What we take: The three-layer vocabulary and the explicit prohibition on inverting the pyramid. "If you have more UI tests than unit tests, your pyramid is upside-down."

Source: Martin Fowler — The Practical Test Pyramid

Google

Google codified the pyramid as the 70/20/10 split in their internal engineering guidelines: 70% small (unit), 20% medium (integration), 10% large (E2E). The Google Testing Blog post "Just Say No to More End-to-End Tests" documents why Google moved away from large E2E suites: they were slow, flaky, and gave poor defect localisation. Test ownership follows the pyramid: small tests are owned by the developer, large tests are owned by a dedicated test engineer.

What we take: The 70/20/10 ratio as a starting target and the ownership model where layer determines who writes the test.

Source: Google Testing Blog — Just Say No to More End-to-End Tests

Spotify

Spotify squads maintain pyramid ownership at squad level: each squad's test inventory shows the current ratio of unit/integration/E2E tests as part of their quality dashboard. Squads with inverted pyramids are flagged in the quality review and given a quarter to correct the shape. The contract layer at Spotify is owned by a dedicated "API quality" function that works across squads.

What we take: Tracking the pyramid shape as a metric on the quality dashboard, so drift from the target is visible over time rather than discovered during a postmortem.

Source: Spotify Engineering Blog

Anatomy

Layer definitions

A written definition of what counts as a unit test, an integration test, and an E2E test in this project — shared vocabulary prevents disagreements at code review.

Target ratios

Explicit percentage targets for each layer (e.g. 70/20/10). Documented in an ADR. Reviewed quarterly.

Ownership map

Who writes and maintains tests at each layer. Developers own unit and integration; QA co-owns contract; QA owns E2E.

Pyramid shape metric

A number (or dashboard panel) that shows the current layer ratios. Tracked at least quarterly so drift is visible.

Inversion policy

A documented rule: if the pyramid inverts (E2E > integration > unit), a corrective action item is raised and owned by the team lead within one sprint.

Common pitfalls

PitfallWhy it happensWhat to do instead
Pyramid inversion E2E tests are easier to write for QA without code access; unit tests require developers to care about testability. Enforce the ownership model: developers are responsible for the unit and integration layers. QA reviews the ratio quarterly and raises a corrective action when it drifts.
Undefined layer boundaries Teams argue about whether a test is "unit" or "integration" at every code review, wasting time on taxonomy instead of coverage. Write a one-page layer definition document. Specify exactly what "isolation" means (no database, no network, no file system) for unit tests.
Ignoring the contract layer The middle layer (integration/contract) is the hardest to own — it spans team boundaries and requires coordination. Assign the contract layer to QA as explicit ownership. Use Pact or OpenAPI-based contract tests rather than integration tests that require both services running.
