Release Sign-off — structured GO/NO-GO decision at every release

Replaces the informal "it seems ready" release decision with a documented checklist and an explicit QA sign-off that is tracked and auditable.

TL;DR

Release sign-off is a structured process: QA runs a smoke checklist (CI gates green, bug bar met, regression suite passed, API acceptance checklist completed for all new endpoints, accessibility scan clean), then issues a formal GO or NO-GO decision. The decision is documented with evidence — not a verbal "it looks good." A NO-GO means the release is blocked; a GO is the QA commitment that the release meets the quality bar. Without a formal sign-off, releases happen based on schedule pressure rather than quality evidence.

What it is

Release sign-off is the formal QA process that determines whether a release is permitted to proceed. It consists of two parts: a smoke checklist that verifies all quality criteria are met, and an explicit GO/NO-GO decision by the QA lead that is documented with evidence.

The smoke checklist for this project covers seven criteria:

  1. All CI quality gates are green (no bypassed gates).
  2. The bug bar is met: zero open Blocker or Critical defects in the release scope.
  3. The regression suite passed in the release candidate build.
  4. API acceptance checklist (0002) is completed for all new or modified endpoints.
  5. The accessibility scan is clean (no Critical violations, serious violations documented and accepted).
  6. Visual regression is approved (no pending diffs in the visual review queue).
  7. Performance is within SLA (p95 response time below threshold for all critical endpoints).

The GO decision is the QA lead's commitment that the release meets the agreed quality bar. It is documented in the release record with: the date, the checklist evidence, and the QA lead's sign-off. This makes the release decision auditable — if a defect is discovered post-release, the release record shows what was checked and what was known at release time.

A NO-GO is a hard block. The release is postponed until the failing criterion is met. NO-GO decisions are documented with the specific criterion that failed and the action required to resolve it.

Why we need it

Without a formal sign-off process, releases happen when the schedule says they happen — not when quality evidence says they should. Schedule pressure is real and legitimate; without a formal quality gate, it consistently wins over quality concerns. A QA engineer who says "I'm not comfortable releasing this" without a documented checklist has a subjective opinion. A QA engineer who says "criterion 2 (bug bar) is not met — there is one open Critical defect" has an objective, documented fact.

Release sign-off also makes quality expectations explicit. New team members and new QA engineers know exactly what must be true before a release can proceed. The checklist is the canonical quality bar — not an informal understanding that varies by team member.

How others do it

Microsoft (ship room)

Microsoft's "ship room" process for Windows and Office releases is the canonical example of structured release sign-off. The ship room involves the product team, QA, security, and localisation leads reviewing all open bugs against the bug bar. Each team lead gives a GO or NO-GO for their area. The final ship decision requires all leads to be GO. Ship room decisions are documented and maintained as release records. Microsoft's process scales to thousands of developers, but the principle — explicit, documented, multi-stakeholder GO/NO-GO — applies at any scale.

What we take: Explicit GO/NO-GO from each quality discipline (QA, security, accessibility). All criteria must be green before release. Document the decision with evidence.

Source: Microsoft — Release management

NASA (mission readiness reviews)

NASA's Mission Readiness Review (MRR) process is the highest-stakes example of structured sign-off. Before any mission proceeds, all systems teams present their readiness status. Any team can call a "hold" — equivalent to a NO-GO — and the mission does not proceed until the hold is resolved. NASA's process requires documented evidence for every readiness claim: not "we believe the system is ready" but "here is the test data showing the system meets specification X." The MRR process is the model for any high-stakes release process where failure is not recoverable.

What we take: Evidence-based readiness. Every checklist item requires documented evidence, not team belief. "It worked in testing" is evidence; "we think it should work" is not.

Source: NASA — Mission Readiness Reviews

Atlassian (Jira release management)

Atlassian's Jira Software includes release tracking features that formalise the sign-off process: release versions with status (unreleased/released), automated checks for unresolved issues assigned to the version, and release notes generation. Atlassian's internal engineering practice requires a release checklist to be completed in Jira before any release is marked as "released." The checklist includes: all critical and blocker bugs resolved or deferred with documented rationale, release notes reviewed, and post-release monitoring enabled.

What we take: Track the release checklist in the issue tracker, not in a separate document. The evidence is attached to the release record where it can be found retrospectively without hunting through documents.

Source: Atlassian — Release management

Anatomy

Smoke checklist (7 criteria)

CI gates / bug bar / regression / API acceptance / accessibility / visual regression / performance. All seven must be green for a GO.

GO/NO-GO decision

Explicit decision by QA lead. GO = release may proceed. NO-GO = release is blocked; documented with the failing criterion and required action.

Evidence record

The completed checklist with links to CI reports, defect lists, and acceptance checklists. Attached to the release record in the issue tracker.

NO-GO escalation

If the release is time-critical and a criterion cannot be met, the escalation path is: QA lead → engineering lead → product lead. The sign-off authority can be transferred with documented acceptance of the risk.

Common pitfalls

PitfallWhy it happensWhat to do instead
Verbal sign-off without evidence QA says "looks good" in the release meeting. No documented checklist, no evidence record. If a defect is discovered post-release, there is no audit trail. Require a written checklist with evidence links for every release. The sign-off is the completed document, not the verbal statement.
NO-GO overridden by schedule QA issues a NO-GO; management overrides it to meet a deadline. The override is undocumented; QA's authority is undermined. Define the escalation path explicitly: overrides are permitted with documented risk acceptance signed by the engineering lead or product lead. The override and the risk acceptance are on record.
Checklist not updated for new criteria The smoke checklist was written 18 months ago and does not include newer quality criteria (e.g. accessibility scanning was added but not added to the checklist). Review and update the smoke checklist each quarter. New quality practices that require release verification must be added to the checklist when they are adopted.
