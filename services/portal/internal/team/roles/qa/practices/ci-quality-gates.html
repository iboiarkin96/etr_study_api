CI Quality Gates — automated pass/fail checks on every PR

Blocks merging when quality criteria are not met, making quality checks automatic and consistent rather than optional.

TL;DR

CI quality gates are automated pass/fail checks that run on every pull request and block merging when quality criteria are not met. Gates include: unit test pass rate (100%), contract test pass, linting (Spectral for OpenAPI, axe for accessibility), code coverage threshold (≥80% line coverage), and visual regression approval. Each gate failure includes an actionable error message — not just a red build. Without automated gates, quality checks become optional and inconsistent: developers skip checks under deadline pressure and the CI pipeline becomes decorative.

What it is

CI quality gates are automated pass/fail checks configured in the CI pipeline that run on every pull request and prevent merging when quality criteria are not met. Unlike advisory checks (which report results but allow the developer to merge anyway), quality gates are hard blocks — the merge button is disabled until all gates pass.

The gate set for this project covers four categories:

  1. Test gates: All unit tests pass (100% pass rate — any failing test blocks merge). Contract tests pass against all consumer pacts. Code coverage meets the threshold (≥80% line coverage). Integration tests pass in the CI environment.
  2. Lint gates: Spectral lints the OpenAPI specification (zero errors, warnings reviewed). ESLint lints JavaScript with the project rule set. axe-core runs a fast accessibility scan on the built portal pages — critical violations block merge.
  3. Security gates: SAST (static application security testing) scan reports no high-severity findings. Dependency audit (npm audit / pip audit) reports no known critical vulnerabilities in dependencies.
  4. Visual regression gate: Any PR that touches shared CSS, layout templates, or UI-kit components triggers a visual regression run. The gate requires explicit human approval of the diff before the PR can merge.

Each gate failure must produce an actionable error message: not just "gate failed" but specifically which test failed, which lint rule was violated, and where the violation is located. A gate that produces an unactionable error message wastes developer time and trains people to ignore CI output.

Why we need it

Without hard-block quality gates, quality checks become optional under deadline pressure. If the CI pipeline reports results but allows the developer to merge anyway, developers who are under pressure will merge failing builds with the intention of fixing them later. The intention is real; the follow-through is not. Quality debt accumulates faster than it is repaid.

Hard-block gates remove the decision. The developer cannot merge a failing build regardless of deadline pressure. This shifts the conversation from "should we skip this check?" to "what is failing and how do we fix it?" — a significantly more productive conversation.

Gates also make quality requirements explicit and visible. New team members see immediately which checks are required and what the standards are. The CI pipeline is the canonical source of quality requirements — not an internal wiki page that may be out of date.

How others do it

Martin Fowler / ThoughtWorks (CI pipeline quality)

Fowler's Continuous Integration essay (2006, updated) identifies the "quality gate" pattern as the cornerstone of CI effectiveness. His key principle: "the build must always be passing." Not "usually passing" or "passing except for known issues" — always passing. ThoughtWorks' CI consulting practice documents the "deployment pipeline" model: a sequence of increasingly expensive quality checks that must all pass before code reaches production. Each stage gates the next; no stage is advisory.

What we take: The build is always green or it is being fixed immediately. No "yellow" state where some gates pass and some are ignored. A partially-green build is a failing build.

Source: Martin Fowler — Continuous Integration

Google (internal CI culture)

Google's internal CI system (Forge/Blaze) enforces hard-block quality gates on every code change submitted to the monorepo. Google's "submit queue" requires all configured tests to pass before a change is committed — there is no mechanism to bypass a failing test. Google's testing culture, described in Software Engineering at Google, explicitly values test suite trustworthiness: a test suite that can be bypassed is not trusted and eventually not maintained. Their "TAP" (Test Automation Platform) runs millions of tests per day with hard-block semantics.

What we take: Design gates so they cannot be bypassed. If there is a bypass mechanism (even an "emergency" one), it will be used routinely under pressure. Hard gates build trust in the test suite.

Source: Software Engineering at Google

GitHub (required status checks)

GitHub's "required status checks" feature is the implementation mechanism for hard-block CI gates in GitHub repositories. Branch protection rules can require specific CI jobs to pass before a PR can merge. GitHub also supports "required reviews" (human approval) and "required deployments" (the code must be deployed to a specified environment before merging). GitHub's documentation on branch protection is the canonical reference for configuring hard-block gates in a GitHub Actions CI pipeline.

What we take: Use branch protection rules to enforce gates — not CI job configuration alone. Branch protection prevents merge regardless of how CI is configured, closing the bypass gap.

Source: GitHub — Protected branches

Anatomy

Test gate

100% unit test pass rate, contract test pass, coverage ≥80%. Failures include the test name, file, and line number of the assertion that failed.

Lint gate

Spectral (OpenAPI), ESLint (JS), axe-core (a11y). Violations include the rule violated, the file, and the line number. Warnings are surfaced but do not block.

Security gate

SAST scan and dependency audit. High and critical severity findings block merge. Medium findings are surfaced as warnings and tracked as issues.

Visual regression gate

Required only on PRs touching shared CSS or UI-kit. A human must explicitly approve the visual diff before the gate passes.

Common pitfalls

PitfallWhy it happensWhat to do instead
Advisory-only gates Gates report results but allow merge regardless. Developers ignore failing advisory gates under pressure. All gates must be hard blocks. Use branch protection rules to enforce this at the platform level, not just in CI configuration.
Unactionable error messages Gate failures report "tests failed" with no detail on which test failed or why. Developers cannot fix what they cannot identify. Every gate failure must include: what failed, where (file + line), and why (which rule/assertion). Configure CI to surface this in the PR comment or check detail view.
Too many gates, too slow All gates run sequentially and the full pipeline takes 30+ minutes. Developers stop waiting for CI and merge optimistically. Run fast gates (lint, unit tests) in parallel first. Run slow gates (integration tests, visual regression) only when fast gates pass. Aim for fast-gate feedback under 5 minutes.
