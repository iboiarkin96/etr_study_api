Contract Testing — prevent API drift between services

Verifies that consumer expectations and provider capabilities stay aligned without requiring both services to be deployed simultaneously.

TL;DR

Contract testing (Pact, 2013) verifies API agreements independently: the consumer records its expectations as a "pact"; the provider verifies it can fulfil them. Unlike integration tests, contract tests run without both services deployed. Without contract tests, API drift between provider and consumer goes undetected until a broken deployment. Every public endpoint has a Pact or OpenAPI-based contract test; provider CI runs consumer pacts on every PR; any drift fails the build.

What it is

Contract testing is a technique for verifying that two collaborating services (a consumer and a provider) agree on the shape of their API interaction. The consumer records its expectations about the provider's API as a "pact" — a set of interaction examples that describe request shapes and expected response shapes. The provider then verifies independently that it can fulfil every interaction in the pact.

The Pact framework (developed by the DiUS team in Australia in 2013) is the most widely adopted implementation. Pact uses "consumer-driven contracts" — the consumer defines the expectations, not the provider. This means the contract reflects what consumers actually need, not what the provider theoretically offers.

An OpenAPI-based alternative to Pact uses the provider's OpenAPI specification as the contract. Tools like Dredd or Schemathesis verify that the running provider conforms to its own specification. This approach does not require consumer-side pact generation but is less precise about what each specific consumer actually needs.

Contract tests run independently — no shared staging environment is needed. The consumer's CI generates pacts; the provider's CI fetches and verifies them. Any change to the provider API that would break a consumer expectation fails the provider's CI pipeline before the change is merged.

Why we need it

Integration environments are slow, expensive, and fragile. Without contract tests, the only way to verify that a consumer and provider agree on the API is to deploy both services simultaneously and run integration tests. This approach is bottlenecked by environment availability, takes significantly longer than unit or contract tests, and provides feedback only after a complete deployment cycle.

API drift — where the provider changes its response shape without telling consumers — is one of the most common causes of production incidents in microservice architectures. Without contract tests, the drift is invisible until a consumer deploys against the changed provider and fails in staging or production.

How others do it

DiUS / Pact (framework originators)

The Pact framework was invented at DiUS in Melbourne in 2013 to solve the integration testing problem in their microservice architecture. The key innovation was consumer-driven contracts: instead of the provider specifying what it offers, consumers specify what they need. This inverts the traditional API ownership model and makes breaking changes visible before they reach production. Pact now supports 10+ languages and has a hosted broker (PactFlow) with enterprise features including can-i-deploy gates.

What we take: Consumer-driven contracts. The consumer's expectations are the source of truth for the contract, not the provider's documentation.

Source: Pact documentation

Martin Fowler

Fowler's "Consumer-Driven Contracts" essay (2011, written with Ian Robinson) predates Pact but describes the same idea. Fowler argues that the contract should be driven by consumer needs because consumers are the ones who break when the provider changes. He describes the pattern as a formal agreement between services that is version-controlled, automated, and enforced in CI. The essay is required reading for teams building microservice architectures.

What we take: The contract is a formal, versioned agreement — not informal documentation. It is enforced in CI, not in a team meeting.

Source: Fowler — Consumer-Driven Contracts

Atlassian

Atlassian's engineering blog describes their adoption of Pact for their Forge (app platform) ecosystem. They run Pact tests on every PR for all services that expose public APIs. The "can-i-deploy" gate in Pact prevents a provider from deploying when any registered consumer's pact would be broken. Atlassian also uses Pact for third-party integrations — consumers outside their control publish pacts to a shared broker that Atlassian's provider CI verifies against.

What we take: The can-i-deploy gate as the deployment guard. A provider cannot merge if any consumer's contract would be violated.

Source: Atlassian Engineering Blog

Anatomy

Consumer pact generation

Consumer-side tests that record expected request/response pairs into a pact file. These tests use a mock provider, not the real service.

Pact broker

A store for published pact files, with version tracking and a can-i-deploy API. Self-hosted or PactFlow (commercial).

Provider verification

Provider-side tests that fetch pacts from the broker and replay them against the running provider. Run in provider CI on every PR.

Can-i-deploy gate

A CI check that queries the broker: can this version of the provider deploy without breaking any registered consumer? Blocks deployment if the answer is no.

Common pitfalls

PitfallWhy it happensWhat to do instead
Provider-driven contracts The provider writes the contract based on what it offers, not what consumers need. Consumers accept whatever the provider documents. Use consumer-driven contracts. Consumers generate the pact from their actual usage; providers verify against consumer expectations.
Contract tests without can-i-deploy Contract tests run in CI but there is no gate that prevents deployment when a consumer's contract is violated. Implement the can-i-deploy check as a deployment gate. Without it, contract tests provide information but don't prevent the breaking change from reaching production.
Using contract tests instead of integration tests Teams replace all integration testing with contract testing, missing the class of defects that only appear when both services run together (e.g. authentication flows, state mutations). Contract tests and integration tests are complementary. Contract tests verify the shape of interactions; integration tests verify behaviour. Both are needed.
