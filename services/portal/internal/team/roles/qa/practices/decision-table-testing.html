Decision Table Testing — model business rule combinations

Maps all combinations of conditions to their expected actions, making combinatorial business logic visible and testable.

TL;DR

Decision tables map combinations of conditions (inputs, states, flags) to expected actions (outputs, decisions). Each column is one combination — one test case. Decision tables are ideal for business logic with multiple interacting conditions: "if premium user AND active subscription AND feature flag enabled, then show the feature." Without a table, combinatorial logic is tested by intuition, and combinations are missed systematically. Decision tables make the complete combination space visible so nothing is overlooked.

What it is

Decision table testing is a black-box technique that models complex business logic as a table of condition-action pairs. Each column in the table represents one unique combination of conditions — one test case. Each row is either a condition (an input or state variable) or an action (the expected output or behaviour for that combination).

The technique is well-suited for business rules with multiple interacting boolean or small-set conditions. Examples from a typical web application include:

  • Feature access gates: "show feature if premium user AND active subscription AND feature flag enabled"
  • Pricing rules: "apply discount if order total > $100 AND loyalty member AND promo code valid"
  • Validation rules: "reject if both fields empty OR both fields have the same value"

For n binary conditions, a complete decision table has 2^n columns (all possible combinations). In practice, rules often constrain which combinations are possible, reducing the table size. The remaining columns after constraint elimination are the test cases.

Decision tables are created before test execution, typically in the Three Amigos session where developer, QA, and PM review the business rule together. The table makes implicit assumptions explicit — "if the user is premium but the subscription is expired, what happens?" — and these questions are resolved before coding starts.

Why we need it

Business logic with multiple interacting conditions produces a combinatorial space that cannot be managed by intuition. A tester who tries to write test cases by reading the requirement narrative will systematically miss combinations — particularly the combinations that are not explicitly described because the author assumed they were obvious.

Decision tables eliminate this ambiguity. Once the table is built, the complete set of required test cases is visible. Missing combinations are visible as empty columns. Contradictory rules are visible as two columns with the same conditions but different actions.

How others do it

ISTQB

ISTQB's Foundation Level defines decision table testing as one of the four core black-box techniques. The syllabus requires candidates to be able to create a decision table from a given business rule and derive test cases from the table. ISTQB distinguishes between limited-entry tables (conditions are Y/N) and extended-entry tables (conditions can have multiple values). The latter is more powerful for complex business rules with multi-value conditions.

What we take: Use limited-entry tables (Y/N conditions) as the default format — they are faster to create and easier to review in Three Amigos sessions.

Source: ISTQB — Foundation Level

Agile Alliance

The Agile Alliance glossary describes decision tables as a specification-by-example technique well-suited to BDD (Behaviour-Driven Development) workflows. In Cucumber and similar tools, decision tables appear as "scenario outline" tables where each row is a test case. The Agile Alliance recommends creating decision tables during backlog refinement, not after, so that ambiguities in the rule are resolved before implementation rather than discovered during testing.

What we take: Create the decision table in the Three Amigos session, not after. The table is as much a requirement artefact as a test artefact.

Source: Agile Alliance — Decision Tables

Amazon

Amazon's internal testing practices (described in conference talks and engineering blog posts) use decision tables extensively for their pricing, eligibility, and recommendation rule engines. Their rule engines have hundreds of conditions, so they use automated combinatorial testing tools (All-Pairs or pairwise testing) to generate the table rather than creating it manually. The automated approach finds the minimum set of test cases that covers all 2-way combinations of conditions.

What we take: For rules with more than 5 conditions, use a pairwise testing tool to generate the table. Manual creation of 2^5 = 32 column tables is error-prone.

Source: Amazon Science — testing at scale

Anatomy

Condition rows

One row per input condition. Conditions should be independent and atomic — each one is a single yes/no (or small-set) question.

Action rows

One row per expected output or behaviour. Actions are the things the system should do differently depending on the combination of conditions.

Test case columns

One column per unique combination of condition values. Each column is a complete test case — conditions are the inputs, actions are the expected outputs.

Infeasible combination annotation

Columns that represent impossible combinations (e.g. active subscription but no user account) are marked as infeasible and removed from the test set.

Common pitfalls

PitfallWhy it happensWhat to do instead
Incomplete combination coverage The table is created with some columns missing because the tester built it by listing "interesting" combinations instead of all combinations. Start with the full 2^n table and then eliminate infeasible combinations. Never start by picking combinations — you will miss the non-obvious ones.
Mixing conditions and constraints Constraints (business rules that make some combinations impossible) are treated as conditions, producing extra rows and confused columns. Separate conditions (input variables) from constraints (rules that eliminate infeasible combinations). Constraints reduce the column count; they are not additional rows.
Decision table not reviewed with developer QA builds the table from the requirement narrative without developer review, and the table contains assumptions that don't match the implementation. Review the decision table with the developer and PM in the Three Amigos session. The table is a requirement artefact — everyone must agree on it before implementation starts.
