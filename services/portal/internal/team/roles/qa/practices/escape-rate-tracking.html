Escape Rate Tracking — measure defects that reach production

Tracks the fraction of defects found by users after release compared to defects found by QA before release — the primary measure of test effectiveness.

TL;DR

Escape rate = (defects found in production) ÷ (defects found in QA + defects found in production) × 100%. A high escape rate means QA is not catching defects before users do — the most expensive possible outcome. Escape rate is tracked per sprint and per feature team. When escape rate rises, root cause analysis identifies which test layer failed to catch the escaped defect. The target escape rate is <10% for the overall backlog and 0% for Blocker and Critical severity defects.

What it is

Escape rate is the fraction of total defects that were found by users in production rather than by QA before release:

escape rate = (prod defects) ÷ (QA defects + prod defects) × 100%

A 0% escape rate means QA caught every defect before release. A 100% escape rate means QA caught nothing and users found everything. The target for this project is <10% overall, and 0% for Blocker and Critical severity defects — no severity-1 or severity-2 defect should ever reach a user.

Escape rate is tracked at two levels:

  • Per sprint: How many defects escaped in this sprint? Tracked in the quality dashboard and reviewed in the retrospective.
  • Per feature team: Which team's features are producing the most escaped defects? Used to target QA investment — a team with a persistently high escape rate needs more test coverage or earlier QA involvement.

When escape rate rises above the target, root cause analysis identifies which test layer failed: did the unit tests not cover the code path? Was there no integration test for this interaction? Did the QA engineer not have a test case for this scenario? The root cause determines the remediation.

Why we need it

A defect found in production costs 10–100× more to fix than a defect found during development or QA testing. This is not merely a cost multiplier — it also includes user impact: a user who encounters a bug loses trust in the product, may not report it, and may leave. The cost of an escaped defect is not only the engineering time to fix it but also the support time, the user trust, and sometimes the revenue impact.

Without escape rate tracking, teams have no objective measure of QA effectiveness. "We shipped on time" is not a quality metric. Escape rate is the primary measure of whether QA is actually preventing defects from reaching users, not just running tests.

How others do it

DORA (DevOps Research & Assessment)

DORA's "Four Keys" framework for software delivery performance includes "change failure rate" — the fraction of deployments that cause incidents requiring a hotfix or rollback. Change failure rate is a superset of escape rate: it measures defects that cause visible failure, not just all defects that reach production. Elite DORA performers achieve change failure rates of 0–15%. DORA's research shows that escape rate and change failure rate are leading indicators of overall delivery performance — teams with high escape rates also deploy less frequently and recover more slowly from incidents.

What we take: Track escape rate as a DORA-adjacent metric. Low escape rate and low change failure rate are correlated — both require strong test coverage and shift-left testing practices.

Source: DORA — State of DevOps research

IBM (Capers Jones research)

Capers Jones' research at IBM and later as an independent researcher established that the cost of fixing a defect increases by an order of magnitude at each subsequent phase: a defect found in requirements costs $1 to fix; the same defect found in production costs $100–$1000. Jones' "defect removal efficiency" metric — the fraction of latent defects removed before deployment — is the inverse of escape rate. Jones documented that best-in-class software organisations achieve defect removal efficiency above 95%, corresponding to an escape rate below 5%.

What we take: Target <5% escape rate for critical functionality. The <10% overall target is a minimum baseline; the best organisations aim significantly lower.

Source: Capers Jones — Software Quality Research

Netflix (defect metrics culture)

Netflix's engineering culture, as described in their tech blog, tracks "customer-impacting incidents" as their primary quality metric — equivalent to escaped critical defects. Netflix's quality goal is not zero defects (they accept that defects will occur) but rapid detection and recovery: mean time to detect (MTTD) and mean time to recover (MTTR). Their "chaos engineering" practice deliberately introduces failures to improve resilience rather than trying to prevent all failures. However, they track escape rate per team and use it to invest in test infrastructure for teams with high rates.

What we take: Track both escape rate (did we miss a defect?) and MTTD (how quickly did we detect the escaped defect?). Both metrics matter: a team that detects and recovers quickly from escaped defects is more resilient than one that prevents all escapes but takes days to detect an incident.

Source: Netflix Tech Blog

Anatomy

Defect source tagging

Every defect is tagged with where it was found: unit test, integration test, QA testing, staging, or production. This is the data that feeds the escape rate calculation.

Sprint escape rate

Calculated after each sprint: prod defects ÷ (QA defects + prod defects) × 100%. Published in the quality dashboard and reviewed in the retrospective.

Escape root cause

For every escaped defect above Minor severity, a root cause note: which test layer should have caught it and why it did not. Feeds test strategy improvement.

Target and alert

Target <10% overall; 0% for Blocker/Critical. Dashboard alerts when the rate exceeds target for two consecutive sprints.

Common pitfalls

PitfallWhy it happensWhat to do instead
Not tagging defect source Defect source (where found) is not captured in the issue tracker. Escape rate cannot be calculated without this data. Make "found in" a required field on every defect. Options: unit test / QA / staging / production. Without this field, escape rate is unmeasurable.
Gaming the metric Teams file known production defects as "QA defects" to keep escape rate low. The metric looks good; the quality does not improve. Use the production incident log as the source of truth for production defects, not the issue tracker alone. Cross-reference incident postmortems with defect records.
Tracking escape rate without root cause Escape rate is tracked and reported but no root cause analysis is done. The number goes up; nothing changes. Require a root cause note for every escaped defect above Minor. The note identifies which test layer should have caught it. Root causes drive test strategy changes.
