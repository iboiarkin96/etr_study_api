Conventional Commits — machine-readable history

A simple grammar over commit subjects turns the git log into the source of truth for changelogs, SemVer bumps, and release notes — automated and impossible to forget.

TL;DR

The Conventional Commits 1.0 specification (conventionalcommits.org), derived from Angular's commit conventions, defines a small grammar: type(scope): subject with a fixed type allow-list (feat, fix, docs, refactor, chore, test, build, ci, perf, style) and an explicit ! or BREAKING CHANGE: footer for breaking changes. Tools like semantic-release, release-please, and changesets read the log, decide whether the next version is a major / minor / patch bump, and generate the CHANGELOG automatically. The git log itself becomes the release ledger.

What it is

Conventional Commits is a specification (1.0.0, current) for commit message structure. The first line follows the grammar type(scope): subject: type is one of a fixed allow-list (feat for new feature, fix for bug fix, docs, refactor, chore, test, build, ci, perf, style); scope is an optional one-word identifier of the component changed; subject is a short imperative-mood description.

Breaking changes are signalled in one of two ways: an exclamation mark after the type — feat(auth)!: drop password login — or a BREAKING CHANGE: footer in the message body. This single bit of information drives the entire SemVer machinery: a commit with feat implies a minor bump, fix implies a patch bump, and any ! implies a major bump.

The grammar is enforced at commit time (commitlint as a pre-commit hook) and at PR time (PR-title linting). Release tools — semantic-release (Node), release-please (Google), python-semantic-release, or changesets for monorepos — read the log between two tags and produce the next version number, a CHANGELOG entry per category, and the git tag itself. The human's job ends at writing a well-formed commit message; the release ledger writes itself.

Why we need it

Without a commit grammar, the answer to "what changed in 2.3.0?" requires re-reading every diff. Changelog generation is manual — a tedious task that gets skipped or generated by ChatGPT after the fact, both of which produce a release ledger that nobody trusts. Version bumps are guesses: someone judges, often wrongly, whether a release is "feature enough" to warrant a minor or "just bug fixes."

Structured commits flip the entire process. The release tool reads the log and produces a deterministic answer: this set of commits demands a major bump because there is one !; the CHANGELOG entries are these, grouped by type; the next git tag is v2.4.0. The release becomes a build artifact, not a meeting. Downstream consumers can also subscribe to specific change types (e.g. "tell me when any feat(api) commit lands") and build automation on top.

How others do it

Angular / Nx — origin of the convention

Google's Angular team published their commit guidelines around 2014; they became the seed for the Conventional Commits 1.0 spec in 2018. Nx (the monorepo build system) and Angular CLI itself both enforce the grammar in CI. Their release pipeline generates the changelog and bumps SemVer from the log; no human edits a CHANGELOG.md.

What we take: The spec works at very large monorepo scale. The Angular monorepo has tens of thousands of conformant commits; the discipline is sustained by tooling (commitlint + CI) rather than human reviewers reminding each other.

Source: conventionalcommits.org

semantic-release — release automation

semantic-release (originally by Stephan Bönnemann) is the most popular automated-release tool in the JS ecosystem. Given a Conventional Commits log, it computes the next version, writes the CHANGELOG, creates the git tag, publishes to npm, drafts the GitHub release, and notifies a Slack channel. The release process is one CI job; no human is in the critical path.

What we take: The full release pipeline can be derived from the commit log alone. Once you trust the grammar, every step downstream — tag, package publish, release notes, notification — is automation. The team gains hours per release and never publishes the wrong SemVer level.

Source: semantic-release.gitbook.io

GitLab — release-please style flow

GitLab's monorepo, and Google's release-please tool which they use for many open-source projects, take a slightly different approach: instead of releasing on every push, a "release PR" is auto-maintained that contains the next version bump and the proposed CHANGELOG. Engineers see the upcoming release inline and merge the PR when ready. The log grammar still drives the content; only the trigger is different.

What we take: For teams not yet ready for continuous release, release-please's "PR for the next release" pattern is a gentler adoption path. The same commit grammar; a human merge of the release PR replaces the automated tag. You get the changelog automation without committing to every-push releases on day one.

Source: github.com/googleapis/release-please

The repeating pattern: small fixed grammar, enforced at commit-time and PR-time, with release automation that turns the log into the changelog. The spec is intentionally small — it is the smallest grammar that still enables full automation.

Anatomy of a Conventional Commit

Type (allow-list of ~10)

Exactly one of feat / fix / docs / refactor / chore / test / build / ci / perf / style. No new types without an RFC.

Scope (optional, one word)

A documented scope vocabulary — usually 10–30 entries, one per major module. Maintained in commitlint.config; CI rejects unknown scopes.

Subject — imperative, ≤ 72 chars

"add SAML support" not "added SAML support." Lower-case start. No trailing period. Hard line limit so it fits in git log --oneline.

Breaking marker

Either ! after the type — feat(api)!: rename endpoint — or a BREAKING CHANGE: footer with a one-paragraph migration note. Either drives a major SemVer bump.

Body + footers

Optional but encouraged for non-trivial changes. Body explains the "why." Footers carry metadata: Refs: #1234, Co-Authored-By: …, BREAKING CHANGE: ….

commitlint at commit + CI

Hook checks the message locally; CI checks the PR title (because squash-merge uses the PR title as the commit message). Both gates use the same config.

Common pitfalls

PitfallWhy it happensWhat to do instead
Scope vocabulary explodesScopes are free-text. Different engineers invent auth, authn, authentication, login, and users for the same module. The "log grouped by scope" view becomes useless and the CHANGELOG reads like a yard sale.Maintain an explicit scope allow-list in commitlint.config.js. CI rejects unknown scopes. Adding a new scope is a small PR with a rationale. 10–30 scopes is the right size; 100+ means you've lost discipline.
Engineers wrote the real commit message in the PR descriptionThe commit messages are wip / fix / address review; the real explanation lives only in the PR description. When the PR is squash-merged, the GitHub default of "use commit messages" produces garbage history.Use "use PR title and body" as the squash-merge default. Lint the PR title with commitlint. Either ban WIP commits or require interactive rebase before merge. The merged commit message is the artifact that survives; treat it accordingly.
BREAKING CHANGE silently buriedAn engineer marks something as fix when it's actually a breaking change, because they're focused on the bug it fixes rather than the API contract it alters. SemVer bumps a patch level; downstream consumers update and break on Monday morning.Review checklist explicitly asks "does this change any public API or wire format?" If yes, the marker must be ! or BREAKING CHANGE:. Pair with a contract test in CI (see practice 12) so breaking changes can't be miscategorized silently.
