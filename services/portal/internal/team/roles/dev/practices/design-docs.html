Design Docs & RFCs — written before code

Move the architecture conversation upstream — from code review to a one-to-four-page document reviewed before a line is written.

TL;DR

A Design Doc — sometimes called an RFC — is a 1-to-4-page document written before non-trivial code, stating the problem, the proposed solution, the alternatives considered, the trade-offs, and the open questions. Google has used Design Docs internally as the primary engineering design artefact for two decades; Malte Ubl's 2020 essay at industrialempathy.com is the canonical public account. HashiCorp, Rust, Ember, and Python (PEPs) all run public RFC processes on the same principle. A code review is a poor place to debate architecture: by then the engineer has invested days and is committed. A Design Doc moves the debate to the cheapest moment to change direction — before code exists.

What it is

A Design Doc (or RFC — Request for Comments) is a short written document, typically 1–4 pages, that an engineer writes before beginning any non-trivial implementation work. The canonical structure, popularised by Google internally and described publicly by Malte Ubl in Design Docs at Google (2020):

  • Context / Background — what problem are we solving, for whom, what is the current state.
  • Goals and non-goals — what this design will achieve, and explicitly what it will not.
  • Proposed design — the recommended approach, with enough detail that a peer could critique the trade-offs.
  • Alternatives considered — at least two other approaches, why each was rejected. This is non-negotiable: a doc presenting a single option is a proposal, not a design.
  • Trade-offs / risks — what we give up, what could go wrong, what we'll do if it does.
  • Open questions — what is undecided and who needs to weigh in.
  • Decision — recorded once review concludes, with the names of the deciders.

The doc is shared, reviewers comment asynchronously over 48–72 hours, the author responds and revises, and the doc is approved before code begins. Final docs live in a durable location (docs/design/ in the repo, or an internal site indexed by date and project) and survive as institutional memory. The Rust, Ember, Python, and HashiCorp RFC processes are public-facing variants of the same idea — proposals live in version control and are reviewed by the community before any reference implementation lands.

Why we need it

Code review is the wrong place to debate architecture. By the time a reviewer opens the PR, the engineer has invested days or weeks. Suggesting a different decomposition, a different boundary, or a different storage model means asking them to throw away that work — which is emotionally expensive, politically expensive, and frequently doesn't happen even when it would be technically correct. The debate gets deferred, the suboptimal design ships, and the team pays for it for months.

A design doc moves the same conversation upstream by an order of magnitude — from "I have built this thing" to "I am proposing to build this thing". The cost of changing direction at the doc stage is hours of writing; at the code stage it is days of rework. Done at scale (HashiCorp, Rust, Google), the doc process also creates project memory: a year later, when someone asks "why didn't we use X instead?", the answer is in the alternatives section.

How others do it

Google (Design Docs)

Google has used Design Docs internally as the primary engineering design artefact since the company's early years. Malte Ubl's 2020 essay Design Docs at Google at industrialempathy.com is the canonical public account: docs are short (1–4 pages), include alternatives, get async review over a few days, and are then stored permanently. Larger projects might have a tree of related docs; tiny changes skip the doc entirely. The cultural force is strong enough that "do you have a design doc?" is the first question on any non-trivial proposal.

What we take: Short and async beats long and synchronous. A 2-page doc reviewed asynchronously catches 90% of the problems a 20-page doc reviewed in a 2-hour meeting catches, at 10% of the cost.

Source: industrialempathy.com — Design Docs at Google (Ubl)

HashiCorp (engineering RFCs)

HashiCorp's engineering RFC process is documented externally (most notably by Mitchell Hashimoto on his blog and Gergely Orosz's Pragmatic Engineer write-up): every cross-team or risky engineering proposal goes through a written RFC that is reviewed asynchronously over a fixed window. The format is deliberate — title, summary, motivation, design, drawbacks, alternatives — and the process is the same whether the author is an engineer, an EM, or the CEO.

What we take: Make the RFC process uniform across seniority. When the CEO and a new hire use the same template and the same review window, the conversation is about ideas, not about authority.

Source: Pragmatic Engineer — Scaling engineering teams via RFCs

Rust (rust-lang/rfcs)

The Rust language's RFC process is run entirely in public on GitHub at rust-lang/rfcs. Every language change, every standard-library addition, every governance change is a numbered, version-controlled RFC. The template is opinionated: Summary, Motivation, Guide-level explanation, Reference-level explanation, Drawbacks, Rationale and alternatives, Prior art, Unresolved questions. Drawbacks and alternatives are required sections — an RFC without them is not eligible for merge. The result is a public archive of every design decision in Rust's history, with a record of why each was chosen and what was rejected.

What we take: The "Drawbacks" and "Alternatives" sections must be required. A template that lets the author skip them produces proposals (single-option pitches), not designs (option comparisons).

Source: github.com/rust-lang/rfcs — public RFC process

The pattern: short, async-reviewed, template-enforced, with alternatives and drawbacks as mandatory sections. The output is both a decision and a durable record of why.

Anatomy of a well-formed design doc

Length: 1–4 pages

Long enough to capture the design, short enough to be read in one sitting. If it exceeds 4 pages, the proposal is probably two designs and should be split. Reviewers' attention is the scarce resource — protect it ruthlessly.

Mandatory "Alternatives considered"

At least two genuine alternatives, each with one paragraph on why it was rejected. A doc with no alternatives is not a design — it is a pitch. The template enforces this; reviewers refuse to approve docs without it.

Async review window: 48–72 hours

The author shares the doc, reviewers comment asynchronously over a fixed window (typically two to three working days). The author responds to all substantive comments inline. No live meeting unless the comment thread is gridlocked.

Recorded decision and deciders

Once review concludes, the doc records the final decision and the names of the people who approved it. Useful both for accountability (who owns this if it goes wrong?) and for future readers (why did we end up here?).

Durable storage in version control

Final docs live in docs/design/ in the repo (or in a versioned design-doc site), numbered or dated, never in a one-off Google Doc that gets orphaned. The doc must be discoverable a year later by an engineer who wasn't on the team when it was written.

Threshold for "needs a doc"

A clear, simple rule — e.g. "any work estimated > 5 engineer-days, any cross-team interface change, any change to a security or data-handling boundary". The rule is written down and applied consistently so the doc requirement is predictable, not arbitrary.

Common pitfalls

Pitfall Why it happens What to do instead
Doc reviewed once, no follow-up after code The design doc is reviewed, approved, and shelved. Implementation diverges from the design in small steps over weeks. By the time the code ships, half of the decisions in the doc are no longer accurate — but nobody updates the doc, so future readers see a fiction. Add a "doc-vs-implementation" review at the end of the project: the author updates the doc to reflect what actually shipped, including a short note on any deliberate divergence. The doc becomes a record of the system, not just a record of the original plan.
Doc written AFTER the code — documentation theatre An engineer ships the feature, then writes a design doc retroactively to satisfy a process requirement. The doc is a description of what was built, with all alternatives invented in hindsight to make the chosen approach look obvious. Reviewers go through the motions but cannot meaningfully change direction. Require the doc to be written before code begins, and gate the work on doc approval. A doc written after code is not a design doc — it is a write-up. The two have different value (the latter is documentation, not architecture review) and should be treated differently.
No alternatives section — single decision presented The author writes the doc as a pitch: "here is what I propose, here is why it's great, please approve". Without alternatives, reviewers cannot meaningfully compare; they can only nod or reject the entire approach. The template lets the author skip the section, so they do. Make the "Alternatives considered" section structurally required: the template includes it with placeholder text the author must replace. Reviewers refuse to approve docs without two real alternatives with one-paragraph trade-off analyses each.
