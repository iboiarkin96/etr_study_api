DDD tactical patterns — aggregates, value objects, services

Concentrate business invariants in aggregate roots; make identifierless things immutable value objects; keep persistence behind repositories.

TL;DR

Tactical Domain-Driven Design is the building-block vocabulary from Eric Evans's 2003 book Domain-Driven Design, refined and made concrete by Vaughn Vernon in Implementing DDD (2013). Every state-changing operation goes through an Aggregate root, which owns a small consistency boundary and its invariants. Identifierless concepts (Money, Address, EmailAddress) become immutable Value Objects. Cross-cutting domain operations that don't belong to one aggregate become Domain Services. Persistence is hidden behind Repositories. Done well, the rules of one consistency unit live in one place; done poorly, business logic is sprinkled across controllers, ORMs, and helpers and every change touches all three.

What it is

Tactical DDD is the set of code-level building blocks that Eric Evans introduced in Domain-Driven Design (2003) and that Vaughn Vernon made concrete in Implementing Domain-Driven Design (2013). The patterns most teams use day-to-day:

  • Entity — an object with an identity that persists over time (a User, an Order). Two Users with the same fields but different IDs are not equal.
  • Value Object — an object defined entirely by its attributes (Money, Address, EmailAddress, DateRange). No ID. Two Moneys with the same amount and currency are equal. Always immutable.
  • Aggregate — a cluster of entities and value objects treated as a single consistency unit. Has exactly one aggregate root entity, and all external access goes through it. The root enforces invariants (an Order's total equals the sum of its lines).
  • Repository — the persistence boundary for an aggregate. OrderRepository.save(order) persists the whole aggregate; findById(id) reconstitutes it. The domain code never sees SQL or an ORM session.
  • Domain Service — a piece of domain logic that doesn't naturally belong to one entity. A PricingService that consults a catalog and tax rules to compute a line price.
  • Domain Event — an immutable record that something meaningful happened (OrderPlaced, PaymentRefunded). Used to communicate between aggregates and bounded contexts without coupling.

The discipline: changes to an aggregate's state happen only through methods on the root, only within a single transaction, and never reach across aggregates synchronously. Cross-aggregate coordination uses domain events (eventually consistent) or domain services that orchestrate but don't own state.

Why we need it

Without tactical DDD, business logic spreads across three layers: a thin controller that "validates input", an ORM with computed columns and triggers, and a pile of helper modules with names like order_utils.py. Any change to an invariant — "an order with status SHIPPED cannot have lines added" — touches all three, and the only way to know whether the invariant actually holds is to run the system and try. The model is anaemic: data lives in one place, behaviour in another, and neither can be reasoned about alone.

Aggregates fix this by concentrating the rules of one consistency unit in one place — the root. The invariants are obvious to a reader (they are methods on the root with English names), the test surface is small (a unit test of the root, no database), and accidentally breaking an invariant becomes much harder because there is no other entry point. The cost is upfront design effort; the return is that subsequent changes are local instead of cross-cutting.

How others do it

Eric Evans / DDD community

Eric Evans's Domain-Driven Design (2003) introduced the vocabulary. The DDD Reference at domainlanguage.com/ddd/reference/ is the short, free, authoritative summary maintained by Evans himself. DDD Europe and the DDD Crew community (ddd-crew.github.io) keep the practice alive with workshops, free guides, and the EventStorming and Context Mapping facilitation methods that feed strategic DDD into tactical implementation.

What we take: Start with strategic DDD (bounded contexts, ubiquitous language) before tactical. Aggregates only make sense inside a bounded context with a clear language; applied to a generic CRUD service they are overhead.

Source: domainlanguage.com — DDD Reference (Evans)

Microsoft eShopOnContainers

Microsoft's "Architecting Cloud Native .NET Applications" guide and the open-source eShopOnContainers reference application implement tactical DDD inside each microservice: aggregates, value objects, domain events, integration events, and CQRS where reads diverge from writes. The Ordering service is the canonical example — an Order aggregate root with OrderItem entities, Address value objects, and a BuyerRepository. The official docs map every Evans/Vernon concept to a concrete C# pattern.

What we take: Reference architectures are most useful as a vocabulary, not a copy-paste. Read eShopOnContainers to see what each pattern looks like in real code; don't import the whole microservice template into a monolith.

Source: Microsoft Learn — DDD & CQRS patterns

DDD Crew (open community)

The DDD Crew, a volunteer community of practitioners, maintains free, version-controlled resources for both strategic and tactical DDD: Context Mapping templates, the Bounded Context Canvas, EventStorming cheat-sheets, and an aggregate design canvas. Teams across industry (ING, Salesforce, Spotify, smaller startups) cite the Crew's materials as their onboarding path into DDD.

What we take: Use the Aggregate Design Canvas before writing the first class. Thirty minutes of canvas work — name, invariants, commands, events, throughput — eliminates 80% of the "this aggregate is wrong" rewrites that happen later.

Source: github.com/ddd-crew — community resources

The pattern: strategic before tactical, aggregate design before code, and treating the patterns as a vocabulary for thinking — not a checklist of classes to create.

Anatomy of a well-formed aggregate

Single aggregate root

Exactly one entity is the root. All external code references the aggregate by the root's ID — never by a child's. The root is the only thing the repository returns. If two roots want to talk, they exchange IDs and events, not object references.

Invariants enforced in the root

Every state-changing method on the root checks the rules before committing the change: order.addLine(line) rejects the call if order.status == SHIPPED. Invariants are not a setter + a service validation — they live on the root and cannot be bypassed.

Value objects are immutable

A Money has no setters. price.add(tax) returns a new Money. Equality is by value, not by reference. Enforced in code by frozen-dataclass / readonly fields / language constructs (records in Java/C#, dataclasses(frozen=True) in Python, struct in Rust).

Repository per aggregate

One repository per aggregate root, not one per table. OrderRepository persists the Order plus all its OrderLines plus its Addresses as a single transactional unit. The domain layer sees repo.save(order); the SQL is hidden behind the boundary.

Aggregate size: rule of one transaction

If two pieces of state must change atomically, they belong in the same aggregate. If they can be eventually consistent, they belong in different aggregates communicating via events. Target: 1–7 entities per aggregate. A "god aggregate" of 30+ entities is a smell — split it along an event boundary.

Domain events on commit

When an aggregate transitions state, it appends a domain event (OrderPlaced) to a list. The repository's save flushes both the state change and the events in one transaction. Downstream aggregates and bounded contexts consume the events asynchronously.

Common pitfalls

Pitfall Why it happens What to do instead
Anaemic domain model — all logic lives in services The entities are data bags (getters, setters, nothing else) and every rule lives in OrderService.shipOrder(). The "domain layer" is in name only — the model is structurally identical to a CRUD app, with an indirection layer added. Move state-changing logic onto the aggregate root as methods with domain names: order.ship(), not orderService.markAsShipped(order). Reserve domain services for genuinely cross-cutting operations (pricing across a catalog, transfers between two accounts).
Aggregate too big — becomes a "god root" The team adds every related entity to one aggregate because "they're related". A Customer aggregate ends up owning orders, invoices, support tickets, and preferences. Concurrency conflicts skyrocket; locks are held across operations that have no real consistency requirement. Re-ask the question "what must change in one transaction?". A Customer's email and address must change atomically — same aggregate. A Customer's order history can be eventually consistent — separate aggregate, linked by customerId and communicating via events.
Leaking persistence into the domain The aggregate root has JPA @OneToMany annotations, SQLAlchemy relationship() calls, or a save() method that calls the database directly. The domain layer and the ORM are now the same object; changes to one ripple into the other, and unit tests need a database. Keep the aggregate POJO/dataclass/struct. The repository is the only thing that knows about ORM and SQL. Domain code is unit-testable without a database; persistence is integration-tested separately.
