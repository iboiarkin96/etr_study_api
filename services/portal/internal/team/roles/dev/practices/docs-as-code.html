Docs-as-Code — docs live next to the code they describe

Documentation in the repo, reviewed in the same PR as the code, built by CI, deployed on every merge.

TL;DR

Docs-as-Code — championed by Anne Gentle and the "Write the Docs" community at writethedocs.org — treats documentation with the same workflow as source code: plain-text files in the repo, PR review, CI gates, versioning by Git tag. The whole GitLab handbook (about-gitlab-com/handbook), Stripe's API docs (stripe.com/docs), and Twilio's developer site run this way. When the doc and the code change in one PR, drift becomes structurally impossible.

What it is

Docs-as-Code is a working agreement, not a tool. It demands four things:

  • Plain text in the repo. Markdown, MDX, reST, or AsciiDoc. Never a binary doc, never a CMS that owns the text.
  • Reviewed in PRs. The same workflow that gates a code change gates a doc change. Reviewers comment inline; merges require approval.
  • Validated in CI. Linters (Vale, markdownlint), link checkers (Lychee, htmlproofer), and accessibility checkers run on every PR. Failures block merge.
  • Versioned with the code. A Git tag at v2.4.0 produces a v2.4.0 docs build. Old versions remain readable; readers can pin docs to the version they run.

The discipline is the point: once you cannot ship code without touching the docs, drift becomes a build failure instead of a slow rot. The static-site generator (Docusaurus, MkDocs Material, Astro Starlight) is an implementation detail; the workflow is the practice.

Why we need it

A parallel CMS (Confluence, Notion) is convenient until someone changes the API and forgets the wiki page. Then the wiki tells the integrator something the code no longer does, and every reader silently absorbs the lie. The team's options at that point are equally bad: update two systems forever, or stop trusting the wiki. Most teams choose the third option — stop reading the wiki — and the doc becomes write-only.

Putting docs in the repo eliminates the choice. The PR for the API change also touches the doc; reviewers see the diff in one place; CI runs the same linters and link-checks on both. The cost is real (no WYSIWYG, no one-click typo fix from the PM), but the benefit — docs that match the code by construction — is the highest-leverage information practice an engineering org can adopt.

How others do it

GitLab — the entire handbook in Git

GitLab's company handbook — 3,000+ pages covering everything from engineering practice to compensation policy — lives in a public Git repo (gitlab-com/www-gitlab-com). Every change is a merge request reviewed by the owning team. CI checks links, style, and structure. It is the company's operating system, rendered as a website.

What we take: The handbook is what survives executive turnover. Treating it as code — branchable, diffable, blame-able — is how the institutional memory remains queryable instead of becoming folklore.

Source: about.gitlab.com/handbook

Stripe — generated from the same OpenAPI spec

Stripe's public API docs at docs.stripe.com are generated from the same OpenAPI source that produces the SDKs in seven languages. A new field added to the spec appears in the docs and the typed SDKs in the same release. CI gates the spec on lint, lint, and semantic-version-bump rules.

What we take: When reference docs are generated from a spec, the doc-vs-code drift class of bug is impossible. The hand-written prose (guides, tutorials) lives in the repo alongside; only narrative content needs human review.

Source: stripe.com/docs

Write the Docs — the community canon

Anne Gentle, Eric Holscher, and the Write the Docs community formalised the term "docs-as-code" around 2014, building on the practice already common in Python and Linux kernel projects. The community runs an annual conference and maintains a guide at writethedocs.org with patterns, tooling comparisons, and case studies from dozens of teams.

What we take: The patterns are not new — the OSS world has lived this way for two decades. Adopting docs-as-code is borrowing a mature playbook, not inventing one. Read the Write the Docs guide before reinventing tooling.

Source: writethedocs.org — Docs as code guide

The shared pattern: docs in version control, PR review as the quality gate, CI as the drift guard, generated content where a spec exists. Teams that diverge — keeping a parallel CMS "for the marketing pages" — see the marketing pages quietly become the canonical reference, and the engineering docs become orphaned.

Anatomy of a docs-as-code setup

Plain-text source in the repo

Markdown / MDX / reST / AsciiDoc. No binary docs. A Confluence link in a README is a code-smell.

Same PR for code + doc

Reviewers reject mismatched PRs ("you changed the endpoint, where's the doc update?"). Encoded in a CODEOWNERS rule that requires a docs-folder approval whenever a public-API file changes.

CI gates

Linter (Vale for prose style, markdownlint for structure), link checker (Lychee), accessibility check. Pipeline budget < 3 minutes; failures block merge.

Single static-site generator

Docusaurus, MkDocs Material, or Astro Starlight. Pick one. Three competing generators in one org is how docs-as-code dies.

Versioning by Git tag

A docs build per release tag; readers can pin docs to v2.4. Old versions stay readable for at least 12 months past EOL.

Generated reference content

API tables, config schemas, CLI flags — generated from the source spec (OpenAPI, JSON Schema, --help). Hand-written only where prose adds something a spec cannot.

Common pitfalls

PitfallWhy it happensWhat to do instead
Docs are in the repo but never reviewed in PR"It's just a typo, ship it." Doc folder has no CODEOWNERS rule, so PRs touching docs/ auto-merge with the code reviewers' rubber-stamp.Add a CODEOWNERS entry that names the doc maintainer for every docs/**/*.md. Make doc approval a required check on PRs that touch any docs file. Drift returns within a quarter without this gate.
Link-checker broken in CI; dead links accumulateLychee flagged 14 dead links last sprint; somebody marked it continue-on-error: true "until we clean up" and the cleanup never happened. Every external link slowly rots; readers click into 404s.Treat link-check failures as P2 bugs, not warnings. Budget a quarterly "link-rot day" where the team fixes whatever has accumulated. Cache external link results for 24h so the build stays fast but the signal stays accurate.
Vale rules so strict everyone disables themAn over-eager initial Vale config flags every passive voice, every simply, every word from a 200-line banned list. Authors add <!-- vale off --> at the top of every file; the linter has zero coverage.Start with five rules, not fifty. Roll out style rules one at a time; each new rule starts as a warning for a sprint, then becomes blocking. Authors should never feel forced to globally disable the linter — that signals the rule is wrong, not the writing.
