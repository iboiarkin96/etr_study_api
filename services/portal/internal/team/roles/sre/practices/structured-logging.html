Structured Logging — machine-readable events, not prose for humans

All log lines emitted as consistent JSON with trace correlation fields — so engineers can query, join, and pivot from metric to log to trace in one operation rather than grepping through free-form text.

TL;DR

Structured logging means every log line is a machine-parseable JSON object with a consistent schema: timestamp, level, service, trace_id, span_id, message, and domain-specific fields. No free-form printf strings. Every log line carries the same trace ID as the distributed trace for that request, enabling engineers to pivot from a metric anomaly → log events → distributed trace in seconds. That pivot is what converts log search from "find a needle in a haystack" into a structured query.

What it is

Structured logging means every log event is a JSON object with a canonical schema. The minimum required fields are:

  • timestamp — ISO 8601 with millisecond precision.
  • level — debug, info, warn, error.
  • service — the emitting service name (from environment config, not hardcoded).
  • trace_id — the OpenTelemetry trace ID for the current request, propagated from the HTTP/gRPC header.
  • span_id — the current span ID within the trace.
  • message — a human-readable description of the event (not the only field; domain fields carry the data).

Domain-specific fields (user_id, order_id, error_code, latency_ms) are added per event type. The schema is enforced by the shared logging library — services don't construct log lines by hand.

Why we need it

Unstructured logs are text for humans. Structured logs are events for machines. The log aggregator (Loki, OpenSearch, BigQuery) can filter, aggregate, and join structured events in milliseconds. When every log line carries the same trace ID as the distributed trace for that request, engineers can pivot from a metric anomaly to every log event in the affected request path in one query. That pivot converts incident diagnosis from hours of log grepping into a 2-minute structured query.

The trace_id field is the key connection: it ties together a distributed trace, all log events from all services involved in that request, and the metric data points recorded during that request window. Without it, logs are silos — with it, they're a join table across the entire observability stack.

How others do it

Datadog

Datadog's platform is built around unified log, metric, and trace correlation through a shared trace_id. Their recommendation: use a single logging library per language (pino for Node.js, structlog for Python, zerolog for Go) that automatically injects the OpenTelemetry trace context into every log line. The library configuration is set once at application startup; individual log calls don't need to pass context manually.

What we take: Automatic trace context injection via logging library middleware is the only scalable approach. Requiring developers to pass trace context manually on every log call produces inconsistent coverage — some log lines have trace IDs, some don't, and the query is useless for the ones that don't.

Source: Datadog — Log Collection

Cloudflare

Cloudflare's observability team enforces structured logging through CI lint rules: any log statement that doesn't call the canonical structured logging function fails the build. Their schema is versioned — when the schema changes, all services are updated in a coordinated migration, not ad hoc. Schema version is a field in every log event, enabling the aggregator to handle schema migrations without breaking dashboards.

What we take: Enforce the schema in CI, not in code review. Code review catches some violations; CI catches all of them. Schema versioning is worth the investment — it prevents the "we changed the field name and broke all the dashboards" incident.

Source: Cloudflare Blog — Engineering

Anatomy

Canonical schema and shared library

One logging library per language, pre-configured with the canonical schema. Services call the library; the library produces compliant JSON. No service constructs log lines by hand.

Automatic trace context injection

The logging library reads the OpenTelemetry context from the current goroutine/thread and injects trace_id and span_id into every log line automatically. Zero manual plumbing per log call.

CI schema enforcement

A linter that fails the build if a log call doesn't go through the canonical library. Catches violations before they reach production and produce un-queryable log events.

Retention and routing policy

Log routing rules: debug logs discarded in production (or sampled at 1%), info logs retained 14 days, error logs retained 90 days. Route to the central aggregator with the retention policy configured — not ad hoc per service.
