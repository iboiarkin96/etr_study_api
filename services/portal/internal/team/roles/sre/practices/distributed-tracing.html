Distributed Tracing — follow one request across every service it touches

A trace_id propagated across every service in a request path, with spans that record latency and context per service, forms the Gantt chart that answers "where is the time going?" for any individual request.

TL;DR

Distributed tracing propagates a trace_id across every service in a request path. Each service emits spans — records of work done within that service, with start time, duration, status, and context. The collected spans form a Gantt-style trace showing exactly where time is spent across the distributed call graph. Implemented via OpenTelemetry with Tempo, Jaeger, or a cloud provider backend. Sampling: 5–10% for normal traffic, 100% for errors. Traces enable the pivot from "the P99 is slow" to "this specific DB query in this specific service at this specific percentile is the bottleneck."

What it is

Distributed tracing works by propagating a trace_id in the headers of every request between services. Each service that receives the request creates a span — a record of the work it did — with:

  • trace_id: Shared across all spans for the same request.
  • span_id: Unique to this service's contribution.
  • parent_span_id: The span that called this service, enabling the Gantt tree structure.
  • start_time and duration: When the work started and how long it took.
  • status: Success or error, with an error message if applicable.
  • attributes: Service-specific context — SQL query text, HTTP endpoint, user_id.

OpenTelemetry is the vendor-neutral standard for trace instrumentation. It provides auto-instrumentation libraries for all major languages and frameworks that instrument HTTP, gRPC, database drivers, and message queues without code changes.

Why we need it

Metrics tell you the aggregate is slow. Logs tell you something went wrong at a specific line of code. Traces tell you which service in a 15-service call graph is the bottleneck for this specific request, at this specific percentile, right now. Without tracing, performance investigations require custom logging patches, redeployments, and cross-team coordination that takes days. With tracing, the bottleneck is identified in one query in seconds.

Traces also surface the "hidden cost" of distributed systems — N+1 database queries that look fast individually but add up to 2 seconds in aggregate, retry storms that are invisible in per-service metrics, and cold-start latency in serverless functions that looks like a random P99 spike.

How others do it

Uber

Uber built Jaeger (now a CNCF project) to address distributed tracing at scale — thousands of services, millions of traces per second. Key design decisions: adaptive sampling (sampling rate adjusts based on traffic volume to ensure rare error paths are always sampled), tail-based sampling for anomalies (collect the full trace for any request that breaches a latency threshold), and trace search by service, operation, tag, and duration range.

What we take: Sample errors at 100% even when sampling normal traffic at 5%. Errors are the traces you most need, and they're the rarest. Dropping errors in the sampler defeats the primary value of tracing.

Source: Jaeger Distributed Tracing

Shopify

Shopify's observability team builds trace-to-log correlation as a first-class feature: every log line in the central aggregator (Loki) carries the trace_id, and every Grafana dashboard has a "Logs for this trace" button that opens the log query pre-filtered by the trace_id. Engineers never need to copy a trace_id and paste it into a log query — the correlation is automatic.

What we take: Build the trace→log→metric pivot into the dashboard as a workflow, not just a possibility. The correlation is only useful if it's one click, not a manual copy-paste operation that most engineers skip because it's too slow.

Source: Shopify Engineering Blog

Anatomy

OpenTelemetry auto-instrumentation

Instrument the HTTP client / gRPC interceptor layer in the shared library once. Every service that uses the shared library gets automatic span generation and context propagation. Individual service teams add manual spans only for custom operations that auto-instrumentation doesn't cover.

Sampling strategy

5–10% head-based sampling for normal traffic. 100% tail-based sampling for errors (any request that returns a 5xx or breaches the SLO latency threshold). The error sample is the most valuable — never drop it.

Trace-to-log correlation

trace_id and span_id injected into every log line automatically (via the logging library middleware). Grafana panels link from span events to the log stream filtered by trace_id. One-click pivot from trace to logs.

DoD requirement

Tracing is a Definition of Done criterion for every new service before it reaches production. A service that generates no traces is invisible in the call graph. No traces = no observability for that service's contribution to latency.

Common pitfalls

PitfallWhy it happensWhat to do instead
Context propagation gaps break the traceOne service doesn't propagate the trace_id to its outbound calls.Use auto-instrumentation middleware for all HTTP/gRPC clients. Manual propagation is unreliable — developers forget, and a trace that breaks in the middle is useless. Auto-instrumentation covers 95% of cases.
100% sampling rate in production"We want to see everything."100% sampling at production traffic volumes produces storage and CPU costs that are prohibitive. 5–10% sampling for normal traffic with 100% for errors gives excellent coverage at manageable cost.
Tracing only on the happy pathInstrumentation is added where it's easiest, not where it's most needed.Ensure all error paths, retries, and timeout handlers are instrumented. The most valuable traces are for the failures — if the error path isn't instrumented, the trace disappears exactly when you need it.
