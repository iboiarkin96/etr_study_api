Event contract template

One contract per event — domain event (another service reacts) or analytics event (a metric consumes it). An event is a wire schema exactly like a response body: named, versioned, owned, and tested for continuity. 1 · Trigger — when exactly it fires

2 · Delivery semantics

  • Guarantee: at-most-once | at-least-once (consumers must dedup by event_id) | exactly-once (name the mechanism).
  • Ordering: guaranteed per <key> | not guaranteed — consumers must not assume order.
  • Emit failure: what happens to the triggering request when the emit fails — blocks | logged and dropped | parked in outbox for retry.

<field><type><yes/no>PII | user content | behavioural | operational<meaning; data classes per data lifecycle>

PII rule: analytics events carry identifiers (client_uuid), never direct PII (full_name, free-text content). If a consumer needs PII, it joins at read time under its own access control — the event stream must stay safe to fan out.

4 · Versioning & compatibility

  • Additive-only within a version. New optional fields do not bump schema_version; removing or renaming a field does — a rename is a removal, same rule as the API schema contract.
  • Consumers tolerate unknown fields. Non-negotiable; it is what makes additive change safe.
  • Version bumps are announced in the producing service's changelog, with a deprecation window for the old version.

5 · Observability

  • Emit metric: events_emitted_total{event="<name>"} — a silent producer is indistinguishable from a broken one.
