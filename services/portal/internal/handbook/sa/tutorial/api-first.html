Design your first endpoint API-first

Install the toolchain, author one OpenAPI fragment, validate it, exercise it against the dev FastAPI, and leave the working tree ready for a spec PR. The point is to feel the API-first loop end-to-end — contract first, handler second — not to ship a real endpoint. Real work uses the same shape; you'll repeat it on every new operation under ADR 0036.

⚙️ 7 phases ⏱ 35–50 min 📚 First-time 💻 Local stack · Python only

–Before you start

  • Python 3.11+ installed (python3 --version reports 3.11 or newer).
  • make available on your shell.
  • You are at the repo root — pwd ends with /study_app. Every command in this tutorial is run from there.
  • curl or any HTTP client for Phase 4. jq is nice but not required.

The toolchain is single-language Python. Everything you install lives inside one .venv/ at the repo root.

0Goal

By the end of this tutorial you will have:

  • A working Python .venv/ with openapi_spec_validator, Connexion + Flask, and the rest of the API requirements.
  • One OpenAPI 3.1 fragment authored at services/portal/internal/services/api/openapi/test/fragments/course/createCourse.yaml.
  • A green make doc-api-swagger-check FILE=course/createCourse.yaml — the validator approves the fragment and resolves every $ref.
  • The fragment rendered as a live Swagger UI on the portal's static server.
  • A running Connexion mock on :8001 that answers curl POST /api/v1/courses with the documented example.
  • A portal Operation spec page stub at the canonical reference path, ready to be filled out.
  • A clean working tree summary — what is staged, what is not, what would land in a spec PR.

Three local processes by the end of Phase 5 (each in its own terminal):

  • :8080 — portal static server, started by make -C services/portal serve. Serves Swagger UI, fragment YAML, the merged spec. GET-only (POST returns 501 Unsupported method).
  • :8001, :8002, … — Connexion mocks, all started at once by make api-mock. One process per canon under openapi/*/: the smoke test canon lives on :8001, product canons (etr_study_app, …) on :8002+ in alphabetical order. Each accepts every documented HTTP method and returns the documented 2xx example.
  • :8000 — optional, FastAPI dev server, started by make run. The real runtime; today it doesn't have handlers for these smoke-test endpoints (Phase 5b shows the parity gate tripping on that).

You will not commit anything in this tutorial. Per the project convention, commits happen only when the author explicitly approves the spec PR.

1Bootstrap the Python toolchain

The repo ships one requirements.txt that pins every Python dep you need, including openapi_spec_validator, jsonschema-path, schemathesis's prerequisites, and FastAPI itself. make setup wires that up in one step.

  1. From the repo root, create the venv and install dependencies:
make setup
  1. Confirm the validator landed inside the venv:
.venv/bin/python -c "import openapi_spec_validator; print(openapi_spec_validator.__version__)"

✓ Check your progress

The command prints a version string (e.g. 0.9.0). If you see ModuleNotFoundError, make setup did not finish — re-run it from the repo root and watch for the line that installs services/api/requirements.txt. If python3 reports a version below 3.11, install a newer Python first (e.g. via pyenv) and re-run make setup.

2Author the OpenAPI fragment

One fragment describes one operation. The path is fixed by convention (ADR 0036 D1): fragments/{resource}/{operationId}.yaml under the canonical tree. Shared schemas, headers, and error responses live next to it under _shared/ and are referenced via relative $ref.

  1. Create the resource directory (skip if it already exists):
mkdir -p services/portal/internal/services/api/openapi/test/fragments/course
  1. Author the fragment at services/portal/internal/services/api/openapi/test/fragments/course/createCourse.yaml with the body below. Copy verbatim for the first pass — change names later when you adapt this to a real endpoint.
openapi: 3.1.0
info:
  title: Study App API
  version: v1
paths:
  /api/v1/courses:
    post:
      operationId: createCourse
      summary: Create a new course.
      tags: [Course]
      requestBody:
        required: true
        content:
          application/json:
            schema: { $ref: "#/components/schemas/CreateCourseRequest" }
            examples:
              minimal: { value: { name: "Introduction to Python" } }
      responses:
        "201":
          description: Created.
          content:
            application/json:
              schema: { $ref: "#/components/schemas/Course" }
              examples:
                minimal:
                  { value: { id: c_abc123, name: "Introduction to Python" } }
        "400": { $ref: "../../_shared/responses/Error400.yaml" }
components:
  schemas:
    CreateCourseRequest:
      type: object
      required: [name]
      properties:
        name: { type: string, minLength: 1, maxLength: 80 }
    Course:
      type: object
      required: [id, name]
      properties:
        id: { type: string, pattern: "^c_[a-z0-9]+$" }
        name: { type: string }

Three details to notice in the shape:

  • operationId: createCourse sits on the method-level object (post:), not the path-level object. It must be unique across the whole API and is what the FastAPI route name should match.
  • $ref: "../../_shared/responses/Error400.yaml" uses a path relative to the fragment file. The validator we'll run in Phase 3 resolves it from this fragment's directory, so the count of ../ matters — two levels up from fragments/course/ lands on openapi/, where _shared/ lives.
  • Every documented response carries an examples block (or $refs a response that does). Those examples double as fixtures for the contract tests later.

✓ Check your progress

ls services/portal/internal/services/api/openapi/test/fragments/course/createCourse.yaml shows the file. Opening the file, you can read off the operation: POST /api/v1/courses, request needs a name (1–80 chars), success returns 201 with an id matching ^c_[a-z0-9]+$, validation errors return 400. If ls says no such file, you saved the fragment at the wrong path — most often services/api/openapi/... (the old location). The canonical path lives under services/portal/internal/services/api/openapi/test/fragments/.

3Validate the fragment

The fragment lints through tools/governance/validate_openapi.py, which wraps openapi_spec_validator and resolves relative $ref from the fragment's directory. You invoke it through a Make target so pre-commit and CI run the same command.

  1. Validate this one fragment by short name (the target resolves it under the canonical fragments tree):
make doc-api-swagger-check FILE=course/createCourse.yaml
  1. Validate every fragment under course/ at once (useful once you've authored more than one):
make doc-api-swagger-check-all DIR=course
  1. Validate the whole fragments tree (the gate CI runs):
make doc-api-swagger-check-all

✓ Check your progress

Each command ends with a green == DOC-API-SWAGGER-CHECK[-ALL]: SUCCESS == banner and a per-file ✅ Valid: …createCourse.yaml line. If you see Unresolvable: ../../_shared/responses/Error400.yaml, the $ref path is wrong — count directories from the fragment up to fragments/ and adjust the number of ../. If you see 'paths' is a required property, your top-level keys are mis-indented; YAML cares. If you see missing required property operationId, you put operationId at path level instead of method level — move it under post:.

4See your design in the browser (Swagger UI)

Validation tells you the fragment is well-formed. To see the API as a human reviewer will read it — endpoints, parameters, schemas, examples, response shapes — open the fragment in Swagger UI. The portal ships a small preview page that loads any fragment by URL query param. It runs entirely off your local static server; no Node.js, no separate process.

  1. From the repo root, start the portal static server (in its own terminal):
make -C services/portal serve
  1. Open the preview page in your browser:
open "http://127.0.0.1:8080/portal/internal/services/api/openapi/test/preview.html"

(open is macOS; on Linux use xdg-open; on Windows paste the URL into your browser.)

By default the page loads merged-spec.json — every fragment under fragments/ rendered as a single Swagger UI, so all of your operations are visible at once. To focus on one fragment, pick it from the dropdown at the top, or append ?spec=fragments/{resource}/{operationId}.yaml to the URL. Both merged-spec.json and the per-fragment manifest are auto-regenerated by the validator (Phase 3), so the dropdown and combined view stay in sync without manual edits.

✓ Check your progress

Swagger UI shows one operation grouped under the Course tag: POST /api/v1/courses. Expanding it reveals: the Idempotency-Key request header marked required; a request body whose example reads { "name": "Introduction to Python" }; three response codes (201 Created, 400 Bad Request, 409 Conflict) each with a worked example. If you see a CORS or 404 error inside the Swagger UI panel, the YAML URL is wrong — the path you pass via ?spec= is relative to preview.html, not to the repo root. If Swagger UI itself fails to load (blank page), check your network for blocked CDN requests against unpkg.com.

5Exercise the contract via the mock

So far you've seen the contract in Swagger UI. Now we send real HTTP at it and watch the documented examples come back — without writing a single line of handler code. This is layer 3 of the toolchain (see API-first toolchain): a Connexion + Flask mock that reads merged-spec.json at startup, registers a route for every documented operation, and on each request returns the first examples.{key}.value from a documented 2xx response.

Two terminals — one for the mocks, one for curl. The portal static server on :8080 from Phase 4 cannot serve POST/PUT/DELETE (it's python -m http.server, GET-only — it returns 501 Unsupported method on writes). Mocks run on their own ports (:8001 for test, :8002 for etr_study_app, and so on) so you can keep the portal and all the mocks up at once.

  1. In a fresh terminal at the repo root, start every mock at once:
make api-mock

You should see startup output like:

== API-MOCK: START ==
🟢 Starting 2 mock server(s):
   · test                     → http://127.0.0.1:8001
   · etr_study_app            → http://127.0.0.1:8002
mock_server · INFO · Loaded 4 operations from …/test/merged-spec.json
  · createCourse   · body-schema · 201[short_track,long_track], 400[…], 409[…]
  · getCourse      · no-body     · 200[default]
  · getUser        · no-body     · 200[default]
  · updateUser     · body-schema · 200[default], 400[…], 409[…]
mock_server · INFO · Loaded 1 operations from …/etr_study_app/merged-spec.json
  · createUser     · body-schema · 201[default], 400[…], 401[…], 409[…], 422[…], 429[…]
INFO:     Uvicorn running on http://127.0.0.1:8001 (Press CTRL+C to quit)
INFO:     Uvicorn running on http://127.0.0.1:8002 (Press CTRL+C to quit)

Ctrl-C stops every mock at once. Leave this terminal open and switch to a SECOND terminal.

  1. From the second terminal, POST the documented body to the mock:
curl -X POST http://127.0.0.1:8001/api/v1/courses \
     -H 'Content-Type: application/json' \
     -d '{"name":"Introduction to Python","length":10}'

The mock answers with the documented 201 example from the fragment:

{"name": "Introduction to Python"}
  1. Try the other operations the same way — every documented method works:
# Both GETs require a documented query parameter.
curl 'http://127.0.0.1:8001/api/v1/courses/test?name=Introduction%20to%20Python'
curl 'http://127.0.0.1:8001/api/v1/users/test?email=test@example.com'

# Missing the required param → 400 in our ErrorBody envelope
curl http://127.0.0.1:8001/api/v1/courses/test
# → {"code":"validation_error","message":"Missing query parameter 'name'"}

# Malformed value → 400 (format check from the schema)
curl 'http://127.0.0.1:8001/api/v1/users/test?email=not-an-email'
# → {"code":"validation_error","message":"'not-an-email' is not a 'email' …"}

curl -X PUT http://127.0.0.1:8001/api/v1/users/test \
     -H 'Content-Type: application/json' \
     -H 'Idempotency-Key: 8f5a2c19-3b4e-4d8a-9f3b-1c2d3e4f5a6b' \
     -d '{"name":"Renamed","email":"renamed@example.com"}'

The same mock is what Swagger UI's «Try it out» button targets — every canon's merged-spec.json ships a servers: entry pointing at its own port (test → :8001, etr_study_app → :8002, …), so browser-side calls land in the matching Connexion process.

Where to open Swagger UI for the mock

Every canon renders in a Swagger UI you can open two ways:

WayURLWhen to use
Portal-hosted preview http://127.0.0.1:8080/portal/internal/services/api/openapi/<canon>/preview.html Rich shell with dark-mode toggle and a spec picker. Requires the portal (make -C services/portal serve) to be up on :8080.
Built-in mock Swagger UI http://127.0.0.1:<port>/ui/ Ships inside the Connexion mock — no portal needed. Bare-bones but always available whenever make api-mock is running.

Common gotcha. Hitting the mock's root — http://127.0.0.1:8002/ — returns 404 Not Found. That's by design: nothing is documented at /, only under the paths declared in the spec. Open /ui/ to browse, or curl one of the declared paths.

Mock troubleshooting

SymptomCauseFix
ERR_CONNECTION_REFUSED in the browser Mock isn't running on that port. Confirm with curl -s http://127.0.0.1:8002/openapi.json | head -1 — { = alive, empty = down. Start make api-mock in a fresh terminal.
404 Not Found at http://127.0.0.1:PORT/ Root path isn't in the spec. This is correct. Open /ui/ or a documented path (e.g. /api/v1/user).

Dynamic overlay — your input reflected in the response

The mock is a systems-analyst tool: on every request it overlays your input onto the picked example, so stakeholders see the exact values you sent. Keys the example already declares are overwritten; keys the schema doesn't declare are never invented.

curl -X POST http://127.0.0.1:8001/api/v1/courses \
     -H 'Content-Type: application/json' \
     -H 'Idempotency-Key: 8f5a2c19-3b4e-4d8a-9f3b-1c2d3e4f5a6b' \
     -d '{"name": "Advanced Systems Design", "length": 16}'
# → 201 + {"name":"Advanced Systems Design","length":16}   ← your values

curl 'http://127.0.0.1:8001/api/v1/courses/test?name=Rust'
# → 200 + {"name":"Rust","length":10}                       ← name from query, length from example

curl 'http://127.0.0.1:8001/api/v1/users/test?email=alice@studyapp.io'
# → 200 + {"id":"u_a1b2c3d4","name":"Alice Example","email":"alice@studyapp.io"}
#                                                              ↑ email from query, id+name from example

Turn overlay off with Prefer: dynamic=off when you want the raw documented example verbatim (for testing that the spec is correctly written, not for UX demos):

curl -X POST http://127.0.0.1:8001/api/v1/courses \
     -H 'Content-Type: application/json' \
     -H 'Idempotency-Key: 8f5a2c19-3b4e-4d8a-9f3b-1c2d3e4f5a6b' \
     -H 'Prefer: dynamic=off' \
     -d '{"name":"whatever","length":42}'
# → 201 + {"name":"Introduction to Python","length":10}    ← documented example, ignored input

Picking a specific example — the Prefer header

By default the mock returns the FIRST documented 2xx example. To pick a different one (other success example, or a documented error), send Prefer: (RFC 7240, the same convention Stripe uses):

# Pick a specific 2xx example (createCourse has minimal + full)
curl -X POST http://127.0.0.1:8001/api/v1/courses \
     -H 'Content-Type: application/json' \
     -H 'Prefer: example=full' \
     -d '{"name":"X","length":5}'
# → 201 + {"name":"Introduction to Rust","length":10}

# Pick a documented error code (createCourse documents 400 with three named examples)
curl -X POST http://127.0.0.1:8001/api/v1/courses \
     -H 'Content-Type: application/json' \
     -H 'Prefer: code=400, example=length_out_of_range' \
     -d '{"name":"X","length":5}'
# → 400 + {"code":"validation_error","message":"Field 'length' must be between 1 and 1000."}

# Pick a documented 409 case (per-resource error file)
curl -X POST http://127.0.0.1:8001/api/v1/courses \
     -H 'Content-Type: application/json' \
     -H 'Prefer: code=409, example=name_already_exists' \
     -d '{"name":"X","length":5}'
# → 409 + {"code":"course_name_already_exists","message":"A course named 'Introduction to Python' already exists."}

The example names match the keys you put in the fragment (look at fragments/course/error-409/name-conflict.yaml — name_already_exists is the YAML key). Tier-B per-resource error files are followed via $ref, so their examples are pickable just like inline ones.

Body validation — invalid payloads return our documented error envelope

The mock validates every POST/PUT/PATCH/DELETE body against the spec's request schema. Anything that doesn't match returns 400 with the shape documented in _shared/schemas/ErrorBody.yaml:

# Missing required field
curl -X POST http://127.0.0.1:8001/api/v1/courses \
     -H 'Content-Type: application/json' \
     -d '{"name":"X"}'
# → 400 + {"code":"validation_error","message":"'length' is a required property"}

# Field out of declared range
curl -X POST http://127.0.0.1:8001/api/v1/courses \
     -H 'Content-Type: application/json' \
     -d '{"name":"X","length":9999}'
# → 400 + {"code":"validation_error","message":"9999 is greater than the maximum of 1000 - 'length'"}

# Wrong format (email)
curl -X PUT http://127.0.0.1:8001/api/v1/users/test \
     -H 'Content-Type: application/json' \
     -H 'Idempotency-Key: 8f5a2c19-3b4e-4d8a-9f3b-1c2d3e4f5a6b' \
     -d '{"name":"X","email":"not-an-email"}'
# → 400 + {"code":"validation_error","message":"'not-an-email' is not a 'email' - 'email'"}

The validator is the spec — change minLength, maximum, format, or required in the fragment, re-run make -C services/portal doc-api-swagger-check-all, restart the mock, and the new rules go live. Two messages differ:

  • Schema-violation 400 (auto-generated by validation): {"code": "validation_error", "message": ": "} — the message is computed at runtime from the schema.
  • Prefer-picked documented 400: returns the EXACT code and message you wrote in the fragment's examples (e.g. "Field 'length' must be between 1 and 1000.").

Use Prefer when you want the front-end to render an EXACT documented message; rely on schema validation for any input the front-end might actually generate.

✓ Check your progress

make api-mock reaches the Uvicorn running on http://127.0.0.1:8001 line and lists every operationId; every curl above returns the documented 2xx body with the right status code; Swagger UI's «Try it out» round-trips through the same mock.

Common stumbles: «Connection refused» — the mock isn't running; check the first terminal didn't crash. «501 Unsupported method» — you hit port 8080 (portal static server) instead of 8001 (mock); change the URL. «merged-spec.json not found» — run make -C services/portal doc-api-swagger-check-all first; it's what writes the file the mock reads. «unknown operation» — you added a fragment but forgot to re-run the validator; the mock only knows what was in merged-spec.json at startup, so kill the mock with Ctrl-C, run the validator, then restart make api-mock.

5bOptional: see the parity gate trip on missing handler

This is layer 5 of the toolchain — the canon ↔ runtime parity diff. It compares merged-spec.json with FastAPI's real /openapi.json. Today it will fail, and that's the right outcome: there are no real handlers yet for the four smoke-test operations, so the runtime spec lacks them.

  1. Start the FastAPI dev server in a THIRD terminal (the real backend, separate from the mock):
make run            # uvicorn on http://127.0.0.1:8000
  1. From any terminal, run the parity gate:
make openapi-check  # canon ↔ runtime diff

Expected: make openapi-check reports drift — the canon contains POST /api/v1/courses, the runtime does not. That drift is the gate working: it blocks an implementation PR from merging until its FastAPI route matches the ratified spec. Once a real handler lands in services/api/app/, this gate flips green for that operation.

✓ Check your progress

You saw make openapi-check fail on a known cause (no handler yet). That's the layer-5 gate doing its job. You can stop here — Phase 5b is informational; the next phase moves on to authoring the human-readable spec page.

6Author the portal Operation spec page

The fragment is the machine-readable contract; the Operation spec page is its human-readable companion. Reviewers read both — the fragment to see field shapes, the page to see §3 acceptance criteria, §11 idempotency, §14 observability, §15 test plan, §19 traceability back to the fragment.

  1. Create the reference directory for the resource:
mkdir -p services/portal/internal/services/api/reference/course/operations
  1. Copy the canonical template to the canonical path:
cp services/portal/internal/handbook/sa/templates/api-spec.html \
   services/portal/internal/services/api/reference/course/operations/post-api-v1-courses.html
  1. Open the copied page and fill the §-numbered sections from the fragment:
  • §3 Acceptance criteria — one Given/When/Then per response status. Example: «Given valid name, When POST /api/v1/courses, Then 201 with body matching Course».
  • §11 Idempotency & concurrency — note that Idempotency-Key is required for writes (per ADR-0006) and that this fragment does not yet declare the header. Mark it as a TODO; you'll add the $ref to _shared/headers/IdempotencyKey.yaml before opening the spec PR.
  • §14 Observability — names of the structured-log events, Prometheus metrics, and alerts this operation must emit (per ADR-0009).
  • §15 Test plan — one row per §3 G/W/T entry. Each row maps to a pytest case name later.
  • §19 Traceability — back-links to the fragment, every ADR cited (at minimum 0003 for the error envelope, 0006 for idempotency, 0036 for the API-first policy), and the contract-test plan sibling once it exists.

✓ Check your progress

The file services/portal/internal/services/api/reference/course/operations/post-api-v1-courses.html opens in your editor and contains §1–§19 placeholders from the template. §3, §11, §14, §15, §19 contain text you wrote — not the template's <…> placeholders. The page need not be perfect; it must be reviewable.

7Hand off for review

You've authored a contract and proved the toolchain accepts it. The remaining step is to surface the artifacts a reviewer would see and stop — without committing. Commits happen only after the author (you) explicitly approves the spec PR scope.

  1. Run the gates one more time, end-to-end:
make doc-api-swagger-check-all
make docs-check
  1. List the working-tree artifacts that the spec PR would contain:
git status --short -- \
  services/portal/internal/services/api/openapi/test/fragments/course/ \
  services/portal/internal/services/api/reference/course/

✓ You're done

  1. The fragment exists, validates, and lives at the canonical path.
  2. The Swagger UI preview renders the operation, parameters, and example responses.
  3. make api-mock answers POST /api/v1/courses on :8001 with the documented 201 example.
  4. The portal Operation spec page exists with §-numbered placeholders filled.
  5. make doc-api-swagger-check-all and make docs-check both exit 0.
  6. (Optional) make run + make openapi-check reports drift on the known cause (no handler yet) — that's the layer-5 gate working.
  7. Nothing is committed. The next decision is yours: stage the fragment + page as a spec PR, or iterate.

That is the API-first loop on this project — fragment → validate → preview → exercise (mock) → spec page → review. Real endpoints widen the surface (more responses, more shared schemas, a separate contract-test plan sibling), but the steps are identical. The recipe form (no narrative, just the seven steps) lives in the Apply API-first how-to.
