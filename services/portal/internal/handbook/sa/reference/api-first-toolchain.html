API-first toolchain — five layers

The five distinct tools that make API-first work on this project, the question each layer answers, the Make target it sits behind, and how they compose. Spec → validator → mock → contract test → parity gate — each one is a server or a script with a different job; none of them replaces another. Read alongside ADR 0036 (policy) and OpenAPI fragment anatomy (YAML keys).

The big picture

Each fragment passes through five sequential layers from authoring to release. Layers 1–3 are «contract-first» work — they exist before any handler is written; layers 4–5 enter once an implementation PR opens and start gating it against the canon.

Which layer answers which question

LayerQuestion it answersToolMake targetStatus
1 · Spec What's the contract? OpenAPI 3.1 YAML (hand-authored) — live
2 · Validator Is the spec syntactically valid OpenAPI 3.1? openapi_spec_validator via tools/governance/validate_openapi.py make doc-api-swagger-check[-all] live
3 · Mock server What would the API feel like to a caller before the handler exists? connexion[flask] via tools/api/mock_server.py make api-mock · fans out to every canon (:8001, :8002, …) live
4 · Contract tests Does the real backend honour the spec for every documented input? schemathesis + pytest (future) make contract-test future
5 · Parity gate Has the runtime drifted from the canonical spec? tools/api/openapi_governance.py make openapi-check live

The five questions are different. A validator that says «spec is OK» tells you nothing about whether a caller can actually use the API — that's what the mock is for. A green mock tells you nothing about whether the real handler matches what the mock pretends — that's what the contract test is for. Each layer is a distinct check.

1 · Spec authoring

You hand-author one YAML fragment per HTTP operation under services/portal/internal/services/api/openapi/test/fragments/{resource}/{operationId}.yaml. No tooling — this is the source of truth (per ADR 0036). Shared building blocks (error envelopes, the Idempotency-Key header) live under openapi/test/_shared/. The full key-by-key reference for what to put in a fragment is at OpenAPI fragment anatomy.

2 · Validator

The validator answers «is this YAML well-formed OpenAPI 3.1?». It does NOT make any HTTP request, run any server, or check whether a handler exists. It only inspects the YAML and resolves $ref chains.

AspectValue
Toolopenapi_spec_validator (Python, already in .venv)
Wrappertools/governance/validate_openapi.py
Make targetmake -C services/portal doc-api-swagger-check FILE=… · doc-api-swagger-check-all [DIR=…]
Side effectsWrites fragments-index.json (manifest for the picker) and merged-spec.json (combined view consumed by Swagger UI and by the mock)
Exit codes0 when every fragment validates; 1 on any failure. Partials (files without a top-level openapi: key) are skipped with a ↺ Partial note and validated transitively via their parent's $ref walk.

3 · Mock server

The mock answers «what would the API feel like to a caller before the handler exists?». make api-mock fans out one Connexion+Flask server per discovered canon under openapi/*/fragments/: test lands on port 8001, product canons (etr_study_app, …) on 8002, 8003, … in alphabetical order. Each server accepts requests for every documented operation on its canon and returns the first documented 2xx examples.{key}.value from the fragment.

AspectValue
ToolConnexion with the Flask extra — single-language Python, no Node.js
Entry pointtools/api/mock_server.py
Make targetmake api-mock — fans out one Flask process per canon; Ctrl-C stops them all
Built-in Swagger UIhttp://127.0.0.1:{PORT}/ui/ ships inside each mock (no portal needed)
Root URLhttp://127.0.0.1:{PORT}/ returns 404 by design — only paths declared in the spec are documented
Readsservices/portal/internal/services/api/openapi/test/merged-spec.json
BehaviourFor each path + method: registers a route. Default response: first documented 2xx example. Dynamic overlay: the caller's request body (POST/PUT/PATCH) and query string (GET/DELETE) are overlaid on top of the example — only keys the example already contains are overwritten; response fields the schema doesn't declare are never invented. Turn off with Prefer: dynamic=off. Body validation: every POST/PUT/PATCH/DELETE body is validated against the spec's request schema BEFORE overlay runs; invalid bodies return 400 in the documented ErrorBody envelope. Prefer header (RFC 7240): Prefer: example=NAME picks a specific documented example; Prefer: code=NNN picks a specific response status; combined (Prefer: code=409, example=name_already_exists) lets the front-end test any documented error case.
Used byFrontend devs (UX-testing the API), Swagger UI «Try it out» (the merged spec ships a servers: entry pointing here), curl smoke tests, future SDK CI.

What the mock DOES — accepts HTTP, returns documented examples, lets a frontend build UI against the contract before any handler is written.

What the mock does NOT do — write to a database, run real business logic, integrate with auth, simulate latency, or remember state between requests (a POST + GET round-trip won't return the same id; Stripe-mock does this, we don't). It's a UX rehearsal of the contract, not a replacement backend.

4 · Contract tests (future)

Contract tests answer «does the real backend honour the spec for every documented input?». They run after layers 1–3 have shipped and a real FastAPI handler exists. Cadence: nightly + on every implementation PR.

AspectValue
Toolschemathesis + pytest
ApproachProperty-based fuzz against the running FastAPI: schemathesis reads the spec, generates inputs that should validate, fires them at the handler, and reports any input the schema accepts but the handler rejects (or vice versa).
Statusfuture — lands once the first ratified spec gets its implementation PR.

5 · Parity gate

The parity gate answers «has the runtime drifted from the canonical spec?». It compares the merged canon (merged-spec.json) against the FastAPI's runtime /openapi.json and fails the build on any unauthorised divergence — added required fields, removed responses, status-code regressions.

AspectValue
Tooltools/api/openapi_governance.py check
Make targetmake openapi-check
Behaviour todayTrips on drift because the canon contains operations (POST /api/v1/courses, etc.) for which no FastAPI handler exists yet. That's the gate working — it blocks merging an implementation PR whose runtime spec doesn't satisfy the ratified canon.

Common confusion. Hitting port 8080 with a POST returns 501 Unsupported method: that's Python's http.server, serving STATIC docs only. Direct curl smoke tests at a mock port (see the make api-mock startup log for the exact canon → port mapping) or the real runtime (8000), not the portal (8080).

Hitting a mock's root (http://127.0.0.1:{PORT}/) returns 404: nothing is documented at /, only under the paths in the spec. Browse the mock via http://127.0.0.1:{PORT}/ui/ (Connexion's built-in Swagger UI) or the portal-hosted preview.html.

Which ports are actually up? One-liner: lsof -nP -iTCP:8001 -iTCP:8002 -iTCP:8080 -sTCP:LISTEN.

Why not collapse the mock into the validator?

Periodically someone suggests it: «the validator already walks every fragment — why not have it serve responses too?» Three reasons:

  • Different shape. Validator is a one-shot CLI that exits 0/1. Mock is a long-running HTTP server. Different lifecycles, different process models.
  • Different consumers. Validator's audience is the spec author + CI. Mock's audience is the frontend dev + the API caller (curl, Postman, future SDK CI).
  • Different failure modes. A broken validator means «I can't tell whether the spec is correct». A broken mock means «the frontend can't develop». Coupling them couples the failure modes.

Industry mirrors this split: Stripe ships stripe-mock as a separate binary, independent of their spec linter; Stoplight ships Spectral (lint) and Prism (mock) as separate tools.

Industry context — who does this

  • Stripe — stripe-mock reads their public OpenAPI spec and serves examples. Used by their 8-language SDK CI, by frontend devs, by their own dashboard's unit tests.
  • Twilio, Plaid, GitHub — same pattern: spec-driven mock backs SDK CI and frontend development until the real backend lands.
  • Postman — sells mock servers as a product line; internally uses spec-driven mocks for design collaboration.
  • Stoplight — built Prism as the de-facto industry mock for OpenAPI; we use the Python equivalent (Connexion) for the single-language constraint.

Quick-start cheat sheet

# Edit a fragment
$EDITOR services/portal/internal/services/api/openapi/test/fragments/course/createCourse.yaml

# Layer 2 — validate; regenerates the manifest + merged spec
make -C services/portal doc-api-swagger-check-all

# Layer 3 — start every mock (terminal A)
make api-mock                           # test → :8001, etr_study_app → :8002, …

# Browse layer 2 output in Swagger UI (terminal B)
make -C services/portal serve           # :8080
open http://127.0.0.1:8080/portal/internal/services/api/openapi/test/preview.html

# Hit the mock with curl (terminal C)
curl -X POST http://127.0.0.1:8001/api/v1/courses \
     -H 'Content-Type: application/json' \
     -d '{"name":"Introduction to Python","length":10}'

# Layer 5 — once a real handler exists, parity gate
make run                                # :8000  (terminal D)
make openapi-check                      # canon ↔ runtime diff
