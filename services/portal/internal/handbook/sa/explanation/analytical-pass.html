The analytical pass — five stages, one spine, six principles

How a systems analyst on Study App turns a business goal into a shippable epic. Five named stages with objective exit criteria; a traceability spine that ties every artifact to the goal it serves; six principles that keep the method honest. This is the discipline every epic on this project runs — not the decomposition of any single one.

🎯 For systems analysts 🧭 Project method (all epics) ⏱ 8–10 min read 📎 Applied in API Analytics epic

Overview

The analytical pass runs before any code is committed to an epic. It exists so a reviewer can walk in either direction — from a failing production test back to the business goal, or from the business goal forward to «which dashboards page if this breaks» — without asking anyone. Every artifact the pass produces hangs from a single traceability spine; every stage has an exit criterion a reviewer can check without re-running the analysis.

Read in order: §2 names the five stages and their exit criteria. §3 draws the traceability spine. §4 lists the six principles that hold it together. §5 lists the epics that were built under this method — the discipline is only as real as its worked examples.

The five stages

Each stage produces named outputs and gates on an objective exit criterion — the kind a reviewer can verify by opening the artifact, not by re-doing the analysis.

StageWhat it producesExit criterion
1. Discovery Problem statement · user-journey map · glossary · stakeholder map. Each persona has at least one Job To Be Done; the glossary covers every term used in any journey.
2. Analytics Capability map · traceability matrix seed · NFR baseline (latency · idempotency · durability · privacy · observability budgets) · risk register. Every capability traces to a JTBD; every NFR has a measurable target with a number, not an adjective.
3. Design One OpenAPI fragment per operation · the Operation spec page §1–§19 · ADRs for contested choices. Spectral lint clean; §3 written as Given/When/Then; every open ADR thread is Accepted or explicitly Deferred with a written trigger condition.
4. Roadmap The dependency DAG · sprint allocation · risk-burn ordering · release-train slots. Every task has acceptance criteria, an owner, and a sprint; the critical path is highlighted.
5. Hand-off Contract-test plan · runbook stub · observability dashboard list · rollout / rollback plan. Engineering can implement the spec without re-asking; QA can author tests from the spec alone.

The traceability spine

The single thread that lets any reviewer walk in either direction — from a failing test back to the business goal, or from the business goal forward to the runbook that opens when it breaks. Every artifact any epic produces hangs from this spine; §19 of every Operation spec page (Traceability) is where the loop closes. 

Business goal (the ETR study loop)
└── Use case (a learner Job To Be Done)
    └── Capability (verb-noun on a resource — «Create note», «List due notes»)
        └── API operation (Method + Path + operationId)
            ├── ADRs (frozen decisions — slot policy, TZ semantics, soft-delete)
            ├── NFRs (latency · idempotency · durability · privacy)
            ├── Contract tests (Given/When/Then ↔ §3 of the spec)
            └── Runbook + dashboard (when production breaks)

The live matrix that instantiates this spine on Study App today lives at Services · api · Traceability matrix. Empty cells are the backlog; a blank cell is a decision, not a styling gap.

Six principles that hold the method together

  1. Contract before code. No FastAPI handler is touched before its Operation spec is frozen and its OpenAPI fragment lints clean. This is the policy of ADR 0036.
  2. Non-functional requirements are numbers. Latency budgets, idempotency windows, retention windows, privacy classes — stated in §6 / §7 of every spec, before code exists. Adjectives are not requirements.
  3. Contested choices become ADRs, not arguments in the spec. Named, owned, ratified once. The spec quotes the ADR. If a review keeps re-litigating the same decision, that is the signal to write the ADR.
  4. Sequence by risk-burn. The DAG is ordered by which artifact removes the most uncertainty next. Wave 1 of any epic is the ADR set that unblocks the widest downstream fan-out.
  5. Observability and rollback live in the spec. §14 (events / metrics / alerts) and §17 (rollout / rollback) are non-optional. The hand-off package exists before the code does.
  6. Defer in writing, with a trigger. P2/P3 decisions ship with explicit trigger conditions for revisiting. Silent deferral is what creates drift; a Deferred ADR with a trigger is a promise, an untouched TODO is not.
