Explanation SA · Foundations API tooling

From YAML to a live mock

Everything that happens between «I saved createCourse.yaml» and «Swagger UI shows the new endpoint, and the mock returns 201 with my payload». Five layers, four processes, two make commands. No magic — plain Python plus one HTML viewer. Written for SAs who write the contract but hand the runtime to someone else.

🎯 For systems analysts 🐍 Python-only stack ⏱ 12–15 min read 📎 Pairs with toolchain reference

Big picture in 15 seconds

Your job as a systems analyst is only one step out of five. The other four are automation. The whole design answers one question: how do we make sure the contract (the API shape) is written before the backend code, and every tool (UI, mock, tests) reads exactly that contract?

🖊️ You write

YAML fragments — one file per HTTP operation.

openapi/test/fragments/

🔍 Validator

Python script — checks each YAML, resolves $refs, merges everything.

tools/governance/validate_openapi.py

📦 Build outputs

Auto-generated JSON — the merged spec and the operation index.

merged-spec.json · fragments-index.json

🎭 🌐 Mock & Swagger UI

Two consumers of the merged spec — one for HTTP, one for browser preview.

:8001+ mocks · :8080 portal

Two commands drive it all:

make -C services/portal doc-api-swagger-check-all   # regen the build outputs
make api-mock                                        # start the mock (auto-reload ON)

The key idea. You edit only the yaml files. Everything else is either auto-generated or launched by a command. If something didn't show up in Swagger UI — either the validator didn't run (step 2) or the mock didn't re-read (step 4).

1 · What YOU write (source of truth)

You author YAML fragments — one file per HTTP operation. Every fragment is a self-contained OpenAPI 3.1 document that describes a single endpoint: path, method, parameters, request body, all possible responses.

File layout under your control

services/portal/internal/services/api/openapi/test/
├── fragments/
│   ├── course/
│   │   ├── createCourse.yaml       # ← one file = one operation
│   │   ├── getCourse.yaml
│   │   ├── error-400/bad-request.yaml     # ← per-resource reusable 400
│   │   └── error-409/name-conflict.yaml
│   └── user/
│       ├── getUser.yaml
│       └── updateUser.yaml
├── _shared/                     # ← cross-resource reusables
│   ├── headers/IdempotencyKey.yaml
│   ├── schemas/ErrorBody.yaml
│   └── responses/Error400.yaml
├── merged-spec.json         # ← auto-generated (don't touch)
├── fragments-index.json     # ← auto-generated (don't touch)
└── preview.html             # ← ready-made viewer (don't touch)

Three tiers of reuse

TierLives atWhen to use
A · Cross-resource_shared/Identical across every resource — error envelope, Idempotency-Key header.
B · Per-resource{resource}/error-NNN/*.yamlAll operations of one resource share the same 4xx (e.g. «course name already exists»).
C · InlineInside the fragment itselfThe response is unique to this one operation.

What a real fragment looks like

openapi: 3.1.0
info:
  title: Study App API
  version: v1
paths:
  /api/v1/courses:
    post:
      operationId: createCourse
      summary: Create a new course.
      parameters:
        - $ref: "../../_shared/headers/IdempotencyKey.yaml"
      requestBody:
        required: true
        content:
          application/json:
            schema: { $ref: "#/components/schemas/CreateCourseRequest" }
      responses:
        "201": { ... }
        "400": { $ref: "./error-400/bad-request.yaml" }
        "409": { $ref: "./error-409/name-conflict.yaml" }
components:
  schemas:
    CreateCourseRequest:
      type: object
      required: [name, length]
      properties:
        name:   { type: string, minLength: 1, maxLength: 80 }
        length: { type: integer, minimum: 1, maximum: 1000 }

Every schema constraint is 3 things at once

A line like maxLength: 80 in a fragment produces:

  1. Documentation in Swagger UI (the user sees the requirement)
  2. Runtime validation in the mock (400 if the payload violates it)
  3. A future contract for the real FastAPI handler (via the parity gate)

You write it once, it works in all three places. That's the SA leverage of API-first.

2 · Validator — turns YAML into a single spec

Triggered by: make -C services/portal doc-api-swagger-check-all, or automatically by uvicorn on every YAML save (auto-reload). The tool is tools/governance/validate_openapi.py.

What the validator does per run

  1. Walks every YAML under openapi/test/fragments/**/*.yaml.
  2. Validates each file as OpenAPI 3.1 using the openapi_spec_validator library. Partials (files without a top-level openapi: key, e.g. error-400/bad-request.yaml) are marked as «partial» and validated transitively through their parents.
  3. Resolves every $ref — file paths relative to the fragment (../../_shared/schemas/ErrorBody.yaml) and local pointers (#/components/…). If a ref is unresolvable, it prints the exact broken path.
  4. Assembles the merged spec — inlines each fragment's path items under one paths: object, prefixing local component names with the operationId to avoid collisions (so Course from createCourse doesn't clash with Course from getCourse). Downgrades to 3.0.3 for Connexion compatibility only (Connexion 3.x cannot read 3.1).
  5. Writes two files:
    • fragments-index.json — list of operations, consumed by the preview's spec picker.
    • merged-spec.json — the full OpenAPI document, consumed by the mock and Swagger UI.

⚠️ Classic mistake #1 — wrong number of ../ in $ref

Count directory levels from the fragment file up to openapi/test/, then add _shared/…:

  • Fragment at fragments/course/createCourse.yaml is 2 levels deep → ../../_shared/…
  • Partial at fragments/course/error-400/bad-request.yaml is 3 levels deep → ../../../_shared/…

The validator's error message is precise: it prints the resolved absolute path so you can see where it looked.

3 · Build outputs — what the validator writes to disk

Two JSON files and one HTML page live inside openapi/test/. The validator creates them. The mock and Swagger UI read them. Your job here is just to look and understand.

fragments-index.json

List of every discovered operation. Consumed by the preview page's spec picker.

{
  "count": 4,
  "items": [
    {
      "spec": "fragments/course/createCourse.yaml",
      "method": "POST",
      "path": "/api/v1/courses",
      "operationId": "createCourse",
      "summary": "Create a new course."
    },
    ...
  ]
}

merged-spec.json

A single OpenAPI document combining every fragment. This is what the mock and Swagger UI read.

{
  "openapi": "3.0.3",
  "info": { ... },
  "servers": [
    { "url": "http://127.0.0.1:8001" }
  ],
  "paths": {
    "/api/v1/courses": { "post": { ... } },
    "/api/v1/courses/test": { "get": { ... } },
    ...
  },
  "components": { ... }
}

The key field is servers. Each canon's merged spec ships its own servers[0].url — http://127.0.0.1:8001 for test, http://127.0.0.1:8002 for etr_study_app, and so on. That's why the Swagger UI «Try it out» button fires requests at the right mock and not at the portal (which is on 8080 and would return 501).

4 · Mock server — answers requests «for real»

Started by: make api-mock (leave it running in its own terminal). One command fans out one Connexion + Flask process per canon under openapi/*/fragments/; Ctrl-C stops them all.
Stack: Connexion (spec-driven routing) + Flask + Uvicorn (ASGI). Pure Python, lives in .venv.

What the mock does at startup

  1. Reads merged-spec.json.
  2. For each path + method under paths:, registers a route (my custom resolver wires in the handler).
  3. Builds an index — per operation: request schema and all documented examples grouped by status code.
  4. Listens on its own port: test → 8001, product canons → 8002, 8003, … in alphabetical order.
  5. Ships a built-in Swagger UI at http://127.0.0.1:{PORT}/ui/ (no portal needed).
  6. Root URL http://127.0.0.1:{PORT}/ returns 404 by design — nothing is documented at /, only under paths declared in the spec.

What the mock does per request

  1. Routing — Connexion matches URL + method to a spec operation.
  2. Input validation (jsonschema, Draft 2020-12): headers, query params, body — all checked against the spec's schemas. Anything invalid → 400 in our ErrorBody envelope.
  3. Response resolution — my resolver picks a documented example: normally the first 2xx, but a Prefer: header can steer the choice.
  4. Dynamic overlay — fields from request.body / request.args are layered on top of the example. What you sent shows up in the response.
  5. JSON response.

Example 1 · Regular POST

curl -X POST :8001/api/v1/courses \
  -H 'Content-Type: application/json' \
  -H 'Idempotency-Key: 8f5a2c19-3b4e-…' \
  -d '{"name":"Advanced","length":16}'

→ 201
{
  "name": "Advanced",   ← from your payload
  "length": 16          ← from your payload
}

Example 2 · Prefer: error scenario

curl -X POST :8001/api/v1/courses \
  -H 'Prefer: code=409, example=name_already_exists' \
  -H 'Content-Type: application/json' \
  -H 'Idempotency-Key: 8f5a2c19-3b4e-…' \
  -d '{"name":"X","length":5}'

→ 409
{
  "code": "course_name_already_exists",
  "message": "A course named '…' already exists."
}

✨ About dynamic overlay

The mock overwrites only keys that already exist in the example. If your request.body contains a field that isn't in the example, the mock ignores it (never invents response fields the schema doesn't document). To disable overlay: Prefer: dynamic=off.

5 · Swagger UI — human-readable view of the contract

Started by: make -C services/portal serve (plain python -m http.server).
URL: http://127.0.0.1:8080/portal/internal/services/api/openapi/test/preview.html

What happens when you open preview.html

  1. Browser fetches the static HTML from the portal (:8080).
  2. The HTML pulls Swagger UI 5 from a CDN (unpkg).
  3. JS fetches merged-spec.json by default (or a specific fragment YAML if the URL has ?spec=…).
  4. Swagger UI renders every operation, schema, example, and request/response example dropdowns.
  5. When you hit «Try it out» → «Execute», JS composes the body from the form and sends it to the URL from the spec's servers: block — that's :8001, the mock.

🔥 Failed to fetch / CORS? ERR_CONNECTION_REFUSED?

Not CORS — the mock for that canon isn't responding on that port. One-line check for a specific port:

curl -s http://127.0.0.1:8002/openapi.json | head -1

Got { — the mock is alive. Empty — you need make api-mock in a fresh terminal (it fans out and starts every canon at once).

Full port audit: lsof -nP -iTCP:8001 -iTCP:8002 -iTCP:8080 -sTCP:LISTEN.

6 · How ONE request travels through the whole system

The full trace: you clicked «Try it out» in Swagger UI on POST /api/v1/courses with {"name": "Advanced", "length": 16}. What happens in 200ms?

  1. Browser composes fetch — Swagger UI read servers[0].url = http://127.0.0.1:8001 from the spec and knows where to send.
  2. HTTP POST to :8001 with body + headers. Cross-origin from 8080 to 8001 works because the mock sets allow_origins=["*"].
  3. Uvicorn (ASGI server) accepts the connection, hands the scope to Connexion.
  4. Connexion routing — matches the URL to operation createCourse.
  5. Connexion validation:
    • Validates Idempotency-Key header against IdempotencyKey.yaml schema.
    • Validates body against CreateCourseRequest schema.
    • Anything invalid → raises BadRequestProblem.
  6. My handler runs only if validation passed. Reads the Prefer: header, picks status + example.
  7. Dynamic overlay: request.get_json() is layered onto the example → final dict.
  8. Return — Flask serialises to JSON, Uvicorn sends 201 + body.
  9. Browser receives the response, Swagger UI renders it in the «Response body» panel. You see your name and length.

🕰 Differential

If validation fails at step 5 → steps 6-8 don't execute. Instead my _bad_request_to_error_body handler runs, which repacks the error into our ErrorBody envelope: {"code": "validation_error", "message": "…"}.

7 · Auto-reload — why YAML edits appear on their own

The mock uses uvicorn --reload combined with the watchfiles library, which watches files and restarts the process on change.

  1. You save createCourse.yaml in your editor.
  2. watchfiles detects the change (~1s). It watches:
    • tools/api/*.py
    • services/portal/internal/services/api/openapi/test/fragments/**/*.yaml
    But not merged-spec.json or fragments-index.json — otherwise there'd be an infinite loop.
  3. Uvicorn kills the old child, spawns a new one.
  4. The mock_server.py module is re-imported. At module level: app = create_app().
  5. create_app() → refresh_spec() — dynamically loads validate_openapi.py, calls regenerate_index() and regenerate_merged_spec(). The refreshed files are written to disk.
  6. build_app() builds a new Connexion app from the fresh merged-spec.json.
  7. Mock listens on :8001 again. Refresh the browser tab on preview.html and you see the new spec.

🎉 Total: 2–3 seconds from Ctrl-S to a new response

No manual make commands between iterations.

⚠️ Fail-soft behaviour

If you save a YAML with a syntax error — the validator crashes during regen, the mock logs a warning, and continues serving the old merged-spec.json. You never end up with a dead mock because of a typo.

8 · Where validation actually lives

A common confusion: «validation from Connexion», «validation from the validator», «schema validation». There are in fact two distinct places where things get checked:

WhenWhoWhat it checksOn failure
At make doc-api-swagger-check / on auto-reload openapi_spec_validator (Python package) That your YAML is itself valid OpenAPI 3.1 — correct keys, types, all $refs resolve. Validator prints ❌ in the terminal; merged-spec is NOT updated.
On every HTTP request to the mock Connexion + jsonschema (inside the mock) That request.body, request.headers, request.query conform to schemas from the spec. 400 + ErrorBody envelope, handler does not run.

Full catalogue of what gets validated at HTTP

What's wrongResponse (HTTP 400)
Malformed JSON (missing closing })"Expecting ',' delimiter: line 3 column 15"
Required field missing"'length' is a required property"
Wrong type (string instead of int)"'' is not of type 'integer' - 'length'"
Number out of range (maximum)"9999 is greater than the maximum of 1000 - 'length'"
String too long (maxLength)"'xxx…' is too long - 'name'"
String empty (minLength: 1)"'' should be non-empty - 'name'"
Wrong format (format: email)"'not-an-email' is not a 'email'"
Regex (pattern) doesn't match"'…' does not match pattern"
Invalid header (Idempotency-Key)"'k7' is too short — minLength: 8"
Required query param missing"Missing query parameter 'name'"

Command cheat sheet

# ─── First run after cloning the repo ─────────────────────────
make setup                                    # create .venv, install deps

# ─── Long-running processes (one per terminal) ────────────────
make -C services/portal serve                 # portal :8080 (Swagger UI + static)
make api-mock                                  # mock :8001 (auto-reload ON)

# ─── Manual validator run (if auto-reload is not running) ─────
make -C services/portal doc-api-swagger-check-all              # every fragment
make -C services/portal doc-api-swagger-check-all DIR=course   # only course/
make -C services/portal doc-api-swagger-check FILE=course/createCourse.yaml

# ─── Open Swagger UI ──────────────────────────────────────────
# Portal-hosted preview (needs `make -C services/portal serve`)
open http://127.0.0.1:8080/portal/internal/services/api/openapi/test/preview.html
open http://127.0.0.1:8080/portal/internal/services/api/openapi/etr_study_app/preview.html

# Built-in Swagger UI shipped inside each mock (no portal needed)
open http://127.0.0.1:8001/ui/     # test canon
open http://127.0.0.1:8002/ui/     # etr_study_app canon

# ─── Focus on one operation ───────────────────────────────────
open 'http://127.0.0.1:8080/portal/internal/services/api/openapi/test/preview.html?spec=fragments/course/createCourse.yaml'

# ─── Check a mock is alive ────────────────────────────────────
curl -s http://127.0.0.1:8001/openapi.json | head -1     # test
curl -s http://127.0.0.1:8002/openapi.json | head -1     # etr_study_app

# ─── Which ports are actually up? ─────────────────────────────
lsof -nP -iTCP:8001 -iTCP:8002 -iTCP:8080 -sTCP:LISTEN

# ─── curl the mocks directly ──────────────────────────────────
# test canon
curl -X POST http://127.0.0.1:8001/api/v1/courses \
     -H 'Content-Type: application/json' \
     -H 'Idempotency-Key: 8f5a2c19-3b4e-4d8a-9f3b-1c2d3e4f5a6b' \
     -d '{"name":"My Course","length":10}'

# etr_study_app canon (real production contract)
curl -X POST http://127.0.0.1:8002/api/v1/user \
     -H 'X-API-Key: sk_test_49d1f7a2b98c4c31af8e6b2b0c9b4d31' \
     -H 'Idempotency-Key: create-user-2026-07-05-01' \
     -H 'Content-Type: application/json' \
     -d '{"system_user_id":"134tg","system_uuid":"b2c3d4e5-0002-4000-8000-000000000002","full_name":"Ivan"}'

# ─── Prefer headers (RFC 7240) ────────────────────────────────
-H 'Prefer: example=long_track'                    # pick a named example
-H 'Prefer: code=409'                              # get the documented 409
-H 'Prefer: code=400, example=name_too_long'       # both at once
-H 'Prefer: dynamic=off'                           # disable overlay, return example as-is

# ─── Kill a stuck process on a port ───────────────────────────
lsof -nP -iTCP:8001 -sTCP:LISTEN | awk 'NR>1 {print $2}' | xargs kill -9

Troubleshooting

SymptomCauseFix
Swagger UI: «Failed to fetch» / browser: ERR_CONNECTION_REFUSEDThe mock for that canon isn't running on that portmake api-mock in a fresh terminal — it fans out and starts every canon at once. Confirm with lsof -nP -iTCP:8001 -iTCP:8002 -sTCP:LISTEN.
Mock returns 404 Not Found at http://127.0.0.1:PORT/The root path isn't in any spec. This is correct behaviour.Open /ui/ for the built-in Swagger UI, or hit a declared path.
curl on 8080: 501 Unsupported methodYou're hitting the portal static server, not a mock.Replace port in URL with the right mock (test → :8001, etr_study_app → :8002, see the make api-mock startup log).
Validator: Unresolvable: ../../_shared/…Wrong number of ../ in $refRecount levels from the fragment up to openapi/test/
Swagger shows the old summary after a YAML editMock didn't re-read merged-spec (auto-reload didn't fire)Confirm the file actually saved, check the mock's log. Or run manually: make -C services/portal doc-api-swagger-check-all + refresh browser
Mock replies 400 Missing query parameterYou added a required query param to the fragment, but forgot to send itAdd ?name=X to the URL
Mock replies Internal Server ErrorLikely my resolver crashed. Look at the traceback in the mock's terminal.Save the traceback, fix the fragment if the error is in the schema
After renaming an operation in YAML, the old operationId in Prefer no longer worksYou renamed operationId → examples are now under the new keyUpdate the name in Prefer, or revert the rename
