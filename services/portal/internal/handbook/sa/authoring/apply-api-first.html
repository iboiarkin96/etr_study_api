How-to Authoring API Architecture

Apply API-first to a new endpoint

Recipe for shipping one new HTTP operation under the API-first policy (ADR 0036): the contract is authored, linted, reviewed, and ratified before the FastAPI handler exists. Two PRs per operation — spec first, code second. Follow the seven steps below from a clean working tree to a merged, observable endpoint.

When this applies

  • You are adding a new HTTP operation under /api/v*/ (new path, new method, or new behaviour worth a separate operationId).
  • You are amending an existing operation in a way that changes the request/response shape, the error envelope, or an idempotency rule.
  • You are back-filling a fragment for a code-first operation that pre-dates ADR 0036. Same steps; the resulting fragment carries x-build-history: code-first in its top matter.

If the change is purely internal (refactor a service, rename a private helper, change SQL), there is no contract change — skip this recipe and ship the code PR alone.

Before you start

  • You have a working dev environment — make setup creates .venv/ and installs services/api/requirements.txt (validator + Connexion + Flask + FastAPI). Three local processes are available: make -C services/portal serve (portal docs on :8080), make api-mock (fans out one Connexion mock per discovered canon — test → :8001, etr_study_app → :8002, next canon → :8003, …), make run (real FastAPI on :8000).
  • The Python toolchain is in the .venv — openapi_spec_validator (fragment lint + $ref resolution), schemathesis (property-based contract fuzz), and FastAPI's own /openapi.json (the runtime spec the parity gate diffs against).
  • You know the resource the operation belongs to (an existing {resource} directory under services/portal/internal/services/api/openapi/test/fragments/) or are creating a new one.
  • The operation's business intent is written down somewhere a reviewer can read — a ticket, a section in an epic, or a sketch in a Discovery doc. The fragment is the contract; you still need the «why».

If any of the four bullets is false, fix it first — API-first does not absolve the upstream work, it just refuses to start without it.

Tools you will use

The whole toolchain is Python — same .venv as the API, no Node.js side-stack. Every tool is invoked through a make target so pre-commit and CI run the same commands you do.

ToolWhat it doesWhen in the flow
openapi_spec_validator pip Validates each fragment as OpenAPI 3.1 and resolves cross-file $ref (e.g. shared error responses) relative to the fragment's directory. Invoked through tools/governance/validate_openapi.py. After you write the fragment, before you open the spec PR. make doc-api-swagger-check FILE=… for one fragment; make doc-api-swagger-check-all DIR={resource} for a sub-tree.
tools/api/openapi_governance.py check Compares the canonical merged spec with the FastAPI runtime /openapi.json. Any breaking change or unauthorised divergence fails the gate. Wired into make openapi-check per ADR-0008. After the handler PR opens, on every push. Runs locally via make openapi-check.
schemathesis pip Property-based fuzz against the canonical spec — generates inputs from the schema, fires them at a running handler, and reports any input the schema accepts that the handler rejects (or vice versa). Pure Python, talks plain HTTP. Nightly + before merge of the handler PR. Backstops the explicit contract tests.
Connexion + Flask pip Spec-driven mock server (layer 3). Reads merged-spec.json and for each documented operation returns the first 2xx examples.{key}.value. Lets you curl the contract before any handler exists. While authoring + during front-end / SDK work. See the 5-layer toolchain reference for how it fits with the parity gate.
FastAPI /openapi.json The runtime claim against which the canon is diffed (layer 5 of the toolchain). Once a real handler ships, its auto-generated spec is what make openapi-check compares to the canon. Always — running whenever make run is up.
Make (aggregator) Wires the gates into one verb. make doc-api-swagger-check-all for fragment lint; make openapi-check for the parity gate; make contract-test for the §3 contract suite. Per ADR-0008, this is the single «is my spec OK?» entry point. Every commit on a branch that touches the API.

Steps

1. Write the OpenAPI fragment

Create one file at services/portal/internal/services/api/openapi/test/fragments/{resource}/{operationId}.yaml. The operationId is camelCase, verb-then-noun (e.g. createCourse, listCourses, getCourse). The fragment describes this one operation only plus any local components/schemas it owns. Re-used schemas live under fragments/_shared/ and are referenced via relative $ref — the validator resolves them from the fragment's directory. For the full key-by-key reference (which keys are mandatory, what each renders to in Swagger UI, the $ref resolution rules), see OpenAPI fragment anatomy.

Every fragment includes:

  • One paths: entry with the method + path.
  • A required operationId, summary, and tags.
  • A request body (if any) with at least one full example.
  • Every documented response code with a schema and at least one example.
  • The standard error-envelope $ref for every 4xx/5xx, per ADR-0003.
  • For writes: x-idempotent: true and the Idempotency-Key header per ADR-0006.

Skeleton (replace placeholders):

openapi: 3.1.0
info:
  title: Study App API
  version: v1
paths:
  /api/v1/{resources}:
    post:
      operationId: createResource
      summary: Create a new resource.
      tags: [Resource]
      parameters:
        - $ref: "../_shared/headers/IdempotencyKey.yaml"
      requestBody:
        required: true
        content:
          application/json:
            schema: { $ref: "#/components/schemas/CreateResourceRequest" }
            examples:
              minimal: { value: { name: example } }
      responses:
        "201":
          description: Created.
          content:
            application/json:
              schema: { $ref: "#/components/schemas/Resource" }
              examples:
                minimal: { value: { id: r_abc, name: example } }
        "400": { $ref: "../_shared/responses/Error400.yaml" }
        "409": { $ref: "../_shared/responses/Error409.yaml" }
components:
  schemas:
    CreateResourceRequest:
      type: object
      required: [name]
      properties:
        name: { type: string, minLength: 1, maxLength: 80 }
    Resource:
      type: object
      required: [id, name]
      properties:
        id:   { type: string, pattern: "^r_[a-z0-9]+$" }
        name: { type: string }

2. Validate the fragment

From the repository root:

# one fragment
make -C services/portal doc-api-swagger-check FILE={resource}/{operationId}.yaml

# every fragment under one resource
make -C services/portal doc-api-swagger-check-all DIR={resource}

# every fragment in the tree
make -C services/portal doc-api-swagger-check-all

Under the hood this calls tools/governance/validate_openapi.py (which wraps openapi_spec_validator with $ref resolution rooted at the fragment's directory). Common findings on a first pass: missing operationId, missing example on a documented response, an unresolved $ref (wrong relative path to _shared/), or an unknown error-envelope shape. Fix them in the fragment — never silence the validator.

3. Exercise the operation via the mock

The mock (layer 3 of the toolchain) lets you curl the contract before any handler exists. It reads merged-spec.json at startup and replies to every documented operation with the first 2xx example from the fragment.

# terminal A — one command fans out mocks for every discovered canon
make api-mock
# → 🟢 Starting N mock server(s):
#      · test              → http://127.0.0.1:8001
#      · etr_study_app     → http://127.0.0.1:8002
# Ctrl-C stops all of them at once.

# terminal B — exercise the new operation on the right port
# (test canon → :8001, product canons → :8002+, in the order shown above)
curl -X POST http://127.0.0.1:8002/api/v1/{resources} \
     -H 'Content-Type: application/json' \
     -H 'Idempotency-Key: 8f5a2c19-3b4e-4d8a-9f3b-1c2d3e4f5a6b' \
     -d '{ "name": "example" }'
# → 201 + documented example body

# Browse every operation of a canon in Swagger UI — two options:
#   1. Portal-hosted:   http://127.0.0.1:8080/portal/internal/services/api/openapi/{canon}/preview.html
#   2. Built-in mock:   http://127.0.0.1:{PORT}/ui/   (Connexion ships its own Swagger UI)
# Note: hitting the mock's root — http://127.0.0.1:{PORT}/ — returns 404 by design.
# Nothing is documented at /, only under the paths declared in the spec.

Drive the operation as a client would (curl, HTTPie, the front-end you're pairing with, or Swagger UI's «Try it out»). The examples you put in the fragment double as request fixtures for the contract tests in step 5; if an example fails the schema in the fragment, the validator from step 2 will reject it before the mock ever loads it.

Auto-reload is ON by default — save a YAML, watchfiles restarts the affected mock in 2–3 seconds. Manual Ctrl-C + make api-mock is only needed when the mock crashes on startup (broken YAML surfaced too late). The full layer breakdown is at API-first toolchain · five layers.

4. Author the portal Operation spec page §1–§19

Copy the template:

cp services/portal/internal/handbook/sa/templates/api-spec.html \
   services/portal/internal/services/api/reference/{resource}/operations/{method}-{path}.html

Fill all nineteen sections. The non-skippable ones for API-first:

  • §3 Acceptance criteria in Given/When/Then. Every line becomes a test name later — write it like a test name.
  • §6 NFRs with numbers: P95 latency budget, idempotency window, retention class, privacy class.
  • §11 Idempotency & concurrency. Must match the fragment's x-idempotent and the header spec.
  • §14 Observability. Names of the structured-log events, Prometheus metrics, and alerts this operation must emit (per ADR-0009).
  • §15 Test plan. Unit + contract + integration + at least one E2E line. Each cell references a §3 G/W/T id.
  • §19 Traceability. Back-link to the fragment, every ADR cited, the contract-test plan sibling.

5. Write the contract-test plan as a sibling page

Copy handbook/sa/templates/test-plan.html to {operationId}-contract-tests.html next to the spec. The contract test is the executable form of §3 — one pytest per Given/When/Then row, with the fragment's examples as fixtures, plus a small schemathesis property-based pass that fuzzes the schema (nightly):

.venv/bin/schemathesis run \
    http://127.0.0.1:8000/openapi.json \
    --endpoint "/api/v1/{resources}" \
    --base-url http://127.0.0.1:8000 \
    --hypothesis-max-examples=50

6. Open the spec PR; ratify; then write the handler

The spec PR contains: the fragment, the portal Operation spec page, the contract-test plan page, and updates to nav-tree-internal.json. It contains no Python handler code. Reviewers focus on the contract.

When the spec PR ratifies (status pill flips draft → ratified in the spec page header), open the implementation PR. Its only review question is «does the running code match the ratified spec». Add the FastAPI route, the service, the repository, the model, the Alembic migration if needed, and the contract tests written against §3.

7. Verify parity and merge

Locally, before pushing:

make -C services/portal doc-api-swagger-check-all   # fragment lint
make openapi-check                                  # canon ↔ FastAPI parity gate
make docs-check                                     # portal autogen + link integrity
make test path=services/api/tests/contract/test_{operationId}.py

CI runs the same gates. If openapi-check reports drift, the handler PR — not the fragment — is wrong. Either align the handler to the spec, or open a follow-up fragment-amendment PR first (and re-ratify) before re-running CI.

Verify

  • Fragment exists at services/portal/internal/services/api/openapi/test/fragments/{resource}/{operationId}.yaml and validates as OpenAPI 3.1.
  • make -C services/portal doc-api-swagger-check-all exits 0 — every fragment in the tree passes.
  • Every documented response status has at least one examples block — they're the contract-test fixtures.
  • Portal Operation spec page exists, has all of §1–§19 filled, and §19 back-links to the fragment + cited ADRs.
  • Contract-test plan sibling exists and lists one row per §3 G/W/T entry.
  • Sidebar nav-tree has entries for both the spec and the contract-test plan; both pages reachable from governance/backlog/plans.html via the right breadcrumb.
  • The handler PR's CI step make openapi-check exits 0 — canon and runtime agree.
  • Contract tests pass against the running handler (make test path=services/api/tests/contract/).
  • make docs-check exits 0.

Anti-patterns

  • Editing the FastAPI handler before the spec ratifies. Defeats the whole policy. If you're already writing the handler, the spec PR is overdue — split the work.
  • Reaching for a Node-based tool (Spectral, Prism, openapi-diff) instead of the Python toolchain. The repo stays single-language on purpose — one .venv, one requirements.txt. New rule? Add it to tools/governance/validate_openapi.py, not a second toolchain.
  • Silencing the validator with an inline override instead of fixing the cause. Open an ADR to amend the rule, or fix the fragment — never look the other way.
  • Documenting one response example for the happy path only. Every documented response code needs at least one example. Otherwise the contract test loses its fixture.
  • Treating x-build-history: code-first as «I can skip the spec PR». It marks a back-fill from a pre-ADR-0036 era only. New work never uses it.
  • Letting openapi-check drift cross a release. Drift is a release-blocker, not a follow-up ticket.

Troubleshooting

SymptomLikely causeFix
Validator reports Unresolvable: ../_shared/responses/Error400.yaml. The $ref path is wrong — too many or too few ../ for the fragment's depth. Count the directory levels from the fragment up to fragments/; the $ref needs that many ../ before _shared/. tools/governance/validate_openapi.py resolves refs relative to the fragment's directory.
Validator reports «operationId missing» on a route that has one. The operationId is on the path-level object, not the method-level object. Move operationId into the post: / get: / etc. block.
make openapi-check reports breaking change «added required field». The FastAPI handler accepts a new required field that wasn't in the ratified spec. If the field is intended, open a fragment-amendment PR first; if not, remove the requirement from the handler.
Contract tests pass locally but fail in CI. The local FastAPI runs against a stale Alembic state and the schema mismatches what CI sees. Run make migrate then make openapi-check locally; the parity gate will surface the mismatch before push.
Schemathesis flags an input the schema clearly allows. Example uses snake_case where the schema declared camelCase (or vice versa). Align the example to the schema's property names verbatim — the validator and schemathesis are both strict.
