Write a postmortem

Recipe for documenting an incident after the all-clear. Copy the postmortem template, fill the timeline, root cause, action items. Blameless prose only — names of systems, not people.

Before you start

  • The incident is closed (all-clear declared, service restored).
  • You have raw timeline data — Slack timestamps, alert fire times, deploy events, log excerpts.
  • You've opened handbook/sa/templates/postmortem.html.

Steps

  1. Name the file. services/<svc>/postmortems/YYYY-MM-DD-<short-title>.html. Date is the incident date.
  2. Set body data-page-type to "reference" — postmortems are reference docs (they document an event, not a procedure).
  3. Fill Metadata: Postmortem · Incident date (with UTC range) · Severity (SEV-1/2/3) · Category · Owner · Last reviewed.
  4. Write Summary — one paragraph: what happened, what was affected, how it was resolved.
  5. Quantify Impact — duration, user-facing failures, error-budget burn, customer signals.
  6. Build the Timeline in UTC — minute-level, who-saw-what-when. Include the alert fire, the page, every hypothesis change, mitigation, all-clear.
  7. Write Root cause — the contributing factors. Multiple causes are normal; name the failure mode for each.
  8. Document Resolution — immediate mitigation, same-day fix, durable fix (often a follow-up PR).
  9. Reflect on Detection — what told us, what did not, what would have caught it earlier.
  10. List Action items — owner, due date, backlog ID. Each answers «would this fix it next time?».
  11. Write Lessons — what we now know that we did not before. Blameless prose only.

Verify

  • Filename matches YYYY-MM-DD-<title>.html.
  • data-page-type="reference" on <body>.
  • Nine canonical sections present in order: metadata-h · summary-h · impact-h · timeline-h · root-cause-h · resolution-h · detection-h · actions-h · lessons-h.
  • Metadata <dl> surfaces Postmortem · Incident date · Severity · Category · Owner · Last reviewed.
  • Timeline rows are UTC.
  • Every action item has owner + due + backlog ID.
  • pytest tests/visual/test_postmortem_canon.py green.

Troubleshooting

  • «Blame creeping in». Replace person names with system names. «The deploy script ran twice» beats «X ran the deploy twice».
  • «Root cause is just one bullet». Always check for contributing factors. The fix-up sequence usually unveils 2–3.
  • «No action items». Then either the incident wasn't worth a postmortem, or you haven't asked «would this fix it next time?» honestly.
