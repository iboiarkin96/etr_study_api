Author and test a new page-type template

When you introduce a new page-type to the portal (a new kind of doc: postmortem, screen-spec, anything not already in the kit), you build two templates and two tests in one PR. This page is the recipe.

Why two templates per page-type

Every page-type in the portal carries two distinct artefacts — they look similar and live in different folders on purpose:

  • ui-kit/pages/templates/doc-<type>.html — visual specimen. A complete, named, plausible example of the page-type (not placeholders) rendered in the UI Kit gallery. Read by designers and front-end maintainers. Sits in the kit sidebar so a reviewer can click «Doc · ADR» and see exactly what an ADR page should look like.
  • handbook/sa/templates/<type>.html — authoring shell. A fillable copy-paste skeleton with <placeholder> markers in every slot, plus a closing «Authoring notes» ghost-card with voice rules / anti-patterns / when-to-use. Read by SREs / SWEs / SAs at write time. Sits next to the other SA templates.

Both files share the same structural skeleton (same <h2> set, same closing aside, same docs-history shape). Only the contents differ — specimen has concrete prose, shell has angle-brackets-marked placeholders.

Before you start

  • You've decided this page-type warrants a canonical form — see the Diátaxis Reference quadrant catalog for the four-quadrant taxonomy before adding a new shape.
  • The existing nearest neighbour is in the kit: look at runbook.html and tutorial.html as concrete prior art before designing yours.
  • You can run the test suite locally: .venv/bin/pytest tests/visual/test_runbook_canon.py -v already works on your branch (otherwise — fix that first).

Steps

1. Design the canon

Pin down the required <h2> sections, their fixed order, the required data-page-type for the Diátaxis quadrant the page-type lives in, and the closing block (Related / Next-step / nothing).

Document the canon in a comment block at the top of the structural test you'll write in step 4. The test IS the canon — there should be no other authoritative list.

Survey at least 5 existing pages of this kind in the portal (or external prior art if none exist here) before declaring section X mandatory.

2. Create the SA authoring shell

cp services/portal/internal/handbook/sa/templates/runbook.html \
   services/portal/internal/handbook/sa/templates/<type>.html

Edit:

  • Title, breadcrumbs, eyebrow pills (use the --accent pill colour for the page-type label).
  • Keep data-page-type="reference" on the body. The SA template documents the form — it sits in the Reference quadrant by convention. The page-type it describes ("how-to" / "tutorial" / "reference" / …) is what live pages declare, not the template.
  • Replace every prose body with <placeholder> markers. Use angle brackets, not square brackets, for placeholders — keeps copy-paste friendly with HTML.
  • Add a closing <aside class="docs-card docs-card--ghost"> «Authoring notes (for the writer — delete before shipping)» summarising the voice rules and anti-patterns.

3. Create the UI Kit specimen

cp services/portal/ui-kit/pages/templates/doc-tutorial.html \
   services/portal/ui-kit/pages/templates/doc-<type>.html

Edit:

  • Replace placeholders with a concrete fake-but-plausible example. An ADR specimen says «ADR 0042: Cache hot-path reads with Redis», not «ADR <NNNN>: <title>». Designers need to see how it actually looks rendered.
  • Use real-looking link targets (placeholder URLs like # are fine; what matters is the markup).
  • Body declares data-page-type="<actual-quadrant>" — match what live pages of this kind use (runbook = "how-to", tutorial = "tutorial", ADR = "reference").
  • Same structural skeleton as the SA shell — h2 sections in the same order with the same id anchors.

4. Add the structural test

Copy the nearest neighbour: tests/visual/test_runbook_canon.py or test_tutorial_canon.py. Adjust:

  • TEMPLATE_PATHS — points to your two new template files.
  • REQUIRED_SECTION_IDS — the canon you pinned in step 1.
  • _live_<type>s() — globs the live pages folder where this page-type ships in the portal.
  • Per-assertion canon: data-page-type, metadata <dl class="svc-meta"> labels, closing aside title, docs-toc reading order.
  • Cross-template agreement test (both templates carry the same canonical sections — catches drift between the SA shell and the UI Kit specimen).

Run it:

.venv/bin/pytest tests/visual/test_<type>_canon.py -v --no-cov

It should be green before you move on. If it isn't, your templates disagree with the canon you declared — fix one of them.

5. Add to the visual corpus

Edit tests/visual/pages.json — add two explicit entries:

{ "id": "tpl-ui-kit-doc-<type>", "path": "portal/ui-kit/pages/templates/doc-<type>.html" },
{ "id": "tpl-sa-<type>",         "path": "portal/internal/handbook/sa/templates/<type>.html" }

Both templates go in the visual corpus — they're the contract for how this kind of page looks. Live pages of this type do NOT go in the visual corpus — their content drift is structural, not pixel-level (the structural test from step 4 covers them).

Seed baselines:

make visual-test-update     # captures the two new PNG sets

Eyeball the new PNGs in tests/visual/baselines/ — confirm they show the page-type rendered cleanly in both themes and on mobile.

6. Wire the sidebar

Add an entry to both nav-trees:

  • UI Kit sidebar — services/frontend/portal/assets_v2/ui-kit/mocks/nav-tree-uikit.json, under the templates section. Alphabetical insertion.
  • Internal portal sidebar — services/frontend/portal/assets_v2/ui-kit/mocks/nav-tree-internal.json, under handbook/sa/templates. Alphabetical insertion. Preserve memory rule sidebar acronym casing for things like ADR / API / SRE.

7. Add to the templates landing

Edit services/portal/ui-kit/pages/templates/index.html — add a <li> entry both to the by-family spine-grid AND to the full-table view. Pick a family — Landing / Diátaxis / Ops / Governance / … — by what readers will look under.

Verify

  • make -C services/portal verify — runs docs-check, docs-a11y-check, AND make visual-test. All three must be green.
  • .venv/bin/pytest tests/visual/test_<type>_canon.py -v --no-cov — structural canon green.
  • make open — open the UI Kit gallery, click your new specimen from the sidebar, eyeball it in light + dark + mobile.
  • git diff --stat tests/visual/baselines/ — only the two new tile-sets should appear, not 30 other PNG files.

Troubleshooting

  • Structural test fails on the SA template — the shell drifted from the UI Kit specimen. Reconcile sections / IDs / order until both pass the same canon.
  • «MISSING baseline» on a brand-new tile — you forgot step 5b (make visual-test-update) after editing pages.json. Run it once.
  • Visual test reports orphan PNGs after step 5 — you removed a page-id from pages.json but the PNGs still sit in baselines/. make visual-test-update sweeps orphans automatically on a full-corpus run.
  • «Templates agree» test fails — your two templates have different h2 sets. Pick one and reconcile.
