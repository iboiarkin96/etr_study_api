Author a new template

Recipe for adding an 11th-and-beyond page-type template to the SA handbook. One file under handbook/sa/templates/, one entry in tests/visual/pages.json, one paired authoring guide, one bespoke canon test (if the genre has a live corpus). Single source of truth — UI Kit twin model retired 2026-06-19.

Before you start

  • You can name the new template's genre («postmortem», «runbook», «test-plan») and explain how it differs from the templates already shipped.
  • There's a real live corpus the template will document — or a concrete plan to seed one. Templates without a live page break the canon-test loop.
  • You've read the templates index and confirmed the form isn't already covered.

Steps

  1. Place the template. Create services/portal/internal/handbook/sa/templates/<slug>.html. Body declares data-page-type="reference" (templates are handbook reference, even when they describe a how-to genre). Skeleton mirrors any sibling — start from runbook.html or how-to.html.
  2. Form-only, no recipe. The template carries placeholder content in each canonical section (<Insert title here>, <one-sentence summary>). All authoring rules — title formula, voice, anti-patterns — belong in the paired authoring guide, not in the template.
  3. Pick canonical section IDs. Required <h2 id="…-h"> anchors in fixed order. Three to nine sections; fewer than three is undercast, more than nine is sprawl. Keep IDs short: before-h, steps-h, verify-h, impact-h, etc.
  4. Add the template-annotator script. The «Enable annotation» toggle ships from ui-kit/components/template-annotator.js — add the module script tag at the bottom (see any existing template).
  5. Register the visual baseline. Add an entry to tests/visual/pages.json: 
    { "id": "tpl-sa-<slug>", "path": "services/portal/internal/handbook/sa/templates/<slug>.html" }
    Run make -C services/portal visual-test-update — three PNGs land under tests/visual/baselines/.
  6. Create the paired authoring guide. handbook/sa/authoring/write-a-<slug>.html (or file-a-<slug> if the verb fits). Itself a canonical how-to with five sections (before-h, steps-h, verify-h, optional troubleshooting-h, related-h). Move title formula, voice rules, anti-patterns here.
  7. Write the bespoke canon test if there's a live corpus. tests/visual/test_<slug>_canon.py — html5lib walk asserting required h2 IDs, metadata terms, data-page-type, history footer, docs-toc order. Copy test_runbook_canon.py as the starting shape. Skip this step if no live corpus yet.
  8. Wire into the sidebar. nav-tree-internal.json — template under SA · Templates, authoring guide under SA · Authoring.
  9. Sanity-check. make -C services/portal verify-local — runs docs-check (drift + a11y + asset refs + nav coverage), then visual-test (pixel-diff vs the new baselines). All green.

Verify

  • Template file exists under handbook/sa/templates/<slug>.html with data-page-type="reference".
  • Template carries placeholder content only — no «Title rules» / «Worked example» / «Do-don't» sections.
  • tests/visual/pages.json has the new entry; baselines exist for three viewports.
  • Authoring guide exists under handbook/sa/authoring/<verb>-a-<slug>.html with data-page-type="how-to" and the five canonical sections.
  • If live corpus exists — tests/visual/test_<slug>_canon.py is parametrised over the template + every live page, all assertions green.
  • Sidebar entries land under SA · Templates and SA · Authoring.
  • make -C services/portal verify-local green.
  • pytest tests/visual/test_sa_templates_shape.py green (generic shape canon auto-discovers the new template).

Troubleshooting

  • «Template feels like a recipe». If it has h2 sections explaining how to author (Title rules, Voice, Anti-patterns), the recipe leaked. Strip those sections; the authoring guide owns them.
  • «No live corpus yet». Skip the bespoke canon test for now — the generic shape canon (test_sa_templates_shape.py) covers the template alone. Add the bespoke canon as soon as the first live page lands.
  • «Section IDs collide with a sibling». Canonical IDs (metadata-h, before-h, steps-h, verify-h) are shared across genres — that's intentional. The bespoke canon asserts which IDs the specific genre requires; overlap is OK.
  • «Visual baseline differs on every run». macOS-Chromium vs Linux-Chromium sub-pixel AA causes 2–7 % drift cross-platform. visual-test is local-only by design — see the quality-gates map footnote under Table B.
