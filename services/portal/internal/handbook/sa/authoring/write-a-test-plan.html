Write a test plan

Recipe for shaping a feature- or release-level test plan: scope, levels, environments, entry/exit criteria, risks. Copy the test-plan template.

Before you start

  • The feature / release is scoped — you know what's in and what's out.
  • You have the acceptance criteria from the requirements / ADR / RFC.
  • You've opened handbook/sa/templates/test-plan.html.

Steps

  1. State the scope — what's in (one list), what's out (another list).
  2. List the test levels — unit · integration · contract · E2E · manual. For each: who owns, where the cases live.
  3. Specify environments — local · staging · prod-equivalent. Which level runs where.
  4. Define entry criteria — what must be true before testing starts (code complete, deployed, fixtures loaded).
  5. Define exit criteria — what must be true to declare done (% pass, performance baseline, no SEV-1/2 open).
  6. List risks — what could miss (areas with low coverage, untested branches, scale assumptions).
  7. Schedule — when each level runs, who's running it.
  8. Cross-link the requirements, ADRs, and the test cases (see Write a test case).

Verify

  • Scope · levels · environments · entry · exit · risks sections all present.
  • Exit criteria are observable and numeric where possible.
  • Risks section names at least three.

Troubleshooting

  • «Exit criteria are aspirational». «All tests pass» is meaningless if there are no tests. State numeric thresholds — pass rate, coverage, no open SEV-1/2.
  • «Risks section says ‹none›». Then the plan isn't honest. Every release has untested edges.
