File a bug report

Recipe for filing a reproducible bug under governance/backlog/. Copy the bug-report template, fill the title, severity, steps-to-reproduce, expected vs actual, attach minimal evidence.

Before you start

  • You can reproduce the bug in a clean environment (or you've tried twice and documented what you tried).
  • You have a one-sentence summary that names what's broken and where.
  • You've checked the backlog for duplicates.
  • You've opened handbook/sa/templates/bug-report.html.

Steps

  1. Write the title. Verb-first, names the broken behaviour. «POST /api/v1/user returns 500 on duplicate email» beats «User bug».
  2. Set severity: SEV-1 (outage), SEV-2 (degraded), SEV-3 (cosmetic).
  3. List steps-to-reproduce. Numbered, atomic, copy-paste-runnable. Anyone reading should reproduce without asking questions.
  4. State Expected vs Actual. Two paragraphs, side by side.
  5. Attach minimal evidence. Log lines, screenshots, curl traces. Strip anything not load-bearing.
  6. Set ownership. Reporter · Assignee · Verifier.
  7. Cross-link. Related ADR / runbook / postmortem. If the bug is a regression, name the commit or release that introduced it.

Verify

  • Title starts with the broken verb-phrase.
  • Severity is one of SEV-1/SEV-2/SEV-3.
  • Steps-to-reproduce is numbered and atomic.
  • Expected and Actual sections both present.
  • At least one piece of evidence attached.
  • Backlog entry created; bug links back to this report.

Troubleshooting

  • «Can't reproduce». Two attempts on clean state minimum. If still flaky, document the flake rate and file as SEV-3 with «intermittent» tag.
  • «Bug or feature request?» If the spec says X and the system does Y — bug. If the spec doesn't mention it and the user wants different behaviour — feature request, file an RFC instead.
