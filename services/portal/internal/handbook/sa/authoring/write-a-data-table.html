Write a data-table spec

Recipe for documenting a tabular data view: column schema, sort/filter behaviour, empty/loading/error states, row actions. Copy the data-table template.

Before you start

  • The table has a stable column schema (no «we'll figure out fields later»).
  • You know the data source — REST endpoint, in-memory JSON, etc.
  • You've opened handbook/sa/templates/data-table.html.

Steps

  1. List columns. Header label, data type, source field, formatter, sortable yes/no, default sort direction.
  2. Specify filter rules. Faceted vs free-text, which columns are filterable, what AND/OR semantics apply.
  3. Define states — loading skeleton, empty (no data), error (request failed), filtered-empty (filters too tight).
  4. Document row actions — click row → drawer/modal/navigate; bulk actions if applicable.
  5. List affordances — column resize, column reorder, sticky header, pagination vs infinite scroll.
  6. Write a11y rules — table semantics (scope, caption), keyboard navigation, screen-reader announcements.

Verify

  • Every column has data type + source field + formatter.
  • All four states (loading / empty / error / filtered-empty) documented.
  • Row click behaviour named explicitly.
  • A11y rules cover keyboard + screen-reader.

Troubleshooting

  • «Schema not stable». Don't document a moving target. Stabilise the contract first, then write the spec.
  • «Filter explosion». ≥ 8 filter facets usually means the table is doing two jobs; split into two views.
