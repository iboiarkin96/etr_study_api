Write a component spec

Recipe for documenting a UI Kit component: anatomy, variants, states, tokens consumed, a11y rules. Copy the component-spec template, fill the live demo, the API table, the tokens reference.

Before you start

  • Component is implemented in services/frontend/portal/assets_v2/ui-kit/components/ with stable CSS/JS API.
  • You have at least one live consumer in the portal — the spec documents what's used, not what could be used.
  • You've opened handbook/sa/templates/component-spec.html.

Steps

  1. Place the file. services/portal/ui-kit/pages/components/<component-name>.html.
  2. Write the lede — one sentence: what the component is, what slot it fills.
  3. Add a live demo at the top — the consumer sees the thing before reading about it.
  4. Document anatomy — labelled parts of the component (with annotation overlay if non-trivial).
  5. List variants and states — each rendered side-by-side, named, with the CSS modifier that produces it.
  6. Specify the API — CSS classes, JS props, data-attributes the consumer sets.
  7. List tokens consumed — colour, spacing, type, motion. Link to tokens.
  8. Write a11y rules — roles, keyboard, focus, contrast.
  9. List sample uses — live pages where the component appears.

Verify

  • Live demo renders inside docs-example wrapper.
  • Anatomy / variants / states / API / tokens / a11y sections all present.
  • Component is listed in UI Kit components index.
  • make -C services/portal verify-local green.

Troubleshooting

  • «Component has no live consumer yet». Don't ship the spec until something uses it — specs without consumers go stale fast.
  • «Variants explode». If you have ≥ 6 modifier classes, split into multiple components.
