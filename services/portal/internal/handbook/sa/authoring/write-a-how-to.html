Write a how-to

The recipe for authoring a how-to page that conforms to the portal's Diátaxis-task canon. Copy the how-to template, then apply this guide to fill it: when the quadrant is right, what the title and lede say, what each canonical section carries, what voice to write in, what to never ship.

Before you start

  • You know the task you're documenting and can name it with an imperative verb («Add a write endpoint», «Rotate a token»). If you can't, pick a different quadrant.
  • The task has a verifiable outcome — a check command, a passing test, a status code, a green CI gate.
  • You've opened handbook/sa/templates/how-to.html and have it side-by-side with this page.
  • You've checked that the reader of this page is not a beginner in the domain — they already know what they're trying to do and need the recipe, not a rehearsal. Beginners read tutorials.

Steps

1 · Confirm the quadrant is how-to

  • Single, well-named task. «Add a write endpoint». «Rotate a token». «Run a load test». If the title is a noun phrase or a topic name, the quadrant is wrong.
  • Verifiable outcome. Without an exit check the page is not a how-to.
  • Conditionals allowed. Unlike tutorials, branches are allowed and labelled inline («If Postgres 15: …», «If you skipped step 2: …»).
  • Not a beginner audience. Reader already knows what they're trying to do.

If any of the above is false — pick a different quadrant: Tutorial (learning, beginner), Reference (lookup, exhaustive spec), or Explanation (understanding, trade-offs).

2 · Write the title (verb-first)

Every how-to title starts with an imperative verb naming the task. The optional worked example or scope qualifier goes in parentheses.

  • «Add a write endpoint (worked example: POST /api/v1/contract)»
  • «Run an API load test»
  • «Verify an API endpoint against its contract»
  • «Rotate a service-account token»

Anti-titles — do not ship these:

  • «Local development» — noun phrase; rename to «Run the API locally».
  • «Docker image and container» — describes a topic, not a task; either split into Explanation + How-to or rename to «Build and run the API as a Docker image».
  • «Accessibility testing playbook» — wraps the verb in a noun; rename to «Run an accessibility audit on a page».
  • «Documentation pipeline (docs-as-code)» — describes an artefact; that's a Reference or Explanation page.

3 · Write the lede (one sentence)

State the deliverable and the smallest precondition. Link out for context, do not embed it.

  • «How to verify any HTTP operation under /api/v1/… against its internal specification. Line-by-line outcomes go in the API endpoint acceptance checklist.»
  • «Register a new deployable artefact in the catalog. Write one catalog-info.yaml, run make catalog-render, the generator updates the entity-card HTML, hub tile, hero ticker, and nav-tree.»

Avoid leading with «In this guide we will…», «Welcome to…», or any context-setting paragraph. The reader already knows what they want; cut to the recipe.

4 · Fill the canonical sections in order

Every how-to carries the same anchors and the same order. Drop optional sections by leaving them out; never reorder.

  1. <section id="before-h"> — Before you start. Three to six bullets: required tools, required permissions, required state (migrations applied, env vars set, branch checked out). Link the prerequisite tutorial or reference if the reader is missing context.
  2. <section id="steps-h"> — Steps. Numbered list, ≤10 top-level items. Conditionals labelled inline. Code snippets in docs-code blocks. Each step is one action with one outcome. Domain-specific sub-steps render as <h3> inside Steps. Link to Reference / Explanation pages inline within the step text — this is the Diátaxis cross-quadrant hinge.
  3. <section id="verify-h"> — Verify. The exit check. A command, a status code, a passing test, an inspect-the-dashboard step. Without this section the page is not a how-to.
  4. <section id="troubleshooting-h"> — Troubleshooting (optional). Two to five common failure modes with one-line fixes. Not a full diagnostic flow — runbook territory.
  5. <section id="related-h"> — Related (optional). Reference for the spec, Explanation for the «why», sibling how-tos. Cross-quadrant hinge that's not inline.
  6. <footer class="docs-history"> — page history rows.

5 · Shape the Verify block

The Verify block carries the contract: «if these checks pass, the task is done». Keep it copy-paste-runnable.

<section aria-labelledby="verify-h">
  <h2 id="verify-h">Verify</h2>
  <ul>
    <li><code><check command></code> exits 0.</li>
    <li><code>GET <url></code> returns <status> with <assertion on body>.</li>
    <li><CI gate> passes (<code>make verify</code>).</li>
    <li><Reference page> reflects the change (link).</li>
  </ul>
</section>

For incident-response how-tos (runbooks), use the runbook skeleton instead: Trigger → Fast triage → Causes → Recovery → Exit criteria → Follow-up. See runbook template.

6 · Apply voice rules

  • Imperative throughout. «Run», «Add», «Confirm», «Open». Never «давайте», «we will», «you can».
  • Second-person sparingly. «You'll see», «You should observe» — only where imperative is awkward.
  • Conditionals labelled inline. «If Postgres 15: …», «If you skipped step 2: …».
  • No «why» on this page. Link to an Explanation page instead. One-sentence rationale is OK; multi-paragraph context belongs in explanation/.
  • Cite real artefacts. Link to actual ADR / RFC / runbook / checklist IDs so the recipe is anchored in production-grade context.
  • Code blocks copy-paste runnable. No $ prompt, no truncation. Real values where possible.

7 · Add the page-history row

        <short summary>.</span>
    </li>
  </ul>
</footer>

Verify

  • The title starts with an imperative verb naming the task.
  • The lede is one sentence — deliverable + smallest precondition.
  • Five canonical anchors are present in order: before-h → steps-h → verify-h → (troubleshooting-h) → (related-h).
  • <body data-page-type="how-to"> is set (live page declares the quadrant; the template carries reference).
  • Verify section names at least one exit check that proves the task is done.
  • Domain-specific sub-headings render as <h3> inside steps-h, never as additional <h2> ahead of verify-h.
  • A page-history footer carries at least one row.
  • pytest tests/visual/test_howto_canon.py passes when the file is parametrised.

Troubleshooting

  • Multi-task megapage. One page = one task. If the page covers 5 tasks, split into 5 pages. Exception: a labelled collection page («Playbooks for docs-frontend changes») with deep anchors per playbook.
  • Reference disguised as How-to. If the page is an inventory («Commands by theme», «Generated paths table»), it's a Reference. Move it.
  • Explanation embedded inline. If a section explains a concept («Docker in one minute», «HICCUPPS-F heuristics»), hoist it to explanation/ and link out.
  • Template in how-to/. Templates describe shape (a reference activity). Templates live in handbook/sa/templates/, never in how-to/.
  • How-to without Verify. Without an exit check, the reader can't confirm success. Always include Verify.
  • How-to duplicating a tutorial. Same topic, two pages. Either the tutorial graduates into the how-to (learning loop done, recipe stays) or the how-to is collapsed back into the tutorial.
  • «Playbook:» prefix in the title. Drop it. Either the page is a playbook or it's not; the chip row already says How-to.
