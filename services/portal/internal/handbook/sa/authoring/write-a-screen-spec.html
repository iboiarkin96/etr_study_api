Write a screen spec

Recipe for documenting a full screen: layout map, anatomy, behavior matrix, responsive rules, accessibility. The screen-spec template is the heaviest in the SA family (17 canonical sections); this guide walks them in order.

Before you start

  • The screen is implemented and reachable in at least one environment.
  • You can name it in one phrase («Backlog cockpit», «User profile», «Settings · security»).
  • You've opened handbook/sa/templates/screen-spec.html.

Steps

  1. Place the file under internal/reference/front/screens/<screen-slug>.html.
  2. Fill the metadata — Screen · Owner · Last reviewed · Live URL.
  3. Add a visual gallery at the top — desktop-light, desktop-dark, mobile.
  4. Document anatomy — labelled regions: header, primary content, sidebar, footer, modals/overlays.
  5. Draw a layout map — grid columns, breakpoints, where each region lives at each breakpoint.
  6. Build the behavior matrix — interactive states for every component on the screen.
  7. Specify the timing budget — first paint target, time-to-interactive target.
  8. Document the data contract — what the screen reads / writes, request shape, error shape.
  9. Specify responsive behaviour — what changes at each breakpoint.
  10. List browser & device support.
  11. Specify accessibility & motion — keyboard map, focus order, reduced-motion overrides.
  12. Add Do's and Don'ts.

Verify

  • 17 canonical sections present in order (see the template).
  • Visual gallery has at least three viewports.
  • Behavior matrix covers every component on the screen.
  • Timing budget has concrete numbers.
  • Accessibility section covers keyboard + focus + screen-reader.

Troubleshooting

  • «17 sections feels excessive». Screens are the heaviest artefact; if 12 of 17 sections are blank, the page is probably a component, not a screen.
  • «Visual gallery drifts». Re-shoot every cycle. Stale screenshots are worse than no screenshots.
