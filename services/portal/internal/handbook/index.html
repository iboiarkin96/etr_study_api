Engineering handbook

Overview

Per RFC 0004, the engineering handbook consolidates the team-facing process and reference content that used to live at services/portal/ root (developer/, howto/, qa/) into one navigable hub under services/portal/internal/handbook/. The audience is the team — anyone shipping code, writing docs, or running quality checks on this repo.

Sections

  • Developer docs — engineering specifications: requirements, schemas and contracts, business logic, error matrices, load testing, local development, the docs pipeline, the container image, and the Make-target catalogue.
  • How-to guides — step-by-step contributor recipes: onboarding from zero to your first endpoint, internal service docs layout, the Make-commands inventory, adding a POST contract, and changing the docs frontend safely.
  • QA — quality-assurance checklists and playbooks: tester onboarding, smoke / endpoint / regression / accessibility / visual / 5xx checks.

Where to write what

Use the matching tree per artifact type so the corpus stays predictable:

  • Architectural decision → ADRs (governance — root services/portal/internal/governance/adr/ until Phase F).
  • Implementation specification → RFCs (governance).
  • Engineering reference page (requirement / schema / contract / process) → handbook/developer.
  • Step-by-step recipe for a contributor → handbook/howto.
  • QA checklist or playbook → handbook/qa.
  • Operational procedure for an alert / incident → internal/sre/runbooks.
  • Documentation principle (the why behind a convention) → SA portal — documentation principles.
