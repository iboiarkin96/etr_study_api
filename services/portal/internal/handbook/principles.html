Documentation principles

Five principles govern every page in this portal: Diátaxis structure, plain-language voice, Docs as Code, PlantUML for diagrams, and internal-first source of truth. Each one is a hard rule, not a guideline.

1 — Diátaxis structure

Every page belongs to exactly one of four quadrants. If you cannot decide the quadrant, the page is not ready to be written. The quadrants are defined by what the reader is trying to do, not by topic.

QuadrantReader isPage answersLives underExample
Tutorial Learning «Walk me through it the first time» tutorials/ Your first API checklist run
How-to Doing «How do I accomplish X?» how-to/ Run an exploratory testing session
Reference Looking up «What is the contract / shape / value?» reference/ API operation spec, DB-table schema
Explanation Understanding «Why is it like this?» explanation/ Test pyramid & layer ownership

Three additional buckets sit outside the quadrants because they cross-cut them:

  • governance/ — normative decisions (ADR, RFC, audits). Prescriptive, not descriptive.
  • services/ — per-service hubs (architecture · API · runbooks · dependencies · on-call) that link into all four quadrants.
  • team/ — role-shaped onboarding hubs and practice pages. Deliberate hybrid of tutorial + how-to + explanation; reader is a person growing into a role.

Anti-patterns: a how-to that drifts into explaining rationale (move the rationale to explanation/). A reference page that teaches (move the teaching to tutorials/). A tutorial that lists every possible flag (link to reference/).

2 — Plain-language voice

Five rules. Apply all of them; do not pick.

  1. Short sentences. Aim for one idea per sentence; cut clauses joined by and / but / which.
  2. Active voice. «The runtime renders the sidebar», not «the sidebar is rendered by the runtime».
  3. Second person and imperative for tutorials and how-tos. «Open the file», «You will see».
  4. One name per concept. Pick the term (e.g. page-head) and use it everywhere. Note synonyms in the glossary only.
  5. Sentence case in headings and labels. Preserve acronym casing: API, AsyncAPI, SRE, gRPC, REST, SQL, ADR, RFC.

The full editorial contract — page skeleton, code-block conventions, table conventions, markup hooks — lives in the Style guide.

3 — Docs as Code

Documentation lives in the application repository, ships through the same pipeline, and is reviewed in the same pull requests. There is no separate documentation system, no separate review workflow, and no export step. See ADR 0001 for the rationale and tooling decisions.

Practical consequences:

  • A code change that breaks a documented behaviour must update the documentation in the same PR.
  • Every doc page goes through make docs-fix (UML render → marker sync → HTML format → pdoc → search index) before merge.
  • CI gate make docs-check blocks merge if any of the above is out of date.
  • Every page records its history inline (date + author + one-line note) — no external changelog.

4 — PlantUML only for diagrams

Every diagram in this portal is authored as a PlantUML source under uml/ and rendered to SVG into rendered/. Mermaid, draw.io, screenshots-as-diagrams and inline ASCII are not accepted. See ADR 0020 for the C4 levels and stencil choices.

Diagram lifecycle: write .puml → run make uml → commit both the source and the rendered SVG. The SVG is the artefact the page imports; the .puml is the artefact reviewers diff.

5 — Internal portal is the source of truth

Per ADR 0026, the internal portal is the canonical version of every service's documentation. Public-facing docs (when they exist) derive from these — never the reverse.

This is why the handbook lives here, not in docs/ at the repository root. The handbook is for the people who write the docs, and those people work inside the internal portal.

How to use these principles

When you sit down to write or review a page, walk this checklist:

  1. Which Diátaxis quadrant is this? If unclear → split or rescope.
  2. Does the voice match the five rules? Read the first paragraph aloud.
  3. Are diagrams PlantUML-sourced with committed SVGs?
  4. Did the related code change land in the same PR?
  5. Did you copy a template rather than start from scratch?
