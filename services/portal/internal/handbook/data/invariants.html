Data invariants — what must hold in the database, always

Cross-cutting rules the database is required to satisfy regardless of which endpoint just wrote to it. Per-table schemas (under data reference) describe each table in isolation; this page describes what holds between tables and what the application layer is responsible for enforcing because the database alone cannot. Every invariant says where it is enforced (FK, UNIQUE, CHECK, app-layer) and what happens if the constraint is somehow violated.

How to read this page

  • An invariant is a rule that must be true at every commit boundary. Mid-transaction states are allowed to violate; final committed states are not.
  • The enforcement column names the line of defence. FK = foreign key constraint; UNIQUE = unique constraint or index; CHECK = column-level CHECK clause; app-layer = the rule is checked in Python before the write, with no DB-side enforcement; convention = the rule is documented but not mechanically enforced.
  • Breakage column names what the user-visible response would be if the invariant were violated. data corruption means there is no graceful path — the row graph becomes inconsistent.

Ownership invariants

Every domain row that can be read or written belongs to exactly one user. The ownership pointer is owner_client_uuid; cross-tenant access is detected here and collapsed to 404 per _shared auth § why-404.

Schedule invariants

Scheduling state is separate from content. The two evolve under different revision counters and are touched by different endpoints.

#InvariantEnforcementBreakage
S-01schedule_revision is strictly monotonic per row; never decreases.app-layer (review action writes current + 1)data corruption
S-02A PATCH to content never touches schedule_revision or next_review_at.app-layer (PATCH router; tested in PATCH spec AC-006)silent schedule drift; user retention regresses without explanation
S-03next_review_at is always in the future relative to the row's last schedule_updated_at (no past-due states persisted at write time).conventionUI shows due-now items that should not yet be due; mild bug, not corruption
S-04A successful review action writes both a conspectus_events row (SCHEDULE_ADVANCED) and updates the schedule fields atomically.app-layer (single transaction in ReviewService.apply)audit gap: postmortem replay cannot reconstruct the slot ladder progression

Idempotency invariants

Idempotency rules apply to every successful write across every endpoint that requires Idempotency-Key.

Where DB-side enforcement is absent

Several invariants above rely on the application layer to enforce them. On SQLite this is acceptable because there is one writer; on Postgres or any multi-writer storage the same rules should be lifted to DB-side checks where possible. Listed below for the future-Postgres reviewer.

  • C-01 (monotonic content_version) — should become a CHECK or a trigger on Postgres. Today, two concurrent writers could theoretically write the same version; SQLite's single-writer serialises this away.
  • S-01 (monotonic schedule_revision) — same shape as C-01; same lift on Postgres.
  • K-04 (replay rejects on hash mismatch) — current implementation reads-then-writes; needs SELECT … FOR UPDATE on Postgres to avoid a TOCTOU window. SQLite's single-writer makes this impossible today.
  • O-03 (body never sets ownership) — pure convention. Adding a row-level security policy in Postgres would mechanically enforce this rather than relying on every router doing the right thing.

Forbidden patterns

  • "The constraint is enforced in the spec." A spec is documentation, not enforcement. Name the enforcement point — FK, UNIQUE, CHECK, app-layer, or convention — explicitly. Conventions are the weakest line of defence.
  • Lowering an invariant to app-layer "because it's simpler". App-layer enforcement is what fails first under concurrency and migration. Don't lower a constraint that the DB can carry.
  • Adding a row without naming its invariants. A new table's spec page must list the invariants it participates in here. If it doesn't have any cross-table rules, say so explicitly.
  • Invariant drift. A constraint removed in a migration but left listed here is worse than not listing it at all. Migrations that drop a constraint must update this page in the same PR.
