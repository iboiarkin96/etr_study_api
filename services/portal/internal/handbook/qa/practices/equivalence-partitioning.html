Equivalence Partitioning — reduce redundant tests

Divides the input space into classes where one representative value per class gives the same defect coverage as testing every value in the class.

TL;DR

Equivalence Partitioning (EP), formalised alongside BVA by Glenford Myers, divides the input domain into classes (partitions) where all values are expected to behave identically. Testing one representative per partition is sufficient — additional values in the same partition add effort without adding defect coverage. EP is most powerful when combined with BVA: EP identifies the classes, BVA tests the boundaries between them. Invalid partitions are as important as valid ones.

What it is

Equivalence Partitioning is a black-box test design technique that divides the input domain of a component or system into groups (equivalence classes or partitions) such that all values within a group are assumed to behave identically with respect to the software under test. Testing one representative value from each partition is considered equivalent to testing all values in that partition.

The technique was formalised by Glenford Myers alongside BVA in 1979. It addresses a fundamental efficiency problem: the input space for most software components is infinite (or at least too large to test exhaustively). EP reduces the infinite input space to a small, finite set of representative test cases without sacrificing defect coverage.

Partitions come in two categories:

  • Valid partitions: Inputs that the system should accept and process correctly. At least one representative from each valid partition must be tested.
  • Invalid partitions: Inputs that the system should reject. At least one representative from each invalid partition must be tested — they verify the error handling path.

For a text field accepting 1–255 characters, the partitions are: EP1 (invalid: length < 1, representative: 0 or empty string), EP2 (valid: 1 ≤ length ≤ 255, representative: any value like 128), EP3 (invalid: length > 255, representative: 300). Three test cases instead of 256.

EP is most powerful when combined with BVA. EP identifies the partitions and their boundaries; BVA then tests the boundary values between partitions. Together they provide systematic coverage of the input domain with a minimal set of test cases.

Why we need it

Without EP, testers either test too few input values (missing entire defect classes) or too many (redundant tests that add effort without adding coverage). Both failure modes reduce quality per hour of test effort.

The invalid partition is the most commonly missed. Teams test that the form accepts valid input but don't verify that it correctly rejects invalid input. EP makes invalid partition testing mandatory — every partition, valid or invalid, requires at least one representative test.

How others do it

Glenford Myers (originator)

Myers introduced EP in The Art of Software Testing alongside BVA. His key insight was that within any equivalence class, the program either works for all values or fails for all values — one test is sufficient to determine which. He emphasised that invalid partitions are equally important to valid ones and that the most common mistake is testing only within the valid range.

What we take: Always identify invalid partitions explicitly. The invalid-input path is tested just as rigorously as the happy path.

Source: ISTQB FL Syllabus

ISTQB

ISTQB's Foundation Level defines EP as a core black-box technique and requires candidates to be able to identify equivalence classes from a requirements description. The ISTQB Advanced Test Analyst adds the concept of "equivalence class combination" — when multiple inputs each have partitions, the combinations also form partitions that need representative test cases. ISTQB certification includes EP questions in every sitting.

What we take: For endpoints with multiple input parameters, apply EP to each parameter independently first, then consider representative combinations for the highest-risk scenarios.

Source: ISTQB — Foundation Level

Atlassian

Atlassian's engineering blog describes how their QA teams use EP for API testing: each API parameter is partitioned into valid and invalid classes, and the test suite must include at least one test case from each partition. For their Jira and Confluence APIs, they maintain partition maps as part of the API documentation — a developer reading the spec can immediately see which test cases are expected for each parameter.

What we take: Embedding partition maps in the API documentation so they are visible to developers during implementation, not just to QA during testing.

Source: Atlassian — Jira development documentation

Anatomy

Partition identification

For each input, list all valid and invalid partitions. Invalid partitions must be listed explicitly — not as an afterthought.

Representative values

One representative value per partition. For valid partitions, choose a value clearly within the partition (not near a boundary). For invalid partitions, choose a clearly invalid value.

Expected outcomes per partition

For each partition: what the system should return. Valid partitions should be accepted; invalid partitions should return a specific error code.

Combination analysis

For endpoints with multiple parameters: identify the highest-risk partition combinations and add representative test cases for them.

Common pitfalls

PitfallWhy it happensWhat to do instead
Missing invalid partitions Testers identify valid partitions and write tests for them, but don't explicitly identify or test invalid partitions. Make invalid partition identification a required step. For every valid partition there is at least one corresponding invalid partition — it must have a test.
Too many representatives per partition Testers don't trust the EP principle and test multiple values per partition "just to be safe." Trust the technique. One representative per partition is sufficient. Use the saved test time for BVA (boundary values) which catches the defects EP misses.
Ignoring partition combinations Each parameter is partitioned independently, but no test cases cover combinations of invalid inputs. For the highest-risk combinations (e.g. both parameters invalid simultaneously), add explicit combination test cases. Pairwise testing tools can generate these efficiently.
