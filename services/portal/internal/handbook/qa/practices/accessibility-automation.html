Accessibility Test Automation — embed a11y in CI

Runs axe-core on every CI build to catch WCAG 2.1 AA violations automatically before they reach users.

TL;DR

Accessibility test automation combines axe-core (automated rule engine) with keyboard and screen-reader testing against WCAG 2.1 Level AA. axe-core runs in CI and catches approximately 30–40% of WCAG failures automatically. Critical violations fail the build; serious violations raise warnings. The remaining 60–70% require manual verification guided by checklist 0005. axe catches missing alt text, duplicate IDs, colour contrast failures, and missing form labels — the most common classes of accessibility defects.

What it is

Accessibility test automation is the practice of running automated tools against the rendered UI to detect violations of WCAG 2.1 Level AA accessibility guidelines. The most widely used tool is axe-core (developed by Deque Systems), a JavaScript library that analyses the DOM and reports accessibility violations as structured JSON.

axe-core is integrated into the CI pipeline and runs on every build against the rendered portal pages. Violations are classified by impact:

  • Critical: Violations that completely prevent users with disabilities from accessing content (e.g. missing alt text on informational images, no focusable elements). Critical violations fail the CI build.
  • Serious: Violations that significantly impair access (e.g. insufficient colour contrast, missing form labels). Serious violations raise a warning and must be resolved before release.
  • Moderate / Minor: Violations that create a poor experience but don't completely block access. Tracked as backlog items.

axe-core catches approximately 30–40% of WCAG 2.1 Level AA failures automatically. The remaining 60–70% require manual testing: keyboard-only navigation testing (can every interactive element be reached and operated with only the keyboard?), screen reader testing (do NVDA/JAWS/VoiceOver users get correct, meaningful information?), and cognitive accessibility testing (is the content readable and navigable without visual cues?). These are guided by checklist 0005.

Why we need it

Accessibility failures are legal liabilities in most jurisdictions. The European Accessibility Act (EAA) requires digital products to meet EN 301 549 (which references WCAG 2.1 Level AA) for all products sold in the EU market by June 2025. In the US, the ADA has been applied to digital products through court precedent. An inaccessible product exposes the company to legal action, regulatory fines, and exclusion from procurement processes that require WCAG compliance.

Beyond legal risk, accessibility affects 15–20% of the global population. Automated scanning catches the most common violations (missing alt text, duplicate IDs, colour contrast failures, missing form labels) before any human tester is involved. Catching these in CI is far cheaper than fixing them post-release.

How others do it

Deque / axe-core (tool inventors)

Deque Systems created axe-core, which has become the de facto standard for automated accessibility testing. axe-core is used by Google Chrome DevTools, Microsoft's accessibility insights tool, and most major CI accessibility integrations. The axe-core rule set is maintained by the accessibility community and covers over 90 WCAG 2.1 success criteria. Deque provides axe-playwright (for Playwright), axe-selenium (for Selenium), and axe-react (for React testing) integrations.

What we take: axe-core is the standard. Use it in CI, not just in DevTools. The CI integration provides evidence of accessibility testing for compliance audits.

Source: Deque — axe-core

W3C / WAI (WCAG standards body)

The Web Accessibility Initiative (WAI) at W3C publishes WCAG 2.1, the normative standard for web accessibility. WCAG 2.1 Level AA is the most commonly required conformance level in legislation and procurement. WAI also publishes ARIA 1.1 (Accessible Rich Internet Applications), which defines how to make dynamic content accessible using semantic attributes. The WAI-ARIA specification is the foundation for the ARIA-related rules in axe-core.

What we take: WCAG 2.1 Level AA is the target conformance level. Passing axe-core is necessary but not sufficient — the standard requires manual testing for subjective criteria.

Source: W3C — WCAG 2.1

Microsoft (Inclusive Design)

Microsoft's Inclusive Design framework and Accessibility Insights tool extend automated accessibility testing with a structured manual testing workflow. Accessibility Insights provides two modes: FastPass (automated axe scan, takes 5 minutes) and Assessment (guided manual testing, takes 2–3 hours per page). Microsoft requires FastPass for every PR that touches UI; Assessment for every major feature before release. Their "accessibility fundamentals" training programme is publicly available and trains both developers and QA engineers.

What we take: FastPass (automated scan) on every PR; Assessment (guided manual) before every major release. The two levels of testing are complementary, not alternatives.

Source: Microsoft — Accessibility Insights

Anatomy

axe-core CI integration

axe-core runs on every CI build against the rendered portal pages. Critical violations fail the build; serious violations raise warnings.

Violation severity policy

Documented policy: critical = build fail; serious = warning + tracked issue; moderate/minor = backlog. No exceptions without a documented waiver.

Manual testing checklist (0005)

Checklist 0005 guides manual keyboard and screen-reader testing for the 60–70% of WCAG criteria that axe cannot automate.

WCAG conformance target

WCAG 2.1 Level AA. The target is documented and referenced in the test strategy so it is not ambiguous which standard is being applied.

Common pitfalls

PitfallWhy it happensWhat to do instead
Treating axe as sufficient Teams run axe, it passes, and they declare the product accessible. axe only catches 30–40% of WCAG failures. Explicitly document that axe is a first filter, not a conformance proof. Checklist 0005 is required for WCAG conformance claims.
Suppressing violations instead of fixing them axe violations block the build and developers suppress them to unblock themselves, with the intention of fixing them later. Require a documented waiver for any suppressed violation. Waivers expire — they must be reviewed and re-approved each quarter.
No keyboard testing axe cannot test keyboard navigability — it cannot verify that all interactive elements are reachable by Tab key only. Include keyboard navigation testing in checklist 0005 for every major feature. Keyboard testing is the highest-value manual accessibility test.
