Boundary Value Analysis — test the edges

Systematically tests the values at and immediately outside the edges of valid input ranges, catching off-by-one errors that random testing misses.

TL;DR

Boundary Value Analysis (BVA), developed by Glenford Myers in 1979, tests the minimum, minimum+1, maximum-1, and maximum values of any bounded input domain — because defects cluster at boundaries, not in the middle. A field that accepts 1–255 characters is most likely to fail at 0, 1, 254, 255, and 256. BVA makes this systematic: every bounded field in the application has a BVA test set that covers these five values at minimum.

What it is

Boundary Value Analysis is a black-box test design technique that selects test values at and immediately around the edges of valid input partitions. The technique was described by Glenford Myers in his 1979 book The Art of Software Testing and remains one of the most effective techniques for finding a specific class of defects: boundary errors.

For any input with a defined valid range [min, max], BVA selects the following test values:

  • min − 1: One below the minimum (invalid input — should be rejected)
  • min: The minimum valid value (should be accepted)
  • min + 1: One above the minimum (should be accepted)
  • max − 1: One below the maximum (should be accepted)
  • max: The maximum valid value (should be accepted)
  • max + 1: One above the maximum (invalid input — should be rejected)

The rationale is that defects are most likely to occur at the boundaries of a domain, not in the middle. A field that accepts 1–255 characters is almost never wrong at 128 characters, but it is frequently wrong at 0, 255, and 256 — the points where the implementation must handle the transition from valid to invalid (or vice versa).

BVA applies to any bounded domain: numeric ranges, string lengths, list sizes, date ranges, price ranges, and array indices. It is used in combination with equivalence partitioning — EP identifies the partitions, BVA selects the test values within and around the partition boundaries.

Why we need it

Off-by-one errors are among the most common defect classes in software. A loop that runs one iteration too many, a validation that accepts 256 characters when the database column is 255 characters wide, a date range that excludes the last day — all of these are boundary defects that random or middle-of-range testing misses entirely.

Without BVA, test case selection is intuitive — testers pick values that "feel right" but systematically avoid the edges where defects actually live. BVA makes boundary testing mandatory and systematic, so the defect class cannot be skipped by accident.

How others do it

Glenford Myers (originator, 1979)

Myers introduced BVA in The Art of Software Testing as a complement to equivalence partitioning. His original insight was that test cases derived from the boundaries of equivalence classes are more productive than test cases derived from the interior. He demonstrated that boundary errors account for a disproportionate share of all defects in programs with numeric computations, string processing, and array manipulation.

What we take: BVA and EP are always used together — EP identifies the partitions, BVA selects the boundary test values. Neither technique alone is sufficient.

Source: ISTQB FL Syllabus (BVA section)

ISTQB

ISTQB's Foundation Level syllabus defines BVA as one of the four core black-box techniques (alongside EP, decision tables, and state transition testing). The ISTQB 2-value BVA selects min and max from each partition; the 3-value BVA (required for ISTQB Advanced) also selects min−1, min+1, max−1, max+1. ISTQB certification exams include BVA questions in every sitting — it is the most frequently examined technique.

What we take: Use 3-value BVA for high-risk fields (the full six-point set); use 2-value BVA for low-risk fields (just min and max).

Source: ISTQB — Foundation Level

Google (Testing at Google)

Google's engineering culture embeds BVA implicitly through their testing guidelines, which require that any function with a numeric parameter includes tests at the minimum, maximum, and one-past-maximum values. The "Fuzz testing" programme at Google is partly an automated form of BVA at scale — the fuzzer is particularly good at finding boundary values that human testers miss. Google's internal test review process flags test suites that lack boundary coverage as incomplete.

What we take: For high-risk fields, complement manual BVA with automated property-based testing or fuzzing — they find boundaries that human analysis misses.

Source: Software Engineering at Google

Anatomy

Input domain identification

For each test area, list all bounded inputs: numeric ranges, string lengths, date ranges, list sizes, enum values.

Boundary value set

For each bounded input: min−1, min, min+1 (lower boundary) and max−1, max, max+1 (upper boundary). Six values per range.

Expected outcomes

For each boundary value: what the system should return (accept and process, reject with a specific error code, or return a boundary-specific result).

Risk-tiered coverage

High-risk fields use 3-value BVA (full six-point set); low-risk fields use 2-value BVA (min and max only).

Common pitfalls

PitfallWhy it happensWhat to do instead
Testing only the valid range Testers test min and max but not min−1 and max+1, missing the invalid-input boundary defects. Always include min−1 and max+1 in the BVA test set. Invalid boundary values are as important as valid ones — they verify rejection behaviour.
BVA without defined expected outcomes The tester knows the boundary values to test but not what the system should return for each. Define expected outcomes before executing. "The API should return 422 with error code FIELD_TOO_LONG when the name field exceeds 255 characters" — not "the API should reject it."
Skipping non-obvious boundaries Testers apply BVA to numeric fields but forget about string length limits, list size limits, and date range boundaries. Include all bounded domains in the BVA analysis: numeric values, string lengths, array/list sizes, date ranges, and pagination limits.
