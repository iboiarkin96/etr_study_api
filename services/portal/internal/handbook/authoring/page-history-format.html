Page history format

Every hand-written page ends with a Page history footer. One row per substantive edit. Skip typo-only changes. This is the audit trail without leaving the page.

Markup

<footer class="docs-history" aria-label="Page history">
  <h2 class="docs-history__title">History</h2>
  <ul class="docs-history__list">
    <li class="docs-history__row">
      <span class="docs-history__date">2026-05-18</span>
      <a class="docs-history__author"
         data-component="author-chip"
         data-person-id="16fc8b78537109162984a2fdbef6e142"
         data-variant="sm">@ivan</a>
      <span class="docs-history__note">One-line note describing the substantive change.</span>
    </li>
  </ul>
      </footer>

Rules

  • Date — ISO format YYYY-MM-DD, the date of the edit (not the file's last-modified timestamp).
  • Author — wrap with the author-chip component; the runtime resolves the chip from data-person-id. Use the maintainer's UUID from team/people/. If unknown (older rows), use «—» as plain text.
  • Note — one sentence, present tense, focuses on what changed. Use <code> spans for symbol/class names. Avoid filler ("updated the page", "minor changes").
  • Order — newest first, top of the list. The list reads top-down as a reverse chronological log.
  • What counts as a substantive edit: new section, removed section, rewritten paragraph, changed contract, migration to a new component. Not: typo fixes, whitespace, lint cleanups, formatting passes by make docs-fix.
  • Maintainer ID — also set on <body data-maintainer-ids="..."> on first authorship. Add additional IDs separated by spaces if a page becomes co-maintained.

Examples

Good:

<span class="docs-history__note">Migrated to UI Kit v3: <code>docs-prose</code> + <code>page-head</code>; sections become h2 with id anchors.</span>

Bad:

<span class="docs-history__note">Updated.</span>
