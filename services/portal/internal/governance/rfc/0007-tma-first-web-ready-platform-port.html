RFC · 0007 Proposed Program: Telegram Mini App

TMA-first, web-ready — a platform port for the Mini App

The Mini App ships inside Telegram's WebView today, but the product roadmap wants a standalone web version installable on a phone as a PWA. Those are the same React application with two entry doors — if every Telegram-specific capability sits behind one boundary. This RFC defines that boundary — a src/platform/ port with a Telegram adapter now and a reserved slot for a Web adapter later — plus the auth and identity invariants on the API side that keep the second door cheap. It changes no user-visible behaviour.

Goals and non-goals

Goals

  • Quarantine every Telegram dependency into one module so the rest of the app is platform-neutral.
  • Fix the auth/identity invariants that make a second login door (web) an additive change, not a migration.
  • Make the boundary enforceable by lint/CI, not by code-review vigilance.

Non-goals

  • Building the web version or the PWA shell (manifest, service worker, web push) — that is a future epic; this RFC only keeps its cost low.
  • Choosing the web login method (email code vs Telegram Login Widget vs OAuth) — deferred to the door's own design slice.
  • Backend hosting — unchanged, tracked separately in the TMA epic.

Background

A Telegram Mini App is a web application: Telegram opens our static bundle in a WebView and hands it a bridge — signed initData for login, themeParams, chrome buttons, haptics. A PWA is the same bundle opened in a browser with an install wrapper. The delta between the two is therefore not «two apps» but «one app, two implementations of a small capability set». The current W1 codebase (see telegram · architecture) calls the Telegram SDK directly from providers and screens; left as-is, every new screen deepens the coupling and the future web epic becomes a rewrite instead of an adapter.

Design

One rule, one folder. All platform capabilities are consumed through a typed port; the Telegram implementation is the only code allowed to import @telegram-apps/* or touch window.Telegram. Full specification in the sections below.

Implementation

  1. Phase 1 (inside the current epic): introduce src/platform/, move existing call-sites behind the port, add the ESLint fence. No behaviour change.
  2. Phase 2 (ongoing): all new screens and features consume ports only — enforced by the fence.
  3. Phase 3 (future web epic): add src/platform/web/, POST /api/v1/auth/web, manifest + service worker.

Validation

  • Local checks: make -C services/telegram verify stays green; the ESLint fence fails on any direct SDK import outside the adapter.
  • CI / automated checks: existing lint / type / build gates; bundle budget unchanged (~46 KB gz — the port is types plus thin re-exports).
  • Operational checks: app still boots both in Telegram and in the plain-browser dev loop (VITE_DEV_INIT_DATA signer).

Rollout plan

  • Rollout phases: Phase 1 is a single refactor PR; Phases 2–3 ride their own epics.
  • Feature flags: none — the adapter is resolved once at boot by environment detection.
  • Rollback strategy: revert the refactor PR; call-sites return to direct SDK usage.
  • Backward compatibility: no wire-level or user-visible change; POST /api/v1/auth/telegram is untouched.

Change management

  • Update this RFC when: a port is added/removed or an invariant (R1–R6) changes.
  • Open a new RFC when: the web door's login method is chosen, or the one-bundle-two-doors model itself is abandoned.

The platform port

A single module owns the boundary. Everything above it imports platform and stays ignorant of where it runs; everything below it is per-platform and disposable.

Two doors, one token

The API side already has the right shape — this RFC freezes it as an invariant rather than an accident. users rows are keyed by an internal client_uuid; the Telegram identity is one linked login system under a composite (system_uuid, system_user_id) key. A future POST /api/v1/auth/web is therefore purely additive: it verifies its own credential kind, links or upserts a login row, and mints the same 24 h JWT with sub = client_uuid. Every other endpoint keeps seeing one token format and never learns which door the caller used. 

Telegram WebView ── initData ──► POST /auth/telegram ─┐
                                                      ├─► same JWT (sub = client_uuid) ─► /api/v1/*
Browser / PWA ── web credential ► POST /auth/web ─────┘            (door-agnostic)

Normative rules

#RuleRationale
R1 No import of @telegram-apps/* and no reference to window.Telegram outside src/platform/telegram/. The whole RFC in one line. Everything else is mechanism.
R2 Every login door exchanges its platform credential for the same JWT shape (sub = client_uuid, 24 h). Doors never mint door-specific tokens. Keeps every non-auth endpoint door-agnostic forever.
R3 Product tables reference users.client_uuid only; telegram_user_id lives solely in the login-link table. A user who later adds an email login keeps one account and all data.
R4 Components read the app's own tokens (--tma-ember-*, --tma-sage-*, motion springs). --tg-* vars are written only by platform/*/theme.ts and read only by base surfaces. The visual identity survives outside Telegram unchanged.
R5 The router knows only internal routes. Entry payloads (startapp today, URL paths tomorrow) are normalised by platform.links.parseEntry() at boot. One deep-link grammar per platform, zero grammar in screens.
R6 Analytics and API payloads never carry platform identifiers (telegram_user_id, usernames). distinct_id stays client_uuid. Already the event-spec invariant; restated so the web door inherits it.

Enforcement — a fence, not a convention

R1 is machine-checked in the existing lint gate; the rest ride code review plus contract tests already planned in the epic.

// eslint.config.js — services/telegram
{
  files: ['src/**'],
  ignores: ['src/platform/telegram/**'],
  rules: {
    'no-restricted-imports': ['error', { patterns: [
      { group: ['@telegram-apps/*'], message: 'Import via src/platform — see RFC 0007 (R1).' },
    ]}],
    'no-restricted-globals': ['error',
      { name: 'Telegram', message: 'Use src/platform — see RFC 0007 (R1).' },
    ],
  },
}

What the web epic adds later — and nothing more

With the port in place, the standalone web/PWA version is a bounded list. Each row becomes a backlog item in a future epic; none requires touching screens, state, styles, or the API surface beyond one new door.
