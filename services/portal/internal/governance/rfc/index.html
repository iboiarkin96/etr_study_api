Requests for Comments

RFCs spell out how we build and run things — implementation detail, design rationale, and operational steps. ADRs record what we decided and why; an RFC usually follows an ADR and provides the specification the ADR intentionally omits. New RFCs start from the RFC template and follow the lifecycle in ADR 0018.

Documentation authoring model

How we document API endpoints, screens, ADRs, runbooks, and how-tos: the closed taxonomy of document types, the section list of every template, the industry programs whose patterns we adopted, and the lint / audit / Definition-of-Done gates that turn the templates into a contract.

RFC 0003

Documentation authoring model

Proposed
Taxonomy, API spec template, screen spec template, why each section exists (Stripe / Atlassian / GitHub Primer / IBM Carbon / Material / Storybook / GOV.UK / IETF / C4), and copy-pasteable section catalogues in Appendix A and B. Companion to ADR 0001 (docs as code), ADR 0025 (external/internal API docs), and ADR 0026 (internal service docs SoT).

Documentation portal information architecture

How the services/portal/ tree splits into a public layer for external integrators (Diátaxis quadrants) and an internal layer for the team (Backstage IDP shape with a Google-SRE-Book-shaped operational hub), with the audience boundary expressed as an IA boundary rather than an auth boundary (GitLab-handbook pattern). Amends ADR 0028's governance-at-root rule from a specific path list to "outside services/".

RFC 0004

Public vs internal portal IA

Proposed
Framework choices, comparative analysis (Stripe / Twilio / GitHub / Cloudflare / AWS / GCP / Spotify / GitLab), data-audience / data-doc-type conventions, six-phase migration plan, and the ADR 0028 amendment.

Docs Search program

Client-side full-text search across all documentation pages. The program covers the index build pipeline, query-time ranking, telemetry, and SLO targets. The architecture decision is in ADR 0027.
