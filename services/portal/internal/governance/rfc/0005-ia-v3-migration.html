RFC · 0005 Proposed

Multi-service IA v3 — entity taxonomy, per-type spine, SA-genres, Foundations, Practices

A growable information architecture for the internal portal: replace the flat services/ directory with a Backstage-subset entity catalog (System / Component / Library / API / Resource), give each entity type its own normative spine matrix, surface the four missing systems-analyst genres (Requirements / Domain / Process / NFR) inside every Component, add a Foundations layer for cross-cutting concepts that ≥ 2 entities consume, and a Practices layer for cross-service ways of working (testing pyramid, docs-as-code, observability discipline, etc.). The scaffold is sized to scale from 4 to 50+ entities without structural rework. Supersedes the flat-services layout described implicitly by ADR 0026 and refines the public/internal split landed in RFC 0004.

Goals and non-goals

Goals.

  1. Replace the flat services/{api,portal,monitoring,ui-kit}/ with an entity-typed catalog: System / Component / Library / API / Resource. Each type has its own spine; library cannot demand a runbook, API cannot demand on-call, Component must have both.
  2. Close the SA blind spot. Every Component carries four normative SA pages: Requirements & use cases, Domain model, Process documentation (BPMN), NFRs & SLOs.
  3. Add a Foundations layer for content that ≥ 2 entities consume — Auth, Errors, Observability, Versioning — so services link instead of duplicating.
  4. Add a Practices layer for cross-service ways of working (testing pyramid, docs-as-code, observability discipline, on-call rotation, postmortem culture, branching & PR review, API versioning practice, security baseline). Each practice carries an adoption matrix back-indexed from each service's catalog-info.yaml.
  5. Make growth mechanical: adding a 5th, 10th, 50th entity should be copy a template, fill the spine, register in catalog-info, push, CI checks the spine. No structural rework when count grows.
  6. Make freshness visible. Surface lifecycle and last-reviewed on every entity page, not only in catalog-info.
  7. Preserve everything that currently works: the 31 ADRs, RFC 0003 authoring model, PlantUML diagram contract, history convention, and the 11 pre-commit doc hooks.

Non-goals.

  • Not switching source format from HTML to Markdown. Tooling decision is deferred to Phase 4 if and only if HTML hand-authoring becomes a measured bottleneck.
  • Not changing the public/internal split. RFC 0004 governs that; this RFC operates strictly inside services/portal/internal/.
  • Not changing the diagram tool. ADR 0020 stands: PlantUML only.
  • Not introducing a new auth boundary. The internal portal remains internal-only per RFC 0004.
  • Not building a Backstage instance. We adopt the catalog data model, not the runtime.

Current state — what we have today

The portal sits at 5.6 / 10 on the 10-axis IA rubric used for this RFC (full audit attached to the PR description). Strengths: real Diátaxis at the top level, dense governance (31 ADR / 4 RFC), mature docs-as-code (11 pre-commit hooks including check-asset-refs, check-h1-structure, catalog-render-check, check-adr-rfc-links). Weaknesses: flat services directory with one-spine-fits-all; SA genres entirely absent; cross-cutting concepts living inside ADRs without a usable reference layer; last-reviewed recorded but never rendered; no scorecards.

A sandbox at services/portal/internal/_preview/ (created 2026-05-18) prototypes the entity-typed catalog, the per-type spine, the onboarding journey, and the scorecards UI. Target IA — eight top-level sections

The internal portal restructures into eight top-level sections:

services/portal/internal/
├── onboarding/        # Day 1 / Week 1 / Month 1 + optional by-role checklists
├── catalog/           # Renamed from services/ — entity-typed catalog
│   ├── systems/       # 1+ umbrellas (owns C4 L1, portal-level ADRs)
│   ├── components/    # Deployable services (4 today, scales to 50+)
│   ├── libraries/     # Non-deployable code (ui-kit, py-sdk)
│   ├── apis/          # First-class contracts (OpenAPI / AsyncAPI / proto)
│   └── resources/     # Infrastructure (databases, buckets, queues)
├── foundations/       # Cross-cutting content (used by ≥ 2 entities)
│   ├── shared-concepts/   # Auth · Errors · Observability · Versioning
│   ├── tutorials/         # Portal tour and other cross-cutting learners
│   ├── how-to/            # Register a Component/Library/API/Resource, write an ADR, deprecate
│   ├── reference/         # Entity descriptor schema, spine matrix, style guide, templates index
│   ├── explanation/       # IA model, taxonomy rationale, Diátaxis blind-spot coverage
│   └── glossary.html      # Single glossary page — every term links here
├── practices/         # Cross-service ways of working (testing pyramid, docs-as-code, …)
├── governance/        # Unchanged: ADR · RFC · Audits · Backlog
├── operations/        # Teams · SLO dashboard · Incidents · Tech radar · On-call schedule
├── handbook/          # "How to author for this portal" (handbook v2)
└── team/              # Redirect-only stub once team/ folds into operations/teams/

The boundary between where a thing lives is formal, not folkloric. The criterion is recorded in foundations/explanation/ia-model.html (created in Phase 1 Step 1.2 alongside the spine matrix):

  • Catalog — material that belongs to exactly one entity (one Component, one API, one Resource).
  • Foundations — material consumed by ≥ 2 entities and with no single entity owner (Auth is consumed by every Component; it isn't owned by any one of them).
  • Practices — a way of working rather than a piece of content. A practice has an adoption matrix ("which service does this at which level"); a Foundations page does not.
  • Governance — normative decisions (ADRs) and proposals (RFCs). Prescriptive, not descriptive.
  • Handbook — instructions for the author, not the reader. Lives next to the scaffolds it explains.

Entity descriptor — catalog-info.yaml v2

Extends the existing Backstage subset already present at services/portal/internal/services/<name>/catalog-info.yaml. New fields are non-breaking: a v1 descriptor renders correctly under v2 (missing fields render as "—" on the tile). 

apiVersion: backstage.io/v1alpha1
kind: Component | Library | API | Resource | System    # five values, not Component-for-everything
metadata:
  name: study-app
  title: study-app
  annotations:
    backstage.io/last-reviewed: "2026-05-18"            # already present
    backstage.io/maintainer: "user:ivan-boyarkin"
    docs.spine-version: "v3"                            # NEW: which spine applies
spec:
  type: service | website | library | openapi | asyncapi | grpc | database | object-store
  lifecycle: experimental | stable | deprecated
  owner: user:ivan-boyarkin | team:platform
  domain: application | data | infrastructure | automation   # NEW: filter axis
  dependsOn: [resource:postgres-main, api:study-api]
  providesApis: [study-api]                             # Backstage-native; unused today, surfaced in v2
  consumesApis: [parser-events]
x-study-app:
  scorecard:                                            # NEW: rendered on tile and hero
    grade: A | B | C | exempt
    spine-completeness: "14 / 14"
    broken-links: 0
  versioning:                                           # NEW
    current: v1
    available: [v1, v2]
    deprecation:
      sunset: null | "2026-09-01"
      migrate-to: null | "study-app v2"
  practices-adopted: [testing-pyramid, docs-as-code, observability-baseline]   # NEW

The renderer at scripts/render_service_descriptors.py is extended in Phase 1 Step 1.4 to consume the new fields. The pre-commit hook catalog-render-check already ensures YAML and HTML do not drift; the new fields ride that channel.

Foundations layer — cross-cutting content

A page belongs in Foundations when (a) at least two entities consume it and (b) no single entity owns it. The starter set has four shared concepts, each one Explanation-quadrant in shape — what the concept is, what contract the portal enforces around it, how a new service plugs into it, which ADR(s) ratify it:

  • foundations/shared-concepts/auth.html — derives from ADR 0005.
  • foundations/shared-concepts/errors.html — derives from ADR 0003.
  • foundations/shared-concepts/observability.html — derives from ADR 0009, ADR 0011, ADR 0023.
  • foundations/shared-concepts/versioning.html — derives from ADR 0004, ADR 0013, ADR 0025.

Foundations also hosts the cross-cutting Diátaxis quadrants (Tutorial = portal tour; How-to = register a new entity; Reference = entity descriptor schema, spine matrix, style guide, templates index; Explanation = IA model, taxonomy rationale, Diátaxis blind-spot coverage), the single-page glossary, and the IA's own governance bookkeeping (authoring workflow, enforcement hooks). The per-Component service-local Diátaxis quadrants do not duplicate Foundations content — they link into it.

Practices layer — cross-service ways of working

A practice is a way the team works the same way across all services: testing pyramid, docs-as-code, observability discipline, on-call rotation, postmortem culture, branching & PR review, API versioning, security baseline. Practices are not content a service produces; they are policy a service adopts. Each practice page has a fixed shape:

  1. What it is — one paragraph plus a PlantUML diagram if shape-shaped (testing pyramid is).
  2. Why we adopted it — rationale and the ADR(s) that ratify it.
  3. Levels we expect — concrete numbers (unit ≥ X %, contract tests for every public API, …).
  4. Adoption matrix — a back-indexed table built from practices-adopted in each entity's catalog-info.yaml. Cell value: basic / standard / advanced / waived.
  5. Tooling that enforces — pytest, coverage gate, mypy, the relevant pre-commit hook.
  6. Antipatterns.
  7. References — industry: Google Testing Blog, SRE Workbook, etc.
  8. History.

Starter set delivered in Phase 2 Step 2.6:

  • practices/testing-pyramid.html — backed by ADR 0002 + 0012.
  • practices/docs-as-code.html — backed by ADR 0001, absorbing handbook/principles §3.
  • practices/observability-discipline.html — backed by ADR 0009, 0011, 0023.
  • practices/api-versioning.html — backed by ADR 0004 + 0007 + 0025.
  • practices/branching-and-pr-review.html — backed by ADR 0017.
  • practices/on-call-rotation.html.
  • practices/postmortem-culture.html.
  • practices/security-baseline.html — backed by ADR 0005 + 0019.

Growth path: add a practice = create one HTML file using handbook/templates/practice.html (new in Phase 2 Step 2.7), append the slug to practices-adopted in any service's catalog-info.yaml that adopts it. The adoption matrix auto-populates.

Per-entity description templates

Every entity type has both a catalog tile (how it appears in catalog/<type>/index.html) and a description page (how the entity describes itself across its spine). Both renderings derive from the same catalog-info.yaml. The templates below are normative — they are what the new handbook/templates/<type>-spec.html scaffolds and what scripts/check_spine_completeness.py verifies.

Component template

Tile fields (rendered from YAML): entity-type pill, lifecycle pill, scorecard grade, name, lede, stack chips, domain pill, owner, last-reviewed, spine completeness, version current/available, broken-links count.

Description spine: 14 pages — Overview, Requirements, Domain, Process, Tutorials (folder), How-to (folder), Reference (folder), Explanation (folder), NFR & SLO, Runbooks, Dependencies, On-call, Migrations, Changelog & Release notes & service-local ADR index.

API template

Tile fields: entity-type pill, protocol pill (OpenAPI / AsyncAPI / gRPC), lifecycle, audience (public / internal), owner-Component, versions, consumers count, compatibility chip, last-reviewed.

Description spine: 5 pages — Overview, Schema (auto-rendered Swagger / AsyncAPI viewer / proto), Consumers, Compatibility, Changelog (+ Migrations if multi-version). APIs do not own runbook, on-call, or SLO — those are the owning Component's responsibility. The API has an availability target delegated to the Component's SLO.

Resource template

Tile fields: entity-type pill, resource-type chip (postgres / clickhouse / s3 / kafka), lifecycle, access tier (public / internal / restricted), backup RPO/RTO, used-by count, owner.

Description spine: 6 pages — Overview, Schema (auto), Access, Backup, Recovery, Runbooks. Resources do not own Diátaxis quadrants — they are for ops, not developers.

Domain model — a page inside a Component, not a catalog entity

The hardest SA page to get right. Lives at catalog/components/<svc>/domain.html. Required sections:

  1. ER PlantUML diagram (entities and relationships, not tables and columns).
  2. Per-entity attribute table (Field · Type · Cardinality · Business rule).
  3. State machines for entities with lifecycle (PlantUML state diagram).
  4. Aggregate boundaries (DDD): which entities are transactionally consistent.
  5. Invariants — business rules in AlwaysTrue: X >= Y form.
  6. Glossary subset — terms used in this domain, linking to foundations/glossary.html.
  7. References — Requirements page, Reference/db schema, relevant ADRs.

Domain model is not DB schema. Schema describes how it is stored; domain describes what it means for the business.

Library template

Tile fields: entity-type pill, lifecycle, owner, stack chips, consumers count, bundle size if frontend, last-reviewed.

Description spine: 4 pages — Overview, Reference (auto: pdoc / TS docs / Storybook), Explanation (design rationale), Changelog. Libraries never have runbook, on-call, NFR.

System template

Tile fields: entity-type pill, scope description, owner. No scorecard (nothing to score).

Description spine: 3 pages — Landscape (C4 L1 PlantUML), portal-level ADR index, strategic roadmap.

Practice template

Not a catalog entity. Single page in practices/<slug>.html following the 8-section shape in Practices layer above.

Compatibility, risks, rollback

Compatibility. Phase 1 adds new files only; existing pages are not moved. Existing URLs do not change. Existing catalog-info.yaml renders correctly with the v2 renderer (missing fields render as "—"). Pre-commit hooks continue to operate; catalog-render-check is updated alongside the renderer.

Risks.

  • Empty-spine debt. Phase 2 introduces new required pages. Components that cannot yet fill all four SA-genres get a stub Explanation-quadrant page with an explicit not yet note and a backlog link.
  • Renderer drift. Extended render_service_descriptors.py must keep producing byte-identical output when fed v1 YAML. Verified by adding a v1 fixture to the script's golden tests.
  • Migration mid-flight. Phase 2 renames services/ → catalog/components/; this changes every existing href. Done as one PR with git mv + sed + asset-refs check.

Rollback. Any phase is fully revertible by git revert of its merge commit; Phase 1 in particular adds only new files. The _preview/ sandbox is retained until Phase 1 ratifies and is then removed.

Open questions — to resolve before Phase 2 starts

  1. Top folder name. catalog/ (proposed) vs keep services/ (less churn, less precise). Recommendation: catalog/.
  2. Practices placement. Top-level practices/ (proposed) vs nested foundations/practices/. Recommendation: top-level — Practices are policy, not content.
  3. Pilot Component for spine v3. api (proposed — richest SA surface) vs portal. Recommendation: api.
  4. Migration speed. Parallel four-Component migration vs one-at-a-time. Recommendation: one-at-a-time, starting with pilot.
  5. Sixth principle in handbook/principles.html. Add "One spine per entity type"? Recommendation: defer to Phase 2 Step 2.7 so principles.html changes only when Handbook v2 lands.
  6. _preview/ fate. Delete after Phase 1 (proposed) vs retain as permanent sandbox. Recommendation: delete — the live tree is now the reference.

Delta vs _preview/

This RFC keeps what the sandbox proposed and adds what the sandbox left open:

  • Kept from preview: entity taxonomy (5 types), per-type spine, onboarding journey, scorecards, scoped search, Foundations layer with shared concepts and cross-cutting Diátaxis quadrants, SA pages inside Component spine, lifecycle pill on tile.
  • Added by this RFC: explicit migration plan with four phases and step-level pre/post/verify contract; catalog-info.yaml v2 schema (domain, scorecard, versioning, practices-adopted); the spine matrix as a normative page (not just sidebar copy); the Practices layer with adoption-matrix mechanism; per-entity description templates (Component, API, Resource, Library, System, Domain model, Practice); supersedes-relationship to ADR 0026; rollback contract.
  • Will be removed: services/portal/internal/_preview/ after Phase 1 ratifies.
