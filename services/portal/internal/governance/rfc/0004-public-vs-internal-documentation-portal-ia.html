RFC · 0004 Proposed

Public vs internal documentation portal IA

A single, defensible information architecture for the whole services/portal/ portal — public Diátaxis layer for external integrators, internal Backstage-shaped layer for the team — grounded in four industry frameworks and delivered across six independently-shippable phases.

Goals and non-goals

This RFC defines a single, defensible information architecture for the whole services/portal/ portal — splitting it cleanly into a public layer for external API consumers and an internal layer for the team — and grounds the choice in the four industry frameworks that have proven the pattern at scale: Diátaxis (public), the Backstage Internal Developer Platform (internal), the Google SRE Book chapter spine (operational hub), and the GitLab handbook single-portal-with-IA-boundary pattern (audience policy). It supersedes the audience-mixed status quo and amends one bullet in ADR 0028 so its "governance at repo root" rule reads as the more general "outside services/".

Goals

  • Adopt Diátaxis as the public-layer organising principle (tutorials / how-to / reference / explanation).
  • Adopt the Backstage Software Catalog + TechDocs per-service spine as the internal-layer organising principle.
  • Adopt the Google SRE Book chapter spine for the internal operational hub (SLOs → monitoring → on-call → incident response → postmortems).
  • Adopt the GitLab handbook single-portal pattern: audience boundary is an IA boundary, not an auth boundary.
  • Consolidate every team-facing page under services/portal/internal/; leave only the public router, release CHANGELOG, generated diagrams, and CSS/JS at services/portal/ root.
  • Introduce two HTML-attribute conventions on every page: data-audience and (for public) data-doc-type.
  • Provide a phased migration (A→F) where each phase ships independently and leaves make verify green.

Non-goals

  • Physical move to services/portal/. That is ADR 0028 Phase 1 — separate backlog item. The IA chosen here is forward-compatible.
  • Filling new pages with full prose. Phases A–F deliver skeletons, indexes, and 1-paragraph stubs with cross-links. Body content for each new hub follows in subsequent backlog items.
  • Auth gating. The public/internal split is structural and labelled, not access-controlled. For a solo monorepo, this is the GitLab-handbook posture: write nothing secret in any layer.
  • Backstage runtime install. We borrow the Software Catalog and TechDocs naming and structuring discipline; we do not run the Backstage application.
  • Visual styling. Existing portal hero / internal-layout CSS is reused. Public pages get no new CSS class; only the existing class="internal-layout" continues to gate the internal sidebar.

Background

Today's services/portal/ tree is audience-mixed and framework-less. Concrete failures:

  • The Swagger sandbox at services/portal/public/reference/api/index.html — the only artifact an external integrator needs — is mislabeled Internal · API Reference in its hero. The page that should be the front door for integrators is hidden behind an internal label.
  • The portal landing at services/portal/index.html mixes three audiences in one hero (external integrators, internal team, "people on this project"). A first-time visitor cannot tell what is for them.
  • There is no data-audience attribute, no public/internal CSS split, no nav branching. class="internal-layout" exists but is a layout flag (gates the sidebar), not an audience flag, and only services/portal/internal/** uses it.
  • ADR 0025 + ADR 0026 already carved a partial split for the API surface (OpenAPI external, services/portal/internal/api/ internal). The whole portal needs the same discipline, grounded in industry frameworks rather than invented locally.
  • Internal operational docs (logging, metrics, observability, debugging) have ADR-level rationale (ADR 0009, ADR 0011, ADR 0023) but no narrative hubs — readers cannot find the "how do we observe production?" page because it does not exist as a coherent surface.
  • Engineering practice docs are scattered: services/portal/internal/handbook/howto/, services/portal/internal/handbook/developer/, services/portal/internal/handbook/qa/ sit at repo-services/portal/ root next to services/portal/internal/, even though they are all team-facing. The split is "by historical accident", not by audience or function.

ADR 0028 (Proposed 2026-05-07) ratifies a destination of services/{api,portal,monitoring} but stages implementation across four phases. Before the folder split, per-service Dockerfiles, root compose, and CI matrix phases ship, the documentation portal itself needs a coherent IA — otherwise services/portal/ inherits today's mixed-audience layout and the architectural separation is cosmetic. This RFC fills that gap.

Design

1. Framework choices

Five decisions, each grounded in a peer-reviewed or billion-user-scale source. The combination is not novel — every large engineering organisation has converged on some variant of it — but stating the framework names explicitly turns an implicit IA into a defensible one.

Decision Framework chosen Alternatives rejected Why this one
Public-layer IA shape Diátaxis quadrants (tutorials / how-to / reference / explanation) Stripe-style product taxonomy; Twilio per-channel; GitHub user-task flat list Stripe and Twilio taxonomies are product-shaped — they work when you sell 12 products. study_app has one API. Diátaxis quadrants give a stable cognitive model regardless of feature count and eliminate the "is this a tutorial or a how-to?" debate. Cloudflare developer docs adopt Diátaxis explicitly inside each product — same pattern at our scale.
Internal-layer IA shape Backstage Software Catalog + TechDocs spine (entity descriptor format) Confluence-style flat wiki; ad-hoc per-service folders The Catalog forces typed entities (Component / API / Resource / System / Domain) with mandatory owner, lifecycle, providesApis, dependsOn, links — exactly the metadata that goes stale in Confluence. We borrow the naming and structuring discipline, not the runtime. The per-service spine matches the page templates Spotify, AWS Lambda, and Google Cloud Run all converged on independently.
Operational hub shape Google SRE Book + Workbook chapter spine (SRE Book TOC, SRE Workbook TOC) One big "operations" page; ITIL-style change management; ad-hoc runbooks index The SRE chapter spine — SLOs → Monitoring → On-call → Incident response → Postmortems → Capacity — is the canonical reference for production operations. Postmortem template (sre.google example postmortem) is the artifact every team eventually writes. Adopting the spine up front means we never invent it ad hoc on the day of an incident.
Audience boundary Single portal, IA-tagged — GitLab handbook pattern Two portals (Stripe-style: docs.stripe.com + internal Confluence/Notion); Backstage hybrid with per-component visibility flags Two portals need authentication and duplicate tooling — overkill for a solo monorepo. Auth-gated TechDocs needs the Backstage runtime. The GitLab handbook proves a single-portal, audience-tagged-via-IA approach works at scale (handbook.gitlab.com is almost entirely public; sensitive items live in private stores instead). For our scale this is the only realistic answer: write nothing secret in any layer; the audience boundary is an IA boundary, not an auth boundary.
Per-service template AWS-Lambda / GCP-Cloud-Run repeating spine: Overview → Quickstart → Concepts → How-to → Tutorials → Reference → Best practices → Release notes Invent a local template; per-service free-form docs Industry consensus — matches what every reviewer already knows from AWS Lambda and Cloud Run service docs. We collapse to six children per entity (index/architecture/api-reference/runbooks/dependencies/on-call) for a one-engineer scope but keep the spine recognisable.

2. Comparative analysis of leading developer portals

We surveyed eight reference portals to confirm the framework choices are typical of the industry, not idiosyncratic. Findings (not a comprehensive teardown — only the IA-relevant pattern):

  • Stripe — gold-standard developer portal. Top level is product-shaped (Payments, Billing, Connect, Tax, …); within each product the recipe is Overview → Quickstart → How it works → Build → Test → Go live → Reference. Three-column layout with persistent language picker. Take: the per-product recipe is functionally Diátaxis (tutorials + how-to + reference + explanation) without the framework name. We adopt the framework name explicitly because it forces page-level discipline that "recipe" does not.
  • Twilio — product-channel-shaped (Messaging, Voice, Video, …). Each product page exposes Quickstart → Tutorials → How-to → API reference → Code samples → Pricing. Take: same shape as Stripe; same observation.
  • GitHub — user-task flat list at top level (Get started, Repositories, Pull requests, Issues, Actions, …); REST and GraphQL API references split off into siblings (/rest, /graphql). Take: separating product narrative from API reference into different top-level trees keeps each tree clean. We mirror the pattern — public Diátaxis quadrants vs internal service catalog — at our scale.
  • Cloudflare developer docs — most explicit Diátaxis-in-the-wild adopter. Top level is product-shaped, but inside each product the four Diátaxis quadrants are explicit headings. Take: directly transplantable to a one-product portal.
  • AWS Lambda — canonical per-service template: What is X? / Getting started / How X works / Configure-Build-Invoke-Deploy / Best practices / Troubleshooting / Code examples / API reference / Limits / Pricing. Take: the repeating spine is the per-service template the industry expects.
  • Google Cloud Run — same repeating spine: Overview / Quickstarts / Concepts / How-to / Tutorials / API reference / Best practices / Pricing / Release notes. Take: same observation; this is the de-facto industry per-service template.
  • Spotify Backstage TechDocs + Software Catalog — internal-portal canonical. Per-service: description, owner, lifecycle, APIs provided/consumed, dependencies, system/domain membership, links, architecture, runbooks, API reference, README. Take: we transplant the catalog metadata fields and the per-service spine without running the Backstage application.
  • GitLab handbook — single-portal team-facing IA at organisation scale. Almost everything is public; sensitive items live elsewhere. Take: proof that audience-boundary-as-IA-boundary works at scale; the only realistic option for a solo monorepo.

4. Conventions

4.1 data-audience attribute

Every page declares its audience on <body>: 

<body data-audience="public">   ← external integrators
        <body data-audience="internal"> ← team

The attribute is a normative classifier read by:

  • scripts/build_docs_search_index.py — emits an audience field per indexed entry.
  • The search UI — exposes a Public / Internal / All filter pill.
  • services/frontend/portal/assets/docs-nav.js — surfaces a public/internal grouping in the top nav.
  • Future tooling that needs to render audience badges or partition outputs.

4.2 data-doc-type attribute (public pages only)

Every page under services/portal/public/ declares its Diátaxis quadrant: 

<body data-audience="public" data-doc-type="tutorial">
<body data-audience="public" data-doc-type="how-to">
<body data-audience="public" data-doc-type="reference">
        <body data-audience="public" data-doc-type="explanation">

The attribute powers Diátaxis-quadrant cards on services/portal/public/index.html, search facets, and any future filtering. It is intentionally not applied to internal pages — the internal layer uses the Backstage entity spine, not Diátaxis quadrants.

4.3 Backstage-style entity card

Every catalog entity page (services/portal/internal/catalog/<service>/index.html) opens with a definition list whose terms mirror the Backstage Software Catalog descriptor spec fields: 

<dl class="catalog-entity-card">
<dt>Owner</dt>        <dd>Ivan Boyarkin</dd>
<dt>Lifecycle</dt>    <dd>experimental | stable | deprecated</dd>
<dt>Provides APIs</dt><dd>HTTP /api/v1 (link to OpenAPI)</dd>
<dt>Depends on</dt>   <dd>SQLite, structlog, FastAPI</dd>
<dt>Links</dt>         <dd>repo, dashboards, runbooks, on-call</dd>
        </dl>

No YAML catalog-info.yaml file is required at this stage — the HTML carries the metadata. A future RFC may introduce machine-readable descriptors if the team adopts the Backstage runtime.

4.4 Per-service spine (six children per entity)

Every services/portal/internal/catalog/<service>/ directory exposes the same six children:

  1. index.html — entity card + overview.
  2. architecture.html — C4 component diagram per ADR 0020.
  3. api-reference.html — for api/, links into services/portal/public/reference/api/; for other services, the relevant interface (e.g., portal's static-asset surface, monitoring's scrape config).
  4. runbooks.html — index page linking into services/portal/internal/sre/runbooks/.
  5. dependencies.html — upstream and downstream entities.
  6. on-call.html — alert routing, escalation, primary contact.

The api/ entity additionally hosts code-reference/ (the relocated pdoc tree). Other services do not need code-reference because they are configuration-only.

4.5 No new public-layer CSS class

Public pages reuse the existing portal hero look (services/frontend/portal/assets/docs.css, services/frontend/portal/assets/home.css) without a new public-layout class. The existing class="internal-layout" continues to gate the internal sidebar; that remains unchanged. The audience attribute alone is sufficient to drive nav, search, and any future audience-aware behaviour.

5. Mapping to existing ADRs

This RFC composes with, rather than replaces, the existing ADR corpus. Mapping:

  • ADR 0001 — Docs as Code: governance principle unchanged. The IA reorganisation is itself a docs-as-code change (versioned, reviewed, link-checked).
  • ADR 0007 — OpenAPI governance: the relocation of openapi-baseline.json to services/portal/public/reference/api/ in Phase B does not affect governance rules; the spec, breaking-change gate, and embedded sandbox stay normative.
  • ADR 0009 — Health + observability: cited from services/portal/internal/sre/{monitoring,observability}.html.
  • ADR 0011 — SLO / SLA / error budget: cited from services/portal/internal/sre/slos.html and from services/portal/public/explanation/slo-error-budget.html (consumer-facing summary).
  • ADR 0023 — Structured logging: cited from services/portal/internal/sre/logging.html.
  • ADR 0024 — Quality assessments: services/portal/internal/governance/audits/ moves to services/portal/internal/governance/audits/ in Phase F. Templates and rubrics unchanged.
  • ADR 0025 — External vs internal API docs: this RFC extends the audience split from the API surface (already partially separated) to the whole portal. ADR 0025 remains authoritative for which artifact is normative for which API audience.
  • ADR 0026 — Internal docs as source of truth: the existing services/portal/internal/api/ hub-and-spoke layout is preserved. The new services/portal/internal/catalog/ layer wraps it: catalog/api/api-reference.html links into the existing internal API specs.
  • ADR 0027 — Search index: extended in Phase B to emit audience and doc_type fields per entry. Single index, filterable view; no separate indexes.
  • ADR 0028 — Monorepo + service boundaries: amended by this RFC (see §6).

6. Amendment to ADR 0028

ADR 0028's "Decision" section currently states (one bullet, paraphrased): "Governance artifacts — ADRs, RFCs, runbooks, audits, CHANGELOG, OpenAPI baseline — remain at repo root (services/portal/internal/governance/adr/, services/portal/internal/governance/rfc/, services/portal/internal/sre/runbooks/, services/portal/internal/governance/audits/, services/portal/CHANGELOG.md, services/portal/public/reference/api/). They are repo-wide concerns, not per-service concerns."

This RFC amends that bullet to: "Governance artifacts — ADRs, RFCs, runbooks, audits, CHANGELOG, OpenAPI baseline — remain outside services/. They are repo-wide concerns, not per-service concerns. Their physical home is services/portal/internal/governance/ (ADRs, RFCs, audits, backlog) and services/portal/internal/sre/runbooks/ (runbooks); the OpenAPI baseline lives at services/portal/public/reference/api/ per ADR 0007; services/portal/CHANGELOG.md stays at services/portal/ root as the release journal."

Why the amendment is conservative: ADR 0028's intent was to prevent governance from being coupled to a specific service (i.e., not inside services/api/ or services/portal/). Both placements — repo-services/portal/ root and services/portal/internal/governance/ — satisfy that intent equally because services/portal/ itself stays at repo root, outside services/. The amendment trades a specific path list for the underlying invariant; ADR 0028's architectural separation is unchanged.

Status of ADR 0028: Proposed (2026-05-07), not yet ratified. Amendments at this stage follow the normal ADR lifecycle (ADR 0018) — edit in place, record the change in the history table, link the discussion Issue.

Implementation

Six phases. Each phase ships independently and leaves make verify green. Phases A–B touch tooling and add new content; Phases C–F move content. The folder-move phases (D, E, F) are paired with one-shot path-rewrite scripts that are deleted after use.

Phase A proposed — RFC 0004 + amend ADR 0028 + backlog + CHANGELOG (text only)

  1. Write this RFC at services/portal/internal/governance/rfc/0004-public-vs-internal-documentation-portal-ia.html.
  2. Amend the governance bullet in ADR 0028; add a history row noting the amendment.
  3. Add a backlog item under services/portal/internal/governance/backlog/ tracking Phases B–F.
  4. Add a 2026-05-08 CHANGELOG entry under Added (this RFC + ADR 0028 amendment).

Phase B proposed — Public Diátaxis layer + relocate Swagger

  1. Create services/portal/public/ with quadrant skeletons: index.html, tutorials/, how-to/, reference/, explanation/.
  2. Move services/portal/public/reference/api/ → services/portal/public/reference/api/. Hero relabel from Internal · API Reference to Public · API Reference. Apply data-audience="public" + data-doc-type="reference".
  3. Refactor services/portal/index.html into a router (two CTAs: integrator → public, team → internal).
  4. One-shot script scripts/apply_docs_audience_attribute.py applies data-audience across all existing pages (idempotent).
  5. Update hard-coded paths: scripts/openapi_governance.py, scripts/spec_consistency.py, scripts/changelog_gate.py, Makefile, .pre-commit-config.yaml.
  6. Extend scripts/build_docs_search_index.py to emit audience + doc_type; update search UI with filter pill.
  7. Update services/frontend/portal/assets/docs-nav.js for public/internal grouping.

Phase C proposed — Internal Service Catalog (Backstage spine)

  1. Create services/portal/internal/catalog/{api,portal,monitoring}/ with the six-child spine each.
  2. Move services/portal/pdoc/ → services/portal/internal/catalog/api/code-reference/. Update Makefile pdoc -o path and .pre-commit-config.yaml exclude patterns.
  3. Wire the catalog into services/portal/internal/index.html and services/frontend/portal/assets/internal-sidebar.js.

Phase D proposed — SRE hub + relocate runbooks

  1. Create services/portal/internal/sre/ spine: index, slos, monitoring, logging, observability, on-call, incident-response, postmortems/, debugging.
  2. Postmortem template at postmortems/_template.html adapted from sre.google example postmortem.
  3. Move services/portal/internal/sre/runbooks/ → services/portal/internal/sre/runbooks/. Update .pre-commit-config.yaml; per-service catalog/<service>/runbooks.html cross-links updated.
  4. Cross-refs: slos ↔ ADR 0011; monitoring + observability ↔ ADR 0009; logging ↔ ADR 0023.
  5. Update internal-sidebar.js.

Phase E proposed — Handbook (move developer / howto / qa)

  1. Create services/portal/internal/handbook/index.html.
  2. Move services/portal/internal/handbook/developer/ → services/portal/internal/handbook/developer/; services/portal/internal/handbook/howto/ → services/portal/internal/handbook/howto/; services/portal/internal/handbook/qa/ → services/portal/internal/handbook/qa/.
  3. Update scripts/sync_docs.py markers, .pre-commit-config.yaml, any nav/portal-data scripts using these prefixes.
  4. One-shot scripts/migrate_handbook_paths.py bulk-fixes cross-refs in every HTML page that links into the moved subtrees. Delete after use.
  5. Update internal-sidebar.js + internal-hub grid.

Phase F proposed — Governance (move adr / rfc / audit / backlog)

  1. Create services/portal/internal/governance/index.html.
  2. Move services/portal/internal/governance/adr/ → services/portal/internal/governance/adr/ (incl. amended ADR 0028); services/portal/internal/governance/rfc/ → services/portal/internal/governance/rfc/ (incl. this RFC); services/portal/internal/governance/audits/ → services/portal/internal/governance/audits/; services/portal/internal/governance/backlog/ → services/portal/internal/governance/backlog/.
  3. Update scripts/check_adr_rfc_links.py (DOCS / "adr" + DOCS / "rfc" paths); .pre-commit-config.yaml file pattern ^services/portal/(adr|rfc)/.*\.html$; any other scripts that hard-code these paths.
  4. One-shot scripts/migrate_governance_paths.py bulk-fixes every cross-ref to the moved trees (most ADR-to-ADR refs are co-relocated and stay valid; refs from outside the moved trees need rewriting).
  5. Replace the aspirational paths in the Phase A amendment of ADR 0028 with the actualised paths.
  6. Update internal-sidebar.js + internal-hub grid.

After Phase F, repo-services/portal/ contains only: index.html, CHANGELOG.md, public/, internal/, uml/, assets/.

Validation

Local checks

  • make verify after every phase — green.
  • make docs-fix regenerates services/frontend/portal/assets/search-index.json; grep for "audience":"public" and "doc_type":"tutorial" after Phase B.
  • scripts/check_adr_rfc_links.py passes after Phase F (script paths updated).
  • Manual smoke: python -m http.server from repo root; visit http://localhost:8000/docs/ — router renders, both CTAs work.

CI / automated checks

  • make verify in CI — same gates as local.
  • Pre-commit hooks updated for new file patterns (Phase B for openapi; Phase D for runbooks; Phase E for handbook; Phase F for governance).

Operational checks

  • Search UI — audience filter pill returns disjoint results for Public vs Internal after Phase B.
  • End-to-end after Phase F: integrator at services/portal/index.html reaches Swagger in ≤ 3 clicks, never touches internal; teammate reaches the SRE logging hub in ≤ 3 clicks.

Rollout plan

  • Rollout phases: A → B → C → D → E → F, one PR per phase. Phase A is text-only; Phases B–F each combine a content scaffold with a path migration scoped to one subtree.
  • Feature flags: none. The IA is structural; runtime behaviour does not change.
  • Rollback strategy: each phase is reversible by git revert. Phases C–F use one-shot migration scripts that are deleted after the migration; the underlying file moves are tracked by git.
  • Backward compatibility: the only consumer-facing path change is services/portal/public/reference/api/openapi-baseline.json → services/portal/public/reference/api/openapi-baseline.json in Phase B. ADR 0007 governance applies; downstream consumers of GHCR images per ADR 0021 are unaffected because the baseline is an artifact path, not a runtime endpoint.

Change management

Update this RFC when the audience boundary, framework choices, or per-service spine changes. Open a new RFC when a fundamentally different IA is needed (e.g., multi-portal split with auth gating, switching from Diátaxis to a product-shaped public layer). Owner after merge: docs maintainer (currently Ivan Boyarkin).

  • Update this RFC when: a new audience appears (e.g., partner integrators with different visibility); a new internal layer is added (e.g., data/ for analytics docs); per-service spine adds or drops a child.
  • Open a new RFC when: physical move to services/portal/ (ADR 0028 Phase 1) makes the path layout obsolete; auth gating becomes a requirement; a separate consumer help-centre is split off from this portal.
