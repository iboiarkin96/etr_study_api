Audit · Docs 2026-05-24 ≈ 7.5 / 10 Baseline run

Principal-Triad Docs Audit — 24 May 2026

Documentation portal reviewed through three independent Principal lenses (SA Systems Analyst, SWE Software Engineer, ARCH Architect), benchmarked against Stripe / AWS / Google / Microsoft / GitLab / Kubernetes / Backstage practices. Each finding ends in a concrete artefact path — what to create, what to fix, where, with which public benchmark. First baseline of this audit type; no prior delta to compute.

1. Executive summary

Overall score: 7.5 / 10 (weighted: SA 33% · SWE 34% · ARCH 33%).

  • SA overall: 6.9 / 10 — strong on requirements, SLO numbers, glossary, risk register; missing reader personas and a domain model that is not just a DB schema.
  • SWE overall: 7.5 / 10 — OpenAPI baseline + contract testing + CHANGELOG gating at production tier; gaps are deprecation-banner as a real component (not a snippet), API migration guides, and prose/link CI lints.
  • ARCH overall: 8.1 / 10 — Backstage catalog-info.yaml across four entities, formal spine matrix, full C4 stack, async RFC process; gaps are runbook_url annotations on Prometheus alerts and aggregated postmortem index.

Top-3 strengths

  1. Backstage-grade entity taxonomy with CI-enforced spine. Four catalog-info.yaml files plus foundations/reference/sa/spine-matrix.html defining required / forbidden pages per kind. Few portals at this size have it; this is Spotify-class.
  2. OpenAPI baseline + contract testing + changelog gate in CI. app/openapi/ + Makefile targets openapi-check / contract-test + scripts/changelog_gate.py on every PR. GitLab-class living-system discipline.
  3. Documented authoring contract. foundations/reference/sa/principles.html states seven load-bearing principles; RFC 0003 ratifies the taxonomy; templates exist for every genre. Most companies have the practice; few write it down.

One-line verdicts

  • SA — «Solid requirements/NFR baseline. Add personas + traceability matrix + a real domain model and this jumps to Staff SA portfolio level.»
  • SWE — «Production-grade API governance. Ship the deprecation banner as a component, wire lychee + Vale, and you’re at GitLab Handbook tier.»
  • ARCH — «Spotify/Netflix-class IA. Two operational hygiene tasks (runbook URLs, postmortem index) finish the picture.»

5. Triad disagreements

SA vs SWE — How «deprecation» should be scored

SWE initial pass flagged deprecation.html as «exists, score 1»; SA on re-read pushed back: the file is a strong Explanation with banner snippet, sunset timeline, ADR-ratification step. The disagreement isn’t about the policy — it’s about whether absence-of-enforcement is a SWE-side or shared concern.

Resolution (closest-principal rule): SWE owns. Score stays 1 because the policy isn’t executable yet — the banner is a snippet, not a component, and CI does not check that deprecated entities carry it. Action lives in Top-5 #2.

ARCH vs SWE — Is «practice radar» the right shape?

ARCH scored ARCH-7 a 2 (per-role practice radars exist and are richly populated). SWE would expect a ThoughtWorks-style tech radar (React 18 in Adopt, etc.). Different artefact serving a different purpose.

Resolution: For a single-product internal portal the practice radar is appropriate — Spotify uses the same pattern. Score remains 2. If/when this portal serves >1 product, a separate tech radar may be added.

SA vs ARCH — Where does the domain model live?

SA wants foundations/explanation/domain-model.html; ARCH points at services/api/code-reference/app/domain.html. Different audiences: SA writes for a BA / non-engineer; ARCH page reflects the implementation.

Resolution: Both should exist. The SA page is the missing one (business invariants, bounded contexts, mapping to ER) and is the Top-5 #5 bundle item.

6. Cross-cutting findings

Items where all three principals flagged the same problem from different angles.

  1. Templates exist; instances don’t. Postmortem template, deprecation policy, traceability description, persona-aware discoverability — all are documented as what good looks like, none have a real example in the portal. This is the single recurring shape of every weak score in this audit.
  2. Principle stated > principle enforced. Several policies (banner mandatory, alert needs runbook URL, link rot must be detected) are written but no CI gate fails when they’re broken. A Staff reviewer will read this as «hopes > contracts».
  3. No live data anywhere. Every radar, every score, every SLO page is a static snapshot. Linking even one of them to live Prometheus / DORA would convert «aspirational» into «monitored». Out of scope for this audit but flagged for the next.

7. Delta vs previous audit

This is the baseline run of the Principal-Triad audit. No previous audit of this exact shape exists, so no per-axis delta to compute.
