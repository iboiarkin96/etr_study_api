Testing Pyramid Assessment — formed 18 May 2026

Audit of how the testing pyramid is described across the portal and whether the on-disk layout under tests/ matches what those pages claim. Triggered by the multi-service IA target tree (2026-05-18): before the catalog spine adopts a per-component testing section, the existing single-service pyramid material has to be coherent.

Overview

Assessment type: services/portal/DX. Formation date: 2026-05-18. Compare the portal's testing-pyramid material to common good practice, score each row (Table 2), and track prioritized gaps (§ Scoring summary).

Headline finding: the portal hosts three mutually inconsistent pyramid models (5-layer, 4-layer, 3-layer) across five files, and the pyramid's canonical page documents two folders (tests/domain/, tests/repositories/) that contain no tests on disk. The doc–reality gap is the single largest defect.

Scope and methodology

Scope

In scope: every portal page that describes the test pyramid, layer taxonomy, or layer ratios; the governing ADRs (0002, 0012); and the on-disk shape of tests/. Out of scope: the quality of the tests themselves, CI configuration, or the load-testing toolchain except where it affects how the pyramid is described.

Method

  1. Reference list — Table 1 describes ten reference practices for documenting a testing pyramid in a service repository, expressed in neutral terms.
  2. Map to repo — Table 2 scores each practice (1–10) against this repository, with concrete evidence: file paths, ADR numbers, and the on-disk contents of tests/.
  3. Summarize — overall score and weighted model in § Scoring summary, then five ranked gaps with workflow status.

Evidence sources

  • explanation/qa/test-pyramid.html — 5-layer model (Manual / System / Contract / Integration / Unit).
  • explanation/qa/test-strategy.html — strategy page; refers out to the pyramid for tactics.
  • team/roles/qa/practices/test-pyramid-design.html — 3-layer Cohn model with 70 / 20 / 10 ratios.
  • governance/adr/0002-testing-policy.html — "one success + one failure per endpoint"; no ratios.
  • governance/adr/0012-testing-strategy-and-load-testing.html — 4-layer split (Automated API / Contract / Manual / Load).
  • Repository: tests/api/v1/, tests/core/, tests/domain/, tests/repositories/, tests/openapi/ — inspected directly.

Table 1: Reference practices

What "good" looks like for documenting a testing pyramid in a service repository. One row per practice.

# Practice Why it matters
1 Single canonical pyramid model One source of truth for the layer taxonomy. Multiple competing models force the reader to guess which is "current" and produce review-time arguments over what counts as which layer.
2 Explicit layer definitions Each layer has a written rule for isolation (no DB / no network / no time / no randomness for unit, real DB for integration, etc.) so a test's correct home is decidable at code review.
3 Documented target ratios A target shape (e.g. Google's 70 / 20 / 10) makes drift visible. "Don't invert" is enforceable only if the target is on the page.
4 Ratios codified in an ADR A practice document that asserts ratios without an ADR is folklore. The ADR is what survives team turnover and what reviewers point at when a PR pushes the wrong way.
5 Layer ↔ folder mapping matches reality If the doc says unit tests live in tests/domain/, the folder has to actually contain unit tests. Aspirational paths in the doc are worse than no path at all — they teach the reader the wrong mental model.
6 Ownership map per layer Who writes / reviews / owns drift at each layer. Without an ownership map the contract layer is everyone's problem, which means nobody's.
7 Pyramid-shape metric on a dashboard A number tracked over time so inversion is detected without a manual audit. "Reviewed quarterly" without a metric is a wish.
8 Cross-document consistency Explanation, governance ADRs, QA practice pages, and runbooks use the same layer count, the same names, and the same ratios. Drift between them is the cheapest defect to detect and the costliest to leave standing.
9 "Internal contract" vs "inter-service contract" disambiguated HTTP / OpenAPI tests inside one service and Pact-style contracts between services are different concepts. Sharing the unqualified word "contract" breaks down once the portal hosts more than one component.
10 Load testing positioned coherently Either inside the pyramid as a top tier (with rationale) or beside it as an orthogonal concern (with rationale). The reader must not have to infer the relationship from absence.

Scoring summary

Narrative overall

Overall: about 2.5 / 10.
To move toward top decile: pick one pyramid model and demote the other two to historical references; back the chosen ratios with an ADR; and close the doc–reality gap on tests/domain/ and tests/repositories/ (either write the missing tests or stop documenting the folders). After those three moves the portal's testing material would score in the 6–7 range without changing a single test.
