Executive summary

Промт-аудит требует, чтобы портал отвечал на 10 осей: entity taxonomy, per-entity Diátaxis, Foundations, SA blind spots, enforcement, versioning, navigation UX, domain genres, doc-as-code, operations. Дерево ниже даёт по каждой оси конкретную ветвь — куда падает артефакт и кто им владеет.

Ключевые сдвиги относительно текущей структуры:

  • Onboarding journey поднят на top-level (был спрятан в Tutorials).
  • Catalog заменяет плоский services/ и делится по типам сущностей (System / Component / Library / API / Resource).
  • Foundations — отдельный «виртуальный продукт» с собственным набором Diátaxis-квадрантов плюс Enforcement и Authoring workflow.
  • Внутри каждого Component появляются SA-страницы — Requirements, Domain model, Process documentation, NFRs — закрывают слепые пятна Diátaxis.
  • Operations вынесена из Governance и включает SLO dashboard, DORA, incidents/postmortems, on-call rotation, tech radar.
  • Из IA уходит top-level раздел roles/ — демотируется до опциональных чек-листов внутри Onboarding.

Легенда дерева

  • [A1] — нода закрывает ось аудита (номер совпадает с рубрикой).

Axis coverage map

Каждая ось аудита покрыта минимум одной веткой дерева. Где покрытия нет — это намеренный скоуп (см. секцию ниже).

ОсьВесЗакрывающие ветвиГлавный артефакт
A1 · Entity taxonomy14% catalog/{systems,components,libraries,apis,resources}/ foundations/reference/entity-descriptor-schema.html + spine-matrix.html
A2 · Per-entity Diátaxis12% catalog/components/<name>/{tutorials,how-to,reference,explanation}/ Шаблон Component (Foundations/Reference/templates/component.html)
A3 · Foundations10% foundations/{shared-concepts,system-landscape,governance}/ foundations/explanation/ia-model.html — формальный критерий cross vs service
A4 · SA blind spots12% catalog/components/<name>/{requirements,domain-model,process,nfrs}.html + operations/risk-register.html Шаблон Requirements (BABOK-aligned)
A5 · Enforcement10% foundations/enforcement/{linter,scorecards,codeowners,stale-detection}.html Pre-merge gate spec
A6 · Versioning & lifecycle8% catalog/components/<name>/{versions,v1,v2,migrations,release-notes}/ + foundations/shared-concepts/versioning.html Compatibility policy на API
A7 · Navigation & onboarding10% onboarding/{day-1,week-1,month-1}.html + scoped search в topbar Onboarding journey как primary CTA на home

A9 · Doc-as-code8% foundations/authoring/{markdown-source,build-pipeline,review-process,diagrams-as-code}.html + templates/ Markdown source contract + diagrams-as-code

Что намеренно не в дереве

  • Top-level roles/ — демотировано. Конкурирует с Diátaxis-навигацией, в Stripe и других топ-компаниях сознательно убрано. Чек-листы остались внутри onboarding/by-role/ как secondary view.
  • «Pricing» / «Quotas» — внутренний портал, не B2B. Если появится публичный self-serve тиринг — добавить в Foundations/Reference как отдельную страницу.
  • «FAQ» — заведомо anti-pattern. Любой вопрос из FAQ имеет правильное место: How-to (как сделать), Reference (что это значит), Troubleshooting (почему не работает). FAQ растворяется по этим квадрантам.
  • Дублирующие Diátaxis-секции на уровне портала — нет общего tutorials/ рядом с catalog/. Только в Foundations и только cross-cutting.
  • Marketing-страницы («Why ETR») — internal-only портал, не нужны. На публичной ветке rendering — добавляются как landing.

Migration phases

Не переписывать всё разом. Phased rollout, каждая фаза самодостаточна и не ломает портал.

Phase 1 · 1 month Foundations skeleton
  • Создать foundations/ с пустыми разделами + index-ы
  • Перенести shared-concepts (Auth, Errors) из дублей по сервисам
  • Описать entity-descriptor-schema.html + spine-matrix.html
  • Добавить onboarding/{day-1,week-1,month-1}.html
  • Поднять Onboarding CTA в hero internal/index.html
Phase 2 · 3 months Catalog re-shape
  • Создать catalog/{systems,components,libraries,apis,resources}/
  • Мигрировать study-app → catalog/components/study-app/
  • UI Kit → catalog/libraries/ui-kit/
  • Вынести study-api как первый first-class API в catalog/apis/
  • Старые URL → 301-редиректы (или canonical link)
Phase 3 · 6 months SA pages + Enforcement
  • Добавить Requirements / Domain / Process / NFR на study-app
  • Включить pre-merge linter (Vale + lychee + Spectral)
  • Внедрить last-reviewed metadata + stale-detection bot
  • CODEOWNERS на docs-разделы
  • Scorecards на тайлах каталога
Phase 4 · 12 months Multi-service scale
  • Добавить второй Component (web-parser или эквивалент)
  • Operations: SLO dashboard, DORA, tech radar, postmortems
  • Versioning UI (v1/v2 switcher) на study-api
  • Deprecation banner механика — pilot на legacy-сервисе
  • Внутренний vs публичный rendering (если применимо)
