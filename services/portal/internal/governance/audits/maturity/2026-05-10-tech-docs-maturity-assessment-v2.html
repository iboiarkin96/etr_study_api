Scope and methodology

Scope

v2 covers the same surface as v1 — the organization of technical documentation across the repository — but rubricated against a broader 144-practice grid. End-user UX of the rendered portal remains out of scope (covered by the DX stream); SRE / security / CI/CD engineering maturity remains out of scope (covered by the Project Maturity stream). Two new axes — Onboarding (Axis 11) and Trust / Compliance / Analytics (Axis 12) — cover surfaces a v1-only reader would not have seen.

Methodology

Every one of the 144 practices is scored 1–10 against an industry benchmark with a public URL anchor. Each AS-IS row carries a confirmed / partial / not-confirmed justification stamp and cites a file:line, a directory/, or a CI workflow path as evidence. A row with no public benchmark URL is ineligible for credit. Severity scale used in the gap tables: SEV-1 baseline practice missing or broken; SEV-2 2023+ industry-standard practice absent or out-of-date; SEV-3 2025–2026 frontier practice absent; SEV-4 polish / tech debt without immediate harm.

Files and directories read for v2 (delta vs v1 set is in the Delta block): Makefile, .pre-commit-config.yaml, .github/workflows/*.yml, .github/PULL_REQUEST_TEMPLATE.md, .github/ISSUE_TEMPLATE/, README.md, CONTRIBUTING.md, CHANGELOG.md, services/portal/public/{tutorials,how-to,reference,explanation}/, services/portal/internal/governance/adr/ (32 ADRs incl. 0018, 0020, 0022, 0024, 0029), services/portal/internal/governance/rfc/ (6 RFCs incl. 0003, 0004), services/portal/internal/catalog/{api,portal,monitoring}/, services/portal/internal/handbook/{howto,developer,qa}/, services/portal/internal/sre/{runbooks,postmortems}/, services/portal/public/reference/api/openapi-baseline.json, scripts/build_docs_search_index.py, scripts/check_adr_rfc_links.py, scripts/check_asset_refs.py, scripts/validate_docs_a11y.py, scripts/validate_docs_design.py, scripts/openapi_governance.py, scripts/repair_docs_html.py, alembic/versions/.

Persona walkthroughs

Five personas exercise five surfaces (one ADR, one API endpoint, one service-catalog entry, one runbook, one tutorial). Friction points are where the persona stops, gives up, or reaches for an external tool.

1. External developer — first API call in 5 minutes

Lands on the public portal, follows public/tutorials/first-api-call.html. Happy path is clear; quickstart works. Friction: error catalog (public/reference/errors/) is a single index page — when the first call fails with an error code, the developer has no per-code page to land on. Multi-language code samples are not auto-generated; only curl is canonical.

2. Internal senior contributor — "why was decision X made?"

Knows ADRs exist; finds internal/governance/adr/ easily; the lifecycle badge (ADR 0018) gives instant status. Supersession links work. Friction: no decision graph — the persona has to walk supersession links one by one to see ADR clusters. MADR 4.0 fields are partial — Decision Drivers and Considered Options as a comparable list are missing on most ADRs, so the persona has to reconstruct "why this option won" from prose.

3. On-call engineer at 3 AM — "metric Y is firing"

Opens the runbook directory; finds the page; structure (alert → diagnosis → actions → escalation) is present. Friction: runbook template is not enforced — page sections vary; no link from runbook back to the catalog page of the affected component, no surfaced SLO state. The on-call has to cross-reference SLO numbers in public/explanation/slo-error-budget.html manually.

4. Hiring panel reviewer — Staff engineer at Stripe / GitLab in 10 minutes

Opens README.md → portal index. Notices: 32 ADRs with lifecycle badges (positive signal); Diátaxis-pure public tree (positive); page-history per page (positive); audit pages with scoring tables (positive — meta-evidence of self-assessment). Friction: opens an API reference, sees Swagger UI 2018-era; opens internal/catalog/, sees HTML triplet rather than catalog-info.yaml; checks .pre-commit-config.yaml, sees no Vale / cspell / lychee. Conclusion: structural thinker with strong governance instincts; tooling stack lags 2025–2026 default.

5. Compliance / Auditor reviewer — privacy + security audit pass

Looks for SECURITY.md at root — absent. Looks for /.well-known/security.txt — absent. Looks for a threat model — finds public/explanation/security-model.html, narrative only, no STRIDE / Threat Dragon artefact. Looks for PII / retention on DB tables — ADR 0029 mandates the contract but fields are pending backfill. Looks for GDPR Article 30 / DPIA records — absent. Friction: the auditor would mark this engagement as insufficient evidence for a privacy / security review, despite the architecture being defensible — the artefacts are missing, not the substance.

Senior SA hiring perspective

A Senior Systems Analyst / Staff PM in an international company evaluating this repository as a portfolio artefact registers, in the first 10 minutes, four positive signals and four concerns. v2 sharpens both lists vs v1.

Positive signals. (1) The 32-ADR / 6-RFC corpus with explicit lifecycle (ADR 0018) reads as a person who thinks in decisions, not in tickets. (2) The Diátaxis-pure public tree (tutorials / how-to / reference / explanation) signals taught information architecture, not accidental folders. (3) OpenAPI drift gate + contract test (scripts/openapi_governance.py) shows comprehension of contract-as-source. (4) Page-level history on every governance page is a discipline most teams skip.

Concerns. (1) HTML-first authoring with custom repair_docs_html.py tooling reads as accidental complexity. (2) Service catalog without catalog-info.yaml reads as un-Backstage. (3) Total absence of Vale / lychee / cspell / Spectral / Lighthouse reads as "I built the cathedral but skipped the inspections". (4) Trust layer — no SECURITY.md, no security.txt, no PIA, no analytics — fails a compliance-aware first read.

Net signal. Strong structural-thinking signal; weaker tooling-adoption signal; a trust gap that is cheap to close. The roadmap below is the conversation a hiring panel would want to have: given limited budget, which moves first, and why? Pre-thinking the triage is itself the Senior SA differentiator.
