Tech Docs Maturity Assessment v1 — formed 10 May 2026

Assessment type: Tech Docs Maturity — Documentation Operations & Knowledge Management. Companion assessments: DX Assessment (1 May 2026) for end-user docs UX, Project Maturity (10 May 2026) for engineering / SRE / security. This audit complements both — its focus is how documentation work is organized: templates, processes, automation, governance, tooling, AI readiness, analytics.

Overview

Reference standards: Diátaxis framework, MADR 4.0, IETF RFC 7322, OpenAPI 3.1, AsyncAPI 3.0, JSON Schema 2020-12, Backstage software-catalog, GitLab Handbook, Stripe API reference, Google / Microsoft style guides, Vale.sh, Spectral, kapa.ai / Mintlify AI, llms.txt.

Overall: ≈ 5.4 / 10. Templates, decision records, and information architecture are punching well above weight (Axis 2 / 9 / 1 score in the upper-middle band thanks to 32 ADRs, 5 ratified RFCs, the public Diátaxis split, and the documentation-authoring RFC 0003). Modern tooling stack, service catalog, and 2025–2026 frontier practices are the structural gaps: there is no SSG, no Backstage catalog-info.yaml, no Vale / lychee / cspell, no llms.txt, no Ask-AI widget. The strongest single signal is Axis 9 (Decision Records, ≈ 7.6 / 10); the weakest is Axis 10 (AI-Augmented Docs, ≈ 1.8 / 10).

Hiring lens. The combined evidence (32 ADRs with lifecycle, 5 RFCs, Diátaxis-pure public tree, page-level history widely deployed, OpenAPI drift gate in CI, ADR 0029 DB documentation contract, homegrown service catalog inspired by Backstage) reads as a Senior Systems Analyst signal of structural thinking. Where it falls short of top-tier IT (Stripe / GitLab / Spotify) is in DocOps automation depth, modern off-the-shelf tooling, and AI-readability — all addressable inside the 12-month roadmap below without a rewrite.

Scope and methodology

Scope

This assessment covers the organization of technical documentation across the repository: ADR / RFC / API / DB / runbook / postmortem templates, the public and internal portal trees under services/portal/, the homegrown service catalog at services/portal/internal/catalog/, the docs-build / docs-check pipeline driven from the Makefile and scripts/, the OpenAPI governance pipeline at services/portal/public/reference/api/, and CI enforcement under .github/workflows/ + .pre-commit-config.yaml. End-user UX of the rendered portal is out of scope (covered by the DX stream); SRE / security / CI/CD engineering maturity is also out of scope (covered by the Project Maturity stream).

Methodology

The 100 practices are grouped into 10 weighted axes summing to 100%. Each row is scored 1–10 against a top-tier benchmark (Stripe, GitLab, Spotify Backstage, Netflix, Twilio, Mintlify, Linear, Vercel) with one of three justification stamps: confirmed (a file path, CI job, or ADR proves the practice), partial (some requirements met, others missing — both facts cited), not-confirmed (the score relies on absence evidence or visual inspection only). No row is scored from a memory of how things should work.

Severity scale used in the gap tables: SEV-1 baseline practice missing or broken; SEV-2 2023+ industry-standard practice absent or out-of-date; SEV-3 2025–2026 frontier practice absent; SEV-4 polish / tech debt without immediate harm.

Files and directories read for this assessment: Makefile, .pre-commit-config.yaml, .github/workflows/ci.yml, .github/workflows/a11y-formal-checks.yml, .github/workflows/dead-code.yml, .github/workflows/changelog-draft.yml, .github/workflows/docs-feedback-cadence.yml, .github/workflows/pr-template-check.yml, services/portal/public/{tutorials,how-to,reference,explanation}/, services/portal/public/reference/api/openapi-baseline.json, services/portal/internal/governance/adr/ (32 ADRs incl. 0018, 0024, 0025, 0026, 0027, 0029), services/portal/internal/governance/rfc/ (5 RFCs incl. 0003 authoring model), services/portal/internal/catalog/{api,portal,monitoring}/, services/portal/internal/handbook/{howto,developer,qa}/, services/portal/internal/sre/{runbooks,postmortems}/, scripts/build_docs_search_index.py, scripts/check_adr_rfc_links.py, scripts/check_asset_refs.py, scripts/validate_docs_a11y.py, scripts/validate_docs_design.py, scripts/validate_docs_html.py, scripts/ensure_docs_maintainers.py, scripts/ensure_docs_page_history.py, scripts/openapi_governance.py, alembic/versions/.

Persona walkthroughs

Three personas were walked through three pages each (one ADR, one API endpoint, one catalog entry) to surface where docs friction shows up in practice.

Persona 1 — New external developer, "make my first request in 5 minutes"

  • Public landing → tutorials → first-api-call.html. Diátaxis quadrants visible — good signal. Lands on the right page on the first click.
  • API reference → openapi-explorer.html. Swagger UI loads (ADR 0022). Try-It works but the UX feels 2018 — no curl / Python / JS tab generated from the spec, no one-click "Copy as curl".
  • Authentication how-to. Concise; no interactive token field. Reader has to jump between three pages to assemble a working request.
  • Verdict: 5-minute-to-first-call goal is realistic but not yet smooth.

Persona 2 — Senior internal contributor, "why was decision X taken?"

  • Internal sidebar → governance → ADR index. 32 ADRs, sorted, with a single template — strong signal. Search via the custom client-side index resolves "openapi" and "logging" instantly.
  • Open ADR 0007. Context, decision, consequences are clear. Missing MADR 4.0 fields ("Decision Drivers", "Considered Options" as a comparable list) slow the reader.
  • Trace supersession. Hyperlink chain works textually. No graph rendering — following a chain that crosses 3 ADRs requires opening 3 tabs.
  • Verdict: the corpus is strong; navigation through it could be visualized.

Persona 3 — On-call engineer at 3 a.m., "what do I do when X is failing?"

  • Internal sidebar → SRE → runbooks index. 11 runbooks with a uniform skeleton — strong signal. Naming is alert-shaped ("tests-failing", "logging-failing").
  • Open 0006-api-security-failing.html. Steps are clear; no embedded escalation matrix beyond a link. Diagnosis ≠ rapid green-amber-red.
  • Open 0009-error-budget-exhaustion.html. Procedure exists; no PagerDuty / Opsgenie wiring documented. Linkage from alert → runbook is implicit.
  • Verdict: the bones are there, but the runbook lacks "press button" decisive UX.

Scoring summary

Narrative overall

Overall: ≈ 5.4 / 10. The repository's documentation is organized at a level that is uncommon for one-author projects — a Diátaxis-pure public tree, a 32-ADR corpus with a formal lifecycle, 5 ratified RFCs, an OpenAPI drift gate, page-level history on every governance page, and a homegrown Backstage-inspired catalog. These read as deliberate structural choices, not retrofits.

The gap to top-tier IT (Stripe / GitLab / Spotify / Mintlify-grade) is concentrated in three clusters: (1) DocOps automation depth — Vale, lychee, cspell, Lighthouse, Spectral, snippet-tests are absent or only partial; (2) off-the-shelf modern tooling — no SSG, Swagger UI instead of Scalar, custom search instead of Algolia / Typesense, HTML-first authoring instead of Markdown-first; (3) 2025–2026 frontier — no llms.txt, no Ask-AI widget, no chunking, no AI-cited answers. Each cluster has a clean addressable path that is laid out in the roadmap.

The single most consequential structural decision pending is whether to migrate authoring source from HTML to Markdown / MDX. That decision unlocks Axis 6 (Modern Tooling) and Axis 10 (AI-Augmented Docs) simultaneously — at the cost of one focused refactor. RFC 0003 already names Astro Starlight as a candidate; treating that RFC as the trigger for the migration would convert two weak axes into mid-band scores in one motion.
