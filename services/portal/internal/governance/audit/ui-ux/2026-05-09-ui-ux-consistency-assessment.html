UI/UX consistency & design-system assessment — Study App (formed 2026-05-09)

Overview

Snapshot: assessment formed on 2026-05-09 against repository commit 9ac59e5 on branch chore/bug-audit-resolution-and-portal-hall. This assessment succeeds the prior UI/UX snapshots (2026-04-24 and 2026-04-23) and is complementary to the 2026-05-01 DX assessment (which scores the documentation programme as a whole). This assessment narrows the lens to one question: is the portal a single, coherent design language with reusable components, or a federation of similar-looking pages with hidden drift?

Headline: overall 6.6 / 10 against a top-tier benchmark (Stripe Docs / Linear / Vercel / GitHub Primer / Radix / Polaris / Material 3 / Apple HIG). The portal is already strong on layout, typography ladder, page-history reuse, and table grammar. It is held back by four systemic gaps: (a) no z-index or shadow token tier (34 hardcoded z-indices, ~76 inline shadow declarations), (b) three parallel status-rendering systems (audit-gap-status--*, score-*, plain-text "Active"), (c) no copy-to-clipboard affordance on any of the 35 pages with <pre><code> blocks, and (d) manual sidebar tree (INTERNAL_SIDEBAR_NAV) keeps cost-to-add at 2-4 files per new entity, with orphan-page risk on every new ADR/RFC/audit page.

Where the repo provides direct evidence (a script, a CI gate, an ADR), the row is confirmed; where the score relies on inspection only, the justification marks it not confirmed.

Scope and methodology

Scope

UI/UX consistency here means: one design language (tokens + components + patterns + content), one canonical implementation per semantic brick, predictable page templates per entity type, and a low cost-to-add for new entities. The benchmark is top-tier developer-docs portals: Stripe Docs is the primary reference (closest jobs-to-be-done overlap); Linear, Vercel, GitHub Primer, Radix Themes, Shopify Polaris, Material 3, and Apple HIG are secondary references for specific patterns. This assessment is not a full DX audit (see 2026-05-01 DX assessment), not a Core Web Vitals audit, and not a WCAG 2.2 AA conformance audit.

Method

  1. Token map: every CSS variable defined in :root and [data-theme="dark"] across docs.css, docs-theme.css, docs-premium.css, docs-syntax-theme.css, docs-site-nav.css, internal-layout.css, public-layout.css, home.css, portal-router.css; matched against literal usage (hex / rgb / px / shadow / z-index) for leakage counts.
  2. Component inventory: for each semantic brick (status pill, callout, page-meta block, page-history, hero, code block, audit tables, drawer/sidebar) all implementations were enumerated to measure how often the canonical class is reused versus duplicated.
  3. Page-template inventory: three sample pages per entity type (ADR, RFC, internal how-to, public how-to, public explanation, API operation, catalog API reference, handbook QA, audit assessment) checked for canonical section order, meta-mount presence, h1 styling, page-history block.
  4. Status vocabulary matrix: every status word rendered across ADR / RFC / audit / page-meta mapped to its CSS class and visual treatment.
  5. Cost-to-add count: last 10 commits adding new entities scanned via git log --diff-filter=A; files-touched-per-entity averaged separately for ADR / RFC / how-to / audit; auto vs. manual sources of truth identified.
  6. Stripe Docs walk-through: identical reader scenario (find a status, read an explanation, jump to a how-to, copy a code sample, return to TOC) compared with our portal to identify concrete benchmark deltas.

Scores are subjective guesses; the goal is alignment, not precision. Where automation already enforces a property (make verify, drift gate, spec lint, link-checker), the score is bumped — we trust the gate, not a recollection. Where a benchmark practice is consciously out of scope at PET-scale, the row is marked accordingly rather than penalised.

What this assessment is not

  • Not a substitute for usability testing with real readers.
  • Not a Core Web Vitals or WCAG 2.2 AA conformance audit — both need separate tooling.
  • Not a redesign proposal. Findings name drifts and gaps; remedy choices belong to follow-up ADRs/RFCs.

Reading the matrix: the portal currently renders status across at least five distinct vocabularies (audit gap, audit score, plain-text active/superseded, RFC inline, spec page-meta). Three of them have CSS classes; two are text-only. A reader scanning the audit index sees no visual difference between an active and a superseded assessment, and there is no shared "status pill" component that can be dropped onto a new entity page.

Action plan: roadmap to top-tier

Four phases, each bounded enough to fit a working week or two. Order matters: foundations (tokens) must precede component consolidation, which must precede polish.

Phase 1 — Foundations (tokens + status vocabulary)

  • Add --z-sticky, --z-popover, --z-dropdown, --z-modal, --z-toast tokens in docs-theme.css; replace ~34 hardcoded values.
  • Extend the shadow ladder to 5 tokens (--shadow-xs / sm / md / lg / xl); replace ~76 inline shadows.
  • Define one canonical status vocabulary (active / draft / proposed / accepted / rejected / superseded / deprecated / beta / stable / experimental) with one CSS class per status.
  • Replace plain-text "Active" / "Superseded" in audit/index.html and audit/docs/index.html with the new component.
  • Definition of done: 0 inline box-shadow, 0 numeric z-index, 0 plain-text status cells; lint enforces all three.

Phase 2 — Components consolidation

  • Migrate callout--warn → docs-callout--warning; remove bare docs-callout instances.
  • Define hero family in RFC 0003 appendix: router / animated / section / minimal; assign each surface to one variant.
  • Inject copy buttons via docs-syntax.js on every <pre><code>; add idle / hover / copied / error states using motion tokens.
  • Decide on profile-page meta: unify with data-maintainer-ids or document as separate sub-system.
  • Definition of done: 1 callout naming scheme, 4 named hero variants, 35 / 35 code blocks have copy buttons.

Phase 3 — Polish & motion

  • State-completeness audit: per-component matrix of idle / hover / focus-visible / active / disabled / loading / error.
  • Skeleton loaders for Scalar/OpenAPI mounts and PlantUML diagrams.
  • Empty-state copy and error-state copy with action affordance.
  • Run axe-core / pa11y over the portal and close gaps until WCAG 2.2 AA conformance is provable.
  • Definition of done: a state matrix lives in internal/front/foundations/; CI runs axe-core nightly.

Phase 4 — Maintainability (cost-to-add ↓)

  • Auto-generate INTERNAL_SIDEBAR_NAV from page front-matter or file tree.
  • Extract shared sidebar utilities into sidebar-utils.js.
  • Add lint rule: every page in services/portal/ appears in either internal or public sidebar (no orphans).
  • Add lint rule: token leak detector (deny hex / numeric z-index / inline shadow).
  • Definition of done: avg files-per-new-entity drops to 1 for ADR / RFC / how-to / catalog; CI catches orphan pages.

12-point self-check checklist (run before every PR touching services/frontend/portal/ or services/portal/)

  1. Every new colour / shadow / radius / spacing / motion / z-index value uses a token; no inline literal.
  2. No font-size declared inside a width-conditional @media query.
  3. Status words use the canonical .docs-status--* class; no plain text in cells that mean state.
  4. Callouts use docs-callout + a variant; no bare docs-callout; no callout--warn alias.
  5. Code blocks ship a copy button (auto-injected); aria-live for the copied confirmation.
  6. Hero blocks use one of the four named variants from RFC 0003 appendix; no new ad-hoc hero.
  7. New page has data-maintainer-ids, docs-page-meta-mount, internal-h1--accent-tail, and a page-history section.
  8. New page is reachable from the sidebar (INTERNAL_SIDEBAR_NAV updated, or auto-derive script run).
  9. Dark theme renders without missing tokens (visual smoke check).
  10. Keyboard reachability: every interactive element has a visible focus-visible style and tab order is logical.
  11. Diagrams are PlantUML (per ADR 0020); no Mermaid.
  12. make verify passes locally before push.
