What shipped

This page used to be a planning document — twenty-eight documentation tasks, four waves, six candidate ADRs, and a 2026-Q3 sprint calendar. The plan ran; the shape it took by the end is different enough that «what we did» reads more cleanly than «what we planned to do». So this page was rewritten as a retrospective on 2026-07-09.

The three durable outcomes of the epic:

  • 16 machine-readable contracts under services/portal/internal/services/api/openapi/etr_study_app/fragments/ — one YAML fragment per endpoint. Every fragment is validated by make api-check; the union is regenerated into merged-spec.json.
  • 16 analyst spec pages under services/portal/internal/services/api/reference/{resource}/operations/ — one 19-section HTML page per endpoint (Business context → Acceptance criteria → Response schema → Observability → Test plan → Traceability).
  • A parity gate that enforces canon ↔ code. make openapi-check lints the runtime spec and checks canon ⊆ shipped code ⊆ canon ∪ exceptions. A silent field rename fails the gate the same way it used to fail the retired snapshot diff.

How the pipeline is wired

A single command, make docs-fix, drives the whole stack. Five layers, each one a distinct artefact, each one machine-checkable.

  1. Analyst spec page. 19 sections of HTML under services/portal/internal/services/api/reference/{resource}/operations/. This is the human view — Business context, Acceptance criteria in Given/When/Then, Response schema tables, Test plan. Rendered on the internal portal. Gated by make docs-check (drift), make docs-html-check, make docs-a11y-check, and tools/docs/spec_consistency.py (operationId ↔ status ↔ oracle).
  2. Canon fragment. One OpenAPI 3.1 YAML per operation under openapi/etr_study_app/fragments/{resource}/. Carries the machine-readable contract — request schema, response schemas, error envelope, examples, and the x-implementation-status tag. Reviewed in its own PR and never regenerated from anything else. Gated by make api-check (validator + $ref resolution).
  3. Merged spec. make api-check concatenates every fragment into merged-spec.json under the same directory. This is a build artefact — never hand-edited, never reviewed as a diff. Downgraded from OpenAPI 3.1 to 3.0.3 in the process so tools that don't speak 3.1 (Connexion, Scalar) still work.
  4. Mock server. make api-mock boots a Connexion + Flask + Uvicorn process (per canon, on its own port — etr_study_app at :8002) that serves the merged spec with dynamic example overlay. The client can post real HTTP against it via Swagger UI at /ui/. This replaces the originally-planned Prism sidecar — Python-only stack, no Node dependency.
  5. FastAPI runtime. The production API under services/api/app/. make docs-fix step 8 dumps its app.openapi() to services/portal/public/reference/api/openapi.json, which the public Scalar explorer reads. make openapi-check then enforces parity: every canon operation tagged shipped must be implemented, and no code operation may exist outside canon or the exceptions allow-list.

The parity gate — canon ⇌ code

The gate lives in tools/api/openapi_governance.py and runs on every make openapi-check. It enforces three invariants:

DirectionWhat must holdWhat breaks it
canon → code Every operation tagged x-implementation-status: shipped (or the tag omitted) must exist in the FastAPI runtime spec with matching operationId, required parameters, required request-body fields, and response codes. Silent field rename, dropped response code, weakening a required response field to optional — all fail the check.
code → canon Every operation in the runtime spec must be in canon OR in KNOWN_NON_CANON_OPERATIONS (currently: /live, /ready). Shipping a public-surface endpoint without a canon fragment fails on next CI.
lint Every operation has an operationId, a summary, and — for writes — a 422 response with examples. Forgetting the 422 when adding a POST/PUT/PATCH/DELETE.

The x-implementation-status: pending tag on the twelve canon-first fragments is what lets the gate stay green today — the canon says «this operation exists», the code doesn't yet, but the tag makes that discrepancy intentional and reviewable. Flipping a fragment from pending to shipped (or dropping the tag) in the same PR that lands the FastAPI handler is the natural checkpoint: the parity gate starts protecting the operation the moment the code exists.
