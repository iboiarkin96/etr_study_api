Working method

The analytical pass that produces this epic runs in five stages. Each stage has named outputs and an objective exit criterion — the kind of criterion a reviewer can check without re-running the analysis themselves.

The five stages

StageWhat it producesExit criterion
1. Discovery Problem statement · user-journey map · glossary · stakeholder map. Each persona has at least one Job To Be Done; the glossary covers every term used in any journey.
2. Analytics Capability map · traceability matrix seed · NFR baseline (latency · idempotency · durability · privacy · observability budgets) · risk register. Every capability traces to a JTBD; every NFR has a measurable target with a number, not an adjective.
3. Design One OpenAPI fragment per operation · the Operation spec page §1–§19 · ADRs for contested choices. Spectral lint clean; §3 written as Given/When/Then; every open ADR thread is Accepted or explicitly Deferred with a written trigger condition.
4. Roadmap The dependency DAG · sprint allocation · risk-burn ordering · release-train slots. Every task has acceptance criteria, an owner, and a sprint; the critical path is highlighted.
5. Hand-off Contract-test plan · runbook stub · observability dashboard list · rollout / rollback plan. Engineering can implement the spec without re-asking; QA can author tests from the spec alone.

Traceability spine

The thread that lets any reviewer walk in either direction — from a failing test back to the business goal, or from the business goal forward to «which dashboards page if this breaks». Every artifact this epic produces hangs from the spine; §19 of every Operation spec page closes the loop.

Business goal (the ETR study loop)
└── Use case (a learner Job To Be Done)
    └── Capability (verb-noun on a resource — «Create note», «List due notes»)
        └── API operation (Method + Path + operationId)
            ├── ADRs (frozen decisions — slot policy, TZ semantics, soft-delete)
            ├── NFRs (latency · idempotency · durability · privacy)
            ├── Contract tests (Given/When/Then ↔ §3 of the spec)
            └── Runbook + dashboard (when production breaks)

Principles that hold the method together

  1. Contract before code. No FastAPI handler is touched before its Operation spec is frozen and its OpenAPI fragment lints clean. This is the policy of ADR 0036.
  2. Non-functional requirements are numbers. Latency budgets, idempotency windows, retention windows, privacy classes — stated in §6 / §7 of every spec, before code exists. Adjectives are not requirements.
  3. Contested choices become ADRs, not arguments in the spec. Six contested choices in this epic — slot policy, TZ semantics, soft-delete, cue-sheet shape, search policy, evening-random — are named, owned, and ratified once. The spec quotes the ADR.
  4. Sequence by risk-burn. Wave 1 is six ADRs because they unblock eleven downstream specs. The DAG is ordered by which artifact removes the most uncertainty next.
  5. Observability and rollback live in the spec. §14 (events / metrics / alerts) and §17 (rollout / rollback) are non-optional. The hand-off package exists before the code does.
  6. Defer in writing, with a trigger. Some decisions (full-text search, schedule preview) are P2/P3 with explicit trigger conditions for revisiting. Silent deferral is what creates drift.

API-first practice — how this epic is built

«API-first» is a phrase you can paint on a wall; the discipline that puts a contract before a handler is what changes behaviour. This epic operationalises it through two artifacts:

  • Policy: ADR 0036 — hand-authored OpenAPI fragments as the single source of truth. Sets what is canonical and what gates the release.
  • Recipe: Handbook · SA · Authoring · Apply API-first. Seven steps, the OpenAPI fragment skeleton, the tools, anti-patterns, and a troubleshooting table.

Source of truth — hybrid, with the fragment as canon

  • Canon: hand-authored fragments at services/api/openapi/test/fragments/. Reviewed in their own PR. Ratification flips the spec status from draft to ratified.
  • Claim: FastAPI's runtime /openapi.json. Documents what the running code does — never the contract.
  • Gate: openapi-diff in CI. Drift fails the build. Tooling enforces the discipline, not good intentions.

Authoring flow

Author OpenAPI fragment
  └→ make doc-api-swagger-check (openapi_spec_validator + $ref resolution)

Single-language Python toolchain — same .venv as the API, no Node.js side-stack. The tutorial at handbook/sa/tutorial/api-first.html walks a first-time reader through these end-to-end; the recipe at handbook/sa/authoring/apply-api-first.html is the recurring form.

Verification checklist

How we know this artifact is done well. Each item is gateable, not «looks fine».

  • make docs-check passes — no autogen drift, no broken internal links.
  • make docs-fix regenerates the Pagefind index; this page and ADR 0036 are searchable.
  • Sidebar shows the new nodes: «API Analytics epic» under Governance → Backlog; «0036 · API-first…» under Governance → ADR; «Apply API-first» under Handbook → SA → Authoring.
  • Breadcrumbs resolve cleanly on each page (Internal → Governance → … or Internal → Handbook → SA → Authoring → …).
  • data-page-type="reference" on both governance pages; "how-to" on the recipe page. ADR 0036 carries the lifecycle popover at status «Proposed».
  • Every Task-ID cell in §5 has an <a> link to its eventual file path (forward link, broken until the file lands — intentional and documented inline).
  • Two-way trace: ADR 0036 links to this epic (SA-002 task) and to the recipe; this epic links to both; the recipe links back to ADR 0036.
  • No <style> blocks, no style="…" attributes — every visual lives in the kit.
  • No «See also», «Sources», or «Bookmarks» trailing blocks. Inline see-also links inside cards are fine.
  • «Implementation in this repository» on ADR 0036 renders as a docs-table, not a nested <ul>.
  • Code in <pre><code> blocks starts at column 0 — no leading HTML indent leaking in.
  • Coverage: §2 lists fourteen endpoints; §5 enumerates twenty-eight task IDs; six new ADR slots (0036–0042) are named, of which 0036 ships in this PR.
