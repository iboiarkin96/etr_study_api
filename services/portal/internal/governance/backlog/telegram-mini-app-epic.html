Telegram Mini App epic

What we're building

A Telegram-native client that covers the ETR daily loop:

  1. Look at today. Open the app → see what's due for review. Nothing to think about, no filters to touch.
  2. Review a conspectus. Tap a card → read the note → mark reviewed / snoozed with a single tap.
  3. Log an error. When retrieval fails during review, add an error-log entry inline — the source conspectus is already known.
  4. See what's next. Glance at the 14-day schedule strip → know when the next wave lands.

The bot half provides the trigger: a daily push at a chosen slot with a deep-link that opens the Mini App on the Today screen. Everything else lives inside the WebApp.

Locked decisions

Five choices ratified up-front. Anything in this table changes only via an explicit follow-up in the History section.

Question Choice Why
Format Bot + Mini App Bot is the notification lane (push «X due today» → deep-link). Mini App is the UX. Bot alone is too poor visually; Mini App alone can't push.
Frontend stack React + @telegram-apps/sdk-react + @telegram-apps/telegram-ui Official SDK; ready-made iOS/Material components that autoswitch on platform; TS-first; largest ecosystem. Vanilla would mean reimplementing every Cell / Section from scratch.
Location in repo services/telegram/ (own service) Separate ship-unit with its own README, Makefile, Dockerfile, deploy target. Matches ADR 0028.
Auth Existing X-API-Key header, simplified The API already enforces a key header on /api/v1. We drop the API_AUTH_STRATEGY switch and keep a plain header + key pair (API_AUTH_HEADER / API_AUTH_KEY). Zero new auth machinery, zero token exchange in the client. Telegram initData → JWT is deferred to P2 — it becomes mandatory the day the audience opens beyond the maintainer.
Audience Maintainer only Cuts the whole «public onboarding + conspectus editor + moderation» stack. If it ever opens up, that's a separate epic — not a scope creep on this one.

Waves — the work plan

Six waves, one PR per wave (W2 ships as two PRs — bootstrap, then the Today screen). Each wave is independently mergeable — end of a wave means the app compiles, tests are green, and the previous wave's contract is preserved.

  1. W1 · Backend: auth simplification & Telegram config. No new auth machinery — the API keeps its existing header-key check, minus the strategy switch. app/core/security.py drops the API_AUTH_STRATEGY branching; config keeps API_AUTH_HEADER (default X-API-Key) + API_AUTH_KEY (renamed from API_MOCK_API_KEY); qa/prod guard changes from «strategy ≠ disabled» to «key must be non-empty». /live and /ready stay open (unchanged). TELEGRAM_BOT_TOKEN + TELEGRAM_CHAT_ID added to settings and env/* — consumed by the bot in W5; register the BotFather test bot now so the values exist. CORS_ALLOW_ORIGINS extended with the Vite dev origin (http://localhost:5173) and the tunnel origin. ADR 0038 «Personal-mode auth: reuse X-API-Key, defer Telegram initData» drafted & accepted. OpenAPI baseline regenerated; make openapi-check green. Exit criteria: pytest covers valid / missing / wrong key on protected routes, and the config guard for empty key in qa/prod.
  2. W2a · Frontend bootstrap. services/telegram/ created — Vite + React 19 + TypeScript. Deps: @telegram-apps/sdk-react, @telegram-apps/telegram-ui, @tanstack/react-query, @tanstack/react-router (type-safe routing + built-in View Transitions), zustand (UI state), framer-motion (interactions + shared-element), posthog-js (analytics). Providers: QueryClient, Router, ThemeProvider (maps themeParams → semantic CSS vars), ViewportProvider (writes --tma-safe-top / --tma-safe-bottom / --tma-viewport-h from viewportStableHeight). Auth: the generated client sends X-API-Key from VITE_API_KEY on every request and a fresh Idempotency-Key (uuid v4) on every write — no token exchange, no storage concern in personal mode. Dev: plain browser tab — mockTelegramEnv() stays only as a 3-line dev-only shim so the SDK boots (theme, viewport); auth never touches initData, so no signed payload and no tunnel are needed for UI work. Cloudflare Tunnel only for on-device dev with real Telegram. i18n scaffold — i18next + ru / en from language_code. Storybook boot for TMA components.
  3. W2b · Today screen (end-to-end). First working screen against real API — Yesterday-digest strip, streak ProgressRing, Recently-reviewed carousel, 90-day HeatmapCalendar, due list with 4 swipe gestures (mark done · snooze · destructive · star). Contract note: snooze is not a separate endpoint — every review outcome, snooze included, is a tag (easy · hard · forgot) on POST /api/v1/conspectuses/{id}/actions/review; the exact UI-action → tag mapping is pinned in the operations spec page, not here.
  4. W3 · Screens: Conspectus, Schedule, Errors, Focus, Search, Profile, Onboarding. Full 8-screen surface — read-only Conspectus with AI micro-actions (Explain differently, Generate quiz), Focus mode (SRS review flow — full-screen card, tap to reveal, Again/Hard/Good/Easy grades, SM-2 next-interval preview), Schedule with 90-day heat-map, Errors with timeline + pattern grouping, Profile with achievements + weekly retention chart, first-run Onboarding, Search / Spotlight (Cmd+K on desktop, pull-down on mobile). All screens hooked to real API via the OpenAPI-generated client (openapi-typescript). Every screen ships loading (content-aware skeleton), empty (with illustration + rotating tip), and error toast states.
  5. W4 · Polish · motion + delight. Framer Motion springs on every mount / dismount. View Transitions API for route changes (Today card → Focus card = shared-element). Haptics vocabulary (7 patterns — selection, impact light/med/heavy, notification success/warning/error). BackButton + MainButton + SettingsButton wired via SDK. Content-aware loading skeletons. Toasts with undo. Milestone celebrations (Lottie for streak 7/30/100). Time-aware copies. Sound cues (opt-in, muted by default). Sentry + PostHog with event spec (review_completed, review_snoozed, error_logged, streak_broken, focus_session_ended, search_opened).
  6. W5 · Bot & deep-linking. Python aiogram handlers colocated inside services/api (shares models, config, and Postgres session). aiogram Dispatcher started on FastAPI lifespan startup; dev uses long-poll, prod uses webhook mounted at POST /api/v1/telegram/webhook with X-Telegram-Bot-Api-Secret-Token header check. /start greets and offers the Mini App launch button. Deep-link scheme uses the modern HTTPS format — https://t.me/<bot_username>/<app_short_name>?startapp=conspectus_<uuid>. The startapp payload is exposed to the WebApp via WebApp.initDataUnsafe.start_param and drives router navigation.
  7. W6 · Deploy, docs, perf budget, DoD. Mini App shipped to Cloudflare Pages (custom domain, HTTPS). Bot webhook served from the API container. Portal gains a service page under services/portal/internal/services/telegram/ — one page per Diátaxis quadrant, plus a public design case-study on the blog. Backlog cockpit reflects «done». make verify extended: builds the Mini App, typechecks, runs Vitest + Playwright, and enforces the performance budget via Lighthouse CI — LCP ≤ 2.0s, TBT ≤ 200ms, bundle ≤ 200 KB gz, a11y ≥ 95. Epic closed.

Non-functional guarantees

Ambition: portfolio-grade. Every point below is a hard gate or a first-class design concern — not an afterthought.

Auth & storage

  • X-API-Key injected at build time via VITE_API_KEY — no token storage in the client at all; nothing in localStorage.
  • The key ships inside the bundle — accepted for personal mode, rotated if leaked; P2's initData → JWT removes it entirely.
  • BiometricManager is used to gate destructive actions (delete conspectus, reset schedule) and streak-break confirms.
  • No DB migration in MVP — users.telegram_* identity columns arrive with P2 auth.

Performance budget

  • LCP ≤ 2.0s on cold LTE (Moto G Power throttle).
  • JS bundle ≤ 200 KB gz initial + code-split routes.
  • No web fonts loaded from network — system stack only.
  • Lighthouse CI in make verify; PR fails on regression.

Motion & a11y

  • Every animation route through motion tokens; prefers-reduced-motion collapses to instant.
  • Tap targets ≥ 44 × 44 CSS px enforced by the cell primitive.
  • Keyboard navigable end-to-end on Telegram Desktop (J/K/E/? shortcuts + Cmd+K).
  • VoiceOver / TalkBack pass on every screen (rotor + live-region).

Observability

  • PostHog event spec: review_completed, review_snoozed, focus_session_ended, error_logged, streak_broken, search_opened, command_ran.
  • Sentry client (opt-in via VITE_SENTRY_DSN) with Telegram context tags.
  • Bot-side: structured logs with correlation_id, joined to API logs.

Dev environment

  • Plain browser tab is the primary dev loop: auth is an X-API-Key header, so no signed initData is ever required. mockTelegramEnv() survives only as a 3-line dev shim that lets the SDK boot (theme + viewport).
  • Cloudflare Tunnel scripts checked in for on-device dev with real Telegram.
  • Storybook for every .tma-* primitive; visual QA lives here, not on the phone.
  • Playwright uses mockTelegramEnv() in CI — no external HTTPS required.

i18n & localisation

  • All copy through i18next; ru + en at MVP, chosen from Telegram's language_code.
  • Numbers use Intl.NumberFormat; dates use Intl.RelativeTimeFormat.
  • Timezone from Intl.DateTimeFormat().resolvedOptions().timeZone, persisted server-side for reminder scheduling.

Definition of Done

  • Opening the bot in Telegram → /start → tap «Open study_app» launches the Mini App.
  • Every /api/v1/* route (except /live, /ready) requires the X-API-Key header — enforced by the existing middleware, covered by tests.
  • All nine MVP screens render on iOS + Android Telegram clients — dark and light themes.
  • Adding an error, marking a review, and reading a conspectus persist against the real Postgres and survive a page reload.

  • Portal has a service page under services/portal/internal/services/telegram/, ADR 0038 is Accepted, and this epic is Delivered in the backlog cockpit.

Risks & open questions

HTTPS in dev

Telegram requires HTTPS for the WebApp URL — but only for on-device sessions. The primary dev loop is a plain browser tab (no tunnel at all). When on-device testing starts: Cloudflare Tunnel; fallback ngrok or hosts-file + mkcert.

Bot polling vs webhook in dev

Webhook needs a public URL. In dev, aiogram long-polling is fine; W5 flips to webhook only for prod. Config lives in env/*.

Deep-linking scope

Default launch opens Today. Per-conspectus deep-links (startapp=conspectus_<uuid>) land in W5 together with the webhook — nothing deep-link-shaped ships before that. Document the URL scheme in the service page — kits and callers rely on it.

API key ships in the bundle

With header-key auth the key is readable by anyone who opens the deployed bundle. Accepted trade-off for personal mode: the key guards one user's study data, is rotated on suspicion, and disappears entirely when P2 moves to initData → JWT. Recorded in ADR 0038.

Reminders scheduler

Deferred to P2 — MVP relies on user opening the bot on their own. Once scheduled, decide: APScheduler in-process vs external cron. Note in the ADR log once chosen.
