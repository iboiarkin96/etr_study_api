What we're building

A Telegram-native client that covers the ETR daily loop:

  1. Look at today. Open the app → see what's due for review. Nothing to think about, no filters to touch.
  2. Review a conspectus. Tap a card → read the note → mark reviewed / snoozed with a single tap.
  3. Log an error. When retrieval fails during review, add an error-log entry inline — the source conspectus is already known.
  4. See what's next. Glance at the 14-day schedule strip → know when the next wave lands.

The bot half provides the trigger: a daily push at a chosen slot with a deep-link that opens the Mini App on the Today screen. Everything else lives inside the WebApp.

Locked decisions

Five choices ratified up-front. Anything in this table changes only via an explicit follow-up in the History section.

Question Choice Why
Format Bot + Mini App Bot is the notification lane (push «X due today» → deep-link). Mini App is the UX. Bot alone is too poor visually; Mini App alone can't push.
Frontend stack React + @telegram-apps/sdk-react + @telegram-apps/telegram-ui Official SDK; ready-made iOS/Material components that autoswitch on platform; TS-first; largest ecosystem. Vanilla would mean reimplementing every Cell / Section from scratch.
Location in repo services/telegram/ (own service) Separate ship-unit with its own README, Makefile, Dockerfile, deploy target. Matches ADR 0028.
Auth JWT (stateless) issued from Telegram initData No session table, easy horizontal scale, no revocation need in personal-use mode. Opaque sessions were rejected as extra state without a matching benefit.
Audience Maintainer only Cuts the whole «public onboarding + conspectus editor + moderation» stack. If it ever opens up, that's a separate epic — not a scope creep on this one.

Waves — the work plan

Six waves, one PR per wave. Each wave is independently mergeable — end of a wave means the app compiles, tests are green, and the previous wave's contract is preserved.

  1. W1 · Backend: auth & identity. ADR 0038 «Telegram initData as primary auth» drafted & accepted. Migration adds users.telegram_user_id BIGINT UNIQUE NULL, telegram_username, telegram_photo_url. Endpoint POST /api/v1/auth/telegram validates the HMAC-SHA256 signature over initData using the bot token as the key, upserts the user, and returns a JWT (24h TTL). Global FastAPI dependency get_current_user attached to every /api/v1/* router. OpenAPI baseline regenerated. Bot token loaded from env/*. Exit criteria: integration tests cover valid initData, tampered initData, expired auth_date, and unknown user.
  2. W2 · Frontend skeleton + Today. services/telegram/ created — Vite + React 19 + TypeScript, @telegram-apps/sdk-react and @telegram-apps/telegram-ui installed. Bootstrap includes initData → JWT exchange, TanStack Query client, router, theme provider that maps Telegram themeParams to CSS custom properties. First screen Today works end-to-end against a locally-run API through ngrok / Cloudflare Tunnel. BotFather test-bot registered.
  3. W3 · Screens: Conspectus, Schedule, Errors. List + detail for conspectuses (read-only). Schedule strip (14-day count-per-day). Error-log list + add-modal. All screens hooked to real API through the OpenAPI-generated client (openapi-typescript-codegen). Empty and error states shipped for each.
  4. W4 · Polish · P2 subset that fits. Haptics on action taps. BackButton and MainButton wired via SDK. Loading skeletons. Route transitions. Error toasts. Sentry client (opt-in) — reports live only when SENTRY_DSN is present.
  5. W5 · Bot & deep-linking. Python aiogram handlers colocated inside services/api (shares models, config, and Postgres session). /start greets and offers the Mini App launch button. Webhook mounted at POST /api/v1/telegram/webhook with a shared-secret header check. Deep-link scheme documented — tg://…?startapp=conspectus_<uuid> opens the Mini App on that record.
  6. W6 · Deploy, docs, DoD. Mini App shipped to Cloudflare Pages (custom domain, HTTPS). Bot webhook served from the API container. Portal gains a service page under services/portal/internal/services/telegram/ — one page per Diátaxis quadrant. Backlog cockpit reflects «done». make verify is extended to build the Mini App and typecheck. Epic closed.

Definition of Done

  • Opening the bot in Telegram → /start → tap «Open study_app» launches the Mini App.
  • Every /api/v1/* route (except /live, /ready) requires a valid Bearer JWT.
  • All five MVP screens render on iOS + Android Telegram clients — dark and light themes.
  • Adding an error, marking a review, and reading a conspectus persist against the real Postgres and survive a page reload.
  • Schedule strip reflects GET /api/v1/schedule/summary without any client-side aggregation.
  • make verify passes (lint · types · openapi · contract · pytest · docs-check · web build).
  • Portal has a service page under services/portal/internal/services/telegram/, ADR 0038 is Accepted, and this epic is Delivered in the backlog cockpit.

Risks & open questions

HTTPS in dev

Telegram requires HTTPS for the WebApp URL — even in dev. Plan: ngrok for the first spike; Cloudflare Tunnel for stable dev sessions. Fallback: hosts-file + mkcert.

Bot polling vs webhook in dev

Webhook needs a public URL. In dev, aiogram long-polling is fine; W5 flips to webhook only for prod. Config lives in env/*.

Deep-linking scope

Only the Today screen is deep-linked in MVP. Per-conspectus deep-links land in W5 via startapp. Document the URL scheme in the service page — kits and callers rely on it.

Reminders scheduler

Deferred to P2 — MVP relies on user opening the bot on their own. Once scheduled, decide: APScheduler in-process vs external cron. Note in the ADR log once chosen.
