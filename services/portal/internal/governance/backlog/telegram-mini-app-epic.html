Telegram Mini App epic

A stylish, Telegram-native mobile client for the ETR daily loop — Today · Conspectus · Schedule · Errors. Thin React app + tiny Python bot on top of the shipped /api/v1 surface. Multi-user by design — every request is authenticated against the Telegram user who opened the app.

What we're building

A Telegram-native client that covers the ETR daily loop:

  1. Look at today. Open the app → see what's due for review. Nothing to think about, no filters to touch.
  2. Review a conspectus. Tap a card → read the note → mark reviewed / snoozed with a single tap.
  3. Log an error. When retrieval fails during review, add an error-log entry inline — the source conspectus is already known.
  4. See what's next. Glance at the 14-day schedule strip → know when the next wave lands.

The bot half provides the trigger: a daily push at a chosen slot with a deep-link that opens the Mini App on the Today screen. Everything else lives inside the WebApp.

Locked decisions

Six choices ratified up-front. Anything in this table changes only via an explicit follow-up in the History section.

Question Choice Why
Format Bot + Mini App Bot is the notification lane (push «X due today» → deep-link). Mini App is the UX. Bot alone is too poor visually; Mini App alone can't push.
Frontend stack React 18 + @telegram-apps/sdk-react + @telegram-apps/telegram-ui Official SDK; ready-made iOS/Material components that autoswitch on platform; TS-first; largest ecosystem. React 18.3.1, not 19 — @telegram-apps/telegram-ui@2.1.13 still peer-locks to ^18.2.0; forcing 19 via --legacy-peer-deps would break lockfile reproducibility. Bump the day telegram-ui ships a 19-compatible build.
Location in repo services/telegram/ (own service) Separate ship-unit with its own README, Makefile, Dockerfile, deploy target. Matches ADR 0028. Single flat package — no web/ subdir, no npm workspace root (one package doesn't earn workspace ceremony).
Auth Telegram initData → JWT · one middleware in front of /api/v1/* The Mini App is opened from inside Telegram, which injects a signed initData payload identifying the user. The client POSTs it to POST /api/v1/auth/telegram; the server HMAC-verifies it against TELEGRAM_BOT_TOKEN, upserts a row in users keyed on telegram_user_id, and returns a 24 h JWT signed with JWT_SECRET. Every subsequent /api/v1/* call carries Authorization: Bearer <jwt>. Nothing sensitive ships in the bundle; identity is real per user from day one.
Audience Multi-user from launch Real Telegram accounts, real concurrent testers. Every request is attributed to a users row via the JWT's sub claim — schedule, conspectus and error data isolate cleanly per user. Public onboarding and conspectus editing stay out of scope, but multi-user identity is a MVP concern, not a P2 concern.
Documentation model Service catalog page written alongside code, not after A client with 9 screens, cross-screen shared components, deep-links and event tracking is cheap to design on paper and expensive to refactor once wired. Every wave that adds runtime surface also adds the catalog page that describes it — services/portal/internal/services/telegram/ starts as W2 alongside the scaffold and grows one Diátaxis quadrant per subsequent wave. Same discipline the other four services in this repo follow.

Waves — the work plan

Seven waves, one PR per wave. Each wave is independently mergeable — end of a wave means the app compiles, tests are green, and the previous wave's contract is preserved. Every wave that adds runtime surface also lands its catalog page: the entity card and design contract in W2, the deploy-topology page in W7, on-call in the wave that first ships production surface. Docs are wave-scope, not backlog-scope.

  1. W1 · Backend prep — Telegram identity, initData → JWT, JWT middleware. Every screen from W3 onwards assumes an authenticated user row keyed on telegram_user_id, so this wave is the foundation nothing else can move without. Four deliverables ship together.

    Identity migration — Alembic revision creates telegram_users, a per-provider identity table with columns telegram_user_uuid, client_uuid (FK → users.client_uuid, unique, ON DELETE CASCADE), telegram_user_id (BigInteger, unique), telegram_username, telegram_photo_url, locale, created_at, updated_at. The core users table is left untouched so future auth providers (Google, Apple) can land as sibling tables instead of bloating it. The existing (system_user_id, system_uuid) wire contract is preserved; the JWT's sub claim binds to users.client_uuid, which the middleware resolves to a users row directly.

    Auth endpoint — POST /api/v1/auth/telegram. Body: raw initData string. Verifier follows the Telegram spec exactly: (1) derive secret_key = HMAC_SHA256(key="WebAppData", msg=bot_token); (2) build the data_check_string — all fields except hash, sorted by key, joined with \n; (3) compare HMAC_SHA256(key=secret_key, msg=data_check_string) against the payload's hash; (4) reject if auth_date is older than 24 h. On pass, upsert on telegram_user_id and mint a 24 h JWT signed with JWT_SECRET (PyJWT, HS256). Response: { jwt, user }. Endpoint stays anonymous — this is how the client obtains its bearer.

    JWT middleware — app/core/security.py checks Authorization: Bearer <jwt> on every /api/v1/* route except /auth/telegram itself. Invalid, expired or missing token → 401. Valid → request.state.current_user resolves to a users row via the JWT's sub claim. /live and /ready remain open per ADR 0009.

    Config & CORS — TELEGRAM_BOT_TOKEN (HMAC key for initData verification), JWT_SECRET (24-h JWT signing), TELEGRAM_CHAT_ID (bot in W6) added to app/core/config.py and env/*. qa/prod guard: JWT_SECRET must be non-default and ≥ 32 bytes. CORS_ALLOW_ORIGINS extended with the Vite dev origin (http://localhost:5173) and the tunnel origin. OpenAPI baseline regenerated with the bearerAuth security scheme; make openapi-check green.

    Exit criteria: pytest covers (a) valid initData → 200 with JWT, (b) tampered initData → 401, (c) replay of expired initData → 401 (auth_date older than 24 h), (d) valid JWT on a protected route → 200, (e) missing / wrong / expired JWT → 401, (f) config guard rejects default JWT_SECRET in qa/prod.
  2. W2 · Service foundation — scaffold + design contract. Two deliverables that carry equal weight in one wave: (a) the empty Vite shell that later waves fill, and (b) the design contract that tells them what to fill it with. Both land before any screen is coded because reshaping a shell already wired to five screens costs an order of magnitude more than editing a design page.

    Scaffold — services/telegram/ created as a flat single-package layout (no web/ subdir, no workspace root — one package doesn't earn workspace ceremony). Vite 6 + React 18.3 + TypeScript, strict mode. Deps installed in one shot so later waves don't each trigger an install pass: @telegram-apps/sdk-react, @telegram-apps/telegram-ui, @tanstack/react-query, @tanstack/react-router (type-safe routing + built-in View Transitions), zustand (UI state), framer-motion (interactions + shared-element), posthog-js (analytics), i18next + react-i18next (ru + en). Providers land as part of the first-screen wave; scaffold just proves tsc --noEmit, eslint, and vite build all pass.

    Design contract — services/portal/internal/services/telegram/ filled with the full Diátaxis quadrant set for a live-runtime service: entity card, architecture (C4-Component + request path), dependencies (runtime + build inputs), on-call, design (9-screen catalog with states, motion + haptics vocabulary, information architecture), how-to (run-locally, test-on-device, add-a-screen, translate-a-string), reference (event spec, deep-link grammar, env vars), explanation (why React + Vite, observability model, deploy topology). ~15 pages, all portfolio-tone, all machine-checked by make docs-check. Same discipline as api, portal, monitoring, datastore and ui-kit.

    Dev: plain browser tab. mockTelegramEnv() boots the SDK with a synthetic theme + viewport; tools/dev/sign_init_data.py mints a signed initData for a fake Telegram user against the local TELEGRAM_BOT_TOKEN. No tunnel required for UI work; on-device testing (real Telegram + real HTTPS) comes in W6.
  3. W3 · Today screen (end-to-end). First working screen against real API. Providers land here (QueryClient, Router, ThemeProvider mapping themeParams → semantic CSS vars, ViewportProvider writing --tma-safe-top / --tma-safe-bottom / --tma-viewport-h), plus the auth bootstrap (POST WebApp.initData → /auth/telegram, cache JWT in CloudStorage) and the generated API client with Authorization: Bearer <jwt> + Idempotency-Key interceptors. Screen itself: Yesterday-digest strip, streak ProgressRing, Recently-reviewed carousel, 90-day HeatmapCalendar, due list with 4 swipe gestures. Contract note: snooze is not a separate endpoint — every review outcome, snooze included, is a tag (easy · hard · forgot) on POST /api/v1/conspectuses/{id}/actions/review; the exact UI-action → tag mapping is pinned in the design contract, not here.
  4. W4 · Remaining screens — Conspectus, Schedule, Errors, Focus, Search, Profile, Onboarding. Full 8-screen surface — read-only Conspectus with AI micro-actions (Explain differently, Generate quiz), Focus mode (SRS review flow — full-screen card, tap to reveal, Again/Hard/Good/Easy grades, SM-2 next-interval preview), Schedule with 90-day heat-map, Errors with timeline + pattern grouping, Profile with achievements + weekly retention chart, first-run Onboarding, Search / Spotlight (Cmd+K on desktop, pull-down on mobile). All screens hooked to real API via the OpenAPI-generated client (openapi-typescript). Every screen ships loading (content-aware skeleton), empty (with illustration + rotating tip), and error toast states.
  5. W5 · Polish · motion + delight. Framer Motion springs on every mount / dismount. View Transitions API for route changes (Today card → Focus card = shared-element). Haptics vocabulary (7 patterns — selection, impact light/med/heavy, notification success/warning/error). BackButton + MainButton + SettingsButton wired via SDK. Content-aware loading skeletons. Toasts with undo. Milestone celebrations (Lottie for streak 7/30/100). Time-aware copies. Sound cues (opt-in, muted by default). Sentry + PostHog with event spec (review_completed, review_snoozed, error_logged, streak_broken, focus_session_ended, search_opened). Event spec doc in the catalog gets its Verified stamp as each event ships.
  6. W6 · Bot & deep-linking. Python aiogram handlers colocated inside services/api (shares models, config, and Postgres session). aiogram Dispatcher started on FastAPI lifespan startup; dev uses long-poll, prod uses webhook mounted at POST /api/v1/telegram/webhook with X-Telegram-Bot-Api-Secret-Token header check. /start greets and offers the Mini App launch button. Deep-link scheme uses the modern HTTPS format — https://t.me/<bot_username>/<app_short_name>?startapp=conspectus_<uuid>. The startapp payload is exposed to the WebApp via WebApp.initDataUnsafe.start_param and drives router navigation. This wave forces the backend hosting decision — on-device Telegram testing needs a public HTTPS endpoint for the API. Candidate ADR 0039 (Cloudflare Tunnel from Mac / Hetzner CX22 VPS / Fly.io) drafted & accepted here.
  7. W7 · Deploy, perf budget, DoD. Mini App shipped to Cloudflare Pages (custom domain, HTTPS). Bot webhook served from the API container per ADR 0039. Backlog cockpit reflects «done». make verify extended: builds the Mini App, typechecks, runs Vitest + Playwright, and enforces the performance budget via Lighthouse CI — LCP ≤ 2.0s, TBT ≤ 200ms, bundle ≤ 200 KB gz, a11y ≥ 95. Deploy-topology page in the catalog is stamped Verified. Epic closed.

Non-functional guarantees

Every point below is either a hard gate (the build fails if it breaks) or a design concern that shapes early decisions, not something to bolt on later.

Auth & storage

  • Trust root: Telegram's HMAC-signed initData. Server verifies against TELEGRAM_BOT_TOKEN; nothing user-supplied is trusted otherwise.
  • JWT (24 h, HS256, signed with JWT_SECRET) is minted at POST /auth/telegram; every /api/v1/* call carries Authorization: Bearer <jwt>.
  • Token stored in WebApp.CloudStorage (cross-device sync per user) with a memory-first hot cache; nothing in localStorage. Silent refresh 5 min before expiry.
  • Identity migration lands in W1: users.telegram_user_id (unique) + telegram_username + telegram_photo_url + locale + tz. Every request resolves to a real users row via the JWT's sub claim.
  • BiometricManager gates destructive actions (delete conspectus, reset schedule) and streak-break confirms.

Performance budget

  • LCP ≤ 2.0s on cold LTE (Moto G Power throttle).
  • JS bundle ≤ 200 KB gz initial + code-split routes.
  • No web fonts loaded from network — system stack only.
  • Lighthouse CI in make verify; PR fails on regression.

Motion & a11y

  • Every animation route through motion tokens; prefers-reduced-motion collapses to instant.
  • Tap targets ≥ 44 × 44 CSS px enforced by the cell primitive.
  • Keyboard navigable end-to-end on Telegram Desktop (J/K/E/? shortcuts + Cmd+K).
  • VoiceOver / TalkBack pass on every screen (rotor + live-region).

Observability

  • PostHog event spec: review_completed, review_snoozed, focus_session_ended, error_logged, streak_broken, search_opened, command_ran. Full contract on the event-spec reference page.
  • Sentry client (opt-in via VITE_SENTRY_DSN) with Telegram context tags.
  • Bot-side: structured logs with correlation_id, joined to API logs.

Dev environment

  • Two dev loops. Fast loop: plain browser tab with mockTelegramEnv() generating deterministic initData payloads signed by a dev-only HMAC helper (tools/dev/sign_init_data.py) that shares TELEGRAM_BOT_TOKEN with the local API. No tunnel, sub-second reload. On-device loop: Cloudflare Tunnel exposes the API over HTTPS so real Telegram clients drive real initData.
  • Storybook for every .tma-* primitive; visual QA lives here, not on the phone.
  • Playwright uses mockTelegramEnv() + the dev signer in CI — no external HTTPS required.

i18n & localisation

  • All copy through i18next; ru + en at MVP, chosen from Telegram's language_code.
  • Numbers use Intl.NumberFormat; dates use Intl.RelativeTimeFormat.
  • Timezone from Intl.DateTimeFormat() → .resolvedOptions().timeZone, persisted server-side for reminder scheduling.

Definition of Done

  • Opening the bot in Telegram → /start → tap «Open study_app» launches the Mini App, and a fresh Telegram account is upserted into users on first open.
  • Every /api/v1/* route (except /live, /ready, /auth/telegram) requires a valid Authorization: Bearer <jwt> — enforced by the JWT middleware, covered by tests.
  • All nine MVP screens render on iOS + Android Telegram clients — dark and light themes — for every tester in the cohort, each seeing only their own data.
  • Adding an error, marking a review, and reading a conspectus persist against the real Postgres, are scoped to the caller's users.id, and survive a page reload.
  • Schedule strip reflects GET /api/v1/schedule/summary without any client-side aggregation.
  • make verify passes (lint · types · openapi · contract · pytest · docs-check · web build).
  • Service catalog at services/portal/internal/services/telegram/ ships the full Diátaxis quadrant set — entity card, architecture, dependencies, on-call, design, how-to, reference, explanation.
  • ADR 0039 (backend hosting) is Accepted; this epic is Delivered in the backlog cockpit.

Risks & open questions

HTTPS in dev

Telegram requires HTTPS for the WebApp URL — but only for on-device sessions. The fast dev loop is a plain browser tab against a locally-signed initData; no tunnel required. When on-device testing starts (W6): Cloudflare Tunnel; fallback ngrok or hosts-file + mkcert.

Bot polling vs webhook in dev

Webhook needs a public URL. In dev, aiogram long-polling is fine; W6 flips to webhook only for prod. Config lives in env/*.

Deep-linking scope

Default launch opens Today. Per-conspectus deep-links (startapp=conspectus_<uuid>) land in W6 together with the webhook — nothing deep-link-shaped ships before that. Grammar is pinned in the deep-link-grammar reference page.

Dev loop needs signed initData

Server rejects any request whose initData doesn't HMAC-verify against TELEGRAM_BOT_TOKEN. The fast dev loop (plain browser tab) uses tools/dev/sign_init_data.py to mint deterministic payloads for the local API — same bot token, same verifier, same result. Real-device testing goes through Cloudflare Tunnel with the real Telegram client generating initData. Never bake the bot token into the bundle; the signer stays server-side.

Backend hosting

API has no cloud production host today (see deployment topology). Three viable paths on the table: Cloudflare Tunnel from Mac ($0 · needs Mac online) / Hetzner CX22 VPS (~€4/mo · 24/7) / Fly.io ($0-5/mo · cloud-native). Decision forced in W6 as ADR 0039 when on-device Telegram testing needs a public HTTPS endpoint.

Reminders scheduler

Deferred to P2 — MVP relies on testers opening the bot on their own. Once scheduled, decide: APScheduler in-process vs external cron. Note in the ADR log once chosen.
