API-first — hand-authored OpenAPI fragments as the single source of truth

Every new HTTP operation in this project ships its contract before its handler. One YAML fragment per operation, hand-authored and reviewed; the running FastAPI's /openapi.json is a claim, not the contract; a Python parity gate (make openapi-check) is the release blocker. This ADR sets the policy; the recipe lives at Handbook · SA · Authoring · Apply API-first.

Context

The Study App API grows by one HTTP operation at a time. Today the working convention is code-first: a FastAPI handler is written, its Pydantic schemas drive a generated /openapi.json, and that generated spec is treated as the contract. For one operation written and reviewed by the same person on the same day, this works. As the API surface grows past a handful of operations and the contributor pool widens, two failure modes appear.

  • Review happens too late. The spec exists only after the handler does, so the contract is reviewed against an already-written implementation. The first opportunity to spot a wrong status code, a missing required field, a leaky internal name, or a misaligned error envelope is in the same PR that ships the wrong behaviour — when the cost of changing the contract is highest.
  • Drift is invisible. When the spec is regenerated from the code, the spec changes whenever the code changes, by definition. There is nothing stable to diff against, so the question «did we break compatibility?» has no mechanical answer.

The industry's name for the opposite practice is API-first: the contract is authored first, in a machine-readable form, reviewed in its own PR, and only then implemented. Stripe, Twilio, and GitHub run their public APIs this way; client SDKs, mock servers, reference docs, and contract tests all derive from the same canonical artifact. Inside the company the spec — not the handler — is the unit of review.

This ADR commits the project to API-first as policy and names the artifacts, paths, and tooling that make it enforceable rather than aspirational. Concrete authoring steps live in the how-to; this ADR is the policy.

Decision

Adopt API-first for every new HTTP operation added to /api/v*/. The policy has six clauses.

  1. D1 — Hand-authored OpenAPI fragments are canonical. One YAML file per operation lives at services/portal/internal/services/api/openapi/test/fragments/{resource}/{operationId}.yaml. Shared schemas, headers, and error responses live under fragments/_shared/ and are referenced via relative $ref. The fragment is what reviewers and contract tests read.
  2. D2 — There is no merged build output. Each fragment stands on its own as a valid OpenAPI 3.1 document, validated in isolation with cross-file $ref resolution. We do not maintain a single merged openapi.yaml alongside the fragments; the merged view, if ever needed downstream, is computed on the fly.
  3. D3 — FastAPI's runtime /openapi.json is a claim, not the truth. It documents what the running code does. Disagreement between the canonical fragments and the runtime claim is drift, and drift fails CI.
  4. D4 — The contract ships before the handler. Every new operation lands as two PRs in order: a spec PR that authors the fragment plus the portal Operation spec page; then an implementation PR whose only review question is «does this match the ratified spec». A handler PR cannot change the contract — only an amendment to the fragment can.
  5. D5 — The toolchain is single-language Python. Fragment lint via openapi_spec_validator (wrapped by tools/governance/validate_openapi.py and surfaced as make doc-api-swagger-check[-all]); canon ↔ runtime parity via openapi_governance.py (surfaced as make openapi-check); property-based contract fuzz via schemathesis against the running FastAPI. No Node.js side-stack — Spectral, Prism, and openapi-diff were considered and rejected as a second toolchain to install. Tool selection is part of the decision; introducing a non-Python tool requires a successor ADR.
  6. D6 — Code-first resources are grandfathered, then back-filled within 60 days. Any operation that exists in the codebase at the date of this ADR but does not yet have a hand-authored fragment is tagged in its eventual fragment with x-build-history: code-first. The back-fill window is two months from the day the toolchain (D5) lands; past that point, the asymmetry is unacceptable and we reopen the decision.

Scope

  • In scope: every new HTTP operation added to /api/v*/ after the date this ADR is ratified. The fragment, the portal Operation spec, the contract-test plan, and the tooling that gates them.
  • In scope: the Python toolchain in this repository's .venv — openapi_spec_validator for fragment lint, schemathesis for property-based contract fuzz, and tools/api/openapi_governance.py for the canon ↔ runtime parity gate. Driven through make doc-api-swagger-check[-all], make openapi-check, and make contract-test.
  • Out of scope: public-API documentation generation. The split between external and internal documentation surfaces is governed by ADR-0025; this ADR only changes what is canonical on the contract side.
  • Out of scope: client SDK generation. The project does not ship client SDKs today. When it does, the SDK build pipeline will read from the merged openapi.yaml output of D2.
  • Out of scope: transport protocols other than HTTP. If the project ever exposes gRPC or GraphQL, a successor ADR adopts the equivalent contract-first artifact and lint chain for that surface.

Alternatives considered MANDATORY · MIN 2

Keep code-first — generate OpenAPI from FastAPI handlers

rejected

Rejected because: the contract becomes a downstream artifact of the implementation. Review happens after the handler exists, so the cheapest moment to fix a contract mistake — before the code is written — is always skipped by construction. Drift between intended and actual contract is invisible because there is no second source to diff against.

Pros considered: zero new tooling; one source file to maintain per operation; the FastAPI/Pydantic chain already produces a usable spec.

Cons that decided it: contract review happens too late; drift cannot be detected mechanically; the SA review has no leverage independent of the handler author.

One hand-authored monolithic openapi.yaml

rejected

Rejected because: PR review on a multi-thousand-line YAML is illegible. Every operation review touches the same file, merge conflicts are frequent, and the diff context spans hundreds of unrelated lines. The fragment-per-operation pattern keeps review at the grain of one operation, which is the right unit of work for both authors and reviewers.

Pros considered: simpler tooling — no merge step; the spec is always internally consistent because it lives in one place.

Cons that decided it: PR grain breaks down; concurrent work serialises on a single file; review noise drowns out the signal.

Treat the portal HTML Operation spec as the contract — no YAML at all

rejected

Rejected because: HTML prose is not machine-checkable. There is no parity diff for prose, no contract-test fixture in HTML, and no SDK generator that reads from it. The portal page is the right form for humans, but the contract still needs a machine-readable canon a tool chain can act on.

Pros considered: no new file format to learn; the portal spec page is already authored anyway; one less artifact per operation.

Cons that decided it: downstream consumers (mock servers, contract tests, future SDK codegen) cannot consume HTML; review is by reading, not by tooling; parity with the running code cannot be gated mechanically.

Consequences

What we get

  • Contract review precedes implementation. The spec PR is reviewed against the requirements; the handler PR is reviewed against the spec. The «which one is right» question disappears.
  • Drift is mechanical. make openapi-check fails the build the moment the runtime spec disagrees with the canonical fragments. Disagreement is found when it costs one commit to fix, not one quarter.
  • Single-language toolchain. Fragment lint, parity gate, and contract fuzz all run inside the same Python .venv as the API. Contributors install one set of dependencies; CI runs one matrix; the contract-first practice doesn't drag a second language stack into the repo.
  • Reviewable grain. One fragment per operation means one PR per operation contract — small, focused, mergeable in minutes.

What it costs

  • ~2 hours of overhead per new operation to author the fragment and align it with the portal spec page. Acceptable at the project's current scale; reconsidered if the API grows past tens of operations.
  • One Python tool to maintain. tools/governance/validate_openapi.py wraps openapi_spec_validator with $ref resolution; it must keep up with the validator's API and any project-specific rules (idempotency tag, error-envelope shape) we add over time.
  • One asymmetric edge during back-fill. Pre-existing code-first operations are grandfathered with an x-build-history: code-first tag for the first 60 days. Readers must understand the tag means «the canon was reconstructed, not authored» and is not the pattern to copy.
  • PR ordering discipline. Two PRs per operation (spec then code). Mitigated by the fact that the spec PR is small enough to review in minutes and authoring it costs roughly what aligning the runtime spec to expectations would have cost anyway.

Compatibility and migration

  • Backwards-compatible. Additive policy. No existing URL changes, no existing handler is touched, no existing portal page is rewritten as a precondition.
  • Migration plan.
    1. Bootstrap services/portal/internal/services/api/openapi/test/fragments/ with the _shared/ sub-tree (headers, schemas, responses) and a README that names the conventions.
    2. Land the toolchain — tools/governance/validate_openapi.py + make doc-api-swagger-check[-all] for fragment lint, make openapi-check for the parity gate — wired into pre-commit and CI.
    3. Back-fill fragments for every code-first operation that exists at the date of this ADR. Each carries x-build-history: code-first in its fragment. Target: within 60 days of the toolchain landing.
    4. From that point on, every new operation lands as spec PR + implementation PR in that order.
  • Rollback strategy. Removing the fragments tree, tools/governance/validate_openapi.py, and the CI gate reverts the project to code-first. No data migration, no URL changes — a pure process rollback. Triggers for rollback: toolchain maintenance burden exceeds 2× the estimate; a better contract format emerges (gRPC, GraphQL) for which a successor ADR is opened.

Validation

The decision is validated when the first new operation ships under this policy — fragment authored, make doc-api-swagger-check clean, portal spec page §1–§19 complete, contract tests pass, make openapi-check clean. Concrete signals to watch over the first quarter after ratification:

  • Spec ratification precedes the handler PR. Every new operation lands as two PRs in order. If a handler PR ever introduces a route without a ratified fragment, the gate failed.
  • The parity gate trips at least once during normal authoring. If make openapi-check never trips in the first month, either nothing real shipped or the gate is misconfigured.
  • Review grain stays small. A typical fragment PR touches one YAML file plus its spec page. If a fragment PR routinely spans more than two operations, the «one operation per fragment» rule is being bent — investigate.
  • Code-first back-fill completes within 60 days. Every operation that existed at the date of this ADR has a back-filled fragment within two months. Past that date, the asymmetry is permanent and the decision is re-opened.
