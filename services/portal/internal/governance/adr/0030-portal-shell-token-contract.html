Context

The repository ships two HTML portals from a single codebase: the public documentation portal (body class="public-layout", audience-facing tutorials and references) and the internal portal (body class="internal-layout", ADRs / RFCs / runbooks / catalog). Per RFC 0004 the two portals share an information architecture spine but differ in identity and density: the public layout is wider and lighter; the internal layout is denser, sticky-sidebar-driven, with audit chrome.

Until 2026-05-09 each portal carried its own copy of the same shell geometry rules — the grid template, sidebar stickiness and scrollbar contract, mobile flatten breakpoint, container max-width and gutters. Two near-identical blocks of CSS lived in public-layout.css and internal-layout.css; whenever a viewport tier changed (e.g. the 2026-05-01 mobile-contract pass at #mobile-contract), both files had to be edited in lockstep, and any drift between them silently broke the visual alignment between the two portals.

We need a single normative source for shell geometry that (a) keeps the two portals in lockstep on the breakpoint canon, (b) preserves each portal's distinct identity (sidebar width, density, container width, padding), and (c) makes adding a third portal (e.g. a future partners branch) a question of declaring a token override rather than copying ~120 lines of geometry.

Decision

1. Single source for shell geometry

All shell geometry — the grid template, the sticky-sidebar contract, the mobile flatten rule, the collapsed-sidebar rule, and the per-portal container box — lives in one file: services/frontend/portal/assets/docs-shell.css. Both portals load it via the @import chain in docs.css. Per-portal differences are encoded as CSS custom properties (tokens) on the body class, never as duplicated geometry rules.

Identity-only styles (sidebar nav internals, drawer chrome, hero / spotlight / backlog widgets, page-title row, color tokens) remain in public-layout.css and internal-layout.css. The split is structural: shell = grid + container box; identity = everything that fills the container.

2. The fourteen-token contract

The shell exposes fourteen tokens. Both portals must define values for all of them at the base scope; per-portal and per-breakpoint overrides update specific tokens without restating geometry.

Token Public default Internal override (≥1025) Purpose
--shell-max-width min(1520px, 100vw) min(1680px, calc(100vw - 24px)) Outer max-width of the shell grid (sidebar + main).
--shell-pad 0 0 72px 24px 12px Padding on the shell at desktop tiers.
--shell-pad-mobile safe-area-inset gutters + 56 px bottom 12 px top + safe-area-inset gutters Padding on the shell at ≤1024 (mobile + tablet flatten).
--sidebar-width 264px 300px Grid track 1 width at ≥1025.
--sidebar-collapsed-width 56px 56px Grid track 1 width when .is-sidebar-collapsed.
--sidebar-bg var(--bg) transparent Sidebar background; transparent for the internal portal so the page chrome shows through.
--sidebar-pad 0 4px 16px 20px 0 Sidebar inner padding (≥1025).
--sidebar-mobile-pad 0 0 0 16px Sidebar inner padding at ≤1024.
--sidebar-mobile-display none block Public portal hides the sidebar on mobile (drawer-only); internal portal flattens it inline.
--sidebar-z 10 2 z-index of the sticky sidebar; public sits above topbar, internal stays below the page-title row.

--container-pad 12px 24px 0 32px at ≥1025 12px 24px 0 32px at ≥1025 Container padding; identical between portals so prose blocks line up.

3. Authoring rules

  • Geometry rules live once. The selectors .public-layout__shell, .internal-layout__shell, .public-layout__sidebar, .internal-layout__sidebar, .public-layout__main, .internal-layout__main, and the .container inside each main, are declared only in docs-shell.css. Adding the same selector to public-layout.css or internal-layout.css is a review-blocker.
  • Breakpoint canon. Geometry rules use the mobile-contract tiers: ≥1025 (desktop), 761–1024 (tablet flatten), ≤760 (phone). No new breakpoints inside docs-shell.css without updating the mobile contract first.
  • Per-portal differences are tokens. When the portals must differ, override the token on the body selector — never branch geometry by selector.
  • Wide widgets opt out at the widget, not the container. When a single widget needs the full main column (HTTP tablet, layer tables, hero blocks), it sets its own width on the widget selector. The container stays the prose column.
  • Identity-only files don't repeat geometry. When editing public-layout.css or internal-layout.css, the file should grow content rules (typography, color, decorative chrome) only — never grid, never sticky positioning.

4. Adding a new portal

To add a third portal (e.g. body class="partners-layout"), the workflow is:

  1. Add the new body selector to the shared selector list in docs-shell.css for each geometry rule.
  2. Add a token block body.partners-layout { ... } overriding only the tokens that differ from the public default.
  3. Create partners-layout.css for identity-only styles and add it to the @import chain in docs.css.
  4. Update this ADR's token table with any new portal-specific values.

Scope

  • In scope: shell grid geometry, sidebar stickiness / scrollbar / collapse, mobile flatten contract, container box (max-width, margin, padding), the fourteen-token contract, and the authoring rules above.
  • Out of scope: typography (font ladder), color tokens, sidebar nav contents (internal-sidebar.js), drawer chrome, hero / spotlight / backlog widgets, syntax highlighter, table-of-contents FAB. These remain in identity files.

Alternatives considered

  1. Keep duplicated geometry in each layout file. Pros: each file is self-contained; reading one portal's CSS gives the full picture. Cons: every breakpoint pass must edit both files; drift is silent; the cost of adding a third portal scales linearly with shell complexity.
  2. Branch geometry by selector instead of tokens. Pros: classic CSS pattern, no custom properties. Cons: the shared rule list explodes (.public-layout__shell { … } .internal-layout__shell { … } per breakpoint × per property); reviewers must mentally diff parallel rule blocks instead of comparing one token line.
  3. Move the shell into a CSS-in-JS framework. Pros: type-safe tokens, theme objects. Cons: the portals are static HTML loaded directly; introducing a JS-only build step for shell CSS would couple every page to a bundler and contradict ADR 0001's "docs render statically with file:// open" baseline.
  4. Use Sass mixins or a build-time preprocessor. Pros: terser source. Cons: adds a build step, splits the source-of-truth between .scss and .css, and obscures cascade behavior (browsers see custom properties live; Sass values are baked at build time).

Consequences

Positive

  • One file owns geometry. Breakpoint passes touch one file; portal drift becomes mechanical to spot in review.
  • Adding a portal is declarative. A new audience needs a token block, not a copy of the shell.
  • Cascade is observable. Custom properties show in DevTools per-element; debugging a layout regression means inspecting one variable list, not grepping two CSS files.
  • Width unification falls out for free. The 2026-05-09 prose-column unification (--container-max-width: 960px on both portals) is a one-line override; the same change in the duplicated regime would have required parallel edits.

Trade-offs

  • The token contract is a public API across portal files. Renaming or removing a token without updating every portal's overrides breaks the shell silently. Mitigated by the up-front list in this ADR's token table and by the rule that geometry selectors live in only one file.
  • Reading any one portal's CSS no longer tells the full geometric story; reviewers must read docs-shell.css + the portal's identity file. Mitigated by the file header in docs-shell.css that names every token.

Compatibility and migration

  • Backward-compatibility impact: none for end-users. The 2026-05-09 extraction is a no-visual-drift refactor; tokens were initialized to the values that the duplicated geometry was already producing.
  • Migration plan:
    1. Land docs-shell.css with both portals' geometry expressed via tokens (done 2026-05-09, commit 450f456).
    2. Strip the duplicated geometry from public-layout.css and internal-layout.css, leaving identity rules (done 2026-05-09).
    3. Unify the prose-column width across portals at ≥1025 via the --container-max-width / --container-margin-inline / --container-pad tokens (done 2026-05-09, commit 0ffe127).
    4. Future portals declare a body class and a token block; no further migration work.
  • Rollback: revert the offending PR. Both portals fall back to their previous duplicated geometry without runtime impact.
