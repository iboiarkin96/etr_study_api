ADR 0038 · TMA screen DNA — Amie signature

A single visual DNA — Amie · warm signature — for every Telegram Mini App screen. This ADR names the choice, records the rejected alternatives (Bear · editorial, Raycast · command) as evidence, and carries one subsection per screen showing how the DNA applies there.

Context

The Mini App is a family of ~12 phone-viewport screens (Today, Focus, Conspectus detail, Schedule, Errors, Onboarding, Profile, Search, plus supporting shells). Without a single visual DNA, each screen drifts toward a different tier-1 aesthetic — one designer chooses Amie warmth, another chooses Bear-editorial, a third chooses Raycast-density — and the product reads as three apps stitched together. Every screen would also re-open the same debate from scratch; deciding once removes that cost.

Three tier-1 DNA takes were on the table when the first three screens were sketched:

  • Amie · signature warmth. One breathing orb, one CTA, editorial peek. Emotional carrier first, information second.
  • Bear · editorial typography. Display digit as hero, story-framed copy, zero chrome. Reader-first.
  • Raycast · command density. Keyboard-first, mono numerics, every row an action. Power-user first.

All three carry the same data and API contract; the choice is purely how the app feels. Because it feels the same on every screen, the choice is made once and applied everywhere — a per-screen re-vote is exactly the drift this ADR prevents.

Decision

Adopt Amie · signature warmth as the single visual DNA for every TMA screen. Four clauses.

  1. D1 — One breathing element per screen. Each screen has exactly one hero element that lives (the orb on Today, the timer ring on Focus, the calendar arc on Schedule, …). This element carries emotion; everything else is a peek.
  2. D2 — One primary CTA per screen. Every screen has one obvious next action. Secondary actions live as peeks or drill-downs, not as competing buttons.
  3. D3 — Editorial peek, not dashboard. Lists are quiet, lined, and one-thumb-scrollable. No charts, no widgets, no dashboard tiles. The app talks like a note, not a control panel.
  4. D4 — Warm ember accent, sage rest. A single accent (ember-500 warm orange) carries streak / progress / attention. A single rest state (sage-500 green) signals «you can breathe». No third accent — additional tones are muted greys. Exception: when the primary CTA is a composite whose semantic difference IS the affordance (e.g. Focus's Again · Hard · Good · Easy grade strip — four tones side-by-side is the whole point of the interaction), the composite may span up to four tones. The exception must be documented per-screen in Per-screen application and does NOT extend to non-composite surfaces on the same screen — everything else still lives on the single-accent budget.

D1–D4 apply to every screen. Individual screens document how their hero, CTA and peek satisfy the four clauses in the Per-screen application section below; documented deviations (with rationale + containment) are permitted for D1 and D4 but must appear there — silent drift is not.

Scope

  • In scope: the visual DNA and its four clauses; the accent + rest palette; the «one breathing element + one CTA + editorial peek» composition rule; the growing per-screen application record.
  • Out of scope: API contract (unchanged across all DNA takes — an implementation detail, not a design decision); accessibility contract (owned by reference/a11y); typography / spacing tokens (owned by design IA).
  • Out of scope: web / desktop port. The DNA is written for a phone-viewport-first Mini App. A desktop port belongs to its own ADR, which decides how (or whether) the DNA scales.

The DNA was decided on Screen 01 · Today as the test surface and re-checked against Screen 02 · Conspectus detail, Screen 03 · Focus, Screen 04 · Schedule, Screen 05 · Errors, and Screen 06 · Search — each screen's rejected sketches are archived below, numbered to match their picked counterpart in Per-screen application. Mocks are the ADR's rationale — not living component references. Nothing on this page updates when production changes.

Why considered: reader-first products (Bear, Ulysses, Craft) show that typography alone can carry identity. Zero chrome + display digit + reflective copy would give the app a calm, journal-like feel that fits a study product.

Why rejected: two independent reasons.

  • Habit loop needs an emotional carrier, not a paragraph. Users open the app in 2 seconds and glance. A display digit + prose demands reading; a breathing orb reads at a glance. For a daily-open habit product, the first-two-seconds tax is the whole game.
  • Editorial voice hides missed days. «Twelfth day» reads the same for streak-12 as for streak-1 unless the reader parses the number. The signature orb makes streak state pre-attentive: full ring = alive, dashed ring = broken, before any digit is read.

Screen 03 · Focus — Duolingo · playful

rejected
9:41•••

SRP — what «reason to change» really means?

Why considered: Duolingo has proven that gamified two-outcome (got-it / missed) grading with a ring-progress orb crushes retention. Two buttons remove decision friction; the ring turns each card into a visible micro-goal; celebration at the end of the ring makes «complete this session» its own reward.

Why rejected: two independent reasons.

  • SRS needs 4-way resolution, not 2. The whole point of SM-2 is that «I remembered» and «I remembered easily» produce different next-intervals. Collapsing to got-it / missed throws away the calibration signal the algorithm depends on. Retention wins from gamification would be swallowed by worse scheduling.
  • Ring orb duplicates Today's breathing element. D1 says one breathing element per screen — Today already owns the orb. Adding a ring-orb on Focus splits the signature; users can no longer identify «the app with the breathing orb» to a friend.

Screen 03 · Focus — Bear · minimal 2-outcome

rejected
9:41•••
03 / 05

SRP — what «reason to change» really means?

Why considered: radical minimalism — prompt at 19 px display weight, two flat buttons, zero chrome. The prompt is the screen. A learning surface that reads like a paragraph you have to answer, not a task queue you have to clear. Fits Bear-editorial DNA at its purest.

Why rejected: two independent reasons.

  • Same 4-way resolution argument as Variant A. Two buttons throw away the calibration signal. Bear's editorial voice cannot rescue an SRS algorithm that has been robbed of its inputs.
  • Flat surface means no «moment». The tap-to-reveal transition is the whole emotional payoff — a flashcard should feel like a physical card flipping. Bear's flat plane offers no place for the 3D flip, no ember bloom, no anticipation. The screen users spend the most seconds in ends up feeling like a to-do checkbox instead of a study ritual.

Why considered: a dedicated /search route composes naturally with the existing routes (Today, Focus, Schedule, Errors), gets its own history entry, and can grow (recent history, filters, saved queries) without stealing space from the surface the user came from. Follows Notion's spatial model — searching feels like «going somewhere», with room to breathe.

Why rejected: two independent reasons.

An overlay preserves the surface and returns focus to the trigger.
  • Encourages scope creep. A route says «this deserves screen area» — the moment you have one, filters, recent-history, saved queries, keyboard-shortcut listing all start feeling in-scope. The overlay's constrained real estate is a scope guard: one input, one target, one keystroke sequence.

Screen 06 · Search — Google · bottom-sheet

rejected
9:41•••

⌕ Search…

Why considered: bottom-sheets are the phone-native modal pattern — Google Search on Android, Apple's «Look Up», Telegram's own reply sheet. Slides up from the thumb, dismisses by swipe-down (matches user muscle memory), leaves the surface visible so the user never loses context.

Why rejected: two independent reasons.

  • Swipe-down fights Telegram Mini App gestures. Telegram's own «pull-to-close-mini-app» gesture already owns swipe-down at the surface level. Adding a sheet whose dismiss gesture IS swipe-down means one of them wins per pixel of drag — the user learns to distrust both. The overlay's Esc / backdrop / Cmd+K close model has zero collision with the host.
  • Half a screen is worse than a full screen or a modal box. The sheet takes ~60% of viewport — enough to be intrusive, not enough to fit a long result list. The centred command box (~560×460 max) is the goldilocks: constrained on desktop (where users have keyboards and screen real estate) and covers the phone viewport (where the box's max-height reaches the safe area).

Consequences

Positive

wins
  • The app feels like one product, not twelve. Every screen inherits the same warmth, the same one-CTA rhythm, the same peek voice.
  • New screens don't re-open the DNA debate. A designer opens Focus / Search / Profile knowing D1–D4 answer «which hero», «which action», «which list style» before any Figma is opened.
  • Signature transfers. The orb becomes the thing users describe to a friend («the one with the breathing circle»). Because DNA is uniform, that recognition transfers to Focus, to Schedule, to every future screen.
  • One accent, one rest state. Ember-500 + sage-500 is the entire semantic palette. Muted greys carry structural weight. Cognitive load stays low.

Trade-offs

costs
  • Power learners don't get keyboard superpowers. Raycast-style density on a laptop-companion would delight a slice of users. We accept the loss because the primary surface is the phone and phone-DAU is what the product is optimised for. A future desktop port can revisit.
  • Text-heavy edge cases feel light. A long conspectus body inside an editorial-Bear frame would breathe more than the same body inside the peek-density frame. Mitigated by Conspectus detail's own «reader mode» surface, which is a per-screen exception documented in the application record — not a DNA break.
  • DNA drift is easy to introduce accidentally. A single PR that adds a second accent tone or a second primary CTA silently degrades the DNA. Guardrail: every screen carries a subsection in Per-screen application declaring how D1–D4 apply; an absent subsection signals the DNA has not been applied.

Revisit — would we do this again?

Yes. Locking the DNA before the first screen is scaffolded — rather than after — keeps every screen honest against D1–D4 from the first commit. Any comparable product benefits from treating DNA as a pre-scaffold decision.

Compatibility and migration

  • Backward compatibility. None to preserve. No prior visual DNA is documented for the Telegram Mini App; every screen shipped before this ADR is treated as an application of Amie by default and is verified against D1–D4 during its next redesign.
  • Migration. Existing screens that do not satisfy one of the four clauses are documented in Per-screen application with an explicit Deviations row that names the clause and the reason. A deviation without a stated reason is a review-time block.
  • Interaction with other ADRs. This ADR does not supersede or amend any other ADR. It sits alongside RFC 0007 — RFC 0007 decides the runtime (Telegram Mini App, phone-viewport-first), this ADR decides the visual language that runtime carries.
  • Downstream consumers. The kit-showcase pages under services/portal/ui-kit/pages/telegram-mini-app/ and the reference pages under services/portal/internal/services/telegram/reference/ point at this ADR for the DNA rationale. A renamed section anchor invalidates those links — deep-link anchors listed under Implementation in this repository are load-bearing and cannot be renamed without updating consumers.

Per-screen application

One subsection per screen. Each subsection records how D1 (breathing element), D2 (primary CTA), D3 (peek shape) and D4 (accent) apply on that surface, together with a frozen visual reference and pointers to the living reference doc and Storybook story. Absence of a subsection means the DNA has not been applied to that screen.

Documented deviations
  • D4 · four tones, not one. Grade buttons are the only place in the app where four semantic tones sit next to each other. Justified because they are ONE composite CTA — the tone difference IS the affordance (green = extend interval, yellow = struggled, red = failed). Reducing to one tone would lose the read-at-a-glance decision. Contained: nothing else on this screen uses tone-inks.

Justified as tone continuity: a row is a quote of the button that produced it, and it cools to success once the miss note is saved. Alternative considered and replaced: the T-20a quiet «Log a miss →» link deep-linking to /errors?prefill_from=session — rejected because it carried only the last miss (N misses, one loggable) and threw the user off the screen; the deep-link survives as a URL contract with no in-app producer.

Timeline rows: signal dot · one-line message · quiet mono time chip. Newest first. No delete (append-only, ADR contract). No filter chips, no tag facets — the shape is the signal, not the filter.
D4 · accent + rest
Ember on the orb (warm state) + the CTA fill. Warning ink on «yesterday» rows, sage on «≥2d» rows — the tone dot goes cooler as the entry ages, so a wall of ember reads as «this week was hard» and a wall of sage reads as «pattern's cooling off». Three tones, all consumed as data-tone attributes on .tma-miss-row; no third accent introduced.
Documented deviations
None — the Amie signature variant ships intact. All four clauses hold at their strict form.
Idempotency contract
POST /api/v1/errors requires an Idempotency-Key header, auto-injected by the client middleware (T-12). Double-submit with the same key returns the same row (server replay per ADR 0006). Mutation is optimistic: temp row prepended immediately, replaced with server row on success or rolled back on error.
Entrance choreography
Same Assemble pattern as Today / Schedule: MissOrb is the hero (fade + soft scale, in place); eyebrow + list header is slot 1; MissLog is slot 2; the CTA is slot 3 — each stacks with the standard 70 ms stagger.

D1 · breathing element
None on this screen — Search is transient (opened, used, closed within a few seconds), not a screen the user dwells on. The DNA hero belongs on the surface the user came from; the overlay is deliberately quiet visual weight. This is a documented deviation: D1 is absent because the surface's role is a shortcut, not a destination.
D2 · primary CTA
None on this screen — the query itself is the action, the highlighted row (Enter / tap) is its target. No button; a button would suggest confirmation on a surface whose whole value is «no confirmation». This is a documented deviation: D2 is absent because the primary interaction IS the input.
D3 · editorial peek
Result rows are the peek: slot-tinted glyph · title with <mark>-wrapped match · muted slot label. One-line rows, ranked earlier-match-first then shorter-title-first. Highlighting is the peek's editorial voice — «here is what matched», not «here is a facet you can filter by».
D4 · accent + rest
The match highlight is the only accent — <mark> renders on the accent ink, no background. Slot glyphs use the four semantic tones (accent/info/success/warn) that already exist for slot labels; no new tokens. Cursor row background is --tma-surface-plate — a rest surface, not a highlight surface.
Documented deviations

Keyboard contract
Cmd+K / Ctrl+K toggles open/close globally (owned by SearchProvider). ↑/↓, Ctrl-J/N (down), Ctrl-P (up) navigate results. Enter opens the highlighted row. Esc closes and restores focus to the trigger element (WAI-ARIA modal contract). Ctrl+K is deliberately NOT bound inside the overlay to avoid fighting the global toggle.
a11y contract
role=dialog + aria-modal=true, aria-activedescendant on the input pointing to the highlighted option, role=listbox / option with aria-selected, and a role=status live region announcing «N matches / no matches / loading» without stealing focus. Backdrop close only fires when mousedown AND mouseup both land on the backdrop — a text-selection drag ending on the backdrop no longer closes the overlay.

Cross-screen gestures

Gestures that apply across multiple screens are pinned here so the DNA stays coherent when the same intent is expressed twice (e.g. reviewing a card via swipe on Today AND grade on Focus). Screen-specific interactions live in each per-screen application card above.

Grade Again Focus forgot Reset schedule from scratch.

Validation
