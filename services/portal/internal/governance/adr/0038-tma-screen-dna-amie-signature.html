ADR 0038 · TMA screen DNA — Amie signature

A single visual DNA — Amie · warm signature — for every Telegram Mini App screen. This ADR names the choice, records the rejected alternatives (Bear · editorial, Raycast · command) as evidence, and carries one subsection per screen showing how the DNA applies there.

Metadata

Status
Accepted · 2026-07-20 · Ivan Boyarkin
Scope
Cross-cutting — applies to every screen in the Telegram Mini App.
Related
RFC 0007 (TMA-first platform port), Telegram service hub.

Context

The Mini App is a family of ~12 phone-viewport screens (Today, Focus, Conspectus detail, Schedule, Errors, Onboarding, Profile, Search, plus supporting shells). Without a single visual DNA, each screen drifts toward a different tier-1 aesthetic — one designer chooses Amie warmth, another chooses Bear-editorial, a third chooses Raycast-density — and the product reads as three apps stitched together. Every screen would also re-open the same debate from scratch; deciding once removes that cost.

Three tier-1 DNA takes were on the table when the first three screens were sketched:

  • Amie · signature warmth. One breathing orb, one CTA, editorial peek. Emotional carrier first, information second.
  • Bear · editorial typography. Display digit as hero, story-framed copy, zero chrome. Reader-first.
  • Raycast · command density. Keyboard-first, mono numerics, every row an action. Power-user first.

All three carry the same data and API contract; the choice is purely how the app feels. Because it feels the same on every screen, the choice is made once and applied everywhere — a per-screen re-vote is exactly the drift this ADR prevents.

Decision

Adopt Amie · signature warmth as the single visual DNA for every TMA screen. Four clauses.

  1. D1 — One breathing element per screen. Each screen has exactly one hero element that lives (the orb on Today, the timer ring on Focus, the calendar arc on Schedule, …). This element carries emotion; everything else is a peek.
  2. D2 — One primary CTA per screen. Every screen has one obvious next action. Secondary actions live as peeks or drill-downs, not as competing buttons.
  3. D3 — Editorial peek, not dashboard. Lists are quiet, lined, and one-thumb-scrollable. No charts, no widgets, no dashboard tiles. The app talks like a note, not a control panel.
  4. D4 — Warm ember accent, sage rest. A single accent (ember-500 warm orange) carries streak / progress / attention. A single rest state (sage-500 green) signals «you can breathe». No third accent — additional tones are muted greys.

D1–D4 apply to every screen. Individual screens document how their hero, CTA and peek satisfy the four clauses in the Per-screen application section below.

Scope

  • In scope: the visual DNA and its four clauses; the accent + rest palette; the «one breathing element + one CTA + editorial peek» composition rule; the growing per-screen application record.
  • Out of scope: API contract (unchanged across all DNA takes — an implementation detail, not a design decision); accessibility contract (owned by reference/a11y); typography / spacing tokens (owned by design IA).
  • Out of scope: web / desktop port. The DNA is written for a phone-viewport-first Mini App. A desktop port belongs to its own ADR, which decides how (or whether) the DNA scales.

Alternatives considered MANDATORY · MIN 2

The DNA was decided on Screen 01 · Today as the test surface and re-checked against Screen 02 · Conspectus detail — each screen's rejected sketches are archived below, numbered to match their picked counterpart in Per-screen application. Mocks are the ADR's rationale — not living component references. Nothing on this page updates when production changes.

Why considered: reader-first products (Bear, Ulysses, Craft) show that typography alone can carry identity. Zero chrome + display digit + reflective copy would give the app a calm, journal-like feel that fits a study product.

Why rejected: two independent reasons.

  • Habit loop needs an emotional carrier, not a paragraph. Users open the app in 2 seconds and glance. A display digit + prose demands reading; a breathing orb reads at a glance. For a daily-open habit product, the first-two-seconds tax is the whole game.
  • Editorial voice hides missed days. «Twelfth day» reads the same for streak-12 as for streak-1 unless the reader parses the number. The signature orb makes streak state pre-attentive: full ring = alive, dashed ring = broken, before any digit is read.

Why considered: the API surface is verb-heavy (review, snooze, focus, defer, promote) and would map naturally to a command palette. Power learners are willing to invest in keyboard fluency and would love it.

Why rejected: two independent reasons.

  • Phone-first · no keyboard. The Mini App runs inside Telegram on a phone. There is no ⌘K, no ⏎, no per-row modifier hint. The whole density argument dissolves when the primary input device is a thumb.
  • Cold-start hostility. Command bars are learned surfaces; they punish first-time users who do not know the vocabulary. The DAU-1 target for a daily-habit product is «open once, feel welcomed, tap once» — the opposite of «read six affordances, memorise three shortcuts».

The rule
Single responsibility

A module has one reason to change — and that reason is a person, not a technical layer. The report module cares who reads reports, not whether the report is a PDF.

Split by actor, not by feature.

Signal in your codebase: two unrelated pull requests keep touching the same file. That means two people share one module.

Why considered: the note is reading material — kicker, display title, body prose, sage pull-quote. No named blocks, no colour-coded rails; the reader carries themselves through. Typography as identity fits a reading surface better than any other screen.

Why rejected: two independent reasons.

  • No warmth at the top means no invitation. The Bear take opens with grey kicker + black title — an article, not a greeting. The warm-reader hero turns «here's your due note» into an invitation while keeping the body just as plain below the fold.
  • The pull-quote steals the one accent. Sage on a quote block spends the rest-state colour on decoration, against D4 — sage signals «you can breathe», not «this sentence matters». The picked take keeps the accent budget on the slot chip and the closing CTA.

Why considered: notes double as reference material — dotted-key meta (source · last · next · errors), a three-line outline, and per-row keyboard hints would make lookups fast for power learners.

Why rejected: two independent reasons.

  • Meta plates bury the text. The console take spends the first screen-height on metadata and outline before a single sentence of the note appears. A reading surface earns attention with the text itself (D3 — the app talks like a note, not a control panel).
  • Same phone-first argument as Screen 01. ⌘K, ⏎ and hint bars assume a keyboard the Mini App never has — the rejection carries over unchanged from the Today comparison.

Consequences

Positive

wins
  • The app feels like one product, not twelve. Every screen inherits the same warmth, the same one-CTA rhythm, the same peek voice.
  • New screens don't re-open the DNA debate. A designer opens Focus / Search / Profile knowing D1–D4 answer «which hero», «which action», «which list style» before any Figma is opened.
  • Signature transfers. The orb becomes the thing users describe to a friend («the one with the breathing circle»). Because DNA is uniform, that recognition transfers to Focus, to Schedule, to every future screen.
  • One accent, one rest state. Ember-500 + sage-500 is the entire semantic palette. Muted greys carry structural weight. Cognitive load stays low.

Trade-offs

costs
  • Power learners don't get keyboard superpowers. Raycast-style density on a laptop-companion would delight a slice of users. We accept the loss because the primary surface is the phone and phone-DAU is what the product is optimised for. A future desktop port can revisit.
  • Text-heavy edge cases feel light. A long conspectus body inside an editorial-Bear frame would breathe more than the same body inside the peek-density frame. Mitigated by Conspectus detail's own «reader mode» surface, which is a per-screen exception documented in the application record — not a DNA break.
  • DNA drift is easy to introduce accidentally. A single PR that adds a second accent tone or a second primary CTA silently degrades the DNA. Guardrail: every screen carries a subsection in Per-screen application declaring how D1–D4 apply; an absent subsection signals the DNA has not been applied.

Revisit — would we do this again?

Yes. Locking the DNA before the first screen is scaffolded — rather than after — keeps every screen honest against D1–D4 from the first commit. Any comparable product benefits from treating DNA as a pre-scaffold decision.

Per-screen application

One subsection per screen. Each subsection records how D1 (breathing element), D2 (primary CTA), D3 (peek shape) and D4 (accent) apply on that surface, together with a frozen visual reference and pointers to the living reference doc and Storybook story. Absence of a subsection means the DNA has not been applied to that screen.

Validation

  • Every documented TMA screen carries a card in Per-screen application. Absence of a card signals the DNA has not been applied to that screen.
  • Grep hygiene: only two accent tokens (ember + sage) surface in services/telegram/src/. A third would flag in code review.
  • One breathing hero per screen. Any screen with two competing hero elements (two orbs, orb + big ring, etc.) requires an inline comment linking this ADR.
  • Kit-showcase iframes render the live Storybook story. Static HTML mocks of hero components on kit-showcase pages are a DNA regression — reviewer looks for class="storybook-embed" or an iframe pointing at /telegram/storybook-static/….
  • make docs-check: passes with this ADR + linked pages; nav-tree + search index consistent.
