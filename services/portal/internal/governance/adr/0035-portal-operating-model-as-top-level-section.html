Portal operating model — top-level section with a self-serve hub

The portal now has 34 ADRs, 16 SA templates, 14 authoring how-to's, the Variant C IA from ADR 0032, a multi-stage delivery pipeline (ADR 0034), Pagefind search (ADR 0033), and a dozen pre-commit hooks. The individual artefacts are documented, but there is no single page an author can open to answer «what are the five rules, what's my next step, what gate will check me, what has changed in the process this month, and what edits drag other files along with them?» This ADR introduces a top-level operating-model/ section — seven router pages plus three new gates that close two coupling-loops (meta-change ↔ change-log, SA-template ↔ UI-kit twin) — so the day-to-day operating model is a place on the portal, not a memory burden on one author.

Context

Three things have grown faster than the surface that explains them:

  • Decision surface. 34 ADRs and the Variant C IA mean placing a new page requires answering six questions (ADR-0032 D1–D6) before any HTML is written. Those questions are documented across handbook/sa/topology.html, handbook/sa/diataxis.html, handbook/sa/document-types.html, and handbook/sa/authoring/internal-service-docs-layout.html — correct individually, but they are not assembled into one Q&A the author can walk.
  • Gate surface. The portal's quality gates are spread across 9 Make targets (docs-fix, docs-check, docs-html-check, docs-design-check, docs-a11y-check, docs-feedback-check, docs-spec-check, docs-nav-check, visual-test), 12 local pre-commit hooks, and 7 CI workflows. Definition-of-done at handbook/sa/authoring/definition-of-done.html lists checklist items per gate, but there is no inventory you can scan to learn the system itself.
  • Coupling surface. Some edits silently require edits to other files. The clearest case: an SA template under services/portal/internal/handbook/sa/templates/ is paired with its UI-kit twin under services/portal/ui-kit/pages/templates/ — both pages are pixel-diffed against the same visual baseline corpus in tests/visual/pages.json. Touching one without the other (or without refreshing the baseline) is a class of bug that is invisible until make visual-test runs, and there is no document that lists the coupling pairs. The same shape applies to meta-changes — a new pre-commit hook, Make target, CI workflow, or template should be announced in a curated «how the process changed this month» log, but there is nowhere to put one and no gate that asks for it.

The micro-documentation is fine. What is missing is the macro: one place where the five operating rules, the routing decision-tree, the lifecycle diagram, the inventory of gates with coupling, and the curated log of meta-changes all live, and a couple of gates that turn the coupling-discipline from «remember this» into «CI refuses to forget».

This pattern has names in adjacent teams: GitLab Handbook calls it Way of Working; Stripe and Shopify call it Operating Model; Backstage calls it a TechDocs landing. The shape is consistent — a single self-serve hub that synthesises rather than re-explains.

Decision

Adopt a top-level Operating model section, parallel to onboarding/ and handbook/, with six router pages and three new or extended gates. Originally ratified with seven pages; the standalone process-glossary was retired on 2026-06-13 (see Update — 2026-06-13).

D1 · Top-level placement, not a handbook sub-tree

The section lives at services/portal/internal/operating-model/ and is the second top-level node in the sidebar (after onboarding/, before tutorials/). It is not nested under handbook/sa/authoring/ because the audience is wider than SA-authoring (it covers any author, any role) and the surface is meta (rules of the portal itself, not SA-craft). Hiding it under handbook is the same mistake as hiding onboarding/ there — discoverability tanks and the section reads as one more reference page rather than the entry point.

D2 · Router, not replacement

Operating-model pages are lightweight routers. They synthesise (5 rules, one Q&A, one lifecycle diagram, one gate inventory, one curated feed) and point at the deep handbook pages for the «why». Theory of Diátaxis stays in handbook/sa/diataxis.html; layout philosophy stays in handbook/sa/topology.html; document-type contents stay in handbook/sa/document-types.html; the master five-step workflow stays in handbook/sa/authoring/add-a-page.html. Operating-model adds incoming links; it does not rewrite existing handbook content.

D3 · Six pages, no more

The shape is fixed at six pages. Adding a seventh requires a successor ADR.

D4 · Five prescriptive rules

Operating-model home declares five rules. Each rule is enforceable; deviations require an ADR.

  1. Where a page lives is determined by data-service (if it is about a service) or data-topic (if it is cross-cutting). Quadrant is metadata (data-page-type), not a folder. → ADR 0032 D1.
  2. Every page declares on <body>: data-page-type, data-service, data-role, data-topic, data-lifecycle, data-updated. Enforced by tools/docs/validate_docs_html.py (this ADR extends it; see D6). → ADR 0032 D7.
  3. Before commit run make -C services/portal docs-fix then make -C services/portal docs-check. If service pages were touched, run make -C services/portal verify on top. Drift = fail.
  4. Template → page. Every new page starts as a copy from handbook/sa/templates/. Structure is not self-expression.
  5. Process changes leave a trail. A new Make target, pre-commit hook, CI workflow, SA template, or operations-class ADR must update operating-model/whats-new-in-how-we-work.html in the same PR (gate D5 below). A change to a SA template must travel with its UI-kit twin per tests/visual/pages.json and a refreshed baseline (gate D5 below).

D5 · Two new pre-commit gates close the coupling loops

  • tools/docs/check_meta_changes_logged.py (pre-commit). Reads git diff --cached --name-only. If staged paths include .pre-commit-config.yaml, .github/workflows/*.yml, services/portal/Makefile, services/portal/internal/handbook/sa/templates/*.html, services/portal/internal/governance/adr/[0-9]{4}-*.html (new or status-changed), or tools/docs/*.py — then operating-model/whats-new-in-how-we-work.html must also be staged. The hook prints which meta-path triggered and an example block to paste into the feed. Runs as a hard fail; META_LOG_SOFT=1 downgrades to a warning for the first week.
  • tools/docs/check_template_twins.py (pre-commit). Reads tests/visual/pages.json, builds a map of twin pairs from tpl-sa-* ↔ tpl-ui-kit-* ids. If staged paths touch one side of a pair without the other, the hook fails with the two paths printed and the command make -C services/portal visual-test-update as the remediation hint.

Neither gate is wired into make docs-check — the drift gate already runs over HEAD for orthogonal reasons, and the staged-diff context these hooks need is meaningful only at commit time. Pre-commit is the right surface.

D6 · Extend validate_docs_html.py for the D7 metadata schema

ADR-0032 W6 promised that tools/docs/validate_docs_html.py would assert the D7 metadata schema on every page under services/portal/internal/ except governance/, onboarding/, autogenerated reference, and code-reference/. That extension is in scope of this ADR — without it, Rule 2 above is convention, not check. The extension preserves the existing FROZEN_DOCS_REL_PATHS and pdoc skip-lists.

D7 · The curated feed is curated

The «what's new in how we work» feed is hand-authored, not generated. Each entry answers four columns: date, what changed, why, and what changes for the author. Auto-generating it from commit history (the obvious shortcut) loses the «what changes for the author» column, which is the only column that does work for the reader.

Scope

  • In scope: the six pages under services/portal/internal/operating-model/; the two new pre-commit gates; the D7 extension to validate_docs_html.py; the sidebar top-level node in nav-tree-internal.json; one pillar CTA on services/portal/internal/index.html; one callout on services/portal/internal/handbook/sa/authoring/index.html; cross-links from onboarding/glossary.html, handbook/sa/process.html, handbook/sa/authoring/add-a-page.html, handbook/sa/authoring/definition-of-done.html, handbook/sa/authoring/internal-service-docs-layout.html.
  • Out of scope: rewriting any existing handbook page. Operating-model adds incoming links; the handbook content stays where it is.
  • Out of scope: editing services/portal/internal/catalog/index.html — that page is generated by tools/docs/build_catalog.py; any manual edit would be reverted by docs-fix and break docs-check.
  • Out of scope: per-PR or per-branch coupling for runtime behaviour, OpenAPI baseline, or other non-portal artefacts. The coupling map starts with twin templates only and is extended one row at a time when a new pattern earns enforcement.

Alternatives considered

Nest the hub under handbook/sa/authoring/

rejected

Why considered: the authoring how-to's already live there; the new pages «look like» more authoring content; one fewer top-level node in the sidebar.

Why rejected: the audience is wider than SA-authoring — a SWE adding a runbook, a QA adding a checklist, an SRE adding a postmortem all need the routing tree and the gate map too. Discoverability for non-SA roles drops sharply when the section hides behind «SA · Authoring». The hub is the front door, not a sub-page; it deserves the visibility of a top-level node.

Auto-generate the «what's new» feed from git history

rejected

Why considered: zero-maintenance, never out of date, no «curator-bottleneck» risk.

Why rejected: a commit log answers «what changed», not «why» or «what changes for the author» — the two columns that make the feed useful. An autogen feed is also a much weaker forcing function for the meta-change gate: the gate's whole job is to make the author write the «why» entry; if the feed populates itself, the gate has nothing to ask for.

Replace overlapping handbook pages with the hub

rejected

Why considered: the handbook contains process (process.html), document-types, page-skeleton, topology — all topics the hub touches. Merging them looks like deduplication.

Why rejected: the handbook pages are deep narrative; the hub pages are routers. Different audiences (deep-dive vs daily-driver), different shapes (essay vs Q&A/table), different update cadences. Merging collapses both. The router-vs-replacement split (D2) is the cleaner contract — it lets each surface evolve on its own clock.

Encode the coupling map in CI only (no docs page)

rejected

Why considered: hooks alone enforce coupling; the gate map page could be dropped as «just a list the script knows».

Why rejected: the docs page is the place a human goes to understand the system before they trip a hook, not after. Without the page, the hook fail message is the only surface that ever shows the coupling rules — which means coupling becomes visible only on failure. The page is the open contract; the hook is its enforcement; both are needed.

Consequences

Positive

wins
  • One entry point for «what are the rules and what is my next step» — measured by the four cold-start questions in §Validation.
  • Coupling-loops (meta-change ↔ feed; SA template ↔ UI-kit twin) become CI-enforced rather than memory-enforced. Adding a future coupling pattern (e.g. nav-tree-internal.json ↔ check_sidebar_coverage) is one row in pages.json/coupling table and one line in the hook.
  • D7 metadata schema becomes a real lint, finishing ADR-0032 W6.
  • Handbook stays clean — operating-model points at it, does not rewrite it. The router/deep-dive split scales as both surfaces grow.

Trade-offs

costs

Three named tradeoffs. Shape from Tradeoff disclosure.

  • A curated feed must be curated. We gain «what changes for the author» — the column auto-generation cannot produce. We lose the option to ignore the meta-log: the gate refuses commits to meta-paths without it. We accept this because the throughput of meta-changes is low (a handful per month) and the gate's whole point is the forcing function.
  • One more top-level sidebar node. We gain discoverability for a section that is the front door. We lose vertical sidebar density for users who never need the hub once they have internalised it. We accept this because the section is finite (seven pages, ADR-bound at D3) and onboarding hubs are exactly what top-level nodes are for.
  • Coupling enforcement is local-only. Pre-commit catches twin-drift and missing-feed at commit time, but a CI matrix on a fresh checkout does not re-run those checks. We accept this because the hook surfaces the failure at the moment the author can fix it cheapest, and the existing PR-review surface is the backstop for anyone who bypassed pre-commit.

Revisit — what triggers a successor?

Re-open this decision when any of the following becomes true: (a) the section needs an eighth page (ADR-bound at D3), (b) coupling enforcement needs to move into CI for an additional safety reason, (c) a second operating model emerges (e.g. a separate public-portal hub with different rules).

Validation

  • After C2: make -C services/portal docs-check passes; the new top-level sidebar node renders on every page.
  • After C7: tools/docs/validate_docs_html.py fails on any page missing the D7 attribute set; once metadata is patched, the tool passes again and Rule 2 is enforced rather than aspirational.
  • After C8: a staged change to .pre-commit-config.yaml without an accompanying edit to operating-model/whats-new-in-how-we-work.html is rejected by pre-commit; the rejection message names the meta-path and prints a feed-entry template. A staged change to handbook/sa/templates/runbook.html without portal/ui-kit/pages/templates/ops-runbook.html is rejected with the twin path and the visual-test-update command. pre-commit run --all-files passes on the current tree.
  • Cold-start probe — open operating-model/index.html in a browser without referring to the plan and answer four questions in 60 seconds:
    1. Where does a new page about monitoring-dashboard latency live? Expected: services/portal/internal/services/monitoring/... with data-service="monitoring", data-page-type="reference".
    2. Which gate fails if data-updated is missing? Expected: docs-html-check via the extended validate_docs_html.py.
    3. If I edited templates/runbook.html, what else must change in the same PR? Expected: ui-kit/pages/templates/ops-runbook.html plus baseline refresh via visual-test-update.
    4. What has changed in the process in the last two weeks? Expected: visible in the home feed.
    If any answer takes longer than 60 seconds, the hub has failed its purpose and the page is iterated.

Update — 2026-06-13

The standalone process-glossary page was retired. D3 now declares six pages instead of seven. Each of the twenty-six process terms moved to the page where it is actually used (gate terms on quality-gates-map, coupling terms on its § D table, metadata terms on page-type-contract); product / domain terms continue to live at onboarding/glossary. The decision itself (top-level Operating-model section) is unchanged; this is a refinement, not a reversal — no successor ADR.

Same update added a 3-minute mental model section to the hub home: one SVG «brain» (Content / Shape / Couplings / Gates) plus one «monorepo map» (four services, root make verify, ci.yml matrix) plus three short paragraphs. The lifecycle page replaced its bulleted ten-station list with a horizontal SVG timeline. Both motivated by the «3-minute pitch» test: if the hub cannot be explained in three minutes, the model is not in the author's head.
