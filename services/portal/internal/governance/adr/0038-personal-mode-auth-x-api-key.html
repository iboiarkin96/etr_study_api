Personal-mode auth — reuse X-API-Key, defer Telegram initData → JWT

The Telegram Mini App epic ships in personal mode — one maintainer, one identity. Keep the API's existing X-API-Key header check as the auth. Drop the API_AUTH_STRATEGY switch, keep a plain API_AUTH_HEADER + API_AUTH_KEY pair. The Telegram initData HMAC → JWT path (originally the top clause of the epic's «Locked decisions») is deferred to the multi-user epic and becomes mandatory before any second user reaches the app. Amends ADR 0005's auth-strategy clause; amends ADR 0010's env-profile guard rule.

Context

The Telegram Mini App epic was drafted with «JWT (stateless) issued from Telegram initData» as a locked auth decision — HMAC-verify the payload the WebApp SDK provides, upsert the user, mint a 24 h JWT, guard every /api/v1/* route. The rationale: no session table, no revocation ceremony, and the day one identity (the Telegram user) equals the day one account.

Three facts changed the calculus when the plan was audited against the shipped code:

  • The API already enforces auth. A middleware in app/main.py gates every /api/v1/* path on X-API-Key. Reusing it is zero code; adding a JWT layer beside it is a new dependency (PyJWT), a new endpoint (POST /api/v1/auth/telegram), a new schema migration (five columns on users), and a new client-side token-storage story (CloudStorage — not localStorage).
  • The MVP audience is one person. Every scope choice in the epic's «Locked decisions» collapses to «maintainer only». Anything the JWT layer would buy — cross-user identity mapping, revocation, per-user rate limits — has no consumer in this epic.
  • Dev-loop cost of initData. The Vite-in-browser dev path (the primary loop this week) has no signed initData. Making it real means either running through a Cloudflare Tunnel with a real Telegram client on every keystroke, or a dev-only signing helper that adds another surface to keep in sync with the verifier. Header-key auth is agnostic to how the page was opened.

The already-shipped auth also carries dead ceremony from a time when we planned multiple strategies: API_AUTH_STRATEGY is a switch between mock_api_key (the only implemented option), disabled, and a placeholder branch that returns 500 for anything else. Simplifying it — one header, one key, one guard — costs less than keeping it around.

Decision

Keep the existing header-key auth, simplified. Defer initData → JWT to the multi-user epic. Four clauses.

  1. D1 — X-API-Key is the only auth for /api/v1/*. Middleware compares the header value against a single expected key. /live and /ready stay open, matching ADR 0009. The Mini App client (services/telegram/web/) injects VITE_API_KEY into every request via an OpenAPI-generated interceptor; every write also carries a fresh Idempotency-Key per ADR 0006.
  2. D2 — Drop API_AUTH_STRATEGY. The switch had one working branch (mock_api_key) and two dead branches (disabled, unknown). It is removed from app/core/security.py and app/core/config.py. Two env vars survive: API_AUTH_HEADER (default X-API-Key) and API_AUTH_KEY (renamed from API_MOCK_API_KEY, same semantics). The qa/prod invariant flips from «strategy ≠ disabled» to «API_AUTH_KEY ≠ local-dev-key».
  3. D3 — The Telegram identity migration is not shipped now. The users.telegram_user_id / telegram_username / telegram_photo_url / locale columns, the POST /api/v1/auth/telegram endpoint, the PyJWT dependency, the WebApp.CloudStorage token dance — all deferred to the multi-user epic. Nothing about them is written in a way that blocks the future migration; when it lands, both auth paths coexist for the switchover window.
  4. D4 — Key ships in the bundle; documented, not hidden. Because there is no token-exchange step, VITE_API_KEY is baked into the Mini App bundle at build time. Anyone who opens the deployed URL can read it. This is acceptable in personal mode — the key guards one user's data, the rotation surface is one file, and the exposure disappears entirely when D3 unwinds. It is not acceptable the day another human logs in; that day the multi-user epic starts.

Scope

  • In scope: the auth middleware and its config knobs, the qa/prod guard, the env-profile files (env/example, env/dev), the two test files that exercise the auth flow, and the portal doc pages that named the old vars.
  • In scope: the TELEGRAM_BOT_TOKEN + TELEGRAM_CHAT_ID settings surface — introduced now so the bot half of the epic (W5) has somewhere to read them. Both empty by default; the bot is disabled until they are set.
  • Out of scope: the Telegram initData HMAC verifier, the JWT mint/verify path, the CloudStorage token layer, and the identity migration. These belong to the multi-user epic and are gated by its opening.
  • Out of scope: the OpenAPI security scheme change — the shipped baseline already models the key as apiKey / header / X-API-Key; nothing to edit beyond regenerating the baseline after the config-var rename.

Alternatives considered MANDATORY · MIN 2

Ship initData → JWT now, as originally planned

rejected

Why considered: the epic's own «Locked decisions» table put JWT there for a reason — it makes the day-one identity equal the Telegram user, and it kills the «key ships in the bundle» risk before it exists.

Why rejected: in personal mode the JWT gets nothing done that the header key does not. The costs — new dependency, new endpoint, five-column schema migration, a token-exchange step that has to work in three dev topologies (plain browser, tunnel, real Telegram) — buy a property (multi-identity) that the epic explicitly forbids in its «Audience: maintainer only» clause. Correct move: build the property when the constraint changes, not before.

Keep the strategy switch, only defer initData

rejected

Why considered: the smallest patch — keep API_AUTH_STRATEGY=mock_api_key as-is, don't touch the dead branches, defer JWT alone. Zero risk of regressions in the existing tests.

Why rejected: the strategy switch is dead ceremony. disabled is explicitly forbidden in qa/prod already; unknown branches return 500. Nothing chooses between them at runtime. Every doc page that mentions it — three ADRs, two runbooks, two authoring pages — is teaching contributors a knob that does not turn. Removing it is one commit and clears the concept from the codebase; keeping it is drag on every reader.

Signed shared secret in a Bearer header (custom JWT-lite)

rejected

Why considered: reuse the Authorization: Bearer … shape so the eventual JWT swap is a client-side no-op (same header, richer payload).

Why rejected: re-implements the shape without reimplementing the safety. A HS256 JWT with a shared secret in the bundle offers no additional security over a raw shared key — you still ship the key, you still can't revoke, and you now have a new signature verifier to maintain. The eventual JWT swap is one interceptor edit away from the header-key setup; there is nothing to «prepare» here.

Consequences

Positive

wins
  • Zero new auth machinery. The middleware, tests, and doc pages already exist. The change is a rename plus a switch removal — smaller blast radius than a fresh integration.
  • Browser-tab dev works out of the box. Vite on localhost:5173 reaches the API with a header set from VITE_API_KEY. No tunnel, no signed initData, no bot-token dependency for UI work.
  • Config surface shrinks. Two env vars become one; the strategy branch and its qa/prod guard collapse into a single non-default-key check.
  • The epic's cut line is achievable. W1 becomes «config tidy + ADR + baseline», not «new endpoint + migration + JWT integration». The 2026-07-17 local-test checkpoint stays reachable.

Trade-offs

costs
  • Key ships in the bundle. Anyone who opens the deployed Mini App URL reads VITE_API_KEY. Acceptable in personal mode; unacceptable the moment audience opens. Mitigation: rotate on suspicion of leak; the rotation cost is a single env write plus a redeploy.
  • No per-user identity yet. The API cannot answer «who is the caller» beyond «someone with the key». Every existing endpoint already models identity via the (system_user_id, system_uuid) pair in the request payload, so business logic keeps working; observability loses the ability to attribute traffic to a Telegram user until D3 unwinds.
  • Multi-user epic must ship before a second user. This is a hard invariant, not a soft aspiration. If a second person is about to log in and the JWT swap is not merged, the swap is the blocker — not «we'll add auth later».

Revisit — would we do this again?

Yes. One refinement worth noting up-front:

  • Write the multi-user auth ADR at the same time as the personal-mode one. This ADR names its own sunset condition — the day a second user shows up — but the ADR that replaces it does not exist yet. Having the successor drafted (even as Proposed) would make the deferral concrete rather than a promise.

Compatibility and migration

  • Backward compatibility. None with API_AUTH_STRATEGY / API_MOCK_API_KEY. Contributors update their local .env once: replace both lines with API_AUTH_KEY=…. The header name (API_AUTH_HEADER) is unchanged; the wire contract is unchanged; the OpenAPI security scheme is unchanged.
  • Migration path for the future JWT swap. The multi-user auth ADR will add a POST /api/v1/auth/telegram endpoint and a JWT verifier as a second middleware layer. Both auth methods coexist during the switchover window; the header-key check is removed the day the last client migrates.
  • Rollback. Revert this commit set. The strategy switch and the old env vars come back; the Mini App keeps working because the header contract did not change. No data-shape rollback needed.
  • Old ADR pointers. ADR 0005's auth-strategy clause (and the API_AUTH_STRATEGY / API_MOCK_API_KEY row in its config table) is superseded by clauses D1–D2 above. ADR 0005 stays Accepted for its wider security-baseline scope. ADR 0010's qa/prod invariant row «API_AUTH_STRATEGY=disabled forbidden» is superseded by «default API_AUTH_KEY forbidden» — same intent, adjusted for the collapsed switch.

Validation

  • Auth contract: curl /live returns 200 without a key; curl /api/v1/conspectuses returns 401 without a key and 401 with a wrong key; curl -H "X-API-Key: local-dev-key" /api/v1/conspectuses?… returns 200 with valid params. Confirmed against dev-tier Postgres on 2026-07-14.
  • Test suite: make test (pytest, testcontainers-python) green — auth cases exercise valid / missing / wrong key and the qa/prod default-key guard.
  • Config guard: APP_ENV=qa API_AUTH_KEY=local-dev-key python -c "from app.core.config import get_settings; get_settings()" raises ValueError: Set a non-default API_AUTH_KEY for qa/prod.
  • OpenAPI parity: make openapi-check passes; services/portal/public/reference/api/openapi.json regenerated with no diff beyond the config field rename in the security scheme's documentation.
  • Docs check: make docs-check passes — nav, catalog, and search index consistent with the ADR addition.
  • Grep hygiene: rg 'API_AUTH_STRATEGY|API_MOCK_API_KEY' returns only history-of-record references (this ADR, ADR 0005's amendment banner, ADR 0010's amendment row, CHANGELOG entry).
