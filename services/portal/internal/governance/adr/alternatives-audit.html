ADR alternatives audit

An ADR without rejected alternatives is a decision dressed as a recommendation. Vol. 2 architect-bar audit flagged "Alternatives discipline" as a gap; this page is the survey of all 30 ADRs against that gap, the explicit punch list of which ones need work, and the bar a new ADR must meet to land green.

The bar

An ADR's Alternatives considered section is STRONG when it names ≥ 2 alternatives and gives each a one-sentence why rejected rationale. It is WEAK when it names alternatives but lacks rationales, or names only one. It is MISSING when there is no alternatives section at all. Decisions without rejected options are not decisions; they are reports.

Priority list — fix order

The five highest-leverage ADRs to upgrade first. Leverage = (reader frequency) × (decision irreversibility). Foundational ADRs that everything else builds on top of go first.

  1. ADR 0003 — Error contract governance. Touched on every endpoint. RFC-7807 problem-details must be named and rejected explicitly so future reviewers know we considered it. Critical.
  2. ADR 0005 — API security defaults. Foundational and externally visible. OAuth and mTLS should be named with the reasons we chose API-key for v1.
  3. ADR 0006 — Idempotency. Touched on every write. Natural-key dedup is a real alternative that deserves a written rejection.
  4. ADR 0011 — SLO/SLA & error budget. Reviewers will ask "why SLO + budget, not SLA-only?". The answer should be in the ADR, not in the author's head.
  5. ADR 0020 — C4 PlantUML. Mermaid is banned by this ADR but the rejection rationale isn't in the document. "Why not Mermaid?" should be a paragraph here.

Rule for new ADRs

  • An ADR PR without an Alternatives section gets bounced at review.
  • An Alternatives section with one option gets bounced at review.
  • An Alternatives section with options but no rejection rationale gets bounced at review.
  • "No alternatives applicable" is acceptable only when the ADR is purely descriptive (e.g. naming a convention that already exists in standard library). Even then, the section says "no alternatives applicable because …", not "this section is absent".
  • Use ADR 0001 as the worked example. Four alternatives, each with one paragraph of rejection rationale.

Forbidden patterns

  • "We considered other options." Name them. Specifically.
  • Listing alternatives without rejection rationale. "We considered X and Y" is a list; "we rejected X because Z" is an ADR.
  • Cargo-culting alternatives from other repos. If you copied "we considered REST, gRPC, GraphQL" from a template without thinking about whether they apply, the reviewer can tell. Tailor to the actual decision.
  • Adding alternatives after the fact. Reconstructed alternatives are obvious to readers (and to future-you). If the section is missing, upgrade it explicitly with a history-row entry; don't pretend it was there from the start.
