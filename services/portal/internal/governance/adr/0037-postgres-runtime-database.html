PostgreSQL as the runtime database (containerized)

Replace SQLite with PostgreSQL 16 running in a docker-compose service as the runtime database for the API. Same schema, same Alembic migrations, same sync SQLAlchemy session; new driver (psycopg[binary] v3), new URL contract (DATABASE_URL), new JSON storage (JSONB). Supersedes the «SQLite + single replica» clause of ADR 0015.

Context

The API has run on SQLite since day one, mounted as a file under var/api/study_app.db and consumed via SQLAlchemy 2.x with check_same_thread=False. It was the right call at the start — zero-install, zero-ops, one file to copy. The container-image ADR explicitly kept SQLite + single-replica in scope and pushed multi-replica to «a Postgres-swap concern, gated by trigger, not date».

Three triggers have now landed:

  • Concurrent-write pressure. The 2026-04-02 SEV-2 postmortem («SQLite write-lock burst») cited SQLITE_BUSY as the saturation signal; capacity snapshot (BL-056) locked that in as the ceiling. Every write path — /v1/conspectuses, /v1/schedule, /v1/errors, event-log inserts — competes for the same file lock.
  • Growing JSON payloads. BL-071 (conspectus / schedule / error-log domain, shipped 2026-07-11) introduced three JSON columns (cue_sheet, bullets, payload). SQLite stores them as TEXT; there is no indexed access, no partial update, no operator to query «all conspectuses whose cue_sheet.section equals X». Postgres JSONB + GIN indices solve this at the storage layer.
  • Integration-test parity with prod. BL-005 already scoped a CI job that boots a throwaway Postgres to catch migration + concurrency regressions. Running one database in tests and a different one at runtime is the exact «mock/prod divergence» the ticket exists to eliminate. Moving runtime to Postgres collapses the two.

The cost of the swap is bounded: schema is generic SQL, Alembic already drives migrations, SQLAlchemy already abstracts the dialect. What changes: the URL, the driver, three column types, the test-boot fixture, and the compose topology. What does not change: the API surface, the models, the request/response contracts, the entrypoint shape.

Decision

Adopt PostgreSQL 16 as the runtime database. The decision has seven clauses.

  1. D1 — Postgres 16 in docker-compose. A dedicated postgres service in the root docker-compose.yml, image postgres:16-alpine, healthcheck via pg_isready, host-mounted data volume at var/postgres/ (gitignored). The api service depends on it with condition: service_healthy.
  2. D2 — DATABASE_URL is the single config knob. The application reads exactly one env var to reach the database: DATABASE_URL=postgresql+psycopg://user:pass@host:port/db. No SQLITE_DB_PATH, no legacy fallback. Env profiles (env/dev, env/qa, env/prod) each carry their own URL.
  3. D3 — Driver: psycopg 3, binary wheel. Dependency psycopg[binary]>=3.2. Chosen over psycopg2 (unmaintained roadmap) and asyncpg (would force an async rewrite of the whole SQLAlchemy layer — out of scope). The SQLAlchemy session stays sync; the API layer stays sync-Depends. Async is a follow-up ADR if ever needed.
  4. D4 — Alembic stays the migration engine, migrations stay in place. The two shipped migrations (20260411_0001_initial_schema, 20260712_0002_add_conspectus_schedule_errorlog) run unchanged on Postgres with two type-level edits: sa.JSON() → postgresql.JSONB() on the three JSON columns (conspectuses.cue_sheet, conspectuses.bullets, conspectus_events.payload). Edited in-place because the migration was shipped one commit ago on this same feature branch and has never run against Postgres — no history to preserve.
  5. D5 — Tests use testcontainers-python. The pytest session fixture spins up a fresh postgres:16-alpine container, sets DATABASE_URL, runs Base.metadata.create_all(), then TRUNCATE … RESTART IDENTITY CASCADE between tests. Rejected: shared docker-compose service (leaks state), in-memory SQLite (defeats the purpose), pytest-postgresql (needs a system-installed pg_ctl).
  6. D6 — Entrypoint waits for Postgres, then migrates, then serves. The container entrypoint (services/api/scripts/container_entrypoint.sh) loops on pg_isready (30 s timeout, ~1 s cadence) as a defensive backstop to the compose healthcheck, then runs alembic upgrade head, then exec uvicorn. Same shape as before; SQLite-specific mkdir removed.
  7. D7 — SQLite is removed, not deprecated. No dual-mode config, no fallback branch. Contributors upgrade in one commit; the var/api/study_app.db path is dropped from all env files and docs. Local dev-data loss is acceptable: the file has always been gitignored dev-scratch. A one-shot dump-and-load recipe lives in the how-to for anyone with dev data worth keeping.

Scope

  • In scope: runtime persistence for the FastAPI service; the compose topology used for local dev, CI, and release smoke; the test harness; every env-file and docs page that names SQLite.
  • In scope: the JSON → JSONB type swap in the three shipped columns (cue_sheet, bullets, payload). Rationale: correctness at the storage layer — indexed access, in-place update, operator support — with no schema change visible to the API.
  • Out of scope: HA / read replicas / connection pooler (PgBouncer). Single-node Postgres matches single-node deploy today; multi-node is a separate ADR when the trigger fires.
  • Out of scope: Async SQLAlchemy migration. The container-image ADR's «shell entrypoint same as host run» property still holds; async is orthogonal.
  • Out of scope: Data migration of prod-tier SQLite (there is no prod-tier Postgres to migrate to either — this ADR builds the destination).

Alternatives considered MANDATORY · MIN 2

Keep SQLite; add WAL + connection-serialisation shim

rejected

Why considered: zero infrastructure change. Turning on WAL mode and serialising writes through a single connection would blunt the SQLITE_BUSY problem at some capacity.

Why rejected: it delays the swap, it does not remove it. The JSON-column problem stays (no indexed access), the mock/prod divergence in BL-005 stays, and the write-lock ceiling only moves — it does not disappear. Every hour spent on the SQLite tuning is an hour not spent on the swap that is coming anyway.

Managed Postgres (Neon / Supabase / RDS)

rejected

Why considered: zero ops, backups included, one-click branching.

Why rejected: the project deploys through docker-compose per ADR 0015 and publishes to GHCR per ADR 0021. Reaching a managed vendor adds a network dependency to the «one docker compose up and it works» promise, splits the ops story between two consoles, and would need per-contributor credentials the project has no story for. Reconsidered when we outgrow single-node; not before.

Async SQLAlchemy + asyncpg in the same ADR

rejected

Why considered: Postgres unlocks async patterns anyway; do both at once.

Why rejected: bundling two independent decisions doubles the review surface and the blast radius. Async touches every route handler, every dependency, every test fixture; sync-Postgres is the smallest atomic change that clears the three triggers. Async waits for its own ADR with its own load-bearing reason.

Consequences

Positive

wins
  • Concurrent writes hold. Row-level MVCC replaces file-level locking; the SQLITE_BUSY ceiling is gone.
  • JSON becomes first-class. JSONB + GIN indices on cue_sheet, bullets, payload enable the query shapes BL-071 documented but could not implement on SQLite.
  • Tests and runtime share a database engine. BL-005's raison d'être — «catch migration + concurrency regressions before merge» — becomes automatic; the testcontainer runs the same Postgres image that runs in prod.
  • Standard tooling. psql, pg_dump, pg_stat_statements, EXPLAIN (ANALYZE, BUFFERS) — every DBA reflex works.

Trade-offs

costs
  • Docker becomes a hard dev-loop dependency. The «one-file SQLite» rescue path is gone. We gain a shared engine across dev / CI / prod; we lose the ability to run the API against a laptop-local file with no daemon. We accept it because Docker is already assumed by ADR 0015 and by the observability compose.
  • Cold-start latency ~5–8 s. Postgres container boot + healthcheck + migration takes visibly longer than «open a file». We gain correctness; we lose the sub-second boot. Mitigation: keep the compose stack up between iterations; docker compose start postgres reuses the volume.
  • Test suite fires up a container. First pytest after a clean docker registry pulls ~150 MB. We gain runtime-parity tests; we lose the pure-Python test suite. Mitigation: session-scoped container fixture (one boot per pytest invocation), TRUNCATE-based per-test cleanup.

Revisit — would we do this again?

Yes. One refinement we would land on day one:

  • Ship a make target for the compose lifecycle. make db-up / db-down / db-reset aliases in the root Makefile so contributors do not have to memorise docker compose up postgres -d. The current plan tucks this into make dev; standalone targets keep the mental model cleaner.

Compatibility and migration

  • Backward compatibility. None with SQLite. Existing dev-tier var/api/study_app.db files are dropped; the API surface, response shapes, and error contracts are unchanged.
  • Migration plan for contributor dev data. One-shot: sqlite3 var/api/study_app.db .dump | pgloader /path/to/dump postgresql://user:pass@localhost/study_app (recipe lives in the how-to). Nobody is expected to have irreplaceable dev data; the fallback is a fresh migrate.
  • Rollback. Revert the seven implementation commits. Because the schema-DDL edits are in-place edits of a never-run-against-Postgres migration, rolling back re-installs the SQLite-side sa.JSON() type without any ALTER. Contributors reconstruct dev data from seeds.
  • Old ADR pointer. ADR 0015's Decision-list clause «Default container paths: SQLite file under /data…» and Scope clause «Keep SQLite + single replica for this ADR's scope» are superseded by clauses D1–D2 above. ADR 0015 stays Accepted for its packaging-artifact scope.

Validation

  • Boot: docker compose up -d results in a healthy postgres service; api starts after the healthcheck flips green; alembic upgrade head completes as part of the entrypoint.
  • Round-trip: smoke of POST /v1/conspectuses with a non-trivial cue_sheet, then GET /v1/conspectuses/{id} — JSONB survives the round-trip byte-for-byte.
  • Test suite: pytest tests/ boots the testcontainer, migrates, exercises the suite, and tears the container down. First run pulls the image; subsequent runs reuse it.
  • Grep hygiene: rg -i 'sqlite' -g '!var' returns only history-of-record references (this ADR, ADR 0015's superseded clause, CHANGELOG entries).
  • Docs check: make docs-check passes — catalog, search index, nav-tree consistent.
