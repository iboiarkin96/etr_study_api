Environment Profiles and Config Governance

Three explicit profiles (dev, qa, prod), a hard split between public profile behavior (committed under env/) and private secrets (uncommitted .env), layered loading on the host and mirrored inside Docker containers, and strict validation that blocks unsafe settings outside dev.

Context

Why this matters: Twelve-factor style apps read config from the environment so the same build can run in dev, staging, and prod. If profiles are fuzzy, teams accidentally ship debug logging, mock auth, or open CORS to production—often discovered by incident, not review.

The config surface has two audiences that behave very differently. Public profile behavior (log level, CORS allowlist, rate limits, feature flags) is safe to commit — reviewers need to see it, CI needs to lint it, contributors need it to onboard. Private secrets (bot tokens, external SaaS keys, personal debug overrides) must never enter git — one commit exposes them permanently in history. Mixing the two into a single file forces every contributor to redact secrets by hand on every commit, which is the exact pattern that ends in leaked keys.

We need explicit profiles (dev, qa, prod), a hard split between the committed and uncommitted layers, a documented template, and validation that rejects unsafe combinations in non-dev environments. The same layering must apply inside Docker containers so a compose-run process reads exactly what the same host-run process would.

Decision

  • Adopt explicit environment profiles: dev, qa, prod only.
  • Two-layer split. env/{dev,qa,prod,example} carry public profile behavior and are committed to git. Root .env carries private secrets and per-developer overrides and is git-ignored. env/example is the canonical template — every key that could be set anywhere in the system has a documented slot there with an empty value for secrets.
  • Precedence (host). services/api/app/core/config.py::_load_env_files() loads in this order, later wins: .env at repo root (override=False, so shell-exported values survive) → env/<APP_ENV> (override=True) → optional ENV_FILE. The _PARENT_WINS_KEYS escape hatch reinstates shell-exported values after profile load for a small allowlist of keys that must not be perturbed by profile reload (DATABASE_URL for tests, rate-limit overrides for local load-tests, JWT / bot-token secrets).
  • Precedence (Docker). docker-compose.yml mirrors the same layering via env_file: directives on each service — env/<profile> first, .env second. The explicit environment: block on the compose service takes precedence over both, and is reserved for values that must differ between host-run and container-run (e.g. APP_HOST=0.0.0.0, DATABASE_URL=…@postgres:5432/…).
  • Secret hygiene. Never commit a real secret to env/*. Empty placeholders are allowed as documentation (they show the key exists) but the real value only lives in .env or in the deployment platform's secret store. CI reviewers reject any PR that adds a non-empty secret to env/*.
  • Automated tests use APP_ENV=qa (not a separate profile). Legacy APP_ENV=test maps to qa.
  • Normalize legacy aliases (for example local -> dev).
  • Expose effective profile in GET /live as app_env.
  • Enforce strict runtime validation for unsafe settings in qa/prod.

Scope

  • In scope: the conventions, rules, and artefacts named in the Decision section above.
  • Out of scope: any system, process, or surface not explicitly listed here.

Alternatives considered

Four alternatives were evaluated before adopting three explicit profiles + a two-layer (public / secret) split + fail-fast validation. Each is real (another serious team in the industry uses it) and explicitly rejected with a reason rooted in our constraints.

No profiles — every setting is an independent env var

rejected

Why considered: the strict twelve-factor reading. No profile name, no validation rules tied to it — every behaviour is one explicit env var that the operator sets. Maximum composability.

Why rejected: with N independent env vars, every unsafe combination is reachable by accident. The matrix of «which combinations are valid?» grows exponentially and lives in nobody's head. A profile name collapses the matrix to «which row of the validation table am I on?», which is a tractable question. Composability we do not need is paid in clarity we do.

External secret/config service (Vault, AWS Parameter Store, GCP Secret Manager)

rejected

Why considered: the «secrets and config out of the env entirely» pattern — the app reads from a vault at boot. Standard for enterprise deployments with rotation requirements.

Why rejected: a vault adds a separate failure mode (vault down ⇒ app cannot boot), a separate auth model (vault credentials are themselves a secret), and a separate local-dev story (vault stub or shared dev secrets). For a solo project with no secret-rotation requirement yet, the env-file + git-ignored .env pattern is strictly cheaper. The day we have multi-tenant secret rotation is the day to revisit.

More profiles: dev, test, staging, prod, preview, …

rejected

Why considered: match a Heroku-style review-app workflow with one profile per deployment kind. More granular guarantees per environment.

Why rejected: three profiles cover «running locally», «running with prod-like constraints» (qa/test), and «running in production». Adding more profiles invites «which one applies?» ambiguity (is the load-test environment qa or its own load?) and grows the validation matrix N×. Three is the smallest set that lets us write meaningful per-profile rules without combinatorial blowup.

Single mixed file — secrets and public config together in .env

rejected

Why considered: the maximally-flat model — one file, one place to look, no «which layer wins?» question. Every developer sees the same thing when they open .env.

Why rejected: pushes every review of «is this rate-limit reasonable?» / «did CORS change?» into a git-ignored file that reviewers never see, and every non-secret behavior change either evades review entirely or forces the author to duplicate the value into a committed doc that immediately drifts. Worse, it makes «forgot to strip a secret before committing» the default failure mode instead of the exceptional one: any time an operator wants to share their .env for reproduction, they have to redact by hand. The two-layer split converts the review question («is this behavior change safe?») into a git-diff-visible artefact, and makes the secret-leak question into a physical one («did the diff touch a file that git tracks?»).

Trade we accept by choosing three explicit profiles + a two-layer split + fail-fast validation. Every new validation rule and every new key needs a placement decision (public or private?) at authoring time. We accept it because each rejected alternative invites a failure mode this ADR exists to prevent: no-profiles ships unsafe combinations silently, external vaults add an outage axis we have not justified, more profiles inflate the matrix faster than the value grows, and a single mixed file guarantees that some contributor will paste a real bot token into a committed doc within the first quarter. Three profiles + a committed template + an uncommitted secret file is the smallest mechanism that is simultaneously auditable, operable, and leak-resistant.

Consequences

Positive

wins
  • Clear branch logic for environment behavior with safer defaults.
  • Reduced risk of accidental insecure production configuration.
  • Better developer onboarding for environment-specific setup.
  • Secret-leak surface reduced to a single git-ignored file — a review of «did this commit touch a secret?» is a one-liner (git diff --stat | grep -v '^ \.env').
  • Docker containers see the same layering the host does — no «works on my Mac, 500s in compose» class of bug.

Trade-offs

costs

Two named tradeoffs accepted on purpose. Shape from Tradeoff disclosure.

  • Fail-fast on unsafe config (startup refuses to boot). We gain a process that cannot serve traffic with mock auth in qa / prod — the validation rule is the gate, not a runtime check that can be silently disabled. We lose the option of «boot in degraded mode and warn» (some teams prefer this for partial-outage scenarios). We accept it because partial-boot with mock auth in production is the exact failure mode this ADR exists to prevent; the «soft warn» pattern always converges on «the warning was ignored, now we have the incident».
  • Profile name baked into the runtime (APP_ENV). We gain a single switch that the validation rules and the /live endpoint can read, so the effective profile is visible end-to-end. We lose the «one binary, infinitely composable env vars» purity that some twelve-factor purists prefer. We accept it because the profile is the load-bearing branch in our validation rules; making it implicit (derive from other vars) is the recipe for «which profile am I actually in?» confusion.
  • Every new key needs a public / private placement decision. We gain a git-diff-visible signal for «did this commit change behavior?» versus «did this commit change a secret?» — reviewers read two very different signals from the two file locations. We lose the «one file, one grep, one edit» simplicity of a mixed layout. We accept it because the placement question is decided once per key, and the answer stays with the key forever; the alternative — asking every reviewer to inspect every diff for accidentally-committed secrets — never converges on «safe» and never scales beyond a solo project.

Revisit — would we do this again?

Yes, with one piece we did not commit to early enough. The three-profile + fail-fast model has caught two unsafe pre-deploys (a leftover LOG_LEVEL=DEBUG in a prod dry-run, a default mock key in a qa smoke). One addition we would commit to on day one if starting today:

  • Validate the env file before docker build, not at runtime. Today the validation runs at app startup; a misconfigured deployment produces a crash-loop pod that the operator has to inspect to learn what went wrong. A CI step that exercises services/api/app/core/config.py's validator with the production env file (with secrets stubbed) would fail the deploy at the pipeline step, not at the pod. The validator already exists; wiring it into pre-deploy CI is a 30-min job that closes the «caught at boot, not at build» gap.

Compatibility and migration

  • Backward-compatibility impact: none — pure governance / convention change with no wire-level effect for existing consumers.
  • Migration plan: apply the rules to new work; bring existing artefacts into compliance during normal touch-ups.
  • Rollback strategy: revert the merge PR; no data or schema changes to undo.
