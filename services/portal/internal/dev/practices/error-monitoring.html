Error monitoring — every exception has an owner

The engineer's try / except doesn't log.error() and forget — it calls capture_exception, the platform fingerprints the stack, routes it to the CODEOWNERS team, and the next standup has a list with names attached.

TL;DR

Sentry (founded by David Cramer in 2008) defined the modern category: every uncaught exception is captured with full stack trace, deduplicated by stack-frame fingerprint, attached to the user that triggered it, and routed to a code-owner. Datadog Error Tracking, Bugsnag, and Rollbar are the same product shape. The developer's job is small: install the SDK, attach a user_id on every captured event, and ensure your code path maps to a CODEOWNERS team. The platform turns errors from a torrent of noise into a finite, owned backlog.

What it is

Error monitoring is the practice of capturing every uncaught exception (and explicit "this should not happen" errors) from every service and every environment into a single tool that fingerprints, deduplicates, and assigns them. Sentry, Datadog Error Tracking, Bugsnag, and Rollbar are the four major products; their data models are identical in shape.

The components are:

  • SDK in every service — auto-captures uncaught exceptions; exposes capture_exception(e) for explicit cases.
  • Fingerprint — a hash of the stack frames (module + function + line) used to group events. Two crashes in the same function are the same issue.
  • Event context — user_id, release_version, environment, request_id, breadcrumbs (the last N log/UI events leading up to the exception).
  • Owner routing — the file path of the top stack frame is matched against a CODEOWNERS-style routing rule; the issue lands in the owning team's inbox.
  • Lifecycle — new → triaged → resolved → regressed (auto-reopened when the same fingerprint reappears after release X).
  • SLA — P1 (user-facing, high volume): 24h to triage. P2 (rare or background): 1 week. Unowned: zero, by policy.

Why we need it

Errors are infinite if you log them and bounded if you fingerprint them. The same null-pointer that fires 100,000 times a day is one issue, not 100,000 issues. Without dedup, the volume is overwhelming and the team gives up. With dedup plus ownership, errors become a finite, ranked backlog that a team can actually drain.

Ownership is the second load-bearing piece. An error that lands in a generic "errors" channel is an error nobody fixes — diffusion of responsibility is a real phenomenon, and a Sentry dashboard everybody can see is a dashboard nobody owns. Routing by CODEOWNERS pushes the issue into a specific team's queue with their name on it, and the team's "open errors" count becomes a visible health metric.

How others do it

Sentry (David Cramer, 2008)

Sentry pioneered the category and its public customer list reads like a Who's Who of engineering: GitHub, Disney, Microsoft, Atlassian, Cloudflare. Their model is the reference: fingerprint by stack, attach user/release/breadcrumbs, route by code owner, lifecycle with auto-regression-detection on release. Open-source SDKs for ~30 languages.

What we take: Attach user.id on every captured event. The single most powerful filter in error monitoring is "is this hitting one user 1000 times, or 1000 users once?" — and you only get to answer it if the user ID is on the event.

Source: Sentry — Welcome

Datadog Error Tracking

Datadog's offering integrates errors with their tracing and logging products — an error in the Error Tracking view links to the trace and the logs from the same request. Fingerprinting is automatic; routing is via integration with PagerDuty / Slack / GitHub. The product's emphasis is on correlation: an error is never just an exception, it's an exception with a trace, a request, and a user attached.

What we take: The error event must carry trace_id. One click from "this error is happening" to "here is the trace of the failing request" is the difference between 5-minute and 30-minute diagnosis. Cross-tool correlation is the whole game.

Source: Datadog — Error Tracking

Bugsnag (mobile/web)

Bugsnag's framing is "stability score" — the percentage of sessions free of any crash. Each release publishes its score and the goal is monotonic improvement. Errors are not a problem to be triaged but a metric to be moved. Releases that drop the score below a threshold are auto-flagged for rollback consideration.

What we take: Treat error-free session rate as a SLI. A release that drops it is a release that introduced regression. The metric is more actionable than the individual issue list — it tells you when to stop the release.

Source: Bugsnag Blog

Anatomy of error monitoring

SDK installed in every service

No service is exempt. Even cron jobs and one-off scripts initialise the SDK — those are the ones that fail silently for months. Bootstrap config in the shared service template.

User context on every event

HTTP middleware calls Sentry.set_user({id: req.user.id, tier: req.user.tier}). Now every captured exception inside that request carries who hit it. The "1 user × 10k times" vs "10k users × 1 time" distinction is the most important triage signal.

Release tagging

CI sends the release SHA + version to the error platform on every deploy. New errors are auto-tagged with the release that introduced them. Regression detection (issue resolved in v1.4, reappears in v1.6) requires this.

CODEOWNERS-driven routing

The platform's routing rules mirror the repo's CODEOWNERS. A change to ownership is a single PR. Unowned code paths are the source of the unowned-error problem and a CI check rejects them.

Weekly triage ritual

30-minute team meeting on Monday. Sort by event count × user count. Assign / fix / mark "known low-impact." Goal: leave the meeting with the new-error count at zero. Track the "open P1 errors" number as a team KPI.

Severity tied to user impact, not log level

P1 = blocks user, >100 users/day or >10 enterprise users. P2 = degraded, single-digit users/day. P3 = log-spam, known recoverable. Severity is not the log call's argument; it's a triage decision.
