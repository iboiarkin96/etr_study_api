Structured Logging — JSON, context, severity

The engineer writes log.info("checkout.submitted", order_id=42, amount_cents=1999) instead of print(f"order {x} done") — every line becomes a queryable event, joined to traces and metrics by the correlation IDs the logger fills in automatically.

TL;DR

Structured logging means every log line is a JSON object with a fixed schema: timestamp, level, service, trace_id, message, plus domain fields. Modern libraries — structlog (Python, Hynek Schlawack), pino (Node.js), zerolog (Go), slog (Go stdlib since 1.21) — make the call-site cost essentially zero. The job of the engineer is to pick a verb-shaped message key, attach the IDs the rest of the system will join on, and stop writing prose into log messages. The payoff is that "find the bug" becomes a SQL-like query instead of a multi-tab grep through gigabytes of free text.

What it is

Structured logging is the practice of emitting each log line as a JSON object with a canonical schema, rather than as a free-form text string. The engineer's job is to pick a verb-shaped event name, attach the relevant IDs as fields, and let the logging library handle the rest.

Every log event consists of:

  • severity — debug / info / warn / error; the ladder is enforced.
  • event / message — a stable, dotted key like checkout.submitted — not a sentence.
  • context — service, env, version from environment, not hard-coded.
  • correlation IDs — trace_id and span_id from the OpenTelemetry context, injected automatically by middleware.
  • domain fields — user_id, order_id, latency_ms, etc., chosen per event.
  • sink/format — JSON to stdout; the platform ships it to Loki / Datadog / OpenSearch.

In practice this is a one-line library call: log.info("payment.refused", reason="insufficient_funds", order_id=order.id). Tools that make this idiomatic: structlog in Python, pino in Node, zerolog and Go's standard-library log/slog in Go. None of them require the developer to construct JSON by hand.

Why we need it

Free-text logs are a write-once medium: somebody grep-debugs them once during an incident, and they are never queried again. That is the worst possible ROI on storage you are already paying for. The minute the lines are JSON with a stable schema, the log store becomes a queryable database — "show me every checkout.submitted event in the last hour where amount_cents>10000 and the trace went through the fraud service" is one query, not an afternoon.

The correlation IDs are the load-bearing piece. Once every log line carries the same trace_id as the distributed trace and the same request_id as the metric tag, the three pillars of observability are joined — pick any one and pivot to the other two. Without correlation, every diagnosis starts from scratch in a different tool.

How others do it

Stripe

Stripe's API emits a structured log line for every request with a request_id that is also returned in the Stripe-Request-Id response header — so the customer's "this charge failed" report comes with the exact log lookup key already in it. Internally every service uses the same canonical logger; severity discipline and field naming are enforced by code review and a shared library, not by convention.

What we take: Expose your request ID to the caller. The fastest support ticket is the one that arrives with the join key. Internally, a single shared logger library is the only way to get a consistent schema — convention is not enough.

Source: Stripe Engineering Blog

structlog (Python, Hynek Schlawack)

structlog popularised "key-value pairs, not f-strings" in the Python world. Its why page is a manifesto for treating logs as events, not as prose: bind contextual fields once at the start of a request (logger.bind(user_id=42, trace_id=…)), and every subsequent log call within that request inherits them. The output format (JSON, key=value, dev-readable) is a render-time decision, not a coding decision.

What we take: Bind context at the boundary (HTTP handler, queue consumer), not at the call site. The engineer writing business logic should not have to remember to pass user_id to every log.info — the library does it.

Source: structlog — why structured logging?

Elastic / Datadog (vendor docs)

Both Elastic Common Schema and Datadog's standard log attributes specify a fixed set of names for the same concept (service.name, trace.id, error.stack_trace). Once your services emit logs that conform, every dashboard, alert, and correlation feature works out of the box. Diverging from the schema is technically allowed and operationally painful.

What we take: Pick a published schema (ECS is the most widely adopted) and follow it field-for-field. Inventing your own naming is a tax you pay every time you set up a new tool.

Source: Elastic Engineering Blog

Anatomy of a structured log event

Severity ladder

Exactly four levels: debug (off in prod or sampled at 1%), info (significant state change), warn (recovered from an error), error (someone needs to look). Anything more granular gets ignored.

Event key, not sentence

The message field is a dotted, stable name (order.shipped), not a templated string. Variable data goes in domain fields. Aggregations over event=order.shipped become trivial; aggregations over "order {id} shipped" require regex.

Bound context at the boundary

HTTP middleware binds request_id, user_id, trace_id once per request. Every log.info inside that request inherits them. Domain code never plumbs context manually.

PII redaction at the emitter

A processor strips or hashes known-sensitive fields (card_number, password, email raw) before serialisation. PII never leaves the process. Allow-list the fields you do want, deny-list everything else.

JSON to stdout, ship by platform

The service writes JSON to stdout; the runtime (k8s logging agent, ECS Firelens) ships it. The service is not aware of Loki vs OpenSearch vs Datadog — that is one config change at the platform layer, not a code change in every service.

Schema CI lint

A linter rejects raw print, console.log, fmt.Println, or log calls that don't go through the shared library. Catches violations at PR time instead of during the next incident.
