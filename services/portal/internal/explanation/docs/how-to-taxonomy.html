Docs Explanation One axis — noun

How-to taxonomy

Why internal/how-to/ is organised by the noun you act on — API, service, docs, QA — and not by who is acting (roles) or what genre the document is (runbook, postmortem). Roles consume how-tos; they don't own folders. Genres get their own subtree when they have different mechanics from a regular recipe.

Background

Until 2026-05-17 the internal/how-to/ tree mixed three different classification axes in one level of folders. api/ and docs/ named the subject of the task. dev/ and qa/ named the role of the author. runbooks/ and postmortems/ named the genre of the document. The three axes overlap silently: an API load-test is a Dev task on an API artefact, and an API-endpoint test is a QA task on the same API artefact — and yet the first lived under dev/ and the second under qa/. Contributors had to pick a side, and the side they picked was inconsistent with the next contributor.

The team-roles axis already exists, separately, under internal/team/roles/ (architect / dev / sa / qa / sre / manager). Each role hub there has a radar of practices, an onboarding page, and links into the relevant how-tos. So when the how-to/ folder also organised by role, the same axis was being asserted twice — once by location, once by linkage — and the two assertions disagreed for leakage cases.

The rule

One axis: the noun the task acts on. Pick the folder by asking "what is this task about?" and not "who will run it?". Top-level folders under how-to/ are nouns:

Folder What it is about Examples
api/ The HTTP surface (endpoints, contracts, OpenAPI artefacts). Add a write endpoint; endpoint testing; load testing.
service/ The running service as a process (env, build, run, register). Local development; Make workflows; Docker image; service catalog.
docs/ The documentation system (pages, pipeline, frontend). Internal docs layout; safe frontend changes; doc-as-code pipeline.
qa/ Quality verification not bound to a single noun above. Accessibility; documentation testing; release smoke; exploratory.
incidents/ Reactive operational work. A different genre from a normal recipe — numbered, templated, triggered by events — so it lives in its own subtree. incidents/runbooks/, incidents/postmortems/.

Why noun, not role

  1. Roles are already modelled elsewhere. The team/roles/ portal owns the role axis, with radars, onboarding, and curated reading lists per role. Duplicating the axis under how-to/ means the same fact is asserted twice — and inconsistently. One source of truth wins.
  2. Tasks frequently span roles. The endpoint-testing recipe is read by QA and by the dev who wrote the endpoint. If the folder name picks a side, the other side has to know to look there anyway — so the folder name buys nothing and costs predictability.
  3. Nouns are stable. The HTTP surface still exists if the QA role goes away; the docs system still exists if "dev" splits into "platform" and "feature". The noun is what survives reorgs.
  4. Reverse lookup still works. The role view is given by the team/roles/<role>/radar.html page and by the filter chips on how-to/index.html. Both cut across the folder tree without needing the tree to be cut that way.

Verb-over-noun is the trap to watch for

The most common mis-categorisation is to file a guide by its verb instead of by the noun it acts on. "Load testing" is a verb; "API" is the noun. The recipe lives under api/, not under a hypothetical load-testing/ folder and not under whichever role drives the verb (Dev, QA — both, depending on the week).

Rule of thumb: if the page title can be rewritten as "X-ing the {noun}", file it under {noun}. Endpoint testing = testing the API → api/. Local development = developing the service → service/. Documentation testing = testing the docs → docs/ (or qa/ when it cuts across; see next section).

When the noun is "everything" — QA

Quality verification that is not bound to a single noun lives in qa/. Accessibility testing applies to every page we ship; exploratory testing applies to the whole product surface; release smoke is the last gate before any release. These are first-class subjects in their own right — the noun is "the deliverable" or "the release", not "the API" or "the docs". Don't try to scatter them across the other folders; the cohesion belongs to QA.

The boundary with verb-shaped subfolders is sharp: endpoint testing is about the API surface and lives under api/; documentation testing is about the docs surface but is currently filed under qa/ because it shares method and process with the rest of the QA recipes. If the docs-testing recipe grows product-specific shape (e.g. portal-page-anatomy assertions), it migrates to docs/. The principle is: file with the smaller cohesive group.

Why incidents/ is its own subtree

Runbooks and postmortems are not regular recipes. They share three properties that normal how-tos don't have:

  • Trigger-driven. You read a runbook because a pager went off, not because you decided to start a task. The reader's state is "I need to recover, fast, right now"; the writing style — preconditions, fast triage, exit criteria — mirrors that.
  • Numbered and templated. RB-0000, RB-0001, RB-0002 …; every runbook follows the same skeleton; the numbering is a stable handle that on-call references in chat ("see RB-0006").
  • Genre-shaped review. Postmortems are blameless, signed off by an incident commander, and audited for action-item follow-through. Runbooks are drilled. Neither lifecycle resembles how a normal how-to is written or maintained.

These properties make incidents/ a natural genre bucket inside How-to. It is the one place where genre wins over noun, because the genre carries the operational meaning.

When something doesn't belong in how-to/ at all

Diátaxis has four buckets; not every "how do I…?"-shaped instinct belongs in How-to. Use this fast filter when filing or reviewing:

If the page is… …it belongs in… Reason
A flat lookup table or generated inventory reference/ Reference is information-oriented and dense; not task-shaped.
"Why we do it this way" / a discussion of trade-offs explanation/ Understanding-oriented; this very page lives there.
"Walk me through end-to-end as a learner" tutorials/ Learning-oriented, time-budgeted, no real-world choices.
A real, completed, scoped task with prerequisites and exit how-to/ Task-oriented, choice-tolerant, doesn't teach the basics.

The make-commands page is the canonical mismatch story: it was filed under how-to/docs/ as "make-commands-inventory" but is a flat, generated list of every Make target. That's Reference, not How-to. It now lives at reference/service/make-commands.html; the task-shaped recipe ("which command to run when") stays in how-to/service/make-commands-and-workflows.html.

Decision recipe

For a new how-to, work through these in order — stop at the first that gives a clear answer:

  1. Is this Reference / Explanation / Tutorial? If yes, file there. If no, it's How-to — proceed.
  2. Is this triggered by an incident? If yes → incidents/runbooks/ (numbered) or incidents/postmortems/. Otherwise proceed.
  3. What noun does this act on? API surface → api/. Running service / build / run / catalog → service/. The docs system / portal / pipeline → docs/.
  4. If no single noun fits — is this quality verification across nouns? If yes → qa/. If still no single home, that's a signal the recipe is two recipes pretending to be one; split it.
  5. Never use a role as the folder. "It's a dev task" is not an answer to step 3 or step 4; pick the noun the dev acts on.

Consequences

  • The hub registry at how-to/index.html filters by five chips — API · Service · Docs · QA · Incidents — and the chip set matches the folders one-to-one. New filter = new folder; new folder = new chip.
  • Adding a new top-level folder is an explicit decision documented here. A security/ folder, for example, would only be added when a critical mass of security-task recipes exists and they don't fit any existing noun. Until then, security tasks file under their noun (e.g. api/api-security-…).
  • The team/roles/ portal continues to be the source of truth for the role axis. A role's reading list is curated there and points into how-to/; the reverse path doesn't exist by design.
  • The incidents/ subtree is the only place where genre overrides noun. Adding a second genre folder (e.g. incidents/drills/) is allowed when a new operational genre emerges; adding a non-incident genre folder elsewhere requires this page to be updated first.
