QA Explanation

Why QA spans the whole SDLC

QA is not a stage between “done coding” and “shipped”. It is a stance the team takes at every stage — and the reason it has to be everywhere is the same reason it cannot be only at the end.

The fallacy of “QA at the end”

A naive process puts testing in a box near the end of the timeline: design, code, then test. The picture is appealing because it suggests a tidy hand-off. It is also wrong, in a way that has been documented for fifty years.

The cost of fixing a defect rises sharply with how late it is caught. A bad assumption corrected in the spec costs an edit. The same assumption discovered after the code is written costs a refactor. Discovered after release, it costs a hotfix, a customer apology, and a regression test that should have existed in the first place. Shifting left — moving testing earlier — is not a slogan; it is the only economic answer to that curve.

The other reason “QA at the end” fails is structural. A test plan written after the spec is signed off can only verify what the spec says. If the spec is silent about a case, the plan is silent too, and the defect ships. The cases that the spec misses are exactly the cases QA is best positioned to surface — at the moment when surfacing them is cheap.

What “shift left” means in practice

The cheapest place to find a defect is in a sentence that has not yet been written into code. QA's most leveraged activity is reading drafts — specs, ADRs, edge-case lists — and asking the questions that surface ambiguity: what happens on retry, what does the empty case return, who is allowed to call this, what error does the client see when authorisation fails. Each question turned into an acceptance criterion is a defect that never has to be filed.

By the time code exists, QA's role has shifted from preventing defects to finding them. By the time release approaches, it has shifted again to confirming that what was promised is what shipped. The activities look different at each stage; the underlying stance — being the voice for the absent caller — is the same.

Three kinds of confidence, three places they come from

A team shipping a release needs three different kinds of confidence, and conflating them is a recurring source of regret.

  1. Confidence in the design Comes from review of the spec and edge cases, before code. Cannot be retrofitted by testing.
  2. Confidence in the implementation Comes from the test pyramid — unit, integration, contract. Automatable; runs on every change.
  3. Confidence in the release Comes from explicit acceptance against the test plan and from regression coverage. Confirms that what we have matches what we promised.

A release with green CI but no acceptance run has the second kind of confidence and not the third — the tests we have pass, but we have not confirmed they are the tests we needed. A release that skips contract tests but runs a careful manual pass has the third without the second — it ships today, but the next change to the API quietly breaks an old client.

Why QA needs independence from “done”

The single most useful property of a QA function is that it is not the same person as the implementer. Not because developers are careless — most are not — but because the writer of a piece of code is the worst-positioned person to find its blind spots. They have already convinced themselves that their interpretation of the spec is the only reasonable one. Otherwise they would have written different code.

This is why QA's sign-off on a release matters even when the developer has “tested it locally”. The sign-off is not a redundancy check; it is a different read of the same artefacts by a person who is allowed to disagree. A process that turns QA into a rubber stamp loses this property, and with it the only systemic defence against the failure mode where a feature is built exactly to the wrong specification.

Risk, not coverage, is the planning unit

Test plans framed around coverage — “test every endpoint, every field, every status code” — sound thorough and are usually impossible. Even a moderate API has more states than can be exhaustively exercised in any realistic time budget.

The planning unit that survives contact with reality is risk: the product of how likely a failure is and how bad it would be if it happened. A payment endpoint with retries deserves an order of magnitude more attention than a read-only listing endpoint, even if both have similar surface area. A regression in error-envelope handling is worse than a regression in success-path latency by 5 ms, even though both are bugs. The plan describes where attention is being spent and what is consciously being left out — and the latter is at least as important as the former.

This is also the principle behind the severity matrix: not every defect is equal, and triaging them by impact rather than by order of discovery is how a finite team ships a finite release.

A hotfix compresses the ceremony, not the gates

Under incident pressure there is a temptation to skip steps. The steps that compress safely are the slow administrative ones — extended discovery, formal sign-off ceremony, full regression sweep. The ones that do not compress are the gates that make the fix correct.

A hotfix without a reproducing test is a hotfix that you cannot tell has worked. A fix without a smoke pass is a fix that may have shipped a worse problem than the one it solved. Compressing those steps trades the incident for a future incident, usually one that arrives faster than the original would have.

Common failure modes

  • “QA is a phase.” The phase ends, the testing ends, and the bugs that needed earlier conversations never get caught.
  • Coverage as a goal in itself. Maximising line coverage at the cost of risk coverage: every if-branch is tested, the wrong cases are not.
  • QA without independence. The same person writes the code and the test; both reflect the same misunderstanding of the spec.
  • Sign-off without reproducibility. A release ships because “we tried it and it worked” — without recorded evidence the team cannot diagnose what they actually tested if a defect surfaces.
  • Defect amnesia. Each escape is fixed but never fed back into checklists or test plans; the same class of bug reappears in a different endpoint a quarter later.
