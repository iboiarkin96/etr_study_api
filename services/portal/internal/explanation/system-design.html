System design — moved

This page used to carry both the cross-service architecture picture and the architectural decision record. Following the Backstage pattern, those two jobs now live in distinct homes. Use the links below.

Where each section moved

Cross-service picture

C4 L1 / L2 diagrams, system-in-one-paragraph, in-the-box / out-of-the-box, governance index → Service catalog · Architecture. The catalog is now the architectural spine: the System entity owns the cross-service view; Components (individual services) own their internals.

Per-service internals (C4 L3)

Domain model, persistence, request lifecycles, schema design notes for each service → its own Architecture page: api · portal · monitoring · ui-kit.

Frozen architectural decisions

The D1–D7 decisions (dual journals, inline body, schedule table shape, SQLite-by-default with PostgreSQL scale path, separate learning_errors, manual edits via events, schedule_policies catalogue) are being promoted to individual ADRs under governance / adr. Until the migration completes, the historical narrative is preserved in the page git history.

Why it moved

The original page tried to do three jobs at once: explain the system, record decisions, and serve as the reference index. That mixes three Diátaxis categories under one URL, which makes it hard to keep current and easy to lose information in. The new layout assigns each job a clear owner:

  • Catalog owns the cross-service map — it is updated whenever a service is added or its wiring changes.
  • Per-service Architecture pages own the L3 component views — they evolve with the service.
  • ADRs own the frozen why of every architectural choice — they are immutable once accepted.

This page is kept as a tombstone so existing links and bookmarks land somewhere useful instead of 404'ing. It will be removed in a future cleanup pass once inbound links have been migrated.
