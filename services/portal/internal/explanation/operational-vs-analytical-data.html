Operational vs analytical data — one database, two workloads

The same tables answer two different kinds of question: «what does this user see right now» and «what happened across users, over time». They pull the schema, the latency budget, and the privacy rules in opposite directions. This page names the boundary while it is still cheap to respect.

The two workloads

OperationalAnalytical
Question shape«Load conspectus X for user Y» — one row, now«How many reviews per week? Which slot transitions dominate?» — many rows, over time
Access patternPoint lookups via PK / owner indexScans and aggregations across owners and dates
Latency budgetP95 in the request path — millisecondsSeconds to minutes are fine; nobody is waiting on a spinner
Schema preferenceNormalised, current-state rowsDenormalised, append-only history
Privacy postureFull PII under the owner's access controlIdentifiers only — aggregates must be safe to share widely

Where this system stands today — honestly

One PostgreSQL instance serves the operational workload; there is no analytical store, no dashboard fed from these tables, and analytical questions are answered by ad-hoc SQL against production tables. At the current scale (single instance, no external traffic) this is the correct amount of infrastructure — a warehouse for a system with no users would be resume-driven architecture.

But the schema already takes sides, and took them deliberately:

  • The append-only audit tables are proto-analytical surfaces. conspectus_review_logs stores before/after state per review — exactly the shape a retention or learning-curve analysis needs, without touching the mutable rows. conspectus_events does the same for content changes.
  • Policy pinning keeps history replayable. Because every review row pins schedule_policy_version, an analysis over old reviews is not silently corrupted when the algorithm changes.
  • Mutable rows hold only current state. conspectus_schedules is overwritten per review — any metric computed from it answers «what is», never «what happened». Metrics belong on the logs.

The boundary artifact: events, not queries

The clean seam between the two worlds is an event contract: the operational side emits a named, versioned, PII-free record at write time; the analytical side consumes events without ever holding a connection to the operational schema. Instrumenting at write time costs one decision per endpoint (§14 of the API spec names its events); reconstructing behaviour later from mutable tables costs an archaeology project.

When the split becomes physical

Three triggers, any one of which justifies moving the analytical workload off the operational instance:

  1. Scan queries appear in the operational latency profile — an aggregation holds locks or IO that a user request pays for.
  2. Retention rules diverge — the data lifecycle wants operational rows purged while analysis wants years of history; one table cannot obey both.
  3. A consumer who must not see PII needs query access — a dashboard, an experiment framework, a second person.

The first physical step is deliberately small: a read replica or a periodic export of the append-only tables — not a warehouse, not a pipeline team. Each step after that should be pulled by a query that hurts, never pushed by architecture fashion.

Forbidden patterns

  • BI-style queries against operational tables in the request path's hours. The cheap version of the split is temporal before it is physical.
  • PII fan-out into analytics. Events carry client_uuid, never full_name or free-text content — the join back to a person happens only under the operational side's access control.
  • Metrics computed from mutable rows. A metric on conspectus_schedules changes retroactively as rows are overwritten; metrics read the append-only logs.
  • Building the analytical stack before the analytical question. The trigger list above is the gate; «we might need it» is not on it.
