SWE Explanation YAML is SoT

Service descriptor policy

Why the catalog uses a small subset of the Backstage descriptor, why catalog-info.yaml is the single source of truth and the entity-card HTML is generated from it, why renderer-only fields live under an x-study-app extension block, and why each omitted Backstage field is deliberately left out. The template is the what; this page is the why.

Background

Backstage's software-catalog descriptor is a YAML file at the root of every component repository. It carries everything an internal developer portal needs to render a service: owner, lifecycle, type, dependencies, provided/consumed APIs, links, tags. It's the closest thing the industry has to a lingua franca for service metadata.

We don't run Backstage. We run a static internal portal (this site) generated from HTML, PlantUML, and pdoc. Four services live in the catalog today (api, portal, ui-kit, monitoring), and until 2026-05-17 each rendered its metadata as a hand-edited HTML <dl class="entity-card__fields"> block inside services/portal/internal/services/<name>/index.html.

The shape of those blocks was Backstage-inspired by convention since RFC 0004, but it was convention by example only — nothing told the next contributor which fields were required, which were optional, or which Backstage fields we'd quietly decided not to use. The Tech Docs Maturity v1/v2 (2026-05-10) audit flagged this as the SEV-1 gap on Axis 2: "no catalog-info.yaml anywhere in the repo, and no written rule for what an internal service descriptor must contain or may omit."

On 2026-05-17 the policy flipped. Each service now has a catalog-info.yaml file alongside its index.html, and scripts/render_service_descriptors.py regenerates the entity-card HTML, the hub tile grid, the hero lifecycle tickers, and the internal nav-tree subtree from those YAMLs. The HTML regions are bracketed by <!-- catalog:start --> / <!-- catalog:end --> markers; a pre-commit hook (make catalog-render-check) fails any commit whose generated regions drift from the YAML.

Why a subset, not the full descriptor

Backstage's descriptor optimises for a single hosted catalog UI that imports YAML from many independent repos. Most fields exist to answer "how does the catalog UI render entity X relative to entity Y" — system grouping, namespace partitioning, label-based filtering, APIs as first-class entities you can navigate to.

We have four services and one repo. We don't have:

  • multiple teams that need a namespace to keep their catalogs separate;
  • a group-of-services layer that would justify spec.system or spec.subcomponentOf;
  • a catalog UI that reads metadata.labels as queryable key/values;
  • a Backstage instance reading spec.providesApis as navigable entities.

Adopting the full descriptor would mean filling those fields with placeholder values that lie about the system's actual structure. The lie would cost us twice: once at author time (every new service has to invent a fake namespace and system) and again at read time (a reader following spec.system arrives at nothing).

The chosen subset keeps the parts that already do work: owner answers "who do I page?", lifecycle drives the hub's lifecycle filter and the hero tickers, dependencies render as the dep-lanes component on the entity page itself, and the last-reviewed annotation drives a freshness pill. Every field has a consumer. apiVersion: backstage.io/v1alpha1 and kind: Component are included not because we run Backstage today, but because the YAML file is now the canonical input — and writing it Backstage-compatible means a future importer is one config flag away.

Why YAML is now the source of truth

The first version of this policy (Phase 1, 2026-05-17 morning) kept HTML as the SoT on the grounds that "shipping YAML without a Backstage importer gives us two source files for the same data — the textbook drift trap." That argument is correct whenever the two files are both hand-edited. It collapses the moment one of them is generated from the other.

The Phase 2 inversion (2026-05-17 afternoon) introduced scripts/render_service_descriptors.py as the build-time generator. That removes the drift trap by construction: the HTML regions inside <!-- catalog:start --> / <!-- catalog:end --> markers are overwritten on every make catalog-render, and the --check mode (wired as a pre-commit hook) fails any commit whose YAML and HTML disagree. There is no longer "two files to keep in sync" — there is one file (YAML) and one derivative artifact (HTML) that CI refuses to ship out of date.

Once that's true, YAML is the strictly better SoT because:

  • Schema you can actually validate. A YAML file has a tree structure that a Python script (or a JSON Schema, or a Pydantic model) can lint field-by-field. An HTML <dl> needs heuristics for every check.
  • Single derivative, multiple targets. The same YAML now drives four artifacts — entity-card HTML, hub tiles, hero tickers, nav-tree subtree. Adding a fifth target (sitemap, search index, deploy manifest) is a function in the generator; with HTML as the SoT, each artifact would need its own scraper.
  • Backstage-portable from day one. The canonical apiVersion, kind, metadata, spec tree is the exact shape a Backstage importer expects. The day we do run Backstage, no schema migration is needed — only renderer-only fields (x-study-app) get stripped.
  • Authoring stays in one place. A contributor adding a new service edits one YAML file, runs make catalog-render, and the four targets all update consistently. The Phase 1 process required hand-editing the entity-card HTML, the hub tile, the hero ticker, and the nav-tree JSON — four places to drift.

Why renderer-only fields sit under x-study-app

The YAML carries two kinds of data: fields a Backstage importer would understand (spec.lifecycle, metadata.tags) and fields that are only meaningful to our renderer (header pills, dependency-lane structure, tile-lede copy, hub display order). Mixing the two would force a future importer to either choke on unknown fields or silently ignore them.

Putting renderer-only fields under a single top-level x-study-app key gives a clean cut: Backstage's spec explicitly says importers must ignore unknown root keys, so an entire renderer-only block disappears for free at import time. The alternative — stuffing renderer-only fields into metadata.annotations with a vendor prefix — works but pollutes the annotations namespace that Backstage itself uses for tool integration (source location, last-reviewed, techdocs refs).

The x- prefix follows the OpenAPI / Backstage convention for vendor extensions and signals "this is not part of the upstream schema" at a glance.

Why a generator rather than a static-site templating step

The entity-card HTML is intricate — dep-lanes with per-chip icons, optional typed links, lifecycle pills, footer chrome. We considered three options for closing the gap between YAML and HTML:

  1. Jinja template + nightly rebuild. Rejected: a template engine forces the whole page through the template; we only want to own a marked region inside an otherwise hand-authored HTML file (the page has handwritten history, spine cards, contextual prose).
  2. Web component that reads YAML at runtime. Rejected: static-site portal, no runtime YAML parser, and the hub tiles need to be present in the HTML for the search index crawler.
  3. Python script with marker-bracketed regions. Adopted. The script rewrites only the bytes between catalog:start / catalog:end markers — handwritten HTML around them is preserved. Idempotent by construction: running it twice in a row produces zero diff.

The marker-region pattern matches how scripts/sync_docs.py, scripts/regenerate_docs.py and friends already work in this repo, so the operational model is familiar.

Why each omitted field is omitted

The omissions table in the schema lists the four Backstage fields we still leave out (now that apiVersion and kind are included). Here is the rationale in narrative form — the part the table can't carry:

metadata.namespace

Namespaces partition the catalog so multiple teams can have a service named auth without colliding. One team, one catalog, four services with distinct names — no consumer for a namespace. Adding it later (say, when a second team contributes a service) is a single-line edit, so the cost of omitting it is zero and the cost of including it now is real (a fake "default" everyone has to type).

metadata.system, spec.system, spec.subcomponentOf

Backstage's System is a group of components that work together to deliver a capability ("checkout system"). Our catalog is flat: four services, no grouping. Modelling systems before the second one exists would invent structure to fit the descriptor instead of the other way round. When we do introduce systems (the most likely first split is "runtime services" vs "documentation pipeline"), the YAML can be extended without a renderer change because spec.system goes straight to a Backstage importer.

spec.providesApis / spec.consumesApis

This is the most tempting field to add, because it sounds like a free dependency graph. But the API contract is already published in two places: the public OpenAPI spec under public/reference/api/ and the per-service api-reference.html page inside each entity's spine. Repeating the API name as a descriptor field adds a third place where the name has to stay correct, and the descriptor field is the one a CI gate is least likely to check. Drift risk with no payoff — the YAML SoT story doesn't change that calculation; only a Backstage instance that turns providesApis into a navigable graph would.

metadata.labels

k8s-style key/value pairs are a query primitive: find all services with tier=critical. We don't have a catalog UI that runs label queries. The same filtering job is done today by entity-card__pills + the data-tags attribute on the hub tile, which the existing svc-filter chip group reads. Labels would be a parallel mechanism with no UI behind it.

Multiple typed links of the same type

Backstage allows any number of metadata.links entries with the same type; our renderer picks only the first type: code, type: data, and type: public entry. This isn't a Backstage omission — it's a rendering constraint to keep the entity card scannable. If you need multiple data references for one service, use the generic type: links bucket (rendered as a middot-separated row) or escalate to an RFC that redesigns the entity-card layout.

Consequences

  • Audit row 19 (catalog-info.yaml) moves to confirmed (8/10): the descriptor subset is ratified, the YAML SoT exists, the generator removes the drift trap, the pre-commit hook enforces it. The remaining 2/10 covers the fact that no Backstage importer actually consumes the YAML yet.
  • New services follow the add-service how-to: write the YAML, run make catalog-render, commit both files. The hand-edit-and-copy-from-the-nearest-existing-service path no longer applies.
  • Reviewers have two objective bars: make catalog-render-check must pass (no YAML/HTML drift), and any new field has to land first in the schema page before being added to a YAML file.
  • Field promotion (turning an optional field into a required one, or adding a brand-new field) is a non-trivial change to the contract. It should land as an ADR if it changes how downstream automation reads the descriptor; as a documented edit to the schema page + generator otherwise.
  • The path to a real Backstage instance is now docs-pipeline-shaped, not codebase-shaped: the YAML files are already in the right shape, and adopting Backstage means pointing an importer at them, not rewriting them.
