Explanation · System · Architectural narrative

Architectural narrative

The shape of this system told as a story: what it is, why it exists in this shape, what we deliberately did not build, what we gave up to get here, and what we would change next. Read this first — every other page on the portal assumes the model this page builds.

About 30 minutes to read One API, one database Target: 99.5% uptime, under 400 ms response

TL;DR

Study App is a spaced-repetition learning loop wrapped around one FastAPI service plus one relational database. The runtime is deliberately the simplest thing that lets the documents tell the truth — because the documents are the deliverable. Three load-bearing decisions earn their keep on the first incident (OpenAPI baseline diff in CI, idempotency at the router, structured logs with request IDs). Three deliberate omissions are the trade we accept (no async, no IdP, no horizontal scale story). Three gaps are next, in this order (Postgres swap, observability beyond logs, retroactive postmortems).

Story arc

01

The journey — six waypoints from gamble to current shape

How the system arrived at its current shape, compressed to six waypoints. Each waypoint is a moment where the model in the team's head shifted; the ADRs and refactors that followed each shift are the receipts.

  1. Start — «this is just a study tool». A solo project to practise spaced repetition. The first commit was a single-file FastAPI app with three endpoints and an SQLite file. No portal, no ADRs, no contract — just code.
  2. Pivot — «the artefact under review is the portal, not the binary». The project's purpose changed: the system became the practical work for a Senior Systems Analyst certification. From that point on, every architectural choice had to be defensible in a panel review, not merely shipped. Cost ratio inverted: the spec became heavier than the code.
  3. Discipline — «contracts are the most expensive thing in the repo». OpenAPI became a committed artefact that CI diffs against the live app. Idempotency moved from per-route ad-hoc handling to a body-hashed router concern. The four _shared/ conventions were extracted so specs link instead of restating.
  4. Maturity — «the portal needs a contract too». Diátaxis adopted as the IA framework (Procida, diataxis.fr). Backstage taxonomy borrowed for the catalog kinds (Software Catalog page). Eight authoring principles + a substance contract committed; make docs-check wired as the pre-commit gate.
  5. Honesty — «runbooks without postmortems are aspirational». Three real incidents written up end-to-end: OpenAPI baseline drift (2026-03-14), SQLite write-lock burst (2026-04-02), docs-fix search-index drift (2026-05-07). Each one changed at least one runbook or ADR.
  6. Now — «posture, not just shape». The portal is read for posture: tradeoffs, citations, measured numbers, production wisdom. The substance contract (five principles) is the working bar. Where the project goes next sits in § 09.

Explain

02

What this system is

Study App is a learning loop wrapped around a single HTTP API. A learner reads source material, distils it into a conspectus (cue sheet + dense paragraph + bullets), and the system schedules a sequence of retrieve passes — spaced repetition with a four-slot ladder. That loop is the product. Everything else exists because the loop needs storage, identity, scheduling, and a way for someone else's app to drive it. The runtime is one FastAPI service plus a relational database; the docs portal is a static site generated alongside it. There is no message bus, no auth service, no microservice mesh, no third-party identity provider on the request path. The full deployable inventory is in the service catalog; the per-resource HTTP surface is in the API reference catalog; the methodology this is built around is in ETR.

Explain

03

Why it exists in this shape

The system is built by one engineer working toward Senior Systems Analyst certification. That constraint is more load-bearing than any technical choice: every architectural decision is made to be defensible in a panel review, not just to ship. The consequence is a documentation discipline that would be excessive for a startup of this scale (a Diátaxis-ordered portal, 30+ ADRs, sectioned API specs, runbooks for every alert) and a runtime that is deliberately the simplest thing that lets those documents tell the truth. The portal is not decoration — it is the artefact under review. The API exists to give the portal something real to describe.

Two implications follow. First, the cost ratio is inverted: writing the spec is heavier than writing the code, and that is on purpose. Second, choices that would otherwise scream "premature" — strict OpenAPI governance, idempotency on every write, structured logs with request IDs from day one — are not premature here. They are the deliverable.

Explain

04

The shape — what you would draw on a whiteboard

Three boxes. Integrator (someone else's mobile or web app) talks HTTP to the api service over a versioned /api/v1 surface. The api service owns a single relational database — a containerized PostgreSQL 16 instance across dev/qa/prod (ADR 0037). That is the entire runtime topology. No queues, no caches yet, no read replicas, no separate workers.

Inside the api service, the layering is conventional and intentional. Routers (services/api/app/api/v1/*.py) handle HTTP and idempotency. Services (services/api/app/services/*.py) hold business rules and raise domain exceptions. Repositories (services/api/app/repositories/*.py) own SQLAlchemy. Each layer is independently testable — the test pyramid page explains why that boundary is the cheapest signal we have. The four cross-cutting concerns that every operation honours — auth, idempotency, the error envelope, and observability conventions — live once, in _shared/, and every spec links them rather than restating them.

Explain

05

What we deliberately did not build

Honesty about scope is the cheapest architectural quality there is. The system has no asynchronous processing — every state change is a synchronous HTTP write inside a single database transaction. There is no separate identity provider; the API key is the tenant, the composite (system_user_id, system_uuid) is the learner, and authorization collapses to ownership comparison (why we return 404, not 403). There is no horizontal sharding plan, no multi-region story, no rate-limit service — the in-process sliding window in services/api/app/core/security.py is enough for the single-tenant integrator the system is sized for. None of these omissions are accidents; each would be added under a specific ADR if the operating envelope changed.

Explain

06

Where the design earns its keep

The single highest-leverage decision was treating the OpenAPI baseline as a committed artefact that CI diffs against the live FastAPI app. Every contract drift becomes a PR comment. The second was making idempotency a body-hashed concern at the router layer rather than something the integrator has to think about; replay rules are now uniform across resources and the SQL schema for idempotency_keys is trivial. The third was structured logs with a request_id threaded from middleware; a one-shot grep in the portal's local Elasticsearch turns "user reports an error" into "show me the row" in under a minute. None of these are clever. All three pay for themselves on the first incident.

Tradeoffs

07

Tradeoffs — what this shape costs us

Four trade-offs accepted on purpose. Knowing them tells you where the model bends if you push it in the wrong direction. Shape and template borrowed from Tradeoff disclosure.

  • Spec-heavier-than-code. We gain a portal that defends itself in a panel review and a contract that CI can diff. We lose feature velocity — every new endpoint pays a fixed spec-authoring tax of ~3× the code time. We accept it because the deliverable is the portal, not the runtime; the day code velocity matters more than spec velocity, the constraint set has shifted enough to revisit.
  • Containerized Postgres, one shared instance. We gain multi-writer semantics (single-writer bottleneck is gone), production parity in dev, and a native JSONB path. We lose the zero-external-dependency make run we had under SQLite — docker compose is now the front door — and we pay an ongoing operational surface (backups, role separation, connection-pool tuning) that we haven't fully instrumented yet. We accept it because SQLite's single-writer ceiling is architectural, not hypothetical — any second concurrent writer serialises — and «zero-friction dev» stopped covering that risk (ADR 0037).
  • Synchronous-only writes. We gain end-to-end traceability of every state change inside a single transaction (request_id propagates cleanly; no eventual-consistency surprises). We lose any feature that would otherwise be naturally async (background scoring, batch re-scheduling). We accept it because the loop is short-lived per request and the integrator does not yet ask for async semantics; the day they do, a job table + a poller is the smallest thing that closes the gap — not a broker.
  • 404-not-403 on cross-tenant access. We gain a uniform unauthorised-access response that does not leak resource existence to attackers. We lose the ability to distinguish «owned but forbidden» from «does not exist» on the client side (why we return 404, not 403). We accept it because the integrator already knows what it owns; for them the distinction is noise, and for the attacker the noise is the point.

What's next

09

Where we would change it next

Three honest gaps, ordered by reader impact. (1) Postgres operational surface — the runtime swap to containerized PostgreSQL 16 landed (ADR 0037), but backup automation, role separation (DDL vs DML), and connection-pool re-measurement are still open follow-ups. (2) Observability beyond logs — Prometheus is wired into /metrics, but there is no live dashboard, no alert routing, and no traces. Section 14 of every operation spec promises metric names that no one is graphing yet. (3) Threat model — security ADRs name controls (rate limit, auth header, error envelope) but no single page names the adversary tiers we are designing against and the attacks we accept as out of scope. Without that, every security decision reads as ad-hoc. These three are the next architectural moves, in this order.

Retrospective

10

Would we do it again — knowing what we know now?

A retrospective in two columns — what we would repeat verbatim, what we would change on day one if we were starting today. Shape borrowed from Tradeoff disclosure §02 (last row).

Would repeat Would change
OpenAPI baseline as a committed artefact diffed in CI. Highest-leverage decision we made. Start with Postgres, not SQLite. The «just use SQLite for now» bet looked free; the postmortem of 2026-04-02 showed it had a measurable ceiling and a deferred-cost penalty, which is exactly what ADR 0037 ended up paying.
Idempotency at the router. Integrators do not have to think about replays. Wire /metrics to a dashboard from day one, not just expose the endpoint. Metrics nobody graphs are imaginary.
Structured logs with request_id threaded from middleware. Pays back on first incident. Write a threat model in week 1, even a thin one. Naming adversary tiers (script kiddies vs motivated attacker vs state actor) makes every later security ADR cite something instead of reading as ad-hoc.
Diátaxis as the IA framework (Procida, diataxis.fr). 95 % of doc portals do not have a quadrant discipline; the one that does feels qualitatively different to navigate. Stand up the postmortem corpus before the runbook corpus. We wrote 10 runbooks and 0 postmortems for the first three months; the asymmetry made the runbooks read as aspirational.

Sources we lean on
