Explanation Vision Horizon: 2027-07

Architecture vision — where this system is going in 12 months

The narrative explains how the system got its current shape and names the next three moves; this page holds the longer arc — the target state one year out, per theme, each with the trigger that starts it. Concrete work items live in the backlog; technology stances live on the tech radar.

The end-state in one paragraph

Twelve months out this is still one modular monolith, one Postgres, one host — deliberately. What changes is not the shape but the trustworthiness: burn-rate alerts that actually fire and page, a database that has survived a rehearsed restore, capacity numbers backed by a real load run, a user-facing Telegram surface exercising the API with real traffic, and analytics events flowing across the operational/analytical seam so product decisions stop being guesses. Explicitly not in this vision: microservices, Kubernetes, multi-region, a data warehouse. Each of those has a named trigger, and none of the triggers is close.

Four themes, each with its trigger

ThemeToday (honest)12-month targetFirst move
1 · Reliability becomes real SLOs written, rules committed, but no live alert has ever fired; restore untested; exhaustion runbook retired as theatre Burn-rate alerts live (backlog item-48); one rehearsed DR drill with a written report; the exhaustion runbook returns with the alerts; a production-readiness checklist gates any future "this is live" claim Wire the Prometheus rules to a notification channel — the smallest change that makes the system able to interrupt its operator
2 · The data seam opens Append-only logs exist; no analytics events, no product metrics; ad-hoc SQL answers analytical questions Write endpoints emit contracted events (event contract); a small set of product metrics (activation, review streak, retention cohort) computed from the logs, published on a page that is re-generated, not hand-edited Instantiate the first event contract for conspectus.reviewed — the highest-signal action in the product
3 · A real user surface API is complete but has no consumer besides tests; Telegram Mini App is a UI-kit prototype on a feature branch One end-to-end user flow (create → review → error log) shipped in the TMA against the real API, used by at least one person who is not the author Promote TMA from Trial by wiring its first screen to a live endpoint — product-direction decisions route through the backlog, not this page
4 · Security grows adversary tiers Controls exist (auth, rate limit, error envelope) and a STRIDE table per component, but no named adversary model; single DB role owns DDL Threat model names the adversary tiers and the accepted-out-of-scope attacks; DML/DDL role split done; encrypted backups as part of theme 1's drill Split the migration role — smallest change with the largest blast-radius reduction

Standing non-goals — and what would change them

  • No second service until a component needs an independent release cadence or failure domain — the triggers named in cross-service patterns.
  • No horizontal api scale until a load run shows a single instance saturating inside the SLO — which first requires theme 1's measurement discipline.
  • No warehouse until an analytical query measurably hurts the operational path — the three triggers in operational vs analytical data.

What would falsify this vision

A vision that cannot be wrong is decoration. This one is falsified if: the TMA direction is abandoned (theme 3 collapses and the API needs a different first consumer); real usage arrives faster than reliability work (themes reorder — 1 jumps ahead of everything); or a second regular contributor joins (the solo-maintainer constraints in the decision process relax, and the review model upgrades before any of the above). The quarterly radar review is also this page's review.
