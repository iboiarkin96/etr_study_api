Observability

Monitoring vs observability

Monitoring answers a fixed question: "is the error rate under 1%?". You define the question in advance and the dashboard tells you yes or no.

Observability is the property of being able to answer questions you did not plan for: "why is this one user seeing 500s on Tuesdays only?". A system is observable when, given a symptom, you can drill down to a cause without adding new code first.

Observability is not a product you install — it is what you get when metrics, logs, traces, and health endpoints are joined together by the same identifiers.

The four pillars in this project

Metrics — Monitoring
Numbers over time. Cheap to store, fast to query, perfect for dashboards and alerts. Bad for "why".
Logs — Logging
Structured events about specific things that happened. Great for "what did this exact request do?". Heavier to store and search.
Traces
A request's path through the system as a tree of spans, each with timing. Great for "where in the request did the time go?". Currently planned but not yet wired — see backlog item-48.
Health endpoints
/healthz says "the process is alive"; /readyz says "the process can serve traffic" (DB up, migrations done, dependencies reachable). Defined by ADR 0009.

The shared key — request_id

The single most useful thing the system does is to attach a request_id to every request at the edge and propagate it through every log line, metric label (when low-cardinality is acceptable), and trace span. With one ID you can pivot:

  1. Alert fires: "POST /conspectuses error rate above 5%."
  2. Open the dashboard: most failed requests in the last 10 minutes are tagged with route POST /conspectuses.
  3. Open Kibana: filter by route : "POST /api/v1/conspectuses" and level : "error"; pick one failing line; copy its request_id.
  4. Filter by that request_id alone: now you see every step that one request took, in order.

This loop — alert → dashboard → log search → single request — is what observability buys you.

"Is the service healthy right now?" — the 60-second check

  1. Open the main dashboard. Look at availability, p95 latency, error rate, saturation. Anything red?
  2. Hit /readyz. If it returns 503, the service knows it cannot serve traffic — read its body for the failed dependency.
  3. Open Kibana: filter level : ("error" or "critical") in the last 15 minutes. Spike? Pick one and follow the request_id.
  4. Check recent deploys (last 1 hour). A deploy + a spike is the most common pairing.

If the answer is "not healthy", switch to incident response.

What good observability feels like

  • You can find out why one specific request failed without redeploying.
  • The same identifier (request_id) appears in metrics labels (within reason), log lines, and trace spans.
  • Each alert links to a dashboard and a runbook; nobody guesses where to look.
  • Health endpoints distinguish "alive" from "ready"; orchestrators do not bounce a healthy pod just because a dependency is slow to start.
  • Adding a new endpoint adds metrics, logs, and a health contribution automatically — not as a checklist item the developer might forget.

Common anti-patterns

  • "We have logs" as a substitute for metrics. Counting log lines at query time is slow and expensive.
  • High-cardinality metric labels. Adding user_id to a Prometheus label kills the database. Cardinal data belongs in logs or traces.
  • Disconnected pillars. Logs without request_id, metrics without service tags, traces nobody reads.
  • Alert without runbook. The on-call person has to invent a procedure at 3 a.m.
  • Health that lies. /healthz returns 200 even when the database is unreachable. Use /readyz for dependency checks.
