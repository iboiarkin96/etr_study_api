Logging

What logs are for

If monitoring tells you that something is wrong, logs tell you what happened to whom. A good log line is the answer to a future question you have not asked yet. We use structured logs — every line is a JSON object with the same key set, so machines can search and filter them, not only humans.

The contract for the log shape and the local Elasticsearch + Kibana setup is defined in ADR 0023 — structured logging and local Elasticsearch. This page explains how to use the contract well.

The structured log contract

Every log line is one JSON object on one line. The required fields are:

FieldTypeMeaning
tsISO-8601 string in UTCWhen the event happened.
levelstringOne of debug, info, warning, error, critical.
msgstringShort human-readable summary. No PII, no full payloads.
loggerstringThe module name that wrote the line.
request_idstringUnique per HTTP request; lets you join all log lines from one request.
trace_idstring (optional)For cross-service requests; same value across services.
servicestringWhich service wrote it: api, portal, etc.
envstringlocal, dev, staging, or prod.

Anything beyond this is a free key — but be consistent. If you log a user identifier as user_id in one place, do not call it uid somewhere else.

Log levels — when to use which

debug
Fine-grained detail used during local development or when chasing a bug. Off in production by default.
info
The normal life of the request: "request started", "user created", "job finished". Should never alert.
warning
The system kept working, but something is suspicious: a retry succeeded after one failure, a deprecated endpoint was used, an input was unusual.
error
One operation failed for a clear reason: a 5xx was returned, a job exhausted its retries, a third-party call failed. Always include the exception type and a short message.
critical
The service or a major part of it cannot continue: database is unreachable, the configuration is invalid at startup. Should always page.

Rule of thumb: if a line at error level appears once a minute and nobody acts on it, it is not an error — it is noise. Lower it, fix the root cause, or aggregate.

Redaction — what must never reach the log

Logs are searchable, exported, and stored. Treat them as broadly visible. Never log:

  • Passwords, password hashes, password resets, or anything that looks like a token (Authorization headers, API keys, JWTs).
  • Full request or response bodies that contain user content.
  • Personal data: full names, emails, phone numbers, IP addresses (when in doubt, hash or truncate).
  • Internal stack traces sent to external sinks; keep them inside the service boundary.

If the logger receives one of these by accident, the redaction filter should mask it ("***"). The filter is a safety net, not a substitute for not logging the value in the first place.

Local Elasticsearch and Kibana

For local development the project ships an Elasticsearch + Kibana pair via the docker-compose file (see ADR 0023 for exact services). The flow is:

  1. The API writes JSON log lines to standard output.
  2. A log shipper reads stdout and forwards to Elasticsearch on the local network.
  3. Kibana provides the UI: search by request_id, filter by level, plot error rates over time.

To open Kibana on a fresh checkout: make logs-up, then visit the URL printed in the terminal. The default index pattern is study-app-*.

Useful Kibana queries

  • All log lines for one request: request_id : "REQ-abc123".
  • All errors in the last hour: level : ("error" or "critical") + time range = "Last 1 hour".
  • Slow endpoints: logger : "api.access" and duration_ms > 500.
  • One specific endpoint: route : "POST /api/v1/conspectuses".
  • Background jobs that failed: service : "api" and logger : "jobs.*" and level : "error".

Save searches you reuse — every saved search is documentation for the next person on call.

Common pitfalls

  • Logging the same event twice. Once at the framework layer and again in the handler. Pick one place.
  • Putting variable data in msg. Use a separate field instead, or full-text search becomes noisy.
  • Free-form objects. Use the same key names everywhere; otherwise filters break.
  • Missing request_id. Without it you cannot reconstruct what one user saw. Generate it at the edge and pass it through.

When logging itself fails

If logs are missing, late, or look corrupted, follow runbook 0003 — logging failing. The most common causes are: shipper container is down, Elasticsearch disk is full, or the log line is not valid JSON because of an unescaped string field.
