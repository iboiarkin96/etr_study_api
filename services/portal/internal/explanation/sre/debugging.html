Debugging

Debugging vs incident response

Incident response is about stopping pain fast. Debugging is about understanding the cause well enough to fix it for good. The two often overlap, but the goals differ: in an incident you may roll back without yet knowing why; in debugging you keep digging until you can explain what happened in one or two sentences.

This page is a generic playbook. For specific known failures, jump straight to a runbook.

The diagnostic loop

  1. Reproduce. A bug you cannot trigger is a bug you cannot fix.
  2. Form a hypothesis. "I think X is happening because Y."
  3. Predict. "If X is true, then I should see Z in the logs / metrics."
  4. Test cheaply. Look at the existing data first. Only add new logging or instrumentation if the existing data cannot answer.
  5. Update the hypothesis. Either the data confirms it, refines it, or kills it. Repeat.

The loop is the same shape as the scientific method. The discipline is to write the hypothesis down before looking at the data, so you do not unconsciously match patterns to whatever you see.

What to read first

For a production-ish issue, follow this order. Each step takes seconds and rules out whole categories of cause:

  1. Health endpoints. GET /healthz and GET /readyz. If readyz is failing, its body tells you which dependency is broken.
  2. The dashboard. Latency, error rate, traffic, saturation. Which of the four is unusual? Since when?
  3. Recent deploys / changes. The last 24 hours. A spike that lines up with a deploy is your prime suspect.
  4. Logs (errors only). Filter Kibana to level : ("error" or "critical") for the last 30 minutes. Look for repeated messages — they cluster better than unique ones.
  5. One failing request. Pick a single failing log line, copy its request_id, and read the full thread. This is where the "aha" usually lives.

Narrowing down — bisection

When you do not know where the problem is, divide the search space in half each step. Examples:

  • Time bisection. If today is broken and last week was fine, ask: was Wednesday broken? If yes, check Monday. Keep halving until you find the first bad day. Then look at what changed that day.
  • Endpoint bisection. If "the API is slow", ask: are all endpoints slow? If only POST /conspectuses, the cause is probably in that handler or its dependencies, not in the framework.
  • User bisection. If only some users see the bug, find the smallest pattern: a country, a feature flag, a payload shape, an old client version.
  • Code bisection. git bisect when you know a commit-range is the culprit and you can reproduce locally.

Cause vs trigger vs root cause

Trigger
The thing that happened just before the symptom. Usually a deploy, a config change, or a traffic spike.
Cause
The technical mechanism that turned the trigger into a failure. "The new code path read a null field."
Contributing factors (root causes)
Why the system allowed the cause to ship. Missing test, missing migration, brittle dependency. There is almost always more than one — that is normal.

Stopping the investigation at "trigger" is too shallow; the same trigger will hit again. Stopping at "cause" is better but still rare to prevent recurrence. The fixes that prevent recurrence usually live in the contributing factors layer.

When to escalate

Escalate before you are exhausted, not after. Triggers:

  • Two hours have passed and you have no working hypothesis.
  • You are about to take an action that is hard to reverse (drop a table, force a migration, push a hotfix without review).
  • Customer impact is growing and your current attempt is not shrinking it.
  • You are tired enough that you might miss something obvious.

Escalation is not failure — it is a tool. Use it.

After you fix it

  • Write down the cause and the fix in plain words. Even if it stays in your own notes, it pays back later.
  • If a runbook would have helped you find it faster, add it (or update an existing one).
  • If the fix changes a contract, an ADR, or a test rule, update those too. The fix is not done until the system knows about it.
  • If this was a SEV-1 or SEV-2, write the postmortem.
