SRE Explanation

Monitoring

Monitoring answers a fixed question you wrote down beforehand — «is the error rate under 1%?», «is p95 latency under 300 ms?». Observability answers questions you didn't plan for. This page is why we put monitoring first: four golden signals (latency, traffic, errors, saturation) on a Prometheus + Grafana + Alertmanager stack, with symptom-based alerts and a runbook for each.

What monitoring covers

Monitoring is the part of observability that uses numbers over time: how many requests per second, how many errors, how long a query takes, how full a queue is. The numbers are stored in a time-series database and shown on dashboards. Some of them have alerts attached, so they wake somebody up when they go wrong.

The wiring (which endpoints expose metrics, what the health probes return) is fixed by ADR 0009 — health, readiness, and observability. This page explains how we use those metrics in practice.

The four golden signals

Borrowed from the SRE Book — Monitoring Distributed Systems. If you only have time to watch four numbers per service, watch these — each glyph mirrors the shape the panel takes on the dashboard.

  1. Latency
    How long

    How long a successful request takes.

    Track
    p50, p95, p99 separately.
    Pitfall
    Averages hide the long tail.
  2. Traffic
    How much

    How much demand the service is handling.

    Track
    Requests/sec, jobs/min — per endpoint.
    Pitfall
    A drop to zero is often a bug, not a quiet day.
  3. Errors
    How many failed

    The rate of failed requests.

    Track
    4xx vs 5xx, split by endpoint.
    Pitfall
    Hiding errors behind a friendly status page.
  4. Saturation
    How full

    How “full” the system is.

    Track
    CPU, memory, DB pool, queue depth.
    Why
    Predicts the next outage.

Where the metrics come from

  • Application metrics — exposed by the API at /metrics in Prometheus format. Counters (requests, errors), histograms (latency), gauges (in-flight requests, queue depth).
  • Health endpoints — /healthz (process is alive) and /readyz (process can accept traffic) per ADR 0009. Used by container orchestrators and as a heartbeat.
  • System metrics — CPU, memory, disk, network — collected by the host or container runtime.
  • Database metrics — connection count, slow query log, replication lag (when relevant).

The collection and display stack

  1. Prometheus scrapes /metrics on a fixed interval and stores the samples.
  2. Grafana reads from Prometheus and shows the dashboards.
  3. Alertmanager reads alert rules evaluated by Prometheus and routes notifications.

The full Prometheus scrape config, the Grafana dashboard JSON, and the alert rule files are tracked in backlog item-48 — they will land in this page as a copy-pasteable reference once that item completes.

Dashboard layout

Each dashboard answers one question and uses the same vertical layout, so reading them feels consistent:

  1. Top — health summary. One row of single-stat panels: availability over the last hour, p95 latency, error rate, saturation flag.
  2. Middle — golden signals. Four time-series panels (latency, traffic, errors, saturation) for the same window.
  3. Bottom — drill-down. Per-endpoint breakdowns, slow query lists, recent deploys.

Time range and refresh interval are left at the dashboard defaults. Pick the same time range when you compare two dashboards — it makes correlation much easier.

Alerting taxonomy

Every alert must answer two questions: “is a user being hurt right now?” and “is there a clear next action?”. We split alerts into three classes:

Page (wakes you up)
Active customer pain. Examples: error rate above 5% for more than 5 minutes, fast burn of the SLO budget, the service is not ready for more than 2 minutes. Always has a runbook.
Ticket (handle next working day)
Slow drift, no immediate user pain. Examples: slow burn of the SLO budget, disk usage trending toward full in days, certificate expiring in two weeks.
Info (dashboard only)
Useful context but not actionable. Examples: deploy just happened, traffic spike from a known job. Never paged.

If an alert does not fit a class, redesign it. “Maybe a page” is the same thing as “noise” in three weeks.

Good vs bad alerts — short examples

  • Good: “Error rate of POST /conspectuses > 5% for 5 minutes” — symptom, time-bounded, clear runbook (0008).
  • Bad: “CPU > 80%” — a cause, not a symptom. The service might still be healthy. Pair with latency or saturation instead.
  • Good: “SLO burn-rate > 14× over 1 hour and > 6× over 6 hours” — multi-window, catches both fast and slow burns.
  • Bad: “There were any 5xx today” — fires constantly, hides real incidents.

When monitoring breaks

Two failure modes are easy to miss:

  • Silent dashboards. Prometheus stopped scraping. Dashboards still render, but the lines went flat. Follow runbook 0008 — observability scrape failing.
  • Quiet alerts. Alertmanager stopped delivering notifications. Use a “deadman” rule that fires every 5 minutes; if the on-call person stops receiving the deadman, alerting itself is broken.
