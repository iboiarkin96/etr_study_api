API Explanation

Why a Definition of Done

A spec is the contract between the analyst who designs an operation and the developer who implements it. The Definition of Done exists so that the moment of handover is unambiguous — and so that ambiguity surfaces as a lint failure rather than as a bug six weeks later.

The problem a DoD solves

Specifications drift. A first draft answers the obvious questions — request shape, response shape, happy path — and is silent about the awkward ones: what happens on retry, who is allowed to call this, what error code the client sees when a precondition fails, what gets logged. Silence reads as “use your judgement” to the developer, and judgement varies by person and by week.

The result is a class of bugs that no test ever catches because no test was written for them: an idempotency key that two endpoints interpret differently, an error envelope whose code field changes shape between operations, an alert that fires on a metric nobody emits. None of these are implementation mistakes. Each is a question the spec did not ask.

A Definition of Done is the answer: a fixed list of questions every spec must answer before it can be handed over. The cost of answering them is borne once, by the author, instead of repeatedly by the readers.

The contract of handover

The DoD describes what the developer is owed when picking up a spec — and, by symmetry, what the analyst owes the rest of the project.

  • Implementability without re-asking. A developer should be able to start coding without coming back to the analyst for clarifications on contract surface, error behaviour, or observability.
  • Testability without invention. Acceptance criteria, edge cases, and non-functional thresholds are concrete enough that QA can write tests directly against them — not against an imagined reading.
  • Traceability without archaeology. A reviewer should be able to follow a requirement back to the methodology that motivated it and forward to the ADR that constrains it, without grepping the repo.

This is why the checklist behind the DoD is mostly about presence, not quality: presence is what a linter can verify. Quality is what peer review is for. The two checks are deliberately separated.

Why a checklist — and why a long one

Checklists are out of fashion in some quarters: they feel bureaucratic, they slow you down, you can game them. Each criticism is true and each is beside the point. The DoD checklist exists for the same reason a pre-flight checklist exists: not because pilots forget how to fly, but because the cost of forgetting one item is high enough that systematic recall is cheaper than relying on attention.

The list is long — twenty items, plus quality gates — because that is the surface of an API operation. Trimming it would not make the surface smaller, only invisible. Each item exists because, in some previous spec, its absence caused a real problem: an error code missing from the catalog, an idempotency rule discovered in production, a metric that did not exist when the alert fired.

Why cross-cutting topics live in shared pages

Idempotency, authentication, pagination, the error envelope, versioning, observability conventions — every operation touches some of them, but the answer is the same for every operation. Inlining the rules in each spec would drift: one operation's idempotency section would say one thing, another's would say another, and there would be no canonical reference to break the tie.

So the DoD requires that each spec link to the shared page rather than restate it. The structural lint enforces the link; the content is maintained in one place. This is the same principle that puts shared types in a library rather than copy-pasted across services: one definition, many uses.

Lint vs review — two layers of done

Two distinct checks gate a spec, and they answer different questions.

  • The linter asks: is every required section present, is every cross-link resolvable, is the operationId unique, is the status badge valid? It runs in CI and is cheap to repeat. It cannot judge whether an acceptance criterion is meaningful.
  • The reviewer asks: does this spec, taken as a whole, describe an operation a developer could implement and a tester could test? They read the prose, follow the user story, sanity-check the edge cases, and form a judgement.

Conflating the two is a common failure mode. A linter-only DoD certifies that a spec is well-formed gibberish; a review-only DoD relies on the reviewer to remember twenty things at once. The DoD is split because the checks are different in kind.

Status as a contract about change

The status badge — draft, in review, approved, implemented, deprecated — is not decorative. It tells a reader what is safe to assume.

An approved spec means a developer can start work without expecting the contract to shift under them. An implemented spec means the OpenAPI document and the live service match the spec; subsequent changes must respect versioning rules. A deprecated spec means new clients should not adopt it; the lifecycle is winding down on a known schedule.

Statuses are coarse on purpose. Finer-grained states would invite arguments — “mostly approved”, “approved-pending-X” — and rob the badge of its load-bearing role: a single word that tells you whether to build on this spec or wait.

Common failure modes

  • “Done is when the lint passes.” The linter checks shape, not substance. A spec full of placeholders can be green. Review is non-optional.
  • “We'll fill in observability later.” Observability deferred is observability never. By the time the alert is needed, the metric does not exist, and the spec has moved to a different author.
  • “Out of scope: none.” Writing None. is allowed; leaving the section empty is not. The presence of an explicit “None.” is the signal that the author considered the question.
  • “This operation is too simple for a full spec.” Simple operations still touch auth, errors, versioning, and observability. The cost of a short spec is small. The cost of a missing one is paid by every future reader.
