🧾 SA reference · Traceability matrix

Traceability matrix — requirement → release

One row per requirement. Six columns: the requirement, the owning team on the hook, the ADR that froze the decision, the API operation that implements it, the test that proves it, and the runbook that responds when it breaks. Empty cells are not stylistic — they are the backlog. IEEE 830 / BABOK v3 shape, kept in lockstep with the catalog.

01

Why a traceability matrix exists at all

A requirement that ships without a test is a hope; a test without a requirement is dead code; an ADR without a requirement is opinion; an API without an ADR is drift. The matrix forces the system to be a coherent chain — broken links are the SA review's primary output.

  • One row per business requirement. Requirements without a row do not exist as far as the portal is concerned. The row is the contract.
  • Empty cell is a finding. A missing ADR, test, or runbook is filed as a ticket with the requirement owner. The audit walks this matrix row by row.
  • Bidirectional. The same chain reads up (ADR → which requirements drive me) and down (requirement → does anyone respond when I break).
  • Source of truth. The matrix is generated from one YAML file (see §05). The HTML you read is rendered from it; never edit cells by hand.
  • Audit cadence. Reviewed quarterly; broken rows generate work items, not commentary.

04

Column definitions

  • ID

    Stable, never reused. REQ-NNN assigned on first acceptance. Retired requirements stay in the matrix with status retired.

  • Requirement

    One sentence in active voice. «A learner can X» / «An author can Y». Acceptance criteria live in the linked spec, not in the cell.

  • Owner

    The single team accountable for the requirement end-to-end — paged on regressions, signs the audit, can say «yes / no» without escalation. Sourced from the YAML owner: field; aligns with Ownership. Empty owner is the most serious gap a row can carry — it means no one will notice when the chain breaks.

  • ADR

    The single decision that froze how this requirement is satisfied. If two ADRs apply, the matrix names the freshest; both link from the requirement spec.

  • API operation

    The HTTP operation, event topic, or pipeline step that implements the requirement. n/a is legal for process-only rows (e.g., REQ-011).

  • Test

    Path to the test that fails when the requirement breaks. End-to-end preferred; unit acceptable if the chain is independently covered.

  • Runbook

    The on-call playbook that fires when production violates the requirement. n/a only for build-time concerns (e.g., REQ-008 publish pipeline).

  • Status

    ok — full chain. wip — in flight. gap — at least one cell missing & ticketed. retired — kept for history.

05

Source — one YAML row per requirement

The matrix renders from governance/traceability/matrix.yaml. CI fails the build when a row references an ADR, API operation, test, or runbook that does not exist on disk — the matrix is structurally enforced, not just documented. 

- id:          REQ-005
summary:     "A learner can submit an attempt and receive a grade within 2 s."
owner:       "learning-team"
spec:        "governance/requirements/REQ-005.html"
adr:         "governance/adr/0021-rubric-pinning.html"
api:
  - operation: "POST /v1/attempts"
    page:      "services/api/reference/operations/post-v1-attempts.html"
test:
  - path:      "tests/assessment/attempt_perf_test.go"
    kind:      "e2e"
runbook:     "foundations/how-to/incidents/runbooks/RB-003-grade-latency.html"
status:      "ok"
last_review: "2026-05-24"
# Gaps appear as nil values; CI fails the build if status is "ok" with any gap.
06

Anti-patterns reviewers reject

  • Hand-edited cells. The HTML is generated. Manual edits silently desync from the YAML and fail the next regen pass.
  • «Status: ok» with empty cells. Either fill the cell or downgrade the status. CI fails on this mismatch.
  • Requirement as a paragraph. One sentence in active voice. Multi-sentence requirements get split into multiple rows.
  • Reusing a retired ID. IDs are permanent. Reusing REQ-007 after retiring it breaks every external reference (incidents, ADRs, postmortems).
  • Skipping the ADR. «We just built it» is not an option for a Tier-1 requirement. No ADR = the row is a gap, even with API + test + runbook filled.
