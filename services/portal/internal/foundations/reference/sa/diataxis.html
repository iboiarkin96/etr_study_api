Diátaxis

Every published page on this portal belongs to exactly one Diátaxis quadrant — Tutorial · How-to · Reference · Explanation — chosen by what the reader is doing, not by topic. Handbook pages (like this one) are the named exception: they compose lenses on purpose, with each section tagged so the composition is visible. This page is the canonical statement of the rule, how it maps to the repo, and the review playbook.

Stance

Our position

Every published page on this portal belongs to exactly one Diátaxis quadrant. A page that tries to satisfy two reader intents at once is split into two linked pages. The four quadrants are chosen by what the reader is doing, not by topic. Handbook / Foundations pages (like this one) are the single named exception: they compose lenses on purpose, with every block labeled.

Rules

The four quadrants

The canonical definitions. These four are exhaustive — there is no fifth quadrant.

QuadrantReader is…Definition & example
📚 Tutorial learning «Walk me through it the first time.» Linear, guaranteed to work end-to-end. Example: Your first API checklist run.
🔧 How-to doing «How do I accomplish X?» Goal-oriented, assumes context. Example: Restart the API service; Run an exploratory testing session.
📖 Reference looking up «What is the contract / shape / value?» Dry, exhaustive, no narrative. Example: OpenAPI spec; Spine matrix; DB-table schema.
💡 Explanation understanding «Why is it like this?» Background, trade-offs, decisions. Example: Why we chose event-sourcing; Test pyramid & layer ownership.

Rationale

Why we picked Diátaxis

Short reasoning for the choice. Full comparison of doc methodologies (Diátaxis vs DITA, Information Mapping, Heroku model, Atlassian role split) lives in /explanation/methodology.html — link out, not duplicate down here.

Alternatives considered:

  • GoodDocs («Five document types») — one too many to remember.
  • Heroku's model (conceptual / reference / how-to) — no dedicated «Tutorial» bucket.
  • Atlassian's role-based split — mixes audience with intent.

Diátaxis was the cleanest fit:

  • Closed taxonomy. Four quadrants — no «Getting started» or «FAQ» bucket that quickly becomes a junk drawer.
  • Reader-intent, not topic. The same topic legitimately appears in all four quadrants (a tutorial about the API, a how-to for the API, the API reference, an explanation of why the API is shaped this way).
  • Industry adoption. Django docs, Cloudflare Workers, Microsoft Learn, GitLab handbook, Stripe docs — all use Diátaxis (some explicitly, some by structural convergence). Hiring a writer who already knows the model is realistic.
  • Pairs with CI and local pre-commit. Every page declares a quadrant in its front-matter or body class; CI can audit the mix per service — are we over-indexed on Reference and under-indexed on Tutorials?

Rules

How quadrants map to the repo

Quadrants aren't a wiki convention — they're encoded at three concrete layers in the repo, so a page's intent is visible from the file tree alone. A reviewer who sees a 🔧 icon on a page full of «why we chose X» rejects the PR; the icon is the contract.

Layer 1 · per service — inside services/<name>/

Each service owns its own quadrant mix. Its catalog-info.yaml declares a nav: block that buckets the service's pages into 📚 / 🔧 / 📖 / 💡. CI counts the mix per service and flags imbalances (e.g. 40 References, 0 Tutorials). See IA contract for the exact nav: schema.

What can live in each per-service bucket:

🗂️ service root
The contract layer — Backstage envelope, service hub, dependencies, changelog, on-call rotation. Never inside docs/.
📚 tutorials/
«Your first …» walkthroughs for a newcomer. Fixed input, one happy path, one verifiable success step. No branching.
🔧 how-to/
Goal-driven recipes the reader already knows they need. Runbooks (one per alert) and dated postmortems live in subfolders here.
📖 reference/
Dry, dense, dated lookup material — OpenAPI & AsyncAPI specs, per-operation pages, events, config, metrics, SLOs, error codes. No «when to use» prose.
💡 explanation/
The «why» layer — architecture, topology, data model, concurrency, security, tradeoffs. ADRs live in cross-cutting governance, not here.
services/checkout/
├── catalog-info.yaml      # Backstage envelope + nav: buckets
├── index.html
├── on-call.html
└── docs/
    ├── tutorials/         # 📚
    │   └── your-first-checkout.html
    ├── how-to/            # 🔧
    │   ├── rotate-secret.html
    │   ├── runbooks/      # one per alert
    │   └── incidents/postmortems/
    ├── reference/         # 📖
    │   ├── openapi.yaml
    │   ├── operations/    # one per HTTP op
    │   ├── events/
    │   ├── metrics.html
    │   └── slos.html
    └── explanation/       # 💡
        ├── architecture.html
        ├── data-model.html
        └── tradeoffs.html

Layer 2 · cross-cutting — shared under foundations/

Material that ≥ 2 services depend on is promoted to foundations/, one folder per quadrant. Tutorials live one level up in onboarding/ because a first-time reader starts at the repo root, not in a service.

Rule of promotion: if a page would otherwise be duplicated across two services, lift it into foundations/ instead of forking. Service pages then link up to the foundation, never across to each other.

internal/
├── onboarding/            # 📚 tutorials
└── foundations/
    ├── how-to/            # 🔧
    ├── reference/         # 📖
    └── explanation/       # 💡

Layer 3 · UI surface — sidebar, templates, card kickers

The same four icons surface everywhere the reader looks: in sidebar section labels, on scaffold filenames under foundations/reference/sa/templates/, and as kickers on every index card. One icon, one meaning, no aliases.

A page whose icon doesn't match its body is a review-blocker — it's a mis-classification, not a styling nit. Authors pick the quadrant by picking the template, before they write a line of prose.

foundations/reference/sa/templates/
├── 📚 tutorial.html
├── 🔧 how-to.html
├── 📖 reference.html
├── 💡 explanation.html
├── adr.html               # governance/ — see below
└── rfc.html               # governance/ — see below

Two axes — disk layout and sidebar are decoupled by design

The file tree above is the on-disk axis: where a page lives for git, refactoring, and ownership. The sidebar is a separate reader axis: the story we tell a person who lands on the portal. They mostly mirror each other, but they don't have to.

disk axis
Stable. Edited only via git mv. Driven by ownership, blast radius, and refactor cost.
sidebar axis
Editorial. Edited in nav-tree-internal.json. Driven by the reader's journey — what to promote, group, or hide.

Industry standard. GitLab handbook, Backstage TechDocs, Stripe Docs, and MkDocs Material all decouple — folder tree says where files live, a nav: declaration says how to present them.

Default aligned, diverge with intent. Keep the sidebar in sync with disk by default; re-group only when the reader story needs it. To stop the two axes from drifting, one CI gate is mandatory: every published .html appears in the sidebar exactly once — or in an explicit hidden-list with a reason. The script is check_sidebar_coverage.py.

Rules

Two genres outside the four quadrants

Diátaxis is a closed taxonomy — and that's its strength. So when a genre genuinely doesn't fit, we don't smuggle it into Reference or Explanation. We carve out exactly two cross-cutting folders, with a written rationale, and keep the four quadrants pure.

governance/
ADRs and RFCs. Prescriptive — they decide; they don't describe. Diátaxis covers the descriptive surface only, so decisions get their own home.
team/
Role hubs and practice radars. The reader is a person growing into a role — a deliberate hybrid of tutorial + how-to + explanation that would fracture if forced into one quadrant.

Apply

Author & reviewer playbook

The work you do on a real PR. Each step has a clear «who / when».

  • Before writing — pick a template. Copy a scaffold from foundations/reference/sa/templates/. The template name commits you to a quadrant before you've written a line of prose.
  • During review — check icon ↔ body. If a 🔧 page is full of «why we chose X», reject the PR. Mis-classification is a review blocker, not a styling nit.
  • Before merge — run check_sidebar_coverage.py. Every published .html must appear in the sidebar exactly once (or sit in the explicit hidden-list with a reason).
  • Quarterly — audit the per-service mix. A service with 0 Tutorials and 40 Reference pages is missing an onboarding path. Count quadrants, treat imbalances as backlog items.
  • When you find yourself wanting a fifth bucket — stop. «Getting started» is a Tutorial. «FAQ» is either Explanation or How-to, split it. Don't invent a junk drawer.

Rationale

Tradeoffs we accept

Diátaxis is a prescriptive framework. The limitations listed below are costs we explicitly choose to bear.

WinCost
Decision at write-time. The hardest IA decision (which quadrant) is forced upfront. Pages can't quietly drift into «explainer + how-to + reference» mush — a reviewer can reject «doesn't fit one quadrant» without subjective debate. Casual readers must learn the model. A new joiner who has never seen Diátaxis has to absorb a 4-axis mental model before navigating the portal feels natural. We mitigate with universal icons, but the friction is real.
Auditable mix. Each service's quadrant mix is countable. A service with 0 Tutorials and 40 Reference pages is a real signal — usually means «no onboarding path». CI can fail on the imbalance. Some pages legitimately straddle. A «what's new in v2» release note has Reference (changelog), Explanation (rationale), and How-to (migration) elements. We split it into three linked pages — cleaner long-term, more friction short-term.
Industry-portable. Hires from Stripe, GitLab, Cloudflare arrive already fluent in the model. Onboarding cost for new authors drops; lateral moves stay readable. Closed taxonomy can pinch. Genres that don't fit (postmortems, status reports, ADRs) need their own buckets outside the four. We carve out governance/ and team/ explicitly — not silently.

Apply

Anti-patterns to catch in PR review

The most common drifts during PR review. Catch them before the merge, not in next quarter's audit.

  • How-to that drifts into Explanation. «How to restart the API … because of the way our process supervisor handles signals.» The rationale belongs in a sibling Explanation page; the how-to should link to it.
  • Reference that teaches. A field table that opens with «Let's see what a User looks like…». Move the teaching to a Tutorial; keep the table dry.
  • Tutorial that lists every flag. «Step 4: run the command with all 27 flags.» Link to Reference; the tutorial picks the 2-3 flags a first-time reader needs.
  • Explanation that prescribes. «Here's why we use X — and you must use X.» Prescription belongs in an ADR / RFC under governance/, with the Explanation linking to it.
  • «Getting started» as a quadrant. «Getting started» is a Tutorial. Name it that. Don't invent a fifth bucket that conflates Tutorial + How-to.
