Nav-tree schema

Authoritative shape of the two files that drive sidebar navigation in the internal portal: nav-tree-internal.json (primary manifest) and internal-sidebar.js (legacy projection). This page is the contract. For the procedure of adding a new entry, see the Update sidebar how-to.

The two files

FileRoleConsumed byStatus
services/frontend/portal/assets_v2/ui-kit/mocks/nav-tree-internal.json Primary manifest (UI Kit v3). Sidebar Web Component ui-kit/components/sidebar.js; pages declare nav source via the data-nav-src attribute on the <aside>. Canonical.
services/frontend/portal/assets/internal-sidebar.js Legacy projection. Still consumed by some shells until full v2 retirement. Lock-step with the JSON; planned for removal in Phase 2.

JSON entry — leaf node

{
  "id": "handbook-templates-adr",
  "label": "ADR",
  "href": "/services/portal/internal/handbook/templates/adr.html"
      }
FieldTypeRequiredRules
idstringyesDot-free, kebab-case, prefixed with the section path. Must be unique within the manifest.
labelstringyesSentence case with acronym preservation (API, AsyncAPI, SRE, gRPC, REST, SQL, ADR, RFC, QA, CI).
hrefstringyes (for leaves)Absolute from the repo root, must start with /services/portal/internal/, must end with .html.
iconstringnoOptional emoji or short label tag rendered before the link.

JSON entry — group node

{
  "id": "handbook-templates",
  "label": "Templates",
  "children": [
    { "id": "handbook-templates-adr", "label": "ADR", "href": "/services/portal/internal/handbook/templates/adr.html" },
    { "id": "handbook-templates-rfc", "label": "RFC", "href": "/services/portal/internal/handbook/templates/rfc.html" }
  ]
      }

A group node has children instead of href; children may themselves be leaves or groups. Nesting is unbounded but the sidebar component visually flattens beyond two levels — keep three-deep as the practical maximum.

JS entry — legacy shape

{ label: "ADR", path: "internal/handbook/templates/adr.html" }
FieldTypeRequiredRules
labelstringyesSame casing rules as the JSON manifest.
pathstringyes (for leaves)Relative to services/portal/, no leading slash.
childrenarrayyes (for groups)Mirrors the JSON manifest's children.

Placement map

Where a new entry belongs by document kind. Authors who already know what they wrote can jump straight to the correct subtree without scanning the whole manifest.

Document kindSubtree
API operation specreference → entity hub → Operations
ADRgovernance → ADRs → topical cluster (Documentation, Security, API contract, …)
RFCgovernance → RFCs
Runbookhow-to → Incidents → Runbooks
Postmortemhow-to → Incidents → Postmortems
Tutorialtutorials → role group
How-tohow-to → topic group
Reference page (other)reference → relevant block (api, data, front, qa, manager, uml)
Handbook / Foundationshandbook or foundations → matching sub-group

Invariants enforced by tooling

  • Valid JSON. python3 -m json.tool must parse the manifest. make docs-check runs this implicitly.
  • Unique id. Duplicate IDs cause the sidebar component to drop entries silently.
  • Reachable href. check-asset-refs pre-commit hook rejects entries whose target file is missing.
  • Acronym casing preserved. Drift is the most common navigation bug; spot-check every PR that touches the manifest.
  • JS ↔ JSON parity. Both files must list the same leaves until the legacy file is retired. make docs-check covers the regression on the JS side.

Anti-patterns

  • Editing the JS file without editing the JSON. Pages on v2 shells will not see the new entry.
  • Relative href in the JSON manifest. Always absolute (/services/portal/internal/…).
  • Acronym lower-casing ("label": "Adr") — fails review even if CI passes.
  • Mixing href and children on the same node. A node is either a leaf or a group, not both.
  • Trailing commas after the last array / object element — common after a deletion.
