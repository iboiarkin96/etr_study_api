Template API

Specification template — operation

The golden template for any internal operation specification under services/portal/internal/api/<resource>/operations/. Copy this file, rename to <method>-<path-with-dashes>.html, then fill every required section in order.

METHOD /api/v1/<path> Status: template v1 · Internal Owner: Ivan Boyarkin operationId: <operationId> Reviewed 2026-04-29 Spec template v1

How to use this template

Required vs optional. Each section is tagged with a badge in its heading and with a data-spec-required="true|false" attribute on its <section> wrapper. Required sections must exist and be non-empty — the lint blocks the PR otherwise. Optional sections may be removed for trivial operations (e.g. a read endpoint usually has no State transitions) — never reorder, only remove.

Authoring rules, lint expectations, and the full Definition of Done for an operation spec are tracked separately in _shared/spec-definition-of-done.html. A spec is considered ready for engineering only when every required section is non-empty and the DoD checklist passes.

Source-of-truth note: this template is the structural contract for analyst-spec pages. Cross-cutting topics (idempotency, error envelope, auth, pagination, versioning, observability conventions, OpenAPI authoring guide) live once under _shared/ — link to anchors there instead of repeating their content per operation.

Lint checklist (block-merge gate)

This block lives only on the template page. Real specs do not duplicate it — the linter (scripts/spec_lint.py) reads data-spec-section / data-spec-required attributes from the spec and applies the rules below.

  • Body attributes: data-spec-status, data-spec-owner, data-spec-operation-id are set on <body>.
  • Status pill in the page-head chip row matches data-spec-status.
  • Every required section exists as <section data-spec-section="…" data-spec-required="true"> with non-empty content (no lone TODO, no empty placeholder).
  • Field tables in sections 7 / 8 are present with all required columns.
  • At least one request example and one response example per success status.
  • Every error code/key in section 12 is also registered in _shared/error-catalog.html (consistency check).
  • Sections 17 (Open questions) and 18 (Out of scope) are non-empty: either a list, or the literal word None.
  • OpenAPI operationId in metadata strip exists in services/portal/public/reference/api/openapi-baseline.json.
  • History section present with at least one entry.
