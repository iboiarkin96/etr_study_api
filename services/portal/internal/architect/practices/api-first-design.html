API-First Design — contract before code

Write, review, and agree the API contract before any implementation begins so teams can work in parallel and integration is predictable.

TL;DR

API-First Design means the OpenAPI 3.1 (or AsyncAPI) contract is written, reviewed, and agreed as the first deliverable of any new API — before a single line of implementation is written. The contract becomes the primary design artifact from which SDKs, mock servers, contract tests, and documentation are generated. Drift between the running code and the spec is caught by contract tests in CI. Teams that adopt API-First reduce integration time by 60–80% because consumers can develop against the mock server while the implementation is in progress.

What it is

API-First Design is a development practice in which the API contract is the first deliverable. Before any implementation, the team writes an OpenAPI 3.1 spec (for REST) or AsyncAPI 2.x spec (for event-driven). The spec is reviewed, agreed, and merged as a PR. From the spec, the following are generated automatically:

  • Mock server (Prism, WireMock) — lets consumers develop against the API before implementation exists.
  • Client SDKs — generated from the spec using OpenAPI Generator, reducing consumer integration time.
  • Contract tests — Dredd or Schemathesis tests that verify the running implementation matches the spec. Fail CI if the code drifts from the contract.
  • Documentation — generated from the spec and published as a reference page on every release.

The spec linting tool (Spectral) enforces API design guidelines: consistent naming, pagination on collection endpoints, standard error envelope, versioning policy.

Why we need it

Teams that code first and document later accumulate API debt that makes every new integration expensive. Without a contract, every consumer reads the source code or sends trial HTTP requests to understand the API. Undocumented breaking changes break consumers silently. Mobile and backend teams cannot work in parallel — the mobile team is blocked waiting for the backend to be "working enough" to integrate against.

API-First eliminates these failure modes. The mock server enables parallel development from day one. The contract test makes breaking changes visible immediately. The generated documentation is always current. At startup speed, the ability for mobile, web, and backend teams to work in parallel is one of the highest-leverage architectural decisions an architect can make.

How others do it

Stripe

Stripe's entire API is designed API-First: the OpenAPI spec is the authoritative source from which the documentation, SDK generators, and changelog are derived. Stripe maintains SDKs for 10+ languages, all generated from the spec. Their API design guide (published publicly) defines conventions for naming, pagination, idempotency keys, and error envelopes. New Stripe API endpoints require an API design review before implementation begins — the spec review is the gate, not the code review.

What we take: The spec review IS the architecture review for API endpoints. Code review validates implementation quality; spec review validates API design decisions. Run them separately.

Source: Stripe — Payment API design

Zalando

Zalando published the Zalando RESTful API Guidelines — 64 rules enforced by the Zally linter on every API spec change. All Zalando APIs are designed API-First with the spec as the contract. New APIs go through a mandatory "API Design Review" before development begins, run by an API guild of 5–10 senior engineers. The spec must pass all Zally rules before the review proceeds.

What we take: An API design review process with published guidelines and automated linting reduces API inconsistency across teams by 80–90%. The linter is the scalable part; the human review is for the decisions the linter cannot make.

Source: Zalando RESTful API Guidelines

Figma

Figma uses API-First for both their public plugin API and internal service APIs. Their public API was designed with a spec-first approach that enabled them to publish comprehensive TypeScript types alongside the REST documentation. Internal APIs follow the same pattern — the spec is the contract that mobile, web, and plugin teams all depend on. Figma's API versioning strategy (documented in their engineering blog) relies on the spec being the source of truth for deprecation timelines.

What we take: Generate TypeScript types (or Python stubs) from the OpenAPI spec and publish them alongside the docs. Consumers who have types catch API misuse at compile time rather than at runtime.

Source: Figma Engineering — Infrastructure

Anatomy

OpenAPI 3.1 Spec

The source of truth for the API. Stored in the repo alongside the service code. Must include: all endpoints, request/response schemas, error codes, authentication requirements, and pagination conventions. Generated artifacts (mock server, SDK, docs) are derived from this file. The spec is the contract — it is not documentation.

Spectral Linting Rules

An automated ruleset (30–60 rules) that enforces API design standards: consistent naming (camelCase fields, kebab-case paths), required pagination on collection endpoints, standard error envelope, versioning in the base path. Run in CI on every spec change. New rules require a PR and a guild review. Start with the OpenAPI recommended rules; add custom rules as patterns emerge.

Mock Server

A server generated from the spec (Prism for OpenAPI, Microcks for AsyncAPI) that returns example responses matching the schema. Deployed to a shared dev environment within 30 minutes of the spec being merged. Consumer teams develop against the mock in parallel with implementation. Mock fidelity target: 95% of use cases covered by spec examples.

Contract Test Gate

A CI job (Dredd or Schemathesis) that hits the running implementation with requests from the spec and verifies the responses match the schema. Runs on every PR that touches the implementation. A failing contract test blocks the PR. No exceptions — a contract test failure means the spec and the code have diverged, which is always a bug.

Common pitfalls

Pitfall Why it happens What to do instead
Spec generated from code rather than written first The team uses a framework annotation-based spec generator (FastAPI auto-docs, Springfox). The spec is generated from the implementation, not the other way round. The API design is driven by implementation convenience, not consumer needs. Write the spec by hand first. Use the annotation-based generator as a validation tool — run it and compare to the handwritten spec to catch divergence. The spec defines the desired API; the generator verifies the implementation matches it.
Contract tests that only test happy paths The contract test suite only covers 2xx responses. Error cases (validation failures, not-found, auth errors) are untested. Consumers encounter undocumented error shapes in production. Include at least one test case per documented error code. Define error response examples in the spec. Use Schemathesis (property-based testing against the spec) to automatically generate edge cases beyond the explicit test scenarios.
API design review happens too late (post-implementation) The spec review is scheduled after the implementation is "mostly done" because it feels inefficient to review the spec before knowing if the implementation is feasible. Design feedback arrives when changes are expensive to make. Run the spec review before implementation begins, with a PoC at most. The spec review is cheap — a 30-minute async review of a YAML file. Implementation feedback is expensive. Move the gate earlier.
