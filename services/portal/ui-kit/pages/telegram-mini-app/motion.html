3 motion personalities below — the same interactions rendered at three different levels of expressiveness. Pick the direction, we productionise.

Motion personality · 3 takes

Variant A · Subtle

Vercel-tier restraint

Fast (120ms), out-easing everywhere. Nothing bounces. Every motion is functional — it exists to show state change, not to entertain. Feels crisp and professional.

hover · 120 ms out
Variant B · Expressive

Arc-tier spring

Med (240ms) with spring easing on every interactive. Slight over-shoot on lift. Ring count-up spring. Toasts slide with a light bounce. Feels alive without being cartoonish.

hover · 400 ms spring
Variant C · Playful

Duolingo-tier delight

Lazy (600ms) durations, celebration Lottie on milestones, mascot bounces, sound cues (opt-in). Every success is a mini-party. Best for retention psychology.

pulse · 1.4 s infinite

Implementation stack

  • CSS transitions for hover / focus / tap states — cheapest, GPU-accelerated.
  • Framer Motion for gesture-driven (drag, swipe, dismiss), shared-element, and orchestrated sequences.
  • View Transitions API for route changes on TanStack Router. 90% browser support in 2026 — polyfill fallback via cross-fade.
  • Lottie only for milestones (streak 7 / 30 / 100 / 365). Cached on disk; ~15 KB gz each.
  • Auto Animate for list add/remove — zero-config sensible defaults.

Rules

  1. Never hard-code milliseconds. Always route through a duration token.
  2. Motion on outcomes, not opens. Opening a screen doesn't celebrate; completing a review does.
  3. One personality per app. Don't mix subtle + playful on different screens — pick one variant globally.
  4. Respect prefers-reduced-motion. Collapse everything to instant. Never bypass this for a «cool» animation.
  5. Motion is a fallback for feedback, not a replacement. Every animation should have a static equivalent (text, colour, badge) that conveys the same info.
