Purpose

The Schedule screen is the antidote to «open the app · surprise: 15 reviews today». It shows the shape of the next 14 days so the user can pre-empt a spike — «Tuesday's heavy, snooze two now». In v1, snoozing happens from Today or Conspectus, not here — this is a look-ahead.

At a glance

Route
/schedule
Entry points
Tap on the schedule strip in Today · direct nav-menu link.
MainButton
Hidden — no primary action on this screen.
Back behaviour
SDK BackButton → Today (or previous screen).
Topbar action
Sync — force refetch. Auto-refreshes on route entry.

Data · API endpoints

14-day preview
  • GET /api/v1/schedule/preview

Returns 14 buckets: { date, review_count, est_minutes, is_peak }. Server does the peak-day heuristic — client only paints.

Peak day drill-down
  • GET /api/v1/conspectuses?due_on=YYYY-MM-DD

Tapping a peak-day cell opens a list of notes due that day — reuses the Today cell layout.

Components used

  • .tma-topbar
  • .tma-section
  • .tma-schedule
  • .tma-cell

States

Loading

14 grey bars shimmer. Peak section shows two skeleton cells.

Loaded · normal load

Peak section says «Load looks balanced». No warn cell.

Loaded · peak detected

Warn cell surfaces the peak day + suggested snooze count.

Empty · no scheduled work

All bars --empty. Peak section becomes an .tma-empty block.

Error

Toast «Couldn't sync» · retry chip in the peak section.

Interactions

  • Tap a bar → open the Today-style list of that day's due conspectuses.
  • Tap topbar Sync → force refetch · haptic impactOccurred('medium').
  • Long-press a bar → tooltip «N reviews · ~M min» (v2).
