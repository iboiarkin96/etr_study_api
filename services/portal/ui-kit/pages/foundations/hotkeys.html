Foundations

Hotkeys

Keyboard first; mouse is a fallback. The kit honours a small, fixed map — three groups, no surprises, no chord conflicts with the OS.

Contract

  1. Single-key shortcuts only fire when no input is focused. Type a slash inside a textarea and you get a slash, not the search palette.
  2. Chord shortcuts use g as the leader (Gmail-style). g s = "go to sidebar", g h = "go home".
  3. Modifier shortcuts use ⌘/Ctrl only where they extend an existing OS metaphor (⌘+K = palette, ⌘+/ = help).
  4. No conflicts with browser defaults. We never rebind ⌘+T, ⌘+W, ⌘+R, etc.

The map

Navigation

KeysAction
/Focus topbar search
⌘+KOpen command palette (when wired)
g→hGo to portal home
g→sFocus sidebar (opens drawer on mobile)
g→tFocus right-rail TOC
EscClose any open overlay (drawer, modal, toc-fab panel)

View & theme

KeysAction
tToggle theme (light ↔ dark) — same as topbar flashlight
[Collapse / expand sidebar
]Collapse / expand TOC right rail
zToggle reading mode (hides chrome — sidebar + TOC + topbar fades to minimal)

Help

KeysAction
?Open this hotkeys map as a modal cheat-sheet
⌘+/Same as ?; the OS-native modifier version

Implementation notes

  • Each binding lives next to the component that owns the action — theme toggle owns t, sidebar owns [, etc. There is no central key-router file.
  • Bindings register on mountX(); unregister is unnecessary because mount is idempotent and the page lifetime is page-load to nav.
  • Chord bindings share a tiny utility (ui-kit/components/hotkey-chord.js, planned) that buffers the leader and times out after 1.2 s.
  • A binding that has no visible affordance — no tooltip, no menu hint — is a bug. Every shortcut shows up in ?.

Status

This page is the contract. The ? cheat-sheet modal and the chord runtime are tracked in the Backlog cockpit — until they ship, the contract is aspirational and the visible bindings are only t, Esc, / on hover-supporting devices. Authoring this page first lets us land the rest piecemeal without churning the canon.
