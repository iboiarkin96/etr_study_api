RFC 0042: Per-service runbook canon — one section order across all services

Pin every runbook in the portal to the same section order — Symptom, First action, Triage, Recovery, Exit criteria, Escalation, Follow-up — and enforce that order with a structural canon test in CI. Today the order drifts between services (Recovery and Triage swap places, some pages have no Exit criteria at all), and an on-call engineer paying the cost of remembering which service uses which order.

Goals

  • One reading order, every runbook. An on-call engineer who has used a runbook in one service can use one in another service without re-learning the layout.
  • Drift caught at commit time. A structural canon test refuses commits that introduce a runbook with sections out of order, or missing one of the seven canonical sections.
  • No bespoke content surgery. Existing runbooks reach canon through a one-time fix-up sweep; from there forward the gate prevents regression.

Background

We have ~25 runbooks across services/api/runbooks/, services/portal/runbooks/, services/monitoring/runbooks/, and handbook/sre/runbooks/. A scan of section IDs in commit 9d3a found three distinct orderings: Symptom → Triage → Recovery (api, 9 pages), Symptom → Recovery → Triage (sre, 6 pages), and a free-form layout (portal, 4 pages). Three more (monitoring) have no «Exit criteria» section at all — recovery just ends and the page assumes you know when to declare the incident closed.

The cost is real but distributed: every postmortem from the last two quarters mentions at least once «I went to the runbook and couldn't find the rollback step» or «the runbook ended without telling me when to declare clear». A canon is the cheapest possible fix.

Detailed design

The seven canonical sections, in order:

  1. Symptom. What the on-call engineer is seeing. One paragraph, no diagnosis.
  2. First action. The single thing to do in the first 60 seconds. Mitigation, not investigation.
  3. Triage. Three to five diagnostic queries / dashboards / log filters that narrow the cause.
  4. Recovery. The step-by-step actions to restore service. Numbered list. Each step has an expected outcome.
  5. Exit criteria. How the engineer knows the incident is over. Concrete signals — not «things look fine».
  6. Escalation. Who to page if Recovery stalls, and at what time threshold.
  7. Follow-up. Postmortem owner, ticket to file, runbook-edit owner if a gap was found during the response.

Each section maps to an <h2 id="…-h"> with a fixed id token: symptom-h, first-action-h, triage-h, recovery-h, exit-h, escalation-h, follow-up-h. The canon test walks every runbook under the four runbook roots, asserts each id is present, in order. Templates for new runbooks ship with all seven sections pre-stubbed.

Implementation

  1. Phase 1 — canon test. Add tests/visual/test_runbook_canon.py (already shipped under BL-047) — extended to enforce the seven-section order across the four runbook roots. Failing the test does not block CI yet (marked xfail).
  2. Phase 2 — fix-up sweep. One PR per service folder: reorder sections, add missing «Exit criteria» where absent, normalise heading IDs. Each PR includes the canon test going xfail → pass for that service.
  3. Phase 3 — hard gate. Remove the xfail markers; canon test becomes a required CI gate. New runbooks must start from the template.
  4. Phase 4 — template lockdown. Add handbook/sa/templates/runbook.html and the UI Kit twin doc-runbook.html to the BL-047 visual corpus so template structure is itself contract-tested.

Validation

  • Mechanical. make -C services/api verify runs the canon test against all four runbook roots; passes on a clean tree, fails on any reordering or missing section.
  • Empirical. Pick three on-call engineers who have not seen this RFC. Ask each to find «how to confirm the incident is over» on a runbook outside their own service. Stopwatch from page-open to correct-section-found. Target: under 10 seconds for all three. Today: 30–60 seconds with two of three giving up before finding it.
  • Postmortem signal. No postmortem in the next two quarters mentions «could not find the rollback / exit step» as a contributing factor. (Current baseline: 3 of last 5 postmortems do.)

Roll-out

  • Week 1 — Phase 1 lands. Canon test on xfail, drift visible but non-blocking. Announce in the change feed.
  • Weeks 2–3 — Phase 2 sweep. One PR per service: api, portal, monitoring, sre-handbook. Reviewer per PR is the service owner. Expected drift: ~25 pages touched.
  • Week 4 — Phase 3 hard gate. Remove xfail markers. Watch CI for one week — any genuine regressions go back through Phase 2.
  • Week 5 — Phase 4 template lockdown. Visual corpus expands with the runbook pair.
  • Rollback strategy. Reverting the canon test removes the gate without touching content. The fix-up commits are independent — each can be reverted in isolation if its service team objects to the result.

Change management

  • Adding a new section to the canon. Open a new RFC superseding this one. The canon set is small precisely because adding to it is expensive — every existing runbook needs the new section, or the gate has to xfail the absences.
  • Service-specific extensions. A service may add extra <h2> sections after Follow-up (typical: service-specific runbook patterns, links to service playbooks). The canon test ignores anything beyond the seven required ids; only their presence and order is enforced.
  • Deprecating a runbook. Move it to archive/runbooks/ with a lifecycle pill set to retired. Archived runbooks are excluded from the canon test scope.
