ADR 0042: Cache hot-path reads with Redis

Introduce a Redis cache for the three highest-RPS read endpoints (GET /catalog/items, GET /user/me, GET /search). Targets a 60 % drop in p95 latency on the cached paths under nominal traffic.

Context

Three read endpoints currently serve ~70 % of total request volume but spend ~85 ms in the database layer at p95 — dominated by joins against the catalog_items and user_profile tables. The traffic is read-heavy (~98 % reads), tolerates seconds-level staleness, and is bounded by a small set of cache keys (under 50 K hot keys at any one time). We're at the latency budget for the public SLO and a planned 3× traffic ramp in Q3 will push us over.

Decision

Stand up a single-shard Redis 7.x deployment behind a read-through cache layer in the API service. Cache TTL is 60 s for catalog reads, 15 s for user reads, 5 s for search results. Cache writes happen on-miss only; eviction is LRU.

Scope

  • In scope: three endpoints listed above, the repository-layer cache wrapper, observability counters (hit / miss / latency), the Redis bootstrap in docker-compose.yml.
  • Out of scope: write-through caching, multi-region replication, cache invalidation on user mutations (covered by ADR 0043), general session storage (different access pattern).

Alternatives considered MANDATORY · MIN 2

  1. In-process LRU cache (cachetools)
    • Rejected because: bytes-per-instance scales with replica count; consistency across replicas is impossible without a coordination layer we'd have to invent.
    • Pros considered: zero-ops, sub-ms hit latency, no extra process.
    • Cons that decided it: hot key duplicated 6× across replicas wastes RAM; cold-replica starvation after any deploy.
  2. CDN-level edge cache (CloudFront / Fastly)
    • Rejected because: requires Vary headers on JWT-bearer requests, which fragments the cache by per-user tokens and erases the hit-rate.
    • Pros considered: zero RPS on the API for cached objects.
    • Cons that decided it: cache key explosion + authenticated requests are 100 % of the hot-path traffic.
  3. Materialised views in PostgreSQL
    • Rejected because: refresh latency (30 s+ on each REFRESH) shifts the cost rather than removing it; the staleness story is worse than Redis TTLs.
    • Pros considered: no new infrastructure component.
    • Cons that decided it: refresh contention with write traffic during peak hours; observability blind-spot during refresh.

Consequences

Positive

  • p95 latency on the three endpoints drops ~60 % (load-test projection): 85 ms → 33 ms.
  • Headroom for the Q3 traffic ramp without horizontal DB scale-out.
  • Cache-hit ratio is a leading indicator of new query regressions (becomes a Grafana SLI).

Trade-offs

  • One new operational surface — Redis HA, backups, failure scenarios. Runbook 0014 — Redis cache failing added in the same PR.
  • Stale-read window of up to 60 s on the catalog endpoint; flagged in the public API docs.
  • Adds a deployment-order constraint: Redis must be reachable before the API container starts accepting traffic.

Compatibility and migration

  • Backward-compatibility impact: none — consumer-visible API contract unchanged; only latency changes.
  • Migration plan: ship the cache layer behind a CACHE_ENABLED feature flag (default off), enable for 5 % of traffic for 24 h, observe hit ratio + error rate, ramp to 100 % over 48 h.
  • Rollback strategy: flip the feature flag off — the wrapper is bypass-able without code rollback. Redis process can be stopped independently; the API degrades to direct-DB reads.

Implementation plan

  1. Add the cache wrapper to the repository layer (services/api/app/cache/redis_cache.py) with a contract test verifying TTL behaviour.
  2. Wire CACHE_ENABLED through the config-profile machinery (ADR 0010) — defaults to false in dev / qa, gated rollout in prod.
  3. Add Grafana panels for cache hits / misses / latency to the Study App Observability dashboard (per ADR 0009).
  4. Update the public API documentation with the staleness window.

Validation

  • Technical checks: contract test tests/cache/test_redis_cache.py covers hit / miss / TTL expiry / connection-failure fallback to DB read.
  • Operational checks: Runbook 0014 — Redis cache failing covers the new failure surface; Grafana dashboard panels flag < 80 % hit ratio.
  • Documentation updates: README.md (operational deps), api/reference/caching.html (staleness window), public API docs.
