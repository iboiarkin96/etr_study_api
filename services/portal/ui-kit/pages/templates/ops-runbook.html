0042 — Portal build failing

Page builds error out on main and the published portal is stuck on the last good commit. Reader impact: stale docs. Author impact: cannot ship updates until the build is recovered.

Metadata

Runbook
0042 — Portal build failing
Category
CI
Severity
SEV-2
Owner
Portal team
Last reviewed
2026-06-12

Impact

  • Published portal is stuck on the previous green commit; readers see stale documentation until the build clears.
  • Authors cannot ship updates — every queued PR is blocked behind the failing job.
  • No production user impact unless the API-facing pages are part of the failing surface.

Trigger

  • make docs-check fails locally.
  • CI status on main turns red on the portal-build job.
  • Author reports «my page didn't publish» after a green PR merge.

Fast triage

  1. Open the failed CI run and scroll to the first ERROR line: 
    gh run view --log-failed
  2. Reproduce locally: make -C services/portal docs-check.
  3. Classify the error — HTML structure validation? Sidebar JSON parse? Pagefind index build? Each has a different recovery path below.

Most common causes

  • An author-edited page failed HTML structure validation (unclosed tag, missing data-page-type, broken heading hierarchy).
  • Sidebar JSON manifest hand-edited and shipped with broken syntax.
  • A page renamed without updating cross-links — link checker fails.
  • New page shipped without make docs-fix — Pagefind index is out of date.
  • Asset path drift after a kit rename — check_asset_refs.py flags missing files.

Recovery steps

Revert if a single commit is responsible

  • Identify the breaking commit on the failed portal pages: git log -- services/portal/internal/.
  • Revert and push to main:
git revert <sha>
git push origin main

Verify: the next CI run is green and the affected pages serve from the published portal.

Patch forward when revert would break other pages

  • Apply the smallest targeted fix on a hotfix branch.
  • Re-run the docs pipeline: make -C services/portal docs-fix.
  • Confirm clean: make -C services/portal docs-check.
  • Merge directly to main.

Pagefind / asset failures

  • Rebuild the search index: make -C services/portal docs-fix (re-runs Pagefind as the last step).
  • Verify visibility: python tools/docs/check_pagefind_visibility.py.

Exit criteria

  • make -C services/portal docs-check passes locally.
  • CI portal-build job is green on main.
  • The previously failing pages are reachable on the published portal at the expected URLs.

Escalation

If 30 minutes of triage didn't restore the build, or the failure mode isn't on the «Most common causes» list:

  1. Capture state — failed CI run URL, make docs-check 2>&1 | tee /tmp/runbook-0042-state.txt, the offending commit hash, and the recovery steps already tried.
  2. Open a SEV-3 incident in the team backlog and link this runbook from it. Portal blackout on the public surface jumps to SEV-2 — readers hit broken pages from external referrers within minutes.
  3. Page on-call (or, for solo work, the project incident commander) with the captured state and the last known-good commit hash.
  4. If the outage is config-only (sidebar JSON, asset path, Pagefind config), revert the offending commit before debugging further — every minute of stuck portal compounds reader confusion.

Follow-up

  • If the fix changes docs policy (templates, validators, gates), add or update an ADR.
  • If it is a new failure pattern, add a short note to the «Most common causes» list above.
  • Add the missed check to the PR review checklist if it would have caught the regression earlier.
  • Open a postmortem if the incident escalated to SEV-2 (how to write one).
