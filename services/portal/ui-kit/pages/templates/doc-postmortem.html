API connection-pool exhaustion on 2026-06-14 (SEV-2, 42 min)

At 14:18 UTC on 2026-06-14, the API service stopped serving new requests because its SQLAlchemy connection pool was fully checked out and no slot freed in time. p95 latency climbed from 110 ms to 4.2 s before requests started failing with HTTP 503. Root cause: a slow N+1 query on the new /catalog/search endpoint held connections for 6–9 s each under nominal traffic, and the default pool of 5 + overflow 10 was never sized for that surface.

Summary

A search endpoint shipped two days earlier (GET /catalog/search?q=…) issued an N+1 query against catalog_items + item_tags: one query per result row to fetch tags. Under regular traffic the endpoint averaged 1.3 q/s with 80–120 tags per item, so each request held a DB connection 6–9 s. The pool (5 base + 10 overflow) saturated within 11 minutes of the morning traffic ramp, every new request blocked on pool.checkout() for the full 30 s timeout, then returned 503. Recovery was triggered manually: a feature flag rolled the endpoint back to a stub response, the pool drained in 90 s, and traffic recovered.

Impact

  • Duration: 42 minutes — 14:18 to 15:00 UTC. Mitigation at 14:55, full recovery at 15:00.
  • User-visible: ~2 100 failed requests across all endpoints (not just /catalog/search — the pool was global). 5xx error rate peaked at 8.4 %.
  • Error-budget burn: 14× the nominal hourly burn rate during the window — about a third of the 30-day budget consumed in 42 minutes.
  • Customer signals: two support tickets opened, no SLA breach (the SLA is monthly).

Timeline

  1. 14:18 UTC — pool saturation begins; pool_checkout_seconds climbs from p50 ≈ 5 ms to p50 ≈ 30 s within four minutes.
  2. 14:22 UTC — first 503 returned. Burn-rate alert fires (14× threshold).
  3. 14:24 UTC — on-call paged. Initial hypothesis: database is down (it wasn't).
  4. 14:31 UTC — Grafana Connections panel shows pool fully checked out. Hypothesis shifts to slow queries.
  5. 14:38 UTC — pg_stat_activity confirms ~14 long-running queries from /catalog/search. Cause located.
  6. 14:55 UTC — feature flag CATALOG_SEARCH_ENABLED flipped off; endpoint returns 503 immediately, no DB hit. Pool starts draining.
  7. 15:00 UTC — error rate back to baseline. Incident closed.

Root cause

The endpoint was shipped without a load test against a representative tag-cardinality distribution. The query path went SELECT items … + N SELECT tags WHERE item_id = … calls inside a single SQLAlchemy session — a textbook N+1 the relationship loader strategy should have flagged. The relationship was defined as lazy="select", the default, which produces the per-row query. No joinedload or selectinload hint was applied, and there was no integration test that asserted query count per request.

Why it slipped review: PR ran a contract test that asserts the response shape, not the SQL count. The reviewer (architect) noted «consider selectinload» as a comment but did not block the merge.

Resolution

  • Immediate (14:55). Disabled the endpoint via the feature flag. Direct mitigation, no code change required.
  • Same day (15:30). Switched the relationship to selectinload in services/api/app/models/catalog.py; one new query per request instead of N. Re-enabled the endpoint behind a 1 % traffic shadow.
  • 2026-06-17. Ramped to 100 % traffic after 72 h of clean pool-checkout metrics.

Detection

  • What detected it. The burn-rate alert (14× hourly on the 5xx ratio) — fired four minutes into the incident. That's the right signal at the right time.
  • What did not detect it. The connection-pool dashboard panel was not paged on; only Grafana-watched. Engineers had to open Grafana manually after the burn-rate page. Adding a paging alert on pool_checkout_seconds_p99 > 5 s for 2 min would have cut MTTD by ~7 min.
  • What should detect it next time. A pre-merge load-test gate that exercises the new endpoint at 5× nominal QPS for 60 s — captures pool saturation before traffic does.

Lessons

  • Shape tests are not load tests. The contract test passed (response shape correct) and the endpoint was 6 s under nominal load. Asserting what comes back isn't asserting how it gets there.
  • «Consider X» is not a review. The architect's comment on the PR was correct. The review pattern that turns a comment into a block is what was missing — a checklist row, not better judgement.
  • Feature flags justify their cost in one incident. The endpoint had a flag because the architect insisted during design review. Without it, the rollback would have been a Git revert + deploy — easily 20 more minutes.
  • One alert that pages, one dashboard that doesn't. The pool dashboard was the right diagnostic, but the alert had to come from somewhere else (burn rate). Two layers, two purposes; both needed.
