Resume banner

Sticky banner at the top of long documentation pages that offers a one-click jump back to where the reader left off. Built automatically on return visits when the previous scroll position is fresh (≤ 14 days) and meaningful (≥ 25 % into the page). One-shot — once actioned or dismissed it stays gone for the rest of the tab.

Source status. This component currently ships as legacy assets at services/frontend/portal/assets/docs-resume.{css,js} — it has not yet been ported to a UI Kit v3 components/ peer. This page is the v3 spec the future port will conform to; the contract below matches the legacy code today.

Example

Static preview of the banner. Live behaviour requires a saved scroll position from a previous visit to the same path.

Continue reading? You stopped here last time on this page.

When to use

  • Long-form documentation that readers often return to mid-read (multi-screen explanations, RFCs, runbooks, API reference deep dives).
  • Pages whose scrollY persistence has product value — short FAQs or landing pages should not load the script.
  • Do not use as a generic toast or notification surface — the role is specifically "resume your previous reading position".

Anatomy

  1. Container — div.docs-resume-banner[role="status"] appended directly to document.body on DOMContentLoaded. position: sticky; top: 12px; so it stays visible at the top of the viewport until dismissed.
  2. Message — div.docs-resume-banner__text: "Continue reading? You stopped here last time on this page." Min-width 220 px so the buttons don't crowd it on narrow viewports.
  3. Actions — div.docs-resume-banner__actions with two buttons:
    • button.docs-resume-banner__btn--primary[data-resume-go] — "Jump to last position". Calls window.scrollTo({ top: savedY, behavior: "smooth" }) then dismisses.
    • button.docs-resume-banner__btn[data-resume-dismiss] — "Dismiss". Sets the session-storage seen flag, plays the leave transition, removes the node.
  4. Leave transition — .is-leaving sets opacity: 0; transform: translateY(-6px); over 180 ms, then the node is removed.

Markup

Authors don't write this markup. The runtime builds it from docs-resume.js when the offer conditions are met (see Reveal conditions).

<div class="docs-resume-banner" role="status">
  <div class="docs-resume-banner__text">
    <strong>Continue reading?</strong> You stopped here last time on this page.
  </div>
  <div class="docs-resume-banner__actions">
    <button type="button"
            class="docs-resume-banner__btn docs-resume-banner__btn--primary"
            data-resume-go>Jump to last position</button>
    <button type="button"
            class="docs-resume-banner__btn"
            data-resume-dismiss>Dismiss</button>
  </div>
      </div>

Reveal conditions

The banner is offered only when all of the following are true (any failure short-circuits the build):

  1. sessionStorage["docs.resume.seen.<path>"] is not "1" — the reader hasn't actioned or dismissed the banner in this tab.
  2. localStorage["docs.resume.<path>"] exists and parses as { y: number, t: number }.
  3. The saved position is fresh: Date.now() − saved.t ≤ 14 days.
  4. The page has scrollable extent: scrollHeight − innerHeight > 0.
  5. The saved position is meaningful: saved.y / extent ≥ 0.25.
  6. The saved position isn't at the very bottom: saved.y ≤ extent − 40.

Storage

KeyStorageValueLifecycle
docs.resume.<path> localStorage JSON { y: number, t: number } — scroll Y in px, write timestamp in ms. Written by a throttled scroll listener (750 ms floor between writes, 100 ms tail debounce) and once more on beforeunload. Read on DOMContentLoaded for the offer decision. TTL 14 days.
docs.resume.seen.<path> sessionStorage "1" Set when the banner is dismissed or its primary action is invoked. Cleared on tab close.

<path> = location.pathname with everything outside [A-Za-z0-9_/-] replaced by _. Storage is per-path so reading two unrelated docs in the same tab keeps independent positions.

States

  • Not offered — one or more reveal conditions failed; no DOM node is ever created.
  • Visible — banner is in the document; primary + dismiss buttons clickable.
  • Leaving — class .is-leaving applied; opacity: 0 + translateY(-6px) over 180 ms, then node removed and session flag set.
  • Suppressed for this tab — session flag is "1"; the next reload will not re-offer until the tab is closed.
  • Expired — saved timestamp is older than 14 days; maybeOfferResume() returns without building.

A11y

  • Container is role="status" — screen readers announce its content on first paint without an explicit aria-live ratchet.
  • Buttons are real <button type="button"> elements with text labels; both are reachable by Tab in document order. Enter/Space activates either.
  • The leave transition is opacity + translateY(-6px) only — short enough to skip a dedicated prefers-reduced-motion rule, but a future port should add @media (prefers-reduced-motion: reduce) for parity with other UI Kit components.
  • The component does not move focus. If keyboard users want to act on it, they reach it via normal tab order; the banner does not steal focus from the page heading on mount.

Do / Don't

DoDon't
Load the runtime only on long pages where saved scroll has product value. Each page that loads it claims a localStorage key. Load it on landing pages, FAQs, or anything fully visible at scrollY = 0. The reveal conditions will short-circuit anyway, but the script still writes scroll snapshots.
Trust the 14-day TTL and the 25 % reveal floor — both exist to suppress the banner when the saved position is stale or trivially close to the top. Shorten the TTL or threshold ad-hoc for one page. Diverging values across pages produce inconsistent reader experience.
Treat the session-storage seen flag as the dismissal contract — once true the banner stays away for the rest of the tab. Re-mount the banner manually after dismissal. The runtime intentionally won't show it twice in the same tab.
When porting to v3 component CSS, rebase the palette onto the standard UI Kit tokens. See the Tokens used mapping. Keep the hard-coded sky-blue palette in v3 — it predates the token system and will drift on theme changes.
