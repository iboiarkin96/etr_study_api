Tooltip

Inline hover/focus hint for controls and glossary terms. One JS-managed bubble is portalled to <body> so it never gets clipped by overflow:hidden ancestors (tables, scrollers, sticky chrome). Triggers are any element with data-tooltip="…" — buttons, chips, badges, icons, headers, terms in prose.

When to use

  • Icon-only buttons — theme toggle, bug report, copy button, drawer trigger. The tooltip carries the label the icon can't.
  • Chips / pills / status badges — short labels whose meaning is non-obvious (e.g. retired, beta, pilot).
  • Column headers / cockpit fields — RFC weight, KPIs, ETR — where one word in the heading hides a definition.
  • Glossary terms in prose — add the docs-tooltip class to get the dotted underline + inverted dark bubble so readers see “lookup available”.

Do not use a tooltip for content that must be read in flow, for error messages, or for paragraph-long explanations — those go in inline prose, a callout, or a popover.

API

There is no JS API to call — tooltips are declarative. The runtime mounts once and observes the DOM for new triggers.

<!-- 1. Plain hint (default) — light bubble, no underline on the trigger -->
<button type="button" data-tooltip="Publish this draft to staging.">Publish</button>

<!-- 2. Glossary term — dotted underline + inverted bubble -->
<span class="docs-tooltip" data-tooltip="Estimated Time to Resolution.">ETR</span>

<!-- 3. Force a placement (default is top with auto-flip) -->
<button type="button"
data-tooltip="Open the navigation drawer."
      data-tooltip-placement="bottom">Menu</button>

Non-focusable hosts (<span>, <div>) automatically receive tabindex="0" so keyboard users can reach the hint — you don't have to add it manually.

Attributes

  • data-tooltip="…" — required. Plain text. Keep it under ~10 words.
  • data-tooltip-placement="top | bottom | left | right" — preferred placement. The runtime still flips if that side would clip; this just sets the first candidate.
  • class="docs-tooltip" — opt-in glossary styling on the trigger (dotted underline + inverted dark bubble).

Placement & flipping

The bubble is positioned with position: fixed. Default placement order is top → bottom → right → left; the runtime picks the first placement that fits within the viewport (minus an 8 px pad). The arrow is shifted horizontally/vertically to remain centred under the trigger when the bubble itself has been pushed against a viewport edge.

Set data-tooltip-placement to bias the order — e.g. tooltips on the right edge of the topbar should prefer bottom.

A11y

  • Reveals on both hover and keyboard focus. Mouse-only tooltips are a footgun — keyboard users get nothing.
  • Bubble has role="tooltip"; the trigger receives aria-describedby while the tooltip is visible, so screen readers announce the hint after the trigger label.
  • Esc dismisses while the trigger is focused.
  • Non-interactive hosts get tabindex="0" so they enter tab order.
  • prefers-reduced-motion — the slide/scale animation collapses to a plain opacity flip via tokens/_base.css.

Markup (rendered into <body>)

<div id="docs-tooltip-bubble" class="docs-tooltip-bubble"
role="tooltip" data-placement="top" style="left:…px; top:…px">
<span class="docs-tooltip-bubble__content">Publish this draft to staging.</span>
<span class="docs-tooltip-bubble__arrow" aria-hidden="true"></span>
      </div>

One shared bubble is created lazily; the same node is repositioned and re-content'd as the user moves between triggers.

States

  • Idle — opacity: 0, off-screen; nothing announced.
  • Showing — 160 ms delay then fade + 2 px slide-up. The delay debounces accidental hovers when the cursor crosses chips quickly.
  • Visible — bubble is positioned next to the active trigger; arrow centred on the trigger.
  • Hiding — 80 ms grace after pointerout / focusout so moving the cursor between trigger and bubble doesn't flicker.

Tokens used

  • --card, --line, --text, --bg (glossary variant inverts to --text/--bg)
  • --radius-sm, --shadow-md
  • --fs-200, --lh-snug
  • --ease-out

Do / Don't

  • Do use a tooltip for an icon-only control or an unfamiliar abbreviation in a column header.
  • Do keep the copy under ~10 words and free of links / formatting (tooltips don't support rich content by design).
  • Don't hide essential information inside a tooltip — if a user must read it to complete a task, surface it in the page.
  • Don't set title on the same element — the native browser tooltip will fight with this component. Use data-tooltip only.
  • Don't use a tooltip as an error message — show inline next to the field and announce via aria-live.

Where it's wired automatically

Several runtime components attach data-tooltip on the elements they render — you get the bubble for free without adding markup:

  • Topbar — theme toggle, bug-report button, drawer trigger, search input.
  • Sidebar — collapse toggle, section carets.
  • Code blocks — Copy button.
  • Diagram lightbox — zoom in/out, reset, close.
  • TOC FAB — open table of contents (mobile).
